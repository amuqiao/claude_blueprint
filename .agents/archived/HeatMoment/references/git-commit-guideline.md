# Git Commit Guideline

## 文档职责

- 本文档用于约束 `HeatMoment` 仓库的提交边界、提交信息写法与提交前最小检查。
- 仅在任务涉及 `git commit`、拆分提交边界、生成 commit message 或提交前自检时读取。

## 1. 提交边界

- 一次提交只处理一个明确主题。
- 不把无关文档、功能实现、规则调整混在同一次提交里。
- 如果改动跨多个主题，先拆分，再提交。

## 2. 提交类型

- `docs:` 文档、规则、说明更新
- `feat:` 新功能或可感知能力增加
- `fix:` 缺陷修复或行为纠正
- `refactor:` 不改变外部行为的结构整理
- `chore:` 零碎维护性调整

## 3. 提交信息写法

- 提交信息优先写“改了什么”，再写对象。
- 保持简短、可扫描，不写空泛标题。

示例：

- `docs: 收敛 heatmoment 0-1 development 文档`
- `feat: 搭建 SwiftData 基础模型与 mock data`
- `fix: 修正首页筛选与 chip 状态不同步`

## 4. 提交前检查

- 确认改动范围和提交主题一致。
- 确认相关入口文档或规则文件已同步更新。
- 完成最小必要验证；无法验证时明确说明原因和剩余风险。

## 5. 默认约束

- 非明确要求下，不做 `amend`。
- 非明确要求下，不改写历史。
- 不为了凑提交而拆出没有独立意义的碎提交。
