# Workflow Guideline

## 文档职责

- 本文档用于承载任务推进与协作判断规则的真源定义。
- 它回答的是“任务应该怎么推进、什么时候先分析、什么时候直接执行、如何给出判断”。

## 适用范围

- 需要决定任务推进方式、分析顺序或协作表达方式时读取。
- 需要判断是否应先澄清边界、是否应给推荐方案、是否应扩展范围时读取。
- 不处理用户背景、测试验证、安全边界或命名细则。

## 核心规则

- 新功能开始前先澄清需求，不做猜测性实现。
- 涉及系统设计、模块边界或长期演进的问题时，先明确职责、边界与模块分工，再进入实现。
- 纯实现任务或明确执行任务，不额外抬高抽象层级。
- 设计类或架构类问题，优先给出推荐方案；如存在合理备选，补充备选方案与权衡分析。
- 不要用“两者都可以”“看情况”替代明确判断；如果结论依赖前提，先说清前提，再给结论。
- 分析实现问题时，优先以代码和实际行为为准；代码是真相，文档是快照。
- 发现超出当前目标的问题时，先记录为 TODO 或风险，不默认扩展范围。

## 与运行时规则的关系

- `rules/workflow.md` 只保留运行时短摘要。
- 如果未来需要阶段化流程、任务分流或更复杂的判断树，应在本文件扩展，而不是把复杂流程直接写进 `rules/`。

## 缺失或冲突时怎么处理

- 如果 `rules/workflow.md` 与本文件冲突，以本文件为准，再回写运行时摘要。
- 如果某条规则实际属于测试、安全或文档表达，应迁回对应真源，不混入本文件。
