# Terminology Normalization Guideline

## 文档职责

- 本文档用于承载术语纠偏与专业表述重述规则的真源定义。
- 它回答的是“当用户对 `SwiftUI`、前端、iOS 交互、设计或实现术语表述不够准确时，应如何降低理解偏差”。

## 适用范围

- 讨论 `SwiftUI`、前端、iOS 交互、原型、页面结构、设计实现时读取。
- 需要判断是否应沿用用户原话、是否应先重述专业表达时读取。
- 不处理任务推进、测试验证、安全边界或 Git 提交。

## 核心规则

- 当用户术语不够准确时，不要机械沿用原话；先按上下文重述为更准确的专业表达，再继续分析、写文档或执行任务。
- 术语纠偏的目标是降低理解偏差，而不是只纠正字面说法；应优先保留用户真实意图。
- 如果存在更标准的术语，优先同时给出“自然语言说法 + 对应专业术语”。
- 在原型、交互和页面结构讨论中，优先澄清交互语义，再讨论技术实现方式。
- 如果问题本质是信息架构或交互流，不要误写成视觉风格问题；如果问题本质是实现约束，不要只停留在交互命名层。
- 对 `SwiftUI`、前端和 iOS 领域的解释，优先使用用户易理解的后端类比或工程类比，再逐步补充专业术语。

## 常见触发场景

- 把页面结构、信息架构、导航关系混称为“UI”或“样式”。
- 把交互语义、组件职责、布局方式混在同一层描述。
- 使用不标准但可推断的 `SwiftUI`、前端或 iOS 术语。
- 先讨论技术实现，但交互语义和页面职责其实尚未对齐。

## 与运行时规则的关系

- `rules/user-background.md` 只保留术语纠偏的最小摘要，不展开长解释。
- `user-background-guideline.md` 负责定义“为什么需要术语适配”和“默认如何面向该用户解释”。
- 本文件负责术语纠偏本身的专项规则；如果后续示例、反例和边界继续增多，应继续扩展本文件，而不是堆回 `rules/`。

## 缺失或冲突时怎么处理

- 如果 `rules/user-background.md`、全局 `AGENTS.md` 与本文件冲突，以本文件为准，再回写摘要。
- 如果某条内容更像用户背景事实，而不是术语纠偏规则，应迁回 `user-background-guideline.md`。
