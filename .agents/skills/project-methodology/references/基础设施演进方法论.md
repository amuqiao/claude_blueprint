# 基础设施演进方法论

> 用于在项目从 POC 到正式迭代的过程中，判断基础设施何时应保持最轻方案、何时值得升级，以及如何提前留出演进缝而不过度设计。

> 主线框架：`Point -> Current Need -> Minimal Solution -> Evolution Seam -> Trigger -> Upgrade -> Verify -> Governance`

## 1. 思考顺序

- 判断基础设施问题时，先按 `Point -> Current Need -> Minimal Solution -> Evolution Seam -> Trigger -> Upgrade -> Verify -> Governance` 的主线顺序思考
- `Point`：先确认当前面对的是一个真实阻塞、一个增长信号，还是只是对未来的担忧
- `Current Need`：先判断当前阶段真正要解决的问题是什么，而不是直接讨论终局架构
- `Minimal Solution`：先选择当前最轻、最小、足够可行的方案
- `Evolution Seam`：再判断哪些地方需要提前留出演进缝，避免后面重写所有模块
- `Trigger`：再明确什么信号出现时，当前方案已经不够用
- `Upgrade`：只有在触发条件成立后，才进入架构或基础设施升级
- `Verify`：升级后验证是否真的解决问题，而不是只是复杂度上升
- `Governance`：最后回写文档、更新约束、明确新旧阶段边界
- `Decision`：不是默认主线；只有在存在多种合理升级方向时，才作为可选分支进入比较与判断
- 不要把“未来可能会复杂”当成“现在必须做重”的理由

## 2. 核心方法

- 先满足当前需求，再考虑下一跳
- 先留演进缝，不要先上终局方案
- 先识别触发信号，再决定是否升级
- 先让模块依赖能力边界，再让它依赖当前实现
- 不要用过度设计替代真正的演进判断

## 3. 当前问题判断

- 第一阶段不要默认当前必须升级基础设施
- 先确认当前问题更像：
  - 一次性 POC 验证需求
  - 当前实现的真实阻塞
  - 性能 / 并发 / 稳定性问题
  - 维护复杂度问题
  - 只是对未来规模的想象
- 如果当前只是“未来也许会大”，通常还不值得直接做重
- 不要把“以后可能需要”直接等同于“现在就应该上”

## 4. 最轻方案优先原则

- 默认优先选择当前阶段最轻、最小、最容易验证的方案
- POC 阶段优先解决“可不可行”，不是“终局是否优雅”
- 只有当当前方案已经明显妨碍主路径时，才考虑升级
- 不要在用户登录注册、极轻后台任务、低并发场景下预设重基础设施
- “当前最轻可行”不是偷懒，而是避免把复杂度提前引入

## 5. 演进缝意识

- 不要一开始就把实现写死到当前基础设施细节
- 优先让模块依赖：
  - 能力边界
  - 接口契约
  - 任务提交能力
  - 状态查询能力
  - 事件或结果边界
- 当前不一定上队列，但应避免让模块直接依赖“只能同步完成”的假设
- 当前不一定上 Kafka，但应避免把所有模块都绑死在某个单点实现上
- 演进缝的目标不是预实现未来，而是降低未来升级时的改动半径

## 6. 升级触发原则

- 升级应由真实信号触发，而不是由想象触发
- 常见触发信号包括：
  - 同步请求已明显拖慢主路径
  - 任务执行时间已超出同步接口可接受范围
  - 并发量增长导致稳定性明显下降
  - 失败重试、任务追踪、任务解耦已成为真实需求
  - 当前实现方式让新模块持续以高成本接入
- 只有在信号持续存在时，升级才值得进入正式讨论

## 7. 升级方式原则

- 升级时，优先选择“下一跳合理方案”，不要默认一步到终局
- 典型升级路径可以是：
  - 同步请求响应
  - 后台任务
  - 队列驱动任务
  - 更成熟的任务系统 / 消息系统
- 每次升级只解决当前最主要的问题，不要顺手引入一整套额外复杂度
- 不要因为要升级一处能力，就把全项目一起重构到最重形态

## 8. 模块影响意识

- 基础设施升级会影响模块开发思路，但不应该让模块层持续直接依赖升级细节
- 模块应尽量依赖能力抽象，而不是依赖“当前用的是同步 / Celery / RabbitMQ / Kafka”
- 如果某次升级会导致大量模块一起改，通常说明演进缝没有留好
- 判断基础设施是否健康，一个重要标准是：新模块接入成本是否仍可控

## 9. 验证与治理意识

- 升级完成后，必须验证：
  - 是否真的解决了原问题
  - 是否引入了新的系统复杂度
  - 是否让模块接入成本下降或至少没有恶化
  - 是否需要更新架构文档、模块约束和开发规则
- 不要把“已经上了更高级中间件”误当成升级成功

## 10. 可视化原则

- 可视化服务理解，不服务装饰
- 涉及演进路径时，优先用阶段链路图、文本树或对比表
- 涉及新旧方案差异时，优先用对比表
- 涉及能力边界时，优先用“模块 -> 能力接口 -> 当前实现”的结构化表达
- 如果没有图会明显抬高理解成本，应明确补图

## 11. 禁止行为

- 不要因为“以后可能会用到”就一开始上重基础设施
- 不要把架构终局当成 POC 起点
- 不要把模块直接绑死在当前实现细节上
- 不要没有真实触发信号就升级
- 不要升级后只留下新组件，不回写边界、规则和文档

## 12. 默认输出要求

- 输出基础设施演进判断时，优先体现：
  - 当前问题是什么
  - 当前最轻可行方案是什么
  - 需要留哪些演进缝
  - 哪些信号会触发下一跳升级
  - 当前是否已经值得升级
  - 升级后要验证什么
  - 升级后应回写什么
- 如果信息不足，先暴露缺口，再继续展开
- 如果存在多个合理升级方向，应先给推荐路径，再说明备选

## 13. 分阶段 Prompt 示例

这一节不是替代上面的规则，而是给出可直接使用的参考模板。

推荐用法：

1. 先把这份规则文档作为高层约束
2. 再按下面的阶段 prompt 去跑具体基础设施问题
3. 默认优先按主线分阶段跑，不要一上来直接讨论 Kafka、RabbitMQ 这类终局名词

### Phase 0：Point

适用场景：

- 你感觉当前架构可能不够了
- 但还不确定这是真问题，还是对未来的焦虑

进入本阶段前，至少确认：

- 当前确实有一个基础设施相关的担忧或阻塞
- 你现在缺的是“要不要升级”的判断

何时足以进入下一阶段：

- 已经知道当前问题是现实阻塞，还是未来想象
- 已经知道下一步应进入 `Current Need`，还是暂时先停

```text
请先不要直接推荐 Kafka、Celery、RabbitMQ 或其他重型方案。

请遵守我提供的《基础设施演进方法论》，先按 Point 阶段帮我判断：我当前这个基础设施问题，是否已经值得进入演进讨论。

当前问题是：
【把你的问题、担忧或信号贴在这里】

请按下面顺序输出：
1. 这更像真实阻塞，还是未来担忧
2. 当前是否已经值得进入演进判断
3. 如果值得，真正的问题是什么
4. 如果还不值得，当前最该继续观察什么
```

### Phase 1：Current Need

适用场景：

- 已经确认当前问题不是纯想象
- 需要先判断当前真正的需求是什么

进入本阶段前，至少确认：

- 当前已经存在真实信号
- 当前需要先澄清问题，而不是直接给中间件名字

何时足以进入下一阶段：

- 已经知道当前最主要的问题到底是延迟、并发、解耦、稳定性还是维护成本
- 已经知道问题影响的是哪条主路径

```text
请遵守我提供的《基础设施演进方法论》，按 Current Need 阶段帮我判断当前真正的问题是什么。

当前背景是：
【把当前实现方式、主路径、问题表现贴在这里】

请按下面顺序输出：
1. 当前真正的问题是什么
2. 它影响的是哪条主路径
3. 当前问题最像性能、并发、稳定性、解耦，还是维护成本问题
4. 如果现在就讨论升级，最容易误判的点是什么
```

### Phase 2：Minimal Solution

适用场景：

- 当前问题已经清楚
- 需要先判断当前最轻可行方案

进入本阶段前，至少确认：

- 当前问题已经被明确
- 当前不是直接追求终局方案，而是找当前最轻可行方案

何时足以进入下一阶段：

- 已经知道当前最轻方案是什么
- 已经知道它能解决什么、解决不了什么
- 已经知道下一步需要留哪些演进缝

```text
请遵守我提供的《基础设施演进方法论》，按 Minimal Solution 阶段帮我判断当前最轻可行方案。

当前问题是：
【把 Current Need 阶段结果贴在这里】

请按下面顺序输出：
1. 当前最轻可行方案是什么
2. 为什么它比更重方案更适合当前阶段
3. 它能解决什么
4. 它解决不了什么
5. 当前最该避免的过度设计是什么
```

### Phase 3：Evolution Seam

适用场景：

- 已经决定先用轻方案
- 但希望未来升级时不要重写全部模块

进入本阶段前，至少确认：

- 当前最轻方案已经成立
- 当前需要考虑的是“如何留缝”，不是“如何提前上重”

何时足以进入下一阶段：

- 已经知道模块层应该依赖哪些能力边界
- 已经知道哪些地方不能写死在当前实现上
- 已经知道后续升级时改动半径应控制在哪

```text
请遵守我提供的《基础设施演进方法论》，按 Evolution Seam 阶段帮我判断当前应该留哪些演进缝。

当前最轻方案是：
【把 Minimal Solution 阶段结果贴在这里】

当前模块或系统背景是：
【把模块接入方式、接口、任务流等贴在这里】

请按下面顺序输出：
1. 当前应该留哪些演进缝
2. 模块层最不应该依赖哪些实现细节
3. 哪些能力边界应先抽出来
4. 如果不留这些缝，未来升级会出现什么问题
```

### Phase 4：Trigger

适用场景：

- 已经有轻方案
- 需要明确什么信号会触发升级

进入本阶段前，至少确认：

- 当前方案已经在运行
- 当前需要明确升级触发条件，而不是提前升级

何时足以进入下一阶段：

- 已经知道哪些信号出现时才值得升级
- 已经知道这些信号如何被观察和确认

```text
请遵守我提供的《基础设施演进方法论》，按 Trigger 阶段帮我判断当前方案的升级触发条件。

当前方案是：
【把当前实现方式贴在这里】

当前系统背景是：
【把并发、任务量、失败率、响应时长、维护问题等贴在这里】

请按下面顺序输出：
1. 哪些信号会触发下一跳升级
2. 哪些信号只是噪音，不应过早触发升级
3. 当前最值得持续观察的指标或现象是什么
4. 当前是否已经触发升级条件
```

### Phase 5：Upgrade

适用场景：

- 触发信号已经成立
- 现在要判断升级到哪一跳最合理

进入本阶段前，至少确认：

- 升级触发条件已经基本成立
- 当前升级是为了解决现实问题，不是为了架构理想化

何时足以进入下一阶段：

- 已经知道当前最合理的升级方向
- 已经知道为什么不是其他更重或更轻的方案
- 已经知道升级影响范围和接入边界

```text
请遵守我提供的《基础设施演进方法论》，按 Upgrade 阶段帮我判断当前最合理的升级方向。

当前问题是：
【把触发信号和真实问题贴在这里】

当前方案是：
【把当前方案贴在这里】

请按下面顺序输出：
1. 当前最合理的升级方向是什么
2. 为什么不是其他更轻或更重的方案
3. 这次升级主要解决什么
4. 它会带来哪些新复杂度
5. 升级后最需要验证什么
```

### Phase 6：Verify

适用场景：

- 已经完成一轮升级
- 需要验证升级是否真的有效

进入本阶段前，至少确认：

- 升级已经实际落地
- 当前需要确认的是“问题是否真的被解决”

何时足以进入下一阶段：

- 已经知道升级是否真的解决了原问题
- 已经知道是否引入了新的复杂度或副作用
- 已经知道是否需要进入 Governance

```text
请遵守我提供的《基础设施演进方法论》，按 Verify 阶段帮我验证这次升级是否真的有效。

当前升级内容是：
【把升级内容贴在这里】

当前验证信息是：
【把上线后表现、问题变化、模块接入变化贴在这里】

请按下面顺序输出：
1. 原问题是否真的被解决
2. 是否引入了新的复杂度或副作用
3. 模块接入成本是降低了、持平了，还是升高了
4. 当前升级是否值得进入 Governance
```

### Phase 7：Governance

适用场景：

- 升级已经过验证
- 需要把它沉淀成长期资产

进入本阶段前，至少确认：

- 升级已经过基本验证
- 当前结果值得沉淀，而不是仍在频繁波动

何时足以形成闭环：

- 相关架构文档、模块约束和规则已明确需要回写的部分
- 已经知道新旧阶段边界如何描述
- 已经知道下一跳升级条件是什么

```text
请遵守我提供的《基础设施演进方法论》，按 Governance 阶段帮我判断这次升级后应沉淀什么。

当前升级结果是：
【把升级结果、验证结果、经验贴在这里】

请按下面顺序输出：
1. 哪些架构文档应回写
2. 哪些模块约束应同步
3. 哪些规则值得沉淀
4. 当前新旧阶段的边界应如何描述
5. 下一跳升级条件应如何定义
```
