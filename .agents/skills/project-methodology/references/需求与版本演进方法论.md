# 需求与版本演进方法论

> 用于在项目已经起盘之后，持续判断新需求如何进入系统、当前版本如何收敛，以及如何在功能增长中避免项目失控。

> 主线框架：`Point -> Fit -> Scope -> Priority -> Slice -> Commit -> Verify -> Governance`

## 1. 思考顺序

- 处理需求和版本问题时，先按 `Point -> Fit -> Scope -> Priority -> Slice -> Commit -> Verify -> Governance` 的主线顺序思考
- `Point`：先确认当前进入系统的是一个什么东西，是新需求、问题反馈、技术债，还是结构性机会
- `Fit`：再判断它属于新模块、旧模块扩展、基础设施升级、规则补强，还是暂不处理
- `Scope`：再判断它是否应该进入当前版本，而不是先问“怎么做”
- `Priority`：再决定它与当前其它事项的相对优先级
- `Slice`：再把它切成当前版本可落地的最小切片
- `Commit`：再明确这轮迭代真正承诺交付什么，不交付什么
- `Verify`：交付后验证需求是否真的被解决，而不是只看代码是否合并
- `Governance`：最后回写文档、清单、规则和下一轮判断，形成持续迭代闭环
- `Decision`：不是默认主线；只有在需求归属、版本安排、优先级或切片方式存在明显分叉时，才作为可选分支进入比较与判断
- 不要让需求未经判断直接冲进当前版本

## 2. 核心方法

- 先判断需求属于哪一类，再判断怎么做
- 先判断当前版本是否该接，再判断实现方案
- 先收敛版本目标，再增加需求
- 先切最小可交付切片，再承诺完整蓝图
- 不要把“有价值”直接等同于“现在就做”

## 3. 需求起点判断

- 第一阶段不要默认当前进来的是“要立即开发的新功能”
- 先确认当前对象更像：
  - 一个真实用户需求
  - 一个产品增强想法
  - 一个技术债修复
  - 一个性能 / 稳定性问题
  - 一个项目级结构问题
  - 一个应该被拒绝或延后的扩张点
- 起点不同，后续处理路径也不同
- 如果当前只是一个很模糊的诉求，应先把它转成可判断对象

## 4. 归属判断意识

- 新进入系统的事项，优先判断它属于：
  - 项目主线需求
  - 某个已有模块扩展
  - 一个新模块
  - 基础设施演进问题
  - 文档 / 规则补齐问题
  - 暂不进入当前系统的问题
- 不要所有问题都按“加功能”处理
- 不要所有问题都按“开新模块”处理
- 如果归属判断不清，后续优先级和切片都会失真

## 5. 当前版本收敛意识

- 当前版本先回答“这轮真正要完成什么”，再回答“还有哪些也很重要”
- 如果一个需求会破坏当前版本边界，应优先延后，而不是顺手塞进来
- 当前版本不应同时承担过多不同性质的目标
- 版本目标越混，项目越容易失控
- 不要让未来路线图污染当前版本边界

## 6. 优先级判断原则

- 优先级不只看“重要”，还应看：
  - 是否影响主链路
  - 是否阻塞后续模块接入
  - 是否暴露项目级结构问题
  - 是否会持续制造高成本技术债
  - 是否适合在当前版本解决
- 高优先级不等于先做完整版本
- 很多高价值需求，更适合先切一个最小切片

## 7. 切片意识

- 需求进入当前版本前，优先切成：
  - 最小可验证切片
  - 最小可交付切片
  - 最小可回写切片
- 如果一个需求切不出当前版本可落地的最小形态，通常说明它还不适合进入当前轮次
- 不要把“完整构想”直接当成本轮交付目标
- 切片的目标不是砍价值，而是让价值更早闭环

## 8. 承诺意识

- 每一轮版本都应明确：
  - 本轮承诺交付什么
  - 本轮明确不交付什么
  - 哪些内容只是为下一轮铺路
- 不要让“顺手一起做”成为版本膨胀入口
- 不要在没有重新评估优先级的前提下不断加码承诺

## 9. 验证与回流意识

- 需求完成后，不要只看“代码写完了”
- 应继续回答：
  - 需求是否真的被解决
  - 当前切片是否已足够
  - 是否暴露出新的模块问题或基础设施问题
  - 是否值得进入下一轮继续扩展
- 如果没有这一层，项目会持续堆完成感，而不是堆真实进展

## 10. 治理与回写意识

- 每轮版本结束后，默认继续回答：
  - 哪些文档要回写
  - 哪些清单要更新
  - 哪些约束值得沉淀成规则
  - 哪些需求应进入下一轮
  - 哪些需求应继续延后或拒绝
- 版本演进不应只留下提交记录，还应留下可持续判断的长期资产

## 11. 可视化原则

- 可视化服务理解，不服务装饰
- 涉及版本边界时，优先用“本轮做 / 不做 / 延后”对照表
- 涉及优先级时，优先用排序表或结构化清单
- 涉及切片时，优先用阶段拆解或最小闭环图
- 涉及需求归属时，优先用“项目 / 模块 / 基建 / 规则”分类表达

## 12. 禁止行为

- 不要把所有有价值的需求都塞进当前版本
- 不要没有归属判断就直接开始设计或开发
- 不要把高优先级误解为必须一次做完整
- 不要把技术债长期伪装成“以后再说”
- 不要让版本目标在开发过程中持续漂移而不显式重判
- 不要只回写代码，不回写版本判断和规则沉淀

## 13. 默认输出要求

- 输出需求或版本判断时，优先体现：
  - 当前进入系统的对象是什么
  - 它属于哪一类问题
  - 它是否值得进入当前版本
  - 它在当前版本中的优先级
  - 它的最小可交付切片是什么
  - 本轮真正承诺交付什么
  - 完成后需要验证和回写什么
- 如果信息不足，先暴露缺口，再继续展开
- 如果存在多个合理版本安排，应先给推荐路径，再说明备选

## 14. 分阶段 Prompt 示例

这一节不是替代上面的规则，而是给出可直接使用的参考模板。

推荐用法：

1. 先把这份规则文档作为高层约束
2. 再按下面的阶段 prompt 去跑具体需求或版本问题
3. 默认优先按主线分阶段跑，不要一上来就直接把需求塞进开发

### Phase 0：Point

适用场景：

- 当前出现一个新想法、新需求或新问题
- 但还不确定它到底是什么性质

进入本阶段前，至少确认：

- 当前确实有一个值得进一步判断的事项
- 你现在缺的是“它是什么”的判断，而不是直接方案

何时足以进入下一阶段：

- 已经知道它更像需求、技术债、结构问题，还是应拒绝项
- 已经知道下一步更适合进入 `Fit`

```text
请先不要直接给开发方案。

请遵守我提供的《需求与版本演进方法论》，先按 Point 阶段帮我判断：当前这个进入系统的事项，到底是什么性质。

当前事项是：
【把需求、问题、想法或反馈贴在这里】

请按下面顺序输出：
1. 这更像新需求、技术债、结构问题，还是暂不处理项
2. 为什么
3. 如果值得继续，下一步更适合进入哪个阶段
```

### Phase 1：Fit

适用场景：

- 已经知道当前事项值得继续判断
- 但还不确定它应落在哪一层

进入本阶段前，至少确认：

- 当前事项已经具备继续判断价值
- 当前不是直接讨论实现，而是先判断归属

何时足以进入下一阶段：

- 已经知道它属于项目、模块、基建、规则，还是应延后
- 已经知道当前更适合进入 `Scope`

```text
请遵守我提供的《需求与版本演进方法论》，按 Fit 阶段帮我判断这个事项应落在哪一层。

当前事项是：
【把事项写在这里】

当前项目背景是：
【把当前项目结构、版本目标、已有模块等贴在这里】

请按下面顺序输出：
1. 它应归属到哪一层
2. 为什么
3. 如果进入当前版本，最容易产生的误判是什么
4. 下一步是否值得进入 Scope
```

### Phase 2：Scope

适用场景：

- 已经知道事项归属
- 需要判断它是否值得进入当前版本

进入本阶段前，至少确认：

- 事项归属已经基本成立
- 当前需要做的是版本边界判断

何时足以进入下一阶段：

- 已经知道它是否进入当前版本
- 已经知道如果不进，是延后、拒绝，还是先观察
- 已经知道如果要进，下一步应进入 `Priority`

```text
请遵守我提供的《需求与版本演进方法论》，按 Scope 阶段帮我判断这个事项是否值得进入当前版本。

当前事项是：
【把事项写在这里】

当前版本背景是：
【把当前版本目标、已承诺事项、边界贴在这里】

请按下面顺序输出：
1. 它是否值得进入当前版本
2. 为什么
3. 如果不进入，应延后、拒绝，还是继续观察
4. 如果进入，会冲击当前版本边界吗
```

### Phase 3：Priority

适用场景：

- 已经决定值得进入当前版本
- 需要判断它在当前版本中的优先级

进入本阶段前，至少确认：

- 事项已经进入当前版本候选
- 当前需要做的是优先级排序，而不是再次讨论归属

何时足以进入下一阶段：

- 已经知道它相对当前其它事项的优先级
- 已经知道它为什么现在做而不是之后做
- 已经知道下一步应如何切片

```text
请遵守我提供的《需求与版本演进方法论》，按 Priority 阶段帮我判断这个事项在当前版本中的优先级。

当前事项是：
【把事项写在这里】

当前版本事项列表是：
【把当前版本其它事项贴在这里】

请按下面顺序输出：
1. 它在当前版本中的优先级
2. 为什么
3. 它是否阻塞其它事项
4. 为什么不是更早或更晚做
```

### Phase 4：Slice

适用场景：

- 已经知道它值得做、也知道优先级
- 现在需要切成当前版本可交付的最小形态

进入本阶段前，至少确认：

- 当前事项已确认进入版本
- 当前需要的是切片，而不是直接扩成完整蓝图

何时足以进入下一阶段：

- 已经知道最小可交付切片是什么
- 已经知道这轮明确不做什么
- 已经知道下一步应进入 `Commit`

```text
请遵守我提供的《需求与版本演进方法论》，按 Slice 阶段帮我把这个事项切成当前版本最小可交付形态。

当前事项是：
【把事项写在这里】

当前背景是：
【把当前优先级、项目状态、相关模块贴在这里】

请按下面顺序输出：
1. 最小可交付切片是什么
2. 这轮明确不做什么
3. 为什么这个切片足以闭环价值
4. 哪些扩展应留到下一轮
```

### Phase 5：Commit

适用场景：

- 已经切好最小版本
- 需要明确本轮承诺交付边界

进入本阶段前，至少确认：

- 当前切片已经基本成立
- 当前需要的是明确承诺，而不是继续扩展愿景

何时足以进入下一阶段：

- 已经知道本轮真正承诺交付什么
- 已经知道本轮明确不交付什么
- 已经知道完成后应如何验证

```text
请遵守我提供的《需求与版本演进方法论》，按 Commit 阶段帮我明确这轮版本真正承诺交付什么。

当前切片是：
【把 Slice 阶段结果贴在这里】

请按下面顺序输出：
1. 本轮真正承诺交付什么
2. 本轮明确不交付什么
3. 哪些内容只是为下一轮铺路
4. 完成后最关键的验证标准是什么
```

### Phase 6：Verify

适用场景：

- 本轮已开发或已交付
- 需要验证需求是否真的被解决

进入本阶段前，至少确认：

- 当前已有可验证结果
- 当前不是只看代码完成，而是看需求闭环

何时足以进入下一阶段：

- 已经知道需求是否真的被解决
- 已经知道当前切片是否足够
- 已经知道哪些东西要进入 Governance

```text
请遵守我提供的《需求与版本演进方法论》，按 Verify 阶段帮我判断这轮交付是否真的解决了原问题。

当前事项是：
【把事项写在这里】

当前交付结果是：
【把实现结果、验证结果、反馈贴在这里】

请按下面顺序输出：
1. 原问题是否真的被解决
2. 当前切片是否已经足够
3. 是否暴露出新的模块、基建或规则问题
4. 当前结果是否足以进入 Governance
```

### Phase 7：Governance

适用场景：

- 本轮需求已经过基本验证
- 需要把它沉淀进长期系统

进入本阶段前，至少确认：

- 当前交付已经过基本验证
- 当前结果值得沉淀，而不是还处于频繁波动状态

何时足以形成闭环：

- 该回写的文档、清单和规则已明确
- 已经知道下一轮哪些事项应进入，哪些继续延后
- 当前版本边界经验已被沉淀

```text
请遵守我提供的《需求与版本演进方法论》，按 Governance 阶段帮我判断这轮需求完成后应沉淀什么。

当前事项是：
【把事项写在这里】

当前结果是：
【把交付、验证、经验、偏差贴在这里】

请按下面顺序输出：
1. 哪些文档应回写
2. 哪些清单应更新
3. 哪些约束值得沉淀成规则
4. 哪些事项应进入下一轮
5. 哪些事项应继续延后或拒绝
```

### Optional Phase：Decision

适用场景：

- 当前事项已经确认值得推进
- 但在版本安排、优先级或切片方式上存在明显分叉

```text
请遵守我提供的《需求与版本演进方法论》，按 Decision 阶段帮我处理这个事项下的关键分叉判断。

当前事项是：
【把事项写在这里】

当前背景是：
【把当前版本目标、约束、团队状态、维护成本等贴在这里】

请按下面顺序输出：
1. 这个问题本质上在决定什么
2. 常见安排方案有哪些
3. 应该从哪些维度比较
4. 当前更推荐哪种路径
5. 哪些信息一旦变化，当前推荐会变化
```
