# 项目成型方法论

> 用于从一个模糊项目起点出发，逐步进入可持续开发闭环时，统一判断顺序、阶段职责和产物意识。

> 主线框架：`Point -> Scope -> Architecture -> Module -> Design -> Build -> Verify -> Governance`

## 1. 思考顺序

- 做项目时，先按 `Point -> Scope -> Architecture -> Module -> Design -> Build -> Verify -> Governance` 的主线顺序思考
- `Point`：先确认当前起点到底是什么，是一个想法、需求、问题、机会点，还是一个已有项目的改造方向
- `Scope`：再判断当前阶段做什么、不做什么，当前版本边界是什么
- `Architecture`：再明确项目级结构、关键分层和关键技术方向
- `Module`：再选择首个最值得落地的核心模块，用它逼出真正的最小基础设施边界
- `Design`：再把模块讲清楚，形成可执行设计，而不是靠写代码来想清楚
- `Build`：围绕当前模块补最小基础设施并实现，而不是一开始铺满全套基建
- `Verify`：实现后做联调、验证和质量确认，确保主路径真的跑通
- `Governance`：最后回写文档、沉淀规则、更新清单，让项目进入下一轮闭环
- `Decision`：不是默认主线；只有在关键选型、结构路线或实现策略存在明显分叉时，才作为可选分支进入比较与判断
- 不要跳过前置阶段，直接进入大规模实现

## 2. 核心方法

- 先定盘，再做项目
- 先选首个核心模块，再判断最小基础设施
- 先把设计讲清楚，再进入实现
- 先让当前节点足以支撑下一节点，再继续推进
- 不要把“项目起盘”和“项目实现”混成一个动作
- 不要把“先做全套基建”误当成稳妥做法

## 3. 起点判断

- 第一阶段不要默认自己已经进入开发
- 先确认当前起点更像：
  - 一个模糊项目想法
  - 一个明确需求
  - 一个老项目补骨架
  - 一个已有项目的新模块
  - 一个架构改造机会
- 起点不同，第一步动作也不同
- 如果当前只是一个点，而不是一个有边界的项目，应先帮助它长成可定盘对象
- 不要把“我想做这个项目”和“这个项目已经可以直接开发”混为一谈

## 4. 范围意识

- 先明确当前版本做什么、不做什么
- 先明确当前阶段的目标，再决定需要哪些产物
- 如果范围不清晰，后面的架构、模块、基建都会失真
- 当前版本范围应尽量能支撑一个首个核心模块落地
- 不要让未来想法污染当前边界

## 5. 项目级结构意识

- 项目级问题应优先回答：
  - 这个项目解决什么问题
  - 当前阶段的边界是什么
  - 大致分层是什么
  - 核心模块有哪些
  - 哪些是项目级长期约束
- 架构设计的目标不是把所有细节写完，而是让你知道首个模块该落在哪一层、依赖哪些边界
- 不要只列技术栈，而不讲分层、边界和理由

## 6. 模块优先原则

- 首个核心模块应优先满足：
  - 能代表主链路
  - 能暴露关键架构判断
  - 能逼出真正的最小基础设施需求
- 不要优先做边缘模块
- 不要在没选模块前先铺全套基建
- 模块不是实现细节，而是逼出工程边界的切口

## 7. 设计优先原则

- 模块设计时，优先讲清：
  - 问题定义
  - 用户流程
  - 页面结构或交互轮廓
  - 前后端契约
  - 数据模型
  - 分层职责
- 如果设计还没闭环，不要急着编码
- 如果页面结构本身会影响设计判断，应尽早通过文字、表格、文本树或 `mock.html` 对齐
- 设计阶段的目标是“实现不再需要靠猜”

## 8. 基建与实现原则

- 正确问题不是“先前端还是先后端”，而是“当前模块真正卡在哪一侧”
- 如果模块先卡在数据模型、接口、异步任务、存储，先补后端最小基础设施
- 如果模块先卡在页面结构、状态组织、交互流转，先补前端最小基础设施
- 如果两边都依赖明确契约，先把交互和契约讲清楚
- 永远围绕当前模块补最小可用基建，不要先做全量基建

## 9. 验证与治理意识

- 实现后必须验证主路径是否真的走通
- 验证不是形式动作，而是确认当前节点是否足以支撑下一轮开发
- 完成实现后，默认继续回答：
  - 哪些文档要回写
  - 哪些规则值得沉淀
  - 哪些清单需要更新
  - 下一个模块是否已具备进入条件
- 不要把代码提交当成项目闭环的终点

## 10. 可视化原则

- 可视化服务理解，不服务装饰
- 涉及主干流程时，优先用阶段清单、流程图或文本树
- 涉及架构关系时，优先用分层图、模块关系图或文本树
- 涉及模块设计时，优先用页面结构图、交互轮廓图、契约表或分层清单
- 涉及并列方案判断时，优先用对比表
- 如果没有图会明显抬高理解成本，应明确补图；否则优先用更轻的表达方式

## 11. 禁止行为

- 不要把系统级 blueprint 误当成项目级起盘方法
- 不要一开始就铺满所有基础设施文档
- 不要在项目级骨架还没稳定时直接进入复杂实现
- 不要没选首个核心模块就先做全套基建
- 不要让后一个节点替前一个节点兜底
- 不要实现完不回写文档和规则
- 不要让项目长期停留在“想到哪做到哪”的状态

## 12. 默认输出要求

- 输出项目方法或项目判断时，优先体现：
  - 当前起点
  - 当前范围
  - 当前所处节点
  - 当前节点目标
  - 当前节点核心产物
  - 当前是否已足以进入下一节点
  - 当前需要补的图示或结构化表达
- 如果信息不足，先暴露缺口，再继续展开
- 如果存在多个合理路径，应先给推荐路径，再说明备选
- 如果当前更适合先沉淀骨架、流程图或清单，不要默认直接展开成长文

## 13. 分阶段 Prompt 示例

这一节不是替代上面的规则，而是给出可直接使用的参考模板。

推荐用法：

1. 先把这份规则文档作为高层约束
2. 再按下面的阶段 prompt 去跑具体项目
3. 默认优先按主线分阶段跑，不要一上来把所有问题混在一次输出里

### Phase 0：Point

适用场景：

- 现在还只是一个项目想法或机会点
- 不确定是否值得起盘
- 不确定它是不是一个“项目”，还是只是一个功能念头

进入本阶段前，至少确认：

- 当前确实有一个值得继续判断的起点
- 你现在缺的是“是否值得进入项目级梳理”的判断，而不是立即写方案

何时足以进入下一阶段：

- 已经能判断这个点是否值得继续项目化
- 如果值得，已经能说清它的核心问题大致是什么
- 已经知道下一步应进入 `Scope`，还是暂时先停

```text
请先不要直接进入项目设计或实现。

请遵守我提供的《项目成型方法论》，先按 Point 阶段帮我判断：我当前这个点，是否值得长成一个项目。

我当前的起点是：
【把你的想法、需求、问题、机会点贴在这里】

请按下面顺序输出：
1. 这更像一个想法、需求、问题，还是一个已经成型的项目
2. 它是否值得继续进入项目级梳理
3. 如果值得，当前项目真正的核心问题可能是什么
4. 如果还不值得，缺的是什么
5. 如果要继续，下一步更适合进入 Scope，还是暂时先停
```

### Phase 1：Scope

适用场景：

- 已经确认值得做
- 但当前范围和版本边界还不清楚

进入本阶段前，至少确认：

- 当前起点已经值得进入项目级梳理
- 当前不是单纯功能念头，而是具备项目化可能
- 已经能描述核心问题和目标对象

何时足以进入下一阶段：

- 当前版本做什么、不做什么已经清楚
- 当前最小范围已能支撑项目级结构判断
- 延后项已被显式隔离，不再污染当前边界

```text
请遵守我提供的《项目成型方法论》，先按 Scope 阶段帮我判断这个项目当前版本的边界。

项目主题是：
【把项目主题写在这里】

当前背景是：
【把用户、场景、目标、约束贴在这里】

请按下面顺序输出：
1. 这个项目当前阶段真正要解决什么
2. 当前版本做什么，不做什么
3. 当前最小可落地范围是什么
4. 哪些想法应该明确延后
5. 当前范围是否已经足以进入 Architecture
```

### Phase 2：Architecture

适用场景：

- 当前范围已经有了
- 现在需要项目级结构和关键方向

进入本阶段前，至少确认：

- 当前范围已经基本成立
- 已经知道当前阶段的目标和边界
- 当前最小范围足以支撑结构判断

何时足以进入下一阶段：

- 已经知道项目的大致分层和核心模块划分
- 已经知道首个核心模块应落在哪一层
- 待定项已被显式标出，而不是继续混在结构里

```text
请遵守我提供的《项目成型方法论》，按 Architecture 阶段帮我定盘这个项目。

项目是：
【把项目写在这里】

当前范围是：
【粘贴 Scope 阶段结果】

请按下面顺序输出：
1. 项目级大致分层
2. 核心模块划分
3. 关键技术方向或待定项
4. 哪些约束属于项目级长期约束
5. 首个核心模块应该落在哪一层
```

### Phase 3：Module

适用场景：

- 架构已有初稿
- 需要确定首个最值得先落地的模块

进入本阶段前，至少确认：

- 架构初稿已足以支撑模块选择
- 关键分层和边界已基本明确
- 当前不是在“选一个看起来顺手的功能”，而是在找能带动整体落地的切口

何时足以进入下一阶段：

- 已经明确首个核心模块及其优先级理由
- 已经知道它会逼出哪些关键判断和最小基建需求
- 已经知道为什么不是其他候选模块先做

```text
请遵守我提供的《项目成型方法论》，按 Module 阶段帮我选择首个核心模块。

项目是：
【把项目写在这里】

当前架构信息是：
【粘贴 Architecture 阶段结果】

请按下面顺序输出：
1. 哪个模块最值得先做
2. 为什么它最适合作为首个模块
3. 它会逼出哪些关键架构判断
4. 它会逼出哪些最小基础设施需求
5. 为什么不应先做其他候选模块
```

### Phase 4：Design

适用场景：

- 已经确定首个模块
- 现在需要把它设计到可执行

进入本阶段前，至少确认：

- 首个核心模块已经选定
- 当前模块目标和上下文已基本清楚
- 当前设计的目标是“让实现不再靠猜”，而不是提前写代码

何时足以进入下一阶段：

- 用户流程、交互轮廓、契约和数据模型已经闭环
- 分层职责已基本清楚
- 当前设计已经足以支撑最小基建判断和实现顺序判断

```text
请遵守我提供的《项目成型方法论》，按 Design 阶段帮我把这个模块讲清楚。

模块是：
【把模块写在这里】

当前背景是：
【把项目上下文和模块目标贴在这里】

请按下面顺序输出：
1. 模块真正要解决的问题
2. 用户流程
3. 页面结构或交互轮廓
4. 前后端接口契约
5. 数据模型
6. 分层职责
7. 哪些部分需要 `mock.html`、表格、结构图或文本树
8. 当前设计是否已经足以进入 Build
```

### Phase 5：Build

适用场景：

- 设计已闭环
- 现在要决定围绕模块补哪些最小基建并开始实现

进入本阶段前，至少确认：

- 设计已基本闭环
- 当前实现不再需要靠补猜测来推进
- 已经能识别当前模块真正卡在哪一侧

何时足以进入下一阶段：

- 当前模块的主路径已经具备实现条件
- 最小基础设施边界已经明确
- 实现顺序已经清楚，不再需要回到“全量基建先做”的思路

```text
请遵守我提供的《项目成型方法论》，按 Build 阶段帮我判断当前模块的最小基础设施和实现顺序。

模块是：
【把模块写在这里】

当前设计是：
【粘贴 Design 阶段结果】

请按下面顺序输出：
1. 当前模块真正卡在哪一侧
2. 需要先补哪些最小基础设施
3. 哪些基础设施现在不用做
4. 最合理的实现顺序
5. 当前实现是否需要先补前端、后端，还是先补契约
```

### Phase 6：Verify

适用场景：

- 已经实现了一轮
- 需要确认主路径和当前质量

进入本阶段前，至少确认：

- 已经有一轮可运行的实现
- 当前已经具备可验证的主路径，而不只是零散代码
- 当前要确认的是“是否足以进入下一轮”，而不是继续凭感觉推进

何时足以进入下一阶段：

- 主路径是否跑通已经被明确验证
- 阻塞问题已被识别，并知道该回退到哪一层修正
- 当前质量已足以进入回写、沉淀和下一轮规划

```text
请遵守我提供的《项目成型方法论》，按 Verify 阶段帮我检查当前模块是否已经足以进入下一轮。

模块是：
【把模块写在这里】

当前实现情况是：
【把当前实现结果、测试情况、联调情况贴在这里】

请按下面顺序输出：
1. 当前主路径是否已经跑通
2. 还有哪些阻塞问题
3. 这些问题更应该回退到 Design、Architecture，还是在实现层修
4. 当前质量是否足以进入 Governance
```

### Phase 7：Governance

适用场景：

- 当前模块已完成一轮
- 需要回写、沉淀、更新规则与清单

进入本阶段前，至少确认：

- 当前这一轮已经经过基本验证
- 当前结果已经值得沉淀，而不是还停留在未稳定状态
- 你现在要解决的是“如何把结果变成长期资产”

何时足以形成闭环：

- 该回写的文档已明确
- 该更新的清单和规则已明确
- 已经知道下一轮最合理进入哪个节点

```text
请遵守我提供的《项目成型方法论》，按 Governance 阶段帮我判断这轮之后应该沉淀什么。

模块或项目是：
【把对象写在这里】

当前结果是：
【把这一轮的实现、验证、经验贴在这里】

请按下面顺序输出：
1. 哪些文档应回写
2. 哪些清单应更新
3. 哪些约束值得沉淀成规则
4. 下一轮最合理进入哪个节点
5. 项目是否已经形成闭环
```

```text
2. 常见方案有哪些

```
