# Drafts Maintaining

> **文档职责**：说明 `drafts/` 草稿箱如何管理、如何流转、何时升格、何时归档。
> **适用场景**：整理文档草稿、整理 prompt 草稿、判断是否需要按主题分类、决定何时迁入正式文档前阅读。
> **目标读者**：本仓库维护者。
> **维护规范**：只要草稿目录结构、生命周期定义、升格路径或整理流程发生变化，就同步更新本文档。

---

## 1. 草稿箱的定位

草稿箱是**中转系统**，不是正式知识库，也不是历史仓库。

它解决 3 件事：

- 先把零散想法接住，不要求一开始就定稿
- 让仍在演化的内容有地方继续收敛
- 在“升格为正式资产”和“退出主线”之间提供清晰路径

判断原则：

- 草稿箱优先回答“这是什么”和“现在成熟到哪一步”
- 正式文档区回答“当前标准是什么”
- 归档区回答“历史上发生过什么”

---

## 2. 管理草稿的四个维度

管理草稿时，至少有 4 个维度：

1. 资产类型：`docs`、`prompts`、`scripts` 等
2. 生命周期：记录中、整理中、候选中、正式、归档
3. 主题：`architecture`、`backend`、`operations`、`writing` 等
4. 版本：`简单版`、`详细版`、`严格版`，或 `v1`、`v2`

优先级：

`资产类型 > 生命周期 > 主题 > 版本`

这意味着：

- 先判断它是文档草稿还是 prompt 草稿
- 再判断它现在处于哪个阶段
- 只有在需要降低检索成本时，才引入主题目录
- 版本通常留在文件内部，不上升为目录主轴

---

## 3. 生命周期是主线，不是主题

本仓库采用“**先分资产类型，再分生命周期，最后按需补主题**”的管理模型。

推荐理解：

```text
想法/尝试 -> wip -> next -> 正式资产
                      \-> archived
```

对本仓库当前结构来说：

- `drafts/docs/`：普通文档草稿，允许零散、试探、未收敛
- `drafts/docs/wip/`：正在重点整理中的文档草稿
- `drafts/docs/next/`：当前主线候选稿，准备继续实践验证
- `drafts/docs/archived/`：已退出当前主线的历史文稿
- `drafts/prompts/wip/`：还在打磨的 prompt 草稿
- `drafts/prompts/next/`：当前主线候选 prompt，已可反复使用
- `drafts/prompts/archived/`：已废弃或已被正式 prompt / skill 吸收的 prompt 草稿

不要把“主题”当成主骨架。  
生命周期比主题更稳定，也更接近每次整理草稿时真正要做的判断。

---

## 4. 主题该放在哪里

结论：

- 主题放在生命周期里面
- 不要先按主题拆，再在每个主题下细分 `wip/next/archived`

推荐结构：

```text
drafts/
  docs/
    wip/
    next/
    archived/
  prompts/
    wip/
      architecture/
      backend/
      operations/
      writing/
    next/
    archived/
```

原因：

- 生命周期判断比主题判断更基础
- 主题边界比生命周期更容易变化
- 过早把主题放到最上层，会制造很多空目录和多余层级

主题的作用只有一个：**降低检索成本**。  
它不是草稿箱的第一组织原则。

---

## 5. 不同阶段是否都需要主题目录

不需要。

当前推荐：

- `wip/`：最容易需要主题目录，因为最杂、最多、最容易堆平铺文件
- `next/`：通常先不按主题拆，除非数量继续明显增长
- `archived/`：通常不按主题拆，目标是留痕，不是高频浏览
- 正式区：按职责维护，不再沿用草稿期的主题分组

判断是否给某一层补主题目录，看这 3 个信号：

- 该层文件已经明显拥挤，平铺浏览成本上升
- 同类文件会被反复一起查看和整理
- 主题边界已经比较稳定，不是今天叫这个、明天又换名字

如果只是觉得“平铺看着不整齐”，不要拆目录。

---

## 6. 当前推荐的主题入口

`drafts/prompts/wip/` 当前推荐保留少量稳定入口：

- `architecture/`：需求分析、架构设计、从方案推进到实现
- `backend/`：接口、测试、验证等后端实现侧 prompt
- `operations/`：部署、服务管理、安全、配置治理、环境检查
- `writing/`：文档输出、讲解、可视化增强

维护规则：

- `wip/` 根目录尽量只保留 `README.md`、整理笔记和少量临时文件
- 新增 prompt 时，优先放入已有主题目录
- 只有某个主题内部继续明显膨胀时，才考虑在该主题下再细分
- 不要为了整齐把目录拆成很深的树

---

## 7. 版本怎么管理

版本是最后一级维度。

优先做法：

- 同一主题的不同表达深度，放在同一个文件里
- 用 `简单版`、`详细版`、`严格版` 或 `阶段 1 / 阶段 2` 表示
- 只有当“已经不是一个任务，而是不同任务”时，才拆成多个文件

不推荐：

- 为了版本建立目录
- 为了 `v1/v2/v3` 长期堆文件，而不回收旧版本

版本的目标是帮助比较和迭代，不是制造新的分类层级。

---

## 8. 何时进入不同阶段

进入 `drafts/docs/wip/` 的信号：

- 这篇草稿会反复影响后续设计判断
- 内容已经不只是随手记录，而是在形成方法论
- 结构、结论或边界还可能继续大改

从 `drafts/docs/wip/` 进入 `drafts/docs/next/` 的信号：

- 当前主线已经比较清楚
- 已经准备拿去真实使用或对照实践验证
- 还不想立即升为正式 `docs/`

Prompt 进入 `drafts/prompts/next/` 的信号：

- 已经具备稳定复用价值
- 输出结构比较清楚
- 会被反复调用，但还不想立即升为正式 `prompts/`

进入 `archived/` 的信号：

- 已被新的主线方案替代
- 内容仍有历史参考价值，但不再继续维护
- 已经被正式文档、正式 prompt、skill 或 command 吸收

---

## 9. 何时迁出草稿箱

草稿不应长期停留在“半正式”状态。

满足以下信号时，应尽快迁出：

- 判断已经稳定
- 已经过实践验证
- 已开始反复被引用
- 已经在约束后续做法，而不再只是想法
- 已能明确归属到某个正式资产

升格路径：

- 使用方式稳定了 → `README.md`
- 方法层稳定了 → `PLAYBOOK.md`
- 设计决策稳定了 → `WHY.md`
- 维护流程稳定了 → `MAINTAINING.md` 或 `RUNTIME-MAINTAINING.md`
- Prompt 稳定且可直接复用 → `prompts/`
- Prompt 已抽象成方法与工作流 → `skills/` 或 `commands/`

如果内容不再进入正式主线，但保留历史价值，则迁入 `archived/`。

---

## 10. 草稿整理流程

当 `drafts/` 目录积累了较多文件时，定期整理。

推荐使用 `/distill-draft` 命令：

```text
# 提炼具体草稿
/distill-draft drafts/docs/某想法.md

# 批量查看所有草稿状态
/distill-draft --batch
```

手动整理时，按下面顺序判断：

1. 先识别资产类型：`docs` 还是 `prompts`
2. 再判断成熟度：仅记录 / 正在打磨 / 当前候选 / 已废弃
3. 再判断影响范围：仅当前项目 / 跨项目通用 / 影响范式 / 影响维护流程
4. 最后才判断是否需要主题归类、是否需要版本整理

常见去向：

- 文档类仅记录 → `drafts/docs/`
- 文档类初步判断 + 会反复影响设计 → `drafts/docs/wip/`
- 文档类已形成当前主线、待实践验证 → `drafts/docs/next/`
- Prompt 类仍在打磨 → `drafts/prompts/wip/`（必要时进入其主题子目录）
- Prompt 类已形成当前主线、待实践验证 → `drafts/prompts/next/`
- Prompt 稳定可复用 → `prompts/`
- Prompt 已被更高层机制吸收 → `skills/`、`commands/` 或 `drafts/prompts/archived/`

整理动作：

- 提炼核心观点
- 补充“为什么”和“什么时候用”
- 合并重复稿、淘汰明显过期稿
- 迁入正式文档或归档目录
- 必要时在原草稿开头标注“已升格到 XXX”

---

## 11. 整理频率

推荐触发时机：

- 每次提交较大改动前，先整理一遍草稿
- 每周或每月定期整理一次
- 当 `drafts/docs/`、`drafts/docs/wip/`、`drafts/docs/next/`、`drafts/prompts/wip/`、`drafts/prompts/next/` 明显积压时，主动触发整理

目标不是把草稿箱整理得像正式文档区，而是避免它长期失控。
