# Claude Blueprint Playbook

> **文档职责**：定义这套 blueprint 的开发范式，回答“先做什么、后做什么、每个阶段产出什么”。
> **适用场景**：新项目启动、老项目纳入规范、新模块设计、实现后回写文档、抽象新规则。
> **目标读者**：使用这套 blueprint 开发和维护项目的人。

---

## 1. 这份文档解决什么问题

这份文档不教你某个命令怎么写，也不教你某个 skill 的格式细节。

它只回答 3 个更上层的问题：

1. 当前项目处在哪个阶段
2. 当前阶段应该先产出什么，再做什么
3. command、skill、agent、hook 在这个阶段分别扮演什么角色

一句话总结：

**先有开发范式，再让 command、skill、agent、hook 服从这个范式。**

边界说明：

- 本文档只负责“开发范式”
- 不负责完整展开“能力系统”边界（见 `docs/能力地图.md`）
- 不负责完整展开“如何与 Claude / Codex 交互”的说明（见 `README.md` 与 `docs/用户心智模型.md`）

## 2. 核心原则

### 2.1 先定盘，再增量开发

新项目或老项目首次纳入规范时，先补齐项目级骨架，再开始模块开发。

项目级骨架最小集合：
- 项目根 `CLAUDE.md`
- `docs/design/INDEX.md`
- `docs/design/架构设计方案.md`
- `docs/实施清单.md`

### 2.2 先设计边界，再写实现代码

遇到新模块、结构性改动、跨层改动时，先设计文档，后实现代码。

遇到小改动、局部修复、纯 bugfix 时，可以先实现，再回写文档。

### 2.3 文档分层，代码分层

不要把所有信息都写进一份文档，也不要把所有逻辑都压进一个目录。

文档分层：
- 项目级架构 → `docs/design/架构设计方案.md`
- 模块级设计 → `docs/design/模块/...`
- 基础设施约定 → `docs/design/基础设施/...`
- 复盘 / 踩坑 / 故障排查 → `docs/专题/...`

### 2.4 最小必要基础设施

先建设撑起首个核心功能所需的最小基础设施，不做预支式过度铺设。

先问：
- 这个功能现在必须依赖哪些基础设施
- 哪些基础设施可以等第二阶段再补

### 2.5 每次实现都要回写文档和抽象规则

代码完成不是终点。

实现后至少做两件事：
- 回写受影响的设计文档
- 判断有没有新的跨项目通用规则需要沉淀

## 3. 标准项目生命周期

这套 blueprint 把项目开发拆成 4 个阶段。

### 阶段 1：项目定盘

**目标**：明确项目范围、技术栈、分层结构和文档骨架，让项目进入可规范化开发状态。

**先想什么**：
- 项目到底解决什么问题
- 当前阶段做什么，不做什么
- 技术栈与关键选型
- 后端 / 前端 / 基础设施大致分层

**主要产物**：
- 项目根 `CLAUDE.md`
- `docs/design/INDEX.md`
- `docs/design/架构设计方案.md`
- `docs/实施清单.md`

**推荐入口**：
- `/init-architecture`

**此阶段不要做的事**：
- 不要一开始就铺满所有基础设施文档
- 不要一开始就把所有模块文档都创建出来
- 不要在没有项目级骨架的情况下直接让 `arch` 或 `/new-module` 处理复杂需求

### 阶段 2：模块设计

**目标**：把一个具体功能模块讲清楚，再进入实现。

**先想什么**：
- 用户要完成什么动作
- 后端需要哪些数据模型和接口
- 前端需要哪些页面、状态和交互
- 这个模块是否影响全局架构

**主要产物**：
- `docs/design/模块/.../设计_v1.md`
- `页面布局_mock.html`（如有前端）

**推荐入口**：
- `/new-module`
- `@arch`（仅在项目级骨架已存在时，用来做架构判断）

**此阶段的默认顺序**：
1. 问题定义
2. 页面/交互轮廓
3. 前后端交互与接口契约
4. 数据模型
5. 分层职责
6. 再进入编码

### 阶段 3：实现落地

**目标**：按既定边界完成代码实现。

**先想什么**：
- 这次改动属于新模块、结构性改动，还是局部修复
- 对应设计文档是否已经足够支撑实现
- 是否只需要补最小基础设施，而不是一次做完全部抽象

**主要产物**：
- 代码
- 验证结果

**推荐入口**：
- 主对话直接实现
- 需要架构判断时调用 `@arch`
- 需要审查时调用 `@rev`

**此阶段的判断规则**：
- 新模块 / 结构性改动 → 先设计，后实现
- 小改动 / bugfix → 可先实现，后回写

### 阶段 4：回写与抽象

**目标**：把实现结果同步回文档，并把可复用经验沉淀下来。

**先想什么**：
- 这次实现有没有偏离原设计
- 哪些文档应该更新
- 有没有新的跨项目规则值得抽象

**主要产物**：
- 更新后的设计文档
- 更新后的 `docs/design/INDEX.md`
- 更新后的 `docs/实施清单.md`
- 新规则（如有）

**推荐入口**：
- `/update-docs`
- `/add-rule`

## 4. 先做什么，后做什么

### 4.1 先做架构，还是先考虑技术栈

先明确项目目标和范围，再同时收敛架构与技术栈。

不要先独立“选技术栈”，也不要脱离目标空画架构。

### 4.2 先做基础设施，还是先做功能模块

先做**支撑当前核心模块的最小基础设施**。

不要为了“以后可能用到”一次性把所有基础设施铺满。

### 4.3 先写设计文档，还是先写项目代码

- 新模块、结构性改动、跨层改动 → 先文档，后代码
- 小改动、局部修复、纯 bugfix → 先代码，后回写文档

### 4.4 到功能设计时，先做后端设计还是先做页面布局

都不要单边先走到底。

更稳的顺序是：
1. 用户流程
2. 页面结构 / 交互轮廓
3. API 契约
4. 数据模型
5. 后端实现与前端接入

### 4.5 项目复杂度上来后，是否需要单独的维护文档

可以，但它不是所有项目一开始就必须存在的文档。

只有当项目开始出现下面这些信号时，才建议新增一份项目级维护文档：

- 核心文档已经明显增多，例如：
  - 架构设计文档
  - 多个模块设计文档
  - 多个基础设施文档
  - 运维文档
  - 实施清单
- 文档之间开始有联动关系
  - 改架构文档会影响模块文档
  - 改基础设施约定会影响运维文档
  - 新增设计文档必须同步更新 `INDEX.md`
- 你开始担心“哪些文档是关键资产、哪些修改必须联动检查”

这时可以新增一份**项目级维护文档**，专门说明：

- 哪些文档是关键资产
- 哪些修改会触发联动更新
- 哪些文档必须长期保持准确
- 哪些内容只应保留为专题 / 复盘，而不是主设计文档

它的定位不是新能力，而是一种**项目文档治理方法**。

因此：
- 不把它归到 `docs/能力地图.md`
- 更适合作为项目方法论的一部分理解
- 等在多个项目里验证稳定后，再考虑是否把它模板化

## 5. 这套范式里，各组件分别干什么

### command

负责**阶段入口和显式工作流**。

例如：
- `/init-architecture`
- `/new-module`
- `/update-docs`
- `/add-rule`

它负责“现在进入哪个阶段”，不负责定义整套方法论。

### skill

负责**某一类任务的思维方式和写法规范**。

例如：
- 设计文档怎么写
- 运维 CLI 怎么组织
- Python 脚本怎么结构化

它负责“这类事情怎么做”，不负责决定“什么时候做”。

### agent

负责**带角色的分析和判断**。

例如：
- `arch` 负责架构判断
- `rev` 负责审查和差异识别

它负责给建议，不负责默默替你启动整条工作流。

### hook

负责**强制和提醒**。

例如：
- 保护配置
- 阻止 `git push`
- 提醒关键文件变更

它不适合承担初始化、补齐骨架、生成文档体系这类主动流程。

## 6. 当前标准范式没有默认引入的能力

`drafts/Claude Code 目录结构完整参考(文字版).md` 列的是 Claude Code 的完整能力版图，不等于这套 blueprint 当前必须全部采用。

当前标准范式默认不把 `rules/` 和 `plugins/` 纳入主路线。

### 6.1 rules

`rules/` 更适合承载按路径、按子域、按文件类型分化的约束。

当前先不默认引入 `rules/`，原因是：
- 跨项目通用约束已经有 `~/.claude/CLAUDE.md`
- 复杂任务规范已经有 `skills/`
- 项目级流程和边界已经有项目根 `CLAUDE.md` 与 `docs/design/...`

当前仓库保留了一个最小 `rules/writing.md` 示例，用于验证 rules 机制可用；这不改变“默认不依赖 rules 分层”的主路线判断。

只有当项目规则开始明显按目录分化，例如：
- `frontend/` 和 `backend/` 有不同约束
- `pipeline/`、`adapters/`、`pages/` 有不同约束
- 项目 `CLAUDE.md` 已经明显变长，继续堆会失控

这时才建议引入项目级 `.claude/rules/`。

### 6.2 plugins

`plugins/` 用来扩展 Claude Code 的能力，不用来定义开发范式。

当前先不默认引入 `plugins/`，原因是：
- 你当前要解决的是开发方法、文档体系、阶段边界
- 这些问题主要靠 `commands`、`skills`、`agents`、`hooks`、`templates` 就能解决

只有当你需要：
- 接入新的外部能力
- 封装新的工具链能力
- 提供当前 Claude Code 默认能力之外的增强

才考虑引入 `plugins/`。

一句话判断：
- `rules`：用来拆项目内分化约束
- `plugins`：用来增加新能力

它们都不是当前标准范式的默认组成部分。

## 7. 新项目和老项目如何进入同一套范式

### 新项目

按顺序：
1. 部署 blueprint 到 `~/.claude`
2. 在项目里运行 `/init-architecture`
3. 补齐项目级骨架
4. 用 `/new-module` 进入第一个模块设计

### 老项目纳入规范

按顺序：
1. 先识别当前项目有没有项目级 `CLAUDE.md`、设计索引、架构文档、实施清单
2. 用 `/init-architecture` 补最小骨架
3. 不要求一次性把所有历史文档补齐
4. 后续在真实迭代中逐步把模块文档和基础设施文档补出来

---

## 8. 命令只是入口，不是范式本身

`/init-architecture` 不需要替你补齐所有未来文档。

它只需要做一件事：

**把项目带到“可以进入统一开发范式”的起点。**

后续文档应随着真实开发逐步生长：
- 项目定盘阶段长项目级文档
- 模块设计阶段长模块文档
- 基础设施复杂度上来后，再长基础设施文档
- 踩坑后再长专题文档

这比一开始造一整套空文档更稳。

---

## 9. 什么时候应该更新这份 Playbook

出现以下任一情况时，应更新本文档：

- 你改变了项目标准生命周期
- 你改变了“先设计还是先实现”的判断规则
- 你改变了 command / skill / agent / hook 的职责边界
- 你发现多个项目里重复出现了新的稳定模式

原则：

- 实现细节变化，不一定改 Playbook
- 方法层变化，必须改 Playbook
