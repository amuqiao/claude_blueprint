# Why This Blueprint Exists

> **文档职责**：记录这个模具仓库的关键设计决策，解释为什么这样组织，而不是记录项目业务内容。
> **适用场景**：修改本仓库结构、增减 hook / skill / agent、或评估是否要公开发布时阅读。
> **目标读者**：仓库维护者，以及想复用这套模具的独立开发者。
> **维护规范**：只要模具层的目录结构、约束分层、hook 策略、agent 数量或角色发生变化，就同步更新对应节。

---

## 1. 这是一个仓库，不是单文档

这套方案的核心价值不在“描述有哪些文件”，而在“这些文件本身可直接被 clone 下来使用”。

如果只保留一份总文档，每次新设备初始化都要手动复制 `skills/`、`agents/`、`commands/`、`templates/` 里的内容到对应路径，容易漏文件，也容易把 frontmatter、目录层级和文件名抄错。

做成 git 仓库后，初始化退化为一次 `git clone`，目录结构和内容一起到位。

但仓库本身不再推荐直接 clone 到 `~/.claude`。

新的使用模型是：
- 本仓库 checkout：配置源
- `~/.claude`：实际运行目录

原因是 `~/.claude` 会持续产生大量运行时数据；把它直接作为 git 工作目录，长期会和配置源混在一起，维护成本更高。

---

## 2. CLAUDE.md 保持短，小而硬

`CLAUDE.md` 只保留跨项目都成立的硬约束：
- 分层架构原则
- 通用工作流约束
- skill 指针
- 极少量写作原则

细节不直接塞进 `CLAUDE.md`，而是下沉到 `skills/`。原因有两个：
- `CLAUDE.md` 会在每个会话中进入上下文，太长会稀释真正高频的全局约束
- 技能型规范天然按任务分类，放进 skill 更容易维护和按需加载

80 行是经验上限。超过这个规模，说明某一类细节还没被拆走。

---

## 3. 技术路线是“工程 OS”，不是“通用使用手册”

这个仓库的目标不是覆盖所有 Claude Code 用户的最佳实践，而是为独立全栈开发者提供一套长期可复用的开发操作系统。

因此这里优先沉淀的是：
- 设计文档结构
- 架构分层约束
- 脚本与运维工具规范
- 文档同步与规则归档流程

不优先追求的是：
- 大量通用 agent 角色
- 覆盖所有场景的 CLI 使用技巧
- 面向团队协作的复杂审批流程

---

## 4. Hooks 负责强制，CLAUDE.md 负责引导

纯靠 `CLAUDE.md` 的约束属于软约束。它适合表达原则，但不适合承担“必须阻止”或“必须提醒”的动作。

因此这里把三类高价值风险收敛到 hook：
- `protect-config.sh`：编辑关键配置或控制文件时强制二次确认
- `notify-file-changed.sh`：关键文件被改后追加同步提醒
- `block-git-push.sh`：阻止 Claude 直接执行 `git push`

设计原则：
- 需要 100% 执行的约束放 hook
- 需要表达判断原则的约束放 `CLAUDE.md`
- 需要大量细节的任务规范放 skill

---

## 5. 只保留两个 agent

这里只保留 `arch` 和 `rev` 两个 agent，是刻意收敛，不是暂时缺失。

选择标准是：
- 任务上下文容易变脏
- 结果相对独立
- 角色边界清晰
- 复用频率高

`arch` 负责方案判断，`rev` 负责合规审查。其余任务默认回到主对话 + skill。

这样做的原因：
- 独立开发者的上下文和流程比团队更短，过多 agent 会带来维护成本
- skill 比 agent 更适合承载写作、脚本、运维类规范
- agent 数量越少，触发边界越清晰，长期漂移越小

---

## 6. rev 必须是 plan 模式

`rev` 的职责是找问题，不是顺手修代码。

因此除了 prompt 里的“不写修复代码”之外，还要在 frontmatter 里显式设置 `permissionMode: plan`，把“不该写”从软约束提升为硬限制。

这能降低两类风险：
- 审查过程顺手改文件，导致 review 和 implementation 混在一起
- Claude 在“帮忙修一下”的冲动下越过审查角色边界

---

## 7. templates 是手动模板，不是假装自动化

`templates/` 目录的目标只是提供稳定起点，不是依赖 Claude Code 的某种隐式自动机制。

因此这里明确采用“手动 `cp` 模板”的方式：
- 行为简单
- 可见性强
- 不依赖隐藏约定

模板文件只提供起点和占位符，不替代项目级判断。

---

## 8. 模具层也要有自己的文档

项目里会有 `docs/专题/` 用来沉淀业务踩坑；模具本身也需要记录为什么这样设计。

没有这类文档时，问题会变成：
- 过几个月后忘记为什么要保留某条约束
- 想公开仓库时说不清设计取舍
- 新增规则时无法判断是延续已有路线，还是在偷偷偏离

`WHY.md` 的作用就是把这些设计判断从脑内记忆变成显式资产。

---

## 9. drafts/ 是模具层的草稿箱

不是每一条想法都值得立刻进入 `WHY.md`、skill 或 command。

因此仓库里单独保留 `drafts/`：
- 先收集零散判断、对话摘录、结构试探
- 等内容成熟后，再决定是否升格为正式决策

整理路径只有两种：
- 设计判断稳定了，提炼进 `WHY.md`
- 已经变成可执行约束了，走 `/add-rule` 归到合适位置

草稿本身不需要删除。它是模具演化过程的一部分，也应该随仓库一起版本化保存。
