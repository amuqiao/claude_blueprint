# iOS App 0-1 工作流

> 把一个 App 想法推进到可开发、可验证的第一版，而不是直接跳进代码或高保真界面。

## 文档职责

本文负责定义 iOS App 从想法到 MVP 开发闭环的工作流：每个阶段解决什么问题，需要什么输入，产出什么结果，做到什么程度就可以进入下一步。

**适用读者**：已经有 App 想法，但不确定如何拆范围、画原型、写设计规格、安排开发顺序的人。  
**不适用于**：iOS 环境安装、SwiftUI 语法学习、Xcode 工程细节、TestFlight 或 App Store 上架流程。

如果你已经进入真实 iOS 工程创建、测试和上架阶段，阅读 [iOS App 从 0 到上架：新手完整指南](ios-app-development-guide.md)。如果你需要理解 SwiftUI 示例项目如何被 Xcode 工程承载并运行，阅读 [SwiftUI 示例项目基础扫盲](swiftui-sample-project-basic-literacy.md)。

## 核心心智模型

不要把 0-1 理解成：

```text
我有一个想法 -> 我开始做
```

更稳的链路是：

```text
我有一个想法
-> 判断它是否足以进入下一阶段
-> 只产出当前阶段的最小必要产物
-> 满足进入条件后再继续推进
```

这套工作流的重点不是把每一步写得很重，而是为每一步建立固定的执行契约：

```text
阶段名：
这一阶段要解决的问题：
输入：
输出：
最小可用：
不要做什么：
进入下一步条件：
```

只要每一阶段都有明确的输入、输出、最小可用产物和进入下一步条件，想法就不会停留在概念层，也不容易在中途被新想法淹没。

## 总体阶段

```text
Idea
-> Product Brief
-> MVP Scope
-> UX Skeleton
-> UI Prototype
-> Design Spec
-> Build Plan
-> Development
-> Verify
```

这条链路按职责分层：

| 阶段 | 核心问题 |
|------|----------|
| Idea | 这个想法是否值得启动 |
| Product Brief | 想法如何变成可讨论的项目定义 |
| MVP Scope | 第一版到底做什么、不做什么 |
| UX Skeleton | 页面结构和主流程是否清楚 |
| UI Prototype | 界面外观和点击体验是否合理 |
| Design Spec | 开发是否还需要靠猜 |
| Build Plan | 代码从哪里开始做 |
| Development | 把设计转成可运行代码 |
| Verify | 当前迭代是否闭环 |

## 阶段执行标准

### 1. Idea

**要解决的问题**：这个东西到底是不是一个值得启动的 App 想法。

**输入**

- 零散想法
- 问题感受
- 功能念头
- 用户场景直觉

**输出**

- 一句话项目定义
- 目标用户
- 核心问题
- 为什么值得做

**最小可用**

只要写清 4 句：

```text
我想做什么：
为谁做：
解决什么问题：
为什么值得单独做成 App：
```

**不要做什么**

- 不要画页面
- 不要定技术栈
- 不要列一堆功能

**进入下一步条件**：已经能一句话说清这个 App 的存在理由。

### 2. Product Brief

**要解决的问题**：把想法从灵感变成可讨论的项目定义。

**输入**

- Idea 阶段的一句话定义
- 初步场景和用户判断

**输出**

- 项目目标
- 用户画像
- 核心价值
- 初步功能方向
- 暂不处理范围

**最小可用**

一页文档即可，回答 5 个问题：

- 这是什么 App
- 给谁用
- 解决什么问题
- 用户为什么会用
- 当前明显不做什么

**不要做什么**

- 不要进入页面细节
- 不要写成完整 PRD
- 不要开始讨论组件

**进入下一步条件**：已经能区分核心价值和顺手想到的附加功能。

### 3. MVP Scope

**要解决的问题**：第一版到底做什么，不做什么。

**输入**

- Product Brief
- 初步功能想法列表

**输出**

- MVP 功能清单
- 非 MVP 清单
- 主链路
- MVP 页面列表

**最小可用**

只要写清：

```text
第一版做：
第一版不做：
主链路：
必须页面：
```

**不要做什么**

- 不要追求完整产品
- 不要把未来想法都塞进 MVP
- 不要开始画高保真原型

**进入下一步条件**：已经能回答“如果只做第一版，用户最核心的一条路径是什么？”

### 4. UX Skeleton

**要解决的问题**：产品结构、页面关系、用户流程是否清楚。

**输入**

- MVP Scope
- 核心场景
- MVP 页面列表

**输出**

- 页面地图
- 主用户流程
- 低保真线框
- 页面入口 / 出口

**最小可用**

至少产出 3 个东西：

- 页面地图
- 一条主流程
- 3-5 个核心页面的低保真线框

**不要做什么**

- 不要先做视觉细节
- 不要先纠结字体颜色
- 不要一开始画所有次级页面

**进入下一步条件**：已经能完整讲清用户从打开 App 到完成核心任务，会经过哪些页面和动作。

### 5. UI Prototype

**要解决的问题**：结构稳定后，界面外观和点击体验是否合理。

**输入**

- 页面地图
- 主流程
- 低保真线框
- 初步视觉方向

**输出**

- 中高保真界面稿
- 可点击原型
- 核心组件视觉样式

**最小可用**

不需要全量页面，只要把主链路上的核心页面做成中保真，并能点击演示主流程。

**不要做什么**

- 不要先做全量视觉系统
- 不要把所有边角页面都画完
- 不要把“好看”放在“可推演”前面

**进入下一步条件**：已经能用原型完整演示主流程，而且页面层级、主要操作、关键反馈没有明显歧义。

### 6. Design Spec

**要解决的问题**：开发时还会不会继续靠猜。

**输入**

- 确认过的原型
- 页面结构
- 组件草图
- 交互规则

**输出**

- Screen spec
- 导航规则
- 页面状态
- 组件契约
- 数据草稿
- 开发约束

**最小可用**

每个核心页面至少写清：

- 页面目标
- 入口 / 出口
- 主状态
- 核心组件
- 用户动作与结果

每个核心组件至少写清：

- 输入
- 状态
- 交互
- 输出事件

**不要做什么**

- 不要把设计文档写成截图堆砌
- 不要重复原型里肉眼可见的东西
- 不要混入太多实现细节

**进入下一步条件**：开发者不看你脑子里的想法，只看文档，也能开始搭页面壳。

### 7. Build Plan

**要解决的问题**：设计已经清楚了，但代码该从哪儿开始。

**输入**

- Design Spec
- 原型
- 技术约束

**输出**

- 页面开发顺序
- 组件优先级
- Mock data
- 第一轮验证点
- 暂缓项

**最小可用**

回答 5 个问题：

- 第一个先做哪个页面
- 哪些组件要优先抽
- Mock data 怎么准备
- 第一轮要验证什么
- 哪些先不做

**不要做什么**

- 不要直接开始全量开发
- 不要同时开太多页面
- 不要一开始就做完整架构

**进入下一步条件**：已经知道第一轮开发先做什么，而不是只有“总之开始写”。

### 8. Development

**要解决的问题**：把设计转成可运行代码。

**输入**

- Build Plan
- Design Spec
- 原型
- Mock data

**输出**

- 页面壳
- 核心组件
- 主链路可运行
- 初步数据联通

**最小可用**

先让主链路跑通，不追求全部功能完整。

**不要做什么**

- 不要边写边改产品定义
- 不要把设计阶段的问题推迟到代码里再想
- 不要过早优化架构

**进入下一步条件**：主路径真的能跑，而不是只写出很多页面碎片。

### 9. Verify

**要解决的问题**：这轮工作是否真的闭环，能否进入下一轮。

**输入**

- 当前实现
- 原型
- 设计文档
- 预期主链路

**输出**

- 验证结论
- 问题清单
- 回写文档
- 下一轮调整建议

**最小可用**

只检查 3 件事：

- 主链路是否跑通
- 设计与实现是否明显偏离
- 下一轮最该修什么

**不要做什么**

- 不要只看能不能编译
- 不要不回写设计文档
- 不要把问题留在脑子里

**进入下一步条件**：知道这一轮做成了什么、没做成什么、下一轮从哪里继续。

## 最小启动模板

如果现在就要开始，不要同时启动整套流程。先写这一张卡：

```text
阶段名：Idea

我想做什么：
为谁做：
解决什么问题：
为什么值得单独做成 App：
当前明显不做什么：
```

写完以后，再进入 `Product Brief`。

## 执行原则

这套工作流可以压缩成一句话：

> 每一步只回答当前阶段必须回答的问题，不提前解决下一阶段的问题。

这样做的目的不是降低标准，而是让每一轮工作都有明确边界。先把最小必要产物做出来，再根据验证结果进入下一阶段或回到上一阶段调整。
