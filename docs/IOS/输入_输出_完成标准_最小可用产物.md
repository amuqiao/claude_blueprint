有，而且这是你现在最缺的一层。

如果没有每一步的 **输入 / 输出 / 完成标准 / 最小可用产物**，方法论就会停留在概念层。你一旦想法很多，就会立刻卡在两个地方：

- 不知道该从哪一步开始
- 不知道当前这一步什么时候算“够了”，可以进入下一步

所以一套可复用的开发范式，最好不是只有阶段名，而是每个阶段都带 4 个东西：

1. `输入`：这一阶段基于什么开始
2. `输出`：这一阶段结束时必须留下什么
3. `最小可用`：做到什么程度就够，不追求一步到位
4. `进入下一步条件`：什么情况下可以继续推进

这 4 个东西比“写得多详细”更重要。

---

**你应该这样理解整条链路**

不是：

```
我有一个想法 -> 我开始做
```

而是：

```
我有一个想法
-> 我先判断它是否足以进入下一阶段
-> 我只产出这一阶段最小必要产物
-> 满足进入条件后再往下走
```

这样你就不会被大量想法淹没。

---

**我建议给每一步都配一张“阶段卡”**

格式固定成这样：

```
阶段名：
这一阶段要解决的问题：
输入：
输出：
最小可用：
不要做什么：
进入下一步条件：
```

下面我直接给你一套可执行版本。

---

## 1. Idea（想法起点）

**要解决的问题**

这个东西到底是不是一个值得启动的 App 想法。

**输入**

- 零散想法
- 问题感受
- 一些功能念头
- 用户场景直觉

**输出**

- 一句话项目定义
- 目标用户
- 核心问题
- 为什么值得做

**最小可用**
只要写清下面 4 句就够：

```
我想做什么：
为谁做：
解决什么问题：
为什么值得单独做成 App：
```

**不要做什么**

- 不要画页面
- 不要定技术栈
- 不要列一堆功能

**进入下一步条件**
你已经能一句话说清这个 App 的存在理由。

---

## 2. Product Brief（项目简述）

**要解决的问题**

把想法从“灵感”变成“可讨论的项目定义”。

**输入**

- Idea 阶段的一句话定义
- 初步场景和用户判断

**输出**

- 项目目标
- 用户画像
- 核心价值
- 初步功能方向
- 暂不处理范围

**最小可用**
一页文档即可，回答这 5 个问题：

- 这是什么 App
- 给谁用
- 解决什么问题
- 用户为什么会用
- 当前明显不做什么

**不要做什么**

- 不要进入页面细节
- 不要写成完整 PRD
- 不要开始讨论组件

**进入下一步条件**
你已经能区分“核心价值”和“顺手想到的附加功能”。

---

## 3. MVP Scope（第一版范围）

**要解决的问题**

第一版到底做什么，不做什么。

**输入**

- Product Brief
- 初步功能想法列表

**输出**

- MVP 功能清单
- 非 MVP 清单
- 主链路
- MVP 页面列表

**最小可用**
只要有下面这张表就够：

```
第一版做：
第一版不做：
主链路：
必须页面：
```

**不要做什么**

- 不要追求“完整产品”
- 不要把未来想法都塞进 MVP
- 不要开始画高保真原型

**进入下一步条件**
你已经能回答：
“如果我只做第一版，用户最核心的一条路径是什么？”

---

## 4. UX Skeleton（结构化原型骨架）

**要解决的问题**

产品结构、页面关系、用户流程是否清楚。

**输入**

- MVP Scope
- 核心场景
- MVP 页面列表

**输出**

- 页面地图
- 主用户流程
- 低保真线框
- 页面入口 / 出口

**最小可用**
至少产出这 3 个：

1. 页面地图
2. 一条主流程
3. 3-5 个核心页面的低保真线框

**不要做什么**

- 不要先做视觉细节
- 不要先纠结字体颜色
- 不要一开始画所有次级页面

**进入下一步条件**
你已经能完整讲一遍：
“用户从打开 App 到完成核心任务，会经过哪些页面和动作。”

---

## 5. UI Prototype（界面原型）

**要解决的问题**

结构稳定后，界面外观和点击体验是否合理。

**输入**

- 页面地图
- 主流程
- 低保真线框
- 初步视觉方向

**输出**

- 中高保真界面稿
- 可点击原型
- 核心组件视觉样式

**最小可用**
不需要全量页面。

只要把 **主链路上的核心页面** 做成中保真，并能点击演示主流程。

**不要做什么**

- 不要先做全量视觉系统
- 不要把所有边角页面都画完
- 不要把“好看”放在“可推演”前面

**进入下一步条件**
你已经能用原型完整演示主流程，而且页面层级、主要操作、关键反馈没有明显歧义。

---

## 6. Design Spec（设计规格文档）

**要解决的问题**

开发时还会不会继续靠猜。

**输入**

- 确认过的原型
- 页面结构
- 组件草图
- 交互规则

**输出**

- screen spec
- 导航规则
- 页面状态
- 组件契约
- 数据草稿
- 开发约束

**最小可用**
每个核心页面至少要写清：

- 页面目标
- 入口 / 出口
- 主状态
- 核心组件
- 用户动作与结果

每个核心组件至少要写清：

- 输入
- 状态
- 交互
- 输出事件

**不要做什么**

- 不要把设计文档写成截图堆砌
- 不要重复原型里肉眼可见的东西
- 不要混入太多实现细节

**进入下一步条件**
开发者不看你脑子里的想法，只看文档，也能开始搭页面壳。

---

## 7. Build Plan（开发实施计划）

**要解决的问题**

设计已经清楚了，但代码该从哪儿开始。

**输入**

- Design Spec
- 原型
- 技术约束

**输出**

- 页面开发顺序
- 组件优先级
- mock data
- 第一轮验证点
- 暂缓项

**最小可用**
只要回答这 5 个问题就够：

- 第一个先做哪个页面
- 哪些组件要优先抽
- mock data 怎么准备
- 第一轮要验证什么
- 哪些先不做

**不要做什么**

- 不要直接开始全量开发
- 不要同时开太多页面
- 不要一开始就做完整架构

**进入下一步条件**
你已经知道“第一周先做什么”，而不是只有“总之开始写”。

---

## 8. Development（开发实现）

**要解决的问题**

把设计转成可运行代码。

**输入**

- Build Plan
- Design Spec
- 原型
- mock data

**输出**

- 页面壳
- 核心组件
- 主链路可运行
- 初步数据联通

**最小可用**
先让主链路跑通，不求全部功能完整。

**不要做什么**

- 不要边写边改产品定义
- 不要把设计阶段的问题推迟到代码里再想
- 不要过早优化架构

**进入下一步条件**
主路径真的能跑，而不是只写出很多页面碎片。

---

## 9. Verify（验证与回看）

**要解决的问题**

这轮工作是否真的闭环，能否进入下一轮。

**输入**

- 当前实现
- 原型
- 设计文档
- 预期主链路

**输出**

- 验证结论
- 问题清单
- 回写文档
- 下一轮调整建议

**最小可用**
只检查 3 件事：

- 主链路是否跑通
- 设计与实现是否明显偏离
- 下一轮最该修什么

**不要做什么**

- 不要只看“能不能编译”
- 不要不回写设计文档
- 不要把问题留在脑子里

**进入下一步条件**
你知道这一轮做成了什么、没做成什么、下一轮从哪里继续。

---

# 最关键的一点

你现在最需要的，不是直接开始写 5 份文档。

而是先建立一个 **“每一阶段都只做最小必要产物”** 的习惯。

你可以把整条工作流压缩成一句执行原则：

> 每一步只回答当前阶段必须回答的问题，不提前解决下一阶段的问题。
> 

这会极大降低你“想法太多、不知道怎么开始”的压力。

# 如果你现在就要开始，第一步怎么做

不要从整套流程一起开。

只做这一张卡：

```
阶段名：Idea（想法起点）

为谁做：
解决什么问题：
为什么值得单独做成 App：
当前明显不做什么：
```

写完以后，再进入 `Product Brief`。
