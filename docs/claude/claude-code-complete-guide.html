Claude Code 新用户完整指南

一个新用户，一个新项目，从安装到交付的完整路径。每个概念在第一次出现时都会先解释清楚再往下走。

1
先建心智模型

Claude Code 是运行在终端的 AI 编程助手。它的能力来自三个来源，每次 session 启动时按顺序叠加：

① 配置层
CLAUDE.md + rules/
session 启动时全量加载，始终在后台约束行为
② 能力层
Skills + Plugins
按需激活：意图匹配或显式调用
③ 对话层
你说的话
实时注入，优先级最高

能力层有两种形态，是本文档里最重要的一对概念：

Skill — 操作手册
一个 SKILL.md 文件。手动放进 ~/.claude/skills/ 即可使用。

触发：意图匹配 或 显式调用
加载：session 启动时只加载索引，匹配到意图时才加载正文
Plugin — 分发包，含 hook
可安装的包，可打包 skills、hooks、agents、MCP servers。通过插件市场安装。

触发：Hook 按事件自动触发（推送型），安装即生效
安装：/plugin install name@marketplace
关键区别：Plugin 里的 hook 是推送型的——按 session 启动、工具调用、session 退出等事件自动触发，不依赖意图匹配，安装即生效。Superpowers、Ralph Loop、Code Simplifier 是 Plugin，因为它们依赖 hook 实现自动化行为。
SkillPlugin
形态单个 SKILL.md 文件含 skills + hooks + agents 的分发包
安装手动放入 ~/.claude/skills//plugin install name@marketplace
触发意图匹配或显式调用Hook 按事件自动触发；skill 部分按意图触发
适合自定义工作流、操作手册需要 hook 自动化或打包分发的场景

3
能力层的五种机制

知道每种东西是什么，才知道装了之后会发生什么。

CLAUDE.md — 最高优先级约束

session 启动时自动全量读入，无条件生效。只放核心规则，控制在 100 行内。超过后 Claude 开始静默忽略，具体规范拆进 rules/。

rules/ — 主题化约束

CLAUDE.md 的模块化拆分。session 启动时全部 .md 文件全量读入，始终在后台约束行为。

frontmatter 里可写 paths 字段，告诉 Claude「只对匹配路径的文件执行」——但内容始终全量加载，paths 不控制是否加载，只是推理时的执行条件。paths: []（空数组）会导致规则从不执行，应删掉该字段。

Skills — 任务型操作手册

每个 skill 是一个目录，入口文件 SKILL.md。session 启动时只加载索引（name + description，约 30-50 token/个）。Claude 识别到任务意图匹配时才完整加载正文。也可显式调用：/skill-name 或自然语言点名。

Plugins — 分发包，含 hook 自动化

通过 /plugin install 安装。Plugin 可打包 skills、hooks、agents、MCP servers。其中 hook 是推送型的——按事件自动触发，安装即生效，不需要意图匹配。

# 浏览已安装的 Plugin
/plugin
# 官方市场安装
/plugin install name@claude-plugins-official
# 添加第三方市场后安装
/plugin marketplace add author/repo
/plugin install name@repo

agents/ — 独立子智能体

有独立上下文窗口和工具权限的子任务执行者，由主 session 委托启动，不污染主上下文。

推荐 Rules 配置

新项目起手四条，放系统级 ~/.claude/rules/，所有项目共享：

~/.claude/rules/git.md全局
---
description: Git 提交规范
---
提交信息必须符合 Conventional Commits 格式：<type>(<scope>): <subject>
type: feat / fix / docs / refactor / test / chore
每个提交只做一件事，功能完成后立即提交，不积攒大提交。
~/.claude/rules/testing.md全局
---
description: 测试要求
---
声明"测试通过"前必须展示实际运行输出，不接受假设性声明。
提交前必须跑完整测试套件。单元测试中外部服务必须 mock。
~/.claude/rules/workflow.md全局
---
description: 工作流约束
---
新功能开始前先澄清需求，不做猜测性实现。
声明任何状态前必须提供新鲜验证证据。
发现超出范围的问题，记录到 TODO，不立即扩展。
~/.claude/rules/security.md全局
---
description: 安全编码规范
---
禁止硬编码密钥、token、密码，只从环境变量读取。
所有用户输入必须校验，SQL 查询必须参数化。
不读取或修改 ~/.ssh、~/.aws 等敏感目录。

常见误解
✗Skill 和 Plugin 是同一回事
✓Skill 是操作手册（意图触发），Plugin 是分发包（含 hook 自动触发）。Superpowers 是 Plugin，不是因为能力强，而是因为它有 hook，装好就自动生效，不依赖你调用。
✗paths 控制规则是否加载进上下文
✓加载永远全量。paths 只告诉 Claude「这条规则只对匹配路径的文件执行」。paths: [] 应删掉。
✗CLAUDE.md 越详细越好
✓超过 100 行后 Claude 开始静默忽略。核心约束留 CLAUDE.md，细节拆进 rules/。
✗上下文压缩后任务就断了
✓Planning with Files 把状态写进文件，压缩后读文件恢复，无缝继续。
✗Rules 和 Skills 功能重复
✓Rules 是约束（始终在后台），Skills 是执行手册（按需激活）。前者说「什么不能做」，后者说「怎么做某件事」。
