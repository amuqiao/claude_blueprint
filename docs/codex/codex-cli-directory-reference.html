配置文件本身已经证明的事实
`~/.codex/config.toml` 的已验证作用

本机配置文件中已经直接出现了全局模型、推理强度、features 开关，以及项目 trust 记录。这意味着：Codex 至少把“用户默认值”和“项目是否已信任”统一收在同一个用户级 TOML 里。

字段/结构状态含义
model已验证默认模型
model_reasoning_effort已验证默认推理强度
[features]已验证特性开关，例如本机出现了 hooks = true
[projects."/abs/path"]已验证项目信任表；本机记录了 trust_level = "trusted"
-c key=value已验证运行时配置覆盖，优先级高于默认文件
--profile已验证从 config profile 载入一组默认值

仓库侧目录，这一轮只保留最小正确说法
为什么这里没有再画满屏 repo 树

上一版 `docs/codex` 把很多 repo 级目录也画成了“硬标准”，但本轮校验不足以支持那种写法。当前真正稳妥、能直接落到文档里的只有两件事：

项目 trust 记录在用户级 `config.toml` 中 当前机器的 `~/.codex/config.toml` 已经记录了多个 [projects."/abs/path"] 表，所以“项目是否被信任”这一层是已验证事实。
user/project rules 文件都被 CLI 正式认识 codex exec --help 暴露了 --ignore-rules，并明确写成“Do not load user or project execpolicy `.rules` files”。这能证明项目侧规则文件存在，但本轮没有继续把具体目录名字写死。
其余 repo 级约定暂时降级处理 例如 .codex/config.toml、AGENTS.md、自定义 prompts 目录、管理员强制层等，本轮不再画成“官方准确目录树”，避免把未证实的实现细节冒充标准。
这页如何阅读
优先看 `~/.codex/` 这是本机已经直接存在的主目录，也是本轮对比 Claude 页面时最可靠的参照面。
再看 `config.toml` 很多“项目层状态”其实先体现在用户级 config 里，比如 trust 记录。
对 repo 侧保持克制 不把当前没验证透的目录命名、自动加载规则、企业层配置写成铁律。
