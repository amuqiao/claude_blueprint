Codex CLI 新用户完整指南

这份页面现在采用与 `docs/claude` 相同的视觉语言，方便横向对比；但内容口径改成了 Codex CLI 自身已验证事实，不再把 Claude Code 的概念直接平移过来。

1
先看定位

Codex CLI 是在终端里运行的本地编码代理。当前安装的 `codex --help` 明确暴露了交互式会话、非交互执行、代码审查、MCP、插件、沙箱、桌面 App、会话恢复与 fork 等能力，所以它不是“单纯生成代码片段”的补全器，而是一套以仓库为工作单位的执行式 CLI。

产品角色
本地终端代理
面向仓库、命令、文件与会话，不只是编辑器补全。
核心入口
`codex`
不带子命令时进入交互式 CLI，会话可持续推进任务。
验证方式
本机 CLI + OpenAI 页面
本页依据 `codex` 帮助输出、当前 `~/.codex/` 实存目录，以及 OpenAI 官方 Codex 页面整理。
这次改版的原则：只保留当前安装能够直接证明、或 OpenAI 官方页面已经明确提到的内容；无法硬证实的目录、命令别名、强制配置层都不再写死。
2
已验证的命令面
`codex --help` 暴露的一级能力
命令作用

`codex` 直接进入会话，适合持续协作、边看边改、带审批地执行命令。
非交互式
`codex exec` 与 `codex review` 更适合脚本、CI、批处理或把输出接进别的流程。
3
配置与信任

当前安装已经直接证明：用户级核心配置文件位于 ~/.codex/config.toml。此外，命令行支持 -c key=value、-m、-p、-s、-a 等运行时覆盖项，所以 Codex 的配置思路是“用户默认值 + 单次运行覆盖”。

本机 `~/.codex/config.toml` 中已出现的真实字段
model = "gpt-5.4"
model_reasoning_effort = "medium"
approvals_reviewer = "user"

[projects."/Users/admin/Downloads/Code/claude_blueprint"]
trust_level = "trusted"

[features]
hooks = true
1
用户全局默认值写在 ~/.codex/config.toml，控制模型、特性开关、以及已信任项目的记录。
2
项目信任状态当前安装把仓库 trust 记录放在用户级 config 的 [projects."绝对路径"] 表里，而不是单独塞进仓库内部状态文件。
3
单次运行覆盖-c、--profile、--model、--sandbox、--ask-for-approval 都是更高优先级的运行时参数。
这次有意删掉的旧说法：requirements.toml、企业强制层、`AGENTS.override.md`、以及若干“越近目录越优先”的扫描规则，这一轮都没有足够硬的证据支持，因此不再作为 Codex CLI 事实写入。

登录与模型

`codex login --help` 已经给出了三类可验证入口：API Key、Access Token、以及设备授权。主命令还暴露了 --model、--oss、--local-provider、--profile 等模型和提供方切换参数。

可直接从 CLI 看到的登录与模型切换方式
能力已验证入口说明

Access Token 登录codex login --with-access-token从 stdin 读取访问令牌

5
审批与沙箱

这部分是 Codex CLI 和其他 coding agent 最容易混淆的地方。当前安装给出的正式参数名是 --ask-for-approval 与 --sandbox，而不是别家的命名。

审批策略
值含义
untrusted只放行“可信命令”，其余升级为用户审批
on-request由模型决定何时请求审批
never不向用户请求审批，失败直接返回模型
on-failureCLI 已标记为 deprecated
沙箱模式
值含义
read-only只读执行
workspace-write允许写工作区
danger-full-access完整访问，本机使用风险最高
高风险旗标：--dangerously-bypass-approvals-and-sandbox 会跳过审批并关闭沙箱。这一轮保留它，但只把它写成“隔离环境专用”，不再包装成日常快捷工作流。
`codex sandbox` 也是正式子命令：当前安装已暴露 macos、linux、windows 三个子命令，说明“沙箱”不是抽象概念，而是 CLI 的明确执行面。
6
扩展能力：MCP、Skills、Plugins

Codex CLI 的可扩展面不能简单等同于 Claude Code，但这三类能力现在都有证据链支持：MCP 来自正式子命令；Skills 在 OpenAI Codex use cases 中被明确提到；Plugins 则同时能从 CLI marketplace 子命令和本机插件缓存中看到。

MCP
`codex mcp`
已验证有 list/get/add/remove/login/logout。`mcp add` 既支持 URL，也支持以命令启动 stdio server。
Skills
`~/.codex/skills/`
本机目录存在，且官方 use cases 已把“Save workflows as skills”作为 Codex 工作流之一。
Plugins
marketplace + manifests
本机插件缓存可见 .codex-plugin/plugin.json、.app.json、.mcp.json、hooks.json。
这一轮保留的最小正确理解
1
MCP 是外部工具接入层通过 codex mcp add 配置，本机 `mcp list --json` 可以枚举已配置服务器。
2
Skill 是可复用工作流单元本机有系统 Skills 目录，官方页面也把“保存工作流为 skills”作为 Codex use case。
3
Plugin 是打包分发单元当前 CLI 公开的是 marketplace 管理面；安装细节和 repo 级目录规范这一轮只写入已看到的 manifest 结构，不再继续脑补。
7
推荐流程
1
先进仓库再启动cd repo && codex。如果不是 Git 仓库，非交互执行可显式加 --skip-git-repo-check。
2
把默认值收进用户 config模型、reasoning effort、特性开关、已信任项目，统一放到 ~/.codex/config.toml。
3
把自动化需求放进 `exec` / `review`codex review --uncommitted 适合提交前检查；codex exec --json 适合脚本和 CI。
4
扩展优先级先接 MCP，再沉淀 Skill，最后才考虑 Plugin 打包。这样最符合当前 CLI 暴露出来的层次。
8
与 Claude Code 的区别
不要直接复用 Claude 的目录心智模型
Codex 的已验证用户目录是 ~/.codex/，并且项目 trust 状态明确存于用户 config 的 [projects] 表里。
参数命名也不同
Codex 公开的是 --ask-for-approval、--sandbox、--dangerously-bypass-approvals-and-sandbox 等参数，不应替换成别家的口径。
因此这份文档刻意没有照抄 Claude 页面里的：requirements.toml、大段 repo 级目录树、未验证 slash 命令集合、以及“某目录一定自动加载”的强断言。
9
本文边界

这不是 OpenAI 官方文档的复刻版，而是一个“对照版可视化整理页”。它的目标不是写得最全，而是写得足够稳：方便和 `docs/claude` 对比时，知道哪些是 Codex CLI 的真实结构，哪些目前只应该保守看待。
