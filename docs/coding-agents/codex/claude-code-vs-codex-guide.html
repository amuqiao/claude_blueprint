关键分工：Instructions 文件 vs 配置文件

这是新用户最常混淆的点。两类文件的职责完全不同：

CLAUDE.md / AGENTS.md — 告诉 AI"应该知道什么"
  • 项目架构、技术栈背景
  • 团队编码规范和约定
  • 常用命令（npm test 等）
  • AI 的行为偏好（语气、输出格式）
  • 业务领域知识
ℹ 这是"上下文注入"，每次会话开始时 AI 会读取它，形成背景知识。
settings.json / config.toml — 告诉工具"可以做什么"
  • 允许/拒绝哪些 bash 命令
  • 使用哪个模型
  • 沙盒模式（文件读写范围）
  • MCP 服务器连接
  • Hooks 配置
⚠ 这是"行为控制"，决定工具本身的能力边界，与 AI 理解的内容无关。

Codex 独有：Trust 机制（必读）

⚠ 踩坑高发区

Codex 的项目级配置（.codex/ 目录下的所有内容）默认不生效，必须显式信任该项目才能加载。

为什么？ 因为 .codex/config.toml 可能来自 git clone 的陌生仓库，里面可能包含恶意 Hook 或危险的 approval_policy = "never"。信任机制防止你 clone 一个项目后 AI 自动执行不安全的操作。

怎么信任？ 在项目根目录运行 Codex 时，它会提示你是否信任该项目，确认后写入信任记录。或在 config.toml 中设置 trust = "trusted"。

✓ Claude Code 没有信任机制，项目配置始终自动加载。两边行为不同，不要混淆。

Instructions 文件的叠加行为

CLAUDE.md 和 AGENTS.md 在多层之间的行为与配置文件不同——它们是叠加（全部读取，拼接到上下文），而不是覆盖。

叠加顺序

Claude Code：~/.claude/CLAUDE.md → CLAUDE.md（项目根）→ src/CLAUDE.md（子目录）— 从外到内依次读取，全部注入。

Codex：优先读取 AGENTS.override.md（如存在）→ AGENTS.md → 子目录 AGENTS.md，有 project_doc_max_bytes（默认 32KB）上限。

⚠ Codex 32KB 限制：所有层的 AGENTS.md 合计超过 32KB 时，内容被截断。最重要的指令要放在文件最前面。

Rule — 权限与行为控制

Rule 控制"AI 被允许做什么操作"。这和 Instructions 不同——Instructions 是 AI 读的知识，Rule 是工具本身的执行策略。

● Claude Code — permissions

配置位置：settings.json 的 permissions 字段

三种动作：

  • allow：直接允许，无需确认
  • deny：直接拒绝
  • ask：每次询问用户

支持通配符：Bash(git *) 匹配所有 git 命令

跨层合并：allow/deny 数组在所有层合并，不互相覆盖

● Codex — approval_policy + sandbox

配置位置：config.toml

审批策略（approval_policy）：

  • on-request：每次询问（默认）
  • never：全部自动批准
  • untrusted：最严格，每步确认

沙盒模式（sandbox_mode）：

  • workspace-write：只能写工作目录
  • read-only：只读
  • danger-full-access：无限制

Claude Code — settings.json permissions 完整模板

.claude/settings.json — 权限配置模板
{
  "permissions": {
    "allow": [
      // Git 操作：允许常用 git 命令，无需确认
      "Bash(git status:*)",
      "Bash(git add:*)",
      "Bash(git diff:*)",
      "Bash(git log:*)",
      "Bash(git branch:*)",
      "Bash(git checkout:*)",

      // 包管理：允许安装和构建
      "Bash(npm install:*)",
      "Bash(npm run:*)",
      "Bash(npm test:*)",

      // 文件读写：允许常规操作
      "Read",
      "Write",
      "Edit"
    ],
    "deny": [
      // 严格禁止：危险操作
      "Bash(rm -rf:*)",
      "Bash(sudo:*)",
      "Bash(curl * | bash:*)",

      // 禁止直接推送到 main
      "Bash(git push * main:*)"
    ],
    "ask": [
      // 每次询问：git commit 需要人工确认
      "Bash(git commit:*)",
      "Bash(git push:*)"
    ]
  }
}

💡 Rule vs Hook 的区别

Rule（permissions）是静态声明——在工具启动前就确定"能做什么"，是能力边界的静态配置。
Hook是动态响应——在特定事件发生时执行脚本，是行为自动化。两者不互斥，常配合使用。

Hook — 事件驱动自动化

Hook 是"当某件事发生时，自动执行某段脚本"。常见用途：命令执行前检查、执行后通知、会话结束时清理。

● Claude Code — hooks 事件
  • PreToolUse：工具调用前（可阻止）
  • PostToolUse：工具调用后
  • Stop：AI 停止响应后
  • Notification：通知事件
  • SessionEnd：会话结束

配置位置：settings.json 的 hooks 字段

● Codex — hooks 事件
  • PreToolUse：工具调用前
  • PostToolUse：工具调用后
  • （事件体系与 Claude Code 基本一致）

配置位置：hooks.json 独立文件，或 config.toml 的 [hooks] 表

注意：项目级 hooks 只在信任状态下加载

Claude Code — hooks 完整模板

Skill 是"教会 AI 一种工作方法"的能力包。AI 根据任务描述自动发现并调用合适的 Skill，无需用户主动触发。

  3. $REPO_ROOT/.agents/skills/
  4. ~/.agents/skills/（全局）

● Codex — Skill

  1. .agents/skills/（当前目录）
  2. 父目录 .agents/skills/
  3. ~/.agents/skills/（全局）
  4. /etc/codex/skills/（系统级）
  5. 内置 Skill

💡 Skill vs Command 的核心区别

Skill：AI 主动发现，自然语言触发，适合"教 AI 怎么做某类工作"。比如"生成 API endpoint 时，按照我们项目的模板来"。
Command：用户主动召唤（用 /commandname），适合"我要触发一个固定操作"。比如 /commit 生成一个规范的 commit message。

# API 脚手架 Skill

## 使用场景
用户请求创建新的 REST API 端点时自动激活。

## 执行步骤

### 1. 理解需求
- 确认端点路径（如 `/api/users/:id`）
- 确认 HTTP 方法（GET/POST/PUT/DELETE）
- 确认请求/响应数据结构

### 2. 生成文件
按以下顺序生成文件：

**路由文件**（packages/api/src/routes/）：
```typescript
import { Router } from 'express';
import { handler } from '../controllers/xxxController';

const router = Router();
router.get('/xxx', handler);
export default router;
```

**Zod Schema**（packages/api/src/schemas/）：
```typescript
import { z } from 'zod';

export const XxxRequestSchema = z.object({
  // 根据需求填写
});
```

决策工具

带着"我要做 X"的问题来查，找到最合适的工具和方法。

决策树 1：我要配置什么能力？

❓ 我想做的是什么？
→ 给 AI 注入项目背景、规范、约定
✅ 写 CLAUDE.md / AGENTS.md（Instructions）
→ 控制 AI 能执行哪些命令
✅ 配置 Rule：Claude Code → permissions，Codex → approval_policy + sandbox_mode
→ 某件事发生时自动执行脚本（如写文件后 lint）
✅ 配置 Hook：Claude Code → settings.json hooks，Codex → hooks.json
→ 教会 AI 一种固定的工作方法（AI 自动选用）
✅ 创建 Skill：.agents/skills/xxx/SKILL.md
→ 我要主动触发一个固定操作（用 / 召唤）
✅ Claude Code → .claude/commands/xxx.md
Codex → 用 Skill 或 codex exec 替代
→ 接入外部服务（GitHub、数据库、Slack）
✅ 配置 MCP：Claude Code → .mcp.json，Codex → config.toml [mcp_servers]

✅ Codex → Profile（config.toml [profiles.xxx]）

决策树 3：Rule 还是 Hook？

→ 我想在 AI 写文件后自动跑测试

排错指南

配置了但没生效？行为和预期不一致？从这里找原因。

高频问题 TOP 10

Codex

问题 1：项目配置完全没有生效

症状：明明在 .codex/config.toml 里改了配置，运行 Codex 后完全没效果。

原因：Codex 的信任机制——项目未被信任，.codex/ 目录被完全跳过。

解决：

  1. 在项目根目录启动 Codex，看到信任提示时确认
  2. 或检查 ~/.codex/config.toml，确认是否有 trust = "untrusted" 误配置
⚠ 用户全局配置（~/.codex/）不受信任机制影响，始终生效。只有项目级 .codex/ 需要信任。
Claude Code

问题 2：MCP 服务器连接失败

症状：配置了 MCP 服务器，但 Claude 说找不到对应工具。

常见原因：

  1. 路径错误：用户级 MCP 是 ~/.claude.json（不是 ~/.claude/mcp.json）
  2. 格式错误：顶级字段是 mcpServers（camelCase），不是 mcp_servers
  3. 环境变量未设置：MCP 配置里引用了 ${GITHUB_TOKEN}，但 shell 里没有这个变量
正确的 ~/.claude.json 格式
{
  "mcpServers": {          ← 注意是 mcpServers，不是 mcp_servers
    "github": {
      "type": "http",
      "url": "https://api.githubcopilot.com/mcp/"
    }
  }
}
两者通用

问题 3：AI 不理解我的项目规范

症状：明明在 CLAUDE.md / AGENTS.md 里写了规范，AI 还是不遵守。

排查步骤：

  1. 确认文件位置：CLAUDE.md 必须在项目根目录，不是 .claude/ 目录里
  2. Codex 检查文件大小：超过 32KB 的内容会被截断，最重要的规范放最前面
  3. 规范写得太模糊：AI 更容易理解具体的"做 X"而不是抽象的"遵循规范"
规范写法的好与坏
❌ 不好的写法（太模糊）：
遵循良好的编码实践。

✅ 好的写法（具体可执行）：
- 所有函数超过 20 行必须拆分
- 使用命名导出，禁止 export default
- API 响应必须用 Zod schema 校验
- 变量名用 camelCase，常量用 UPPER_SNAKE_CASE
Claude Code

问题 4：permissions 配置了 deny 但 AI 还是执行了

原因：多层 permissions 数组合并时，某一层的 allow 覆盖了另一层的 deny。

规则：deny 的优先级高于 allow。如果全局 deny 了 Bash(sudo:*)，项目层 allow 它也没用。但要确认 deny 确实在生效的那一层。

调试方法：用 /permissions 命令（如果支持）查看当前生效的权限组合，或检查各层 settings.json。

两者通用

问题 5：Hook 没有被触发

排查清单：

  • Codex：项目 hooks.json 只在信任状态下加载，先确认项目已信任
  • matcher 字段是否正确匹配工具名（大小写敏感）
  • Hook 脚本是否有执行权限：chmod +x .claude/hooks/xxx.sh
  • 脚本路径是相对路径时，相对的是工作目录还是配置文件所在目录？
  • Codex：检查是否同时有 hooks.json 和 [hooks] 表（只能二选一）
两者通用

问题 6：Skill 没有被 AI 自动调用

排查步骤：

  1. 检查 SKILL.md 的 description 字段是否足够具体，包含触发词
  2. 确认文件路径：必须是 .agents/skills/xxx/SKILL.md，目录层级要对
  3. description 过于宽泛会导致 AI 不知道该在什么场景用
Skill description 的好与坏
❌ 太模糊：
description: "帮助处理代码相关任务"

✅ 具体且包含触发词：
description: >
  当用户要求创建 API 端点、新增接口、添加路由时，
  按照项目规范自动生成路由文件、Controller 和测试文件。
  触发词：create endpoint、新增接口、添加 API。
Claude Code

问题 7：Command 调用时上下文命令输出为空

症状：Command 文件里用了 !`git diff HEAD`，但输出是空的。

原因：shell 命令没有在正确的工作目录执行，或命令本身有错误。

解决：在命令里加绝对路径，或确认工作目录是项目根。可以先在终端手动执行确认命令有输出。

Codex

问题 8：AGENTS.md 内容被截断

症状：AGENTS.md 下半部分的规范 AI 完全不理解，像是没读到。

原因：所有层的 AGENTS.md 合计超过 project_doc_max_bytes（默认 32KB）。

解决方案：

  • 将最重要的内容移到文件最前面
  • 将详细参考资料移到 Skill 文件中（按需加载，不占 AGENTS.md 额度）
  • 在 config.toml 中增大限制：project_doc_max_bytes = 65536（64KB）
两者通用

问题 9：敏感信息（API Key）泄露到配置文件

🚨 高危：绝对不要把 API Key、数据库密码等敏感信息写死在配置文件里，特别是要提交 git 的文件。

正确做法：使用环境变量引用

正确的敏感信息处理方式
# Claude Code .mcp.json ── 引用环境变量
{
  "env": {
    "GITHUB_TOKEN": "${GITHUB_TOKEN}"  ← 从环境变量读取
  }
}

# Codex config.toml ── 使用 env: 前缀
[mcp_servers.github]
env = { GITHUB_TOKEN = "env:GITHUB_TOKEN" }  ← env: 前缀表示读环境变量

# 在 shell profile 里设置（~/.zshrc 或 ~/.bashrc）
export GITHUB_TOKEN="ghp_xxxxxxxxxxxx"
export DATABASE_URL="postgres://user:pass@host/db"
两者通用

问题 10：团队成员配置行为不一致

症状：同一个项目，A 成员的 AI 行为和 B 成员不同。

排查：

  • 检查每个人的全局配置（~/.claude/settings.json 或 ~/.codex/config.toml）是否有不同的默认值覆盖了项目配置
  • Codex：检查每个人是否都信任了该项目（未信任则项目配置不生效）
  • 检查 .claude/settings.local.json 是否被某人误提交了 git

预防措施：

  • 在 .gitignore 里加入 .claude/settings.local.json
  • 在项目 README 里写清楚：clone 后需要信任项目（Codex）、需要安装哪些依赖
  • 在 CLAUDE.md / AGENTS.md 里写明团队约定，用文档而非靠个人记忆
