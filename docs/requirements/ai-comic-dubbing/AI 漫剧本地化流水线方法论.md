# AI 漫剧本地化流水线方法论

> 把「mp4 + srt 输入，目标语言 mp4 + srt 输出」这一反复出现的生产问题，抽象成对象清楚、节点稳定、每步可干预、技术路线可替换的方法论。

```text
AI 漫剧本地化流水线方法论
├── Node 1：Ingest           输入解析与校验
├── Node 2：Segmentation     对白分段建模
├── Node 3：Localization     文本本地化
├── Node 4：Speaker Binding  角色与音色绑定
├── Node 5：TTS Generation   配音生成
├── Node 6：Alignment        时长对齐
├── Node 7：Audio Mix        音轨混音
├── Node 8：Subtitle Render  字幕渲染
└── Node 9：Export           审核与导出
```

---

## 核心命题

这个问题不是「翻译字幕 + 调一次 TTS」，而是把原视频的对白生产链完整重走一遍。

每个节点都可能出错，且错误会向下游传播。全自动是理想态，但每个节点必须支持人工干预——不是简单的 yes/no 确认，而是真正能修改输入、调整参数、重新执行、覆盖结果。

方法论的核心转化是：

```text
mp4 + srt（原语言或已译）
  -> 解析与校验
  -> 对白分段建模（Segment 是最小处理单元）
  -> 文本本地化
  -> 角色识别 + 音色绑定
  -> 目标语言配音生成
  -> 时长对齐
  -> 音轨混音（保留背景音）
  -> 字幕渲染（软字幕 / 硬字幕覆盖）
  -> 人工审核 + 导出
```

技术路线默认混合：第三方 API 快速验证，开源方案并行评测，自研后置。

## 方法论节点

### Node 1：Ingest — 输入解析与校验

**这个节点解决什么问题**

所有后续节点依赖输入数据的完整性。如果 mp4 编码异常、srt 时间轴错位、字幕段和视频时长不匹配，越往下游跑，问题越难定位。

**输入**

| 输入 | 说明 |
| --- | --- |
| `mp4` | 原视频，可能带软字幕轨或硬字幕 |
| `srt` | 原语言字幕 **或** 已翻译好的目标语言字幕（两种情况均需支持） |
| 元信息配置 | 源语言、目标语言（支持多选，默认 2 种，可扩充）、是否已翻译 |

**输出**

| 输出 | 说明 |
| --- | --- |
| 媒体元信息 | 视频时长、分辨率、帧率、音轨采样率、声道数 |
| 字幕类型判断 | 软字幕 / 硬字幕，及其位置区域估计 |
| SRT 校验报告 | 时间越界、重叠、空文本、乱序、编码异常的列表 |
| 原始 Segment 列表 | 每条 srt 条目转成结构化对白段 |

**可能遇到的问题**

| 问题 | 原因 | 处理建议 |
| --- | --- | --- |
| srt 时间轴与音轨不对齐 | 字幕是从第三方来源拼来的，和视频制作批次不同 | 自动检测偏移量，提供全局时移修正入口 |
| srt 编码异常 | 非 UTF-8 编码，日韩文常见 | 自动检测并转换，失败则提示手动修正 |
| mp4 含多条音轨 | 原视频已有多语言轨 | 列出所有轨道，让用户选择目标提取轨 |
| 硬字幕位置识别失败 | 字幕位置非标准底部或透明度较低 | 提供手动框选字幕区域的入口 |
| srt 已是目标语言 | 用户上传的已是译文，不需要再翻译 | 标记字段 `is_translated: true`，跳过 Node 3 翻译步骤 |

**人工干预入口**

- 修正 srt 全局时间偏移（毫秒级滑块）
- 手动标注硬字幕区域（画框选择）
- 选择要处理的音轨
- 修正 srt 中异常段（时间重叠、空文本）
- 切换「已翻译 / 未翻译」模式

**技术选型**

- 视频/音频元信息提取：`ffprobe`
- SRT 解析与校验：`pysrt` / `srt`
- 硬字幕检测：`PaddleOCR` / `EasyOCR`
- 编码检测：`chardet` / `charset-normalizer`

详细对比和选型依据见 [`AI配音替换技术坐标与方案选型.md`](./AI配音替换技术坐标与方案选型.md#node-1-ingest-所需技术)。

### Node 2：Segmentation — 对白分段建模

**这个节点解决什么问题**

SRT 的分段服务于字幕阅读，不天然服务于配音生产。一条 srt 可能太长（一句话跨多个镜头），也可能太短（被拆成两行）。这个节点把 srt 条目转成「可生成、可编辑、可独立重试」的最小配音单元 Segment。

**输入**

- 原始 Segment 列表（来自 Node 1）
- 可选：原视频音轨（用于辅助对齐验证）

**输出**

每个 Segment 包含：

```text
id
source_index         原 SRT 序号（可多对一，支持合并）
start_time_ms
end_time_ms
duration_ms          时间窗长度
source_text          原对白文本
target_text          目标语言文本（此时为空，等 Node 3 填充）
speaker_id           说话人（此时可能为空，等 Node 4 填充）
voice_profile_id     音色配置（等 Node 4 填充）
merge_group          合并组 ID（若多条 srt 合并为一段）
split_from           若从某条 srt 拆分，记录来源 index
duration_status      ok / tight / overflow（预估）
generation_status    pending
review_status        pending
```

**可能遇到的问题**

| 问题 | 原因 | 处理建议 |
| --- | --- | --- |
| 单条 srt 文本过长 | 字幕编辑时没有考虑配音长度 | 标记为「建议拆分」，提供可视化拆分入口 |
| 相邻两条 srt 语义上是一句话 | 字幕为阅读换行 | 标记为「建议合并」，提供合并入口 |
| 时间窗太短（<0.5s） | 快速短句或标点段 | 标记为「极短段」，提示可能无法生成自然配音 |
| 说话人切换频繁 | 多人快速来回对话 | 每条 segment 独立处理，speaker_id 后续绑定 |

**人工干预入口**

- 合并相邻 segment（拖拽或勾选合并）
- 拆分单条 segment（选择断句位置）
- 修改 source_text（修正 OCR 或 srt 错误）
- 调整单条 start_time / end_time（毫秒精度）
- 锁定某条 segment（锁定后不被批量操作覆盖）

**技术选型**

- 辅助对齐验证（可选）：`WhisperX`
- 说话人辅助识别（可选）：`pyannote-audio`
- 文本长度预估：基于目标语言字符/音节速率的规则

详细对比和选型依据见 [`AI配音替换技术坐标与方案选型.md`](./AI配音替换技术坐标与方案选型.md#node-2-segmentation-所需技术)。

### Node 3：Localization — 文本本地化

**这个节点解决什么问题**

把每个 Segment 的 source_text 转成目标语言 target_text，同时兼顾角色语气、配音时长约束、术语一致性。如果用户上传的 srt 已经是目标语言（`is_translated: true`），本节点跳过自动翻译，直接进入人工校对模式。

**输入**

- Segment 列表（含 source_text、时间窗、duration_ms）
- 源语言、目标语言
- 可选：角色词汇表、术语表、口吻约束（如「角色 A 说话简短粗暴」）
- `is_translated` 标记

**输出**

- 每个 Segment 的 target_text
- 每个 Segment 的 `length_warning`（预估目标语音时长是否超出时间窗）
- 翻译置信度或质量标记（可选）

**可能遇到的问题**

| 问题 | 原因 | 处理建议 |
| --- | --- | --- |
| 目标语言比原语言长很多 | 如日→法、日→英，音节密度差异大 | 自动标记 `length_warning`，提示缩短或走时间拉伸 |
| 角色口吻丢失 | 机器翻译抹平语气 | 支持角色口吻约束注入 LLM prompt |
| 专有名词错译 | 漫剧角色名、地名、招式名 | 支持术语表绑定，优先替换 |
| 上下文断裂 | 按单条 segment 翻译，不知道前后文 | 支持「批量上下文翻译」模式，多条 segment 一起送入 LLM |
| 已翻译 srt 有错误 | 用户自己准备的译文质量不稳定 | 提供逐条文本编辑入口，支持重新生成单条翻译 |

**人工干预入口**

- 逐条修改 target_text
- 锁定已确认的 target_text（锁定后批量重跑不覆盖）
- 对单条或选中多条重新翻译（选择 provider）
- 调整角色口吻约束（全局或单角色）
- 编辑术语表（实时生效）
- 切换翻译 provider 对比结果

**技术选型**

- 主力翻译：OpenAI GPT-4o / Claude / Gemini
- 轻量/高速翻译：DeepL API / Google Translate API
- 开源翻译：`Helsinki-NLP/opus-mt`
- 术语表管理：自建键值表 + 翻译后后处理替换
- 时长预估：目标语言字符/音节速率规则 + TTS 试算

详细对比和选型依据见 [`AI配音替换技术坐标与方案选型.md`](./AI配音替换技术坐标与方案选型.md#node-3-localization-所需技术)。

### Node 4：Speaker Binding — 角色与音色绑定

**这个节点解决什么问题**

多角色对白中，每条 Segment 必须知道「谁说的、用什么声音说」，否则同一角色跨句声音会随机漂移。这个节点建立 Speaker → VoiceProfile 的稳定映射。

**输入**

- Segment 列表（此时 speaker_id 可能为空）
- 可选：原视频音轨（用于自动说话人识别）
- 可选：用户提供的角色参考音频（用于音色克隆）

**输出**

- 每个 Segment 的 speaker_id
- Speaker 列表（含名称、描述）
- VoiceProfile 列表（含 provider、voice_id、参考音频路径、克隆参数）
- Speaker → VoiceProfile 映射关系

**说话人识别：两种模式**

| 模式 | 适用场景 | 做法 |
| --- | --- | --- |
| 人工标注 | segment 数量少、角色少、质量要求高 | 界面上逐条或批量指定 speaker |
| 自动识别 | segment 多、角色明确、有原音轨 | 用 diarization 工具聚类说话人，人工核对和命名 |

两种模式可混用：先跑自动识别，再人工修正错误标注。

**音色来源：两种模式**

| 模式 | 适用场景 | 做法 |
| --- | --- | --- |
| 从原视频提取 | 原音轨干净，角色声音清晰 | 分离人声，提取每个角色的参考音频片段 |
| 用户提供参考音频 | 有更干净的角色录音 | 直接上传，绑定到 VoiceProfile |

**可能遇到的问题**

| 问题 | 原因 | 处理建议 |
| --- | --- | --- |
| 自动识别说话人数量不对 | 背景音干扰、说话人声音相近 | 提供手动合并 / 拆分说话人簇的入口 |
| 原音轨有背景音，参考音频不干净 | 漫剧配乐覆盖人声 | 先做人声分离，再提取参考片段 |
| 角色数量多，绑定工作量大 | 长片 | 支持「复制音色配置到同角色所有 segment」 |
| 同一角色多种情绪，音色差异大 | TTS 克隆后情绪风格不稳定 | VoiceProfile 支持绑定情绪参数或多情绪音色变体 |

**人工干预入口**

- 手动为每条 segment 指定 speaker
- 批量为多条 segment 指定同一 speaker
- 新建 / 编辑 / 删除 Speaker
- 上传或替换参考音频
- 试听 VoiceProfile 效果（用固定测试文本）
- 修改 Speaker → VoiceProfile 映射

**技术选型**

- 说话人自动识别：`pyannote-audio`
- 辅助对齐 + 说话人：`WhisperX`
- 人声分离：`Demucs` / `UVR5`
- 第三方音色管理：ElevenLabs Voice Library

详细对比和选型依据见 [`AI配音替换技术坐标与方案选型.md`](./AI配音替换技术坐标与方案选型.md#node-4-speaker-binding-所需技术)。

### Node 5：TTS Generation — 配音生成

**这个节点解决什么问题**

按每条 Segment 的 target_text + VoiceProfile，调用 TTS / 音色克隆能力，生成目标语言配音片段。这是质量最不稳定的节点，也是成本最高的节点，因此必须支持单句重生成和 provider 切换。

**输入**

- Segment（含 target_text、speaker_id、voice_profile_id、duration_ms）
- VoiceProfile（含 provider、voice_id 或参考音频、语速参数、情绪参数）

**输出**

- 每条 Segment 对应的音频片段（wav / mp3）
- 生成音频的实际时长 actual_duration_ms
- generation_status（success / failed / timeout）
- provider 调用记录（耗时、成本、参数）

**可能遇到的问题**

| 问题 | 原因 | 处理建议 |
| --- | --- | --- |
| 生成音频超出时间窗 | 目标语言比原语言长 | 进入 Node 6 时长对齐处理 |
| 生成音频远短于时间窗 | 文本过短或语速过慢 | 标记 `too_short`，可加静默或调语速 |
| 错读专有名词 | 人名、地名、招式名在 TTS 中读音错误 | 支持音素标注（phoneme hint）注入 |
| 情绪平淡 | TTS 默认无情绪 | 支持情绪参数或 SSML 标注注入 |
| provider 调用失败 | 网络、配额、参数错误 | 自动重试 3 次，失败后标记，支持手动重试或切换 provider |
| 音色跨句不稳定 | 每次生成有随机性 | 固定 seed / voice_id，同一角色统一参数 |

**人工干预入口**

- 对单条 segment 重新生成（可修改 target_text、语速、情绪参数后重生成）
- 批量重生成选中 segment
- 切换 provider 重生成（对比结果后选择）
- 手动上传替换音频（用户自己录制或处理的音频）
- 试听生成结果（单条或连续多条）
- 添加音素标注修正读音
- 调整语速参数（0.8x ~ 1.3x）

**技术选型**

- 第三方 TTS（推荐 MVP）：ElevenLabs API
- 第三方备选：OpenAI TTS / Azure TTS / Google TTS
- 开源音色克隆：`CosyVoice` / `OpenVoice`
- 开源少样本微调：`GPT-SoVITS`
- SSML / 音素支持：Azure TTS SSML / ElevenLabs pronunciation

详细对比和选型依据见 [`AI配音替换技术坐标与方案选型.md`](./AI配音替换技术坐标与方案选型.md#node-5-tts-generation-所需技术)。

### Node 6：Alignment — 时长对齐

**这个节点解决什么问题**

生成的配音片段和原 SRT 时间窗之间会存在时长差异。这个节点判断差异是否可接受，并选择对齐策略——轻微超时可拉伸，明显超时必须退回修文本或调整时间窗，而不是强行压缩到失真。

**输入**

- Segment（含 start_time_ms、end_time_ms、duration_ms）
- 生成音频片段（含 actual_duration_ms）

**输出**

- 对齐后的音频片段（处理后）
- duration_status：`ok` / `tight` / `overflow` / `severe_overflow`
- 对齐策略记录（time_stretch / silence_pad / manual_trim / flagged）

**对齐策略分级**

| 状态 | 判断条件 | 推荐策略 |
| --- | --- | --- |
| `ok` | 实际时长 ≤ 时间窗 * 1.05 | 直接使用，末尾补静默 |
| `tight` | 时间窗 * 1.05 < 实际时长 ≤ 时间窗 * 1.2 | time-stretch 轻压（不超过 1.2x）|
| `overflow` | 时间窗 * 1.2 < 实际时长 ≤ 时间窗 * 1.5 | 标记，建议缩短文本或拓展时间窗 |
| `severe_overflow` | 实际时长 > 时间窗 * 1.5 | 必须退回修文本，不建议强压 |

注：比例阈值可在系统配置中调整。

**可能遇到的问题**

| 问题 | 原因 | 处理建议 |
| --- | --- | --- |
| 大量 overflow | 目标语言整体比原语言长 | 全局语速参数调整，或整体放宽时间窗 |
| time-stretch 导致声音失真 | 压缩比过大 | 超过 1.2x 不建议拉伸，强制提示修文本 |
| 相邻两条 segment 时间窗紧挨 | 无法向后借时间 | 标记，提示合并相邻段或整体重排时间轴 |
| 原音乐节拍敏感位置 | 超时会打乱节拍感 | 支持手动指定「节拍锁定点」，不允许跨越 |

**人工干预入口**

- 查看所有 overflow / severe_overflow 列表
- 对单条 segment 手动调整时间窗（拖拽时间轴）
- 手动触发 time-stretch（指定目标时长）
- 退回 Node 3 修改 target_text（缩短文本后重生成）
- 批量放宽时间窗（全局 +N 毫秒容差）
- 手动上传对齐后的音频

**技术选型**

- Time-stretch：`librosa` / `ffmpeg atempo` / `Rubber Band Library`
- 静默填充：`ffmpeg`
- 时长测量：`ffprobe` / `librosa`

详细对比和选型依据见 [`AI配音替换技术坐标与方案选型.md`](./AI配音替换技术坐标与方案选型.md#node-6-alignment-所需技术)。

### Node 7：Audio Mix — 音轨混音

**这个节点解决什么问题**

把生成的对白音频片段按时间轴合并成目标音轨，同时保留原视频的背景音乐和音效，让成品听感自然。这个节点是「用什么方式处理原音轨」决策最复杂的地方。

**输入**

- 原视频音轨
- 所有 Segment 的对齐后音频片段 + 时间戳
- 混音策略配置

**输出**

- 目标语言音轨文件（wav）
- 混音日志（每条 segment 的混音位置、音量参数）

**混音策略选项（按保守到激进排列）**

| 策略 | 做法 | 优点 | 缺点 | 适用场景 |
| --- | --- | --- | --- | --- |
| 压低原音叠加 | 原音轨整体降低（-12dB 左右），对白时间段叠加目标配音 | 实现简单，背景音保留 | 原对白仍可听见 | MVP 第一版 |
| 人声分离后替换 | 先分离人声和背景音，背景音保留，对白轨替换为目标配音 | 原对白完全消失，背景音干净 | 人声分离有质量损失 | 主流推荐路线 |
| 全轨替换 | 直接用目标配音替换整条音轨 | 最简单 | 背景音全部丢失 | 不推荐，除非原视频无背景音 |

**可能遇到的问题**

| 问题 | 原因 | 处理建议 |
| --- | --- | --- |
| 人声分离后背景音有残留人声 | 分离模型不完美 | 可接受轻微残留；严重时对该段手动处理 |
| 对白音频和背景音音量不平衡 | TTS 输出音量和原视频音量基准不同 | 对目标配音做响度归一化（loudness normalization）|
| 背景音乐节拍在对白段被压低 | 压低策略影响节拍感 | 支持对特定时间段单独设置压低参数 |
| 多说话人同时发言（重叠对白） | 漫剧偶有重叠音效 | 标记为「重叠段」，当前版本建议人工处理 |

**人工干预入口**

- 选择混音策略（全局或按 segment）
- 调整对白音量和背景音音量比例
- 对单个时间段微调 ducking 参数
- 替换某段的对白音频后重新混音
- 局部预览混音效果（选择时间段）

**技术选型**

- 音轨合并 / 混音：`ffmpeg amix` / `pydub`
- 人声分离：`Demucs` / `UVR5` / `MDX-Net`
- 响度归一化：`pyloudnorm` / `ffmpeg loudnorm`

详细对比和选型依据见 [`AI配音替换技术坐标与方案选型.md`](./AI配音替换技术坐标与方案选型.md#node-7-audio-mix-所需技术)。

### Node 8：Subtitle Render — 字幕渲染

**这个节点解决什么问题**

把目标语言字幕嵌入视频。软字幕和硬字幕处理路径完全不同——软字幕直接替换字幕轨即可，硬字幕需要先遮盖原字幕区域再叠加新字幕。

**输入**

- 原视频（或已合并目标音轨的视频）
- 目标语言 Segment 列表（含 target_text、时间戳）
- 字幕类型（软字幕 / 硬字幕，来自 Node 1 判断）
- 字幕样式配置（字体、大小、颜色、位置）

**输出**

- 含目标语言软字幕的视频（或字幕轨文件）
- 或含硬字幕遮盖 + 新字幕的视频
- 目标语言 srt 文件

**软字幕和硬字幕处理路径**

```text
软字幕路径：
原 srt 目标语言版本
  -> ffmpeg 替换字幕轨
  -> 输出含目标字幕的 mp4

硬字幕路径：
Node 1 检测到的硬字幕区域
  -> 生成遮罩图层（纯色块或模糊）
  -> 叠加到视频对应时间段
  -> 在遮罩区域上方渲染新字幕文本
  -> 烧录进视频
```

**可能遇到的问题**

| 问题 | 原因 | 处理建议 |
| --- | --- | --- |
| 硬字幕遮盖区域不准 | Node 1 检测有偏差 | 提供手动调整遮罩区域入口 |
| 遮罩颜色和画面背景不融合 | 字幕背景色多变 | 支持选择遮罩类型：纯色 / 模糊 / 透明叠字 |
| 目标语言字幕超出显示宽度 | 长文本换行逻辑不同 | 自动换行 + 字号动态调整；或提示手动缩短 |
| 字幕时间和音频不完全对齐 | 时长对齐后时间轴有微调 | 字幕时间戳以最终 Segment 时间为准，不用原 srt 时间 |
| 软字幕轨格式不兼容播放器 | mp4 软字幕格式多样 | 默认输出 SRT 外挂 + 嵌入 mov_text，同时输出独立 srt |

**人工干预入口**

- 调整遮罩区域（位置、大小）
- 修改字幕样式（字体、颜色、位置、大小）
- 手动编辑单条字幕文本（不触发重新翻译）
- 调整字幕时间戳（微调显示时机）
- 选择输出格式（软字幕 / 硬字幕 / 两者都要）

**技术选型**

- 软字幕嵌入：`ffmpeg`
- 硬字幕烧录：`ffmpeg subtitles` filter
- 字幕遮罩：`ffmpeg drawbox` + `overlay`
- 字幕格式转换：`pysrt` / `ass`
- 高质量字幕样式：ASS 格式 + `libass`

详细对比和选型依据见 [`AI配音替换技术坐标与方案选型.md`](./AI配音替换技术坐标与方案选型.md#node-8-subtitle-render-所需技术)。

### Node 9：Export — 审核与导出

**这个节点解决什么问题**

在最终导出前，提供整体质量确认入口，确保没有漏处理的 segment、未确认的 overflow 和未审核的音频。导出后生成可追踪版本，避免覆盖已审核成果。

**输入**

- 所有 Segment 的最终状态（review_status）
- 目标音轨文件
- 字幕渲染后的视频（或待合并状态）

**输出**

- 目标语言 mp4（含目标音轨 + 字幕）
- 目标语言 srt 文件
- 可选：目标音轨 wav 独立文件
- 处理报告（segment 数量、超时段统计、provider 使用记录、耗时、成本估算）

**导出前检查项**

| 检查项 | 触发条件 | 处理方式 |
| --- | --- | --- |
| 有 segment 生成失败 | generation_status = failed | 提示列表，要求重生成或手动上传 |
| 有 segment 未审核 | review_status = pending | 提示数量，支持批量确认或逐条审核 |
| 有 severe_overflow 未处理 | duration_status = severe_overflow | 强警告，建议处理后再导出（可强制跳过）|
| 有 segment 未绑定音色 | voice_profile_id 为空 | 必须处理，无法跳过 |

**人工干预入口**

- 全局预览（全片播放）
- 按状态过滤查看 segment（可只看 overflow / failed / pending）
- 对任意 segment 做最后修正后局部重渲染
- 选择导出内容（视频 / srt / 音轨 / 全部）
- 选择字幕嵌入方式（软 / 硬 / 外挂）
- 导出为版本快照（版本号 + 时间戳，不覆盖历史版本）

**技术选型**

- 视频合成导出：`ffmpeg`
- 版本管理：对象存储（S3 / MinIO）+ 数据库版本记录
- 处理报告生成：后端汇总 + JSON/PDF 输出

详细对比和选型依据见 [`AI配音替换技术坐标与方案选型.md`](./AI配音替换技术坐标与方案选型.md#node-9-export-所需技术)。

## 全流程总览

```text
输入：mp4 + srt（原语言 或 目标语言）
          │
          ▼
Node 1：Ingest
  解析媒体元信息、校验 srt、判断字幕类型
  输出：媒体元信息、Segment 列表、校验报告
          │
          ▼
Node 2：Segmentation
  建立最小配音单元，支持合并 / 拆分
  输出：结构化 Segment 列表（含时间窗、文本、状态）
          │
          ▼ （若 is_translated = true，跳过自动翻译，直接进入人工校对）
Node 3：Localization
  文本本地化，支持多语言、角色口吻、术语表
  输出：每条 Segment 的 target_text + 长度预警
          │
          ▼
Node 4：Speaker Binding
  角色识别（自动 或 人工）+ 音色绑定（克隆 或 选库）
  输出：Speaker 列表、VoiceProfile 列表、Segment ↔ Speaker 映射
          │
          ▼
Node 5：TTS Generation
  按 Segment + VoiceProfile 生成配音片段
  输出：每条 Segment 的音频文件 + 实际时长
          │
          ▼
Node 6：Alignment
  检测时长差异，分级处理（time-stretch / 标记 / 退回修文本）
  输出：对齐后音频 + duration_status
          │
          ▼
Node 7：Audio Mix
  人声分离（可选）+ 对白替换 + 背景音保留 + 响度归一化
  输出：目标语言音轨
          │
          ▼
Node 8：Subtitle Render
  软字幕替换 或 硬字幕遮盖 + 新字幕渲染
  输出：含字幕的视频 + 目标语言 srt
          │
          ▼
Node 9：Export
  全局质量检查 + 人工确认 + 版本导出
  输出：目标语言 mp4 + srt + 处理报告
```

## 人工干预原则

每个节点的人工干预不是 yes/no 按钮，而是真正能修改输入、调整参数、重新执行的编辑能力。

| 原则 | 说明 |
| --- | --- |
| 锁定机制 | 人工确认的 segment 不被后续批量操作覆盖 |
| 局部重跑 | 修改单条 segment 后只重跑该条，不强制整片重跑 |
| 状态可见 | 每条 segment 的每个节点状态随时可查 |
| 回退能力 | 任意节点的结果都可以退回上一节点重新处理 |
| 版本保护 | 每次导出生成独立版本，历史版本不覆盖 |

## 适用边界

**适合使用本方法论的场景**

- 输入是 mp4 + srt，目标是输出目标语言 mp4 + srt
- 需要多语言输出，且质量要可人工干预
- 需要对多角色配音保持音色稳定
- 是持续生产场景（多集、多语种）而不是一次性处理

**不适合的场景**

- 真人强口型同步（lip-sync）需求——本方法不覆盖口型驱动，漫剧宽松，真人短剧需要单独引入 lip-sync 能力
- 只需要字幕翻译而不替换配音——直接用翻译工具，不需要本流水线
- 完全黑盒全自动（无审核需求）——本方法论假设质量问题必须可定位和修正
- 原视频无 srt、需要从零生成字幕——可用 WhisperX 生成初始 srt 后再进入本流水线，但属于前置步骤
