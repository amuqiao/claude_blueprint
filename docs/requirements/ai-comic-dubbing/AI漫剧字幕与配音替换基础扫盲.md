# AI 漫剧字幕与配音替换基础扫盲

本文用于把“用 SRT 和 MP4 做 AI 漫剧字幕替换、配音替换、音色相似”这个需求对象化，先建立可讨论的最小内部模型，再进入方案选型。

本文使用 [`主题基础扫盲方法论`](../../方法论/理解类/主题基础扫盲方法论.md) 的思路：先定住对象，再解释它为什么存在、靠什么成立、边界在哪里。

## 一句话理解

这个需求不是“翻译一份字幕文件”，而是把原视频里的对白生产链重新走一遍：以 `srt` 为对白时间轴，把文本本地化、生成目标语言配音、替换字幕显示，并让声音尽量像原角色。

## 当前需求复述

你现在有两类输入：

- 一段处理过的其他语种 `srt`，可能是英文、法文、日文、韩文、印尼文等。
- 一段 `mp4` 素材，素材本身带字幕和配音。

你希望最终做到：

- 替换视频字幕。
- 替换视频配音。
- 新配音的音色尽量接近原角色。
- 以 `srt` 中的对白段落作为基本处理单元。
- 能处理不同语种导致的分段、时长、语序和表达长度差异。
- 优先考虑第三方实现、开源方案和自研方案之间的组合，而不是一开始只押单一路线。

这个需求所在的业务背景是 AI 漫剧平台。平台不仅要生成结果，还需要服务持续生产，所以后续要考虑任务编排、人工审核、单句重生成、角色音色管理和批量导出。

## 这个问题的真实对象

表面对象是 `srt + mp4`，真实对象是“对白驱动的视频本地化流水线”。

```text
mp4 原素材
+ srt 对白时间轴
-> 文本本地化
-> 字幕重排
-> 角色音色匹配
-> 目标语言配音
-> 音频时长对齐
-> 视频混音和字幕输出
-> 人工质检
```

这里的核心不是某一个模型，而是多个能力之间的协作关系。

| 对象 | 在需求中的作用 |
| --- | --- |
| `srt` | 对白切片、时间窗和字幕文本的主索引 |
| `mp4` | 画面、原音轨、背景音、原字幕和成片容器 |
| 对白段 | 最小生产单元，承载文本、时间、角色和配音 |
| 翻译/本地化 | 把原对白变成目标语言，但要兼顾角色语气和时长 |
| TTS/音色克隆 | 生成目标语言语音，并尽量保留原角色声音特征 |
| 音频对齐 | 让生成语音尽量落入原对白时间窗 |
| 字幕渲染 | 输出软字幕或烧录字幕，必要时遮盖原硬字幕 |
| 人工审核 | 修正机器无法稳定处理的分段、语气、节奏和错译 |

## 为什么不能只做“翻译 SRT + TTS”

`srt` 是文本时间轴，不是完整配音剧本。

如果只把每条字幕翻译后直接送进 TTS，会遇到几个稳定问题：

| 问题 | 原因 |
| --- | --- |
| 目标语音超时 | 不同语言表达长度不同，同一含义在法语、日语、英语里的音节和语速差异很大 |
| 分段不自然 | 原 `srt` 分段服务字幕阅读，不一定服务配音停顿 |
| 语气丢失 | 字幕文本通常不包含情绪、停顿、重音和角色状态 |
| 角色音色漂移 | 多角色对白需要稳定 speaker identity，不能每句随机克隆 |
| 原背景音损坏 | 直接替换整条音轨会丢失音乐、环境音和音效 |
| 硬字幕残留 | 如果原字幕已经烧录进画面，单纯添加新字幕会叠字 |

所以本需求应被理解为“对白生产工程”，不是单点模型调用。

## 内部结构

一个可用的 AI 漫剧配音替换系统至少包含 7 层。

| 层级 | 责任 | 典型能力 |
| --- | --- | --- |
| 输入解析层 | 读取视频、字幕和音轨 | `ffmpeg`、SRT parser、媒体元信息提取 |
| 对白建模层 | 把字幕段变成可处理任务 | segment、speaker、time window、text version |
| 文本本地化层 | 翻译并改写对白 | 机器翻译、LLM 改写、术语表、角色口吻 |
| 音色与角色层 | 维护角色和声音关系 | speaker diarization、voice clone、voice library |
| 语音生成层 | 生成目标语言对白音频 | TTS、voice cloning、prosody control |
| 对齐与混音层 | 让音频、字幕、画面对齐 | time-stretch、silence trim、ducking、mixing |
| 审核导出层 | 人工修正并输出成片 | 局部预览、单句重生成、字幕导出、视频渲染 |

这 7 层决定了平台的工程形态：后端不是简单接一个 TTS 接口，而是要调度一组可替换的媒体处理和 AI 任务。

## SRT 作为基本处理单元的含义

以 `srt` 为基本处理单元，意味着系统的主数据结构应该围绕字幕段建立，而不是围绕整段音频建立。

一个字幕段至少需要抽象成：

| 字段 | 含义 |
| --- | --- |
| `index` | 原 SRT 序号 |
| `start_time` / `end_time` | 原对白时间窗 |
| `source_text` | 原对白文本 |
| `target_text` | 目标语言文本 |
| `speaker_id` | 角色或说话人 |
| `voice_profile_id` | 目标音色配置 |
| `generated_audio` | 当前生成的配音片段 |
| `duration_status` | 是否超时、过短或可接受 |
| `review_status` | 是否已人工确认 |

后续的合并、拆分、重生成、审核和导出，都围绕这个结构进行。

## 关键边界

### 软字幕和硬字幕不是一回事

如果 `mp4` 里是软字幕轨，可以直接替换或新增字幕轨。
如果字幕已经烧录在画面里，系统必须选择遮盖、重排、裁切或视频修复。第一版更适合用字幕区域遮罩和重新排版，不建议把视频 inpainting 作为 MVP 必做能力。

### 音色相似不等于完全复刻

平台目标应定义为“角色声音风格接近且稳定”，不是保证声纹完全一致。真实声纹复刻涉及授权、合规和伦理风险，也会受原素材干净程度、背景音、人声分离质量影响。

### 配音替换不等于口型同步

漫剧通常比真人口型同步宽松。第一阶段应优先保证字幕、语义、角色音色和节奏可接受；如果后续进入真人短剧或强口型画面，再单独考虑 lip-sync。

### 全自动不是第一阶段最优目标

配音质量问题通常集中在语气、停顿、错译、名字读音、分段和时长。AI 可以生成初稿，但人工审核工作台应从第一版就进入设计。

## 最小心智模型

可以把这个需求理解成一个“带人工审核的对白流水线”：

```text
SRT 提供时间窗
LLM/翻译提供目标文本
TTS/音色克隆提供声音
音频处理负责对齐和混音
视频处理负责字幕和导出
人工审核负责质量兜底
```

只要这个模型稳定，后续无论接第三方 API、开源模型，还是自研训练，都可以放进同一套工程框架里讨论。
