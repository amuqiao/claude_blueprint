# AI 漫剧平台 MVP 落地方案

本文把 AI 漫剧字幕与配音替换需求收敛为第一版可执行的工程方案。目标不是一次性做完整自动化平台，而是先实现可验证、可审核、可替换 provider 的 MVP。

## 方案结论

第一版建议建设 **SRT 驱动的人审半自动配音替换流水线**。

```text
上传 mp4 + srt
-> 解析媒体和字幕
-> 按 SRT 建立对白段
-> 翻译/本地化目标文本
-> 绑定角色和音色
-> 生成目标语言配音
-> 对齐每段时长
-> 混音替换原配音
-> 替换或覆盖字幕
-> 人工审核
-> 导出成片
```

MVP 的重点是跑通生产闭环，而不是追求全自动完美成片。

## 产品形态

第一版应做成创作者或运营人员使用的任务工作台。

核心页面和能力：

| 模块 | 能力 |
| --- | --- |
| 任务上传 | 上传 `mp4`、上传或粘贴 `srt`、选择源语言和目标语言 |
| 对白列表 | 按 SRT 段展示时间、原文、目标文本、角色、状态 |
| 文本编辑 | 修改翻译、合并/拆分长对白、锁定不再自动改写的文本 |
| 角色音色 | 为说话人绑定 voice profile，试听和替换音色 |
| 单句生成 | 对单条对白生成、重生成、试听和接受结果 |
| 局部预览 | 预览某一段或某一小段连续剧情 |
| 质量状态 | 标识超时、过短、未生成、待审核、已确认、失败 |
| 导出 | 输出目标语言视频、字幕文件和音频轨 |

第一版不建议只做一个“上传后等待成片”的黑盒按钮，因为质量问题出现后用户必须能定位和修正。

## 后端架构

你有 FastAPI 开发经验，后端可以按任务编排服务设计。

```text
FastAPI API 层
-> Job Service
-> Segment Service
-> Provider Adapter
   -> Translation Provider
   -> TTS / Dubbing Provider
   -> ASR / Alignment Provider
   -> Audio Processing Provider
   -> Video Rendering Provider
-> Worker Queue
-> Object Storage
-> Database
```

### 关键对象

| 对象 | 说明 |
| --- | --- |
| `Project` | 一个漫剧本地化项目，包含源视频、语言、输出配置 |
| `Job` | 一次处理任务，如翻译、配音、混音、导出 |
| `Segment` | 一个 SRT 对白段，是最小编辑和生成单元 |
| `Speaker` | 视频中的角色或说话人 |
| `VoiceProfile` | 第三方 voice id、开源参考音频或自定义音色配置 |
| `Asset` | 视频、字幕、音频片段、预览文件、导出文件 |
| `ReviewStatus` | 人工审核状态 |

### Segment 最小字段

```text
id
project_id
source_index
start_time_ms
end_time_ms
source_text
target_text
speaker_id
voice_profile_id
generated_audio_asset_id
duration_ms
duration_status
generation_status
review_status
```

先保持字段够用，不要一开始设计过度复杂的剧本系统。

## Provider 设计

MVP 需要把第三方和开源能力封装成统一 provider，避免业务流程被某一个厂商或模型绑定。

| Provider | 输入 | 输出 |
| --- | --- | --- |
| `SubtitleParser` | `srt` 文件 | `Segment[]` |
| `TranslationProvider` | `source_text`、源语言、目标语言、角色上下文 | `target_text` |
| `TTSProvider` | `target_text`、`voice_profile`、语言、时长目标 | 音频片段 |
| `AlignmentProvider` | 原音频、SRT、生成音频 | 对齐建议和时长状态 |
| `AudioMixProvider` | 原音轨、生成对白、时间轴 | 混音后的目标音轨 |
| `VideoRenderProvider` | 原视频、目标音轨、目标字幕 | 成片 |

第一版可以先实现第三方 `TTSProvider` 和本地 `ffmpeg` 渲染，开源模型作为并行实验 provider。

## 任务流程

### 1. 上传和解析

- 用户上传 `mp4` 和 `srt`。
- 后端保存原始文件，提取视频时长、音轨、分辨率等媒体信息。
- 解析 `srt` 生成 `Segment` 列表。
- 校验 SRT 时间是否越界、重叠、空文本或异常排序。

### 2. 文本本地化

- 对每条 `Segment` 生成目标语言文本。
- 对大段对白标记为“可能需要拆分”。
- 对目标文本过长的片段标记为“可能超时”。
- 保留人工编辑入口，人工修改后可锁定文本。

### 3. 角色和音色绑定

- MVP 可以先由人工为角色绑定音色。
- 如果原视频多说话人明显，再引入 WhisperX 或其他 diarization 能力辅助识别。
- 每个 `Speaker` 绑定一个 `VoiceProfile`，避免同角色多句声音漂移。

### 4. 配音生成

- 按 `Segment` 调用 TTS provider。
- 生成结果写入音频 asset。
- 记录耗时、失败原因、provider、voice profile 和参数。
- 支持单句重生成，不强制整片重跑。

### 5. 时长对齐

- 比较生成音频时长和原 SRT 时间窗。
- 轻微超时可做 time-stretch 或语速调整。
- 明显超时应提示改短文本、拆分片段或允许跨段处理。
- 不建议静默强行压缩到严重失真。

### 6. 混音和字幕输出

- 第一版优先保留原背景音和音效。
- 如果没有可靠人声分离，先采用压低原音轨并叠加目标配音的保守方案。
- 后续可加入人声分离能力，把原对白压低或移除，保留背景音乐和音效。
- 字幕优先输出目标语言软字幕；如果原视频硬字幕明显，第一版使用字幕区域遮罩和新字幕覆盖。

### 7. 审核和导出

- 人工逐段审核文本、音频、时长和字幕。
- 所有关键状态通过后允许导出。
- 导出目标视频、目标 `srt`、目标音轨和处理报告。

## MVP 范围

第一版包含：

- `mp4 + srt` 上传。
- SRT 解析和 segment 列表。
- 目标文本生成和人工编辑。
- 人工角色/音色绑定。
- 单句和批量配音生成。
- 时长状态检测。
- 局部预览。
- 字幕覆盖或软字幕输出。
- 视频导出。
- 失败重试和任务状态。

第一版不包含：

- 完整自研 TTS 模型训练。
- 强口型同步。
- 高质量硬字幕擦除。
- 完全自动多角色识别。
- 全自动质量验收。
- 复杂多人重叠说话处理。

这些能力可以进入后续版本，但不应阻塞 MVP。

## 技术实施建议

### 后端

- 使用 FastAPI 提供项目、任务、segment、asset 和导出接口。
- 使用异步 worker 处理翻译、TTS、混音和渲染等长任务。
- 使用对象存储保存原视频、音频片段、预览文件和成片。
- 数据库保存项目状态、segment 状态、provider 调用记录和审核状态。
- 所有 provider 调用都记录输入摘要、输出 asset、耗时、错误和成本。

### 媒体处理

- 使用 `ffmpeg` 做音轨提取、音频拼接、混音、字幕渲染和视频导出。
- SRT 解析使用成熟 parser，不建议手写脆弱字符串解析。
- 音频片段统一转成后续混音需要的采样率和声道。
- 每次导出生成可追踪版本，避免覆盖用户已审核结果。

### AI 能力

- MVP 可优先接第三方 TTS / dubbing API。
- 开源实验线单独跑，不阻塞主流程。
- 翻译和本地化可用 LLM，但必须允许人工编辑和锁定。
- 音色克隆需保留授权字段和来源记录。

## 测试和验收

准备 3 到 5 条短漫剧片段，每条 30 秒到 2 分钟。

覆盖场景：

- 单人对白。
- 多人对话。
- 大段对白。
- 快速短句来回。
- 带背景音乐和音效。
- 英语、日语、韩语、法语、印尼语至少覆盖其中几类。

验收口径：

- 能从 `srt` 正确生成 segment。
- 能生成目标语言文本，并允许人工修改。
- 能按角色绑定音色并生成配音。
- 能标记生成音频超出时间窗的片段。
- 能单句重生成，不需要整片重跑。
- 能导出目标视频和目标字幕。
- 失败任务有明确状态和错误原因。
- 人工审核通过的片段不会被后续自动任务静默覆盖。

## 演进路径

| 阶段 | 目标 |
| --- | --- |
| MVP | 端到端跑通，人工可修，质量可验收 |
| V1 | 增加开源 TTS provider 对比、成本统计、批量任务 |
| V2 | 引入人声分离、自动说话人识别、字幕遮罩模板 |
| V3 | 沉淀角色音色库、质量评分、半自动分段优化 |
| V4 | 基于真实数据做模型微调或训练 |

后续是否自研模型，应由平台数据决定：当第三方或开源 provider 在某些语言、角色类型、成本或稳定性上形成明确瓶颈，再针对瓶颈做训练或微调。

## 当前默认决策

- 文档和产品方案默认服务 AI 漫剧生产，不优先服务真人强口型视频。
- 技术路线默认混合路线。
- 工作流默认人审半自动。
- 数据结构默认以 `srt` segment 为核心。
- 后端默认 FastAPI + worker queue + provider adapter。
- MVP 默认先接第三方能力，开源模型并行评测，自研模型后置。
