# Chapters 短文本语种识别方案输出的需求拆解与补充说明

本文用于对 [方案分析与建议](./方案分析与建议.md) 的输出做二次拆解，把推荐方案中的前提、边界、状态和待确认问题整理成可进入研发沟通与验收确认的需求补充。

## 输入对象

本文不直接分析原始口头需求，而是分析方案文档已经形成的推荐输出。

方案文档的核心结论是：

```text
优先推荐后端统一检测编排。

有效对白字符数 < 第三方检测门槛
-> 调用 AI 语种识别
-> AI 返回正常且满足可信口径
-> 返回统一正常状态，前端展示绿字

AI 失败 / 超时 / 无结果 / 低置信度 / 检测异常
-> 返回非正常状态
-> 前端不得展示为绿色正常状态

有效对白字符数 >= 第三方检测门槛
-> 继续走既有第三方检测逻辑
```

因此，本文的任务不是重新比较方案，而是把“后端统一检测编排”这个方案转成更明确的需求口径。

## 当前理解

推荐方案已经把实现路径收敛到后端统一编排，但仍有一些需求层面的关键点不能直接跳过。

当前可以理解为：

- 系统需要在后端统一判断对白应走第三方检测还是 AI 检测。
- 短对白 AI 检测结果要被收敛成现有前端可消费的统一检测状态。
- 绿色正常状态只能来自明确的正常检测结果，不能来自未检测、失败或未知。
- 八字符及以上对白继续沿用既有第三方检测逻辑，本次不改变其业务语义。
- 方案是否覆盖详情、导出、统计、历史补检和检测来源展示，仍需要在需求层面确认。

这里需要拆解的重点，不是“要不要做 AI 检测”，而是“推荐方案成立后，系统应如何定义对象、状态、边界和验收口径”。

## 需求对象

| 层级 | 内容 |
| --- | --- |
| 方案对象 | 后端统一检测编排 |
| 需求对象 | 短对白检测状态补齐与统一状态返回 |
| 直接改变 | 低于第三方检测门槛的有效对白检测来源与结果状态 |
| 间接影响 | 前端绿字展示、详情页、导出、统计、异常筛选、历史补检 |
| 不应默认纳入 | 全量检测平台重构、第三方检测替换、翻译质量检测规则重写 |

本需求应把“AI 检测”理解为短对白检测来源补充，而不是新的展示捷径。

## 从方案输出拆出的隐含假设

| 隐含假设 | 风险 | 需要补充确认 |
| --- | --- | --- |
| 后端可以拿到用于判断门槛的对白原文 | 如果后端拿到的是处理后文本，字符数判断可能与页面展示不一致 | 明确字符数判断基于哪个字段 |
| 字符门槛可以稳定表达为 `< 8` | 如果第三方规则实际是“少于或等于八字符”或按字节计算，分流会错误 | 明确第三方限制的精确口径 |
| 有效对白可以被规则识别 | 空白、标点、变量、标签等内容若进入 AI 检测，可能产生误判和浪费 | 明确有效对白过滤规则 |
| AI 检测结果可以映射到统一状态 | 如果 AI 返回的是概率、解释或多候选语种，需要定义映射规则 | 明确正常、异常、低置信度、无结果的状态映射 |
| 现有绿色状态可复用 | 如果绿字当前隐含“第三方检测通过”，直接复用会造成来源误解 | 明确是否展示检测来源 |
| 同步后端编排时延可接受 | 如果 AI 响应慢，用户会感知检测卡顿 | 明确超时时间、是否允许异步、是否允许重试 |
| AI 检测结果需要进入统一结果体系 | 如果只服务列表展示，后端统一编排可能超出当前范围 | 明确是否覆盖详情、导出、统计和历史记录 |

## 场景样例

| 场景 | 输入样例 | 期望需求口径 |
| --- | --- | --- |
| 短中文对白 | `是`、`好的` | 属于有效对白时走 AI 检测，正常后展示绿字 |
| 短英文对白 | `OK`、`Hi` | 属于有效对白时走 AI 检测，结果按目标语种规则判断 |
| 七字符对白 | 任意七字符有效对白 | 低于八字符门槛时走 AI 检测 |
| 八字符对白 | 任意八字符有效对白 | 默认继续走第三方检测，除非业务确认八字符也受限 |
| 空白内容 | 空字符串、仅空格 | 应跳过检测或标识不适用，不应展示为正常 |
| 纯符号内容 | `...`、`?!` | 需要确认是否属于有效对白，默认不宜直接走语种识别 |
| 占位符内容 | `{name}`、`%s` | 需要确认是否跳过检测，避免 AI 误判为自然语言 |
| AI 正常 | 返回语种匹配且满足置信度 | 映射为统一正常状态，前端可绿字展示 |
| AI 异常 | 语种不匹配、置信度不足 | 映射为异常或非正常状态，不绿字 |
| AI 无结果 | 超时、服务失败、空返回 | 映射为失败、待重试或未检测，不绿字 |

## 状态语义补充

推荐方案中最需要固定的是状态语义。后端统一编排只有在状态可解释时才成立。

| 后端状态 | 业务含义 | 前端表现要求 |
| --- | --- | --- |
| 正常 | 已完成检测，结果符合业务判定标准 | 可以展示绿色正常状态 |
| 异常 | 已完成检测，结果不符合业务判定标准 | 展示异常状态，不能绿字 |
| 检测失败 | 检测过程失败，没有可信结果 | 展示失败或可重试状态，不能绿字 |
| 检测超时 | 检测未在规定时间内完成 | 展示超时、待重试或未检测状态，不能绿字 |
| 低置信度 | AI 返回结果但可信度不足 | 展示待确认或异常状态，不能绿字 |
| 不适用 | 内容不属于有效对白或无需语种识别 | 展示跳过或不适用状态，不能混同为正常 |
| 第三方检测结果 | 八字符及以上对白沿用既有结果 | 保持现有展示语义 |

绿字展示的需求口径应收敛为：只有统一状态为“正常”时才绿字，不根据检测来源或字符长度直接绿字。

## 需求补充说明

基于推荐方案，当前需求可以补充为：

- 后端检测编排需要增加短对白分流逻辑，对低于第三方检测门槛的有效对白调用 AI 语种识别。
- 分流逻辑不能影响八字符及以上对白的既有第三方检测流程。
- AI 检测结果必须映射为统一检测状态，前端不应自行根据字符长度判断绿字。
- AI 检测正常时，统一状态返回正常，前端按现有绿色正常状态展示。
- AI 检测失败、超时、无结果、低置信度或检测异常时，统一状态不得返回正常。
- 非有效对白应按确认后的规则跳过、标识不适用或进入单独状态，不能默认正常。
- 如果短对白结果会进入详情、导出、统计或异常筛选，这些入口必须使用同一套统一状态。

## 澄清问题

- 第三方八字符限制的精确规则是什么：`< 8`、`<= 8`、按字符数、按字节数，还是按第三方内部 token 计算。
- 后端用于分流的文本字段是哪一个：原始对白、去空格后对白、去标签后对白，还是展示文本。
- 有效对白过滤规则是什么：空白、纯标点、纯数字、表情、音效词、变量、占位符、富文本标签是否进入 AI 检测。
- AI 结果正常的判定标准是什么：只要求语种识别匹配，还是需要同时检查翻译内容、占位符、格式符和漏译。
- AI 是否必须返回置信度，低于多少应进入低置信度或异常状态。
- AI 调用超时时间是多少，超时后是否自动重试，重试期间前端展示什么状态。
- 是否需要在列表、详情、导出或日志中标识检测来源为 AI。
- 短对白 AI 检测结果是否进入统计通过率、异常汇总和导出结果。
- 历史存量短对白是否需要补检；如果需要，补检是否覆盖原有状态。
- 如果 AI 服务不可用，短对白最终展示为未检测、检测失败、待重试，还是不适用。

## 可验收口径

在上述问题确认后，验收可以按以下口径收敛：

- 低于门槛的有效对白会进入 AI 检测分支。
- AI 检测正常时，后端返回统一正常状态，前端展示绿字。
- AI 检测失败、超时、无结果、低置信度或异常时，前端不展示绿字。
- 八字符及以上对白仍走既有第三方检测链路。
- 非有效对白不会被误判为正常。
- 如果确认覆盖详情、导出和统计，这些入口与列表展示状态一致。
