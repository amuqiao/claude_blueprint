# 工作流参考

> 本文档把 `PLAYBOOK.md` 的方法层，和 `commands / agents / skills / templates` 的执行层串起来，回答“这一轮开发应该怎么走”。

---

## 总主线

标准项目生命周期建议按这条主线执行：

1. 项目定盘：`/init-architecture`
2. 模块设计：`/new-module`
3. 实现落地：主对话实现 + 按需 `@arch` / `@rev`
4. 回写与抽象：`/update-docs` + `/add-rule`

---

## 场景 1：新项目启动

### 目标

让项目进入“可规范化开发”的起点状态。

### 先做什么

- 明确项目目标与范围
- 收敛技术栈与关键选型
- 补齐项目级文档骨架

### 推荐入口

```text
/init-architecture
```

### 主要产物

- 项目根 `CLAUDE.md`
- `docs/design/INDEX.md`
- `docs/design/架构设计方案.md`
- `docs/实施清单.md`

### 此阶段不要做什么

- 不要一开始就铺满所有基础设施文档
- 不要一开始就创建所有模块文档

---

## 场景 2：老项目首次纳入规范

### 目标

给老项目补最小骨架，让后续迭代能进入同一套流程。

### 推荐入口

```text
/init-architecture
```

### 原则

- 不要求一次性补齐所有历史文档
- 只补最小骨架
- 后续在真实开发中逐步补模块文档和基础设施文档

---

## 场景 3：新增功能模块

### 目标

先把模块讲清楚，再进入实现。

### 推荐入口

```text
/new-module 模块名称
```

### 标准顺序

1. 读取需求输入
2. 用 `@arch` 判断是否影响全局架构
3. 明确页面 / 交互轮廓
4. 写模块设计文档
5. 更新 `docs/design/INDEX.md`
6. 更新 `docs/实施清单.md`

### 关键判断

- 影响全局分层或核心数据流 → 回写 `docs/design/架构设计方案.md`
- 只影响单模块 → 写入对应模块设计文档

---

## 场景 4：代码实现

### 目标

按已知边界完成代码实现。

### 推荐入口

- 主对话直接实现
- 需要判断时用 `@arch`
- 需要审查时用 `@rev`

### 判断规则

- 新模块 / 结构性改动 → 先文档，后代码
- 小改动 / 局部修复 / 纯 bugfix → 可先代码，后回写

---

## 场景 5：实现完成后回写

### 目标

同步文档状态，并把可复用经验抽象出来。

### 推荐入口

```text
/update-docs 模块名称
```

### 至少检查什么

- 设计文档是否和实现一致
- `docs/design/INDEX.md` 是否应更新状态
- `docs/实施清单.md` 是否应更新状态
- 是否出现了新的跨项目通用规则

如果出现新的跨项目规则，再执行：

```text
/add-rule 用自然语言描述这条约束
```

---

## 场景 6：不知道该用哪个能力

### 你已经明确阶段

用 `command`：

- 初始化 → `/init-architecture`
- 建模块 → `/new-module`
- 同步文档 → `/update-docs`
- 归档规则 → `/add-rule`

### 你需要一个判断

用 `agent`：

- 架构判断 → `@arch`
- 规范审查 → `@rev`

### 你要完成某类任务，但更在乎规范

依赖 `skill`：

- 设计文档
- 写作
- 运维脚本
- Python 脚本
