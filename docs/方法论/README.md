# 方法论文档

> 本目录维护可跨主题、跨项目复用的方法论；单篇方法论负责沉淀方法，本文负责维护方法论之间的选择关系和新增范式。

## 目录职责

本目录只放长期可复用的方法论文档。

适合进入本目录的内容，通常具备稳定的认知节点、操作节点或判断标准，能够帮助后续主题分析、文档写作或项目理解降低成本。

本目录不放具体主题说明、项目教程、临时笔记、工具使用记录或一次性任务复盘。

## 当前方法论

### 理解类

- [`理解类/主题基础扫盲方法论.md`](理解类/主题基础扫盲方法论.md)
  用于把模糊主题从名称、术语或碎片资料中定住，让它成为可以继续讨论和深入的稳定对象。

- [`理解类/主题认知坐标建模方法论.md`](理解类/主题认知坐标建模方法论.md)
  用于把已经成形的主题放入关系结构中，解释它的位置、比较对象、机制原因和取舍边界。

- [`理解类/工具能力使用与排错方法论.md`](理解类/工具能力使用与排错方法论.md)
  用于说明由多个能力、入口、配置和规则组成的工具体系，让读者能区分、会使用、能选择、可排错。

### 需求与方案类

- [`需求与方案类/需求拆解与补充说明方法论.md`](需求与方案类/需求拆解与补充说明方法论.md)
  用于把一句话或几段话的粗略需求，拆成可讨论、可确认、可验收的需求说明，但不进入解决方案设计。

- [`需求与方案类/方案分析与建议方法论.md`](需求与方案类/方案分析与建议方法论.md)
  用于在需求已经拆清楚后，生成候选方案、评估取舍，并给出有前提和风险说明的推荐建议。

## 如何选择

方法论之间不是固定流程，而是按问题触发。

```text
主题还没有成形
└── 使用：主题基础扫盲方法论

主题已经成形，但缺少关系、比较和取舍判断
└── 使用：主题认知坐标建模方法论

面对一组工具能力、配置入口、优先级规则和排错问题
└── 使用：工具能力使用与排错方法论

用户只给了一句话、几段话或标题式需求，需要拆清楚需求对象、问题、范围、期望和待确认点
└── 使用：需求拆解与补充说明方法论

需求已经清楚，但实现路径不唯一，需要生成候选方案、评估取舍并形成推荐建议
└── 使用：方案分析与建议方法论
```

同一个真实主题可以组合使用多篇方法论，但组合关系由当前问题决定，不写死在单篇方法论里。

## 新增范式

新增方法论前，先判断五件事：

```text
1. 是否有反复出现的问题类型
2. 是否能抽象出稳定节点
3. 是否能说明核心转化
4. 是否能给出完成判断
5. 是否能说明适用边界
```

方法论文档可以参考以下版式，但不把它当作强制模板：

```text
标题
一句话主旨
主干模型
核心命题
方法论节点
判断标准
误用
适用边界
```

这里的“主干模型”不是文档目录，也不是装饰图，而是方法论的结构压缩。它应展示这套方法由哪些核心节点组成，以及每个节点解决什么问题。

如果主干模型能降低理解成本，优先使用简单的文字树：

```text
方法论名称
├── Node A：节点名
│   └── 这个节点解决什么问题
├── Node B：节点名
│   └── 这个节点解决什么问题
└── Node C：节点名
    └── 这个节点解决什么问题
```

具体文档可以按主题需要增删章节，但应保持以下顺序：

```text
先说明它解决什么问题
再说明它如何工作
再说明如何判断是否完成
最后说明它不适合什么
```

不要为了满足版式而强行制造节点、图示或章节。如果一个方法只能写成步骤清单，尚未形成稳定抽象，不应进入本目录。

## 维护规则

- 单篇方法论应独立可读，只维护自身的核心命题、输入输出、节点、判断标准、误用和适用边界
- 单篇方法论不维护与其他方法论的交叉引用，方法论之间的选择、边界和组合关系统一由本文维护
- 新增方法论时，先判断它是否具备跨主题复用价值，再补充到“当前方法论”和“如何选择”
- 不为了目录完整而新增方法论；只有当某种思考方式反复出现，并且能抽象出稳定节点时，才进入本目录
- 草稿箱可以周期性整理，但整理目标是筛选、合并、归档和少量升格；不要把普通文档、提示词、一次性判断过程都包装成方法论
