# 方法论文档

> 本目录维护可跨主题、跨项目复用的方法论；单篇方法论负责沉淀方法，本文负责维护方法论之间的选择关系和治理入口。

## 目录职责

本目录只放长期可复用的方法论文档。

适合进入本目录的内容，通常具备稳定的认知节点、操作节点或判断标准，能够帮助后续主题分析、文档写作或项目理解降低成本。

本目录不放具体主题说明、项目教程、临时笔记、工具使用记录或一次性任务复盘。

## 如何选择

方法论之间不是固定流程，而是按问题触发。

```text
主题还没有成形
└── 使用：主题基础扫盲方法论

主题已经成形，但缺少关系、比较和取舍判断
└── 使用：主题认知坐标建模方法论

面对一组工具能力、配置入口、优先级规则和排错问题
└── 使用：工具能力使用与排错方法论

用户只给了一句话、几段话或标题式需求，需要拆清楚需求对象、问题、范围、期望和待确认点
└── 使用：需求拆解与补充说明方法论

需求已经清楚，但实现路径不唯一，需要生成候选方案、评估取舍并形成推荐建议
└── 使用：方案分析与建议方法论

项目开始持续开发，启动、验证、生成、清理、构建等动作开始重复，需要判断是否统一开发入口
└── 使用：项目开发入口设计方法论

开始一个 Flutter 应用或重构 Flutter 子工程，需要判断版本边界、架构、交互、界面、状态、原型、实现和验证主干
└── 使用：Flutter 项目开发主干方法论

有一个模糊能力种子，需要构想原生定义、能力范围、画像、场景、抽象升级和边界代价
└── 使用：超能力构想与抽象升级方法论

有反复出现的思考方式，想判断是否值得升格为方法论
└── 使用：方法论新增范式

已有方法论文档需要检查对象、命题、节点、表达和边界是否成立
└── 使用：方法论文档质检方法论
```

同一个真实主题可以组合使用多篇方法论，但组合关系由当前问题决定，不写死在单篇方法论里。

## 新增与质检

新增方法论前，应先使用 [`方法论治理类/方法论新增范式.md`](方法论治理类/方法论新增范式.md) 判断它是否具备长期复用价值。

新增方法论的基本判断包括：

```text
是否有反复出现的问题类型
是否能形成清楚的方法对象
是否能抽象出稳定节点
是否能说明完成判断
是否能切清适用边界
```

已有方法论如果出现对象漂移、命题不清、节点臃肿、表达流程化或边界膨胀，应使用 [`方法论治理类/方法论文档质检方法论.md`](方法论治理类/方法论文档质检方法论.md) 做质检，而不是继续在本文中堆规则。

## 维护规则

- 单篇方法论应独立可读，只维护自身的核心命题、输入输出、节点、判断标准、误用和适用边界
- 单篇方法论不维护与其他方法论的交叉引用，方法论之间的选择、边界和组合关系统一由本文维护
- 新增分类目录时，应同时补充该目录的 `README.md`，说明目录职责、当前文档和维护边界
- 新增方法论时，先判断它是否具备跨主题复用价值，再补充到“当前方法论”和“如何选择”
- 不为了目录完整而新增方法论；只有当某种思考方式反复出现，并且能抽象出稳定节点时，才进入本目录
- 草稿箱可以周期性整理，但整理目标是筛选、合并、归档和少量升格；不要把普通文档、提示词、一次性判断过程都包装成方法论
