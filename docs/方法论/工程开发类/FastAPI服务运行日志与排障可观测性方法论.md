# FastAPI 服务运行日志与排障可观测性方法论

> 指导轻量 FastAPI 服务建立统一日志入口、安全边界、请求追踪和业务摘要，让服务交付到容器或 K8s 后能被直接采集、快速排查，并避免把运行日志治理过早做成重型可观测平台。

```text
FastAPI 服务运行日志与排障可观测性
├── Sink：日志出口
│   └── 先固定日志写到 stdout/stderr，不写本地文件，让容器和 K8s 能直接采集
├── Core：日志核心
│   └── 建立轻量 logging 模块，统一级别、格式、logger 获取和上下文注入
├── Guard：安全边界
│   └── 明确哪些字段禁止入日志，尤其是密钥、原文、请求体和内部证据明细
├── Trace：请求追踪
│   └── 用中间件生成或透传 request_id，记录请求状态、耗时和异常
├── Signal：业务信号
│   └── 在业务入口记录聚合摘要，而不是记录完整请求体或逐条敏感内容
├── Verify：验证闭环
│   └── 用编译、接口、日志内容和回归脚本确认日志既有用又不泄漏
└── Scale：升级边界
    └── 只有采集、检索、指标、链路或告警需求明确时，才升级到 JSON、metrics、tracing 或日志平台适配
```

## 核心命题

FastAPI 服务运行日志与排障可观测性处理的不是“在代码里到处加 print”，也不是完整可观测平台建设，而是“一个轻量 API 服务如何形成可被容器平台采集、可按请求串联、可支撑业务排查、又不泄漏敏感数据的最小运行日志体系”。

小型 FastAPI 服务经常有两种误区：一种是只依赖 `uvicorn` 默认日志，服务能启动、能看到 access log，但线上排查时缺少请求 ID、耗时、业务原因分布和认证失败原因；另一种是一开始就引入复杂可观测平台，把日志、指标、链路、告警、日志文件和采集配置混成一个大工程。

这套方法论的任务，是把日志能力收敛成“轻量统一模块 + 安全边界 + 请求中间件 + 业务摘要 + 验证闭环”。它面向容器化和 K8s 的默认现实：应用应把日志写到 `stdout/stderr`，采集、存储、查询和告警由运行平台或运维系统承接。

它的核心转化是：

```text
零散输出 -> 统一日志出口 -> 日志核心模块 -> 安全边界 -> 请求追踪 -> 业务摘要 -> 验证闭环 -> 可选升级
```

## 方法论节点

这套方法论不是 FastAPI 教程，也不是 K8s 日志平台教程。它关注的是服务代码内部应该沉淀什么日志能力，以及哪些能力应交给运行平台。

### Sink：日志出口

先固定日志出口。

容器化服务的默认日志出口应是 `stdout/stderr`，而不是本地日志文件。这样 `kubectl logs`、容器运行时和集群日志采集组件都能按标准路径获取日志。对轻量 FastAPI 服务来说，写文件通常会引入额外目录挂载、权限、轮转、磁盘占用和 sidecar 配置，反而让排查变重。

`Sink` 要回答：日志输出到哪里，谁负责采集，是否需要挂载日志目录，是否能通过容器标准日志查看。

默认判断是：

```text
应用只负责写 stdout/stderr
运行平台负责采集、存储、检索和保留周期
项目不维护日志文件目录和日志轮转
```

只有在部署环境明确要求文件采集，或已有统一 sidecar 文件采集规范时，才考虑本地文件日志。否则文件日志不应成为 FastAPI 服务的默认设计。

### Core：日志核心

建立一个轻量日志核心模块。

核心模块的职责不是封装所有可观测能力，而是统一基础行为：日志级别、格式、输出 handler、logger 获取方式和请求上下文注入。它应优先使用 Python 标准 `logging`，避免为了第一阶段日志治理引入额外依赖。

轻量 FastAPI 服务通常可以沉淀一个统一日志入口，固定这些基础判断：

```text
日志配置在哪里初始化
各模块如何获取 logger
request_id 如何进入日志上下文
日志字段和格式如何保持一致
```

`Core` 的完成标准是：项目里不同文件不会各自随意配置 logging，不会同时出现 `print`、默认 logger、局部 handler 和重复格式；服务启动后，应用日志、请求日志、认证日志和异常日志都进入同一输出流和同一格式约定。

第一阶段可以使用文本键值格式，等日志平台解析需求明确后再升级 JSON。不要为了“生产化”一开始就把格式、采集、索引、trace、metrics 全部塞进核心模块。

### Guard：安全边界

明确禁止入日志的字段。

日志治理必须先定安全边界，否则“为了方便排查”会逐渐把请求体、密钥和内部细节都写进日志。对 FastAPI 服务，尤其是对外 API 或处理文本、文件、用户内容的服务，安全边界应成为日志模块的一部分，而不是事后提醒。

常见禁止项：

```text
API Key、JWT、Cookie、Authorization
完整请求体
用户原文 text、文件内容、生成内容
normalized_text 等可反推出原文的中间字段
完整内部 evidence 或模型明细
数据库连接串、云密钥、第三方 token
```

认证失败日志只记录事件和原因，例如 `missing_api_key`、`invalid_api_key`、`auth_not_configured`，不记录密钥原文。异常日志可以记录异常类型和堆栈，但要避免把敏感参数拼进异常消息。

`Guard` 的判断不是让日志变少，而是让日志保持可共享、可采集、可长期保存。

### Trace：请求追踪

用中间件承载跨接口请求日志。

FastAPI 中间件是包在每个 HTTP 请求外层的通用逻辑：请求进入具体路由前先经过中间件，路由返回响应后再经过中间件。因此它适合处理所有接口共同需要的日志行为。

请求日志中间件应负责：

```text
读取 X-Request-ID，没有则生成 request_id
把 request_id 放入请求上下文
响应头返回 X-Request-ID
记录 method、path、status_code、duration_ms
未捕获异常时记录 request_failed 和堆栈
```

中间件不应解析完整请求体，也不应承担业务日志判断。它只处理通用请求维度，让任何接口都有最小可追踪性。

`Trace` 的判断重点是：当线上出现一次失败、慢请求或调用方反馈时，能否用同一个 `request_id` 串起请求日志、认证日志、业务摘要和异常堆栈。

### Signal：业务信号

在业务入口记录聚合摘要。

请求日志只能说明“哪个接口、状态码、耗时”；业务排查还需要知道“这次业务处理了多少数据、结果大致如何、失败原因集中在哪里”。这些信息只有业务入口知道，不应该硬塞到中间件里。

同步 batch 或能力型 API 的业务摘要通常记录：

```text
batch_size
success_count / failed_count 或 match_count / mismatch_count
reason_counts
expected_type_counts 或 language_counts
duration_ms
```

不同接口形态的业务信号重点可以不同：

| 场景 | 业务摘要重点 |
| --- | --- |
| 同步 batch | 规模、成功失败计数、reason 分布和总耗时 |
| 单项能力接口 | 能力类型、结果类别、关键外部依赖耗时，不记录输入原文 |
| 异步 job 入口 | `task_id`、提交规模和状态变化，不把任务明细塞进请求日志 |
| 认证或限流 | 事件类型和失败原因，不记录凭证、token 或完整请求头 |

业务摘要应记录聚合信号，而不是逐条输入、完整请求体或完整内部证据。对于文本检测、翻译、生成、导入等服务，输入内容本身可能是业务敏感数据；日志一旦进入集群日志系统，访问面通常比接口本身更大。

`Signal` 的完成标准是：不看请求体也能知道一次业务调用的规模、结果分布和主要原因；但日志中没有足以还原用户输入或密钥的信息。

### Verify：验证闭环

验证日志模块不能只看代码能不能跑。

日志能力的验证应同时覆盖功能、排障价值和安全边界：

```text
静态编译或测试通过
服务启动后出现 app_start
健康检查返回 X-Request-ID，并出现 request_completed
业务接口透传 X-Request-ID，并出现业务摘要
认证失败出现 auth_failed，但不包含密钥
日志不包含请求原文、完整请求体或敏感字段
现有冒烟脚本或回归脚本仍通过
```

如果只验证接口返回，没有检查日志内容，说明日志方法没有闭环。如果日志有内容但泄漏了输入文本或 API Key，说明 `Guard` 没完成。

### Scale：升级边界

按需求升级，不把第一阶段做成平台。

第一阶段的目标是让服务在容器和 K8s 中可被采集、可按请求排查、可看业务摘要。后续升级应由明确需求触发。

常见升级信号：

| 升级项 | 触发条件 |
| --- | --- |
| JSON 日志 | 日志平台需要结构化字段检索和索引 |
| healthz 降噪 | K8s probe 高频刷屏，影响有效日志阅读 |
| metrics | 需要延迟分位、错误率、吞吐、reason 计数和告警 |
| tracing | 一次请求跨多个服务或外部模型调用，需要链路串联 |
| Sentry / 错误平台 | 需要异常聚合、告警、版本关联和负责人分派 |
| 审计日志 | 需要记录用户、资源、操作和合规留痕，且与运行日志分离 |

`Scale` 的核心是分层：运行日志、业务摘要、指标、链路、审计和告警是不同能力。可以逐步装配，不应在轻量服务第一阶段一次性混成“日志模块”。

## 判断标准

FastAPI 服务运行日志与排障可观测性是否完成，只看一个标准：

**服务部署到容器或 K8s 后，维护者能通过标准容器日志看到启动、请求、异常、认证失败和关键业务摘要，并能用 request_id 串联排查，同时日志中不包含密钥、完整请求体或业务敏感原文。**

如果只能看到 `uvicorn` access log，但看不到业务规模、结果分布或认证失败原因，说明日志能力不足。

如果业务日志记录了完整文本、请求体、API Key 或模型细节，说明安全边界失败。

如果服务需要挂载日志目录、手工找文件或额外解释日志在哪里，说明日志出口没有收敛到容器标准路径。

如果第一阶段就引入复杂采集、链路、指标和告警，而服务还没有稳定请求日志和业务摘要，说明升级边界过早打开。

## Few-shot Learning 示例组

### 正例：短文本检测 FastAPI 服务

一个同步 batch 文本检测服务需要交给运维部署到 K8s。合适做法是新增轻量日志模块，输出到 `stdout/stderr`；用中间件生成或透传 `X-Request-ID`；在 batch 接口记录 `batch_size`、`match_count`、`mismatch_count` 和 `reason_counts`；认证失败只记录失败原因。

判断：

```text
Sink：stdout/stderr，K8s 标准采集
Core：统一日志入口和上下文注入
Guard：不记 X-API-Key、text、请求体、完整 evidence
Trace：中间件记录 request_completed / request_failed
Signal：batch 聚合摘要，不记原文
Verify：healthz、batch、认证失败和 1000 条回归都验证
Scale：第一阶段不做 JSON、metrics、tracing
```

### 反例：只依赖 Uvicorn 默认日志

服务上线后只能看到 `GET /healthz 200` 和 `POST /batch 200`，但不知道一次 batch 处理了多少条、耗时多少、哪些原因导致 mismatch，也无法用调用方反馈的 ID 串联业务日志。

问题不在于 `uvicorn` 日志不可用，而是它只提供框架层 access log，不能替代服务自己的请求追踪和业务摘要。

### 反例：为了排查方便记录完整请求体

一个文本服务把每次请求的 `text`、`X-API-Key` 和完整 `evidence` 都写入日志。短期看排查方便，长期会让日志系统保存业务内容和密钥，扩大敏感数据暴露面。

正确做法是记录聚合原因分布和请求 ID；需要复现时通过受控样本、业务 ID 或专门调试流程获取输入，不把生产请求体默认写进日志。

### 边界例：何时升级 JSON 日志

如果日志只通过 `kubectl logs` 临时查看，文本键值格式已经足够。如果日志平台需要按 `event`、`request_id`、`status_code`、`reason_counts` 检索和聚合，就应升级为 JSON 日志或平台约定格式。

升级 JSON 不改变方法主线：仍然先保证出口、核心模块、安全边界、请求追踪和业务摘要成立。

## 误用

FastAPI 服务运行日志与排障可观测性不是日志平台建设方法。

不要在轻量服务第一阶段展开 ELK、Loki、Fluent Bit、OpenTelemetry、Prometheus、Sentry 的部署细节。那些属于运行平台、监控告警或链路追踪方法，只在明确触发时作为升级分支进入。

它也不是要求所有日志都进中间件。

中间件只处理请求级通用日志。业务摘要必须由业务入口记录，因为只有业务层知道数量、原因、结果分布和可安全公开的聚合字段。

它也不是鼓励多打日志。

有效日志不是越多越好，而是排查时能回答关键问题：服务是否启动、请求是否到达、认证是否通过、耗时在哪里、业务结果如何、异常在哪里、是否能按 request_id 串起来。

它也不是审计日志方法。

运行日志用于排障和运维观察；审计日志用于合规和操作追溯。两者字段、保留周期、权限和查询方式不同，不应混成一个日志模块。

## 适用边界

这套方法适合处理轻量到中型 FastAPI API 服务，尤其是：

```text
服务会部署到 Docker 或 K8s
希望 kubectl logs 或容器日志能直接排查
已有 LOG_LEVEL，但缺少统一日志模块
需要 request_id、耗时、状态码和异常堆栈
存在同步 batch、认证失败或业务 reason 分布
处理文本、文件或用户输入，需要日志安全边界
```

这套方法不适合处理纯 CLI 工具、没有 HTTP 请求生命周期的离线脚本、完整可观测平台建设、跨服务 tracing 体系设计、审计合规平台或日志采集组件部署教程。

如果问题是“FastAPI 服务接口和同步/异步任务如何设计”，应使用 FastAPI 独立服务接口与任务架构方法论。

如果问题是“服务如何本地启动、Dockerfile 交付、compose 配置和健康检查验证”，应使用项目部署入口与配置加载方法论。

如果问题是“部署路径稳定后如何写发布、维护和排障手册”，应使用项目部署文档与发布手册方法论。
