# 工程开发类方法论

> 本目录存放用于指导项目开发主干、工程推进顺序、模块落地和验证治理的方法论。

## 当前文档

- [`新服务架构龙骨与能力装配方法论.md`](新服务架构龙骨与能力装配方法论.md)
  用于从 0-1 构建新服务时，先建立服务龙骨，再按需求触发接口、任务、数据、缓存、存储、前端、安全、部署等能力积木，并判断它们的架构权重和装配顺序。

- [`全栈项目开发主干方法论.md`](全栈项目开发主干方法论.md)
  用于在轻量 API、独立后端能力服务、客户端应用和完整全栈平台之间判断项目类型、复杂度深度、必需主干节点和可补强的枝叶方法论。

- [`FastAPI独立服务接口与任务架构方法论.md`](FastAPI独立服务接口与任务架构方法论.md)
  用于指导小型 FastAPI 独立服务在同步请求、同步 batch、异步任务、任务队列和工作流升级之间建立稳定架构判断。

- [`Flutter项目开发主干方法论.md`](Flutter项目开发主干方法论.md)
  用于把 Flutter 应用从需求、概念图、旧代码或模糊想法，推进到可运行、可验证、可持续迭代的工程主干。

- [`项目部署入口与配置加载方法论.md`](项目部署入口与配置加载方法论.md)
  用于统一项目的本地开发、Dockerfile 独立、docker compose 全量和 docker compose 依赖部署，并切清服务边界、配置加载和验证入口。

- [`项目部署文档与发布手册方法论.md`](项目部署文档与发布手册方法论.md)
  用于把已经稳定的部署路径沉淀为可查、可复制、可维护的项目级部署文档和发布手册。

## 维护边界

工程开发类方法论负责回答“项目或模块应该按什么主干推进、服务如何从 0-1 起盘、部署形态如何选择、每个节点产出什么、做到什么程度进入下一步”。

其中，全栈项目开发主干方法论承担主干定义职责，只判断项目类型、复杂度深度、必需主干节点和枝叶补强关系，不维护目录索引，也不替代 FastAPI、Flutter、部署入口等具体子方法论。

新服务架构龙骨与能力装配方法论承担新服务起盘模具职责，重点判断服务龙骨、架构分层、能力积木、触发条件、架构权重和装配顺序；具体能力的落地细节仍回到对应枝叶方法论。

如果问题仍停留在需求拆解、方案比较或开发入口脚本设计，应回到上层方法论目录选择需求与方案类方法。

如果问题只是具体工具、命令、插件或报错使用说明，应使用理解类中的工具能力使用与排错方法论，不应沉淀到工程开发类。
