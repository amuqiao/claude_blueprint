# FastAPI 统一响应信封架构设计文档

```text
Version: 1.0.1
Status: Accepted / Development Baseline
Date: 2026-06-21
Change Policy: 进入开发后，任何 API 合同、顶层字段、HTTP 状态码策略、业务码分段、异常映射、包装范围、OpenAPI 展示结构、request_id/server_time 语义变更都必须升版本并记录变更原因。
Change Reason: 明确业务码分段为推荐治理示例，具体业务域和码段必须根据项目实际业务注册表落地。
```

本文设计一套基于 FastAPI + Pydantic V2 的统一响应信封体系，用于保证 REST API 在成功、失败、校验异常和系统异常场景下都返回一致、可追踪、可治理的 JSON 外层结构。

## 1. 文档职责

本文负责定义统一响应信封的目标、边界、响应结构、HTTP 状态码策略、业务码规范、请求链路、模块职责和验证标准。本文定稿后作为开发基线使用，开发实现必须以本文定义的外部 API 合同为准。

本文不负责具体业务领域错误码全集设计，不负责数据库事务、认证鉴权、日志平台、链路追踪平台的完整实现，也不覆盖文件下载、流式响应、WebSocket 等非标准 JSON API 场景。

## 2. 背景与目标

在企业级后端服务中，如果每个接口自行拼装响应结构，容易出现字段不一致、错误格式不统一、异常泄漏、前端适配成本升高、问题排查困难等问题。

统一响应信封的目标是把接口外层协议收敛到框架层，让业务接口只表达业务结果。

核心目标：

- 所有标准 JSON API 成功响应使用同一外层结构。
- 所有错误响应使用同一外层结构。
- 业务接口只返回原始业务数据，禁止手动拼接 `code`、`msg`、`data`、`request_id`。
- 系统异常、业务异常、参数校验异常统一转换为错误信封。
- 每个响应都携带 `request_id`，便于日志检索和链路追踪。
- 每个响应都携带 `server_time`，便于客户端校准展示时间和排查问题。
- HTTP 状态码描述请求层级结果，业务 `code` 描述业务层级结果。
- 成功响应使用 HTTP `200`，业务 `code` 固定为 `"0"`。

## 3. 总体原则

统一响应信封遵循以下原则：

```text
业务接口负责业务结果
框架层负责响应包装
异常处理层负责错误转换
工厂层负责对象创建
模型层负责结构约束
```

业务代码不直接感知响应信封结构。接口返回值进入成功包装层后，由成功包装层包装为成功信封；接口抛出的异常进入全局异常处理器后，由异常处理器包装为错误信封。

HTTP 状态码和业务 `code` 必须分层隔离：

```text
HTTP status: 描述请求在协议层是否成功，以及失败类型属于 4xx 还是 5xx
business code: 描述具体业务结果或业务错误原因
```

## 4. 响应信封结构

### 4.1 统一字段

成功响应和错误响应共享同一组顶层字段：

| 字段 | 类型 | 必填 | 说明 |
|---|---:|---:|---|
| `code` | `str` | 是 | 独立业务码，成功固定为 `"0"` |
| `msg` | `str` | 是 | 面向调用方的结果说明 |
| `data` | `T \| null` | 是 | 成功时为业务数据，失败时为错误详情或 `null` |
| `request_id` | `str` | 是 | 单次请求追踪 ID |
| `server_time` | `str` | 是 | 服务端生成响应信封的时间，使用 ISO 8601 格式 |

字段约束：

- 标准 JSON API 响应顶层字段固定为 `code`、`msg`、`data`、`request_id`、`server_time`。
- 第一阶段不得增加新的顶层字段；如果需要增加 `trace_id`、`details`、`meta` 等字段，必须升版本并记录变更原因。
- `data` 字段必须存在。成功但无业务数据时使用 `null`，不得省略字段。
- `msg` 是面向调用方的结果说明，不作为客户端分支判断依据；客户端只能根据 HTTP 状态码和业务 `code` 做逻辑判断。
- 标准 JSON API 的响应媒体类型为 `application/json`。

### 4.2 成功响应

成功响应使用 HTTP `200`，响应体使用 `HttpEnvelope[T]` 表达，业务 `code` 固定为 `"0"`。

示例：

```json
{
  "code": "0",
  "msg": "success",
  "data": {
    "id": 10001,
    "name": "Alice"
  },
  "request_id": "01J2Z8N8F6Q2Q6GZ9W4R2R0P8X",
  "server_time": "2026-06-21T22:30:00+08:00"
}
```

### 4.3 错误响应

错误响应使用对应 HTTP `4xx` 或 `5xx` 状态码，响应体使用 `ErrorEnvelope` 表达，业务 `code` 用于细分具体错误原因。

示例：

```json
{
  "code": "300004",
  "msg": "user not found",
  "data": {
    "user_id": 10001
  },
  "request_id": "01J2Z8N8F6Q2Q6GZ9W4R2R0P8X",
  "server_time": "2026-06-21T22:30:00+08:00"
}
```

错误详情约束：

- 错误响应的 `data` 只允许放置可安全暴露给调用方的结构化错误详情。
- 参数校验错误的 `data` 应使用稳定结构表达字段路径、错误类型和错误说明，避免直接透出未治理的框架内部错误对象。
- 未捕获系统异常、第三方异常和超时默认不向调用方暴露内部堆栈、SQL、密钥、请求头、上游原始响应体等敏感信息。
- 需要暴露调试信息时，只能通过受控日志和 `request_id` 检索，不通过响应信封返回。

## 5. HTTP 状态码策略

HTTP 状态码负责描述请求层级结果，必须用于网关、监控、日志、客户端拦截器和第三方调用方判断请求是否成功。

处理规则：

| 场景 | HTTP 状态码 | 说明 |
|---|---:|---|
| 正常返回 | `200` | 请求成功，业务 `code="0"` |
| 参数错误 | `400` | 请求参数不合法或请求体校验失败 |
| 未认证 | `401` | 缺少身份凭证或身份凭证无效 |
| 无权限 | `403` | 已认证，但无权访问目标资源 |
| 资源不存在 | `404` | 请求的资源不存在 |
| 业务状态冲突 | `409` | 请求语义有效，但与当前资源状态冲突 |
| 服务内部异常 | `500` | 服务崩溃、未捕获异常或内部处理失败 |
| 上游服务异常 | `502` | 第三方服务返回异常结果 |
| 上游服务超时 | `504` | 第三方服务或下游依赖超时 |

约束：

- 不把所有业务失败都包装成 HTTP `200`。
- 不把 HTTP 状态码拼进业务 `code`。
- HTTP `4xx` 表示调用方请求存在问题或当前请求无法被业务接受。
- HTTP `5xx` 表示服务端或服务端依赖出现问题。
- 前端拦截器应先按 HTTP 状态码分层处理，再按业务 `code` 细分提示和流程。
- 标准 JSON API 不使用 HTTP `204`。无业务返回值的成功请求仍返回 HTTP `200`，且响应体中 `data=null`。
- 如果少数接口必须使用 HTTP `204`，该接口属于响应信封例外，必须在路由声明和 OpenAPI 中显式标注。

## 6. 业务 `code` 编码规范

业务 `code` 是独立业务码，不承担 HTTP 状态码职责。

固定规则：

- 成功响应的业务 `code` 必须为 `"0"`。
- 失败响应的业务 `code` 必须为非 `"0"`。
- 业务 `code` 必须集中维护，不允许接口内部临时编写魔法数字。
- 同一个业务 `code` 在系统内必须具有唯一语义。
- HTTP 状态码相同的错误，可以通过业务 `code` 继续细分。
- 不依赖业务 `code` 的数值大小做逻辑判断，只允许做等值匹配。

推荐优先使用字符串数字编码。以下分段是业务码治理示例，不是强制业务域划分：

```text
"0": 成功
"100000"-"199999": 通用请求类错误
"200000"-"299999": 认证与权限类错误
"300000"-"399999": 用户域错误
"400000"-"499999": 订单域错误
"900000"-"999999": 系统与基础设施类错误
```

具体项目必须根据实际业务域、团队边界和错误治理方式设计业务码注册表。是否存在用户域、订单域，以及各业务域使用哪些码段，应由项目实际业务决定；本文示例只用于说明业务码可以按领域分段治理。

业务 `code` 使用字符串，是因为它本质上是稳定标识符，不参与数学计算。字符串数字可以保留分段治理能力，同时为后续扩展 `"USER_NOT_FOUND"`、`"AUTH_TOKEN_EXPIRED"` 这类语义码留下兼容空间。

示例编码：

| HTTP 状态码 | 业务 `code` | 说明 |
|---:|---:|---|
| `200` | `"0"` | 成功 |
| `400` | `"100001"` | 请求参数校验失败 |
| `400` | `"100002"` | 请求追踪 ID 非法 |
| `401` | `"200001"` | 未认证 |
| `403` | `"200003"` | 无权限 |
| `404` | `"300004"` | 用户不存在 |
| `409` | `"400009"` | 订单状态冲突 |
| `500` | `"900500"` | 未处理系统异常 |
| `502` | `"900502"` | 第三方服务异常 |
| `504` | `"900504"` | 第三方服务超时 |

## 7. HTTP 状态码与业务码关系

HTTP 状态码和业务 `code` 是两层判断，不互相替代。

```text
第一层：HTTP status 判断请求层级结果
第二层：business code 判断业务层级结果
```

典型处理方式：

- 网关、APM、日志平台通过 HTTP 状态码快速统计 `2xx`、`4xx`、`5xx`。
- 前端拦截器遇到 HTTP `4xx` 时进入客户端问题处理分支，再根据业务 `code` 展示具体提示。
- 前端拦截器遇到 HTTP `5xx` 时进入服务端问题处理分支，统一展示系统错误或重试提示。
- 第三方调用方可以基于 HTTP 状态码遵循标准 REST 语义，不必先解析响应体才能判断请求失败类型。

## 8. `request_id` 规则

`request_id` 是所有响应的必填字段。

生成规则：

- 如果请求头中存在合法的 `X-Request-ID`，服务端沿用该值。
- 如果请求头中不存在 `X-Request-ID`，服务端在请求入口生成新的 `request_id`。
- 响应头中也应写回同一个 `X-Request-ID`。
- 日志记录、异常处理、响应信封必须使用同一个 `request_id`。

合法性规则：

- `X-Request-ID` 长度必须在 `1` 到 `128` 个字符之间。
- `X-Request-ID` 只允许使用 ASCII 字母、数字、点号、下划线、冒号和连字符。
- 非法 `X-Request-ID` 不做静默替换，返回 HTTP `400`，业务 `code="100002"`。
- 服务端生成的 `request_id` 应使用 ULID、UUIDv7 或同等级的唯一 ID 格式，并满足上述合法性规则。

## 9. `server_time` 规则

`server_time` 是所有响应的必填字段。

生成规则：

- `server_time` 表示服务端生成响应信封的时间。
- `server_time` 使用 ISO 8601 字符串格式，例如 `2026-06-21T22:30:00+08:00`。
- `server_time` 必须包含时区偏移，不使用无时区的本地时间字符串。
- 成功响应和错误响应都必须携带 `server_time`。
- `server_time` 用于客户端展示校准、联调排查、日志对齐和第三方调用方确认响应生成时间。
- `server_time` 不替代业务数据中的业务时间字段，例如订单创建时间、支付过期时间、任务执行时间。
- 关键业务判断仍以后端计算结果为准，客户端不得仅依赖 `server_time` 自行完成安全敏感判断。

## 10. 总体请求链路

```text
Client
  -> RequestId Middleware
  -> FastAPI Route
  -> Business Handler
       -> return raw data -> Success Envelope Layer
       -> raise BusinessException / ValidationError / Exception -> Global Exception Handler
  -> Response Header: X-Request-ID
  -> Client
```

成功链路：

```text
业务接口返回裸数据
  -> 成功包装层读取 request_id
  -> 生成 server_time
  -> success_resp(data, request_id, server_time)
  -> HttpEnvelope[T]
  -> HTTP 200 JSONResponse
```

失败链路：

```text
业务接口抛出异常
  -> 全局异常处理器识别异常类型
  -> 生成 server_time
  -> error_resp(code, msg, request_id, server_time, data)
  -> ErrorEnvelope
  -> HTTP 4xx / 5xx JSONResponse
```

## 11. 模块职责

### 11.1 Envelope Models

模型层负责定义响应信封结构。

建议模型：

- `HttpEnvelope[T]`：成功响应泛型模型。
- `ErrorEnvelope`：错误响应模型。

模型层只描述结构，不包含业务判断。

### 11.2 Response Factory

工厂层负责统一创建响应对象。

建议函数：

- `success_resp(data, request_id, server_time, code="0", msg="success")`
- `error_resp(code, msg, request_id, server_time, data=None)`

工厂层负责消除重复字段拼接，业务代码不直接实例化响应信封。

### 11.3 RequestId Middleware

请求追踪中间件负责创建、读取和回写 `request_id`。

职责：

- 从 `X-Request-ID` 读取请求 ID。
- 请求头缺失时生成请求 ID。
- 将请求 ID 写入请求上下文。
- 将请求 ID 写入响应头。

### 11.4 Success Envelope Layer

成功包装层负责自动包装标准 JSON 响应。它是逻辑职责，不强制只能实现为普通 ASGI middleware。

实现约束：

- 如果需要包装业务接口的原始返回值，应优先使用 FastAPI `APIRoute` 包装、路由装饰器或同等级的框架扩展点。
- 普通 Starlette / ASGI middleware 在 `call_next` 之后通常只能看到已经序列化后的 `Response`，不得依赖解析任意响应体作为主要包装机制。
- 如果项目最终选择 middleware 方案，必须在测试中覆盖响应体读取、媒体类型判断、异常路径、流式响应和大响应体场景。

职责：

- 拦截业务接口返回的原始业务数据。
- 判断响应是否属于可包装的 JSON API 响应。
- 生成响应信封的 `server_time`。
- 调用 `success_resp` 生成 `HttpEnvelope[T]`。
- 返回 HTTP `200`。
- 保持业务接口无侵入。
- 对已明确跳过包装的响应保持原样，并在路由或配置中显式声明。
- 不对已经构造好的信封做二次包装；业务接口仍禁止手写信封，测试应覆盖该违规场景。

不应包装的场景：

- `/docs`
- `/redoc`
- `/openapi.json`
- 文件下载响应
- 流式响应
- 已经明确声明跳过包装的内部健康检查接口

### 11.5 Global Exception Handlers

异常处理层作为统一错误外观，负责把不同异常转换为 `ErrorEnvelope`。

必须覆盖：

- `BusinessException`
- FastAPI / Pydantic 参数校验异常
- 未捕获系统异常
- 第三方服务异常或超时

处理原则：

- 业务异常按异常自带的 HTTP 状态码和业务码返回。
- 参数校验异常返回 HTTP `400`，业务 `code="100001"`。
- 必须覆盖 FastAPI 默认参数校验响应，不能保留默认 HTTP `422` 外观。
- 未捕获系统异常返回 HTTP `500`，业务 `code="900500"`。
- 响应模型序列化或响应校验失败属于服务端实现错误，返回 HTTP `500`，业务 `code="900500"`。
- 第三方服务异常返回 HTTP `502`，业务 `code="900502"`。
- 第三方服务超时返回 HTTP `504`，业务 `code="900504"`。
- 未捕获系统异常不向调用方暴露内部堆栈和敏感信息。
- 异常必须记录日志，并携带 `request_id`。
- 错误信封必须生成 `server_time`，且格式与成功信封一致。

## 12. 业务接口编写规范

业务接口只允许做两件事：

```text
返回原始业务数据
抛出业务异常
```

允许示例：

```python
@router.get("/users/{user_id}")
async def get_user(user_id: int):
    return {"id": user_id, "name": "Alice"}
```

允许示例：

```python
@router.get("/users/{user_id}")
async def get_user(user_id: int):
    raise BusinessException(
        http_status=404,
        code="300004",
        msg="user not found",
        data={"user_id": user_id},
    )
```

禁止示例：

```python
@router.get("/users/{user_id}")
async def get_user(user_id: int):
    return {
        "code": "0",
        "msg": "success",
        "data": {"id": user_id},
        "request_id": "manual-id",
        "server_time": "2026-06-21T22:30:00+08:00",
    }
```

## 13. 设计模式映射

设计模式用于解释职责划分，不作为堆砌概念的目标。

| 设计模式 | 落点 | 作用 |
|---|---|---|
| Response Envelope Pattern | `HttpEnvelope[T]`、`ErrorEnvelope` | 固化统一响应外层结构 |
| 模板方法思想 | Envelope 基础字段定义 | 强制成功和错误响应共享统一顶层字段 |
| 装饰器思想 | 成功包装层 | 对业务返回值进行透明包装 |
| 简单工厂 | `success_resp`、`error_resp` | 集中创建标准响应对象 |
| 外观模式 | 全局异常处理器 | 收敛不同异常来源，统一输出错误信封 |

## 14. OpenAPI 文档策略

成功包装层会在运行时统一包装接口返回值，因此需要单独处理 OpenAPI 展示问题。

推荐策略：

- 对外正式 API 应声明最终响应信封结构。
- 内部实现仍保持业务接口返回裸数据。
- 可以通过统一响应模型工具函数生成 `HttpEnvelope[BizModel]`，避免每个接口重复声明。
- 错误响应需要在 OpenAPI 中声明常见 HTTP `4xx`、`5xx` 响应结构。
- 请求头 `X-Request-ID` 的格式约束应在 OpenAPI 中声明。
- 如果存在响应信封例外接口，必须在 OpenAPI 中显式展示真实响应结构。

需要注意：

- 如果只依赖中间件运行时包装，OpenAPI 可能展示裸业务模型。
- 如果项目要求 SDK 生成或强契约联调，必须让 OpenAPI 展示信封后的结构。
- 第三方调用方应能通过 HTTP 状态码先判断请求失败类型，再读取业务 `code` 做细分处理。

## 15. 扩展边界

后续可以扩展的能力：

- 业务码注册表。
- 多语言 `msg` 映射。
- 与日志系统绑定 `request_id`。
- 与链路追踪系统绑定 trace id。
- 统一分页结构。
- 统一批量错误详情结构。

扩展约束：

- 不得破坏现有顶层字段。
- 不得让业务接口重新手动拼装信封。
- 不得为异常添加静默兜底逻辑。
- 新业务码必须在集中定义处登记，并保持唯一语义。
- 不得将所有异常统一压成 HTTP `200`。

## 16. 验证标准

实现完成后至少验证以下场景：

| 场景 | 预期 |
|---|---|
| 正常接口返回 `dict` | HTTP `200`，自动包装为 `HttpEnvelope`，业务 `code="0"` |
| 正常接口返回 `list` | HTTP `200`，自动包装为 `HttpEnvelope`，业务 `code="0"` |
| 业务异常 | 返回对应 HTTP `4xx` 或 `5xx`，并返回对应业务 `code` |
| 参数校验异常 | HTTP `400`，业务 `code="100001"` |
| 未捕获系统异常 | HTTP `500`，业务 `code="900500"` |
| 第三方服务异常 | HTTP `502`，业务 `code="900502"` |
| 第三方服务超时 | HTTP `504`，业务 `code="900504"` |
| 请求头携带 `X-Request-ID` | 响应信封和响应头沿用该值 |
| 请求头携带非法 `X-Request-ID` | HTTP `400`，业务 `code="100002"`，不静默替换 |
| 请求头不携带 `X-Request-ID` | 服务端生成并写回 |
| 任意成功或错误响应 | 响应信封包含 ISO 8601 格式的 `server_time` |
| 成功接口无业务返回值 | HTTP `200`，业务 `code="0"`，`data=null` |
| `/openapi.json` | 不被业务响应信封包装 |
| 文件或流式响应 | 不被业务响应信封包装 |
| OpenAPI 正式 API 响应 | 展示信封后的成功结构和常见错误结构 |

## 17. 最小交付清单

第一阶段实现应包含：

- `HttpEnvelope[T]`
- `ErrorEnvelope`
- 统一业务码定义
- `BusinessException`
- `success_resp`
- `error_resp`
- `RequestId Middleware`
- `ServerTime` 生成工具或统一响应时间生成逻辑
- `Success Envelope Layer` 及其实现落点说明
- 成功包装层对应测试
- 全局异常处理器
- 示例接口
- 单元测试或集成测试

完成标准：

- 所有标准 JSON API 输出结构一致。
- 业务接口没有手动拼装响应信封。
- 成功、业务异常、校验异常、系统异常都能返回统一结构。
- 成功响应使用 HTTP `200`，业务 `code="0"`。
- 客户端问题使用对应 HTTP `4xx`，并用业务 `code` 细分。
- 服务端或第三方依赖问题使用对应 HTTP `5xx`，并用业务 `code` 细分。
- `request_id` 在响应体、响应头和日志中保持一致。
- `server_time` 在成功和错误响应中都存在，并使用 ISO 8601 字符串格式。
