# FastAPI 服务鉴权与接入边界方法论

> 指导轻量 FastAPI 服务按接口暴露面、服务角色、调用主体、凭证模式、密钥与会话生命周期、授权限流、浏览器安全、身份透传、失败可观测和阶段演进，形成可验证的接入边界。

```text
FastAPI 服务鉴权与接入边界
├── Surface：接口暴露面
│   └── 区分公开接口、认证入口、受保护业务接口、文件产物、长连接、内部接口和运维接口
├── Role：服务角色与状态属性
│   └── 区分无状态 API 网关、Headless Service、平台服务、基础设施层服务，以及有状态和无状态的边界
├── Actor：调用主体分叉
│   └── 判断服务是纯机器调用，还是存在终端用户、登录态、租户和资源权限
├── Credential：凭证模式
│   └── 按机器调用、用户登录、混合调用选择 API Key、JWT、Session、mTLS、Auth 服务或网关 SSO
├── Secret：密钥与会话生命周期
│   └── 固定 API Key、access token、refresh token、session 和签名密钥的生成、刷新、撤销与轮换边界
├── Policy：权限、授权与限流策略
│   └── 区分认证、授权、资源权限、配额、限流和防滥用
├── Browser：浏览器接入边界
│   └── 明确 CORS 不替代鉴权，Cookie 登录态必须处理 CSRF 和 Cookie 安全属性
├── Transport：传输安全与身份透传
│   └── 判断 HTTPS、反向代理、可信 Header、内部 JWT、mTLS 和多跳身份透传边界
├── Failure：失败与可观测
│   └── 统一 401、403、429、503、账号枚举防护、鉴权依赖故障、审计日志和敏感信息脱敏
├── Verify：验证闭环
│   └── 按服务形态验证鉴权、授权、限流、CORS/CSRF、身份透传、长连接、文件产物、HTTPS、日志和文档一致
└── Stage：演进触发
    └── 判断从内部工具、第三方接入、前端接入、用户系统到多租户平台时何时升级鉴权模型
```

## 核心命题

FastAPI 服务鉴权与接入边界处理的不是“给接口加一个 `Depends`”，也不是完整企业安全体系建设，而是“一个轻量 API 服务如何在对外接入前，形成清楚、可实现、可验证、可维护的访问边界”。

轻量 FastAPI 服务经常先从内部工具、模型能力、数据处理或第三方集成起步。早期为了快，容易只做一个共享 API Key；后来一旦出现多个调用方、浏览器前端、外部合作方、生产部署、限流、审计或密钥泄漏风险，原来的临时鉴权就会变成接口契约、运维和排障的共同隐患。

这套方法论的任务，是把鉴权问题从“某个项目当前怎么接入”抽象为一组稳定判断：先定接口暴露面，再识别服务角色与状态属性，再判断调用主体是否包含终端用户，然后选择凭证模式，固定密钥或会话生命周期、权限与限流、浏览器安全、传输与身份透传、失败可观测、验证闭环和升级触发条件。

它的核心转化是：

```text
接口可访问 -> 暴露面分级 -> 服务角色与状态属性 -> 调用主体分叉 -> 凭证模式选择 -> 密钥/会话生命周期 -> 权限与限流 -> 浏览器与传输边界 -> 失败可观测 -> 接入验证 -> 演进触发
```

## 方法论节点

这套方法论不是 OAuth、JWT、CSRF 或 FastAPI Security 的教程，也不是完整 IAM 设计。具体项目可以使用 `APIKeyHeader`、`HTTPBearer`、中间件、依赖注入、网关、Auth 服务或反向代理实现，但方法论只关心这些实现是否服务于清楚的接入边界。

### Surface：接口暴露面

先区分哪些接口可以公开访问，哪些接口必须受保护。

接口暴露面至少应分成以下几类：

| 接口类型 | 典型例子 | 默认判断 |
| --- | --- | --- |
| 公开接口 | `/health`、公开说明页、静态 OpenAPI 文档、第三方 OAuth 回调入口 | 可以不鉴权，但不能泄漏内部状态和敏感配置；OAuth 回调必须校验 `state` 和来源 |
| 认证入口 | 注册、登录、刷新 token、登出、找回密码、SSO 回调 | 不要求已有登录态，但必须有防爆破、防 CSRF、回调校验、刷新和撤销边界 |
| 受保护业务接口 | 翻译、生成、导入、导出、任务提交、结果查询、用户资源读写 | 默认必须鉴权，并绑定调用方、用户、租户或资源身份 |
| 文件与产物接口 | 上传文件、下载结果、预签名 URL、临时访问凭证 | 必须绑定用户、租户、任务或调用方；临时 URL 要有有效期、作用域和撤销边界 |
| 长连接接口 | WebSocket、SSE、长轮询、实时任务推送 | 握手或建立连接前完成鉴权，并说明连接中 token 过期、断线重连和权限变化如何处理 |
| 内部接口 | worker 回调、管理任务、重算、补偿、调试入口、服务间回调 | 默认不对公网暴露，必要时做内网、网关、mTLS、内部 JWT 或签名 header |
| 运维接口 | readiness、metrics、debug、admin | 默认按部署环境隔离，不能直接复用普通业务 Key 或普通用户 token |

`Surface` 的完成标准是：新增接口时能立即判断它属于哪一类，是否需要认证，是否允许公网访问，是否会暴露任务状态、数据产物、错误细节或内部配置。

如果项目只有“健康检查不鉴权、其他接口都鉴权”这一条规则，也可以成立；但它必须写成明确的接口表或路由策略，而不是靠开发者记忆。

### Role：服务角色与状态属性

再判断当前对象在系统拓扑中承担什么角色，以及状态归属在哪里。

鉴权边界不能只看服务是否用了 FastAPI，也不能只看它有没有数据库。无状态 API 网关、Kubernetes Headless Service、平台服务和基础设施层服务虽然都可能出现在同一条调用链里，但它们承担的安全职责不同。

| 服务角色 | 本质对象 | 状态判断 | 鉴权与接入边界 |
| --- | --- | --- | --- |
| 无状态 API 网关 | 统一入口、路由、TLS 终止、认证、限流、身份透传 | 网关进程本身不保存业务状态；策略、配额和配置可以由外部存储承载 | 适合做入口认证、统一限流、签名校验和身份透传；后端要明确是否信任网关以及如何防绕过 |
| Headless Service | Kubernetes 服务发现机制，直接暴露后端 Pod DNS，不提供负载均衡虚拟 IP | Headless 本身不是有状态或无状态；常用于 StatefulSet，但状态属于后端 Pod 和持久化存储 | 不是安全边界，也不是 API 网关；不应因为有 Headless Service 就省略后端鉴权、网络策略或 mTLS 判断 |
| 平台服务（Platform Service） | 面向用户、租户、业务方或内部团队提供稳定能力的平台 API | 通常是应用进程无状态、平台数据有状态；状态在数据库、任务表、对象存储或队列中 | 需要用户、租户、角色、配额、审计和资源级授权；API Key 只能表达调用方，不能替代平台权限模型 |
| 基础设施层服务 | 数据库、Redis、消息队列、对象存储、模型运行时、配置中心等基础能力 | 多数是有状态基础设施，或承载关键共享状态 | 不应直接暴露给普通业务调用方；通常使用网络隔离、IAM、mTLS、账号权限和最小授权，而不是普通业务 API Key |

有状态和无状态应至少分三层判断：

```text
进程是否无状态：实例重启或横向扩缩容是否丢失业务语义
数据是否有状态：数据库、队列、缓存、对象存储或任务表是否保存业务状态
会话是否有状态：认证是否依赖服务端 session，还是依赖可验证 token 或外部会话存储
```

“应用进程无状态”不等于“系统无状态”。一个 FastAPI 平台服务可以做到进程无状态，方便多副本部署；但只要它依赖任务表、用户表、租户配置、审计日志或对象存储，系统整体就是有状态的。鉴权设计应围绕状态归属做判断：凭证在哪里校验，身份如何透传，权限策略在哪里存储，限流计数在哪里累计，审计记录在哪里落库。

`Role` 的完成标准是：读者能说清当前对象是入口网关、服务发现、平台能力服务还是基础设施依赖；能区分进程无状态、数据有状态和会话有状态；不会把 Headless Service 当成网关，也不会把 API Gateway 的入口鉴权误认为后端服务天然安全。

### Actor：调用主体分叉

在选择凭证之前，先判断服务是否存在终端用户。

很多鉴权混乱不是因为选错了 JWT 或 API Key，而是没有先区分“机器在调用能力”还是“用户在操作资源”。API Key 适合表达调用方或系统身份；JWT、Session 或 SSO 适合表达用户、租户、角色和资源权限。二者可以组合，但不能互相替代。

默认分叉可以这样判断：

```text
服务是否有终端用户（人）？
├── 否：纯机器调用
│   ├── 内部服务间：mTLS、网关透传、内部 JWT、签名 Header 或受控 API Key
│   ├── 受信任第三方：独立 API Key、调用方配额、轮换撤销和接入联调
│   └── 公开开放 API：Key 注册发放、配额分级、滥用防护、欠费或违规停用
└── 是：有用户登录态
    ├── 浏览器 Web：Session/Cookie 或 BFF 模式，必须处理 CSRF、Cookie 安全属性和用户级权限
    ├── 移动端 / SPA：JWT Bearer、refresh token、设备维度撤销和强制失效
    └── 混合调用：网关统一入口、用户身份透传、服务间身份校验和资源级授权
```

纯机器调用也要继续细分。内部服务间调用的主要问题是防止绕过网关、伪造内部 Header 和跨环境误连；受信任第三方的主要问题是独立 Key、配额、轮换、审计和联调；公开开放 API 的主要问题是自助注册、Key 发放、套餐配额、滥用治理、欠费停用和服务条款。

有用户系统时，认证入口本身也要进入 `Surface` 判断。注册、登录、刷新 token、登出、SSO 回调和找回密码通常不能要求已有登录态，但必须有防爆破、CSRF 或回调 `state` 校验、token 生命周期和账号风险控制。业务接口则不能只验证“用户已登录”，还要判断用户是否有权操作目标租户、资源或记录。

混合调用需要额外固定凭证优先级和入口隔离。一个接口如果同时允许用户 JWT 和内部服务凭证，必须说明两种凭证分别代表谁、权限如何映射、审计归属写谁、冲突时以哪个身份为准。更稳的默认做法是用不同路由、不同网关路径或不同 audience 隔离用户调用和服务调用；只有确实需要同一路由复用时，才允许双凭证共存，并在代码中显式分支处理。

`Actor` 的完成标准是：读者能先把服务归到纯机器调用、有用户登录态或混合调用；知道 API Key、用户 token、Session 和网关身份各自表达的主体；能说明同一路由是否允许多种凭证共存，以及凭证冲突时如何处理；不会把“调用方通过认证”误认为“用户有权操作资源”。

### Credential：凭证模式

根据调用主体、信任关系和部署形态选择凭证模式。

轻量服务最常见的默认模式是 API Key。它适合内部系统、服务间调用、受信任第三方和没有用户登录态的能力型 API。API Key 的核心对象是“调用方”，不是终端用户。

常见凭证模式可以这样判断：

| 凭证模式 | 适合场景 | 风险与边界 |
| --- | --- | --- |
| API Key | 服务间调用、内部系统、受信任第三方、小型能力服务 | 不表达用户身份；不应放进浏览器、移动端包体或公开仓库 |
| 内部 JWT / 签名 Header | 网关到后端、服务间多跳调用、需要身份透传 | 后端必须验证签名、来源、过期时间和受信任 issuer，不能盲信普通 header |
| JWT Bearer | 移动端、SPA、开放 API 用户态访问、有用户身份和租户声明 | 需要签名校验、过期处理、发行方约束、刷新机制和强制失效策略 |
| Session / Cookie | 浏览器后台管理、同站点 Web 应用、BFF 模式 | 需要 CSRF、Cookie 安全属性、服务端撤销能力和会话生命周期 |
| mTLS | 高信任服务间调用、内网平台、强服务身份绑定 | 证书签发、轮换和部署成本更高，不直接表达终端用户权限 |
| 独立 Auth 服务 | 多应用共享登录、账号体系开始复用、需要统一 token 签发和撤销 | 需要清楚划分 Auth 服务、业务服务和网关的信任边界 |
| 网关 SSO / API Gateway 鉴权 | 统一入口、多个后端服务、企业登录、集中认证和限流 | 后端要明确身份透传格式、签名校验、防绕过和服务间调用规则 |

有用户系统时，不应只写“升级到 JWT”。至少要判断三条路线：

| 路线 | 适合阶段 | 信任边界 |
| --- | --- | --- |
| 服务自签 JWT | 单个应用、账号体系简单、后端自己负责登录和 token 签发 | 业务服务同时承担 Auth 和 Resource Server；要处理 refresh token、撤销和密钥轮换 |
| 独立 Auth 服务 | 多个应用共享用户、需要统一登录、统一撤销或账号风控 | Auth 服务签发 token，业务服务只校验 token 和做资源授权 |
| 网关 SSO | 已有企业 SSO、统一 API Gateway、多个后端服务入口集中 | 网关完成入口认证，但后端仍要校验可信透传、防直连绕过和资源级权限 |

社会化登录或 OAuth2 第三方登录只负责“用户是谁、是否授权登录”。它不自动解决业务系统里的角色、租户、资源权限和会话撤销。第三方 OAuth 回调应被视为认证入口，必须校验 `state`、回调地址、授权码交换和账号绑定策略。

混合调用场景中，不要让内部服务凭证绕过用户授权。服务凭证可以证明“哪个服务在调用”，但如果调用是代表某个用户发起，还应携带可验证的用户上下文、delegation 信息或明确的后台任务身份，并在审计中区分 `actor_user`、`actor_service` 和 `on_behalf_of`。

不要把 API Key 误用成用户权限系统。只要需求开始出现用户级权限、租户隔离、操作审计、角色差异或按用户计费，就应从共享 Key 进入 JWT、Session、独立 Auth 服务或网关身份体系，而不是继续扩展 Key 的含义。

### Secret：密钥与会话生命周期

密钥、token 和 session 都不是配置字符串，而是有生命周期的安全对象。

API Key 至少需要回答：

```text
谁生成
在哪里配置
如何分配给调用方
是否每个调用方独立
如何撤销和轮换
生产 Key 如何避免进入仓库、前端、日志和截图
服务启动时没有有效密钥是 fail-open 还是 fail-closed
```

API Key 的生命周期要求取决于调用方类型：

| 机器调用类型 | Key 生命周期重点 |
| --- | --- |
| 内部服务间 | 环境隔离、配置中心或 secret 管理、服务身份绑定、防止跨环境复用 |
| 受信任第三方 | 每个第三方独立 Key、接入登记、定期轮换、泄漏撤销和联调记录 |
| 公开开放 API | 自助申请、Key 展示一次、重置、套餐配额、欠费停用、滥用封禁和审计 |

轻量服务可以用环境变量维护有效 Key 列表，但生产环境应尽量做到调用方独立 Key。独立 Key 才能支持按调用方限流、禁用、审计和灰度迁移。多个调用方长期共用一个生产 Key，会让泄漏追踪、撤销和配额治理失效。

JWT 和 Session 也要有生命周期判断：

| 对象 | 生命周期重点 |
| --- | --- |
| Access Token | 短有效期、签名校验、issuer/audience 校验、过期后的刷新路径 |
| Refresh Token | 更长有效期、服务端存储或可撤销记录、旋转、防重放、设备维度管理 |
| Session | 服务端撤销、过期时间、闲置超时、Cookie 安全属性、登出后失效 |
| 签名密钥 | 轮换计划、kid、旧 token 兼容窗口、泄漏后的批量失效策略 |

强制失效不是 JWT 的天然能力。用户改密、账号封禁、管理员踢下线、租户停用或设备丢失时，如果仍需要立即失效，就必须有服务端状态：session 存储、refresh token 表、token version、黑名单、设备会话表或网关撤销机制。

强制失效方案应和阶段匹配。轻量单服务可以优先使用 `token_version`、`session_version` 或 refresh token 表作为最小可行方案；多设备登录需要设备会话表；跨应用统一登录或企业 SSO 再考虑集中撤销、网关撤销或专门的 token introspection。不要为了一个内部工具一开始就引入全局黑名单和复杂撤销平台。

浏览器、移动端和服务间调用的生命周期不应混用。浏览器 Cookie Session 更适合服务端撤销和同站点 Web；移动端和 SPA 常用 Bearer token 与 refresh token；服务间调用更适合 API Key、mTLS、内部 JWT 或签名请求。

密钥校验应避免把真实 Key、token 或 session id 写进错误响应、日志或调试输出。认证失败日志只记录 `missing_api_key`、`invalid_api_key`、`expired_token`、`invalid_session`、`auth_not_configured` 这类原因，不记录凭证原文。

### Policy：权限、授权与限流策略

认证只回答“调用方或用户是谁”，不自动回答“它能做什么、能调用多少次、能操作哪条资源”。

当服务只有一个受保护能力时，API Key 通过即获得完整调用权限是可接受的最小策略。但只要出现多调用方、多接口、多环境、多租户、不同配额、用户角色或资源归属，就应把授权与限流作为独立策略建模。

策略至少应覆盖：

```text
调用方身份：Key、client_id、tenant_id、user_id、service_id
接口权限：哪些路径、方法或能力允许访问
用户权限：角色、租户、组织、资源归属和操作范围
环境隔离：测试 Key 不能访问生产服务，测试用户不能操作生产租户
配额限流：按 Key、用户、IP、租户、套餐或服务维度计算
重试退避：429 是否返回 Retry-After
异常滥用：短时间失败、爆破、超大 payload 和高频轮询如何处理
```

授权模型可以从轻到重分层：

| 授权模型 | 适合场景 | 判断重点 |
| --- | --- | --- |
| 能力级权限 | API Key 是否能调用某类接口 | 适合纯机器调用和能力型 API |
| RBAC | 用户按角色访问后台、管理台或平台能力 | 角色只表达粗粒度职责，不应直接替代资源归属判断 |
| 资源级权限 | 用户只能访问自己的数据、任务、文件或项目 | 每次读写资源前校验 owner、tenant、project 或 sharing 关系 |
| ABAC | 按租户、环境、资源属性、用户属性、时间或状态判断权限 | 适合多租户、企业规则、复杂审批和上下文权限 |
| Scope / Claim | OAuth2、JWT 或开放 API 中限制 token 能力范围 | scope 只是声明，业务服务仍要校验资源和租户边界 |

对有用户系统的服务，“JWT 通过”只说明 token 有效，不说明用户有权操作目标记录。业务接口应在 service 或 domain 层校验资源归属、租户边界和操作权限，不要只在路由层做登录校验。

对能力型 API，按 API Key 维度限流通常比按 IP 更稳定，因为调用方可能经过代理、网关或 NAT。对浏览器或用户态系统，通常还需要业务后端做用户级、租户级或资源级限流，不能只依赖下游能力服务的 Key 级限流。

公开开放 API 还要把配额和商业状态纳入策略：免费额度、套餐等级、超额降级、欠费停用、违规封禁和申诉恢复都应明确落到调用方或租户维度。

配额和账单状态不应在每个请求里实时查询外部账单系统。更稳的边界是：鉴权或限流层读取本地数据库、Redis 或网关缓存中的配额快照；账单、套餐和封禁状态变化通过后台任务、事件或管理操作更新缓存；请求路径只做快速判断和计数，避免认证链路被账单服务拖垮。

### Browser：浏览器接入边界

CORS 不是鉴权，CSRF 也不能被 CORS 替代。

CORS 只控制浏览器是否允许某个来源读取响应，不阻止非浏览器客户端直接发请求。生产环境即使配置了严格 `ALLOWED_ORIGINS`，业务接口仍然需要真实鉴权。

浏览器场景的默认判断是：

```text
真实 API Key 不进入前端代码
前端通过自己的业务后端代理调用能力服务
业务后端保存服务 Key，并叠加用户级权限和限流
CORS 只开放明确需要的前端来源
生产环境不使用宽泛通配来源
```

如果一个服务必须被浏览器直接调用，应优先判断是否需要短期临时 token、用户态 JWT、网关签发凭证或更细的作用域，而不是把长期 API Key 发给浏览器。

有浏览器登录态时，必须判断 CSRF：

| 浏览器鉴权形态 | CSRF 判断 |
| --- | --- |
| Cookie Session | 默认需要 CSRF 防护，尤其是会改变状态的 `POST`、`PUT`、`PATCH`、`DELETE` |
| SameSite Cookie | `Lax` 可以降低跨站请求风险，但不能替代所有 CSRF token 场景；跨站嵌入或第三方登录回调要单独判断 |
| CSRF Token | 服务端生成 token，前端通过隐藏字段或自定义 header 回传，服务端校验 token 与 session 关系 |
| Double Submit Cookie | Cookie 中放一份 token，请求 header 或 body 中回传一份，服务端比对；适合部分无服务端 session 场景 |
| Bearer Token in Header | 通常不受传统 Cookie CSRF 影响，但要防 XSS、token 存储泄漏和恶意脚本调用 |

认证入口也要进入浏览器安全判断。登录、登出、刷新 token、OAuth 回调、账号绑定、改密和找回密码都可能改变认证状态，应分别判断 CSRF、回调 `state`、重放、防爆破和账号枚举风险。

认证接口不应通过错误消息暴露账号是否存在。登录、找回密码、注册校验和 OAuth 绑定失败时，应避免直接返回“用户不存在”“邮箱已注册但未激活”这类可批量枚举的信息；对外响应应保持模糊，详细原因进入受控日志或后台风控。

### Transport：传输安全与身份透传

只要凭证通过请求头、Cookie 或查询参数传输，生产环境就应使用 HTTPS。

开发环境可以使用 HTTP，本地联调也可以简化证书；但测试、预发和生产环境应明确由应用、反向代理、负载均衡或网关中的哪一层终止 TLS。

`Transport` 要回答：

```text
外部访问是否全程 HTTPS
反向代理是否正确透传 Host、scheme 和 client IP
应用是否依赖 X-Forwarded-* 判断安全上下文
文档是否禁止通过 URL query 传长期凭证
错误页、重定向和 OpenAPI 文档是否意外暴露内部地址
```

对多数 FastAPI 服务，TLS 终止通常在 Nginx、Ingress、负载均衡或 API Gateway。应用本身不一定负责证书，但文档和部署配置必须说清楚生产访问入口不能明文传输凭证。

当网关或上游服务完成入口鉴权后，身份透传必须防伪造。不要让后端盲信普通客户端可写的 `X-User-Id`、`X-Tenant-Id`、`X-Role` 或 `X-Client-Id`。

常见透传方式可以这样判断：

| 透传方式 | 适合场景 | 防伪造要求 |
| --- | --- | --- |
| 受信任网关 Header | 网关和后端在受控内网，后端只接受网关流量 | 网络层防直连，清理外部同名 Header，必要时加内部签名 |
| 内部 JWT | 网关或上游服务签发短期内部 token | 后端校验签名、issuer、audience、过期时间和 key id |
| mTLS + Header | 服务身份由证书保证，用户身份通过 Header 传递 | 校验证书身份，并限制哪些服务可以传递用户身份 |
| Signed Request | 每次请求带时间戳、nonce 和签名 | 防重放、时钟窗口、签名密钥轮换和调用方独立密钥 |

多跳调用时要区分用户身份和服务身份。`A -> B -> C` 中，B 调用 C 时既可能代表用户，也可能代表服务自身。文档和代码应说明是委托调用、服务调用还是后台任务调用；否则容易出现 B 伪造用户、C 误信 Header、审计日志归属不清的问题。

长连接和文件产物不是普通短请求，但仍应回到同一组接入边界判断：

| 场景 | 鉴权判断重点 |
| --- | --- |
| WebSocket / SSE / 长轮询 | 握手或建立连接前完成认证；连接上下文固定用户、租户、服务和权限快照；token 过期、用户禁用或权限变化时，明确断开、要求重连，或在下一次订阅和消息操作时重新校验 |
| 文件上传 | 上传凭证限制文件大小、类型、目标租户、目标任务和有效期，避免跨租户写入或无限制临时凭证 |
| 文件下载 / 预签名 URL | 下载链接绑定任务、用户、租户或调用方，设置短有效期；说明链接泄漏后是否可撤销、是否允许二次传播 |

不要把一次握手成功理解成无限期授权，也不要把拿到临时 URL 理解成永久访问资源。

### Failure：失败与可观测

鉴权失败要既可排查，又不泄漏。

常见状态码应稳定：

| 状态码 | 含义 | 使用边界 |
| --- | --- | --- |
| `401` | 未认证、凭证缺失或无效 | 调用方应检查凭证 |
| `403` | 已认证但无权限 | 调用方身份成立，但不能访问该资源或能力 |
| `429` | 触发限流 | 应返回或记录退避信息 |
| `422` | 请求参数不合法 | 不应混同为鉴权失败 |
| `500` | 服务内部错误 | 不暴露密钥、内部配置或堆栈给调用方 |
| `503` | 鉴权依赖或上游认证服务不可用 | 区分 Auth 服务、JWKS、session 存储、网关依赖故障和普通业务失败 |

响应可以给调用方明确的处理方向，但不要泄漏有效 Key 列表、签名算法细节、内部配置名或过多判定过程。认证接口响应也不应区分“用户不存在”和“密码错误”这类可用于账号枚举的信息。日志侧应记录 `request_id`、调用方标识、接口、状态码、失败原因和限流信息，且必须脱敏凭证。

鉴权依赖不可用时必须明确 fail-closed 还是降级。默认应 fail-closed，避免 Auth 服务超时、JWKS 端点不可达、session 存储宕机时放行请求；只有公开健康检查、静态公开资源或明确允许匿名降级的接口才可以继续访问。相关日志和告警应与业务 500 区分，方便排查认证链路故障。

审计日志不同于普通运行日志。涉及登录、登出、Key 创建、Key 撤销、权限变更、资源访问、越权拒绝、管理员操作和高风险下载时，至少应记录操作者、服务身份、租户、资源、操作类型、结果、时间和 request_id；审计日志应避免记录凭证原文，并考虑防篡改、保留周期和查询权限。

如果服务已经建立运行日志与排障可观测性，应让鉴权日志进入同一 `request_id` 链路；不要单独用 `print` 或临时日志把安全事件散落在各处。

### Verify：验证闭环

鉴权方法必须通过最小验证闭环落地。

最小验证应按服务形态分组覆盖：

```text
所有服务必测
- 公开接口：不带凭证能访问，但不泄漏敏感信息
- 受保护接口：缺失凭证返回 401
- 受保护接口：错误凭证返回 401
- 受保护接口：正确凭证可以访问
- 无权限接口：已认证但无权限返回 403
- 限流：超过阈值返回 429，并带退避约定
- HTTPS：生产入口不会明文传输凭证
- 日志：认证失败、授权失败和限流可排查，但不出现凭证原文
- 文档：接口表、请求头、状态码、Key 管理、token/session 生命周期和接入建议与实现一致

有用户系统时追加
- 认证入口：登录、刷新、登出、OAuth 回调、防爆破和 CSRF/state 校验可验证
- 资源级权限：用户不能访问其他用户、租户或项目的资源
- 资源级越权：用 user_A 的 token 访问 user_B 的 task_id、file_id 或 project_id，应返回 403 或 404，且不泄漏资源是否存在
- CSRF：Cookie 登录态下状态变更接口不能被跨站提交

有浏览器前端时追加
- CORS：只允许预期来源

有服务间或网关透传时追加
- 身份透传：伪造 X-User-Id / X-Tenant-Id 不会绕过网关或后端校验

有长连接时追加
- 长连接：WebSocket 握手缺失或过期凭证会被拒绝，连接中权限变化有明确处理

有文件产物时追加
- 文件产物：过期、越权或跨租户的预签名 URL / 下载链接不能访问资源
```

验证可以是单元测试、集成测试、`curl` 冒烟脚本、浏览器联调、网关联调或上线前清单。资源级越权测试应至少构造两个合法主体，而不是只测匿名访问；例如 user_A 创建任务后，用 user_B 的 token 访问该 `task_id`，应返回 `403` 或 `404`，并且响应不暴露资源归属。关键不在形式，而在它能发现“接口漏鉴权、CORS 被误当安全边界、CSRF 缺失、密钥泄漏、身份 Header 可伪造、IDOR、限流维度错误、文档与实现不一致”这些高风险问题。

### Stage：演进触发

鉴权模型应随服务暴露面和调用主体演进，不应一开始过重，也不应在阶段变化后继续沿用旧模型。

常见演进路径可以这样判断：

| 阶段 | 典型形态 | 当前足够的做法 | 升级触发条件 |
| --- | --- | --- | --- |
| 内部工具 | 单个团队或内部后端调用 | 受控 API Key、内网访问、最小日志和 HTTPS 入口 | 多调用方、跨环境调用、需要审计或限流 |
| 少数第三方 | 指定外部合作方接入 | 独立 API Key、配额、轮换撤销、接入文档和联调清单 | 第三方增多、需要自助接入、套餐或滥用治理 |
| 公开开放 API | 任意开发者可申请调用 | Key 注册发放、配额分级、封禁、账单或服务条款 | 出现终端用户、资源权限或平台账号体系 |
| 前端接入 | 浏览器、SPA、移动端使用能力 | BFF 代理、用户 token、CORS、CSRF 和前端安全边界 | 用户数据、租户隔离、角色权限或设备管理 |
| 有用户系统 | 注册登录、用户资源、账号状态 | JWT/Session/Auth 服务、refresh token、最小强制失效和资源级授权 | 多应用共享登录、企业 SSO、多租户、跨服务权限或必须立即吊销 token |
| 多租户平台 | 用户、组织、租户、套餐、审计 | 独立 Auth、网关 SSO、RBAC/ABAC、租户级配额和审计 | 进入企业 IAM、合规审计、服务网格或平台治理 |

`Stage` 的作用不是鼓励所有项目一次性建设完整 IAM，而是让读者知道当前模型何时还够用、何时必须升级。如果服务已经进入多租户平台，却仍只靠一个共享 API Key 或只校验登录态，说明鉴权模型已经滞后。

多租户平台之后，本方法论只保留入口边界判断，不继续展开企业 IAM、合规审计、服务网格、零信任网络或平台治理细节。进入这些阶段时，应切换到更专门的身份治理、审计合规、服务网格或平台工程方法，而不是继续把所有规则塞进单个 FastAPI 服务鉴权文档。

## 判断标准

FastAPI 服务鉴权与接入边界是否成立，只看一个标准：

**任何新增或对外开放的接口，都能被明确归类到暴露面，识别服务角色、状态属性和调用主体，匹配合适凭证模式，具备密钥、token 或 session 生命周期，明确认证、授权、限流、浏览器和传输边界，并能通过最小验证证明当前阶段的文档与实现一致。**

如果只有业务接口加了一个 Key 校验，但健康检查、任务结果、调试接口或 OpenAPI 文档是否暴露不清楚，说明 `Surface` 没完成。

如果把 Headless Service 当成 API 网关，或把应用进程无状态误认为系统没有业务状态，说明 `Role` 没完成。

如果服务已经有终端用户、租户或用户资源，但仍按纯 API Key 机器调用建模，说明 `Actor` 没完成。

如果 API Key 被前端持有，多个调用方长期共用一个生产 Key，或用户系统只写“使用 JWT”却没有签发、刷新、撤销和授权边界，说明 `Credential` 和 `Secret` 没完成。

如果用户登录成功后可以访问其他用户、租户或项目的资源，说明 `Policy` 没完成。

如果 CORS 配置严格但接口本身没有鉴权，或 Cookie 登录态没有 CSRF 防护，说明 `Browser` 被误用成安全边界。

如果后端盲信外部可伪造的身份 Header，WebSocket 握手后不再处理 token 过期，或预签名下载链接没有作用域和有效期，说明 `Transport` 的身份透传和资源访问边界没完成。

如果认证失败日志记录了真实 Key、JWT、session id 或完整请求头，或者 Auth 服务不可用时放行了受保护接口，说明 `Failure` 的安全边界失败。

如果文档写了鉴权、限流和 HTTPS，但没有任何测试、脚本或联调清单验证，说明 `Verify` 没完成。

如果服务已经从内部工具演进到第三方开放、前端接入或多租户平台，但鉴权模型仍停留在旧阶段，说明 `Stage` 没完成。

## Few-shot Learning 示例组

### 正例：翻译能力 API

一个翻译服务提供健康检查、同步翻译、异步任务提交、任务状态和结果查询。合适做法是让 `/health` 公开但只返回存活状态；所有翻译和任务接口都要求 `X-API-Key`；不同调用方分配独立 Key；限流按 Key 计算；浏览器前端通过业务后端代理；生产入口启用 HTTPS；错误响应使用 `401`、`429` 和 `trace_id`；上线前用缺失 Key、错误 Key、正确 Key、限流和 CORS 做联调。

判断：

```text
Surface：health 公开，业务和任务接口受保护
Role：轻量能力服务，进程无状态，任务和结果状态在后端存储
Actor：纯机器调用或业务后端代理调用，不直接面对终端用户
Credential：API Key 适合服务间和受信任第三方
Secret：每个调用方独立 Key，生产 Key 不进仓库和前端
Policy：按 Key 限流，同步和异步接口配额不同
Browser：前端通过后端代理，不暴露真实 Key
Transport：测试、预发、生产使用 HTTPS
Failure：401/429 清楚，日志记录原因但不记录 Key
Verify：接口、限流、CORS、日志和文档一致
```

### 正例：有用户系统的平台 API

一个平台服务提供注册登录、用户项目、文件处理任务和结果查询。合适做法是把注册、登录、刷新 token、登出和 OAuth 回调归为认证入口；浏览器后台可使用 Session/Cookie + CSRF，移动端或 SPA 可使用 JWT + refresh token；业务接口不只校验 token，还要校验用户是否属于对应租户、是否有项目权限、是否能读取目标任务和结果；服务间调用使用网关透传的内部 JWT 或签名 Header，后端校验 issuer、audience 和过期时间。

判断：

```text
Surface：认证入口和业务接口分开归类
Role：平台服务进程无状态，用户、租户、任务和审计数据有状态
Actor：终端用户 + 服务间混合调用
Credential：Session/JWT/Auth 服务或网关 SSO，而不是共享 API Key
Secret：access token、refresh token、session 和签名密钥都有生命周期
Policy：RBAC + 租户边界 + 资源级权限
Browser：Cookie 登录态必须处理 CSRF，OAuth 回调校验 state
Transport：身份 Header 必须由网关签发或内部 JWT 防伪造
Verify：覆盖登录、刷新、登出、越权访问、CSRF 和身份透传伪造
```

### 反例：把 CORS 当成鉴权

一个服务只配置了 `ALLOWED_ORIGINS=https://example.com`，但业务接口不校验凭证。浏览器来源限制看起来严格，实际任何脚本、服务端程序或命令行都可以直接调用接口。

问题不在 CORS 配置本身，而在于它只处理浏览器策略，不处理调用方身份。

### 反例：JWT 通过就允许访问资源

一个用户已经登录，JWT 校验通过，于是后端允许他读取任意 `task_id` 的结果。这里认证是成立的，但授权失败了。正确做法是继续校验任务属于当前用户、租户或项目，并把越权访问返回 `403` 或资源不可见的业务错误。

### 反例：共享生产 Key

三个第三方长期共用同一个生产 API Key。某个调用方高频调用导致所有调用方一起被限流，Key 泄漏后也无法判断来源，只能整体停用，影响所有接入方。

正确方向是按调用方分配独立 Key，并把限流、审计、撤销和轮换都绑定到调用方。

### 边界例：何时从 API Key 升级

如果服务只是被一个内部后端调用，API Key 足够。若服务开始直接面对用户、需要区分角色、租户、资源权限、用户级配额或审计记录，就不应继续扩展共享 Key，而应升级到 JWT、Session、网关身份或更完整的授权模型。

### 边界例：公开开放 API

当 API 从少数第三方接入升级为任何开发者都能申请 Key，鉴权对象就不再只是“一个受信任调用方”。此时需要 Key 注册发放、套餐配额、超额处理、欠费或违规停用、滥用检测和审计记录。继续用手工环境变量维护 Key 列表会很快失效。

### 边界例：网关已经鉴权

如果 API Gateway 已经完成统一认证、签名校验、限流和身份透传，FastAPI 服务可以不重复实现完整鉴权链路，但必须明确后端信任边界：服务是否只接受网关流量，是否校验内部 header，是否防止绕过网关直连。

### 边界例：Headless Service 不是网关

一个 Kubernetes Headless Service 为多个后端 Pod 提供稳定 DNS。它可以服务于 StatefulSet，也可以用于让调用方直连 Pod，但它不负责认证、TLS 终止、限流或身份透传。因此它不能替代 API Gateway，也不能成为“接口已经受保护”的证据。

判断重点是：Headless Service 解决服务发现，不解决访问控制。访问控制仍应由网络策略、mTLS、后端鉴权、网关入口或基础设施权限共同完成。

### 边界例：平台服务进程无状态但系统有状态

一个平台 API 的 FastAPI 实例可以无本地 session、无本地文件、可横向扩缩容，因此进程是无状态的。但它依赖用户表、租户表、任务表、对象存储和审计日志，所以平台系统是有状态的。

此时鉴权不能只判断“服务是否无状态”，而要判断身份、权限、限流计数和审计状态分别存在哪里，以及多副本部署时是否一致。

## 误用

FastAPI 服务鉴权与接入边界不是安全百科。

不要把 OAuth 细节、密码学算法、KMS 使用、WAF、零信任网络、审计平台和合规流程全部塞进轻量服务鉴权方法。只有当它们改变接口接入边界时，才作为升级分支进入。

它也不是“所有接口都必须复杂鉴权”。

健康检查、只读公开资源、开发环境临时接口可以有不同策略；关键是暴露面被明确归类，并且不会泄漏内部状态或敏感数据。

它也不是“API Key 永远够用”。

API Key 是轻量服务的常见默认模式，不是用户权限系统。出现用户态、租户态、资源态或合规审计需求时，应升级凭证和授权模型。

它也不是服务拓扑百科。

无状态 API 网关、Headless Service、平台服务和基础设施层服务只在它们改变鉴权责任、信任边界、状态归属和访问路径时进入本方法论。完整 Kubernetes 服务发现、服务网格、平台工程或基础设施架构设计，应由更专门的方法论处理。

它也不是完整审计平台设计。

审计日志只在它影响鉴权、授权、追责和高风险操作留痕时进入本方法论；日志存储、防篡改、检索、归档和合规报表应由专门的审计或可观测体系承接。

它也不是只写接入文档。

接入说明必须和代码、配置、测试、部署入口一致。文档写得再完整，如果接口实际漏鉴权、Key 泄漏到日志、生产没有 HTTPS 或限流维度错误，方法论都没有完成。

## 适用边界

这套方法适合处理轻量 FastAPI 能力服务，尤其是：

```text
服务需要对内部系统、第三方或前端后端开放 API
当前采用或准备采用 X-API-Key、JWT、Session、Auth 服务或网关鉴权
需要判断哪些接口公开、哪些接口受保护、哪些接口属于认证入口
需要区分纯机器调用、有用户系统和混合调用
需要补齐密钥管理、token/session 生命周期、授权、限流、CORS、CSRF、HTTPS、身份透传、长连接、文件产物和审计边界
需要把项目级安全接入说明抽象成可复用方法
需要在上线前做鉴权和接入边界检查
```

这套方法不适合替代完整 IAM、OAuth 授权服务器、企业网关治理、合规审计体系、WAF 策略或密码学设计。

如果问题是“接口同步、异步、任务状态和结果查询怎么设计”，应使用 FastAPI 独立服务接口与任务架构方法论。

如果问题是“认证失败、请求日志和敏感信息如何进入运行日志”，应和 FastAPI 服务运行日志与排障可观测性方法论组合使用。

如果问题是“生产部署、环境变量加载和 HTTPS 入口如何组织”，应使用项目部署入口与配置加载方法论。
