# FastAPI 独立服务接口与任务架构方法论

> 指导小型 FastAPI 独立服务在同步请求、同步 batch、异步任务、任务队列和工作流升级之间建立稳定架构判断；当模式成熟后，再沉淀为项目级接口与任务设计契约。

```text
FastAPI 独立服务接口与任务架构
├── Scope：服务边界
│   └── 判断当前服务是否是独立能力服务，以及它对外暴露什么能力
├── Interface：接口形态
│   └── 区分同步请求、异步任务、流式响应、状态查询和结果查询的职责
├── Batch：批量语义
│   └── 判断 batch 是同步批处理还是异步任务入口，以及规模边界在哪里
├── Task：任务模型
│   └── 判断是否进入通用 Job 模型；具体 job_id、状态、投递和恢复语义读取规则真源
├── Queue：队列执行
│   └── 判断普通单步骤执行器 task 是否足以支撑异步任务
├── Workflow：工作流升级
│   └── 只有多步骤、并行汇聚或步骤级可观测时，才升级到任务编排与步骤状态模型
└── Contract：设计契约
    └── 当判断结果稳定后，沉淀为项目级接口和通用 Job 接入契约
```

## 核心命题

FastAPI 独立服务接口与任务架构处理的不是“FastAPI 项目目录怎么摆”，而是“一个小型后端能力服务应该如何稳定区分同步请求、同步 batch、异步任务、任务状态和队列执行形态”。

小型服务经常围绕一个清晰能力展开，例如翻译、生成、转换、导入、分析或处理文件。它们的复杂度通常不在 HTTP 框架本身，而在请求耗时、批量规模、后台执行、结果查询、失败明细和后续扩展之间的边界不清。

如果每个服务都临时设计同步接口、异步任务入口、任务表、结果表和队列模式，项目之间会很快出现接口风格、状态命名、结果查询和排错方式不一致。如果所有服务一开始都套用重型工作流，又会让小项目过早承担 Canvas、步骤表和可观测工作流的复杂度。

这套方法论的任务，是把小型 FastAPI 服务的接口与任务架构收敛成一组稳定判断：外部契约尽量统一，内部执行按复杂度分级。

它的核心转化是：

```text
独立服务能力 -> 交互模式判断 -> batch 边界 -> 任务模型统一 -> 队列执行分级 -> 工作流升级边界 -> 设计契约沉淀
```

## 方法论节点

这套方法论不是 FastAPI 教程，也不是某个执行器的 workflow 模板。它关注的是新增独立服务时，如何快速判断接口形态和任务执行形态，避免每次从零设计。

### Scope：服务边界

先判断当前后端是否适合被视为独立能力服务。

独立服务通常围绕一个可命名能力展开，对外提供有限接口，对内拥有自己的配置、执行逻辑、结果表达和验证方式。它可以依赖数据库、Redis、文件存储或第三方模型，但不应把多个无关业务域混成一个方法对象。

`Scope` 要回答：这个服务提供什么能力，输入输出是否稳定，是否需要持久化结果，调用方是谁，调用是否稳定，是否需要作为可复用能力暴露。

如果服务边界还只是临时脚本、一次性实验或多个业务域的混合入口，不应急着沉淀接口与任务契约。

### Interface：接口形态

区分同步请求、异步任务、流式响应、状态查询和结果查询。

同步、异步、流式和离线批处理是接口交互模式；`batch` 是输入规模和处理单位语义。`batch` 可以同步返回，也可以作为异步任务入口，不能仅凭 `batch` 这个词判断接口形态。

同步接口适合小规模、耗时可控、结果体可控、结果立即返回的请求，也可以承载受限规模的同步 batch。异步任务适合耗时不可控、规模较大、需要后台排队或需要结果持久化的请求。流式响应适合需要边处理边返回、前端增量渲染或持续展示进度的请求。

状态查询和结果查询应承担不同职责：前者回答任务是否完成，后者回答产物是什么、哪些成功、哪些失败。

对于小型独立服务，外部接口可以优先稳定为：

```text
POST /{domain}/sync
POST /{domain}/jobs
GET  /{domain}/jobs/{job_id}
GET  /{domain}/jobs/{job_id}/results
```

这组接口是设计契约的候选形态，不是强制模板。`sync` 可以处理单项请求或受限规模的同步 batch；`jobs` 表达异步任务提交，适合大批量、耗时不可控、需要排队或需要持久化结果的 batch。

真正需要固定的是职责边界：同步接口不伪装成长任务，异步提交不直接承诺结果，状态接口不承载大结果，结果接口不反推任务生命周期。轮询、长轮询、Webhook、SSE 或 WebSocket 属于结果获取或前端感知策略，不应和同步/异步接口形态混成同一个判断。

主架构可以默认收敛到 `sync + jobs + status/results`，但落地时仍应按需求触发交互方式分支，避免把默认接口契约用成一刀切模板：

| 交互方式 | 适用触发条件 | 对架构判断的影响 |
| --- | --- | --- |
| 同步请求-响应 | 短耗时、小结果、前端一次 loading 可以完成 | 使用 `sync`，可承载受限规模同步 batch |
| 异步 job + 轮询 | 大批量、耗时不可控、需要后台排队、需要任务状态 | 使用 `jobs`、`job_id`、状态查询和结果查询；具体状态语义读取通用 Job 规则 |
| 异步 job + Webhook + 轮询兜底 | 第三方接入方需要完成通知，且能提供可回调地址 | 保持任务模型不变，额外增加回调地址、签名、重试、幂等和轮询兜底约束 |
| 流式响应或 SSE | 需要生成式内容、日志、进度事件或前端增量展示 | 优先解决用户感知和持续反馈；如果还需要任务持久化，应与通用 Job 模型组合，而不是替代任务模型 |
| WebSocket | 需要强实时、双向通信或会话型交互 | 不作为普通独立能力服务默认选项，只有连接管理和双向状态确实成立时才引入 |
| 离线批处理 | 超大批量、长周期处理、结果文件下载、非实时业务 | 通常超出轻量同步 batch 和普通异步 job，应单独固定提交、状态、结果文件和保留周期 |

这组枚举的作用是限制随意设计，而不是鼓励每个服务都支持所有模式。每次新增服务时，应先落到最小可用的默认架构；只有当用户体验、第三方接入方式、实时性或数据规模提出明确要求时，才进入对应分支。

### Batch：批量语义

判断 batch 的规模和执行语义。

`batch` 不等于必然异步。小规模 batch 可以走同步接口，只要有明确的最大输入项数、总 payload、总 token、总文件大小、并发限制、超时边界和失败明细。大规模 batch 或耗时不可控的 batch 应进入异步任务。

批量语义要回答：批量单位是什么，部分失败如何表达，结果是否按输入项对齐，是否允许重复提交，是否需要缓存、限流或去重。

同步 batch 要让调用方能在一次 loading 中拿到完整结果和部分失败明细；异步 job 形式的 batch 要让调用方能围绕 `job_id` 查询状态和结果，具体状态名与终态语义读取通用 Job 规则。

如果 batch 只是为了让接口看起来统一，而没有输入规模、失败表达和结果顺序约束，后续很容易在调用方和服务端之间产生歧义。

### Task：任务模型

判断异步任务是否需要进入通用 Job 模型。

异步任务入口一旦返回 `job_id`，它就不只是后台执行，而是一个可查询、可排错、可恢复的协作对象。具体 `job_id`、状态机、终态、投递和恢复语义以 [`rules/backend/jobs/async-job.md`](../../../../rules/backend/jobs/async-job.md) 为准，方法论只判断是否需要这类协作对象。

任务状态应回答任务整体处于什么阶段；进度字段应回答总量、已完成和失败数量；结果表或结果字段应回答每个输入项的产物和错误。批量任务执行完成但存在部分失败时，整体是否进入成功终态由通用 Job 规则和项目结果契约共同约束。不要让一个状态字段同时承担队列状态、业务结果和错误详情。

当项目只需要一次性后台执行时，任务模型可以很轻；当需要前端进度、重试、审计或多业务域复用时，任务模型应向统一任务中心靠拢。

### Queue：队列执行

判断普通单步骤队列任务是否足够。

多数小型服务的异步任务只需要单步骤队列执行：提交后由 worker 执行核心业务逻辑，写入结果，更新任务状态。此时普通执行器 task 已经足够，不需要一开始引入 workflow。

单步骤队列任务适合整体重跑不会造成不可接受副作用，或已具备幂等、去重、补偿机制；同时没有内部步骤依赖、不需要并行拆分和结果汇聚、不需要步骤级进度的任务。它的设计重点是任务创建、事务提交、worker 可见性、结果持久化和失败兜底，而不是编排拓扑。

如果为了统一维护而把所有任务都写成执行器 workflow，会把小型服务的主要复杂度从业务能力转移到工作流框架上。

### Workflow：工作流升级

只有任务复杂度超过单步骤队列时，才升级到工作流。

工作流升级的典型信号是：任务天然分成多个阶段，阶段之间有依赖关系，需要并行处理和结果汇聚，需要步骤级进度，需要断线重连后恢复状态，或者失败时不能只展示整体失败。

此时可以进入任务编排形态，用执行计划表达顺序、并行或汇聚关系，用步骤状态模型表达每个阶段的可观测状态。具体项目可以用步骤表、步骤记录表或其他命名实现，不应把某个项目的表名写成方法论节点。

多步骤 workflow 是复杂任务的升级形态，不是小型服务的默认起点。统一维护应优先统一任务契约、状态模型和目录边界，而不是强制统一所有内部执行机制。

### Contract：设计契约

当接口与任务判断稳定后，沉淀为项目级设计契约。

设计契约不是方法论起点，而是复用结果。它应固定当前项目或同类小型服务默认采用的接口职责、通用 Job 接入方式、结果表达、错误结构、队列模式和升级条件；状态命名、投递和恢复语义不在方法论中另起事实源。

契约沉淀的目标，是让下一次新增独立服务时不再重新争论同步接口、同步 batch、异步任务、任务状态和队列执行形态；但它仍应保留升级路径，避免把简单服务强行做成任务平台。

## 判断标准

FastAPI 独立服务接口与任务架构是否成立，只看一个标准：

**新增一个小型服务时，读者是否能快速判断同步请求、同步 batch、异步任务、任务模型和队列执行形态，并知道什么时候保持单步骤任务、什么时候升级到工作流。**

如果同步接口可能长时间阻塞，或同步 batch 没有规模和超时边界，说明 `Interface` 或 `Batch` 没完成。

如果异步任务入口返回了 `job_id`，但状态、进度、结果和失败明细没有稳定关系，说明 `Task` 没完成。

如果任务只有一个执行阶段，却被强行套入 Canvas 和步骤表，说明 `Queue` 与 `Workflow` 边界没完成。

如果多个小型服务的接口路径、状态命名、结果查询、前端获取结果策略和失败表达各不相同，说明还没有沉淀为可复用设计契约。

## Few-shot Learning 示例组

### 正例：轻量翻译服务

一个翻译服务同时支持小规模即时翻译和大规模批量翻译。合适做法是保留同步接口处理受限规模的即时请求和同步 batch，用异步 job 提交大任务，并提供状态查询和结果查询。

判断：

```text
Scope：独立翻译能力服务
Interface：sync 直接返回，jobs 返回 job_id
Batch：小规模同步 batch，大规模 batch 进入异步 job
Task：状态机和终态语义读取通用 Job 规则，结果层表达成功失败计数
Queue：普通执行器 task 足够
Workflow：暂不需要 Canvas
Contract：可沉淀为小型服务默认接口契约
```

### 反例：所有异步任务默认 Canvas

一个服务只有一次文件转换或一次 LLM 调用，却一开始引入工作流编排、步骤状态表、汇聚回调和步骤级实时进度。

问题不在于 Canvas 不强，而在于任务没有多步骤、并行汇聚或步骤级可观测需求。此时统一接口和任务状态更重要，普通单步骤队列任务更合适。

### 边界例：多步骤生成平台

如果任务需要先分析输入，再并行生成多个结果，最后汇总并实时推送进度，就已经超过普通单步骤异步任务。此时应升级为 workflow 与步骤级状态，而不是继续把多个阶段塞进一个长函数。

### 边界例：同步 batch 与异步 job

一个服务支持一次提交 10 条文本并立即返回翻译结果，也支持一次提交 10 万条文本并后台处理。前者是同步 batch，前端只需要一次 loading 和部分失败展示；后者是异步 job，前端需要保存 `job_id`，再通过轮询、SSE、Webhook 或其他策略获取状态和结果。

判断重点不是是否叫 `batch`，而是规模、耗时、结果体、前端体验和是否需要持久化任务。

### 边界例：异步 job 与实时反馈

一个生成服务需要后台执行较长任务，同时前端希望实时看到进度。此时不应直接把任务改成 WebSocket 服务，也不应因为有 SSE 就取消任务模型。更合适的判断是：任务生命周期仍由通用 Job 状态和结果查询承载；实时反馈可以作为 SSE、流式响应或 WebSocket 分支补充，用来改善前端感知。

如果实时通道断开后仍需要恢复状态，说明任务模型仍然是主契约；如果连接本身就是业务会话的一部分，才考虑把 WebSocket 升为核心交互方式。

## 误用

FastAPI 独立服务接口与任务架构不是 FastAPI 教程。

不要解释路由、依赖注入、Pydantic、执行器或 workflow 的基础概念。只有当它们影响同步/异步边界、任务模型或工作流升级判断时，才进入方法论。

它也不是项目脚手架模板。

目录结构、文件名和接口路径可以作为设计契约沉淀，但方法论本身不强制所有项目照抄同一套文件树。

它也不是要求所有异步任务都使用同一个执行器 workflow。

Celery、Taskiq 或其他执行器可以作为异步任务基础设施；workflow 只在多步骤、并行汇聚或步骤级可观测时成为合适升级形态。

它也不是大型任务平台设计。

如果项目已经进入多租户任务中心、跨服务工作流、复杂调度、分布式锁、配额计费或生产级任务运维，应进入更专门的任务平台或运维架构方法。

它也不是前后端交互模式总览。

同步请求、异步任务、流式响应、轮询、Webhook、SSE 和 WebSocket 会影响接口与任务契约，但本方法论只吸收它们对 FastAPI 独立服务架构的判断影响；完整交互模式应由专门的接口交互方法论承载。

## 适用边界

这套方法适合处理小型 FastAPI 独立服务，尤其是：

```text
服务围绕单一能力对外提供接口
同时存在即时请求和批量请求
批量请求可能需要后台执行
需要统一 job_id、状态查询和结果查询
希望多个小型服务保持一致的任务队列设计
需要保留从单步骤任务升级到多步骤工作流的路径
```

这套方法不适合处理纯同步 CRUD 后台、一次性脚本、没有服务边界的实验项目、完整微服务治理、复杂工作流平台、生产运维调度系统或具体框架使用教程。

如果问题是“项目如何选择本地开发、Dockerfile、compose 和配置加载方式”，应使用项目部署入口与配置加载方法论。

如果问题是“常用启动、验证、生成、清理和构建动作是否需要统一入口”，应使用项目开发入口设计方法论。
