# 项目入口规则

项目入口负责让开发者和 Agent 不读实现也能知道如何启动、验证、停止和排障。入口规则定义操作契约，不定义脚本目录拓扑，也不替代部署形态文档。

## 入口选择

按项目复杂度选择入口：

| 场景 | 推荐入口 |
| --- | --- |
| 只有少量稳定命令 | README 命令说明或包管理脚本 |
| 有生成、测试、构建、清理等重复动作 | `scripts/dev.sh` 或等价入口 |
| 有 API、worker、scheduler、前端等后台服务 | 服务生命周期脚本 |
| 有测试/预发发布链路 | 独立发布脚本或部署入口 |

不要把本地开发、生产发布、旧工程清理、数据库重置和远程环境操作塞进同一个入口。

当脚本数量增长时，入口脚本应保持为稳定门面，具体实现下沉到职责明确的子目录。脚本拆分规则见 [`script-topology.md`](script-topology.md)。进入 Pod、容器或远端环境后执行的只读排障命令，读取 [`runtime-troubleshooting.md`](runtime-troubleshooting.md)。

## 命令契约

入口脚本必须说明：

- 命令名称和执行位置。
- 每个命令的成功标准。
- 会修改哪些文件、缓存、依赖或外部状态。
- 不管理哪些旧工程、生产配置、外部依赖或远程资源。
- 失败后应查看哪个日志、状态或验证入口。

脚本至少支持 `help` 或无参数帮助。危险命令必须显式命名，例如 `clean-cache`、`reset-local-db`，不要隐藏在默认启动流程里。

## 服务生命周期

当项目有后台服务时，`start / stop / restart / status / logs` 必须围绕服务清单组织。

每个服务应输出：

- 服务名和用途。
- PID 文件位置。
- 日志文件位置。
- 端口和访问 URL。
- 健康检查或状态探测入口。
- 外部依赖状态摘要。

具体技术栈的排障信号由对应专项规则定义。FastAPI 服务读取 `../fastapi/observability/logging.md`，部署后验收读取 `../deployment/service-deployment.md`，运行时诊断入口读取 [`runtime-troubleshooting.md`](runtime-troubleshooting.md)。

## 保护边界

入口默认不得触碰：

- 生产环境。
- 旧工程目录。
- 远程数据库。
- 未经确认的删除、重置、覆盖动作。
- 脚本职责外的外部依赖启动和停止。

需要状态改变的命令，应在输出中明确副作用。失败时不要吞错，应快速暴露并提示排查入口。
