# Shell 脚本规则

Shell 脚本应优先面向可重复执行、可理解、可排错、可迁移的自动化场景设计，而不是只满足一次性命令封装。

## 适用边界

- 本规则默认面向 Linux 环境和 Bash 脚本。
- 如果脚本需要兼容 macOS、POSIX sh、zsh 或其他运行环境，必须在脚本帮助信息或注释中明确说明。
- 复杂脚本应在顶部注释或帮助信息中声明 Bash 版本要求，例如 `Requires: Bash 4.x+`。
- 脚本可以使用 Linux 常见命令和 Bash 能力；依赖的非系统默认命令应在帮助信息或脚本顶部注释中列出，并在运行时通过 `command -v` 检查。

## 基本要求

- 脚本应使用明确的运行时参数承接输入，不应把目标路径、模式开关或危险操作硬编码在脚本主体中。
- 面向复用的脚本必须支持 `-h` 和 `--help`，并在帮助信息中说明用途、参数、示例和执行结果。
- 参数名、命令名、路径、环境变量、文件名、exit code、stderr、stdout 等技术字面量必须保留英文，不要翻译成中文。
- 面向中文使用者的脚本，可以使用中文注释、中文帮助信息和中文终端输出，以降低理解成本。
- 脚本输出应可扫描：关键结果、写入路径、后续动作、错误原因要分层清楚，不要把调试信息和用户操作提示混在一起。
- 脚本如果设计为可重复执行，应在帮助信息中说明是否幂等，以及重复执行会覆盖、跳过还是追加已有结果。

## 参数设计

- 常用参数应同时支持短参数和长参数，例如 `-h` / `--help`、`-t` / `--target`。
- 布尔开关使用 `--all`、`--dry-run`、`--force` 等明确语义，不使用含糊缩写。
- 需要多个输入时，应支持重复值或列表值，例如 `--rules git.md testing.md`。
- 参数应承接本次执行的选择，而不是强迫用户重复输入长期固定配置；长期固定配置应进入配置区或外部配置来源。
- 不传必要参数时，可以进入交互输入；但非交互场景必须能通过参数完整执行。
- 检测到非 TTY 环境时，脚本不应尝试读取交互输入，应直接报错并提示用户通过参数传入，例如 `[ ! -t 0 ]`。
- 互斥参数必须显式校验，例如 `--all` 和 `--rules` 不应同时生效。
- 缺少必要参数、未知参数、非法路径、非法选择都应快速失败并输出明确错误。

## 配置区规则

脚本如果需要支持多个长期固定的运行环境，例如 `local`、`dev`、`test`，可以在脚本顶部设立独立配置区，集中维护各环境对应的 URL、namespace、默认开关或其他固定参数。用户通过运行时参数传入环境标识，例如 `--env dev`，脚本内部通过查表获取对应配置，避免每次执行都手动输入全部参数。

配置区应遵守：

- 配置区应与主逻辑物理隔离，并通过注释边界明确标识。
- 配置区只承载长期固定、低频修改的配置映射，不承载本次执行才决定的临时输入。
- 配置区的修改不应影响脚本主逻辑；主逻辑应只通过环境标识读取配置。
- 脚本应在启动时校验所选环境是否存在；不存在时快速失败，并列出可用环境。
- 生产环境密钥、真实 token、生产数据库密码等敏感配置不应写入脚本配置区。
- 如果脚本会进入版本控制，配置区中不应包含真实密钥；可以使用占位符（例如 `REPLACE_ME`），也可以保存环境变量名并通过间接引用读取真实值。
- 本地开发或临时脚本确需放置低风险测试密钥时，应明确标注用途、环境和不可复用边界，避免被误用到生产环境。

配置区示例：

```bash
# ============================================================
# 环境配置 - 只修改本区配置，不修改下方脚本逻辑
# ============================================================

declare -A ENV_API_URL=(
  [local]="http://localhost:8080"
  [dev]="https://dev.example.com"
  [test]="https://test.example.com"
)

# 真实 token 不应进入版本控制；通过环境变量或 Secret Manager 注入。
declare -A ENV_TOKEN_NAME=(
  [local]="LOCAL_API_TOKEN"
  [dev]="DEV_API_TOKEN"
  [test]="TEST_API_TOKEN"
)

# ============================================================
# 以下为脚本逻辑，通常不需要修改
# ============================================================

# 示例省略了环境存在性和必填环境变量校验；完整脚本应在读取前完成校验。
# 通过配置区中的变量名间接读取环境变量值。
token_var="${ENV_TOKEN_NAME[$ENV]}"
token="${!token_var}"
```

## 环境变量规则

脚本可以通过环境变量读取敏感值、CI 注入值或机器本地差异配置。环境变量适合承载不应写入脚本主体、配置区或版本控制的值。

环境变量使用应遵守：

- 帮助信息必须列出脚本会读取哪些环境变量，并说明每个变量的用途、是否必填、适用环境。
- 敏感环境变量只说明变量名和用途，不在日志、帮助信息、错误信息或调试输出中打印真实值。
- 必填环境变量应在主流程开始前集中校验；缺失时快速失败，并提示缺少的变量名。
- 可选环境变量应说明默认行为；不要用静默默认值掩盖配置错误。
- 如果环境变量名称来自配置区映射，脚本应先校验环境存在，再按映射读取对应变量。
- 使用间接引用读取环境变量时，若变量未设置，`set -u` 会在读取时直接报错退出；必填环境变量的存在性校验应在间接引用之前完成。
- `.env` 文件、Secret Manager、CI secret 等外部来源可以作为环境变量的注入方式；脚本不应自行猜测多个隐式注入来源，帮助信息应明确说明期望的注入方式，例如通过 `export` 设置环境变量或通过指定 `.env` 文件加载。

## 帮助信息

`-h` / `--help` 至少应包含：

- 脚本用途：这个脚本解决什么问题。
- 运行环境：例如 Linux、Bash，以及必要的外部命令依赖。
- 用法示例：展示最常用的 2-4 个命令。
- 参数说明：每个参数的含义、是否必填、输入格式。
- 环境变量：列出会读取的环境变量、用途、是否必填，以及敏感值注入方式。
- 执行结果：脚本会创建、复制、修改或输出什么。
- 交互模式：如果支持交互选择，应说明可输入的格式。
- 幂等性和副作用：说明重复执行、覆盖文件、删除文件或修改外部资源的行为。

帮助信息应面向真实使用流程组织，不只罗列参数。

### 子命令型脚本帮助信息

带子命令的脚本，例如 `jobs.sh summary`、`jobs.sh inspect <id>`，
顶层 `-h` / `--help` 应提供完整入口说明，而不是只输出参数解析器生成的
usage 和选项列表。

推荐大纲：

```text
用法
简介
作用域
命令
选项
环境变量
输出
常用示例
保护边界
Exit Codes
```

各章节要求：

- 用法：展示顶层调用形式，例如 `jobs.sh [-h] {summary,inspect,events} ...`。
- 简介：用一句话说明脚本职责，例如 `Job 只读查询与排障入口。`。
- 作用域：说明脚本能做什么、不能做什么；查询、排障、部署、修复类脚本应明确副作用边界。
- 命令：列出所有子命令及一句话说明；子命令专属选项放到对应子命令的帮助信息中。
- 选项：列出顶层选项，例如 `-h` / `--help`、`--env`、`--json` 等全局选项。
- 环境变量：列出脚本会读取的环境变量；没有环境变量依赖时可以省略本节。
- 输出：说明 stdout、stderr、表格输出、JSON 输出、大字段截断等行为。
- 常用示例：给出 2-6 个真实高频命令，优先覆盖常见成功路径和排障入口。
- 保护边界：说明只读、dry-run、不会修改状态、不会删除资源、不会触发重试等安全边界。
- Exit Codes：需要被 CI、自动化或排障流程调用的脚本，应说明主要退出码。

章节可以按脚本复杂度裁剪：

- 简单脚本通常保留 `用法`、`简介`、`选项`、`常用示例`。
- 子命令脚本应增加 `命令`。
- 查询和排障脚本应增加 `作用域`、`输出`、`保护边界`。
- 有副作用脚本必须说明 `保护边界`、`幂等性` 和 `Exit Codes`。
- 依赖环境变量的脚本必须说明 `环境变量`。

如果参数解析器自动生成了 `命令` 区，业务扩展说明不应再次使用同名标题造成
重复；可以使用 `命令说明`、`排障入口` 或更具体的标题承接补充说明。

带子命令的脚本在未传 command 时不应隐式执行默认子命令；应快速失败，
输出简短 usage、缺失的 command 名称或错误原因，并返回非 0 exit code。
`-h` / `--help` 应输出完整帮助到 stdout 并返回 0；缺少 command、未知
command、非法参数等错误路径应输出到 stderr 并返回非 0。

帮助信息模板：

```text
Usage:
  script.sh --target PATH [--dry-run] [--force]

Purpose:
  说明脚本解决的问题。

Environment:
  Requires: Linux, Bash 4.x+
  Dependencies: rsync, jq

Options:
  -t, --target PATH   必填，目标目录
  --dry-run           只展示将执行的操作
  --force             允许覆盖已有文件
  -h, --help          显示帮助信息

Environment Variables:
  API_TOKEN           必填，用于访问 API；不要打印真实值

Output:
  stdout: 正常结果、生成路径、可复制内容
  stderr: 错误原因、缺失参数、非法状态

Idempotency:
  重复执行会覆盖 / 跳过 / 追加哪些内容。

Exit Codes:
  0  成功
  1  参数或配置错误
  2  外部依赖缺失
  3  执行失败
```

## 脚本注释

- 顶部注释应说明脚本职责、运行环境、外部依赖和危险副作用。
- 配置区注释应说明哪些配置可以修改、哪些配置不应修改，以及配置变更的影响范围。
- 函数注释应说明输入、输出、副作用和失败条件；不需要为简单函数强行添加注释。
- 复杂判断前应解释业务意图、输入边界或状态假设，不要逐行复述命令行为。
- 涉及删除、覆盖、远程调用、生产资源修改等高风险操作的代码块，必须用注释说明影响范围和触发条件。
- 注释应跟随代码维护；如果注释与实现不一致，应优先修正注释或删除误导性注释。

## 输出规则

- 成功输出应明确说明实际完成了什么，例如复制了哪些文件、写入了哪个目录、下一步做什么。
- 可复制内容应使用清晰边界标记，例如 `----- Copy the following ... -----` 和 `----- End -----`。
- 错误输出应写到 stderr，并包含可定位原因。
- 输出中可以使用中文解释动作，但命令、参数、路径、文件名、Markdown 标题等技术对象保持原文。
- 不要静默吞错；脚本失败时应让错误暴露，并返回非 0 exit code。
- 如果支持 `--dry-run`，应输出将要执行的操作、目标路径和影响范围，而不是静默退出。
- 如果支持调试模式，应通过 `--verbose`、`--debug` 或 `DEBUG=1` 显式开启，不应默认输出内部执行细节。

## 实现约束

- 默认使用 `set -euo pipefail`，避免未定义变量、管道失败或中间命令失败被忽略。
- 路径应基于脚本自身位置解析，避免依赖调用者当前工作目录。
- 推荐使用以下方式定位脚本目录：

```bash
SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
```

- 写文件前应明确目标路径；必要时创建目录，但不要擅自创建无关目录结构。
- 复制、覆盖、删除等有副作用操作，应在输出中明确说明影响范围。
- 脚本如果产生临时文件、锁文件或占用外部资源，应使用 `trap` 注册清理逻辑，确保中断或异常退出后不留残余。
- 不为了“更稳”添加 silent fallback、默认吞错或自动降级逻辑；异常应快速暴露。
- 脚本如果不支持并发执行，应在启动时检查锁文件或进程；发现冲突时应快速失败并提示，不要静默等待、覆盖已有进程或继续执行。
- 依赖外部命令时，应在主流程开始前集中检查；缺失依赖应快速失败并说明缺少哪个命令。
- 脚本启动阶段建议按以下顺序执行：解析参数 → 校验参数合法性与互斥关系 → 检查外部命令依赖 → 校验配置区环境标识 → 校验必填环境变量 → 进入主流程。每一步失败都应快速退出，不进入下一阶段。

## 验证要求

声明脚本完成前，至少验证：

- `bash -n script.sh` 语法检查通过。
- 推荐使用 `shellcheck script.sh` 补充静态分析，发现 `bash -n` 无法检测的引用、展开和管道问题。
- `script.sh -h` 和 `script.sh --help` 输出正常。
- 关键主流程能在临时目录或测试目标中跑通。
- 如果脚本支持 `--env`，应验证传入未定义环境名时快速失败，并输出可用环境列表。
- 错误参数或缺失参数能按预期失败，且返回非 0 exit code。
- 错误信息写入 stderr，正常结果写入 stdout。

错误路径验证可使用以下形式：

```bash
out_file="$(mktemp)"
err_file="$(mktemp)"

if script.sh --invalid-arg >"$out_file" 2>"$err_file"; then
  echo "Expected command to fail" >&2
  exit 1
fi

test ! -s "$out_file"
test -s "$err_file"
```
