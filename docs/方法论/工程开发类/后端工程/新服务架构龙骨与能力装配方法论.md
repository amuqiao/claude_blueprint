# 新服务架构龙骨与能力装配方法论

> 面向从 0-1 构建新服务的场景，先建立稳定的服务龙骨，再按需求触发能力积木，并根据架构权重决定前置、后置和沉淀方式。

```text
新服务架构龙骨与能力装配
├── Keel：服务龙骨
│   └── 定住服务定位、能力边界、接口入口、配置入口、运行入口、验证入口和文档入口
├── Layer：架构分层
│   └── 根据服务复杂度建立合适分层和目录结构，避免屎山，也避免过度 DDD
├── Block：能力积木
│   └── 识别接口、任务、数据、缓存、存储、前端、安全、部署、评测等可装配能力
├── Trigger：触发条件
│   └── 用需求信号决定是否引入某个能力，而不是按技术偏好或模板硬套
├── Weight：架构权重
│   └── 区分骨架级、结构级和增强级能力，决定哪些必须前置，哪些可以后置
├── Order：装配顺序
│   └── 按龙骨、接口、状态、数据、任务、基础设施、部署、安全和上线增强逐步装配
├── Draft：架构草稿演进
│   └── 先写骨架稿，再随开发补成成型稿和维护稿，避免一次性空泛大设计
├── Contract：项目契约
│   └── 把稳定装配结果回写为 README、API、架构、部署、验证和模块设计文档
├── Evolve：方法论树迭代
│   └── 将反复出现的能力节点沉淀为枝叶方法论，并回挂到主干
└── Boundary：边界控制
    └── 防止所有服务都套完整平台，也防止关键结构能力后置过久
```

## 核心命题

新服务架构龙骨与能力装配方法论处理的不是“如何写某个框架的代码”，而是“拿到一个需求后，如何从 0-1 构建一个不乱、不重、可迭代的新服务”。

很多新服务的失败不在于技术不会用，而在于起盘时没有稳定的装配模型：

```text
服务边界没定住就开始写接口
接口形态没定住就引入队列或数据库
目录结构随手增长，后续变成屎山
轻量服务过早套完整 DDD 或平台架构
需要任务生命周期时仍用长同步请求硬撑
每个项目的部署、配置、验证方式都不一样
架构文档一开始写得很满，但实现后很快漂移
```

这套方法论的任务，是提供一套“服务龙骨 + 能力积木 + 装配”的心智模型：服务龙骨先定住，能力积木按需求触发，积木有架构权重和装配顺序，稳定范式再沉淀成方法论文档树。

它的核心转化是：

```text
需求 -> 服务龙骨 -> 架构分层 -> 能力积木 -> 触发判断 -> 权重排序 -> 渐进装配 -> 项目契约 -> 方法论回挂
```

## 心智模型：服务龙骨 + 能力积木 + 装配

本文用“服务龙骨 + 能力积木 + 装配”理解新服务的 0-1 构建。

```text
服务龙骨
└── 任何服务都必须先定住的基础工程骨架

能力积木
└── 可以按需求挂载的工程能力模块

装配
└── 根据触发条件、架构权重和装配顺序，把能力积木接到服务龙骨上
```

后续节点统一使用以下表达：

| 术语 | 含义 | 对应节点 |
| --- | --- | --- |
| 服务龙骨 | 服务定位、能力边界、接口、配置、运行、验证和文档入口 | `Keel` |
| 架构分层 | 服务内部职责边界和目录组织原则 | `Layer` |
| 能力积木 | 接口、任务、数据、缓存、存储、前端、安全、部署等可装配能力 | `Block` |
| 触发条件 | 引入某个能力积木的需求信号 | `Trigger` |
| 架构权重 | 能力对架构的影响程度，用于判断前置或后置 | `Weight` |
| 装配顺序 | 多个能力积木挂载到服务龙骨上的推荐先后关系 | `Order` |
| 架构草稿演进 | 架构文档从骨架稿到成型稿、维护稿的渐进过程 | `Draft` |
| 项目契约 | 稳定装配结果沉淀出的文档、接口、部署和验证规则 | `Contract` |
| 方法论树 | 主干方法论和枝叶方法论的维护关系 | `Evolve` |

## 方法论定位

本文适合在“准备新建一个服务或把脚本升级为服务”时使用。

本文主要面向 API 服务、后端能力服务和可服务化的全栈项目。纯客户端应用应使用客户端工程方法论，不应把客户端项目强行套入新服务装配模型。

它和全栈项目开发主干方法论的关系是：

```text
全栈项目开发主干方法论
└── 更关注项目推进主干：定盘、架构、切片、契约、基建、实现、验证、治理

新服务架构龙骨与能力装配方法论
└── 更关注新服务起盘模型：服务龙骨、架构分层、能力积木、触发条件、架构权重和装配顺序
```

本文不替代具体枝叶方法论。部署、接口任务、Flutter、发布手册等具体范式仍由对应方法论承载。

## 方法论节点

### Keel：服务龙骨

先建立所有新服务都应该具备的基础龙骨。

龙骨不是重型架构，而是防止项目一开始就散掉的最小工程骨架。即使服务很小，也应该能回答这些问题：

| 龙骨项 | 要回答的问题 | 最小产物 |
| --- | --- | --- |
| 服务定位 | 这个服务解决什么问题，不解决什么问题 | README 中的定位说明 |
| 能力边界 | 对外提供什么能力，调用方是谁 | 能力清单和不做清单 |
| 接口入口 | 调用方如何进入能力 | API、CLI、文件输入、前端入口或内部函数入口 |
| 配置入口 | 配置从哪里来，默认值和运行时覆盖如何区分 | `.env.example`、配置说明或默认配置 |
| 运行入口 | 本地怎么跑，依赖服务怎么跑，容器怎么跑 | 启动命令、脚本或 compose 说明 |
| 验证入口 | 怎么证明主路径成立 | curl、脚本、评测命令、测试或冒烟路径 |
| 文档入口 | 后续从哪里理解和维护项目 | README、API 文档、架构说明或部署说明 |

龙骨的目标不是一次性完整，而是让项目从第一天起有稳定方向。

如果一个服务无法说清定位、入口、配置、运行和验证方式，就不应急着讨论目录结构、队列、数据库或前端。

### Layer：架构分层

根据服务复杂度建立合适的架构分层和目录结构。

分层的目标是防止职责混乱，而不是追求形式完整。目录结构是分层原则的物理投影，不是独立的审美选择。

可以按服务复杂度渐进分层：

| 服务形态 | 合适分层 | 判断重点 |
| --- | --- | --- |
| 轻量服务 | 入口、配置、核心逻辑、验证脚本 | 不让所有逻辑挤在一个不可维护文件里，但不强行拆过多层 |
| 中型后端服务 | API、schemas、services、repositories、db、core、tasks | API 不写业务，Service 不感知 HTTP，Repository 封装数据访问 |
| 完整平台 | frontend、backend、pipeline、repositories、infrastructure、core、models、schemas | 前后端、任务、共享基础设施、外部服务和功能模块边界清楚 |

分层判断应关注调用方向和职责边界：

```text
入口层负责接收请求、参数校验和委托
Service 层负责业务编排，不直接感知 HTTP 细节
Repository 层负责数据访问，不包含业务判断
Infrastructure 层负责数据库、Redis、存储、外部 API、模型网关等基础设施
Models 层承载 ORM 或数据模型
Schemas 层承载跨层请求响应和数据契约
Tasks / Pipeline 层承载后台任务入口、编排和 worker 执行边界
```

不要在小服务中强行引入 DDD、聚合根、领域事件和复杂模块边界。也不要在服务已经出现多个接口、数据库、任务、缓存和外部调用后，继续把所有逻辑堆在 `main.py` 或单个 service 文件里。

### Block：能力积木

识别新服务可能装配的能力积木。

能力积木不是技术清单，而是会改变服务形态、开发注意力或项目契约的能力节点。

下表是架构选项提示表，不是完整技术清单，也不是工具教程。它的作用是暴露高频架构选项，让开发者和 Agent 在起盘时不会遗漏会显著影响架构的能力。

常见能力积木包括：

| 能力积木 | 关注点 |
| --- | --- |
| 同步接口 | 请求响应、超时、错误结构、输入输出契约 |
| 同步 batch | 输入规模、部分失败、结果顺序、性能和失败明细 |
| 异步 job | task_id、状态查询、结果查询、进度和失败恢复 |
| 任务队列 / 工作流编排 | worker、broker、并发、重试、普通 task、Canvas / chain / group / chord、步骤状态、结果汇聚和部署边界 |
| 数据库 / 数据模型 | 数据模型、事务、查询、历史、审计、持久化和字段演进 |
| Alembic 迁移 | 数据库版本管理、迁移顺序、环境一致性和发布回滚风险 |
| Repository | 数据访问封装、ORM 边界、测试和服务层解耦 |
| 缓存 | key 设计、TTL、失效策略、一致性和成本优化 |
| 文件存储 | 上传、产物、路径、生命周期、下载和部署挂载 |
| 前端界面 | 用户流程、状态、API 契约、构建和联调 |
| 进度感知 | 轮询、SSE、Webhook、WebSocket、断线恢复 |
| 基础设施服务 | DB、Redis、对象存储、模型网关、第三方 API |
| 认证鉴权 | X-API-Key、JWT、多用户、权限、对外接入和错误暴露边界 |
| 限流与安全 | 滑动窗口、配额、防滥用、日志和错误暴露 |
| 部署配置 | 本地、Dockerfile、compose、env、健康检查和验证 |
| 评测与验收 | 测试数据、冒烟脚本、性能指标和人工复核 |
| 发布维护 | 发布手册、回滚、备份、监控、排障和复盘 |

能力积木可以组合，但不应一次性全上。装配前必须先看触发条件和架构权重。

### Trigger：触发条件

用需求信号决定是否引入某个能力。

能力积木不应该由偏好触发，也不应该因为某个项目用过就默认照搬。

触发条件表同样是架构选项提示表。它用于帮助开发者判断“是否应该提醒 Agent 考虑某个能力”，不替代对应能力的具体范式。

常见触发条件：

| 需求信号 | 触发的能力 |
| --- | --- |
| 请求耗时短、结果小、调用方可等待 | 同步接口 |
| 一次处理多条输入，但仍能在可控时间内返回 | 同步 batch |
| 耗时不可控、大批量、需要进度或结果恢复 | 异步 job |
| 需要后台排队、并发控制、重试、步骤编排或结果汇聚 | 任务队列 / 工作流编排 |
| 需要保存任务、结果、历史、用户、审计或配置 | 数据库 |
| 数据库字段会随版本长期演进 | Alembic 迁移 |
| Service 层开始直接写 SQL，或 ORM 对象泄漏到上层 | Repository |
| 外部调用成本高、重复输入多、结果可复用 | 缓存 |
| 输入或输出包含图片、文件、压缩包、大结果 | 文件存储 |
| 用户需要人工操作、多步骤提交、任务查看或结果管理 | 前端界面 |
| 前端需要持续看到任务进度、日志或增量结果 | 进度感知 |
| 多个模块共享 DB、Redis、Storage、AI client 或外部 API | 基础设施层 |
| 对外开放、多调用方、多用户或生产环境 | 认证鉴权、限流、安全日志 |
| 服务需要交付给别人运行或上线 | 部署配置、发布维护 |
| 算法或规则容易误判，需要持续调参 | 评测与验收 |

触发条件应尽量写进架构文档或模块设计文档，避免后续开发者不知道某个能力为什么存在。

### Weight：架构权重

区分能力积木对架构的影响程度。

同一个能力在不同项目中权重可能不同。安全、限流、缓存不是永远后置；当项目对外、生产化或成本敏感时，它们会升级。

可以先分为三类：

| 权重 | 含义 | 示例 |
| --- | --- | --- |
| 骨架级 | 不做就无法稳定起盘 | 服务定位、接口入口、配置入口、运行入口、验证入口、最小文档入口 |
| 结构级 | 一旦触发，会改变接口、目录、数据、部署或验证方式 | 分层架构、数据库、Repository、Alembic、异步 job、任务队列、文件存储、前端、基础设施层 |
| 增强级 | 重要但不一定阻塞 0-1 起盘，通常提升安全、质量、运维或上线成熟度 | JWT、X-API-Key、限流、审计、监控、发布手册、管理后台 |

权重不是固定标签，而是由场景决定：

```text
内部临时服务中的 X-API-Key 可以是增强级
对外 API 中的 X-API-Key 或 JWT 是结构级
本地脚本中的缓存可以是实现细节
高成本模型服务中的缓存可能是结构级
原型服务中的发布手册可以后置
上线服务中的发布手册和回滚策略应前置
```

判断权重的核心问题是：这个能力如果晚加，会不会导致接口、数据、目录、部署或用户路径大规模返工。

### Order：装配顺序

按稳定顺序装配能力，而不是想到什么加什么。

推荐顺序：

```text
1. 定服务龙骨
2. 定接口范式
3. 判断 batch、状态和任务生命周期
4. 判断数据模型、Repository、Alembic、缓存和文件存储
5. 判断任务队列、worker、进度感知和前端
6. 定配置加载、运行入口、部署形态和验证闭环
7. 补认证、限流、日志、发布手册和上线增强
8. 回写项目契约和方法论沉淀
```

部署配置比较特殊：它不是最后才考虑。

从 0-1 阶段开始，服务就需要最小运行、配置和验证入口；但生产发布、回滚、监控、备份和安全加固可以随着上线需求逐步补齐。

因此部署可以分两层：

```text
最小部署龙骨：启动、配置、健康检查、验证入口
上线部署治理：发布手册、回滚、监控、备份、安全和排障
```

### Draft：架构草稿演进

架构文档不应一次性写成“大而全”的说明书。

`Draft` 关注架构文档如何随开发逐步成型；`Contract` 关注稳定结果最终沉淀成哪些项目协作契约。二者都和文档有关，但判断重点不同。

更合适的方式是先写能压住方向的骨架稿，再随着实现、验证和复盘逐步补成成型稿和维护稿。架构文档可以分三阶段：

| 阶段 | 目标 | 内容 |
| --- | --- | --- |
| 骨架稿 | 让项目方向不乱 | 服务定位、边界、主链路、关键能力、暂不做什么 |
| 成型稿 | 让实现有边界 | 分层、接口范式、数据模型、任务模型、配置部署、验证方式 |
| 维护稿 | 让项目可持续迭代 | 否决方案、踩坑、约束、扩展点、复盘、下一阶段计划 |

骨架稿不追求完整，但必须压住方向。

成型稿不追求百科，但必须支撑实现。

维护稿不追求一次写完，而是随着开发、验证、部署和复盘不断回写。

如果架构文档一开始就写了很多细节，却没有定住服务边界、接口范式和装配顺序，说明它只是信息堆叠，不是有效的架构草稿演进。

### Contract：项目契约

把稳定装配结果沉淀为项目契约。

不同项目不需要同样多的文档，但每个项目都应该有能让后续开发继续推进的契约。

常见契约包括：

```text
README：服务定位、运行方式、验证入口
API 文档：接口路径、请求响应、错误结构、示例
架构文档：分层、模块、基础设施、关键选型和否决方案
部署文档：部署形态、配置加载、服务边界、健康检查
模块设计：核心切片、数据模型、任务模型、前后端契约
验收脚本：冒烟测试、评测数据、性能指标、人工复核入口
复盘文档：踩坑、故障、修复原则和后续治理点
```

契约的目标是：下一次新增模块、修 bug、部署或交给 Agent 继续开发时，不需要重新猜项目规则。

### Evolve：方法论树迭代

把反复出现的能力节点沉淀为枝叶方法论。

总方法论只维护龙骨、能力分类、触发条件、架构权重和装配顺序。它不应该吞掉所有枝叶细节。

当某个能力节点多次出现，并且具备稳定误区、稳定判断动作和稳定完成标准时，可以升格为独立方法论。是否值得新增，应回到方法论新增范式判断。

枝叶方法论成熟后，应回挂到主干对应节点；主干只说明何时需要它，枝叶说明如何做好它。具体挂载关系由工程开发类 `README.md` 维护，本文不维护路由表。

### Boundary：边界控制

控制装配边界，防止两个极端。

第一个极端是所有服务都套完整平台：

```text
轻量同步 API 不应默认引入前端、Celery、PostgreSQL、Repository、Alembic 和完整发布手册
```

第二个极端是关键结构能力后置过久：

```text
已经需要任务状态和结果恢复，却仍用长同步请求硬撑
已经有多张表和字段演进，却没有迁移管理
Service 层已经直接写复杂 SQL，却不抽 Repository
项目已经交付别人运行，却没有稳定配置、部署和验证入口
```

边界判断看一个问题：

**这个能力现在不引入，后面会不会造成接口、数据、目录、部署或用户路径的大规模返工。**

如果会，它通常是结构级，应前置。

如果不会，它可以先作为增强级后置，但应记录触发条件。

## 判断标准

新服务架构龙骨与能力装配是否完成，只看一个标准：

**给定一个新需求，读者或 Agent 是否能先搭出服务龙骨，再判断需要哪些能力积木、哪些必须前置、哪些可以后置，并把稳定结果沉淀为项目契约。**

如果服务定位、接口入口、配置入口、运行入口和验证入口都不清楚，说明 `Keel` 没完成。

如果目录结构只是按文件类型或个人习惯随手增长，说明 `Layer` 没完成。

如果能力积木是因为模板或偏好引入，而不是由需求信号触发，说明 `Trigger` 没完成。

如果把认证、限流、发布手册等增强能力一律当作可有可无，或把所有服务都做成完整平台，说明 `Weight` 没完成。

如果架构文档一开始很大但无法指导实现，说明 `Draft` 没完成。

## Few-shot Learning 示例组

### 正例：短文本语种符合性检测服务

一个服务只需要判断 `expected_language + text` 是否匹配，不需要历史、用户、任务队列和前端。

合适装配：

```text
Keel：服务定位、batch API、配置、运行、验证和文档入口
Layer：入口、配置、detectors、评测脚本即可
Block：同步 batch、评测脚本、轻量部署
Weight：数据库、队列、前端、复杂认证暂不引入
Contract：README、API 文档、部署说明、评测入口
```

### 正例：标签批量翻译服务

一个服务既支持小规模同步翻译，也支持大规模异步翻译，需要缓存、数据库、任务状态和结果查询。

合适装配：

```text
Keel：翻译能力边界、sync/batch/job 接口、配置、运行、验证
Layer：api、schemas、services、repositories、db、core、tasks
Block：同步接口、异步 job、数据库、Repository、缓存、Celery、部署入口
Weight：接口任务、数据库、队列、配置部署是结构级；安全接入按调用场景判断
Contract：接口文档、架构文档、配置加载规范、部署指南、验收脚本
```

### 正例：多语种海报标题生成平台

一个平台包含前端、后端、Celery 流水线、SSE、文件存储、任务历史和多功能扩展。

合适装配：

```text
Keel：平台定位、Hub + Spoke 扩展边界、任务主链路、文件产物和前端入口
Layer：frontend、backend、pipeline、repositories、infrastructure、models、schemas
Block：前端、文件存储、任务队列、步骤状态、SSE、数据库、模型网关、部署治理
Weight：多个结构级能力叠加，必须形成项目级架构契约
Contract：架构设计、模块设计、接口契约、部署手册、实施清单、专题复盘
```

### 反例：轻量服务强行 DDD

一个服务只有一个同步接口和少量纯函数逻辑，却一开始拆出 domain、application、infrastructure、repository、unit of work、领域事件和复杂聚合。

问题不是分层意识太强，而是 `Weight` 判断失败。这个服务需要清楚边界和验证入口，但不需要重型领域建模。

### 反例：中型服务继续堆 main.py

一个服务已经有多个接口、数据库、缓存、外部 API 和后台任务，但所有逻辑仍堆在入口文件或单个 service 中。

问题不是项目还小，而是结构能力已经触发。此时应引入合适分层、Repository、配置治理和任务边界。

### 边界例：认证鉴权是否前置

内部临时脚本服务可以先不做 JWT，只保留最小运行和验证入口。

但如果服务要对外开放、接入多个调用方、进入生产环境或涉及用户隔离，认证鉴权就从增强级升级为结构级，应前置到接口和错误契约设计中。

## 误用

新服务架构龙骨与能力装配不是脚手架模板。

不要把本文理解为所有项目都必须生成同一套目录、同一套技术栈和同一组文档。

它也不是 DDD 方法论。

分层是为了控制职责和演进成本，不是为了给每个小服务套复杂领域模型。

它也不是全栈平台蓝图。

`title` 这类项目可以作为复杂度上限样本，但不应成为所有服务的默认形态。

它也不是枝叶方法论合集。

本文只说明能力积木何时触发、权重如何判断、装配顺序如何安排；具体部署、接口任务、任务队列、前端开发、发布手册等细节应由对应方法论承载。

## 适用边界

这套方法适合处理：

```text
从 0-1 新建一个 API 服务、后端服务或全栈平台
把脚本或实验能力升级为可调用服务
需要根据需求复杂度决定是否引入 DB、队列、缓存、文件、前端、部署和安全
希望多个项目共享同一套起盘标准和架构装配判断
希望把重复的工程节点逐步沉淀为方法论树
```

这套方法不适合处理单个函数实现、具体框架 API 教程、纯业务需求文档、生产级运维平台设计或大型组织架构治理。
