# 项目部署入口与配置加载方法论

> 指导项目判断本地开发、独立镜像、compose 全量和 compose 依赖部署之间的边界，稳定服务管理、配置加载和验证闭环；当部署路径成熟后，再沉淀为项目级部署契约、速查说明或必要脚本。

```text
项目部署入口与配置加载
├── Shape：部署形态
│   └── 区分本地开发、Dockerfile 独立、compose 全量和 compose 依赖部署
├── Service：服务分类
│   └── 切清应用进程、依赖服务、辅助服务和一次性任务分别由谁管理
├── Config：配置加载
│   └── 固定 env 文件、compose 覆盖和运行时变量之间的优先级
├── Entry：启动入口
│   └── 统一本地脚本、镜像命令和 compose 命令的使用边界
├── Verify：部署验证
│   └── 通过配置检查、健康检查、镜像独立验证和冒烟测试确认部署成立
└── Boundary：维护边界
    └── 说明哪些部署形态、服务和脚本属于当前项目维护范围
```

## 核心命题

项目部署入口与配置加载处理的不是“怎么写 Docker 命令”，而是“一个项目应该如何稳定区分部署形态、服务边界和配置来源”。

很多项目部署混乱，不是因为缺少命令，而是本地开发、容器全量启动、只启动依赖服务、独立镜像交付和环境变量覆盖规则混在一起。开发者不知道哪些服务应该本地跑，哪些服务应该交给 compose，哪些配置来自 env 文件，哪些配置被 compose 覆盖。

这套方法论的任务，是把部署从零散命令收敛成一组稳定判断，让项目能长期维护多种启动方式，而不是每次部署都重新猜配置和服务边界。

它本身不是部署契约文档模板，也不默认要求项目产出总控脚本。项目尚未成型时，它用于辅助判断部署形态和配置边界；项目部署路径稳定后，才适合把判断结果沉淀为项目级部署契约文档、速查说明或必要脚本。

它的核心转化是：

```text
零散部署命令 -> 部署形态判断 -> 服务边界切分 -> 配置优先级固定 -> 入口边界明确 -> 部署验证闭环
```

## 方法论节点

项目部署入口与配置加载不是 DevOps 大全。它只处理常见项目在本地开发和基础容器化部署中反复出现的入口、服务和配置问题。

### Shape：部署形态

先判断项目当前维护哪些部署形态。

常见形态通常只有四类：本地开发启动、Dockerfile 独立部署、docker compose 全量部署、docker compose 依赖部署。

本地开发启动适合让前端、后端、worker、scheduler 等应用进程运行在宿主机，方便调试、热更新和查看日志。compose 依赖部署只启动数据库、缓存、消息队列等依赖服务。compose 全量部署由 compose 管理应用和依赖服务。Dockerfile 独立部署只交付单个应用镜像，依赖服务由外部提供。

形态判断的重点不是覆盖所有部署可能性，而是明确当前项目真正维护哪些方式。没有验收过的方式不应作为已支持方案沉淀进项目级部署契约。

### Service：服务分类

切清每类服务由谁启动、谁停止、谁验证。

应用进程、依赖服务、辅助服务和一次性任务不应混成同一种启动对象。应用进程通常需要本地开发入口或应用镜像；依赖服务更适合 compose 管理；辅助服务按是否必须参与主链路决定；迁移、初始化、构建和验证这类一次性任务应显式执行，不应隐式塞进默认启动。

服务分类完成后，读者应能看清：本地开发时哪些服务跑宿主机，哪些服务跑容器；compose 全量时哪些服务同属一个编排边界；独立镜像部署时哪些依赖必须外部提供。

### Config：配置加载

固定配置来源和覆盖优先级。

配置文件负责表达环境差异，compose environment 只负责运行形态导致的必要覆盖。不要让同一配置在多个 env 文件和多个 yml 文件之间形成多对多关系。

推荐优先级是：

```text
运行时显式环境变量
> docker-compose.yml environment
> ENV_FILE 指定的 env 文件
> .env
> 应用默认值
```

在 compose 视角下，可以简化理解为：

```text
docker-compose.yml environment
> ${ENV_FILE} 指定的 env 文件
> .env
```

其中，compose environment 应只覆盖容器网络地址、容器内部端口或运行形态必须不同的值。业务配置、密钥、限流、模型、第三方服务参数等应来自 env 文件或运行时注入。

配置治理必须同时回答：哪个文件是模板，哪个文件是真实配置，哪个配置来源会进入运行时，哪些内容不能进入镜像构建上下文。镜像不应携带真实配置，运行时配置应由 env 文件、compose 或运行时环境注入。

### Entry：启动入口

统一每种部署形态的入口命令。

本地开发入口应说明它管理哪些应用进程、不管理哪些依赖服务。compose 依赖部署应明确只启动依赖服务。compose 全量部署应明确服务都在同一个编排边界内。Dockerfile 独立部署应明确镜像只包含应用服务，依赖由外部提供。

入口命令的目标不是隐藏所有复杂度，而是让开发者知道当前使用的是哪种部署形态，以及这条命令会启动、停止或修改哪些对象。

入口是否需要落成脚本，应回到项目开发入口设计方法论判断。部署方法论负责说明入口边界，脚本只是其中一种实现形态，不是默认产物。

### Verify：部署验证

为每种部署形态提供最小验证闭环。

部署验证应覆盖配置是否可用、服务是否启动、健康检查是否通过、关键接口是否可访问，以及 Dockerfile 是否能脱离 compose 独立运行。

当验证动作具有长期复用价值时，可以沉淀为少量维护入口，例如配置检查、镜像独立验证和冒烟测试。不要把一次性排查命令都放进 `scripts/`，也不要把验证脚本写成部署教程。

### Boundary：维护边界

说明当前项目维护哪些部署方式，不维护哪些部署方式。

如果当前项目只验收了本地开发、compose 全量和 Dockerfile 独立部署，就不要把生产 compose、K8s、云平台 Secrets 或 CI/CD 流水线写成已支持方案。相邻方案可以记录为未来方向，但不应污染当前部署入口。

当项目需要沉淀部署契约时，维护边界应明确主文档、速查文档、配置规范和脚本目录各自承担什么职责，避免多个文档重复解释同一部署形态。项目尚未成型时，可以先保留判断结论，不必过早拆出多份文档。

维护边界也应覆盖忽略规则：版本库和镜像构建上下文都应排除真实配置、本地状态和运行产物，避免配置治理只停留在运行时加载层面。

当部署路径已经稳定，并且需要面向多人维护、日常发布、按变更类型操作、备份恢复或故障排查沉淀项目级参考手册时，应进入项目部署文档与发布手册方法论。部署手册不仅要记录部署形态和命令，还应提供面向维护任务的直达入口，让开发者按当前任务、变更类型或故障现象找到对应操作。

## 判断标准

项目部署入口与配置加载是否完成，只看一个标准：

**读者是否能根据当前场景选择正确部署形态，并清楚知道服务由谁管理、配置从哪里来、哪些值会被覆盖、如何验证部署成功。**

如果读者仍然需要在多个 README、compose 文件、env 文件和脚本之间猜启动方式，说明部署入口没有完成。

如果同一配置在 env 文件和 compose environment 中互相覆盖却没有边界说明，说明配置加载没有完成。

如果部署脚本可以运行，但不知道它属于本地开发、全量 compose、依赖 compose 还是独立镜像验证，说明入口契约没有完成。

## Few-shot Learning 示例组

### 正例：后端服务本地开发

一个后端项目包含 API、worker、监控面板、PostgreSQL 和 Redis。合适做法是让 API、worker 和监控面板由本地开发脚本管理，让 PostgreSQL 和 Redis 由 compose 依赖部署提供。

判断：

```text
Shape：本地开发启动 + compose 依赖部署
Service：应用进程跑宿主机，依赖服务跑容器
Config：env 文件表达环境差异，compose 只负责依赖服务
Entry：本地脚本管理应用进程，compose 命令管理依赖
Verify：脚本状态、健康检查和冒烟测试共同确认可用
```

### 反例：配置来源多对多

一个项目同时在 `.env`、`.env.dev`、`docker-compose.yml` 和启动脚本里维护数据库地址、端口、密钥和业务参数。开发者切换环境时不知道哪个值最终生效。

问题不在于配置文件数量，而在于没有区分环境差异和运行形态覆盖。应固定加载优先级，并限制 compose environment 只覆盖运行形态必须不同的值。

### 边界例：K8s 或云平台部署

如果项目已经进入 K8s、云平台 Secrets、滚动发布、自动扩缩容或完整 CI/CD 流水线，本方法只提供服务边界和配置优先级的前置判断，不替代生产运维平台设计。

## 误用

项目部署入口与配置加载不是 Docker 教程。

不要解释 Docker、compose、env 文件或 shell 命令的基础概念。只有当它们影响部署形态、服务边界、配置优先级或验证闭环时，才进入方法论。

项目部署入口与配置加载也不是生产运维平台设计。

它不处理 K8s 资源建模、云平台 Secrets、灰度发布、监控告警和 CI/CD 完整流水线。那些问题应进入相邻的运维或交付方法。

项目部署入口与配置加载也不是脚本仓库。

脚本只保留长期维护价值的检查和验证入口。一次性排查命令、临时实验脚本和重复清理脚本不应沉淀为项目标准入口。

它也不应被当作固定交付模板。

它先提供判断框架，帮助项目识别部署形态、服务边界、配置优先级和验证闭环。只有当部署方式稳定、需要多人或 Agent 复用时，才把判断结果固化为项目级文档、速查说明或必要脚本。

## 适用边界

这套方法适合处理常见前后端或后端项目的启动和部署治理，尤其是：

```text
项目同时存在本地开发和容器化部署
compose 既可能全量启动，也可能只启动依赖服务
Dockerfile 需要独立交付或验证
env 文件和 compose environment 之间存在覆盖关系
项目可能需要沉淀少量部署检查或验证入口
```

这套方法不适合处理纯工具库、无服务进程的脚本项目、一次性实验部署、完整生产运维平台设计、K8s 资源编排教学或云平台配置教程。

如果问题只是“不知道某个 Docker 或 compose 命令怎么用”，应使用工具能力使用与排错方法论。

如果问题是“项目常用启动、验证、生成、清理和构建动作如何统一入口”，应使用项目开发入口设计方法论。

如果问题是“已经稳定的部署路径如何写成可查、可复制、可维护的部署文档和发布手册”，应使用项目部署文档与发布手册方法论。
