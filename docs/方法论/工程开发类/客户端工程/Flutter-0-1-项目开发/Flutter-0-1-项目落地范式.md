# Flutter 0-1 项目落地范式

> 把 Flutter 新项目从需求意图、技术决策、产品切片、组件架构、项目骨架、分层实现、验证闭环到演进回写，压缩成一条稳定主线和可枚举分支。

```text
Flutter 0-1 项目落地范式
├── Intent：需求意图
│   └── 理解并复述项目真正要解决什么，本次不解决什么
├── Decision：技术决策
│   └── 锁定会影响架构、目录、依赖和演进成本的关键选择
├── Slice：产品切片
│   └── 把需求压缩成可判断、可验收、可维护的设计单元
├── Architecture：组件架构
│   └── 把 Slice 转成页面结构、组件边界、状态归属和数据流
├── Foundation：项目骨架
│   └── 建出可运行、可继续写业务的 Flutter 工程基础
├── Implementation：分层实现
│   └── 按数据层、状态层、UI 层、导航和错误处理完成代码
├── Verification：验证闭环
│   └── 用测试和人工验收确认行为符合 Slice 与架构文档
└── Evolution：演进回写
    └── 代码变化后同步回写项目文档、测试和变更记录
```

## 核心命题

Flutter 0-1 项目开发的问题，不是缺少某一份规范，而是容易把三类文档混在一起：

```text
指导规范包  ->  说明如何判断、如何写、如何实现
项目文档    ->  记录这个具体项目真实决定了什么
全景图      ->  展示指导规范包当前有哪些节点和参考物料
```

指导规范包不应该整包复制进真实项目。真实项目应该建立自己的 `docs/`，沉淀当前项目的 `Tech Decision`、`Slice`、组件架构、验收和变更记录。指导规范包的作用，是在项目文档写不清、代码边界混乱、阶段顺序漂移时，提供修正标准。

因此，这份范式处理的对象不是“某个 Flutter 项目的全部文档”，而是“如何把一套 Flutter 0-1 指导规范落到真实项目开发过程里”。

它的核心转化是：

```text
指导规范 -> 项目事实 -> 代码实现 -> 验证结果 -> 文档回写
```

## 方法论节点

### Intent：需求意图

先把需求转成可复述的项目意图。

这个节点解决的是“还没理解清楚就开始写 Slice 或代码”的问题。开发前需要写清楚：

```text
用户真正要解决什么
当前版本要交付什么
本次明确不做什么
哪些问题还需要确认
```

Intent 不等于产品长文档。它只需要让后续 `Decision`、`Slice` 和 `Architecture` 不再基于猜测推进。

### Decision：技术决策

锁定会影响项目结构和演进成本的关键技术选择。

这个节点处理的是不可随意漂移的选择，例如状态管理、本地数据库、导航、缓存、测试 Mock、构建工具和平台约束。选择一旦进入项目 `Tech Decision`，后续代码和文档应按这个选择推进；如果发现选择不合适，先记录问题，再通过变更管理调整。

Decision 的目标不是列出所有依赖，而是把“换了会返工”的选择提前压实。

### Slice：产品切片

把需求压缩成可判断、可实现、可验收的设计单元。

一个 Slice 单元默认是一份 Markdown 文档。目录只负责分类，不代表一个 Slice。开发前 Slice 是实现依据；代码回写 Slice 是维护基线。两者都服务于同一个目标：让后续实现和验收有稳定对象，而不是只靠口头理解。

Slice 不写类名、方法名、Provider 名和文件路径。它描述产品行为、状态、边界和验收标准。

### Architecture：组件架构

把 Slice 转成页面、组件、状态归属和数据流。

这个节点解决的是“产品设计有了，但 Widget 怎么拆、状态归谁、数据怎么流仍然靠临场发挥”的问题。组件架构文档应该回答：

```text
有哪些页面
每个页面由哪些组件组成
组件之间的职责边界是什么
状态由谁拥有，谁只消费
数据从哪里来，错误和空态在哪里展示
```

Architecture 是从产品文档进入代码实现前的最后一道结构确认。它不是代码实现说明，也不是把目录树提前写死。

### Foundation：项目骨架

建出可以开始写业务代码的 Flutter 工程基础。

这个节点包括 `flutter create` 之后的基础接入：目录主线、`ProviderScope`、路由入口、主题、错误基础类型、数据库入口、测试基础设施和 `pubspec.yaml` 组织方式。它的目标是让项目进入稳定业务开发状态。

Foundation 不是一次性创建所有目录。目录和基础设施按真实职责创建；没有数据层、没有跨 feature 复用组件、没有 CI 或发布需求时，不提前制造空壳。

### Implementation：分层实现

按已经确认的文档和架构写代码。

实现顺序通常是：

```text
数据模型 / Repository
状态管理 / Notifier / Provider
页面和组件
导航接入
错误、加载、空态
```

Implementation 的关键不是“所有项目必须长成同一个目录”，而是依赖方向稳定：UI 不直接访问数据库或 API，跨 feature 复用逻辑不藏在单个 feature 里，基础设施不反向依赖业务代码。

### Verification：验证闭环

用测试和人工验收确认行为成立。

验证对象来自 Slice 的 `Acceptance` 和组件架构文档，不是临时想起什么测什么。最小验证应覆盖：

```text
核心数据逻辑
关键 Widget 状态
主要用户链路
Slice Acceptance 中的可观察结果
```

如果测试失败暴露的是产品定义矛盾，回到 Slice；如果暴露的是组件边界不清，回到 Architecture；如果暴露的是代码实现问题，修代码并补测试。

### Evolution：演进回写

把真实变化同步回项目文档和规范判断。

项目从 1 到 100 的关键，不是继续堆新功能，而是每次变更都能回答：

```text
哪个 Slice 变了
哪个组件架构变了
哪个数据或路由契约变了
测试是否覆盖了变化
是否需要更新项目变更记录
是否暴露出指导规范包的缺口
```

当真实项目反复出现同类缺口时，才考虑回到指导规范包新增或修正规范。不要因为单个项目的一次性特殊情况，直接把指导规范包写成项目经验合集。

## 节点物料关系

| 节点 | 项目内产物 | 指导规范参考 | 是否必选 |
|------|------------|--------------|----------|
| Intent | 需求理解、边界和待确认点 | `Flutter-Slice-写作规范.md` | 必选 |
| Decision | `docs/tech-decisions/tech-decision.md` | `Flutter-Tech-Decision-模板.md` | 必选 |
| Slice | `docs/slices/**/*.md` | `Flutter-Slice-写作规范.md` | 必选 |
| Architecture | `docs/architecture/pages/*.md` | `Flutter-组件架构文档写作规范.md` | 有 UI 或页面结构时必选 |
| Foundation | 可运行的 Flutter 工程骨架 | `Flutter-项目目录结构规范.md`、项目初始化规范 | 必选 |
| Implementation | `lib/`、`test/` 中的项目代码 | 数据层、UI、导航、错误处理规范 | 必选 |
| Verification | 测试、验收记录、回归结论 | `Flutter-测试规范.md` | 必选 |
| Evolution | 变更记录、回写后的项目文档 | `Flutter-变更管理规范.md` | 演进期必选 |

表里的“指导规范参考”是参考物料，不是项目内必须复制的文件。项目内真正需要长期维护的是产物列。

## 默认主线

长期维护的 Flutter 项目，默认按下面顺序推进：

```text
Intent
-> Decision
-> Slice
-> Architecture
-> Foundation
-> Implementation
-> Verification
-> Evolution
```

这条主线不要求每个节点都写成长文档，但不允许跳过判断。可以轻量化，不能无意识缺失。

## 可枚举分支

不同项目可以裁剪执行强度，但不能破坏节点职责。

| 场景 | 推荐分支 | 不能裁剪的部分 |
|------|----------|----------------|
| 小型原型或单页工具 | Intent -> Decision 轻量版 -> Slice 轻量版 -> Foundation 轻量版 -> Implementation -> Verification 轻量版 | 需求边界、关键选型、可观察验收结果 |
| 已有代码需要补文档 | Intent -> 代码事实梳理 -> Slice Backfill -> Architecture Backfill -> Verification -> Evolution | 不能把错误实现包装成产品规则 |
| 暂无数据层 | Intent -> Decision -> Slice -> Architecture -> Foundation -> UI Implementation -> Verification | UI 不能直接埋未来会阻碍数据层接入的隐式契约 |
| 2 人以上协作 | 默认主线 + Git / PR 工作流 | PR 粒度和 Slice / 变更记录的对应关系 |
| 有 CI | 默认主线 + CI 质量门禁 | `analyze`、测试、生成代码和覆盖率阈值 |
| 准备外部分发 | 默认主线 + 发布与环境规范 | 环境、包名、签名、版本号和灰度渠道 |

分支的目的不是减少责任，而是根据当前阶段选择合适重量。

## 完成判断

这套范式是否落地，不看项目里复制了多少规范文档，只看下面四个判断是否成立：

```text
能说清楚当前项目为什么这样做
能找到每个关键决定对应的项目文档
能从项目文档追到代码和测试
能在代码变化后知道哪些文档必须回写
```

阶段性判断如下：

| 判断点 | 成立标准 |
|--------|----------|
| 可以开始编码 | Intent、Decision、当前 Slice、必要组件架构和 Foundation 已经足够支撑实现 |
| 可以交付内测 | 核心 Slice 的 Acceptance 已验证，测试和人工验收能解释主要行为 |
| 可以进入演进 | 变更传播路径清楚，新增功能、修复和架构调整知道回写哪里 |
| 可以反哺规范包 | 同类缺口在多个项目或多个阶段反复出现，已经不是单项目特例 |

## 示例校准

**正例：新功能开发**

先复述需求和边界，再写功能 Slice；如果涉及多个功能关系，补设计原则 Slice；进入实现前写页面组件架构；代码按项目目录和分层规范落地；最后用测试和验收记录回查 Slice。

**反例：整包复制指导规范**

把本目录下的所有指导规范复制到新项目 `docs/`，但没有写当前项目的真实技术决策、真实 Slice 和真实组件架构。这看起来文档很多，实际没有形成项目事实来源。

**边界例：一次性原型**

单页面验证原型可以不建立完整 `architecture/` 和 `data/`，但仍然需要保留最小 Intent、关键选型和可观察验收结果。否则原型一旦被继续维护，就会变成没有边界的代码堆积。

## 误用

不要把 `flutter_project_full_map.html` 当成项目开发步骤。它是指导规范包的状态图，不是项目内的事实来源。

不要把 `README.md` 当成落地方法。README 是目录入口和索引，不负责承载主线判断。

不要把参考规范当成必须照抄的项目模板。真实项目只维护自己的决策、设计、架构、测试和变更记录。

不要因为规范包暂时缺某份文档就停止项目推进。先按本范式完成项目内判断和落地，再评估这个缺口是否值得反哺为新的指导规范。

不要在 0-1 阶段提前写大量演进期规范。CI、PR、发布、性能基线、线上问题回流等文档，应等触发条件成立后再写。

## 适用边界

本范式适用于 Flutter 新项目从 0-1 起盘，以及从第一个可运行版本进入持续演进的早期阶段。

它不替代具体项目的产品需求文档、UI 设计稿、接口文档、发布手册和团队协作制度。它也不要求所有 Flutter 项目采用同一种目录深度或架构重量。

当问题变成“如何新增一篇可长期复用的方法论”时，应切换到 `方法论新增范式.md`。当问题变成“某个具体项目怎么写某份 Slice、组件架构或测试”，应回到对应项目文档，而不是继续扩展这份范式。
