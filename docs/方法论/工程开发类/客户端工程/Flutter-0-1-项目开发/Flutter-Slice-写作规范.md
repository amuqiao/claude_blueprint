# Flutter-Slice-写作规范

> Slice 不是功能清单，而是把一个产品对象、交互关系或验收边界压缩成可判断、可实现、可维护的设计单元。

---

## 什么是 Slice

Slice 是产品设计的最小可交付单元。它不描述代码怎么写，只描述产品应该是什么、用户体验是什么、什么情况下算做对了。

一份好的 slice 应该能直接回答三个问题：

```
这个功能 / 交互的边界在哪里？
有哪些情况需要处理？
怎么判断做对了？
```

如果一份文档回答不了这三个问题，它还不是 slice，只是笔记。

---

## Slice 的三种类型

写 slice 之前先判断类型，类型决定结构和写法。

| 类型 | 回答的问题 | 典型场景 |
|------|-----------|---------|
| 功能 slice | 这个功能做什么 | 列表页、编辑页、分类管理、统计页 |
| 设计原则 slice | 这些功能之间如何理解和协同 | 两个功能看起来都像筛选但职责不同、一个状态归谁拥有 |
| 验证 slice | 怎么确认多个 slice 没有互相打架 | 跨页面验收、回归检查 |

**判断卡住了怎么办**

问自己：这份文档的读者是谁？

```
读者是"要做这个功能的人"   -> 功能 slice
读者是"要理解这两个功能关系的人" -> 设计原则 slice
读者是"要验收整个链路的人"  -> 验证 slice
```

一份文档只服务一种读者。如果想同时服务两种，拆成两份。

## 功能 Slice

### 结构

```
Purpose        这个功能存在的原因，用户任务是什么
Experience Model  用户操作链路，自然语言描述
UI Structure   用户能看到什么，布局关系
Interaction    用户操作后发生什么
State Model    有哪些状态，状态之间如何转换
Edge Cases     边界情况和异常情况
Acceptance     怎么判断做对了
```

### 写作原则

**先写用户任务，再写页面结构。** 不要从"页面有一个按钮"开始写，要从"用户想做什么"开始写。

**State Model 只写产品状态。** 不写代码字段，不写变量名。写"用户选中了一个分类"，不写 `selectedCategory: String?`。

**Acceptance 要能直接变成测试用例。** 每一条验收都应该对应一个可操作的用户行为和一个可观察的结果。

**边界模糊时的判断：** 如果你在写功能 slice 时发现需要解释另一个功能，停下来，那部分内容属于设计原则 slice。

### 示例判断

```
"记录列表页显示用户记录，按分组倒序排列"
-> 功能 slice，描述列表页功能是什么

"索引点选和分类筛选都像是在筛选，但职责不同"
-> 不属于功能 slice，属于设计原则 slice
```

## 设计原则 Slice

### 什么时候需要写设计原则 Slice

出现以下情况时，说明需要一份设计原则 slice：

```
两个功能看起来都在做同一件事，但真实职责不同
一个 UI 动作容易被误解成状态变更
一个状态不清楚归哪个功能拥有
空结果不清楚是任务空态还是轻量反馈
```

如果没有这类混淆，不需要强行写设计原则 slice。

### 结构

```
Purpose        本文解释什么，不解释什么
Core Principle 一句话压缩核心设计原则
Terminology    把口语需求转成稳定设计术语
Experience Model  主要链路的用户体验
Relationship Model  不同对象之间的关系：谁负责什么，谁不负责什么
Situation Matrix  枚举主要情况，尤其是会改变判断路径的情况
Edge Cases     边界情况
Acceptance     怎么判断设计原则被正确实现了
```

### 写作原则

**开头一句话必须能压缩整份文档的核心判断。** 如果写不出来，说明设计原则还没想清楚，先不要写文档。

**Terminology 是这类文档最重要的部分。** 口语里的"点筛"、"筛选"、"锚点"在不同人脑子里是不同的东西。Terminology 的任务是消除这种歧义，给后续所有描述建立统一语言。

**Relationship Model 要说清楚边界。** 不只说"谁负责什么"，还要说"谁不负责什么"。负面约束往往比正面描述更重要。

**不写代码字段和实现路径。** 设计原则 slice 的读者是理解设计意图的人，不是实现代码的人。一旦写进类名、方法名、provider 名，文档就和具体实现绑定了，实现一改文档就过期。

**边界模糊时的判断：**

```
"索引点选负责定位，分类筛选负责可见性"
-> 设计原则 slice，说的是两个功能的关系

"索引点选日期后，列表滚动到当天第一条记录"
-> 这句话出现在功能 slice 里，不在设计原则 slice 里
```

## 验证 Slice

### 什么时候需要写验证 Slice

功能 slice 基本稳定之后，需要验证多个功能组合起来是否一致时写。不要在功能 slice 还在频繁变动时写验证 slice，否则验证 slice 会一直在追着上游跑。

### 结构

```
Verification Scope   验证哪些功能的哪些组合
Critical Paths       最重要的几条用户链路
State Matrix         跨功能的状态组合矩阵
Cross Slice Consistency  检查多个 slice 是否有矛盾
Regression Risks     最容易回归的风险点
Acceptance           验收通过的标准
```

### 写作原则

**只写可验证的结果。** 每一条都应该对应一个用户行为和一个屏幕状态，不重新解释设计原则。

**发现上游矛盾时，回写上游。** 验证 slice 不是打补丁的地方。如果发现两个功能 slice 互相矛盾，回到上游修，不要在验证文档里临时解释。

**State Matrix 关注组合情况。** 单个功能的状态矩阵在功能 slice 里写，验证 slice 里的矩阵关注多个功能同时生效时的情况。

## 实现细节边界

Slice 默认不写实现。以下三种情况例外：

```
1. 项目已有稳定实现事实，不写会导致误读
   例："索引作为列表页 section 展开，不是全屏页面"

2. 平台存在明确主流范式
   例："使用系统原生分享面板，不自定义"

3. 实现方式本身就是体验边界
   例："列表滚动定位使用原生 GlobalKey，不引入额外依赖"
```

**禁止写入 slice 的内容：**

```
类名、方法名、变量名、provider 名
为了实现方便倒推出来的产品规则   <- 这是最常见的错误
尚未验证的技术方案
与用户体验无关的内部数据结构
文件路径和重构计划
```

**"为了实现方便倒推产品规则"是什么意思**

举例：代码里用 `GlobalKey` 按日期定位，于是文档写"列表锚点是日期"。但真正的设计意图是"定位到当天第一条记录"，日期只是实现中的中间变量。这种情况下文档描述的是实现，不是设计，后续换实现方式时文档就会误导人。

## 文件命名和目录

```
slices/
├── features/           功能 slice
├── design-principles/  设计原则 slice
└── verification/       验证 slice
```

命名规则：

```
功能 slice           用功能名         item-list.md
设计原则 slice       包含 design      item-list-filter-design.md
验证 slice          包含 verification cross-slice-verification.md
```

暂时放扁平目录也可以，但文档类型必须在正文开头写清楚。

---

## 维护规则

```
交互语义变了      先改设计原则 slice，再检查相关功能 slice 是否需要同步
功能行为变了      改对应功能 slice
验证发现矛盾      回到上游 slice 修，不在验证文档里打补丁
```

写作顺序：先解释"为什么这样理解"，再枚举"有哪些情况"，最后写"如何验收"。顺序反了，文档就会变成规则清单，失去判断依据。
