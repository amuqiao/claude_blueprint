# Flutter Tech Decision 模板

> 这份模板用于记录阶段 0 的技术选型结论和依据。复制到具体项目后，写入本文档的选型即视为当前项目锁定；如果发现选型问题，先记录在 Open Questions 里，有充分依据再发起修改。

---

## 写 Tech Decision 文档的时机

在阶段 0 开始写，在写第一行业务代码之前完成。

判断"是否需要写进 Tech Decision"的标准：

```
换掉这个选型，需要重写超过一个文件的代码   -> 写进来
只是换一个工具函数、换一个 UI 库           -> 不需要
```

---

## 基础设施选型原则

基础设施默认优先使用成熟包或 Flutter / Dart 官方能力，不自定义实现通用能力。

```text
状态管理、路由、数据库、缓存、网络、序列化、权限、分享、图片处理、测试 Mock
  -> 优先选择成熟生态方案
  -> 不为了“可控”或“简单”临时自研框架级能力
```

允许自定义实现的情况只有三类：

```text
1. 官方能力已经足够简单，例如原生 GlobalKey 能稳定解决当前滚动定位
2. 成熟包引入的复杂度明显超过当前问题，例如为了一个工具函数引入大型依赖
3. 项目有明确差异化需求，成熟包不能满足，并且已经写清维护成本和替换边界
```

Tech Decision 里每个基础设施选型都要记录：

```text
选择了哪个成熟包 / 官方能力
放弃了哪些备选项
为什么不自定义实现
如果未来要替换，影响哪些模块
```

## 示例选型

### 状态管理：Riverpod

**选型原因**
- 编译期类型安全，Provider 引用错误在编译时暴露，不在运行时
- `AsyncNotifier` 对异步数据的支持比 Provider 原生更完整
- 代码生成（`@riverpod`）减少样板代码

**放弃的选项**
- Bloc：适合大型团队，事件/状态分离带来额外结构成本，对当前项目规模偏重
- Provider（原版）：没有编译期安全保障，`context.watch` 的使用位置限制多

**如果换掉的代价**
重写所有 Notifier、所有 `ref.watch` 调用和页面状态层。

---

### 本地数据库：Drift

**选型原因**
- 类型安全的 SQL：表定义和查询都在 Dart 里，列名拼错编译就报错
- Stream 支持：`watchAll()` 直接返回 `Stream`，和 Riverpod 的响应式模型天然配合
- Migration API 完整，支持结构化的版本迁移

**放弃的选项**
- Isar：写入性能更高，但查询 API 是 Isar 自己的 DSL，和 SQL 思维差异大，学习成本高
- Hive：适合简单 key-value，不适合有关联查询（记录 + 分类）的场景

**如果换掉的代价**
重写所有 DAO、所有 Repository 实现和所有 migration。

---

### 导航：GoRouter

**选型原因**
- Flutter 官方维护，和 Navigator 2.0 无缝集成
- 声明式路由配置，路由树结构清晰
- 深链接支持完整，路由路径即 URL

**放弃的选项**
- Navigator 2.0（原生）：手动实现太繁琐，样板代码多
- AutoRoute：代码生成方案，功能完整但依赖额外构建步骤

**如果换掉的代价**
重构所有页面入口、路由参数传递和跳转调用。

---

### 滚动定位：待定（scrollable_positioned_list 或原生 GlobalKey）

**问题描述**
索引项被点选后，列表需要滚动定位到目标分组的第一条记录。

**选项对比**

| | scrollable_positioned_list | 原生 GlobalKey |
|---|---|---|
| 实现复杂度 | 低，直接调用 `scrollTo(index)` | 中，需要手动计算偏移 |
| 依赖风险 | 引入第三方包，需要维护 | 零依赖 |
| 适用条件 | 列表项等高或高度可预估 | 任意列表 |

**决定时机**
首个列表实现完成后，用真实数据验证定位精度，再做最终决定。

**当前约定**
在决定前，列表组件的滚动定位接口对外保持语义稳定（"定位到目标分组第一条记录"），内部实现可以替换。

---

### 图片/资源缓存：cached_network_image + flutter_cache_manager

**选型原因**
- `cached_network_image` 是 Flutter 生态事实标准，维护活跃
- `flutter_cache_manager` 提供缓存策略控制（过期时间、最大缓存大小）
- 占位图、错误图、渐进加载开箱即用

**如果换掉的代价**
重构所有图片展示组件和资源生命周期管理。

---

### 测试 Mock：mocktail

**选型原因**
- 不需要代码生成，`build_runner` 不需要额外配置
- API 和 mockito 相似，迁移成本低
- 适合中小规模项目快速迭代

**如果换掉的代价**
重写所有 Mock 类定义（通常只需要改继承方式，成本较低）。

```
    待决定：由产品路线图决定，未决定前不引入 Dio 或任何网络层依赖
```
