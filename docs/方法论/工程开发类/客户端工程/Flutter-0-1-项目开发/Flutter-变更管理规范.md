# Flutter 变更管理规范

> 变更不是问题，没有传播路径的变更才是问题。一个 Slice 改了，组件架构、数据层、UI 层、测试各自要不要跟着改，必须在动手之前就想清楚。

---

## 为什么需要变更管理规范

这个项目的文档和代码分为以下层级：

```
产品层     Slice 设计文档
结构层     组件架构文档
数据层     数据层设计文档 + 错误处理规范
UI 层      UI 组件规范 + 导航规范
基础设施层  项目目录结构规范
验证层     测试
```

任何一层发生变更，都可能影响其他层。没有规范的情况下，最常见的结局是：代码改了，文档没改；或者文档改了，代码只改了一半。三个月后没人知道哪个是真的。

这份规范只做一件事：给每一类变更定义传播路径，让"改一个地方"变成"按顺序改对的几个地方"。

**哪些变更还需要额外检查：**

```
路由路径变了    -> 同步检查导航规范的路由常量定义
新增错误类型    -> 同步更新错误处理规范的 AppError 子类和 AppErrorMessage
目录结构调整    -> 同步更新项目目录结构规范
```

---

## 变更的四种来源

在开始讨论传播路径之前，先区分变更是从哪里发起的：

```
产品变更    用户需求变了，体验要调整       从 Slice 开始往下传播
架构变更    组件拆分方式变了              从组件架构文档开始，向上检查 Slice，向下同步代码
数据变更    数据模型或 API 字段变了       从数据层开始，向上检查 Slice，向 UI 层传播
修复变更    发现 bug，修复实现            从代码开始，检查是否需要同步文档
```

方向搞反了会导致文档和实现脱节。产品变更必须先改 Slice，不能先改代码然后回来补文档。

---

## 产品变更的传播路径

产品变更 = 用户体验变了、交互逻辑变了、验收条件变了。

### 传播顺序

```
1. 更新 Slice
   -> 如果是交互语义变更，先改设计原则 Slice
   -> 如果是功能行为变更，改功能 Slice
   -> 更新 Acceptance 条件

2. 检查组件架构文档
   -> State Ownership 是否需要调整（有没有新的状态、状态归属是否变化）
   -> Component Tree 是否需要调整（新增 / 删除组件）
   -> Boundary Rules 是否受影响

3. 检查数据层
   -> Domain Model 是否需要新增字段
   -> Repository 接口是否需要新增方法
   -> Provider 是否需要调整状态结构

4. 更新 UI 层
   -> 按照更新后的组件架构文档调整组件

5. 更新测试
   -> 更新受影响的 Widget 测试
   -> 更新受影响的集成测试
   -> 验证新的 Acceptance 条件有对应测试覆盖
```

### 检查清单

每次产品变更完成后，对照以下清单确认没有遗漏：

```
[ ] Slice 的 Acceptance 条件已更新
[ ] 组件架构文档的 State Ownership 已检查
[ ] Domain Model 字段和产品状态一一对应
[ ] 受影响的测试已更新，没有测试依赖旧的行为
[ ] Open Questions 里的相关问题已标记已解决或已调整
```

---

## 架构变更的传播路径

架构变更 = 组件拆分方式变了、状态归属调整了、层级关系重组了。

### 传播顺序

```
1. 更新组件架构文档
   -> 先在文档里确定新的 Component Tree 和 State Ownership
   -> 在动代码之前，文档必须是最新的

2. 向上检查 Slice
   -> 架构变更有没有改变产品行为（如果改变了，这是产品变更，不是纯架构变更）
   -> 通常架构变更不影响 Slice，确认一下即可

3. 更新代码
   -> 按新的 Component Tree 重构 Widget
   -> 按新的 State Ownership 移动状态

4. 更新测试
   -> 组件路径变了，Widget 测试的 finder 可能需要更新
   -> 验证功能行为没有变化
```

### 架构变更的特殊风险

架构变更最容易引入的问题是：重构了组件结构，但测试没有完整覆盖，导致某个边界情况的行为静默改变。

每次架构变更完成后，必须跑完整的 Widget 测试套件，确认没有测试失败。如果测试失败了，要判断是测试需要更新（因为组件结构变了）还是功能行为被意外改变了（需要修复）。

## 数据变更的传播路径

数据变更 = Domain Model 字段变了、Repository 接口变了、数据库表结构变了、API 格式变了。

### 传播顺序

```
1. 判断是否影响产品行为
   -> 如果影响（新字段改变了 UI 展示），先更新 Slice
   -> 如果不影响（内部字段重命名、优化查询），直接从步骤 2 开始

2. 更新数据模型
   -> 更新 Domain Model
   -> 如果是表结构变更，写 migration（不要忘记）
   -> 如果是 DTO 变更，更新序列化代码

3. 更新 Repository 接口
   -> 接口变了，同步更新实现

4. 更新 Notifier
   -> Repository 接口变了，检查所有使用该接口的 Notifier
   -> 字段名变了，检查所有引用该字段的地方

5. 更新 UI 层
   -> 新字段需要展示，更新对应组件
   -> 字段删除了，清理组件里的引用

6. 更新测试
   -> 数据层单元测试
   -> 如果 UI 展示变了，更新 Widget 测试
```

### 数据变更的特殊情况：废弃字段

不要保留废弃字段"以防万一"。废弃字段一旦留下来，以后没人敢删，会永久存在于代码库。

正确做法：确认没有使用方后，在同一次 PR 里删除字段、删除相关代码、更新测试。

---

## 修复变更的传播路径

修复变更 = 发现 bug，修复实现，行为应当符合原始 Slice 的 Acceptance 条件。

### 传播顺序

```
1. 确认 bug 的性质
   -> 实现和 Slice 不符    -> 修复代码，不改文档
   -> Slice 本身有歧义     -> 先澄清 Slice，再修复代码
   -> Slice 描述的行为确实需要调整 -> 按产品变更流程处理

2. 修复代码

3. 补充测试
   -> 这个 bug 为什么没有被测试发现？
   -> 补充对应的测试，防止回归

4. 检查是否有类似问题
   -> 同一个模式的 bug 是否在其他地方也存在？
```

修复变更不需要更新 Slice，除非 bug 的根因是 Slice 描述有歧义导致实现偏差。

---

## 不允许的变更模式

以下几种变更模式会导致文档和实现脱节，必须避免：

```
先写代码再补文档
  -> 结果：文档变成实现的注脚，而不是实现的依据
  -> 正确做法：先更新文档，再写代码

在验证 Slice 里打补丁
  -> 结果：上游 Slice 的矛盾被掩盖，下次改动还会撞上
  -> 正确做法：发现矛盾回到上游 Slice 修

改了代码但不更新 Open Questions
  -> 结果：Open Questions 里记录的未决问题永远不会被关闭
  -> 正确做法：变更完成后检查 Open Questions，标记已解决的问题

只改接口不改实现（或只改实现不改接口）
  -> 结果：接口文档和实际行为不一致
  -> 正确做法：接口和实现在同一次变更里同步更新
```

---

## 变更的最小单元

一次变更应该是原子的：一个 Slice 的一处改动，加上它引发的所有下游更新，应该在一次完整的工作单元里完成，而不是分散在多次提交里。

判断一次变更是否完整的标准：

```
所有受影响的文档都已更新
所有受影响的代码都已更新
所有受影响的测试都已更新，没有测试失败
```

如果这三条不能同时满足，这次变更还没有完成。
