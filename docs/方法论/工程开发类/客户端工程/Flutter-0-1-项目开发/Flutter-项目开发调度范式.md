# Flutter 项目开发调度范式

> 当 Flutter 项目、需求、模块、基础设施或文档问题进入开发过程时，先判断它属于哪一层，再调用对应规范和产物，而不是让所有问题都走同一条流程。

## 文档定位

本文不是 `Flutter-项目推进的正确顺序.md` 的替代品，也不是某个功能的 Slice 写作模板。

它解决的是一个更前置的问题：

```text
我现在面对的是项目起盘问题、版本收敛问题、功能模块问题、基础设施问题，还是文档回写问题？
```

只有先回答这个问题，后续才知道应该读哪份规范、产出哪份项目文档、做到什么程度可以停。

新项目第一次使用这套指导规范时，只需要先读两份文档：

```text
README.md                    负责维护规范包索引，告诉你有哪些文档
Flutter-项目开发调度范式.md   负责建立调度心智，告诉你当前该进入哪一层
```

本文不维护具体规范索引，也不维护任何真实项目的文档路径。需要查具体规范时，回到 `README.md`；需要记录真实项目决策时，写入该项目自己的 `docs/`。

## 核心命题

Flutter 0-1 开发容易混乱，不是因为缺少规范，而是因为不同层级的问题被压成了一条线：

```text
项目起盘
需求分析
版本取舍
功能设计
模块实现
基础设施演进
文档回写
```

这些动作都属于开发过程，但它们不是同一个层级。把它们写成一条固定流程，会让开发者误以为每次遇到需求、写模块、补文档都要从头跑完整项目流程。

本文的核心转化是：

```text
线性流程思维 -> 分层调度思维
```

先定位层级，再进入对应循环。

## 总体心智模型

```text
Flutter 项目开发调度范式
├── Project：项目起盘层
│   └── 从想法、目标和约束进入可持续开发骨架
├── Version：版本收敛层
│   └── 判断当前轮做什么、不做什么、什么延后
├── Feature：功能模块层
│   └── 把需求落成 Slice、组件架构、实现和验收
├── Infrastructure：基础设施层
│   └── 判断路由、状态、数据库、CI、发布等能力何时升级
└── Governance：治理回写层
    └── 让代码、测试、项目文档和指导规范保持一致
```

这五层不是五个固定阶段。它们是开发过程中的五个判断入口。

## 第一判断：我现在站在哪一层

拿到一个新问题时，先做下面的分流。

| 当前问题 | 所属层级 | 最小下一步 | 主要产物 |
|----------|----------|------------|----------|
| 项目还只是想法，不知道怎么起盘 | Project | 明确项目目标、当前版本边界和首个核心模块 | 项目 README、Tech Decision、最小工程骨架 |
| 新需求进入，不确定该不该做 | Version | 判断是否进入当前版本，是否延后或拒绝 | 版本边界、需求取舍记录 |
| 已确定要做某个功能或模块 | Feature | 写 Slice，再写组件架构，再实现和验收 | Slice、组件架构文档、代码、测试 |
| 现有能力不够，开始影响开发 | Infrastructure | 判断是否是真实阻塞，是否需要升级 | 基础设施决策、升级触发条件、实现约束 |
| 代码、文档、测试开始不一致 | Governance | 判断哪些项目文档和规范需要回写 | 变更记录、Backfill Slice、测试补充 |

判断完成后，只进入对应层级的最小充分路径，不强行跑完整套规范。

## Project：项目起盘层

Project 层只处理一个问题：项目如何从模糊想法进入可持续开发状态。

适用场景：

```text
新建 Flutter 项目
项目目标还不清楚
技术选型还没锁定
还没有第一个核心模块
工程骨架还不能支撑业务开发
```

Project 层的最小主线：

```text
项目目标
-> 当前版本边界
-> 技术选型
-> 首个核心模块
-> 最小项目骨架
-> 进入 Feature 层
```

这里会调用：

```text
Flutter-项目推进的正确顺序.md
Flutter-Tech-Decision-模板.md
Flutter-项目目录结构规范.md
```

Project 层不负责把所有功能提前设计完，也不负责提前铺满 CI、发布、性能、线上问题回流等演进期规范。

## Version：版本收敛层

Version 层处理的是“当前轮到底做什么”的问题。

适用场景：

```text
新需求不断进入
当前版本边界开始变模糊
一个需求有价值，但不确定现在该不该做
多个功能都重要，但开发顺序需要判断
```

Version 层的最小主线：

```text
进入系统的事项是什么
-> 属于新功能、旧功能扩展、基础设施、技术债还是暂不处理
-> 是否进入当前版本
-> 当前版本最小可交付形态是什么
-> 进入 Feature / Infrastructure / Governance 对应层级
```

Version 层不直接写 Widget，也不直接写 Slice 细节。它先决定一个事项是否值得进入当前版本；进入后，才交给 Feature 层切 Slice。

## Feature：功能模块层

Feature 层处理的是“一个功能如何健康接入项目”的问题。

适用场景：

```text
已经决定要做某个功能
需要写 Slice
需要设计页面和组件
需要实现数据、状态、UI、导航和错误处理
需要验收一个功能是否完成
```

Feature 层的最小主线：

```text
Request Understanding
-> Slice
-> Component Architecture
-> Implementation
-> Verification
-> Governance
```

这里会调用：

```text
Flutter-Slice-写作规范.md
Flutter-组件架构文档写作规范.md
Flutter-数据层设计规范.md
Flutter-UI组件规范.md
Flutter-导航规范.md
Flutter-错误处理规范.md
Flutter-测试规范.md
```

Feature 层是最常用的开发循环。它不是 Project 层的重复执行，不需要每次都重新做全局技术选型或重建项目骨架。

## Infrastructure：基础设施层

Infrastructure 层处理的是“现有底层能力是否该升级”的问题。

适用场景：

```text
路由、状态管理、数据库、缓存、构建、CI、发布、性能开始成为阻塞
某个功能接入时暴露现有工程能力不足
继续用最轻方案会让后续维护成本明显升高
```

Infrastructure 层的最小主线：

```text
当前阻塞是什么
-> 是否是真实需求，而不是未来想象
-> 当前最轻方案是什么
-> 需要留哪些演进空间
-> 什么条件触发升级
-> 升级后如何验证和回写
```

基础设施规范不应该在 0-1 开始时全部写满。CI、Git / PR、发布与环境、性能基线、线上问题回流等文档，应在触发条件成立后再补。

## Governance：治理回写层

Governance 层处理的是“项目资产如何保持一致”的问题。

适用场景：

```text
代码已经改了，但 Slice 没更新
实现行为和原设计不一致
测试覆盖跟不上
项目文档和真实代码脱节
多次出现同类问题，说明指导规范可能缺口
```

Governance 层的最小主线：

```text
变化发生在哪里
-> 哪些 Slice / 架构 / 技术决策受影响
-> 哪些测试需要补充或调整
-> 是否需要记录变更
-> 是否反哺指导规范包
```

这里会调用：

```text
Flutter-变更管理规范.md
Flutter-Slice-写作规范.md 中的 Backfill 模式
Flutter-测试规范.md
```

Governance 层不是事后补形式。它决定项目能否从 1 继续演进到 100。

## 场景用法

### 新项目 0-1 起盘

```text
Project
-> 明确项目目标和版本边界
-> 写 Tech Decision
-> 建最小项目骨架
-> 选择首个核心 Feature
-> 进入 Feature 层
```

### 新需求进入

```text
Version
-> 判断需求性质
-> 判断是否进入当前版本
-> 如果进入，切成最小可交付形态
-> 进入 Feature 层
```

### 设计并实现一个功能模块

```text
Feature
-> Request Understanding
-> Slice
-> 组件架构
-> 分层实现
-> 测试和验收
-> Governance 回写
```

### 基础设施开始不够用

```text
Infrastructure
-> 判断是否真实阻塞
-> 先给最轻方案
-> 明确升级触发条件
-> 触发后补对应规范和实现
-> Governance 回写
```

### 已有代码需要补文档

```text
Governance
-> 从真实代码反推行为
-> Backfill Slice
-> Backfill Architecture
-> 补测试
-> 标出实现与合理产品行为的冲突
```

## 指导规范包与项目文档的关系

指导规范包负责回答：

```text
应该如何判断
应该如何写文档
应该如何组织代码
常见错误是什么
做到什么程度算完成
```

真实项目文档负责回答：

```text
这个项目实际选择了什么
这个版本实际承诺什么
这个功能实际怎么工作
这个模块实际如何接入
这次变更实际影响哪里
```

不要把指导规范包整包复制进项目。真实项目应建立自己的 `docs/`，只维护当前项目真实存在的决策、设计、架构、验收和变更记录。

## 完成判断

这套调度范式是否成立，看开发者能不能稳定回答五个问题：

```text
当前问题属于哪一层
当前层级的最小下一步是什么
当前应该调用哪份规范
当前应产出哪份项目文档或代码结果
完成后需要回写什么
```

如果开发者仍然不知道“我是该写 Tech Decision、写 Slice、设计组件、升级基础设施，还是回写文档”，说明调度还没有完成。

## 误用

不要把本文当成每个任务都必须完整执行的流程清单。

不要在 Feature 层问题里重新做 Project 层决策，例如每写一个功能都重新讨论状态管理或数据库选型。

不要在 Version 层还没判断清楚时，直接进入 Slice 和实现。

不要在 Infrastructure 层没有真实阻塞时，提前补 CI、发布、性能、线上回流等演进期规范。

不要把 Governance 当成最后补文档。它是代码、测试、项目文档和指导规范保持一致的闭环。

## 适用边界

本文适用于 Flutter 项目从 0-1 起盘到早期持续演进的开发调度。

它不替代具体规范文档，不替代项目 README，不替代 `flutter_project_full_map.html`。全景图可以由人按需要维护；本文只负责建立开发者判断当前层级和调用规范的心智模型。

当问题已经明确属于某一层，应停止继续展开本文，直接进入对应规范或项目文档。
