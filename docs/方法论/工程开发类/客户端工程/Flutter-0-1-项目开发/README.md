## 待补与条件触发规范

不是所有规范都应该在 0-1 阶段一次写完。判断一份规范是否值得单独成文，看三个条件：

```
没有它会踩坑      -> 不写大概率出问题
现有规范没有覆盖  -> 不重复已有文档
当前阶段用得上    -> 有执行主体和使用场景
```

当前唯一无条件待补的是 **Flutter-项目初始化规范**：它负责从 `flutter create` 到 Riverpod / Drift / GoRouter 接入完成，让项目进入可以开始写业务代码的状态。

以下规范按触发条件处理：

| 规范 | 触发条件 | 处理方式 |
|------|---------|---------|
| Flutter-CI质量门禁规范 | 项目接入 CI | 有 CI 时必选 |
| Flutter-Git-PR工作流规范 | 2 人以上协作 | 2 人以上必选 |
| Flutter-发布与环境规范 | 准备 TestFlight / 内测 / 上线 | 外部分发前必选 |
| Flutter-性能基线规范 | 有真实 profiling 数据 | 延后评估 |
| Flutter-线上问题回流规范 | 项目上线后产生真实问题 | 延后评估 |
| 安全与隐私约束 | 产品涉及敏感数据或权限 | 写入对应 Slice 的 Edge Cases |
| 设计系统与可访问性约束 | UI 规范需要扩展 | 并入 Flutter-UI 组件规范 |

完整分层见 [Flutter 0-1 项目开发文档全景图](./flutter_project_full_map.html)。

## 项目技术选型

以下是本方法论默认推荐的阶段 0 选型。具体项目应在 Tech Decision 文档中记录最终结论、放弃选项和未决问题。

| 类别 | 选型 |
|------|------|
| 状态管理 | Riverpod |
| 本地数据库 | Drift |
| 导航 | GoRouter |
| 图片缓存 | cached_network_image + flutter_cache_manager |
| 滚动定位 | scrollable_positioned_list 或原生 GlobalKey |
| 测试 Mock | mocktail |

已经写入具体项目 Tech Decision 的选型，视为当前项目已锁定，不要在项目中途随意引入替代方案。如果发现选型有问题，先记录在 Open Questions 里，等有充分依据再发起正式讨论。

## 新成员入门路径

**第一天**：读本文档 + 项目推进顺序 + Tech Decision 模板 + Slice 写作规范，建立整体认知。

**开始第一个功能之前**：读组件架构文档写作规范 + 数据层设计规范 + UI 组件规范 + 项目目录结构规范。

**遇到具体问题时**：查上面的文档地图，找对应的规范。

**改现有功能时**：先读变更管理规范，确认传播路径，再动手。

## 规范的维护约定

规范不是一次写完就不动的文档。以下情况需要更新规范：

```
发现规范描述和实际情况有出入      -> 更新规范，让规范追上现实
发现新的常见错误模式             -> 在对应规范的"禁止"部分补充
两份规范之间有矛盾               -> 在变更管理规范的框架下解决，不在一份规范里打补丁
新增了本文档没有覆盖的重要约定    -> 新建规范文档，在本索引里登记
```

规范的可信度来自于它和实际代码的一致性。规范和代码脱节比没有规范更危险。
