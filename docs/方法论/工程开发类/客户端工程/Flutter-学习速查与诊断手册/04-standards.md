# Flutter 开发规范
> 以下规范由实践验证，可直接指导实现判断。每条规范包含：触发信号、推荐方案、禁止做法、验收标准。

---

## 一、导航（Navigation）

### 1.1 CupertinoSliverNavigationBar 大标题与紧凑标题分工

**触发信号**：出现以下任一现象时使用本方案：
- 展开态同时看到两个相同标题
- `largeTitle` 和 `middle` 文案重复，用户无法判断哪个是页面主标题
- 为隐藏标题手写滚动监听、透明度计算或局部状态同步

**推荐方案**：让 Flutter 原生导航栏状态处理标题显隐，不要手写：

```dart
CupertinoSliverNavigationBar(
  alwaysShowMiddle: false,  // 展开态隐藏 middle
  largeTitle: Text('页面主标题'),
  middle: PageTitleMenu(),  // 折叠态显示紧凑标题或标题菜单
)
```

状态分工：
```
展开态 → largeTitle 表达页面身份，middle 不显示
折叠态 → middle 表达紧凑标题或标题菜单，largeTitle 已折叠
```

**禁止做法**：
- 展开态同时高强调显示两个相同标题
- 把 `middle` 和 `largeTitle` 都当作页面身份标题
- 为此场景手写滚动阈值、透明度或标题显隐状态
- 把筛选入口提升成与页面标题同级的第二视觉中心

**验收标准**：
- 展开态只出现一个页面主标题
- 折叠态仍可找到标题菜单或筛选入口
- 不需要手写滚动监听控制标题显隐
- 标题菜单点击区域不小于 44×44

**适用边界**：仅适用于 `CupertinoSliverNavigationBar` 的 large title 页面。不适用于完全自定义导航栏、`AppBar` / `SliverAppBar` Material 折叠标题。

---

### 1.2 Sheet 不承接主导航

**触发信号**：页面需要临时承接筛选、设置、选择器或确认任务。

**推荐方案**：`showModalBottomSheet` 只承接当前上下文的轻量任务：

```
当前页面 → 打开 sheet → 完成选择/取消/关闭 → 回到当前页面上下文
```

**禁止做法**：
- 在 sheet 中承接多层主导航
- 让 sheet 打开后改变底层页面所属主任务
- 把设置、统计、编辑等长期页面伪装成临时 sheet

**验收标准**：
- sheet 关闭后用户回到原页面上下文
- sheet 内有清晰关闭路径
- 有未保存输入时，关闭前必须确认或保留草稿

## 二、交互（Interaction）

### 2.1 单屏单主动作

**触发信号**：出现以下任一现象时处理：
- 同一屏存在两个执行同一任务的主入口
- 空状态主按钮和底部 FAB 同时指向新建
- 筛选无结果时主按钮仍然是新建
- 装饰图标看起来像可点击按钮

**推荐方案**：先判断当前屏幕状态，再确定唯一主动作：

```
首次空状态
  → 主动作：新建第一条内容
  → 不显示全局 FAB

正常列表
  → 主动作：全局 FAB 新建
  → 列表项操作作为次级动作

筛选无结果
  → 主动作：清除筛选
  → 已有内容时，FAB 可保留为次级全局入口

加载失败
  → 主动作：重试
  → 不引导创建内容
```

**禁止做法**：
- 同一状态下出现两个视觉等级相同的主按钮
- 空状态插画使用 `plus`、`edit`、FAB 或 filled button 造型
- 用颜色同时表达心情、主按钮和状态含义

**验收标准**：
- 用户不用阅读说明也能判断下一步
- 当前状态只有一个 primary CTA
- 次级动作在视觉重量、位置或文案上明显弱于主动作
- 所有触控目标不小于 44×44

---

### 2.2 筛选状态必须外显

**触发信号**：用户通过日期、标签或其他条件改变列表内容。

**推荐方案**：筛选生效后，页面必须持续显示 active chip 或等价状态条：

```
选择筛选条件 → 列表刷新 → active chip 可见
→ 用户可单独清除某个条件
→ 用户可清除全部条件
```

**禁止做法**：
- 筛选只存在于 sheet 内部
- 关闭筛选面板后用户看不出当前不是全部数据
- 新建、编辑、删除后静默清除用户筛选

**验收标准**：
- 筛选条件能在主页面直接看见
- 每个筛选条件有独立清除路径
- 筛选空态解释的是"当前条件无结果"，不是"没有数据"

## 三、状态（State）

### 3.1 状态分层

**触发信号**：页面同时处理数据库数据、筛选条件、UI 展开状态、加载错误、反馈等多种状态。

**推荐方案**：按生命周期和所有权分层管理：

```
Persistent State（持久化状态）
  → 数据库、用户设置、长期保存的业务对象
  → 存 Drift / SharedPreferences

Page State（页面状态）
  → 当前筛选、展开/收起、选中项、当前年份
  → 存 NotifierProvider 或 StatefulWidget

Derived State（派生状态）
  → 从持久化数据 + 页面状态计算出来的列表、统计、空状态判断
  → 不存储，按需计算，可单独测试

Ephemeral State（一次性状态）
  → loading、saving、SnackBar 反馈、确认弹窗
  → 消费后清除，不持久化
```

**实现模板**：

```dart
// 页面状态类
class PageState {
  const PageState({
    this.filter = const ItemFilter(),
    required this.selectedYear,
    this.isExpanded = false,
    this.feedback,   // Ephemeral State
  });

  final ItemFilter filter;
  final int selectedYear;
  final bool isExpanded;
  final PageFeedback? feedback;
}

// 派生模型单独构建，脱离 Widget 可测试
final model = PresentationModel.build(
  filteredItems: items,
  filter: state.filter,
);
```

**禁止做法**：
- 把筛选条件写入业务实体
- 把派生结果持久化为业务事实
- 让 Widget 分支到处重复计算同一派生逻辑
- 把一次性 SnackBar 状态长期保留在持久化数据中

**验收标准**：
- 每个状态能说清所有权和生命周期
- 页面刷新不会破坏持久化数据
- 派生模型可以脱离 Widget 进行测试
- 一次性交互反馈被消费后不会重复出现

---

### 3.2 筛选是页面状态，不是数据状态

**触发信号**：列表、时间轴、统计按条件展示数据。

**推荐方案**：筛选对象属于页面状态，持久化对象只保存业务事实：

```
业务实体（Item）
  → 保存发生时间、内容、类型、标签
  → 存 Drift

筛选条件（ItemFilter）
  → 保存当前页面如何查看 Item
  → 存 Page State（Provider 或 StatefulWidget）
```

**禁止做法**：
- 在业务实体中保存"当前是否被筛选命中"
- 在数据库中保存当前页面 active chip
- 因新建或编辑记录静默重置筛选

**验收标准**：
- 同一条业务数据可被多个页面状态以不同方式展示
- 清除筛选不会修改任何业务实体
- 筛选条件可以独立测试

## 四、组件（Components）

### 4.1 成熟组件优先

**触发信号**：需要实现导航、滚动、输入、弹窗、选择器、按钮、列表、反馈等通用 UI。

**推荐方案**：优先使用 Flutter / Cupertino / Material 已提供的组件和参数：

```
导航    → go_router / CupertinoSliverNavigationBar / AppBar
滚动    → CustomScrollView / SliverList / ListView / SingleChildScrollView
浮层    → showModalBottomSheet / showCupertinoDialog / PopupMenuButton
输入    → TextField / Form / 系统日期时间选择器
反馈    → SnackBar / AlertDialog / CircularProgressIndicator
```

**禁止做法**：
- 为通用按钮、基础输入、普通弹窗重造组件体系
- 用手写布局状态替代组件已有参数
- 把一次性页面布局提前抽成全局组件

**验收标准**：
- 能指出使用的 Flutter 成熟组件或参数
- 自定义部分有明确产品识别理由
- 组件边界清楚，不把业务状态、数据库查询和样式细节混在一起

---

### 4.2 自定义组件边界

**触发信号**：某个 UI 对象具有产品识别度，成熟组件无法自然表达。

**推荐方案**：只有满足以下全部条件时才自定义：

```
✓ 成熟组件无法表达
✓ 对象构成产品识别
✓ 状态边界清楚
✓ 后续可以通过 token 或参数调整
```

适合自定义的对象：时间轴轨道、心情节点、带 pointer 的气泡、热力图 cell 组合。

不适合自定义的对象：基础导航系统、普通按钮、普通输入框、普通弹窗、普通选择菜单。

**验收标准**：
- 自定义组件职责一句话可说明
- 输入参数明确，不直接读取无关全局状态
- 样式依赖 theme token，不散落魔法数
- 可通过 Widget Test 或视觉检查验证关键状态

## 五、快速诊断

### "导航混乱" 问题诊断

| 症状 | 最可能的规范违反 | 修复 |
|---|---|---|
| Sheet 打开后，底层页面的导航栏消失或界面变了 | 违反 1.2 规则：Sheet 在里面做了导航决策 | Sheet 只能完成本地任务（选择、输入、确认），不能改变底层页面身份；完成后 dismiss，回到原页面上下文 |
| 用户打开 Sheet 选择，关闭后底层列表状态丢失了 | Sheet 没有暂存选择，用户的前序操作被遗忘 | 在 Sheet 关闭前，把选择保存到 Page State（StatefulWidget 或 NotifierProvider），dismiss 时带回选择结果 |

### "交互不清" 问题诊断

| 症状 | 最可能的规范违反 | 修复 |
|---|---|---|
| 空状态页面有多个相同权重的按钮，用户不知道点哪个 | 违反 2.1 规则：单屏单主动作 | 判断当前状态（首次、列表、无结果、加载失败），为每个状态指定唯一的主动作（primary CTA）；其他按钮降级为次级 |
| 用户进行了筛选，但没有任何提示当前在筛选状态，一会儿就忘了 | 违反 2.2 规则：筛选状态不外显 | 筛选生效后，页面顶部或列表上方加 active chip/tag 显示当前筛选条件，用户可单独清除条件 |
| 同一个操作（如删除）在不同页面的位置/样式完全不同 | 缺少规范约束 | 建立"删除操作"的统一模式：位置、确认流程、视觉反馈都保持一致 |

### "状态管理混乱" 问题诊断

| 症状 | 最可能的状态分层错误 | 修复 |
|---|---|---|
| 数据库有数据，但列表删除后，重新点开还是老数据 | 从持久化状态读数据，但派生逻辑缓存了结果 | 确保派生状态（filtered items）每次都重新计算，不要 cache；如需 cache，应该由 Provider 管理，不在 Widget 里 |
| SnackBar 或加载动画显示，用户返回后重新进页面，又显示一遍 | Ephemeral State（一次性）被错误地存在了持久化数据里 | SnackBar/loading/dialog 状态只应存在 Page State 的本次 build 周期，消费后立即清除；不应进数据库或 SharedPreferences |
| 筛选条件在数据库记录中出现，新建记录时条件被意外应用 | 违反 3.2 规则：筛选污染了业务数据 | 筛选条件只存 Page State，不存业务实体；业务表只记录事实，不记录"当前如何查看" |
