# Flutter 项目质检方法论

> 判断一个 Flutter 项目的架构、状态、组件和代码组织是否支撑持续迭代,还是在走向屎山。

```text
Flutter 项目质检
├── Scope Clarity：边界清晰度
│   └── 判断当前版本边界是否清楚,是否有功能膨胀或范围漂移
├── Architecture Integrity：架构完整性
│   └── 检查职责边界是否清晰,是否有跨层调用,依赖方向是否合理
├── State Management：状态管理
│   └── 判断状态归属是否清晰,是否混淆不同生命周期的状态
├── Component Quality：组件质量
│   └── 检查组件职责是否单一,是否过度自定义或过度抽象
├── Code Organization：代码组织
│   └── 判断目录结构、文件职责、命名是否清晰可预测
├── Configurability：可配置性
│   └── 检查是否有硬编码泛滥,主题是否统一管理,是否有魔法数字
├── Maintainability：可维护性
│   └── 判断是否容易定位、修改、扩展,是否有明显技术债
└── Conclusion：质检结论
    └── 给出问题优先级和优化方向建议
```

## 核心命题

Flutter 项目质检方法论处理的不是"Flutter 代码写得对不对",而是"这个 Flutter 项目的架构、状态、组件和代码组织是否支撑持续迭代,还是在走向屎山"。

Flutter 项目的常见问题通常不是缺少功能,而是质量腐化:架构边界模糊,UI、状态、数据层混在一起;状态管理混乱,不知道状态应该放哪里;组件职责不清,万能组件或过度自定义;硬编码泛滥,尺寸、颜色、文字写死;代码组织混乱,找不到对应文件;技术债累积,临时方案变成长期实现。

这套方法论的任务,是系统化判断 Flutter 项目的质量状态,识别哪些地方会阻碍迭代、容易出错或难以维护,并给出优化方向。

它的核心转化是:

```text
Flutter 项目当前状态 -> 架构完整性判断 -> 状态管理判断 -> 组件质量判断
-> 代码组织判断 -> 可维护性判断 -> 问题清单与优化方向
```

这里的"质检",不是挑语法错误,也不是统一代码风格,而是判断项目是否能持续推进,还是需要重构。

## 方法论节点

Flutter 项目质检不是逐文件检查,而是围绕架构、状态、组件、组织和可维护性进行系统判断。

### Scope Clarity：边界清晰度

判断当前版本边界是否清楚。

Flutter 项目容易在迭代中范围膨胀:最初定的是 MVP,但实现过程中不断加功能,导致首版迟迟无法收敛;或者概念图里的"未来能力"混进了当前实现,导致代码复杂度超出必要范围。

质检时应判断:

```text
当前版本边界是否明确
是否有明确延后的功能却已经开始实现
是否有概念图里的"未来能力"污染了当前代码
是否能清楚说明"这一版做什么、不做什么"
是否有功能膨胀导致核心路径不清
```

如果团队不能快速说明当前版本边界,说明 `Scope Clarity` 有问题。

### Architecture Integrity：架构完整性

检查职责边界是否清晰。

Flutter 应用不是一组页面文件。好的架构应该让 UI、状态、数据、路由、主题和平台能力各司其职,不互相兜底。

质检时应判断:

```text
是否有清晰的分层结构
UI 层是否只负责展示和组合
状态层是否只负责业务状态
数据层是否只负责读写和转换
路由层是否只负责页面入口
主题层是否只负责复用样式
是否有跨层调用(如 UI 直接操作数据库)
依赖方向是否合理(如数据层依赖 UI 层)
是否有某个文件或模块承担了多个职责
```

**架构完整性的典型问题**:

| 问题类型 | 表现 | 后果 |
|---------|------|------|
| 跨层调用 | Widget 里直接写 SQL 查询 | 无法测试,无法复用,改数据库要改 UI |
| 职责混淆 | 页面文件里既有 UI 也有数据逻辑也有路由 | 无法维护,改一处影响多处 |
| 依赖倒置 | 底层模块依赖上层模块 | 无法复用底层,循环依赖 |
| 无分层 | 所有代码放在 `lib/` 根目录 | 找不到文件,不知道代码归属 |

如果改一个数据源需要改遍所有 UI,说明 `Architecture Integrity` 有问题。

#### 技术栈适配性

判断当前技术栈是否匹配项目规模和复杂度。

Flutter 项目的技术栈选择会直接影响架构设计。随着项目规模增长,技术栈需要相应演进。技术栈过重会增加复杂度,技术栈过轻会限制扩展。

**技术栈演进判断**:

| 技术领域 | 小型项目 (< 10 页面) | 中型项目 (10-30 页面) | 大型项目 (30+ 页面) |
|---------|---------------------|---------------------|-------------------|
| 状态管理 | `StatefulWidget` | `Provider` / `Riverpod` | `Bloc` / `Redux` |
| 数据持久化 | `SharedPreferences` | `Hive` / `Sqflite` | `Isar` / `Drift` |
| 网络层 | `http` 包 | `Dio` + 拦截器 | `GraphQL` / `gRPC` |
| 路由 | `Navigator.push` | `go_router` | 深层链接 + 动态路由 |
| 依赖注入 | 手动传参 | `Provider` / `get_it` | 完整 DI 容器 |

**质检时应判断**:

```text
当前技术栈是否匹配项目规模?
是否有小项目使用了过重的技术栈?
是否有大项目仍在使用过轻的技术栈?
技术栈是否阻碍了功能实现或性能优化?
是否有明确的技术栈演进计划?
技术栈切换的成本和收益是否清楚?
```

**技术栈过重的表现**:

| 场景 | 表现 | 后果 |
|------|------|------|
| 状态管理 | 10 页面项目用 Bloc,写了大量 Event/State 但只是简单 UI 切换 | 增加复杂度,新人学习成本高 |
| 架构设计 | 5 页面项目配置了完整 DI 容器和 6 层架构 | 为了架构而架构,维护成本高 |
| 网络层 | 简单 API 调用引入 GraphQL 和复杂缓存策略 | 增加调试难度,收益不明显 |

**技术栈过轻的表现**:

| 场景 | 表现 | 后果 |
|------|------|------|
| 状态管理 | 30 页面项目仍用 `StatefulWidget`,状态管理混乱 | 状态难以共享,重复代码多 |
| 网络层 | 所有 API 调用都写在页面里,没有统一处理 | 无法统一错误处理、重试、缓存 |
| 路由 | 复杂跳转逻辑仍用 `Navigator.push`,无法管理路由栈 | 深层链接无法实现,路由混乱 |

#### 抽象层级适配性

判断当前抽象层级是否合理,既不过度抽象也不抽象不足。

抽象应服务业务,而不是增加复杂度。小型项目不需要复杂架构,但必须有基本分层;大型项目需要完整架构,但不应过早引入不必要的复杂度。

**抽象层级判断**:

| 项目规模 | 合适的抽象 | 过度抽象 | 抽象不足 |
|---------|-----------|---------|---------|
| 小型 (< 10 页面) | UI/Data 分离,简单 State | 引入 DDD,每个字段都写 Cubit | UI 里直接写 SQL,没有任何分层 |
| 中型 (10-30 页面) | Provider + Repository 层 | 完整 Clean Architecture,大量接口 | 所有逻辑都在 Widget,没有复用 |
| 大型 (30+ 页面) | Bloc + 完整分层 + DI | 过早微服务化,模块间 RPC 调用 | 没有模块划分,所有代码混在一起 |

**质检时应判断**:

```text
分层是否清晰但不过度?
是否有为了架构而架构的过度设计?
是否有明显缺少的抽象层?
抽象是否服务业务,还是增加复杂度?
新人是否能理解当前抽象层级?
是否有重复代码但没有抽象?
是否为了"统一"而过度抽象?
```

**过度抽象的表现**:

| 场景 | 表现 | 问题 |
|------|------|------|
| 简单功能 | 一个计数器页面,写了 Repository、UseCase、Entity、DTO 等 6 层 | 增加理解成本,简单改动要改 6 个文件 |
| 组件设计 | 为了"统一",把所有原生组件都包装成自定义组件 | 丢失原生能力,维护成本高 |
| 架构设计 | 小项目引入事件总线、领域事件、CQRS | 复杂度远超业务需求 |

**抽象不足的表现**:

| 场景 | 表现 | 问题 |
|------|------|------|
| 职责混淆 | UI 层和数据层混在一起,Widget 里写 API 调用 | 无法测试,无法复用 |
| 重复代码 | 3 个页面各自实现了相似的列表逻辑 | 维护成本高,不一致风险 |
| 缺少抽象 | 没有任何业务逻辑层,所有代码都在 Widget | 无法单独测试业务逻辑 |

**合适抽象的判断标准**:

```text
能否在 10 分钟内向新人解释清楚分层逻辑?
改一个功能是否需要改动过多文件(如超过 5 个)?
是否存在明显的重复代码但没有抽象?
抽象是否真的降低了复杂度,还是增加了复杂度?
是否能快速定位业务逻辑在哪里?
```

### State Management：状态管理

判断状态归属是否清晰。

Flutter 的复杂度经常来自状态混淆。持久化数据、页面状态、组件局部状态、派生状态和一次性交互状态如果混在一个地方,后续会很快变得难以维护。

质检时应判断:

```text
是否区分了持久化数据、页面状态、组件状态、派生状态
状态是否放在了合理的层级
是否有应该共享的状态却无法共享(如所有页面各自维护用户信息)
是否有应该局部的状态却放在了全局
是否有状态冗余(同一份数据在多处维护)
状态流转是否可追踪
状态变化是否能触发 UI 更新
```

**状态管理的典型问题**:

| 问题类型 | 表现 | 后果 |
|---------|------|------|
| 全部放页面 | 所有状态都在 `StatefulWidget` 里 | 页面间无法共享,状态无法持久化 |
| 混淆生命周期 | 持久化数据和临时 UI 状态放一起 | 不知道什么该存什么不该存 |
| 状态冗余 | 用户信息在每个页面各维护一份 | 更新时要改遍所有页面,容易不一致 |
| 派生状态存储 | 把计算结果当原始状态存起来 | 容易不同步,增加维护成本 |

如果改一个数据要同步更新多个地方,说明 `State Management` 有问题。

### Component Quality：组件质量

检查组件职责是否单一。

Flutter 组件容易出现两个极端:过度自定义(应该用原生组件却自己写)或过度抽象(万能组件,100 个参数)。

质检时应判断:

```text
是否优先使用了原生 Material/Cupertino 组件
是否有不必要的自定义组件
是否有过度抽象的"万能组件"
组件职责是否单一
组件参数是否过多(超过 10 个?)
组件是否容易理解和使用
组件是否有明确的适用场景
是否有重复的组件实现
```

**组件质量的典型问题**:

| 问题类型 | 表现 | 后果 |
|---------|------|------|
| 过度自定义 | 为了改个颜色自己写整个按钮组件 | 维护成本高,丢失原生能力(如无障碍) |
| 万能组件 | 一个组件处理所有场景,100 个配置参数 | 无法理解,无法维护,改一处影响所有 |
| 职责不清 | 一个组件既负责数据获取也负责 UI 展示 | 无法复用,无法测试 |
| 重复实现 | 多个页面各自实现了类似的卡片组件 | 维护成本高,样式不一致 |

**组件复用判断**:

```text
标准 UI 模式优先查原生组件库
复杂交互优先查第三方库
只有原生和第三方都不满足时才自定义
自定义组件应保持单一职责
组件参数应限定在必要范围
```

#### 过度自定义检查

判断是否有本该用现成组件或库却自己实现的问题。

Flutter 项目容易出现过度自定义,尤其是在不理解原生实现方式时,容易把简单需求搞复杂。

**质检时应判断**:

```text
是否有本该用原生组件却自己实现的?
是否有本该用第三方库却自己写的?
实现前是否查过原生组件文档?
是否因为需求描述不准确导致实现偏离?
是否理解了视觉效果的真实实现方式?
```

**视觉欺骗性导致的过度实现**:

前端开发中,很多视觉效果具有欺骗性。如果不理解原生实现方式,容易误判为"需要自定义",导致过度实现。

| 视觉效果 | 错误理解 | 正确实现 | 过度实现的后果 |
|---------|---------|---------|--------------|
| 透明导航栏 + 滚动 | "让文字出现在导航栏下方" | `extendBodyBehindAppBar: true` | 自己处理层级、滚动、安全区域,复杂且易出错 |
| 卡片阴影 | "需要自己画阴影" | `Card` 组件自带 `elevation` | 自己用 `CustomPaint` 画,性能差且不一致 |
| 底部弹出选择 | "需要自己实现弹层" | `showModalBottomSheet` | 自己处理动画、遮罩、手势,易出错 |
| 列表滑动删除 | "需要自己处理手势" | `Dismissible` 组件 | 自己写手势识别和动画,复杂且不流畅 |
| 下拉刷新 | "需要自己实现刷新逻辑" | `RefreshIndicator` | 自己处理下拉检测、加载状态,不符合平台规范 |

**过度自定义的判断标准**:

```text
实现前是否先查 Material/Cupertino 组件文档?
是否先在 pub.dev 搜索相关功能的第三方库?
需求描述是否准确反映了视觉效果的实现方式?
是否因为"看起来需要自定义"就直接动手写?
自定义实现是否比原生方案更复杂?
```

**避免过度自定义的流程**:

```text
1. 理解视觉效果的本质,而不是表面描述
2. 查原生组件文档,确认是否有直接支持
3. 查第三方库,确认是否有成熟方案
4. 只有确认没有现成方案时,才考虑自定义
5. 自定义时,优先组合现有组件,而不是从零实现
```

如果新人看组件代码半小时还不知道怎么用,说明 `Component Quality` 有问题。

### Code Organization：代码组织

判断目录结构、文件职责、命名是否清晰可预测。

Flutter 项目容易在迭代中目录混乱:所有文件放一个目录,或者目录很多但找不到文件应该放哪里,或者文件命名不准确导致无法快速定位。

质检时应判断:

```text
目录结构是否清晰
是否有明确的模块划分
文件职责是否单一
文件命名是否准确表达内容
是否容易找到对应文件
是否有巨型文件(超过 500 行?)
是否有职责不清的文件
是否有临时文件或废弃文件未清理
```

**代码组织的典型问题**:

| 问题类型 | 表现 | 后果 |
|---------|------|------|
| 扁平结构 | 所有文件放 `lib/` 根目录 | 找不到文件,不知道归属 |
| 过深嵌套 | 目录层级超过 5 层 | 路径太长,难以导航 |
| 巨型文件 | 单个文件超过 1000 行 | 难以理解,难以修改 |
| 命名不准确 | `page1.dart`、`utils.dart`、`common.dart` | 不知道里面是什么 |
| 废弃文件未清理 | `xxx_old.dart`、`xxx_backup.dart` | 不知道哪个是当前版本 |

**目录组织参考**:

```text
lib/
├── core/          # 核心基础设施(路由、主题、常量)
├── data/          # 数据层(API、数据库、模型)
├── features/      # 功能模块,按业务划分
│   ├── auth/
│   ├── home/
│   └── profile/
├── shared/        # 共享组件和工具
└── main.dart
```

如果找一个功能的实现文件要翻 5 分钟,说明 `Code Organization` 有问题。

### Configurability：可配置性

检查是否有硬编码泛滥。

Flutter 项目容易硬编码尺寸、颜色、文字,导致无法适配不同机型、无法换主题、无法国际化。

质检时应判断:

```text
是否有大量硬编码尺寸(如 width: 375)
是否有大量硬编码颜色(如 Color(0xFF123456))
是否有硬编码文字(如 Text('标题'))
是否有魔法数字(如 padding: 8 但不知道为什么是 8)
主题是否统一管理
尺寸是否响应式计算
颜色是否从主题获取
文字是否支持国际化
```

**可配置性的典型问题**:

| 问题类型 | 表现 | 后果 |
|---------|------|------|
| 硬编码尺寸 | `Container(width: 375, height: 50)` | 在不同机型上显示异常 |
| 硬编码颜色 | `Color(0xFF123456)` 散布在所有文件 | 换主题要改遍所有文件 |
| 硬编码文字 | `Text('登录')` | 无法国际化 |
| 魔法数字 | `padding: EdgeInsets.all(8)` | 不知道为什么是 8,不敢改 |

**可配置性改进方向**:

| 场景 | ❌ 硬编码 | ✅ 可配置 |
|------|---------|---------|
| 尺寸 | `width: 375` | `width: MediaQuery.of(context).size.width * 0.9` |
| 颜色 | `Color(0xFF123456)` | `Theme.of(context).primaryColor` |
| 文字 | `Text('标题')` | `Text(AppStrings.title)` |
| 间距 | `padding: EdgeInsets.all(8)` | `padding: EdgeInsets.all(AppSpacing.small)` |

如果换个主题颜色要改 100 个文件,说明 `Configurability` 有问题。

### Maintainability：可维护性

判断是否容易定位、修改、扩展。

可维护性是综合指标。一个可维护的项目应该让新人能快速理解、让修改不会引入新问题、让扩展不需要大范围重构。

质检时应判断:

```text
新人是否能快速理解项目结构
是否容易定位问题
是否容易修改功能
修改是否容易引入新问题
是否容易扩展新功能
是否有明显的技术债
是否有临时方案变成长期实现
是否有大量 TODO 注释
是否有大量被注释掉的代码
文档是否和代码同步
```

**可维护性的典型问题**:

| 问题类型 | 表现 | 后果 |
|---------|------|------|
| 改一处影响多处 | 改一个数据结构要改 10 个文件 | 不敢改,容易出错 |
| 无法定位问题 | 出错了不知道是哪个文件的问题 | 调试时间长 |
| 技术债累积 | 大量 TODO 和临时方案 | 越来越难维护 |
| 文档不同步 | 注释和代码不一致 | 误导新人 |
| 僵尸代码 | 大量注释掉的代码 | 不知道能不能删 |

**可维护性检查清单**:

```text
能否在 5 分钟内向新人解释清楚项目结构?
改一个功能是否会影响其他功能?
是否有明确的技术债清单?
是否有定期重构计划?
文档是否和代码同步?
是否有单元测试覆盖核心逻辑?
```

如果团队成员都说"这个项目不敢动",说明 `Maintainability` 有问题。

### Conclusion：质检结论

给出问题优先级和优化方向。

质检不只输出问题清单,应给出哪些问题优先解决、哪些问题可以延后、应该采用什么优化策略。

结论应包括:

```text
问题清单:按维度列举发现的问题
优先级:哪些问题会阻碍迭代,应优先解决
优化方向:建议采用什么策略(重构、增量改进、文档补充等)
风险评估:当前状态下继续开发的风险
下一步:具体的可执行建议
```

**优先级判断**:

| 优先级 | 判断标准 | 处理方式 |
|--------|---------|---------|
| P0 - 立即处理 | 阻碍当前开发,继续推进会累积更多债务 | 停止新功能,优先重构 |
| P1 - 近期处理 | 影响开发效率,但不阻碍推进 | 在下个迭代周期处理 |
| P2 - 计划处理 | 有改进空间,但不紧急 | 纳入技术债清单,按优先级排期 |
| P3 - 观察 | 当前可接受,未来可能需要优化 | 记录备案,定期复查 |

**优化策略**:

| 问题类型 | 建议策略 |
|---------|---------|
| 架构混乱 | 先建立分层边界,再逐模块迁移 |
| 状态混乱 | 先定义状态归属规则,再重构状态管理 |
| 组件质量差 | 先建立组件库,再替换临时实现 |
| 硬编码泛滥 | 先统一主题,再批量替换硬编码 |
| 目录混乱 | 先定义目录规范,再逐步迁移 |
| 技术债多 | 建立技术债清单,每个迭代分配 20% 时间还债 |

## 判断标准

Flutter 项目质检是否完成,只看一个标准:

**是否已经明确指出项目在架构、状态、组件、组织等维度的问题,并给出优先级和可执行的优化方向。**

如果只列出"架构不清晰""代码很乱"这类泛泛评价,质检没有完成。

如果能具体指出"UI 层和数据层混在一起,建议先定义分层边界,再把数据逻辑迁移到独立的 repository 层",质检已经成立。

如果能判断"当前状态下继续开发风险较高,建议优先重构 XX 模块",质检已经可以指导下一步行动。

## Few-shot Learning 示例组

### 正例：定期质检防止腐化

一个 Flutter 项目开发了 3 个月,功能在持续增加,但团队感觉开发越来越慢。使用质检方法论后发现:

**问题清单**:
- 架构: UI 层和数据层混在一起,页面文件既有 Widget 也有 API 调用
- 技术栈: 已有 15 个页面,但仍在用 `StatefulWidget`,状态管理混乱(技术栈过轻)
- 抽象层级: 抽象不足,没有任何业务逻辑抽象,所有代码都在 Widget
- 状态: 所有状态都放在页面里,导致用户信息在每个页面各维护一份
- 组件: 有 3 个页面各自实现了类似的卡片组件,样式不一致
- **过度自定义**: 自己实现了底部弹层、下拉刷新,实际上 `showModalBottomSheet` 和 `RefreshIndicator` 原生支持
- 硬编码: 尺寸和颜色全部硬编码,在不同机型上显示异常
- 组织: 所有文件放在 `lib/` 根目录,找文件要翻半天

**优先级判断**:
- P0: 架构混乱 + 技术栈过轻 + 抽象不足,导致无法测试,改一个数据源要改遍所有页面
- P1: 状态冗余导致数据不一致,用户信息更新后有的页面没刷新
- P2: 硬编码导致适配问题,但不阻碍开发
- P3: 目录组织可优化,但当前规模尚可接受

**优化方向**:
1. 立即:
   - 建立分层架构,定义 UI、State、Data 职责边界
   - 引入 Provider/Riverpod,替换 StatefulWidget(技术栈演进)
   - 抽取 Repository 层,分离业务逻辑和 UI(建立合适抽象)
2. 近期: 统一用户信息状态,建立全局状态管理
3. 计划: 建立主题系统,替换硬编码
4. 观察: 定义目录规范,新代码按规范写,旧代码逐步迁移

这个顺序能防止继续累积技术债,同时不阻塞业务推进。

### 反例：只列问题不给方向

一个质检报告只写了:
```text
- 架构不清晰
- 代码很乱
- 状态管理有问题
- 组件质量不高
- 需要重构
```

这种质检没有价值,因为:
- 不知道具体哪里有问题
- 不知道优先级
- 不知道怎么改
- 不知道改到什么程度
- 无法指导下一步行动

正确的质检应该具体指出:
```text
架构问题: `lib/pages/home_page.dart` 里同时包含 UI、状态、API 调用,
应该拆分为 home_page.dart(UI)、home_state.dart(状态)、home_repository.dart(数据)

状态问题: 用户信息在 home_page、profile_page、settings_page 各维护一份,
应该提取为全局 UserState,使用 Provider 或 Riverpod 管理

组件问题: diary_card、mood_card、note_card 实现了相似功能但样式不一致,
应该抽象为通用 ContentCard 组件,通过参数区分展示形态
```

### 反例：视觉欺骗性导致过度实现

一个项目需要实现"透明导航栏 + 内容滚动到导航栏下方"的效果。

**错误理解**:
- 开发者看到视觉效果,描述为"让标题文字出现在导航栏下方"
- 认为这需要自定义实现,因为"原生导航栏是不透明的"
- 开始自己处理层级关系、滚动监听、安全区域适配

**实现过程**:
```dart
// 错误做法:自己实现复杂的层级和滚动
Stack(
  children: [
    // 内容层
    ListView(...),
    // 自定义透明导航栏
    Positioned(
      top: 0,
      child: Container(
        // 手动处理安全区域
        // 手动处理透明度
        // 手动处理滚动监听
        // ...大量自定义代码
      ),
    ),
  ],
)
```

**问题**:
- 复杂度高:需要自己处理安全区域、滚动监听、透明度渐变
- 易出错:滚动性能差,键盘弹出时布局错乱
- 不符合平台规范:丢失了原生导航栏的手势和动画

**正确做法**:

这个视觉效果的本质是"让内容延伸到导航栏下方",Flutter 原生支持:

```dart
Scaffold(
  extendBodyBehindAppBar: true,  // 关键:让 body 延伸到 AppBar 下方
  appBar: AppBar(
    backgroundColor: Colors.transparent,  // 透明背景
    elevation: 0,
  ),
  body: ListView(...),
)
```

只需要 2 行配置,无需自定义。

**这个案例说明**:
- 视觉效果具有欺骗性,表面描述不等于实现方式
- 不理解原生实现方式,容易误判为"需要自定义"
- 实现前应该先查原生组件文档,理解视觉效果的本质
- 过度自定义会把简单需求搞复杂

### 反例：技术栈过重和过度抽象

一个只有 5 个页面的小型日记应用,使用了:
- Bloc 状态管理,每个页面都写了大量 Event、State、Bloc
- 完整的 Clean Architecture,写了 Entity、UseCase、Repository、DataSource 等 6 层
- 引入了 get_it 依赖注入容器,所有依赖都需要注册
- 为了"统一",把所有原生组件都包装成自定义组件

**问题分析**:
- **技术栈过重**: 5 个页面不需要 Bloc,Provider 足够;完整 DI 容器增加复杂度
- **过度抽象**: 简单的 CRUD 操作写了 6 层,改一个字段要改 6 个文件
- **维护成本高**: 新人需要学习 Bloc、Clean Architecture、DI,入门成本高
- **收益不明显**: 复杂度增加了,但项目规模不需要这些

**正确做法**:
- 状态管理: 用 Provider,简单够用
- 分层: UI/State/Data 三层,有 Repository 但不引入完整 Clean Architecture
- 依赖注入: 手动传参或简单的 Provider,不需要完整 DI 容器
- 组件: 优先用原生组件,只在必要时自定义

### 边界例：新项目不需要质检

一个 Flutter 项目刚开始开发,只有 1-2 个页面,代码不到 500 行。

这种场景不需要使用质检方法论,因为:
- 规模太小,问题还没暴露
- 架构还在建立中,不适合质检
- 应该使用"Flutter 项目开发主干方法论"指导开发

只有当项目开发到一定规模(如 5+ 个页面、2000+ 行代码、2+ 个月开发时间),才需要定期质检。

## 误用

Flutter 项目质检不是 lint 检查。

lint 检查关注语法和代码风格,质检关注架构和可维护性。`flutter analyze` 不能替代质检方法论。

Flutter 项目质检也不是逐文件审查。

质检关注系统性问题,不是逐行挑毛病。应该先看整体架构、状态管理、组件质量,再看局部实现。

Flutter 项目质检也不是 Code Review。

Code Review 关注单次改动是否合理,质检关注整体项目质量是否健康。两者可以配合,但不能互相替代。

Flutter 项目质检也不是重构计划。

质检输出问题和方向,重构计划需要进一步细化为具体任务、排期和验收标准。质检是重构的前提,但不是重构本身。

Flutter 项目质检也不是越严格越好。

质检应该区分优先级,而不是要求项目完美。P0 问题必须解决,P3 问题可以接受。过度追求完美会阻碍业务推进。

## 适用边界

这套方法适合用于:

```text
定期质检,防止架构腐化
接手别人的 Flutter 项目,评估质量
项目感觉开始混乱,不知道问题在哪
重构前,诊断当前问题并确定重构范围
Code Review 时系统化检查
```

这套方法不适合用于:

```text
Flutter 基础语法教学
Dart lint 规则配置
0-1 项目起盘(应使用 Flutter 项目开发主干方法论)
单个文件的代码审查
静态 Demo 项目
不到 500 行代码的小项目
```

如果问题是"不知道 Flutter 怎么开始开发",应使用 Flutter 项目开发主干方法论。

如果问题是"某个方法论节点需要单独抽象",应使用方法论新增范式判断是否新增子方法论。

如果问题是"代码格式不统一",应配置 `dart format` 和 lint 规则,不需要用质检方法论。
