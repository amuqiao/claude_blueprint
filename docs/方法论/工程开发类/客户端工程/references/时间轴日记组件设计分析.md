# 时间轴日记组件设计分析

本文用 Python / FastAPI 的分层视角解释 Flutter 时间轴日记组件设计：核心不是“把页面拆成很多组件”，而是把**状态、交互、绘制**三类职责放到正确的位置。

---

## 先建立心智模型：组件树就是对象组合

你的理解方向是对的：Flutter 里每个组件都是一个类，组件树就是对象组合。

```text
HomeScreen
  └─ HomeTimelineViewport
       └─ HomeTimeline
            └─ TimelineBubble
```

这和 Python 里大对象持有小对象完全一样：

```python
class HomeScreen:
    timeline_viewport: HomeTimelineViewport

class HomeTimelineViewport:
    timeline: HomeTimeline

class HomeTimeline:
    bubbles: list[TimelineBubble]
```

但 Flutter 组件比普通 Python 对象多一个重要维度：它不只是在组织数据和逻辑，还要决定**谁持有状态、谁响应交互、谁负责画出来**。

---

## 关键区别：Flutter 组件要同时切三类职责

Python 对象通常只有一个核心职责：

```text
Python 对象 → 数据和逻辑
```

Flutter 组件设计需要额外回答三个问题：

```text
这个组件 → 持有数据/状态吗？
        → 响应用户交互吗？
        → 负责画自己吗？
```

所以不能简单理解成“每个职责一个组件”。更准确的说法是：**每种职责都要找到合适的承载者**。

```text
职责           →  承载者
────────────────────────────────────────────
管数据/状态    →  Controller / StateNotifier
管交互逻辑     →  StatefulWidget
管样式/画图    →  StatelessWidget
```

这里最容易混淆的一点是：`Controller` 不是 `Widget`。它是挂在 Widget 树旁边的普通 Dart 逻辑对象，负责管理状态和业务行为；Widget 树负责把状态变成可交互、可绘制的界面。

---

## 三层职责和三种组件并不冲突

这里容易产生一个误解：前面说“数据层、交互层、样式层”，后面又说 `Controller`、`StatefulWidget`、`StatelessWidget`，它们是不是两套互相冲突的分类？

它们不冲突，只是观察角度不同。

```text
三层职责  →  概念分类：有哪些事情需要有人负责
三种角色  →  实现分工：具体由谁来承担这些事情
```

对应关系可以这样理解：

```text
数据层  →  HomeController   →  纯 Dart 类，不是 Widget
交互层  →  StatefulWidget   →  既持有交互状态，也参与渲染
样式层  →  StatelessWidget  →  只负责画，不存业务状态
```

唯一需要特别说明的是交互层。`HomeTimelineViewport` 既要管理 `ScrollController`、滚动定位和用户手势，也必须通过 `build()` 把自己的子树画出来。

这是 Flutter 和后端分层最大的不同：在后端里，业务逻辑、查询逻辑和响应结构可以拆得更干净；但在 Flutter 里，响应交互的组件通常也必须参与渲染。也就是说，`StatefulWidget` 的职责重心是交互状态，但它不能完全脱离绘制。

所以更准确的说法是：

```text
数据层   →  完全独立，跟画面无关
交互层   →  负责交互逻辑，但必须顺带渲染自己
样式层   →  只负责视觉呈现，完全不管业务逻辑
```

这三层不是三个完全平行、互不接触的抽屉，而是职责重心不同。判断一个类放在哪里，关键不是看它“有没有 build”，而是看它主要变化原因是什么：数据规则变化、交互规则变化，还是视觉样式变化。

---

## StatelessWidget 与 StatefulWidget 的边界

Flutter 组件最关键的区分是 `StatelessWidget` 和 `StatefulWidget`。

| 类型 | 职责 | 类比 |
|---|---|---|
| `StatelessWidget` | 纯绘制，给什么数据就画什么，自己不存状态 | Python 纯函数，没有 `self.state` |
| `StatefulWidget` | 自己持有局部状态，状态变化后触发重新绘制 | Python 类实例，有 `self.state` |

放到时间轴日记里：

```text
TimelineBubble
  → StatelessWidget
  → 只接收日记内容、颜色、圆角、图片等参数
  → 不知道滚动位置，不知道筛选条件，不知道热力图状态

HomeTimelineViewport
  → StatefulWidget
  → 持有 ScrollController
  → 管理滚动位置、定位目标、用户滚动交互

HomeController
  → 普通 Dart 类
  → 管理当前筛选、时间锚点、可见数据
  → 不是 Widget，不负责画界面
```

用 Python 视角类比，可以写成：

```python
class HomeController:
    """管数据和业务状态，不画东西。"""
    state: HomeState


class TimelineViewport:
    """管交互，也会参与 build。"""
    scroll_controller: ScrollController

    def build(self):
        ...


class TimelineBubble:
    """只画东西，不存业务状态。"""
    def build(self, data):
        ...
```

这个边界比“拆组件”本身更重要。拆组件只是手段，真正要避免的是让一个对象同时承担状态、交互、样式三种变化原因。

## 类比 FastAPI 的分层

时间轴日记的组件拆分，本质上和 FastAPI 中 router / service / repository / schema 的分层相通。

| Flutter 角色 | FastAPI 类比 | 职责 |
|---|---|---|
| `HomeScreen` | Router / Endpoint | 组织页面，连接状态和 UI |
| `HomeController` | Service Layer | 管理业务状态，例如筛选、时间锚点 |
| `TimelineAnchorResolver` | Repository / Query | 根据日期锚点找到对应日记条目 |
| `HomeTimelineViewport` | Pagination / Cursor | 管理滚动位置和用户交互 |
| `TimelineCard / TimelineBubble` | Pydantic Schema / Response Model | 纯展示，把数据渲染出来 |

原来的问题在于 `HomeScreen` 容易把三类职责全混在一起，就像把数据库查询、业务逻辑、分页计算和响应结构都写在一个路由函数里。

```python
# 反模式：一个路由函数做了所有事
@app.get("/home")
async def home(db: Session, user_id: int, tag: str, date: str):
    entries = db.query(...)          # 数据查询
    anchor = resolve_anchor(date)    # 业务逻辑
    scroll_offset = calc_offset(...) # 交互/位置细节
    return render_page(...)          # 样式输出
```

Flutter 里对应的反模式就是：页面组件既保存筛选状态，又计算目标日记，又直接控制滚动，还顺手决定气泡样式。

## 时间轴日记的推荐设计

### 1. 数据层：状态与查询分离

`HomeController` 只表达“当前用户想看什么”，不要直接处理具体滚动位置。

```python
class HomeState:
    current_tag: str | None
    time_anchor: date | None
    heatmap_visible: bool
```

`TimelineAnchorResolver` 负责把用户意图转换成具体数据对象。

```python
class TimelineAnchorResolver:
    def resolve(anchor: date) -> EntryId:
        # 找到这一天第一条可见的日记
        ...
```

这样可以把两个问题分开：

```text
“我想看 6 月 3 日”          → 用户意图
“6 月 3 日第一条是 entry_123” → 数据查询结果
```

### 2. 交互层：定位逻辑独立成 Viewport

`HomeTimelineViewport` 是这个设计里最关键的中间层。它不决定数据是什么，也不决定气泡长什么样；它只负责“内容应该出现在屏幕哪里”。

```text
用户点击热力图日期
    ↓
HomeController 更新 time_anchor = 6/3
    ↓
TimelineAnchorResolver 解析出 entry_id = entry_123
    ↓
HomeTimelineViewport.scrollToEntry(entry_123)
    ↓
计算目标位置 = 导航栏底部 + 阅读呼吸间隔
```

类比 FastAPI：这就像把分页游标或 `OFFSET` 计算从业务逻辑里抽出来。路由只表达“我要哪一页”，分页服务负责“这一页从哪里开始”。

这个拆分的价值是：以后要调整定位后的留白、吸附规则、滚动动画，只需要改 `HomeTimelineViewport`，不需要碰数据状态和气泡样式。

### 3. 样式层：TimelineBubble 保持纯 UI

`TimelineBubble` 只负责把输入数据画出来。

```text
TimelineBubble 关心：
  - 圆角半径
  - 气泡尾巴朝向
  - 背景颜色 token
  - 图片网格布局
  - 文本间距

TimelineBubble 不关心：
  - 当前是否按标签筛选
  - 热力图是否展开
  - 当前滚动到了哪里
  - 点击日期后要滚到哪一条
```

类比 Pydantic `ResponseModel`：它只定义输出结构，不负责数据库查询，也不决定业务流程。

## 整体调用链

```text
用户操作：点击热力图日期
        │
        ▼
HomeScreen
  类比 Router：接收用户动作，连接状态与界面
        │
        ▼
HomeController
  类比 Service：更新业务状态
  time_anchor = 6/3
        │
        ▼
TimelineAnchorResolver
  类比 Repository：把日期解析成具体日记
  6/3 → entry_123
        │
        ▼
HomeTimelineViewport
  类比 Pagination / Cursor：管理物理滚动位置
  entry_123 → scroll offset
        │
        ▼
TimelineCard / TimelineBubble
  类比 Response Schema：纯展示
```

这个调用链里，每一层只处理一种变化原因：

```text
数据变了       → Controller / Resolver
交互方式变了   → Viewport
视觉样式变了   → TimelineBubble / TimelineCard
页面组合变了   → HomeScreen
```

## 实践判断标准

### 1. 先问“状态归谁”，再问“组件怎么拆”

组件拆分不是按文件数量拆，而是按变化原因拆。每写一个新组件，先判断它是否需要持有状态。

```text
只接收参数并绘制           → StatelessWidget
需要 ScrollController      → StatefulWidget
需要管理业务状态           → Controller / StateNotifier
```

### 2. 交互参数要语义化

“定位后留一点空白”不要散落成硬编码数字。它应该是一个有业务语义的设计变量。

```text
TimelineRhythm.readingGap
TimelineRhythm.anchorTopPadding
```

这和后端里不用到处写 `LIMIT 20`，而是统一使用 `settings.PAGE_SIZE` 是同一个道理。

### 3. 先定义组件接口，再写内部实现

`HomeTimelineViewport` 对外应该暴露清晰语义，而不是泄漏内部滚动细节。

```text
scrollToEntry(entryId)
scrollToDate(date)
```

外层只表达意图：滚到某条日记或某一天。至于怎么计算 offset、怎么避开导航栏、动画多长，属于 Viewport 内部实现。

### 4. 样式 token 集中管理

颜色、间距、字号、圆角这些视觉变量应集中管理，避免散落在各个组件里。

```text
TimelineColors.bubbleBackground
TimelineSpacing.cardGap
TimelineRadius.bubble
```

这样 `TimelineBubble` 仍然是纯 UI，但不会变成一堆难以维护的魔法数字。

---

## 总结

这篇设计的重点不是“Flutter 里每个东西都是组件”，而是：**Widget 树负责组织界面，Controller 负责业务状态，StatefulWidget 承担局部交互状态，StatelessWidget 负责纯绘制**。

所以更准确的理解是：

```text
HomeController        → 管数据，不是 Widget
HomeTimelineViewport  → 管滚动交互，是 StatefulWidget
TimelineBubble        → 管视觉呈现，是 StatelessWidget
```

这就是 Flutter 版本的关注点分离。它和 FastAPI 分层的底层逻辑一致，只是前端额外多了一个后端通常不需要处理的维度：**物理位置和交互状态**。
