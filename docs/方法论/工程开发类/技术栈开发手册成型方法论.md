# 技术栈开发手册成型方法论

> 把一种语言、框架或技术栈的开发经验，组织成认知清楚、架构稳定、选型有边界、规范可判断、AI 可执行的长期开发手册。

```text
技术栈开发手册成型方法论
├── Cognition：认知模型
│   └── 回答“应该如何理解这门技术”
├── Architecture：架构模型
│   └── 回答“代码应该如何组织”
├── Selection：技术选型
│   └── 回答“默认用什么工具，何时演进”
├── Standard：工程规范
│   └── 回答“具体问题应该如何判断”
└── Protocol：AI 实现协议
    └── 回答“如何让 AI 按正确边界写代码”
```

## 核心命题

技术栈开发手册不是工具清单、目录说明、代码规范或 Prompt 模板的简单拼接，而是帮助开发者和 AI Agent 用同一套工程心智进入项目的长期上下文。

当一个技术栈会被反复用于项目开发时，真正需要沉淀的不是“某个功能怎么写”，而是：

```text
如何理解这门技术
如何组织代码
默认采用哪些成熟范式
遇到具体工程问题如何判断
如何把需求稳定交给 AI 实现
```

这套方法论的核心转化是：

```text
零散开发经验 -> 技术栈认知模型 -> 架构归位规则 -> 选型演进边界 -> 工程判断标准 -> AI 实现协议
```

它适合用来编写 `Flutter`、`FastAPI`、`React`、`SwiftUI`、`Node.js` 等技术栈的开发手册。不同技术栈的具体工具、目录和规范会变化，但手册成型的主干节点应保持稳定。

## 方法论节点

### Cognition：认知模型

先建立理解这门技术的正确心智模型。

认知模型负责回答“这门技术本质上是什么”，避免开发者或 AI Agent 用错误范式写代码。很多工程问题不是因为不会调用 API，而是因为把一个技术栈误理解成另一个技术栈。

认知模型通常回答：

```text
这门技术主要解决什么问题
它最重要的抽象是什么
它的运行模型或渲染模型是什么
它适合什么场景，不适合什么场景
哪些其他技术栈经验不能机械迁移
```

不同技术栈的认知模型应压缩到能改变判断的核心差异上，而不是展开通用术语背景。

例如：

| 技术栈 | 认知模型重点 |
| --- | --- |
| Flutter | 声明式 UI、Widget Tree、状态驱动渲染、跨平台适配 |
| FastAPI | 类型驱动 API、依赖注入、请求生命周期、异步与同步边界 |
| React | 组件树、单向数据流、状态驱动渲染、副作用管理 |

如果手册一开始就进入目录、库和代码示例，而没有先说明技术栈的核心抽象，后续规范很容易退化成零散经验。

### Architecture：架构模型

规定代码如何分层、归位和依赖。

架构模型负责回答“代码应该放在哪里、谁可以依赖谁、模块边界如何判断”。它不是展示当前目录，而是把目录背后的职责边界和依赖方向写清楚。

架构模型通常回答：

```text
推荐目录结构是什么
每个目录承担什么职责
业务模块如何切分
基础设施能力放在哪里
UI、接口、业务逻辑、数据访问之间如何依赖
哪些跨层调用或循环依赖禁止出现
```

目录结构应服务架构判断，不应为了看起来完整而提前制造复杂分层。对于小项目，可以写轻量架构；对于持续演进项目，应明确哪些目录是主干，哪些目录是复杂度触发后再出现的补强结构。

架构模型的重点不是统一所有技术栈的目录名，而是统一判断方式：

| 技术栈 | 架构模型重点 |
| --- | --- |
| Flutter | `app/`、`core/`、`features/`，以及 `presentation/domain/data` 分层 |
| FastAPI | `routers/`、`schemas/`、`services/`、`repositories/`、`models/` 的职责边界 |
| React | `features/`、`components/`、`hooks/`、`services/`、`routes/` 的模块归位 |

如果手册只列目录树，没有说明目录职责和依赖方向，说明架构模型没有完成。

### Selection：技术选型

明确默认工具链、成熟范式和演进边界。

技术选型负责回答“默认用什么、为什么用、什么时候升级或替换”。它不是依赖清单，而是把工具选择和项目复杂度、团队维护成本、平台成熟范式绑定起来。

技术选型通常回答：

```text
默认框架和核心库是什么
状态管理、路由、数据访问、测试、构建分别使用什么
优先使用官方能力还是社区包
什么时候保持轻量实现
什么时候允许升级到更复杂方案
哪些依赖暂不引入或明确禁用
```

选型层应贯彻成熟范式优先原则。通用问题优先使用目标技术栈已有的框架能力、组件体系和社区验证方案；自定义实现只用于产品差异、核心业务规则、关键交互差异或领域模型表达。

技术选型要写清演进触发条件，而不是只写当前版本使用什么。没有触发条件的选型容易变成静态快照，后续项目复杂度上升时仍然缺少判断依据。

典型触发信号包括：

```text
重复样板代码开始扩散
跨模块状态同步变复杂
本地数据与远程数据来源开始并存
测试、构建或生成动作开始重复
现有轻量方案开始让职责边界模糊
```

### Standard：工程规范

沉淀具体问题的判断标准。

工程规范负责回答“遇到具体开发问题时怎么判断”。它不是把所有代码风格写满，而是优先沉淀高频、易误判、会影响可维护性的规则。

工程规范通常覆盖：

```text
命名与文件拆分
模块边界与依赖方向
状态管理或数据流规则
错误处理与异常表达
测试与验证入口
性能、可访问性、安全或兼容性边界
文档与注释要求
```

规范层应尽量写成判断句，而不是偏好句。

弱规范通常是：

```text
代码要清晰
组件不要太大
接口要规范
测试要充分
```

有效规范应说明判断信号：

```text
当一个组件同时承担布局、数据请求、业务计算和副作用协调时，应拆分职责。
当 API 返回结构需要被多个前端页面消费时，应沉淀稳定响应模型，而不是让页面各自解析。
当状态只服务单个交互控件时，不应升级为全局状态。
```

工程规范不追求一次性覆盖所有细节。手册应优先写会改变实现路径、评审结论或 AI 输出质量的规则。

### Protocol：AI 实现协议

把需求转成 AI 可执行、可验收的代码任务。

AI 实现协议负责回答“如何告诉 AI 写代码”。它不是普通 Prompt 模板，而是让 AI 在明确上下文、边界、涉及文件、禁止行为和验收标准后执行工程任务。

AI 实现协议通常回答：

```text
任务应如何描述
必须提供哪些上下文
涉及哪些文件或模块
哪些行为明确禁止
完成后如何验收
需要运行哪些验证命令
无法验证时如何说明风险
```

推荐任务格式：

```text
目标：这次要完成什么
上下文：相关产品意图、架构位置和已有约束
涉及文件：优先修改哪些文件，哪些文件不要碰
边界：本次只做什么，不做什么
不要做：禁止重构、禁止引依赖、禁止改接口等
验收标准：用户如何判断完成
验证命令：完成后至少运行什么
```

AI 实现协议应和前四层相互引用。认知模型防止 AI 用错技术范式，架构模型防止 AI 放错目录，技术选型防止 AI 私自引依赖，工程规范防止 AI 生成不可维护实现。

如果一份开发手册只写“请按项目规范实现”，但没有给出任务格式、边界表达和验收方式，说明它还不能稳定支撑 AI 协作。

## 判断标准

技术栈开发手册是否成型，只看一个标准：

**读者或 AI Agent 是否能基于这份手册，用正确技术范式、清晰架构边界、稳定工具选择、明确工程规则和可验收任务协议，持续完成同类项目开发。**

如果读者看完后只知道用了哪些库，但不知道为什么这样组织代码，说明手册没有完成。

如果 AI 仍然需要在每次任务中重新猜目录、状态管理、依赖选择和验证命令，说明手册没有完成。

如果换到 `Flutter`、`FastAPI`、`React` 等不同技术栈时，只需要替换具体工具和规则，而五层主干仍然成立，说明方法论已经成立。

完成后的手册至少应让读者看清：

```text
这门技术应如何思考
项目代码应如何组织
默认工具链是什么
复杂度上升时如何演进
具体工程问题如何判断
AI 应如何接收和完成任务
```

## Few-shot Learning 示例组

### 正例：Flutter 开发手册

一个 Flutter 项目面向 `iOS` 和 `macOS`，需要让开发者和 AI Agent 稳定完成页面、状态、路由、数据持久化和验证任务。

合适手册主干：

```text
Cognition：声明式 UI、Widget Tree、状态驱动渲染、平台适配
Architecture：app/core/features，presentation/domain/data 分层
Selection：Riverpod、go_router、Drift、intl、Flutter 原生组件优先
Standard：Widget 拆分、Provider 边界、主题 token、响应式布局、无障碍
Protocol：Screen、Widget、Provider、Repository、Widget Test 的任务模板
```

这份手册不需要展开每个 Flutter API 的用法，而应说明 Flutter 项目中哪些判断不能交给临时实现决定。

### 正例：FastAPI 开发手册

一个 FastAPI 项目需要稳定接入 API、数据库、后台任务、错误处理和测试。

合适手册主干：

```text
Cognition：类型驱动 API、依赖注入、请求生命周期、同步/异步边界
Architecture：routers/schemas/services/repositories/models 分层
Selection：Pydantic、SQLAlchemy、Alembic、pytest、httpx、统一日志
Standard：DTO 边界、事务边界、错误模型、鉴权策略、迁移规则
Protocol：Router、Schema、Service、Migration、API Test 的任务模板
```

这份手册应帮助 AI 判断接口层、业务层和数据访问层的职责，而不是只给出 FastAPI 装饰器示例。

### 反例：把依赖清单包装成开发手册

一份文档只写：

```text
Flutter 使用 Riverpod、go_router、Drift
FastAPI 使用 Pydantic、SQLAlchemy、Alembic
React 使用 React Router、TanStack Query、Zustand
```

问题：

```text
没有建立技术栈认知模型
没有说明目录职责和依赖方向
没有说明为什么选择这些工具
没有给出复杂度上升时的演进边界
不能指导 AI 如何执行具体任务
```

这种文档更像技术栈速查，不是开发手册。

### 边界例：具体项目实施手册

一个项目已经确定所有技术选型和目录结构，只需要说明本项目如何启动、配置、部署和排障。

判断：

```text
如果目标是沉淀某种技术栈的可迁移开发方法，使用本方法论。
如果目标是说明某个项目如何运行、部署、维护，应写项目级 README、部署文档或发布手册。
```

项目实施手册可以引用技术栈开发手册，但不应把所有项目配置细节塞回技术栈方法论。

## 误用

技术栈开发手册成型方法论不是“所有技术栈统一目录模板”。

不同技术栈的成熟范式不同，目录结构、测试方式、构建工具和模块边界也不同。方法论统一的是手册成型的判断主干，不是强行统一文件夹名称。

技术栈开发手册成型方法论也不是工具百科。

手册不应逐条解释框架 API、命令参数或库函数。只有当某个工具选择会改变架构、边界、演进或 AI 实现方式时，才需要进入手册主干。

技术栈开发手册成型方法论也不是代码规范大全。

规范层应优先沉淀会影响判断的高频规则，不要把格式化、命名偏好和低风险风格细节堆满整篇文档。能交给 formatter、linter 或框架默认规则处理的内容，应下沉到工具配置或项目规范。

技术栈开发手册成型方法论也不是 Prompt 模板集合。

AI 实现协议需要任务边界、上下文、禁止行为和验收标准。单纯收集 Prompt 句式不能保证 AI 理解项目架构，也不能替代前四层的工程上下文。

技术栈开发手册成型方法论也不是具体项目开发主干方法论。

它负责建立某个技术栈的长期开发手册；项目从 0-1 起盘、模块切片、版本收敛、部署入口和运行排障，仍应回到对应工程开发方法论。

## 适用边界

这套方法适合处理：

```text
需要为一种技术栈建立长期开发手册
同一技术栈会反复用于多个项目
开发者和 AI Agent 需要共享工程上下文
零散经验已经开始重复出现
项目经常因为目录、选型、状态、测试或 AI 任务边界产生分歧
```

这套方法不适合处理具体库 API 教程、单个项目的临时开发记录、一次性任务复盘、纯依赖清单、纯 Prompt 模板、生产部署操作手册或完整 CI/CD 流水线设计。

如果问题是“一个新项目应该如何起盘”，应使用全栈项目开发主干方法论或对应技术栈的项目开发主干方法论。

如果问题是“一个服务如何从 0-1 建立架构龙骨”，应使用新服务架构龙骨与能力装配方法论。

如果问题是“项目如何启动、验证、生成和构建”，应使用项目开发入口设计方法论。

如果问题是“稳定部署路径如何写成发布手册”，应使用项目部署文档与发布手册方法论。
