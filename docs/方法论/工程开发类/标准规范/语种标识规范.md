# 语种标识规范

本文定义一套可跨项目复用的语种标识规则，用于统一语种输入、内部流通、业务支持边界和外部系统适配。

## 文档职责

本文负责说明：

- 语种标识应采用什么主线标准。
- 不同历史体系的语种值如何归一化到主线标准。
- 项目如何区分“格式合法”和“业务支持”。
- 中文相关 tag 应如何处理。
- 新增或调整语种支持时应遵循什么维护规则。

本文不负责定义某个具体服务当前支持哪些语种。具体支持范围应由项目自己的接口、配置或代码源头声明，例如 `SUPPORTED_LANGUAGES`、`/supported-languages` 或等价机制。

## 核心结论

跨项目语种标识统一采用 `BCP 47 / IETF language tag` 作为内部主线格式。

典型示例：

| Tag | 含义 |
| --- | --- |
| `en-US` | English, United States |
| `zh-CN` | Chinese, China |
| `zh-Hans` | Chinese, Simplified script |
| `zh-Hant` | Chinese, Traditional script |
| `ar` | Arabic |
| `pt-PT` | Portuguese, Portugal |

规范化后的 tag 应采用 canonical form：

- `language` 使用小写，例如 `en`、`zh`、`pt`。
- `script` 使用首字母大写，例如 `Hans`、`Hant`、`Latn`。
- `region` 使用大写，例如 `US`、`CN`、`TW`。
- 分隔符统一使用 `-`，不使用 `_`。

## 主线标准

语种标识以 `BCP 47 / RFC 5646` 为主线标准。

选择它的原因是互操作性最好：

- HTTP `Accept-Language` 使用该体系。
- HTML `lang` 属性使用该体系。
- CLDR / ICU 可与其稳定转换。
- Java、Swift、Kotlin、JavaScript 等主流生态均能处理。
- 主流云服务的翻译、语音、NLP 接口通常接受或返回该体系的 tag。

因此，项目内部不应直接以 POSIX locale、Windows 私有 tag、ISO 639 三字母码或自由文本作为长期流通格式。它们应先经过归一化，再进入业务逻辑。

## 标准边界

语种处理需要区分三个层级：

| 层级 | 说明 | 示例 |
| --- | --- | --- |
| 格式合法 | tag 符合 BCP 47 语法 | `pt-BR` 是合法 tag |
| 项目声明支持 | 项目明确把该 tag 纳入支持列表 | `pt-PT` 在某项目中被列入 `SUPPORTED_LANGUAGES` |
| 能力真实覆盖 | 检测、翻译、TTS、分词或模型实际能处理该语言 | 模型能识别 `pt`，但未必区分 `pt-PT` 和 `pt-BR` |

业务代码不能只根据“格式合法”判断语种可用。一个合法 tag 只有进入项目支持列表，并且底层能力真实覆盖，才应视为可用。

## 处理流程

语种值应按以下流程处理：

```text
任意输入
  |
  v
输入识别
  - POSIX locale
  - Windows / .NET CultureInfo
  - ICU Locale ID
  - ISO 639-2/3
  - BCP 47
  - 自由文本或非规范写法
  |
  v
归一化
  - 去空白
  - 统一分隔符
  - 修正大小写
  - 映射历史码
  - 转换扩展语法
  |
  v
BCP 47 canonical form
  |
  +--> 业务支持校验
  |
  +--> 输出适配
```

内部业务逻辑应尽量只接触归一化后的 BCP 47 tag。对接外部系统时，如目标系统要求其他格式，再在输出适配层转换。

## 输入归一化规则

通用归一化至少包含以下步骤：

1. 去掉首尾空白。
2. 将 `_` 替换为 `-`。
3. 识别并移除 POSIX encoding 后缀，例如 `.UTF-8`。
4. 将 `language` 子标签转为小写。
5. 将 `script` 子标签转为首字母大写。
6. 将 `region` 子标签转为大写。
7. 映射历史废弃码和私有兼容码。
8. 校验是否符合 BCP 47 语法。
9. 根据项目支持列表判断是否可用。

示例：

| 输入 | 归一化结果 | 说明 |
| --- | --- | --- |
| ` en_us ` | `en-US` | 去空白，统一分隔符和大小写 |
| `ZH-cn` | `zh-CN` | 修正大小写 |
| `zh_CN.UTF-8` | `zh-CN` | POSIX locale 转换 |
| `zh_Hans_CN` | `zh-Hans-CN` | ICU locale 转换 |
| `iw` | `he` | 历史废弃码转换 |
| `root` | `und` | ICU root locale 转为未确定语种 |

## 历史体系映射

### POSIX Locale

POSIX locale 常见于 Linux 服务、Python `locale`、数据库 collation 和系统环境变量。

| POSIX 输入 | BCP 47 | 说明 |
| --- | --- | --- |
| `zh_CN.UTF-8` | `zh-CN` | 去掉 encoding，`_` 转 `-` |
| `zh_CN` | `zh-CN` | 分隔符转换 |
| `en_US.UTF-8` | `en-US` | 去掉 encoding |
| `C` | `und` | POSIX 特殊值，表示未指定语言环境 |
| `POSIX` | `und` | 同上 |

### Windows / .NET

Windows 和旧版 .NET 系统中可能出现私有或历史兼容 tag。

| 输入 | BCP 47 | 说明 |
| --- | --- | --- |
| `zh-CHS` | `zh-Hans` | 微软私有 tag，简体中文 |
| `zh-CHT` | `zh-Hant` | 微软私有 tag，繁体中文 |
| `zh-CN` | `zh-CN` | 已是合法 BCP 47 tag |
| `zh-TW` | `zh-TW` | 已是合法 BCP 47 tag |
| `iw` | `he` | 希伯来语旧码 |
| `in` | `id` | 印度尼西亚语旧码 |
| `ji` | `yi` | 意第绪语旧码 |

### ICU Locale ID

ICU Locale ID 常见于 Java、Android、Chrome 和本地化系统。

| ICU 输入 | BCP 47 | 说明 |
| --- | --- | --- |
| `zh_CN` | `zh-CN` | 分隔符转换 |
| `zh_Hans_CN` | `zh-Hans-CN` | 保留 script 和 region |
| `zh@calendar=chinese` | `zh-u-ca-chinese` | ICU keyword 转 BCP 47 Unicode extension |
| `root` | `und` | root locale 转未确定语种 |

### ISO 639-2/3

ISO 三字母码常见于图书馆、语料库、学术系统和旧数据资产。

| ISO 639-2/3 | BCP 47 | 说明 |
| --- | --- | --- |
| `eng` | `en` | English |
| `zho` | `zh` | Chinese 宏语言 |
| `ara` | `ar` | Arabic |
| `cmn` | `cmn` 或 `zh` | 普通话；严格语言学场景可保留 `cmn` |
| `yue` | `yue` | 粤语；工业系统支持程度需要单独确认 |

三字母码不应被简单截断成两字母码。应通过明确映射表处理。

### 非规范写法

人工配置、接口调用和旧数据中可能出现非规范写法。

| 输入 | 推荐归一化 | 说明 |
| --- | --- | --- |
| `zh-cn` | `zh-CN` | region 大写 |
| `ZH-CN` | `zh-CN` | language 小写 |
| `EN_US` | `en-US` | 分隔符和大小写修正 |
| `english` | `en` | 需要语言名映射表 |
| `chinese` | `zh` | 需要语言名映射表 |
| `mandarin` | `cmn` 或 `zh` | 需按业务约定决定 |

自由文本语言名必须谨慎处理。若项目没有维护稳定映射表，应拒绝该类输入，而不是猜测。

## 中文 tag 处理

中文是语种标识中最容易混淆的部分，原因是它可能按地区、书写系统、宏语言或具体语言变体表达。

| 输入或场景 | 常见含义 | 推荐主线 tag |
| --- | --- | --- |
| `zh-CN` | 中国大陆中文，工业系统通常隐含简体 | `zh-CN` |
| `zh-TW` | 台湾中文，通常为繁体 | `zh-TW` |
| `zh-HK` | 香港中文，通常为繁体 | `zh-HK` |
| `zh-Hans` | 简体中文，不绑定地区 | `zh-Hans` |
| `zh-Hant` | 繁体中文，不绑定地区 | `zh-Hant` |
| `zh-CHS` | Windows 简体中文私有 tag | `zh-Hans` |
| `zh-CHT` | Windows 繁体中文私有 tag | `zh-Hant` |
| `cmn` | 普通话 | 按业务选择 `cmn` 或 `zh` |
| `yue` | 粤语 | `yue` |

通用建议：

- 对普通业务接口，优先接受 `zh-CN`、`zh-TW`、`zh-HK`。
- 对需要区分书写系统的本地化、字体、展示或文本转换系统，可使用 `zh-Hans` 和 `zh-Hant`。
- 对 NLP、语料库或语言学系统，才考虑暴露 `cmn`、`yue` 等更细粒度 tag。
- 不要假设所有 `zh` 都等于 `zh-CN`。如果业务需要默认值，应在项目支持列表中明确声明。

## 业务支持列表

每个项目都应维护一个明确的支持列表，作为业务能力边界。

推荐字段：

| 字段 | 说明 |
| --- | --- |
| `tag` | 对外暴露的 BCP 47 canonical tag |
| `base_language` | 检测、模型或处理逻辑使用的基础语言 |
| `english_name` | 英文名称 |
| `native_name` | 本地名称，可选 |
| `aliases` | 可接受的兼容输入，可选 |
| `capabilities` | 支持能力，例如 `detect`、`translate`、`tts` |

示例：

```json
{
  "tag": "zh-CN",
  "base_language": "zh",
  "english_name": "Chinese",
  "native_name": "简体中文",
  "aliases": ["zh", "zh-Hans", "zh_CN"],
  "capabilities": ["detect"]
}
```

支持列表应是业务判断的源头。归一化只能说明输入应被解释成什么 tag，不能替项目决定是否支持。

## 不支持语种的处理

当输入 tag 格式非法时，应返回“非法语种格式”类错误。

当输入 tag 格式合法但项目不支持时，应返回“不支持该语种”类错误。

两者不应混用：

| 情况 | 示例 | 推荐结果 |
| --- | --- | --- |
| 格式非法 | `en_us_extra_bad` | `invalid_language_tag` |
| 格式合法但未支持 | `pt-BR` | `unsupported_language` |
| 可归一化且已支持 | `en_us` -> `en-US` | 进入业务逻辑 |

如果接口不区分错误码，也应在日志或内部结果中保留该差异，方便排查调用问题和扩展支持范围。

## 输出适配

内部主线格式应保持 BCP 47。只有对接下游系统时，才根据目标系统要求转换。

| 目标系统 | 输出示例 | 说明 |
| --- | --- | --- |
| Web / HTTP | `zh-CN` | 直接使用 BCP 47 |
| HTML | `zh-CN` | 用于 `lang` 属性 |
| ICU / Java | `zh_CN` 或 `zh-Hans-CN` | 视目标 API 要求决定 |
| POSIX | `zh_CN.UTF-8` | 用于系统 locale 时补充 encoding |
| 旧 Windows / .NET | `zh-CHS` 或 `zh-CHT` | 仅兼容旧系统时使用 |

输出适配不应反向污染内部格式。项目内部仍应保存和流通 BCP 47 tag。

## 维护规则

新增或调整语种支持时，应至少完成以下检查：

1. 对外 tag 是否采用 BCP 47 canonical form。
2. 是否需要兼容历史输入或别名。
3. 归一化后是否会与现有 tag 冲突。
4. 底层能力是否真实支持该语言或地区变体。
5. 是否需要区分 `language`、`script`、`region`。
6. 是否更新接口文档、测试数据和支持列表。
7. 是否明确不支持时的响应行为。

新增地区变体时尤其要谨慎。例如 `en-GB`、`pt-BR`、`es-MX` 在格式上合法，但如果底层能力只按 `en`、`pt`、`es` 判断，就需要明确它们是否只是别名，还是独立支持项。

## 推荐落地方式

项目中建议至少拆分三类职责：

| 职责 | 说明 |
| --- | --- |
| `normalize_language_tag` | 把外部输入归一化为 BCP 47 canonical form |
| `SUPPORTED_LANGUAGES` | 声明当前项目支持的 tag 和能力边界 |
| `adapt_language_tag_for_target` | 对接特定下游系统时做输出转换 |

不要把归一化规则、业务支持列表和下游适配逻辑混在一起。三者变化原因不同，混在一起会导致新增语种、兼容旧系统和调整业务能力时互相影响。
