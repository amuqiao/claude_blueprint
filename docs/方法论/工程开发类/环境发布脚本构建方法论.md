# 环境发布脚本构建方法论

> 指导项目把高风险、可重复的环境发布流程沉淀为安全、可检查、可回退、可复制到其他项目的脚本入口，让开发者按固定命令完成发布准备、状态判断和发布提交，而不是每次手工切分支、猜 CI 触发规则和覆盖运维配置。

```text
环境发布脚本构建
├── Scope：链路边界
│   └── 明确脚本负责发布链路的哪一段，不把 CI、平台操作和业务验证混进脚本
├── Contract：命令契约
│   └── 固定少量入口，让开发者知道什么时候 prepare、status 和 push
├── Configuration：配置收敛
│   └── 把项目差异集中到顶部配置区，避免通过长参数维护项目差异
├── Isolation：工作区隔离
│   └── 用 tmp 副本承载切分支、merge、commit 和 push，避免污染日常开发目录
├── Protection：发布保护
│   └── 识别发布控制文件、构建风险文件和业务代码文件，分别处理
├── State：状态判断
│   └── 让脚本在任意时刻都能说明 prepare 和 push 是否可执行
├── Recovery：失败恢复
│   └── 失败时输出原因和下一步，而不是只暴露底层命令错误
└── Handoff：发布交接
    └── 脚本结束后把后续动作交接给 CI、运维平台或人工验证
```

## 核心命题

环境发布脚本构建处理的不是“怎么写 Shell 脚本”，而是“如何把多人、多分支、多配置的环境发布流程，收敛成一个稳定、安全、可复用的工程入口”。

很多发布事故不来自某条命令写错，而来自发布流程没有边界：开发者在日常开发目录里切测试分支，测试分支的 `Dockerfile` 或 CI 配置被业务分支覆盖，本地未 push 的提交被误以为已经发布，发布失败后不知道该继续合并、重新准备还是删除临时目录。

这套方法论的任务，是把发布脚本从“命令自动化”提升为“发布状态管理”。脚本不只是帮人少敲命令，而是承担发布前检查、工作区隔离、风险文件保护、状态判断和失败恢复这些稳定职责。

它的核心转化是：

```text
手工发布步骤 -> 发布链路边界 -> 固定命令契约 -> 项目配置收敛 -> 工作区隔离 -> 风险文件保护 -> 状态判断 -> 失败恢复 -> 发布后交接
```

## 方法论节点

环境发布脚本构建不是 Shell 最佳实践大全，也不是 CI/CD 平台设计。它只处理已经反复出现、值得脚本化的环境发布流程，尤其适合测试环境、预发环境或内部交付环境。

### Scope：链路边界

先明确脚本负责发布链路的哪一段。

环境发布通常横跨本地 Git、远端分支、CI 构建、镜像仓库、运维平台、服务验证和回滚策略。发布脚本不应默认吞掉整条链路。它应该先说清自己负责什么，不负责什么。

一个安全的发布脚本通常只负责其中一段稳定链路，例如：

```text
主开发分支代码
-> 合入测试发布分支
-> 提交带 CI 标记的发布 commit
-> 推送测试发布分支触发 CI
```

CI 镜像构建、Kuboard 或 K8s 切镜像、服务验证、业务回归和生产发布可以作为后续交接动作，但不应在脚本没有足够上下文和权限时强行自动化。

`Scope` 完成后，维护者应能回答：脚本会修改哪个分支、会推送到哪里、会触发什么 CI、不会修改哪些平台资源、发布后还需要人工做什么。

### Contract：命令契约

固定少量、稳定、语义清楚的命令入口。

发布脚本不应把复杂度转移给使用者，让他们每次拼一长串参数。日常入口应尽量少，并且每个入口有明确职责。

推荐契约是：

```bash
./scripts/release-<env>.sh          # prepare：准备发布，不提交、不推送
./scripts/release-<env>.sh status   # 查看主目录和发布副本状态
./scripts/release-<env>.sh push     # 提交并推送，触发 CI
./scripts/release-<env>.sh --help   # 查看帮助
```

其中 `<env>` 表示目标环境，例如测试环境可以命名为 `release-test.sh`，预发环境可以命名为 `release-staging.sh`。脚本名可以随项目调整，但命令语义应保持稳定。

`prepare` 负责安全准备，`status` 负责随时判断，`push` 负责最终发布。不要同时保留 `push`、`--push`、`-p` 等多套入口；入口越多，团队越容易形成不同操作习惯，脚本说明也会漂移。

命令契约还应限制多余参数。发布脚本属于高风险入口，宁可拒绝模糊输入，也不要猜用户想做什么。

### Configuration：配置收敛

把项目差异集中到脚本顶部配置区。

跨项目复用时，发布脚本不应依赖长参数，也不应把项目名、远端名、主分支、测试分支、镜像提示和保护文件散落在函数逻辑里。复制到新项目后，维护者应优先只改顶部配置区。

稳定配置通常分为几类：

```text
项目身份
  例如 PROJECT_NAME，用于生成 tmp 目录、日志提示或默认标识

Git 边界
  例如 REMOTE_NAME、MAIN_BRANCH、TARGET_BRANCH、SOURCE_REF
  用于固定从哪里发布、合入哪里、推送到哪里

隔离目录
  例如 RELEASE_TMP_DIR
  用于固定临时发布副本位置，避免污染日常开发目录

构建触发
  例如 BUILD_MODE
  用于区分代码产物、运行环境产物或其他 CI 触发模式

风险治理
  例如 PROTECTED_FILES、RISK_FILES
  用于区分自动保护、人工确认和正常合入的文件

发布交接
  例如 CODE_IMAGE_NEXT_STEP、OS_IMAGE_NEXT_STEP
  用于说明 push 后交给 CI、平台或人工验证的下一步
```

仓库地址和本地项目路径通常不应作为配置项。脚本放在项目内时，应从当前 Git 仓库读取仓库根目录和 `REMOTE_NAME` 对应的远端地址。这样复制到不同机器或不同目录时不容易失效，也能避免把发布推到硬编码的错误仓库。

配置收敛的目标不是做成万能脚本，而是形成“复制脚本到新项目，只改顶部配置区”的维护协议。

### Isolation：工作区隔离

把高风险 Git 操作放到临时发布副本中执行。

发布脚本如果直接在日常开发目录里切测试分支、merge、解决冲突和 commit，很容易让主开发工作区进入混乱状态。更稳妥的做法是：主项目目录只做检查和复制来源，测试分支合并、提交和推送都发生在 tmp 发布目录。

典型隔离模型是：

```text
主项目目录
  只允许在 MAIN_BRANCH 执行
  只做状态检查、fetch、复制来源

tmp 发布目录
  允许切换 TARGET_BRANCH
  允许 merge SOURCE_REF
  允许 commit 和 push TARGET_BRANCH
```

tmp 发布目录应满足几个安全条件：不能是根目录，不能等于主项目目录，不能放在主项目内部，名称应包含可识别的 `release-` 标记，最好位于项目同级 `tmp` 目录下。这样发布副本出问题时可以删除重建，不会影响日常开发。

### Protection：发布保护

区分发布控制文件、构建风险文件和业务代码文件。

测试发布分支经常包含运维维护的 CI、Dockerfile、基础镜像或平台配置。如果直接把主开发分支完整覆盖到测试分支，可能把运维配置冲掉，导致 CI 构建错误或镜像类型错误。

发布脚本应至少维护三类文件：

```text
发布控制文件
  例如 .gitlab-ci.yml、Dockerfile、Dockerfile_OS
  合并时默认保留测试分支版本

构建风险文件
  例如 .dockerignore、.gitignore、pyproject.toml、uv.lock、.python-version
  发生变化时提示人工确认

业务代码文件
  例如 main.py、src/*、app/*、config.py
  正常随主开发分支合入，但发布后需要业务验证
```

保护策略不应隐藏风险。发布控制文件可以自动保留目标分支版本；构建风险文件更适合提示人工确认；业务冲突应停下来让开发者解决。

### State：状态判断

让脚本在任意时刻都能判断下一步能不能做。

发布脚本不能只提供 `prepare` 和 `push`，还应提供 `status`。因为发布流程可能被打断：prepare 后人工检查，解决冲突后继续，CI 前发现本地分支落后，或者 tmp 发布目录已经干净但用户忘记是否准备过。

`status` 应同时输出：

```text
主项目目录状态
tmp 发布目录状态
远端名和远端地址
来源引用和目标分支
构建模式和 CI 标记
保护文件和风险文件
prepare 是否可执行
push 是否可执行
下一步建议
```

`status` 必须是只读入口。它可以读取主项目目录、tmp 发布目录、远端配置和 Git 状态，但不应切分支、修改暂存区、创建提交、删除目录或推送远端。开发者应能在发布前、发布中断后和发布完成后反复执行 `status`，并相信它只提供判断，不改变现场。

判断输出应面向开发者，而不是只贴 `git status`。例如“主项目工作区不干净，所以 prepare 不可执行”，“tmp 发布目录没有待提交 merge，所以 push 不可执行”，“tmp 仍有冲突，解决并 git add 后再 push”。

### Recovery：失败恢复

失败时输出原因和下一步。

发布脚本失败不可怕，最危险的是失败后用户不知道当前状态。高风险脚本应把失败当成流程的一部分设计，而不是只依赖 `set -e` 中断。

常见失败应有明确恢复路径：

```text
主开发工作区不干净
  先提交或清理工作区，再重新 prepare

本地主分支领先远端
  先 push 主分支，避免发布遗漏本地提交

tmp 发布目录已有未完成 merge
  先检查 tmp 状态，确认后 push 或删除 tmp 重来

发布控制文件冲突
  自动保留目标分支版本，再提示检查 diff

业务代码冲突
  停止在 tmp 目录，人工解决冲突并 git add 后再 push

SOURCE_REF 在 prepare 后变化
  重新 prepare，避免发布过期代码
```

恢复提示应直接给下一条可执行命令，但不要替用户做危险操作。比如可以提示“删除 tmp 后重新 prepare”，但不要在失败时自动删除用户可能正在解决冲突的 tmp 目录。

### Handoff：发布交接

脚本完成后明确交接给谁。

发布脚本通常不是发布链路的终点。它推送目标分支后，后续可能由 CI 构建镜像，由钉钉或 GitLab 通知镜像版本，由 Kuboard 或 K8s 手动切镜像，再由开发或测试验证服务。

脚本结束时应输出下一步，尤其要区分发布产物和后续操作。常见交接对象包括：

```text
代码产物
  例如应用镜像、前端构建产物或代码包
  下一步通常是等待 CI 产物版本，并更新目标环境使用的代码镜像或服务版本

运行环境产物
  例如基础镜像、运行环境镜像或依赖层镜像
  下一步通常需要运维确认是否更新工作容器、基础环境或平台模板

平台切换动作
  例如 Kuboard、K8s、云平台或内部发布平台中的镜像版本调整
  下一步应说明由谁操作、改哪个对象、不要改哪个对象

人工验证动作
  例如健康检查、核心接口验证、业务回归或日志观察
  下一步应说明发布后如何确认服务真的可用
```

`BUILD_MODE=ci` 和 `BUILD_MODE=os` 只是其中一种实现方式：前者常用于代码产物，后者常用于运行环境产物。如果项目没有运行环境镜像，就不需要保留 `os` 分支；如果项目有更多产物类型，也应先在交接模型中说清角色、对象和验证方式，再决定是否扩展脚本配置。

如果代码产物和运行环境产物由不同角色维护，脚本必须在提示中说清楚，避免开发者把代码镜像、基础镜像和平台容器对象混用。

## 判断标准

环境发布脚本构建是否完成，只看一个标准：

**开发者在不理解全部 Git、CI 和运维细节的情况下，能否通过固定命令安全完成发布准备、状态判断和发布提交，并且在失败时知道下一步做什么。**

如果脚本能执行，但用户不知道它会在哪个目录切分支、推送哪个分支、触发哪个 CI，说明 `Scope` 和 `Contract` 没完成。

如果复制到新项目后需要在函数逻辑里到处改项目名、分支名、容器名和保护文件，说明 `Configuration` 没完成。

如果脚本会让主开发目录进入测试分支或 merge 状态，说明 `Isolation` 没完成。

如果发布分支的 Dockerfile、CI 配置或运维维护文件容易被主分支覆盖，说明 `Protection` 没完成。

如果用户必须靠猜测判断能不能 push，说明 `State` 没完成。

如果脚本失败后只留下底层错误，没有原因和下一步，说明 `Recovery` 没完成。

## Few-shot Learning 示例组

### 正例：测试环境代码镜像发布

一个项目由开发维护 `dev` 分支，由运维维护 `test` 分支上的 CI 和 Dockerfile。开发只负责发布代码镜像，运行环境镜像由运维维护。

合适做法是提供固定脚本：

```bash
./scripts/release-test.sh
./scripts/release-test.sh status
./scripts/release-test.sh push
```

脚本在主项目 `dev` 目录发起，把仓库复制到同级 `tmp` 发布目录，在副本里切到 `test`，合入 `origin/dev`，自动保留 `test` 分支上的发版控制文件。默认提交信息包含 `[ci build]`，只触发代码镜像构建。发布成功后提示开发者去平台更新代码镜像，不要改 OS 工作容器镜像。

判断：

```text
Scope：脚本只负责 Git 合并、提交和推送，不负责平台切镜像
Contract：prepare/status/push 固定
Configuration：项目名、分支、tmp 目录、保护文件集中配置
Isolation：test 分支操作只发生在 tmp 副本
Protection：保留 test 分支上的发版控制文件
State：status 判断 prepare 和 push 是否可执行
Recovery：冲突、未 push、过期 SOURCE_REF 都有下一步
Handoff：push 后交接给 CI 和 Kuboard 人工更新
```

### 反例：一键脚本直接在 dev 目录切 test

一个脚本在当前目录执行 `git switch test && git merge dev && git push`。它能少敲命令，但主开发目录会被切到测试分支，一旦冲突或失败，开发者的日常工作区就留在 merge 状态。脚本也没有保护测试分支上的 Dockerfile，容易覆盖运维配置。

问题不在于缺少更多 Shell 技巧，而在于没有建立发布隔离、风险文件保护和失败恢复模型。

### 边界例：完整生产 CI/CD 平台

如果项目已经有完整 CI/CD、环境审批、镜像签名、自动灰度、回滚策略和平台化发布单，本方法只能作为“发布脚本入口设计”的前置判断，不替代生产交付平台设计。

此时重点应转向发布流程治理、权限审批、流水线状态机、回滚策略和可观测性，而不是继续强化本地 Shell 脚本。

## 误用

环境发布脚本构建不是 Shell 编程教程。

不要把 `set -euo pipefail`、数组、函数、引号和 `shellcheck` 基础规则写成方法论主体。它们是实现约束，只有在影响发布安全、输入边界或失败恢复时才需要进入文档。

环境发布脚本构建也不是把所有发布动作都一键自动化。

脚本不应在没有清晰权限、状态判断和回滚策略时自动操作运维平台、删除临时目录、覆盖配置、切生产流量或修改数据。高风险动作可以提示下一步，但不要默认替用户执行。

它也不是长参数工具。

如果一个脚本必须通过大量参数传入项目路径、仓库地址、分支名、镜像名和容器名，说明项目差异没有收敛。更适合的方式是把脚本复制到项目内，维护顶部配置区，日常命令保持稳定。

它也不是部署文档的替代品。

脚本负责执行和判断，文档负责解释发布链路、镜像版本、平台操作、注意事项和排障路径。当发布流程需要交接给新人或跨角色协作时，仍应使用项目部署文档与发布手册方法论沉淀说明。

## 适用边界

这套方法适合处理以下场景：

```text
测试环境或预发环境需要从主开发分支合入发布分支
发布分支存在运维维护的 CI、Dockerfile 或镜像配置
开发者只负责代码镜像，运行环境镜像由运维维护
发布前需要人工检查 diff，再推送触发 CI
发布过程要避免污染日常开发分支
脚本需要复制到多个项目复用
```

这套方法不适合处理一次性实验脚本、纯本地开发启动脚本、简单工具库发布、生产级发布平台设计、数据库迁移治理、蓝绿或灰度发布策略、K8s 资源编排和监控告警体系。

如果问题是“项目有哪些本地启动、构建、验证和清理入口”，应使用项目开发入口设计方法论。

如果问题是“本地开发、Dockerfile、compose 和配置加载边界怎么设计”，应使用项目部署入口与配置加载方法论。

如果问题是“已经稳定的发布路径如何写成可查的操作手册”，应使用项目部署文档与发布手册方法论。
