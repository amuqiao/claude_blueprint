# 项目开发入口设计方法论

> 把项目里反复出现的启动、验证、生成、清理和构建动作，收敛成边界清楚、命令稳定、风险可控的开发入口。

```text
项目开发入口设计
├── Surface：入口面
│   └── 判断开发入口应采用文档、脚本、包管理命令、任务运行器还是服务管理脚本
├── Scope：作用域
│   └── 切清入口管理哪个项目、子工程、服务或命令集合
├── Mode：运行模式
│   └── 区分前台应用、后台服务、一次性任务、代码生成、构建验证和清理动作
├── Contract：命令契约
│   └── 固定命令名称、参数、输出、失败行为和验证标准
└── Guard：保护边界
    └── 防止误删、误清、误改配置、误操作旧工程或生产环境
```

## 核心命题

项目开发入口设计处理的不是“要不要写一个总控脚本”，而是“一个项目的常用开发动作是否已经需要统一入口，以及这个入口应该长成什么样”。

真实项目里，开发动作常常分散在说明文档、协作约定、包管理脚本、构建工具、根目录脚本和子工程命令中。

当这些入口没有被设计，问题不会立刻表现为“不能运行”，而是表现为入口没有形成稳定判断：新人不知道该从哪里开始，Agent 不知道改完后如何验证，旧工程、新工程、本地开发、构建生成和清理动作容易互相混淆。

项目开发入口设计的任务，是把这些零散动作收敛成一个可判断的入口面。这个入口可能是文档、生态脚本、任务运行器、轻量脚本，也可能是服务生命周期脚本。脚本只是载体之一，不是默认答案。

它的核心转化是：

```text
零散动作 -> 入口面判断 -> 作用域切分 -> 运行模式归类 -> 命令契约固定 -> 风险边界保护
```

## 方法论节点

项目开发入口设计不是固定脚本模板。它先判断入口形态，再决定是否需要脚本，以及脚本应该轻到什么程度、重到什么程度。

### Surface：入口面

先判断项目需要哪种开发入口。

入口面不是具体命令，而是开发者和 Agent 接触项目时的操作界面。一个项目可以有多个入口，但必须知道哪个入口负责什么。

入口面可以是文档、生态脚本、任务运行器、轻量脚本或服务生命周期脚本。判断重点不是选哪种工具，而是哪种入口能最小化协作成本。

入口面判断关注三件事：高频动作是否已经分散，读者是否能不看源码就知道如何启动和验证，新增脚本是否真的降低认知成本。

如果动作很少、命令稳定、误用风险低，文档入口已经足够。不要为了“看起来工程化”而新增脚本。

如果动作开始重复、路径容易写错、验证命令需要组合，轻量脚本才有价值。

如果项目存在多个后台服务，需要统一 `start / stop / restart / status / logs`，才进入服务生命周期脚本设计，不再按轻量开发入口判断。

### Scope：作用域

切清开发入口管理什么，不管理什么。

很多总控脚本失控，不是因为命令写错，而是因为作用域一开始没有定住。脚本既管旧项目，又管新项目；既管开发，又管发布；既管本地，又管生产；最后任何改动都可能误伤。

作用域判断关注入口管理什么、不管理什么，以及它是否会触碰旧项目、生产配置、数据库、账号、远程服务或有副作用的本地产物。

有效作用域应能一句话说清楚，例如“只管理新子工程，不管理旧工程；只做本地开发验证，不做部署”。

作用域越清楚，脚本越容易长期维护。作用域说不清时，应先写文档，不应先写脚本。

### Mode：运行模式

区分项目动作属于哪种运行模式。

`Surface` 解决“入口采用什么载体”，`Mode` 解决“动作属于什么运行类型”。服务生命周期脚本只有在 `Mode` 判断为后台服务生命周期时，才是合适的入口面。

不同模式需要不同入口。把所有动作都塞进 `start / stop / restart`，会把前台应用、后台服务、一次性任务和构建验证混成一类。

常见运行模式包括：

```text
持续运行：需要进程、端口、日志、停止机制或刷新策略
一次性任务：关注执行位置、退出状态和失败行为
状态改变动作：会生成文件、安装依赖、删除缓存或改变构建产物
```

模式判断关注动作是否持续运行、是否需要进程管理、修改代码后如何刷新，以及它是否会改变项目状态。

前台 App 不应默认套用后台服务脚本。只有存在可后台托管的服务进程时，`start / stop / restart / status / logs` 才是合适抽象。

### Contract：命令契约

固定命令名称、参数、输出和失败行为。

入口一旦进入脚本或任务运行器，就不只是命令集合，而是项目协作接口。接口不稳定，后续文档、Agent、CI 和开发者都会反复适配。

命令契约关注命令名称、执行位置、参数、成功标准、失败行为、输出可读性、副作用说明和旧入口迁移方式。

命令集合不应追求跨项目统一，而应让高频动作、验证动作和有副作用动作拥有稳定名称。

如果命令会改动仓库文件，例如代码生成、格式化、依赖安装，应在文档或脚本帮助中明确说明。

当入口落地为脚本时，应提供简短使用说明，让读者知道如何调用、何时成功、何时失败以及哪些动作有副作用。使用说明服务入口契约，不承担工具教程职责。

### Guard：保护边界

为高风险动作加边界。

开发入口越方便，越要防止误操作。保护边界不是让脚本变复杂，而是让脚本在关键位置拒绝危险默认行为。

保护边界通常体现在三处：固定执行对象，拒绝危险默认行为，让清理、重置、删除、数据库变更、远程环境操作等高风险动作显式命名。

脚本内注释应只解释非显然逻辑和高风险边界。不要为工具常识、普通命令或交互按键写成教学型注释。

保护边界的目标不是让脚本替代人的判断，而是防止最容易发生、代价最高的误操作。

## 判断标准

项目开发入口设计是否完成，只看一个标准：

**读者或 Agent 是否能用一个稳定入口完成当前项目的启动、验证、生成、构建和必要清理，并且知道哪些动作不属于这个入口。**

如果读者仍然需要在多个文档和配置文件之间猜命令，说明入口没有完成。

如果脚本可以执行，但不知道它管哪个项目、不管哪个项目、什么时候需要重启、什么时候会删除缓存，说明入口没有完成。

如果入口能让人清楚判断日常开发、代码刷新、提交验证、代码生成、完整构建、缓存异常处理，以及哪些旧项目或生产环境不属于这个入口，说明这套入口设计已经成立。

## Few-shot Learning 示例组

### 正例：新子工程轻量入口

一个仓库里同时保留旧原生工程和新跨平台子工程。用户只想管理新子工程，不想碰旧工程。

合适入口：一个只覆盖新子工程的轻量脚本，提供运行、验证、生成和构建入口，不管理旧工程和后台服务。

判断：

```text
Surface：轻量 shell 脚本
Scope：只管理新跨平台子工程
Mode：前台 App + 一次性验证 + 代码生成 + 构建
Contract：固定命令名和执行路径
Guard：不碰旧原生工程，不做后台服务管理
```

这个场景不需要 `start / stop / restart / status / logs`。

### 反例：把所有项目都套成总控脚本

一个项目只有一个 `npm test` 和一个 `npm run build`，README 已经清楚说明命令，却新增一个 `dev.sh`，里面只是转发两条命令。

问题：

```text
没有降低认知成本
增加了一个需要维护的新入口
脚本命名大于实际职责
未来 README 和脚本容易不一致
```

这个场景更适合保留文档入口，不需要脚本。

### 边界例：多服务后端进入服务生命周期脚本

一个项目需要同时启动 API、worker、scheduler、前端 dev server，并且每个服务都有端口、PID、日志和状态检查。

合适入口：进入服务生命周期脚本设计，让启动、停止、重启、状态检查和日志查看围绕服务进程组织。

判断：

```text
Mode 已经从一次性命令进入后台服务生命周期
需要 PID、日志、端口冲突检查和 stale PID 清理
应进入服务生命周期脚本设计，而不是普通命令聚合脚本
```

## 误用

项目开发入口设计不是“每个项目都要有总控脚本”。

有些项目只需要 README 中的三条命令。此时新增脚本不会提升协作效率，只会制造新的同步成本。

项目开发入口设计也不是把所有命令塞进一个文件。

入口设计强调边界和选择。旧工程、新工程、本地开发、生产部署、构建验证、缓存清理，不应因为都能写成 shell 命令就放进同一个脚本。

项目开发入口设计也不是服务管理脚本模板。

只有当项目真的存在后台服务生命周期，才需要 `start / stop / restart / status / logs`。前台 App、代码生成和一次性验证命令，通常不需要 PID、端口和日志管理。

项目开发入口设计也不是替代 CI。

本地入口解决本地开发和本地验证；CI 解决自动化验证和交付。二者命令可以复用，但职责不应混为一谈。

## 适用边界

这套方法适合处理已经进入持续开发的项目，尤其是：

```text
仓库内有多个子工程
旧项目和新项目并存
开发命令开始分散
代码生成、验证、构建、清理动作重复出现
Agent 或新人容易不知道从哪里启动和验证
某些命令有误操作风险
```

这套方法不适合处理只有一两个稳定命令的小项目、一次性脚本或临时迁移脚本、生产运维平台设计、CI/CD 完整流水线设计、具体 shell 语法教学。

如果问题只是“不知道某个工具怎么用”，应使用工具能力使用与排错方法论。

如果问题已经明确为多服务后台进程管理，应进入服务生命周期脚本设计，而不是继续停留在轻量开发入口设计。
