# 项目部署文档与发布手册方法论

> 指导项目把已经稳定的部署路径沉淀为可查、可复制、可维护的部署手册，让开发者能按运行方式、变更类型和故障场景选择正确操作，而不是每次靠读代码和经验部署。

```text
项目部署文档与发布手册
├── Role：文档职责
│   └── 说明这份文档服务首次部署、日常发布、迁移恢复还是排障维护
├── Access：任务入口
│   └── 让维护者按当前任务、变更类型或故障现象直达对应操作
├── Mode：运行方式
│   └── 列清项目实际验收过的本地开发、容器部署、独立镜像或服务器部署方式
├── Config：配置说明
│   └── 固定模板文件、真实配置、密钥、覆盖优先级和必须修改项
├── Operation：常用操作
│   └── 提供启动、停止、重启、状态、日志和验证等可复制命令
├── Release：变更发布
│   └── 按前端、后端、数据库、配置、镜像和容器结构等变更类型给出发布路径
├── Data：数据安全
│   └── 说明迁移、备份、恢复、持久化目录、危险命令和不可逆操作
├── Troubleshoot：故障排查
│   └── 把高频错误映射到检查命令、判断依据和修复动作
└── Maintain：同步维护
    └── 规定修改服务、配置、端口、脚本、迁移和部署形态时如何同步更新文档
```

## 核心命题

项目部署文档与发布手册处理的不是“项目应该如何设计部署入口”，而是“当部署路径已经稳定后，如何把它写成长期维护者可以直接使用的操作参考”。

多项目维护时，部署风险往往不来自某条命令本身，而来自维护者不知道当前项目这次改动属于哪类发布：只改前端是否需要重建后端，后端代码和 worker 是否共用镜像，数据库 migration 是否需要先执行，配置变更是否要重建容器，Dockerfile 或 compose 变更是否需要全量重建，哪些命令会删除数据。

如果部署文档只记录首次启动命令，后续维护者仍然需要读代码、猜 compose、翻脚本和回忆经验。好的部署手册应像项目级参考手册：先把已支持的运行方式讲清，再把常用操作、变更发布、数据安全和故障排查沉淀成可复制的操作路径。

它的核心转化是：

```text
稳定部署路径 -> 文档职责定位 -> 任务入口设计 -> 运行方式归档 -> 配置与密钥说明 -> 常用操作速查 -> 按变更类型发布 -> 数据安全与排障 -> 同步维护规则
```

## 方法论节点

项目部署文档与发布手册不是部署命令合集，也不是把 README 写长。它关注的是：维护者面对不同部署场景时，能不能快速找到正确入口、理解影响范围，并避免误删数据、漏跑迁移、漏重建服务或遗漏配置更新。

这套主线可以作为部署手册的默认骨架，但不是固定目录模板。落地时应先保留 `Role`、`Access`、`Mode`、`Config`、`Operation` 和 `Maintain` 这类基础判断，再根据项目是否存在前端、数据库、worker、持久化数据、独立镜像、compose 全量部署或长期服务器运维，触发 `Release`、`Data`、`Troubleshoot` 等重点分支。

### Role：文档职责

先固定这份部署文档承担什么职责。

部署文档可能服务首次搭建、本地开发、服务器部署、日常发布、服务器迁移、灾难恢复、故障排查或交接维护。职责不清时，文档会同时堆入安装教程、命令速查、架构说明和排障记录，读者反而找不到当前要做的操作。

`Role` 要回答：目标读者是谁，文档覆盖哪些场景，哪些操作可以直接照做，哪些内容只作为说明，修改哪些项目文件时必须同步更新本文档。

如果项目只需要单人临时运行，不需要多人、跨项目或 Agent 复用，不必急着沉淀完整部署手册；保留轻量启动说明即可。

### Access：任务入口

把部署手册组织成可直达的维护入口。

部署手册不应要求读者从头读到尾。对长期维护者来说，最重要的入口不是“系统怎么部署”，而是“我现在要做什么”。

`Access` 要回答：首次部署、日常发布、配置变更、代码变更、数据库迁移、容器结构变更、回滚、备份恢复和故障排查分别应该进入哪里。

好的任务入口通常按三类问题组织：

```text
我要做什么：首次部署、本地启动、服务器部署、重启服务、查看日志、回滚
我改了什么：前端、后端、worker、数据库、配置、Dockerfile、compose、依赖
我看到什么：启动失败、健康检查失败、配置不生效、502、数据库连接失败、任务卡住
```

任务入口不是目录美化，而是减少维护者的搜索和判断成本。它应放在部署手册靠前位置，让读者能直接跳到可复制命令。首次部署可以线性展开，但日常发布、故障排查和风险操作应优先提供直达入口。

### Mode：运行方式

列清项目实际支持的运行方式。

运行方式应来自已经验收过的部署路径，而不是理论上可行的部署想象。常见方式包括本地开发、容器化依赖、本地 Docker 全量部署、Dockerfile 独立镜像、服务器 compose 部署、云平台部署或 CI/CD 发布。

每种运行方式应说明它启动哪些服务、依赖哪些外部资源、暴露哪些端口、适合什么场景、如何验证成功。没有验证过的 K8s、生产 compose、云平台 Secrets 或自动发布流程，不应写成已支持方案。

`Mode` 的重点不是覆盖所有部署可能性，而是让维护者知道当前项目真正维护哪些部署方式，以及这些方式之间是否可以共存、共享数据或互相影响。

部署手册应有默认主线，但不同项目形态会触发不同重点，不能把完整手册结构用成一刀切模板：

| 项目形态 | 部署手册重点 |
| --- | --- |
| 简单单服务 | 运行方式、配置项、启动停止、健康检查、镜像重建 |
| 单服务 + 外部依赖 | 依赖地址、配置来源、连接验证、依赖不可用排查 |
| 前后端项目 | 前端和后端发布分支、代理关系、构建产物、访问入口 |
| 后端 + worker / scheduler | 共用镜像、队列依赖、后台任务中断风险、重启顺序 |
| 有数据库 migration | 迁移命令、幂等性、备份、回滚、不可逆风险 |
| 有持久化文件或对象存储 | 存储路径、挂载、备份、恢复、清理风险 |
| Dockerfile 独立镜像 | env 注入、外部依赖、端口映射、镜像独立验证 |
| compose 全量部署 | profile、服务组、网络、卷、健康检查、全量重建边界 |
| compose 依赖部署 | 依赖服务启动、应用进程本地运行、配置地址差异 |
| 长期服务器部署 | 密钥、端口、防火墙、拉代码、发布、回滚、备份 |

这张表的作用是把“视项目情况而定”收敛成可枚举分支。项目只应吸收与自身服务形态和部署风险相关的重点，不应为了完整感填入不存在的前端、数据库、队列或生产运维内容。

### Config：配置说明

固定配置文件、密钥和覆盖关系。

部署手册必须说明哪个文件是模板，哪个文件是真实配置，哪些文件不能提交，密钥通过 shell 环境变量、env 文件、secrets 文件、CI/CD 变量还是平台 Secrets 提供。

配置说明不应只列字段名，还应说明运行时最终生效的覆盖关系。例如 `.env` 负责本地默认值，`docker-compose.yml environment` 覆盖容器网络地址，`--env-file` 提供密钥替换值，运行时显式环境变量优先级最高。

这里不重新发明配置优先级，而是把项目已经确定的配置加载规则写成部署手册中的可查规则。

对维护者最有价值的是“必须修改项”和“不要修改项”：哪些字段上线前必须替换，哪些字段会被 compose 覆盖，哪些默认值适合本地但不适合服务器，哪些引号、变量替换或路径写法会导致常见故障。

### Operation：常用操作

把高频操作沉淀成可复制命令。

部署手册应为每种主要运行方式提供最小常用操作：启动、停止、重启、查看状态、查看日志、进入容器、执行健康检查、验证配置和访问关键入口。

这些命令不是为了展示工具用法，而是为了固定当前项目的正确操作对象。比如同样是 `docker compose stop`，是否需要带 profile、是否只影响依赖服务、是否会停止前端和 worker，都必须在项目语境中讲清。

常用操作应避免无限扩张。一次性排查命令、临时实验命令和只有当前机器适用的命令，不应进入主手册；如果必须保留，应放入故障排查或项目备注，而不是污染日常操作区。

### Release：变更发布

按变更类型提供发布路径。

这是部署手册区别于普通启动说明的核心。维护者通常不是从“我要部署”开始，而是从“我这次改了什么”开始：前端代码、后端代码、worker、数据库 migration、模型配置、静态资源、环境变量、Dockerfile、compose、端口、挂载或健康检查。

`Release` 要把变更类型映射成操作路径，说明需要构建哪些镜像、重启哪些服务、是否要执行 migration、是否需要刷新前端代理、是否可以只重启单个服务，什么时候走保守全量发布。

如果项目只有单个服务且没有数据库、前端或后台进程，`Release` 可以只说明代码更新后如何重建和重启；如果项目存在多个构建产物、多个运行进程或数据库迁移，才需要发布矩阵。

如果项目有前端、后端、worker、scheduler 或多个服务共用同一镜像，部署文档必须显式说明它们的发布耦合关系。否则维护者很容易只重建一个服务，遗漏共用镜像或后台进程。

### Data：数据安全

把数据持久化和危险操作讲清楚。

部署手册必须标出哪些数据会持久化，哪些数据会随容器重建丢失，哪些目录、卷、数据库、对象存储或缓存属于项目数据边界。

数据库迁移、备份、恢复和回滚应明确风险等级。幂等 migration 可以作为日常发布步骤；不可逆 migration、手工改表、删除卷、清空存储、覆盖配置和强制重建，应写清前置确认和恢复路径。

`Data` 的目标不是写成数据库运维教程，而是防止维护者在正常发布、回滚、迁移或排障时误删数据、漏备份或误以为容器重建等于数据重置。

如果项目没有数据库、卷、上传文件或外部持久化资源，`Data` 可以退化为说明“无持久化数据”和危险命令边界，不必强行加入备份恢复章节。

### Troubleshoot：故障排查

沉淀高频故障的判断路径。

好的故障排查不是报错大全，而是把高频错误映射到可执行检查：看哪个日志，查哪个健康检查，执行哪个配置验证，如何区分网络不可达、密钥错误、数据库未就绪、反向代理缓存、worker 中断或数据卷丢失。

每个故障条目应尽量包含：

```text
现象
检查命令
判断依据
修复动作
是否有数据风险
```

如果一个排障动作只在某次事故中成立，或者依赖某台机器的临时状态，不应上升为主部署手册内容。它可以进入事故复盘或项目备注。

### Maintain：同步维护

规定部署文档何时必须更新。

部署手册的价值取决于它是否和项目真实部署路径保持一致。修改 `Dockerfile`、`docker-compose.yml`、启动脚本、端口、环境变量、配置模板、服务名、镜像名、数据库 migration、存储路径、健康检查或发布流程时，都应同步检查部署文档。

`Maintain` 应明确文档与 `.env.example`、compose 文件、脚本、CI/CD、迁移目录和 README 的关系，避免多个文档重复维护同一事实。

同步维护的重点不是保持文档好看，而是确保部署命令、配置模板、服务名和真实运行方式不发生事实漂移。

如果文档不能随项目变化更新，它会比没有文档更危险，因为维护者会复制已经过期的命令。

## 判断标准

项目部署文档与发布手册是否成立，只看一个标准：

**维护者在不重新阅读完整代码的情况下，是否能根据“当前要做什么”找到对应操作，并知道这组命令会影响哪些服务、配置和数据。**

如果维护者必须从头阅读部署文档才能找到当前任务对应命令，说明 `Access` 没完成。

如果读者只知道首次启动，却不知道代码、配置、数据库或 Docker 变更后如何发布，说明 `Release` 没完成。

如果文档列了很多命令，但没有说明这些命令属于哪种运行方式、影响哪些服务、是否会删除数据，说明 `Mode`、`Operation` 或 `Data` 没完成。

如果配置模板、真实配置、密钥来源和运行时覆盖关系没有讲清，说明 `Config` 没完成。

如果部署路径已经变化，但文档没有同步维护规则，说明 `Maintain` 没完成。

## Few-shot Learning 示例组

### 正例：前后端加后端任务的 Docker 项目

一个项目包含前端、后端、worker、数据库和 Redis，支持本地开发与服务器 compose 部署。合适的部署手册应先说明运行方式和配置覆盖，再提供日常状态、日志、重启命令，最后按前端、后端、数据库、配置和 compose 变更类型给出发布命令。

判断：

```text
Role：服务首次部署、日常发布、迁移恢复和排障
Access：按首次部署、日常发布、变更类型和故障现象提供直达入口
Mode：本地开发 + Docker 全量 + 服务器部署
Config：模板、真实配置、密钥和 compose 覆盖关系清楚
Operation：状态、日志、启动、停止、验证命令可复制
Release：按前端、后端、DB、配置、Docker 变更分类发布
Data：迁移、卷、备份、危险命令有边界
Troubleshoot：高频错误有检查命令和修复动作
Maintain：改 compose、env、服务、迁移时同步文档
```

### 反例：只有首次启动命令

一个部署文档只写了 `docker compose up -d --build` 和几个环境变量，但项目实际有前端、后端、worker、数据库迁移和配置覆盖。维护者每次发布仍然要猜哪些镜像需要重建、哪些服务要重启、是否要执行 migration。

问题不在于命令太少，而在于文档没有按运行方式、变更类型和数据风险建立判断。它只能帮助首次启动，不能作为长期发布手册。

### 边界例：简单无数据库服务

一个服务没有前端、数据库、worker 和持久化数据，只提供单个 HTTP API。它仍然可以有部署手册，但应保持轻量：运行方式、配置项、启动停止、健康检查、镜像重建和常见错误即可。

此时不应为了套模板加入数据库迁移、备份恢复、前端发布或队列排障。方法论提供判断动作，不要求所有项目填满所有栏目。

## 误用

项目部署文档与发布手册方法论不是 Docker 或 Linux 教程。

不要解释 Docker、compose、shell、SSH、数据库或反向代理的基础概念。只有当这些内容影响当前项目的部署判断、操作路径或风险边界时，才进入部署手册。

它也不是部署架构设计方法。

部署形态、服务边界、配置加载优先级和验证闭环应先由项目部署入口与配置加载方法论判断。本方法论只处理这些判断稳定后，如何写成项目级部署文档和发布手册。

它也不是命令垃圾桶。

不要把所有用过的命令都塞进部署文档。部署手册只保留长期可复用、能降低误操作风险、能帮助维护者完成常见场景的命令。

它也不是固定模板。

主线提供默认骨架，项目形态分支决定重点章节。简单服务可以只有轻量部署说明；复杂项目才需要完整运行方式、发布矩阵、数据安全和排障章节。文档规模应跟项目部署风险匹配。

它也不是线性说明书。

首次部署可以按步骤线性展开，但日常维护入口应靠前，按当前任务、变更类型和故障现象组织。配置说明、架构背景和工具解释不能挡在常用发布命令前面。

## 适用边界

这套方法适合处理已经有稳定部署路径、需要多人维护、跨项目复用或交给 Agent 操作的服务型项目，尤其是：

```text
项目存在多种运行方式
项目有前端、后端、worker、scheduler 或多个容器服务
代码、配置、数据库、镜像或 compose 变更会触发不同发布动作
项目有真实配置、密钥、持久化数据或备份恢复要求
维护者需要按场景复制命令完成部署、发布、回滚或排障
```

这套方法不适合处理一次性实验、纯工具库、没有服务进程的脚本项目、尚未稳定的部署探索、完整生产运维平台设计或 CI/CD 平台搭建。

如果问题是“项目当前应该支持哪些部署形态、配置从哪里来、服务由谁管理”，应先使用项目部署入口与配置加载方法论。

如果问题是“常用启动、验证、生成、清理和构建动作是否需要统一脚本入口”，应使用项目开发入口设计方法论。
