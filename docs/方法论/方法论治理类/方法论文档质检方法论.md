# 方法论文档质检方法论

> 检查一篇方法论文档是否真正形成方法，而不是退化成说明书、待办清单、模板目录或过度抽象的概念堆叠。

```text
方法论文档质检
├── Object：质检对象
│   └── 确认这篇文档处理的对象是否清楚
├── Thesis：核心命题
│   └── 判断它是否说清要解决什么认知问题
├── Node：节点质量
│   └── 检查节点是否必要、稳定、独立、不过度流程化
├── Expression：表达方式
│   └── 判断表达和示例是否服务理解，而不是堆格式或下禁令
└── Boundary：边界归属
    └── 确认适用边界、误用边界和目录归属是否稳定
```

## 核心命题

方法论文档质检处理的不是“文档写得是否完整”，而是“这篇文档是否真的形成了可迁移的方法”。

一篇方法论文档很容易出现四类问题：看起来像方法论，实际只是步骤清单；节点很多，但没有解释力；抽象很高，但读者不知道如何判断；边界不清，导致它和相邻方法论互相覆盖。

方法论文档质检的任务，是检查文档是否能独立帮助读者完成一次稳定的认知转化。

它的核心转化是：

```text
方法论文档 -> 对象检查 -> 命题检查 -> 节点检查 -> 表达检查 -> 边界检查
```

这里的“质检”，不是把文档改得更长，而是判断哪些内容应保留、删减、下沉、拆分或改写。

## 方法论节点

方法论文档质检不是按清单打勾，而是围绕方法是否成立进行判断。

### Object：质检对象

确认这篇方法论处理的对象是否清楚。

对象不清时，后续节点会漂移。方法论文档必须让读者知道它处理的是主题、需求、方案、工具体系、文档、流程，还是某种协作问题。

质检时应判断：

```text
对象是否明确
对象是否过大或过小
对象和相邻方法论是否混淆
单篇文档是否能独立说明对象前提
```

如果对象尚未稳定，就不应继续优化节点，而应先回到对象定义。

### Thesis：核心命题

判断文档是否说清它要解决什么认知问题。

核心命题不是摘要，而是这套方法成立的理由。它应说明：读者为什么不能只靠堆信息、照流程或看示例解决问题，必须使用这套方法。

质检时应判断：

```text
它要纠正哪个常见误区
它要完成什么认知转化
核心转化是否能用一句结构化表达压缩
读者是否能知道何时该用这套方法
```

如果核心命题只是“介绍某某方法”，说明它还没有形成方法论。

### Node：节点质量

检查节点是否必要、稳定、独立、不过度流程化。

节点是方法论的骨架。节点质量决定文档是方法，还是流程清单。

质检时应判断：

```text
每个节点是否是稳定判断动作
节点英文是否准确、常用、单一语义且可迁移
节点英文是否避免晦涩、罕见、学术化或临时造词
中文解释是否准确落地该节点在当前方法论中的含义
英文节点和中文解释是否互补，而不是彼此脱节或互相补救
是否存在两个并列动作硬塞进一个节点名
是否存在输出栏目冒充方法节点
是否为了减少节点而过度抽象
是否为了完整感而增加无必要节点
是否形成稳定主线，而不是松散经验集合
是否把稳定主线误写成所有场景都必须照填的固定模板
是否对高频、可归类、会改变判断重点的场景只用“视情况而定”泛泛带过
```

英文节点可以是名词、动词或稳定合成词，不强求词性一致。质检重点不是形式统一，而是英文是否提供稳定抽象、中文是否提供语境落地，两者合起来是否降低理解成本。

节点保留与否，不看数量多少，而看缺少它后方法论是否失去解释力。

好的方法论通常需要在“稳定主线”和“可枚举分支”之间取得平衡。主线提供默认判断骨架，分支处理高频、可迁移、会改变关注点的典型场景。如果只有主线而没有分支，容易变成一刀切模板；如果只有分支而没有主线，容易退化成案例堆叠或经验集合。

### Expression：表达方式

判断表达是否服务理解，而不是堆格式或下禁令。

方法论文档可以使用文字树、表格、示例、对比、清单和图示，但这些都是表达工具，不是方法本身。

质检时应判断：

```text
开头是否先建立心智模型
主干模型是否降低理解成本
面向 LLM 复用的文档是否显式处理 Few-shot Learning
示例是否帮助理解，而不是替代方法
是否提供必要的正例、反例或边界例来校准抽象节点
示例是否具有经典性，而不是临时项目样例
示例是否覆盖典型误用，而不是只展示顺利场景
示例是否足够精炼，避免把方法论膨胀成案例库
表格和清单是否服务判断，而不是堆信息
是否把模型本身能理解的术语、工具、命令、文件、步骤或常识性枚举展开成教学型清单
是否用过多禁止式表达把方法论规定死
是否把方法论写成线性流水线
是否用精炼分支校准常见变体，而不是只说“按情况调整”
分支是否服务判断，而不是堆案例、堆术语或堆场景
```

好的表达应给读者判断力，而不是只给读者填空格式。

可枚举分支是表达工具，不是强制栏目。只有当某类场景高频、稳定、可迁移，并且会改变方法判断重点时，才适合进入方法论文档；如果只是为了显得全面而罗列可能性，应收束或删除。

教学型清单指的是：为了显得完整，把模型和目标读者通常已经能理解的概念、工具、命令、文件、步骤或术语逐条展开，但这些展开没有增加判断力。质检时应判断这些内容是否真的改变边界、暴露误用或压实抽象；如果只是常识性说明，应收束为判断句或删除。

`Few-shot Learning` 是方法论文档面向 LLM 复用时的注意力锚点。质检重点是示例是否用最少样本降低抽象漂移，而不是补充通用术语背景、工具背景或命令含义。对于容易被模型误用的方法论，可考虑：

```text
正例：按本方法正确处理一个典型场景
反例：看起来像用了方法，但没有完成关键判断
边界例：什么时候不该用这套方法，或应该切换到相邻方法
```

如果示例只是复述方法节点，没有暴露误区、边界或判断动作，它对质检帮助有限。

经典案例通常具备：

```text
高频：在真实工程协作中反复出现
稳定：不依赖单个项目的临时上下文
可迁移：换业务后仍能类比使用
能暴露误区：能展示方法要纠正的错误做法
边界清楚：能说明何时不该继续使用本方法
足够短：服务方法理解，而不是变成案例库
```

如果示例只能在某个项目、某次需求或某个业务名词下成立，它更适合作为项目案例或参考资料，不应作为方法论文档的主案例。

同时也要检查是否混淆了层级：`Few-shot Learning` 可以作为元类层的表达治理概念，但不应被误升为具体业务方法论的业务判断节点。

### Boundary：边界归属

确认适用边界、误用边界和目录归属是否稳定。

方法论必须说明自己不处理什么。边界不清时，文档会膨胀成万能模板，也会和相邻方法论重复。

质检时应判断：

```text
适用场景是否清楚
不适用场景是否清楚
误用是否能防止文档滑向相邻类型
是否应该归属当前目录
是否应该拆分、合并、下沉为参考资料或移动到其他类别
```

如果一篇方法论需要依赖其他索引页才能被理解，说明它的单篇独立性不足。

## 质检结论类型

质检不只输出“好”或“不好”，应明确下一步处理方式。

常见结论包括：

| 结论 | 含义 |
| --- | --- |
| 保持现状 | 方法论已经稳定，只需避免无关扩写 |
| 小幅优化 | 主干成立，只需调整命名、表述、示例组或判断标准 |
| 重写主干 | 核心命题成立，但节点无法支撑方法 |
| 拆分文档 | 一篇文档塞入了两个以上不同对象或方法 |
| 合并文档 | 与已有方法论差异不足，独立维护会造成重复 |
| 下沉资料 | 内容更像示例、参考、教程或经验，不应作为方法论 |

## 判断标准

方法论文档质检是否完成，只看一个标准：

**是否已经能判断这篇文档应保持、优化、重写、拆分、合并还是下沉，并说明这个判断来自对象、命题、节点、表达或边界中的哪一类问题。**

如果只指出“写得不清楚”，但不能说明问题出在哪个层面，质检没有完成。

如果能明确指出文档是否形成稳定方法，并给出可执行的下一步处理方式，质检已经成立。

## 误用

方法论文档质检不是润色。

润色只改善语言；质检要判断方法是否成立。

方法论文档质检也不是统一模板。

不同方法论可以有不同节点、不同表达方式和不同示例密度。质检关注的是解释力、稳定性和边界，而不是格式一致。示例组也不是强制栏目；只有当抽象节点容易发散或误用时，才需要用精炼示例校准。

方法论文档质检也不是背景补充。

质检不应要求文档解释模型本身已经理解的概念、工具、命令、文件或步骤。补充常识性说明会污染上下文，让方法论退化成教学型清单。

方法论文档质检也不是越抽象越好。

过度抽象会抹掉必要动作，流程化又会让方法论退化成待办清单。质检应在二者之间判断平衡点。

## 适用边界

方法论文档质检适合用于检查 `docs/方法论/` 下的单篇方法论文档。

当文档还只是普通主题说明、项目教程、配置手册或一次性复盘时，应先判断它是否值得升格为方法论，而不是直接按方法论文档质检。

当质检结论需要新增一篇方法论时，应回到方法论新增范式，先判断是否有反复出现的问题类型和稳定节点。
