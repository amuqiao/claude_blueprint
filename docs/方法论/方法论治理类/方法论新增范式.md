# 方法论新增范式

> 把反复出现的思考方式，升格成对象清楚、节点稳定、边界明确、可长期复用的方法论。

```text
方法论新增范式
├── Problem：问题类型
│   └── 判断是否存在反复出现、值得抽象的问题
├── Object：方法对象
│   └── 确认这套方法处理的对象是什么
├── Node：稳定节点
│   └── 抽象出必要、独立、可复用的判断动作
├── Standard：完成判断
│   └── 说明这套方法做到什么程度才算成立
└── Boundary：适用边界
    └── 切清它适合什么、不适合什么
```

## 核心命题

新增方法论处理的不是“把一段经验写成文档”，而是“判断某种思考方式是否已经稳定到值得长期复用”。

普通说明、临时提示词、一次性复盘、项目经验和方法论很容易混在一起。方法论新增范式的任务，是把真正可迁移的思考方式抽出来，让它成为后续主题分析、需求分析、方案判断或文档写作可以反复调用的认知工具。

它的核心转化是：

```text
重复问题 -> 稳定对象 -> 方法节点 -> 判断标准 -> 适用边界
```

这里的“方法论”，不是流程清单，也不是输出模板，而是一组能帮助读者判断、解释和迁移的稳定动作。

## 方法论节点

新增方法论不是把已有文档换个标题，而是判断它是否具备成为方法论的条件。

### Problem：问题类型

先判断是否存在反复出现的问题类型。

如果一个问题只在单个项目、单次任务或单个上下文中成立，它更适合写成项目文档、经验记录或任务复盘，不应升格为方法论。

有效的问题类型通常具备：

```text
反复出现
跨主题或跨项目复用
有稳定误区
需要判断而不是只需执行
继续堆信息不能自然解决
```

### Object：方法对象

确认这套方法处理的对象是什么。

方法论必须有对象。对象可以是主题、需求、方案、工具体系、文档、流程或协作问题。对象不清，节点就会漂移；对象过大，方法论会膨胀成百科；对象过小，方法论会退化成一次性技巧。

对象判断通常回答：

```text
这套方法处理什么对象
它不处理什么对象
这个对象为什么值得单独建模
它和相邻对象有什么区别
```

### Node：稳定节点

抽象出必要、独立、可复用的判断动作。

节点不是输出目录，也不是待办步骤。好的节点应表达读者理解或判断问题时必须执行的动作。

节点设计应遵守：

- 英文节点使用高度抽象、常用、单一语义的词。
- 中文解释说明该节点在当前方法论中的具体含义。
- 不用临时造词、业务缩写或只在单篇文档里成立的词。
- 不把两个并列动作硬塞进一个节点名，例如 `A / B`，除非二者共同表达一个不可拆开的抽象动作。
- 不用流程化、待办化的命名替代方法论节点。

节点应在“不过度流程化”和“不过度抽象”之间保持平衡。缺少某个动作会损失解释力时，应保留节点；只是为了显得完整而增加栏目时，应删除。

### Standard：完成判断

说明这套方法做到什么程度才算成立。

判断标准不是验收清单，而是帮助读者判断方法是否真正完成了认知转化。

有效的完成判断通常回答：

```text
读者完成后应能看清什么
哪些现象说明方法没有完成
做到什么程度就可以停止
什么情况说明已经进入其他方法论
```

### Boundary：适用边界

切清它适合什么、不适合什么。

方法论一旦没有边界，就会膨胀成万能模板。边界应说明它的适用前提、失效场景和相邻方法论之间的区别。

边界通常回答：

```text
适合处理什么问题
不适合处理什么问题
什么时候显得过轻
什么时候显得过重
什么时候应该切换到其他方法
```

## 输出形态

新增方法论文档可以参考以下骨架，但不把它当作强制模板：

```text
标题
一句话主旨
主干模型
核心命题
方法论节点
判断标准
误用
适用边界
```

主干模型不是文档目录，而是方法的结构压缩。它应展示这套方法由哪些核心节点组成，以及每个节点解决什么问题。

如果主干模型能降低理解成本，优先使用简单的文字树：

```text
方法论名称
├── Node A：节点名
│   └── 这个节点解决什么问题
├── Node B：节点名
│   └── 这个节点解决什么问题
└── Node C：节点名
    └── 这个节点解决什么问题
```

具体文档可以按主题需要增删章节，但应保持以下表达顺序：

```text
先说明它解决什么问题
再说明它如何工作
再说明如何判断是否完成
最后说明它不适合什么
```

## 判断标准

新增方法论是否成立，只看一个标准：

**它是否把一种反复出现的问题，抽象成了对象清楚、节点稳定、判断明确、边界可迁移的方法。**

如果文档只是步骤清单、模板目录、经验记录或一次性任务复盘，说明它还不是方法论。

如果文档能帮助读者在新问题中复用同一组判断动作，而不是只能照抄原案例，方法论已经成立。

## 误用

方法论新增不是文档包装。

不能把普通说明、临时提示词、具体项目经验或一次性判断过程包装成方法论。

方法论新增也不是节点堆叠。

不要为了看起来完整而强行制造节点、图示或章节。如果一个方法只能写成步骤清单，说明它尚未形成稳定抽象。

方法论新增也不是过度抽象。

不要为了减少节点而把必要动作折叠到过大的抽象词里；对比表、示例、清单、图示都可以作为表达工具，关键是它们是否服务于方法论的解释目标。

## 适用边界

方法论新增范式适合用于判断一种思考方式是否值得进入 `docs/方法论/`。

当内容只是具体主题说明、项目教程、临时笔记、工具使用记录或一次性任务复盘时，不应使用这套方法强行升格。

当已有方法论出现节点漂移、边界膨胀或表达退化时，应使用方法论文档质检方法论，而不是重新新增一篇方法论。
