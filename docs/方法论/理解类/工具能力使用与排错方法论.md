# 工具能力使用与排错方法论

> 让工具能力从“知道有这些东西”，变成“能区分、会使用、能选择、可排错”。

```text
工具能力使用与排错
├── Distinguish：区分
│   └── 分清概念、命名、边界和映射关系
├── Use：使用
│   └── 说明入口、路径、位置和格式
├── Decide：决策
│   └── 判断用什么、放哪层、何时不用
└── Debug：排错
    └── 解释优先级、覆盖规则、信任机制和常见陷阱
```

## 核心命题

工具能力使用与排错处理的不是“读者不知道这个工具是什么”，而是“读者面对一组能力时不知道如何正确使用”。

有些主题不是单一概念，而是一套由能力、入口、配置、文件位置、权限、优先级和运行规则组成的系统。读者即使知道每个名词，也可能无法判断它们之间的分工、该把配置放在哪里、什么时候用哪个能力、出问题时从哪里查。

它的核心转化是：

```text
能力清单 -> 使用地图 -> 独立操作与判断
```

这里的“使用地图”，不是功能大全，也不是教程步骤，而是帮助读者在真实场景中做四件事：分清对象、找到路径、做出选择、定位问题。

## 方法论节点

工具能力使用与排错不是固定写作模板，但它有稳定的方法论节点。

这些节点不是栏目要求，而是判断一套工具能力是否真正可被掌握的抽象动作。

### Distinguish：区分

先把容易混在一起的能力切开。

工具体系里常见的问题，不是概念太少，而是概念太多且命名相近。比如同一个系统里可能同时存在 `Instruction`、`Rule`、`Skill`、`Command`、`Plugin`、`Agent`、`MCP`、`Profile`、`Settings`、`Memory` 等对象。

区分要完成的不是逐个下定义，而是建立映射关系和边界感：

```text
它是什么
它不是什么
它和相邻能力怎么分工
它在不同工具里是否有命名差异
它解决的问题属于哪一类
```

如果区分没有完成，后面的使用路径会失真；读者会把不同层级、不同职责、不同生命周期的能力混用。

### Use：使用

说明能力如何落到具体操作。

使用不是教程步骤，也不是泛泛说“可以配置”“可以扩展”“可以调用”，而是判断能力是否已经落到可找到、可触发、可验证的操作路径。

有效的使用说明通常回答：

```text
从哪里创建
放在哪个目录或配置文件
最小格式是什么
怎么触发
怎么验证它生效
```

对配置型、CLI 型、插件型工具来说，使用说明必须落到文件和命令层面。没有路径、格式和验证方式，读者仍然只能停留在概念理解。

### Decide：决策

帮助读者判断什么时候用什么。

工具能力越多，越需要决策说明。否则读者会把所有能力都当成“都可以试试”的选项，最终导致配置分散、职责重叠、维护困难。

决策说明要回答：

```text
这个场景优先用哪个能力
为什么不用另一个相近能力
配置应该放在个人层、项目层还是局部层
什么时候不应该抽象、不应该新建、不应该全局化
```

决策的重点不是给唯一答案，而是给判断依据。好的决策说明能让读者在新场景里迁移，而不是只会照抄一个案例。

### Debug：排错

说明能力失效时如何判断。

工具能力体系常见的问题不是“完全不能用”，而是“看起来配置了，但没有按预期生效”。这类问题通常来自优先级、覆盖规则、权限、信任机制、作用域、缓存或触发条件。

排错说明要优先解释系统规则，再列常见陷阱：

```text
谁覆盖谁
哪个配置优先级更高
当前文件是否被信任
当前能力是否在正确作用域内
触发条件是否满足
如何确认它已经被加载
```

没有排错节点，文档只能帮助读者在理想路径中使用工具，不能帮助读者在真实环境中独立解决问题。

## 判断标准

工具能力使用与排错是否完成，只看一个标准：

**读者是否已经能在具体场景中区分能力、找到操作路径、做出选择判断，并在能力未生效时定位优先级、作用域或信任机制问题。**

如果读者看完只知道工具“有哪些能力”，但不知道该用哪个、放哪里、怎么确认生效，说明文档没有完成。

如果读者不记得全部细节，但能按场景查到路径、按决策规则选择能力、按排错顺序定位问题，说明文档已经成立。

## 误用

工具能力使用与排错不是功能大全。

功能大全追求覆盖；使用与排错追求可操作、可选择、可定位。

工具能力使用与排错也不是入门教程。

教程通常带读者完成一条固定路径；使用与排错要让读者面对不同场景时能自己判断。

`Distinguish / Use / Decide / Debug` 是方法论节点，不是输出目录。主题不同，真正需要展开的节点权重也不同。对于概念混乱的工具，区分要更重；对于配置层级多的工具，决策和排错要更重；对于新手 onboarding 文档，使用路径和格式示例要更重。

## 适用场景

这套方法适合说明由多个能力、入口、配置和规则组成的工具体系，例如：

```text
CLI 工具能力体系
开发者平台配置体系
插件 / Skill / Agent / MCP 扩展体系
CI / 部署 / 权限 / 环境配置体系
团队 onboarding 知识库
```

## 适用边界

工具能力使用与排错适合处理“知道有这些能力，但不知道如何正确使用”的问题。

当一个主题存在多个能力入口、配置文件、作用域、优先级、权限、触发条件或常见失效路径时，应使用这套方法。

工具能力使用与排错不适合替代基础概念解释、生态比较、产品介绍或完整项目教程。它的重点是让读者在真实场景中能区分能力、找到路径、做出选择并定位问题。

如果能力入口很少、规则简单、没有明显选择和排错问题，使用这套方法会显得过重。
