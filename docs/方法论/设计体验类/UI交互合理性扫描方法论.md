# UI 交互合理性扫描方法论

> 像扫描核一样观察一个界面、组件或流程，先自行建立必要心智模型，再判断 UI 和交互是否符合用户理解、操作路径和体验连续性。

```text
UI 交互合理性扫描
├── Input：输入界定
│   └── 明确当前可观察材料、用户诉求和技术语境是什么，以及能从中判断到什么程度
├── Frame：心智建模
│   └── 自行建立产品类型、用户任务、页面角色和主要使用情境
├── Structure：结构识别
│   └── 看清页面对象、信息层级、操作入口和状态边界
├── Affordance：可理解性判断
│   └── 判断用户是否能看懂什么可点、点了会发生什么、当前处于什么状态
├── Continuity：连续性判断
│   └── 判断展开、收起、跳转、筛选、弹层和反馈是否形成自然因果
├── Rhythm：视觉节奏判断
│   └── 判断空间、密度、对齐、比例、留白和强调是否服务任务
├── Stress：压力测试
│   └── 用极端内容、设备尺寸、动态字体、空态和异常状态检验体验是否稳
├── Direction：设计转化
│   └── 把诊断问题转成结构、状态、视觉、动效或组件层面的设计动作
└── Judgment：诊断输出
    └── 给出问题分级、原因解释、调整方向和可选下一步
```

## 核心命题

UI 交互合理性扫描处理的不是“这个界面好不好看”，而是“这个界面是否能被用户自然理解、顺畅操作，并在状态变化中保持连续”。

真实产品迭代中，体验问题经常不是单个元素的问题，而是局部元素和整体心智模型之间失配。例如：

```text
标题像按钮，但用户不知道它能筛选
图标看起来是装饰，但实际承担关键入口
展开面板像外来模块，不像从触发点生长出来
卡片内容看起来漂亮，但图片比例破坏阅读节奏
组件各自合理，但组合后页面主次关系混乱
```

这套方法论的任务，是在输入可能不完整的情况下，先建立足够使用的临时心智模型，再用稳定观察节点扫描界面，判断体验问题出在哪里。

它的核心转化是：

```text
界面材料 / 用户诉求
-> 输入界定
-> 临时心智模型
-> 结构识别
-> 可理解性判断
-> 连续性判断
-> 视觉节奏判断
-> 压力测试
-> 设计转化
-> 诊断建议
```

这套方法论不要求输入必须完整。输入可以是截图、录屏、代码、设计稿、产品描述、用户反馈、单个组件，也可以是一段模糊的体验抱怨或用户自己的优化建议。上下文越多，判断越具体；上下文不足时，方法论应显式说明假设，而不是停止分析。

## 方法论节点

这些节点不是固定输出模板，而是面对 UI 与交互问题时应执行的判断动作。简单问题可以轻量使用，复杂页面需要完整扫描。

### Input：输入界定

先判断当前能观察到什么。

UI 交互扫描经常面对不完整输入。只有一张截图时，能判断视觉层级、信息密度、入口显性和部分状态表达；有录屏时，才能判断动效、连续性和操作反馈；有代码时，才能判断组件边界、状态来源和可维护性；有用户建议时，还要判断这句话背后真正想改变的是状态、位置、显示规则、交互入口还是视觉节奏。

`Input` 通常回答：

```text
当前输入是什么：截图、录屏、代码、设计稿、自然语言描述或真实页面
是否包含用户原始诉求，哪些词可能存在歧义
输入覆盖的是整体页面、局部组件、单次交互还是完整流程
输入是否包含平台、框架、组件体系或代码语境
哪些判断可以直接做
哪些判断只能基于假设
哪些信息缺失会影响结论可信度
```

输入界定不是降低要求，而是防止把不可见内容说成已验证。

如果输入包含口语化建议、模糊体验反馈或用户自己的优化想法，应在 `Input` 中按需做诉求澄清。诉求澄清不是固定主流程节点，而是输入阶段的分支动作；没有用户建议时，方法论仍应自动建立心智模型并扫描界面。

诉求澄清的任务，是把用户的口语化体验表达转成可分析、可交接、可实现的交互语义。

UI 交互扫描不只处理已有界面，也经常处理用户自己的优化建议。用户可能不熟悉具体平台、框架或客户端术语，因此会使用“消失”“上面”“标题菜单”“怪”“更自然”这类体验词。方法论应先澄清这些词在当前界面中对应什么状态、触发条件、布局位置、组件角色和显示规则，再进入后续分析。

按需诉求澄清通常回答：

```text
用户原话想解决什么体验问题
原话中的模糊词分别可能指什么
应该把诉求改写成哪些 UI 状态
状态变化由什么触发：滚动、点击、展开、收起、路由切换、筛选变化或加载完成
每个状态下显示什么、隐藏什么、替代什么
关键元素承担什么角色：页面身份、当前范围、筛选入口、主操作、状态提示或临时工具
当前技术语境下应使用哪些常用术语表达，避免建议太泛
```

诉求澄清不是替用户改需求，也不是提前写实现方案。它只是把“我感觉应该这样”转成大模型和开发者能稳定识别的交互描述，避免后续建议因为语言模糊而被错误实现。

当输入包含具体技术栈时，输出应使用对应语境中能帮助定位问题的常用术语。术语应服务状态、组件边界、交互触发、布局约束和验证方式，而不是展开成某个框架的术语清单。

术语适配不是堆技术名词。只有当术语能让建议更接近当前代码、设计工具或平台语境时，才应进入输出；如果术语不改变判断，应使用更通用的体验语言。

### Frame：心智建模

在分析界面前，先自行建立必要心智模型。

心智模型不是项目文档复述。即使没有项目文档，也应从产品类型、用户任务、页面角色、使用频率和上下文推断出一个临时模型。

`Frame` 通常回答：

```text
这个界面属于什么产品类型
用户来到这里主要想完成什么
当前页面在产品中承担什么角色
用户第一眼应该先理解什么
哪些对象是内容，哪些对象是工具，哪些对象是状态
这个场景更需要沉浸阅读、快速操作、精确管理还是探索回看
```

如果已有项目文档、用户旅程或设计原则，应把它们作为上下文吸收；如果没有，也应先建立可解释的临时模型。后续所有判断都要回到这个模型，而不是只看单个元素漂不漂亮。

### Structure：结构识别

识别页面中的对象、层级和边界。

很多体验问题来自结构混淆：标题同时像入口，筛选状态藏在面板里，工具按钮和内容区争夺视觉中心，展开区不像属于触发入口。

`Structure` 通常观察：

```text
页面身份
主要内容
主动作
次级工具
状态提示
筛选或范围
弹层、展开区或临时面板
固定元素和滚动内容
```

结构识别的目标不是列组件，而是看清用户会如何分组理解这些对象。如果用户必须猜“这个东西到底属于导航、内容、筛选还是装饰”，结构就没有成立。

### Affordance：可理解性判断

判断界面是否让用户知道可以做什么。

`Affordance` 不只看按钮是否明显，也看一个元素承担的语义是否和它的形态一致。可点元素必须有可点感；状态元素不能长得像按钮；关键入口不能只靠用户猜。

`Affordance` 通常回答：

```text
什么元素可点击、可拖动、可展开、可编辑
用户能否在不读说明的情况下理解入口用途
图标、文字、颜色、边框和位置是否共同表达同一语义
当前状态是否可见，例如选中、展开、筛选、禁用、加载
操作前后是否有反馈
icon-only 控件是否有可访问语义和足够触达面积
```

如果一个元素看起来像 A，实际做 B，或用户必须试错才能知道它能做什么，说明可理解性不足。

### Continuity：连续性判断

判断状态变化是否有自然因果。

交互割裂通常不是因为某个面板不好看，而是因为用户感受不到它从哪里来、为什么出现、关闭后状态还在不在、选择后影响了哪里。

`Continuity` 通常观察：

```text
触发点和结果区域是否空间上有关联
展开、收起、弹出、跳转是否符合用户预期
动效是否表达因果，而不是单纯装饰
筛选、选择、编辑、删除后的状态是否持续可见
关闭面板是否等于取消操作，还是仅隐藏工具
返回后是否保留滚动位置、筛选状态和用户上下文
```

连续性成立时，用户会觉得“这是刚才那个动作带来的结果”。连续性不成立时，用户会觉得界面突然插入、突然跳走或状态丢失。

### Rhythm：视觉节奏判断

判断视觉组织是否服务任务。

视觉节奏不是审美偏好，而是阅读、比较和操作成本。间距、密度、对齐、字号、图片比例、留白和强调关系都应服务当前页面任务。

`Rhythm` 通常观察：

```text
内容密度是否匹配使用场景
标题、正文、标签、图片和操作入口是否有清楚层级
空间留白是否表达分组，而不是制造割裂
对齐关系是否帮助用户扫描
图片比例是否稳定，是否压迫正文或破坏卡片节奏
主色、情绪色、状态色是否各司其职
装饰是否抢走内容注意力
```

视觉节奏问题常表现为“单个组件都没错，但整体不顺”。这类问题应先找层级和分组原因，再决定是否调整尺寸、间距、颜色或比例。

### Stress：压力测试

用边界条件检验界面是否稳。

一个界面在理想数据下成立，不代表体验成立。UI 交互扫描必须用常见压力条件检查它是否会变形、误导或失去可用性。

常用压力条件包括：

```text
窄屏、小屏、横屏和大屏
动态字体放大
长标题、长正文、多标签
无图片、单图、多图、极宽图、极高图
空状态、筛选空态、加载态、错误态
快速连续点击
滚动到顶部、底部和展开区附近
亮色、暗色和低对比环境
```

压力测试不是要求每次都完整跑设备矩阵，而是用边界思维暴露“当前设计是否只在理想状态下好看”。

### Direction：设计转化

把扫描结论转成可执行的设计动作。

UI 交互合理性扫描不能停在“哪里怪”或“哪里不顺”。诊断应进一步判断：这个问题应该通过调整结构、状态、视觉、动效还是组件边界解决。否则建议容易变成泛泛的审美意见，或直接跳到代码改法。

`Direction` 通常回答：

```text
问题对应的设计变量是什么：层级、密度、对齐、色彩、字号、图标、动效、状态或组件边界
问题在当前技术语境中应如何命名，避免只停留在泛化描述
应该先调整关系，还是先调整单个元素
应该强化、弱化、合并、拆分、靠近、远离、固定、隐藏还是延迟出现
哪些变量会影响用户理解和操作，哪些只是风格偏好
推荐方向是什么
不推荐方向是什么
调整后的体验应如何被验证
```

设计转化不是完整视觉系统设计。它不负责定义品牌气质、组件规范或全局设计语言，而是把当前体验问题落到足够具体的设计变量上，让下一步设计或实现有明确抓手。

### Judgment：诊断输出

把观察结果转成可行动的判断。

有效诊断不应只说“丑”“割裂”“不自然”，也不应直接跳到代码修改。它要说明问题属于什么层级，为什么会发生，影响用户什么理解或操作，以及可以怎样调整。

`Judgment` 通常输出：

```text
主要问题
问题层级：心智模型、结构、可理解性、连续性、视觉节奏或边界状态
用户感知到的症状
造成症状的原因
澄清后的交互语义：仅在输入包含模糊诉求时输出
对应设计变量
建议调整方向
不推荐调整方向
建议先改什么、后改什么
哪些内容不应一起改
验证方式
```

复杂问题可以给出多个候选方向，但应明确推荐方向和前提。诊断输出的目标是让下一步设计或实现有方向，而不是制造更多可能性。

## 输出形态

UI 交互合理性扫描的输出可以按场景轻重变化。

### 轻量输出

适合用户只问一个局部问题，例如“标题和导航为什么怪”。

```text
我的理解
主要问题
为什么会这样
建议优先调整
不要一起改的内容
```

### 标准输出

适合分析一个页面、截图或交互片段。

```text
临时心智模型
诉求澄清：仅在输入包含模糊诉求时输出
结构判断
可理解性问题
连续性问题
视觉节奏问题
压力风险
设计转化
建议顺序
```

### 实施前输出

适合准备进入代码修改前。

```text
本轮目标
本轮不改
具体调整点
影响文件或组件
验收场景
回退边界
```

实施前输出属于执行准备，不是本方法论的必要结尾。方法论的核心是完成体验判断；是否进入实现，由项目阶段、版本边界和用户意图决定。

## 判断标准

UI 交互合理性扫描是否完成，只看一个标准：

**是否已经建立足够使用的用户心智模型，并能解释界面体验问题的来源、影响和调整方向。**

如果只列出视觉偏好，说明扫描没有完成。

如果只指出“哪里不好看”，但不能说明它破坏了什么任务、层级、状态或连续性，说明扫描没有完成。

如果直接给出代码改法，却没有说明用户为什么会更容易理解或操作，说明扫描没有完成。

如果能让读者看清：

```text
用户原本期待什么
当前界面让用户误解什么
用户口语诉求应如何转成稳定交互语义：仅在输入包含模糊诉求时需要
问题发生在哪个体验层级
应该优先调整哪个关系
问题应落到哪些设计变量
哪些东西不应被误改
```

则扫描已经成立。

## Few-shot Learning 示例组

### 正例：列表筛选面板割裂

用户说“点筛选后弹出来一块，像突然插进来的，不像原来页面的一部分”。

合适做法不是马上重做导航，而是先建模：

```text
页面角色：内容列表或管理列表
筛选入口角色：范围控制工具
筛选面板角色：临时控制层
用户期待：从筛选入口自然展开，并能理解它会影响当前列表结果
```

再判断问题可能来自：

```text
触发点和展开区空间关系弱
面板出现后没有和原筛选入口形成连续
已选条件没有持续可见
关闭面板和清除筛选语义可能混淆
面板背景、边界、间距或高度让它像外来模块
列表内容跳动过大，用户不知道结果范围是否已经变化
```

此时设计转化应落在触发点连续性、空间紧凑度和状态提示上，例如让面板贴近筛选入口、保留已选条件可见性、区分关闭和清除，而不是直接改成另一套筛选入口。

### 正例：按需把大标题诉求转成折叠态标题策略

用户说“我希望主页大标题消失的时候，正上方能够显示标题菜单”。

当输入包含这类口语化建议时，合适做法不是直接让模型“在上方加一个标题菜单”，而是先澄清这句话背后的交互语义：

```text
用户原话中的“消失”：更可能是大标题随滚动进入折叠态，而不是被删除
用户原话中的“正上方”：更可能是顶部导航栏中间区域，而不是滚动内容顶部
用户原话中的“标题菜单”：更可能是紧凑标题 + 筛选入口的组合控件，而不是普通标题文本
```

再转成更稳定的设计表达：

```text
展开态：只显示大标题 `时刻`，主页重点是时间轴。
折叠态：才显示中间的 `时刻 + 筛选` 紧凑标题菜单。
触发条件：大标题随滚动离开主要视区，顶部导航进入 collapsed 状态。
显示规则：展开态避免重复标题；折叠态保留页面身份和筛选入口。
交互意图：用户滚动后仍知道当前页面和当前范围，并能继续触达筛选。
```

如果分析 Flutter 代码，可以进一步使用当前语境术语表达为：

```text
这不是新增一个独立顶部标题，而是处理 `SliverAppBar` 或滚动标题区域的 expanded / collapsed 状态切换。
紧凑标题菜单应属于折叠态导航区域，而不是时间轴内容区的新组件。
```

这种转译能让后续实现模型知道应处理的是状态策略、导航区域和筛选入口显隐关系，而不是随意添加一个顶部元素。

### 反例：把体验问题当成样式或组件重构

用户说“这个卡片看着不顺，不知道该先看哪里”，分析者直接调整圆角、阴影、渐变和颜色，或者把卡片拆成新的组件结构。

这可能让局部看起来更精致，但没有解释用户为什么不知道先看哪里，也没有判断问题来自信息层级、内容密度、图片比例、主操作位置还是状态表达。

问题在于：看起来做了 UI 优化，实际跳过了 `Frame`、`Structure` 和 `Rhythm` 的关键判断。正确做法应先说明卡片承担什么任务、用户第一眼应看什么、哪些元素在争夺注意力，再决定是调间距、字号、比例、强调关系，还是需要改变组件结构。

### 边界例：明确工程错误

如果问题是明确的布局报错、图片加载异常、点击事件没有触发、路由跳转失败或接口错误，应优先使用对应工程排错方法。

例如 Flutter 中的 `RenderFlex overflowed`、SwiftUI 中的 `Layout cycle` 或 React 中的事件处理函数未触发，都不是 UI 交互合理性扫描的主对象。UI 交互合理性扫描可以帮助判断这些错误造成的体验影响，但不替代具体技术排障。

## 误用

UI 交互合理性扫描不是审美投票。

不要把个人喜欢的风格当成结论。所有判断都应回到用户任务、页面角色、状态表达和操作路径。

UI 交互合理性扫描也不是强制小步优化。

它可以分析局部，也可以分析整体；可以导向微调，也可以导向大改。是否小步执行，是项目节奏和风险控制问题，不是这套方法论的限制。

UI 交互合理性扫描也不是项目文档复述。

已有文档可以作为上下文，但方法论不能依赖项目文档才能工作。没有文档时，也应基于输入材料和产品常识建立临时模型。

UI 交互合理性扫描也不是术语堆砌。

术语的作用是让诊断和建议更贴近当前平台、框架和组件语境，而不是把体验问题改写成技术教程。不能因为输入是 Flutter、SwiftUI 或 React 代码，就跳过用户任务、状态表达和交互连续性判断。

UI 交互合理性扫描也不是代码实现方案。

它可以给出实施前建议，但核心价值是判断体验是否合理。进入代码实现时，应结合工程架构、组件边界、验证入口和回退策略。

## 适用边界

适合使用：

- 分析一个页面、组件、截图、录屏或交互流程是否合理。
- 用户反馈“怪”“割裂”“不顺”“不知道点哪里”，但原因还不清楚。
- 用户提出口语化优化建议，需要转成清楚的状态、触发、显示规则和技术语境表达。
- 设计和代码都能跑，但体验质量不稳定。
- 准备做 UI / 交互优化前，需要先判断问题层级。
- 没有完整项目文档，但需要快速建立临时心智模型。

不适合使用：

- 纯工程 bug 排查。
- 纯视觉风格选择，例如只问某个颜色好不好看。
- 已经明确的实现任务，不需要体验判断。
- 完整产品战略、需求拆解或技术方案设计。

当问题已经从体验判断进入“怎么实现、改哪些文件、如何验证”时，应切换到对应工程开发、方案分析或项目方法论。
