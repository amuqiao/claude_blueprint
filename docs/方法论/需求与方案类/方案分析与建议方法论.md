# 方案分析与建议方法论

> 把已经明确的需求，推进成可比较、可落地、可验证的执行方案，并给出有前提、有取舍、有闭环的推荐建议。

```text
方案分析与建议
├── Goal：目标校准
│   └── 确认方案必须解决什么、不能扩大什么、如何验收
├── Capability：能力调研
│   └── 找到可用的第三方服务、开源依赖、本地模型、平台能力或自研路径
├── Option：候选方案
│   └── 把不同能力组合成可执行方案，而不是只写方向
├── Architecture：落地形态
│   └── 明确服务形态、接口契约、状态模型、数据流和失败处理
├── Constraint：约束条件
│   └── 判断业务、技术、资源、成本、时延和稳定性约束
├── Impact：影响范围
│   └── 看清方案会影响哪些系统、用户、数据、流程和测试口径
├── Tradeoff：取舍判断
│   └── 比较每个方案用什么代价换来什么收益
├── Evolution：演进路径
│   └── 判断当前版本怎么落地，后续如何升级
└── Recommendation：建议闭环
    └── 给出推荐方案、接口草案、依赖选择、验证方式、风险和回退方式
```

## 核心命题

方案分析处理的不是“需求是否清楚”，而是“需求已经清楚，但还没有形成能执行、能比较、能验收的落地方案”。

真实工程里，方案文档最常见的问题不是没有方向，而是只有方向。例如：

```text
推荐后端统一编排
推荐 AI 检测兜底
推荐异步任务处理
推荐接入 OCR 能力
推荐优化展示状态
```

这些表达只能说明大致倾向，不能指导研发开工。研发仍然不知道：用什么服务、什么包、什么模型、什么接口、什么入参、什么返回状态、失败如何处理、如何验收、如何回退。

方案分析与建议的任务，是把明确需求转成 **可执行方案空间**，再基于能力调研、落地形态、约束、影响和取舍形成推荐。

它的核心转化是：

```text
明确需求
-> 目标校准
-> 能力调研
-> 候选方案
-> 落地形态
-> 约束条件
-> 影响范围
-> 取舍判断
-> 演进路径
-> 建议闭环
```

这里的“方案”，不是抽象方向，也不是完整代码实现，而是足够让研发、测试、产品继续推进的执行设计。

有效方案至少要回答：

```text
用什么能力解决问题
为什么选它而不是别的能力
系统边界放在哪里
接口或交互契约大致是什么
状态如何表达
失败如何兜底
如何验证方案成立
如果前提不成立如何回退
```

## 方法论节点

这些节点不是固定输出目录，而是面对明确需求时应执行的判断动作。简单需求可以轻量表达，复杂需求必须展开到足够可执行。

### Goal：目标校准

先确认方案必须解决什么问题。

需求已经明确，不代表方案目标天然清楚。方案分析前，需要把必须满足的目标、不能顺手扩大的范围、验收口径和成功标准定住。

目标校准通常回答：

```text
这个方案必须解决什么问题
哪些内容不属于本次方案目标
哪些结果是验收必须看到的
哪些体验、性能、数据或流程不能被破坏
如果需求有优先级，哪个目标最重要
```

目标校准不是重新拆需求。如果目标仍然说不清，应回到需求澄清，而不是继续设计方案。

### Capability：能力调研

先了解有哪些现实可用的能力，再组合方案。

很多方案写得虚，是因为没有进入实践层面调研。比如“做 OCR”不能只写“接入 OCR 能力”；至少要比较 AI 大模型图文识别、云 OCR 服务、Python OCR 包、本地模型或现有内部能力。又比如“短文本语种识别”不能只写“AI 检测”，需要说明能否用 Python 依赖包、本地模型、第三方 API 或大模型兜底。

能力调研通常覆盖：

```text
已有系统能力
第三方服务或 SaaS 能力
开源依赖包
本地模型
云模型或大模型能力
自研规则或算法
人工运营或半自动兜底
```

能力调研不是简单列名词，而是要让读者知道这些能力是否能解决当前需求。

有效的能力调研通常回答：

```text
这个能力具体叫什么
接入方式是什么
输入输出是什么
是否支持当前数据形态
是否适合当前边界值和异常场景
性能、成本、准确率和稳定性大致如何
部署和运维成本是什么
主要风险是什么
```

如果需要使用依赖包、模型或第三方服务，应尽量给出具体名称和试用路径。例如：

```text
OCR：PaddleOCR、Tesseract、云 OCR API、多模态大模型
语种识别：lingua-language-detector、fastText lid.176、langdetect、第三方语言检测 API、大模型分类
文本审核：规则词库、分类模型、第三方审核 API、大模型审核
```

这些名称不是结论，而是进入方案比较的候选能力。最终推荐必须结合项目语言、部署环境、准确率、成本和维护能力判断。

### Option：候选方案

把能力组合成可执行方案，而不是直接认定唯一做法。

候选方案应从“方向”升级为“可执行路径”。例如“后端统一编排”仍然太粗，需要进一步说明它是同步接口、异步任务、批量任务、内部服务，还是已有接口扩展。

候选方案通常来自：

```text
前端展示层处理
后端同步接口处理
后端异步任务处理
数据模型或状态模型调整
配置化或策略化处理
调用已有内部能力
接入第三方服务
引入开源依赖包
部署本地模型
调用大模型能力
短期兼容方案
长期平台化方案
```

候选方案不是越多越好。无明显可行性、明显违背约束或无法验收的想法，不应为了凑数量进入比较。如果经过判断只有一个现实可行方案，也要说明其他路径为什么不成立。

### Architecture：落地形态

把推荐方向落到系统可以执行的形态。

这是防止方案浮在半空中的关键节点。方案文档不一定要写到完整技术设计，但必须给出足够明确的工程轮廓，让研发知道下一步怎么设计和验证。

落地形态通常回答：

```text
服务放在哪里
是新增接口、扩展旧接口，还是新增后台任务
接口入参是什么
接口返回哪些核心字段
状态模型有哪些状态
核心处理流程是什么
失败、超时、空结果、低置信度如何处理
是否需要配置项、开关、阈值或灰度策略
是否需要日志、监控、审计或人工复核入口
```

对后端方案，至少应给出接口草案或调用契约。例如：

```text
POST /language-detection/detect

Request:
{
  "text": "OK",
  "source_locale": "en",
  "target_locale": "zh-CN",
  "scene": "chapters_dialogue"
}

Response:
{
  "status": "normal | abnormal | failed | timeout | low_confidence | not_applicable",
  "detected_language": "en",
  "confidence": 0.93,
  "provider": "lingua | fasttext | llm | third_party",
  "display_as_normal": true,
  "reason": null
}
```

接口草案不是最终接口设计，但它能暴露方案是否真的想清楚了。只要方案涉及前后端协作、服务调用、任务流转或状态展示，就应给出类似契约。

### Constraint：约束条件

判断哪些条件会限制方案选择。

约束不是方案的附属说明，而是决定方案能否成立的硬条件。忽略约束，方案会看起来可行，落地时才暴露成本和风险。

常见约束包括：

```text
业务规则和合规要求
已有系统能力和接口边界
数据来源、准确性和一致性
第三方服务限制
开源依赖许可证和供应链风险
模型部署资源和运行成本
性能、时延和稳定性要求
上线窗口和资源投入
兼容历史数据或旧流程的要求
```

约束条件要区分硬约束和软约束。硬约束决定方案是否可行；软约束影响方案优先级。

### Impact：影响范围

看清方案会影响哪些系统、用户、数据和流程。

方案不是孤立改动。一个看似局部的实现，可能影响展示、统计、导出、权限、缓存、监控、客服解释、运营流程或测试口径。

影响范围通常回答：

```text
影响哪些页面、接口、任务或数据链路
影响哪些用户角色或使用场景
是否改变已有状态、统计、导出或通知语义
是否影响历史数据、缓存或兼容逻辑
是否新增依赖、服务、任务、配置或监控
测试和验收需要覆盖哪些路径
```

影响范围不是为了制造复杂度，而是防止方案只在局部成立。

### Tradeoff：取舍判断

比较每个方案用什么代价换来什么收益。

方案很少只有“好”和“不好”。更常见的是，一个方案低成本但短期，另一个方案更稳但投入更高；一个方案体验好但依赖外部能力，另一个方案可控但覆盖有限。

取舍判断通常比较：

```text
实现成本
交付速度
准确率或业务效果
用户体验
性能和时延
稳定性和可观测性
数据一致性
依赖风险和维护成本
扩展能力
失败后的影响
```

取舍判断不追求绝对最优，也不是选能力最强的方案，而是把每个方案在当前阶段的收益、代价、风险和演进空间讲清楚，让推荐有依据。

### Evolution：演进路径

判断当前阶段做到哪一层，以及后续如何升级。

方案分析不能只看眼前，也不能一开始就追求最终形态。很多工程方案需要分阶段推进：短期先满足确定需求，中期补齐稳定性和维护性，长期再进入平台化、架构化或体系化演进。

演进路径通常回答：

```text
当前版本最小可落地方案是什么
短期方案能支撑到什么边界
什么条件会触发中期升级
长期方案是否有必要现在进入设计
当前实现是否保留升级路径
当前方案是否会锁死未来演进
```

演进路径不是鼓励过度设计。它的作用是防止两类问题：只看眼前导致后续推倒重来，或者只看最终形态导致当前需求成本过高。

### Recommendation：建议闭环

给出推荐方案、适用前提、落地草案、风险、验证和回退方式。

建议结论不能只写“推荐方案 A”。它至少要说明为什么推荐、在什么条件下推荐、如何落地、如何验证、需要接受什么风险、哪些情况应该换方案。

有效的建议结论通常包括：

```text
推荐哪个方案
为什么它更适合当前需求
具体采用哪些能力、包、模型、服务或平台能力
接口、任务、状态或数据契约的草案是什么
它依赖哪些前提
主要风险是什么
如何验证它是否有效
失败或前提不成立时如何回退
后续满足什么条件时需要升级
```

如果多个方案都可行，应说明推荐优先级；如果当前信息不足以推荐，应明确缺少哪些调研或试验结果，而不是假装已经能判断。

## 表达方式

方案分析可以使用能力调研表、候选方案表、接口草案、状态矩阵、演进表、取舍表、影响矩阵或建议摘要。这些只是表达工具，不是固定模板。

### 能力调研表

| 能力 | 接入方式 | 适用场景 | 主要风险 |
| --- | --- | --- | --- |
| 开源依赖包 | 项目内安装依赖，直接调用 | 轻量、低成本、可离线 | 准确率、维护状态、许可证和供应链风险 |
| 本地模型 | 服务内加载模型或独立模型服务 | 数据不出内网、可控性高 | 部署资源、模型更新和性能成本 |
| 第三方 API | 调用外部服务 | 能力成熟、接入快 | 成本、限流、数据合规和稳定性 |
| 大模型能力 | 调用文本、多模态或推理模型 | 复杂语义、低样本、兜底判断 | 成本、时延、稳定性和结果可解释性 |

### 候选方案表

| 方案 | 核心做法 | 落地形态 | 适用前提 | 主要风险 |
| --- | --- | --- | --- | --- |
| 保守方案 | 在现有能力上补局部逻辑 | 扩展旧接口或配置 | 现有能力足以支撑需求 | 后续扩展性有限 |
| 工程方案 | 引入依赖包或内部服务 | 新增服务函数或 API | 需求需要稳定复用 | 需要评估准确率和维护成本 |
| 模型方案 | 接入本地模型或大模型 | 模型服务或外部 API | 规则和普通包覆盖不足 | 成本、时延和可解释性风险 |
| 异步方案 | 任务队列补检或回写 | 后台任务和状态流转 | 批量、耗时、失败重试要求高 | 状态复杂，用户等待成本高 |

### 接口草案

```text
接口：
请求：
响应：
状态：
错误码：
幂等或重试：
超时策略：
```

接口草案用于暴露方案边界，不要求替代正式接口设计。

### 状态矩阵

| 状态 | 业务含义 | 展示或后续动作 |
| --- | --- | --- |
| normal | 已检测且正常 | 可展示正常状态 |
| abnormal | 已检测但异常 | 展示异常并进入问题处理 |
| failed | 检测失败 | 展示失败或允许重试 |
| timeout | 检测超时 | 展示待重试或未完成 |
| low_confidence | 结果可信度不足 | 不按正常处理，必要时人工确认 |
| not_applicable | 不适用检测 | 跳过或标识不适用 |

### 建议摘要

```text
推荐方案：
推荐理由：
具体能力：
落地形态：
接口或任务草案：
状态模型：
适用前提：
主要风险：
验证方式：
回退方式：
演进条件：
```

表达结构应根据当前方案复杂度裁剪。简单需求可以轻量表达，但不能省略让方案可执行的关键信息。

## 示例组

本示例组采用 `Few-shot Learning` 思路，用经典工程案例校准方案分析的落地颗粒度。

这里不重新定义 `Few-shot Learning`，也不把它作为 `Goal / Capability / Option / Architecture / Constraint / Impact / Tradeoff / Evolution / Recommendation` 的业务节点。它只作为示例校准机制，帮助读者和模型理解方案分析不能只给方向，也不需要替代完整技术设计；它应给出足够具体的能力选择、接口草案、状态模型、验证方式和回退路径。

### 正例：OCR 能力接入

#### 已明确需求

```text
用户上传图片后，系统需要识别图片中的文字。
识别结果用于后续人工校对和结构化录入。
图片可能来自截图、扫描件和手机拍照。
当前后端使用 Python 技术栈。
```

#### 能力调研

| 能力 | 具体选择 | 优点 | 风险 |
| --- | --- | --- | --- |
| Python OCR 包 | `PaddleOCR` | 中文场景支持较好，可本地部署 | 模型体积、部署资源和版本兼容需要评估 |
| 传统 OCR | `Tesseract` | 成熟、轻量、离线可用 | 中文、复杂版式和低质量图片效果可能不足 |
| 云 OCR API | 云厂商 OCR 服务 | 接入快，工程维护成本低 | 成本、限流、数据合规和外部稳定性风险 |
| 多模态大模型 | 图文识别模型 | 对复杂图片和语义理解更强 | 成本、时延、稳定性和结构化一致性风险 |
| 本地模型服务 | 自部署 OCR 模型 | 数据不出内网，可控性高 | 需要模型运维、监控和推理资源 |

#### 候选方案

| 方案 | 核心做法 | 落地形态 | 主要风险 |
| --- | --- | --- | --- |
| 本地 OCR 同步识别 | 后端接收图片后调用 `PaddleOCR` 返回文本块 | `POST /ocr/recognize` | 大图和复杂版式可能耗时较长 |
| 云 OCR 接入 | 后端转发图片到云 OCR API，再统一返回结构 | 外部 API 编排 | 数据合规、成本和限流依赖外部服务 |
| OCR + 大模型修正 | 先 OCR，再用大模型修正或结构化 | 两段式处理链路 | 成本和时延上升，错误归因更复杂 |
| 异步 OCR 任务 | 图片入队识别，完成后回写结果 | 后台任务 + 状态查询 | 用户需要等待，状态流转更复杂 |

#### 推荐方案

如果图片量可控且有数据不出内网要求，短期推荐 “`PaddleOCR` 本地识别 + FastAPI 同步接口 + 超时后可切换异步任务”。

接口草案：

```text
POST /ocr/recognize

Request:
{
  "image_id": "img_123",
  "image_url": "https://example.com/a.png",
  "scene": "document_import"
}

Response:
{
  "status": "succeeded | failed | timeout | low_confidence",
  "provider": "paddleocr",
  "blocks": [
    {
      "text": "发票号码",
      "confidence": 0.96,
      "bbox": [10, 20, 120, 42]
    }
  ],
  "reason": null
}
```

这个方案已经说明了用什么能力、服务接口是什么、返回状态是什么、失败如何表达。后续详细设计可以继续展开图片存储、任务队列、模型部署和字段结构。

#### 取舍判断

只写“接入 OCR”无法支撑研发落地，因为 OCR 本身有多种能力来源。`PaddleOCR` 本地方案更适合数据不出内网、需要控制成本的场景；云 OCR 更适合快速验证但依赖外部服务；多模态大模型适合复杂图文理解，但不适合默认承担所有图片的基础识别。

当前推荐把 OCR 基础识别与后续结构化修正分开：先用稳定 OCR 能力产生可校对文本块，再根据业务复杂度决定是否引入大模型修正。这样能避免一开始把成本、时延和错误归因都压到大模型上。

### 反例：空泛方案

下面这种写法不是合格的方案分析：

```text
推荐后端统一接入 OCR 能力。
复杂图片使用 AI 兜底。
后续可以异步化和平台化。
```

它的问题是：

- 没有说明 OCR 能力具体来自 `PaddleOCR`、`Tesseract`、云 OCR API、本地模型还是多模态大模型。
- 没有说明服务形态是同步接口、异步任务还是已有接口扩展。
- 没有接口草案、输入输出、状态模型和失败处理。
- 没有说明为什么不选其他能力。
- 没有验证方式和回退方式。

这种输出只是方向判断，不足以让研发进入落地。

### 边界例：已经进入详细设计

如果团队已经确定使用 `PaddleOCR`，并开始讨论模型部署目录、图片缓存表、任务表结构、类设计、重试队列和排期拆分，说明问题已经进入详细设计或实施计划。此时方案分析应停止在能力选择、系统边界、接口草案和状态闭环层面，把更细的结构交给技术设计文档。

## 判断标准

方案分析是否完成，只看一个标准：

**读者是否已经知道推荐方案为什么成立、具体用什么能力落地、接口或任务契约大致是什么、状态如何闭环、失败如何处理、如何验证、如何回退。**

如果只写“后端统一编排”“AI 兜底”“接入 OCR”“增加检测能力”，但没有说明用什么包、什么服务、什么模型、什么接口、什么状态和什么验收方式，方案分析没有完成。

如果只罗列多个方案，却没有形成取舍和推荐，方案分析没有完成。

如果给出了推荐方向，但研发仍然不知道下一步该调研、安装、调用、封装或验证什么，方案分析没有完成。

如果文档已经开始展开完整数据库结构、类结构、排期拆分和代码实现细节，说明它进入了详细设计或实施计划阶段。方案分析可以给接口草案、状态模型和关键流程，但不必替代完整技术设计。

## 误用

方案分析不是需求澄清。

如果需求对象、验收口径、边界和状态仍然不清楚，应先回到需求拆解，而不是用方案替需求做决定。

方案分析也不是空泛方向判断。

只说“前端处理”“后端处理”“AI 检测”“异步补偿”“平台化治理”，没有具体能力、落地形态、接口契约和验证方式，不是可执行方案。

方案分析也不是完整技术设计。

它不负责展开全部数据库表、类结构、目录结构、任务排期或代码实现细节。它负责把需求推进到足够可执行的方案层：具体能力、系统边界、接口草案、状态模型、关键流程、验证和回退。

方案分析也不是方案包装。

不要先选定一个主观方案，再补理由证明它正确。方案分析要先调研可用能力，打开可行路径，再根据约束和取舍收敛。

方案分析也不是固定表格。

`Goal / Capability / Option / Architecture / Constraint / Impact / Tradeoff / Evolution / Recommendation` 是方法论节点，不是输出目录。真实需求不同，节点权重可以不同，表达结构也可以不同。

## 适用边界

方案分析与建议适合处理需求已经明确、但实现路径、能力选择或落地形态不唯一的场景。

当团队需要在多个实现路径、第三方服务、开源依赖、本地模型、大模型能力、职责分层、短期和长期方案之间做判断时，应使用这套方法。

如果需求一次性、范围小、技术路径唯一，可以轻量处理 `Capability`、`Evolution` 和 `Tradeoff`，但仍应说明具体做法、接口或交互契约、状态和验收方式。

当需求仍然模糊、验收标准不清、状态语义未定时，这套方法会显得过早；此时应先做需求拆解。

当方案已经被明确选定，并需要进入数据库设计、类设计、任务拆分、开发排期或测试用例设计时，应切换到对应的技术设计或实施方法。
