# 方案分析与建议方法论

> 把已经拆清楚的需求，转成候选方案、评估判断和推荐建议。

```text
方案分析与建议
├── Input：承接需求
│   └── 确认需求目标、范围、约束和待确认点是否足以进入方案阶段
├── Options：生成方案
│   └── 构造多个能回应需求的可行路径
├── Criteria：设定标准
│   └── 明确本次比较方案时真正重要的判断维度
├── Evaluate：评估取舍
│   └── 分析不同方案的收益、成本、风险和适用条件
└── Advice：形成建议
    └── 给出当前最合适的推荐方向、理由和保留条件
```

## 核心命题

方案分析处理的不是“把需求继续写细”，也不是“直接进入技术设计”，而是回答一个中间问题：

```text
在这个需求已经基本清楚的前提下，应该选择什么解决路径。
```

需求拆解关注的是需求是否被说清楚：

```text
问题是什么
影响谁
期望什么行为
边界在哪里
还有什么需要确认
```

方案分析关注的是选择是否有依据：

```text
有哪些可行路径
每条路径解决到什么程度
各自代价和风险是什么
为什么推荐其中一种
推荐结论依赖哪些前提
```

它的核心转化是：

```text
需求说明 -> 候选方案 -> 评估维度 -> 取舍判断 -> 推荐建议
```

这里的“建议”不是最终技术设计，也不是研发任务拆分。它只是让后续方案设计、排期、评审有一个清晰的选择依据。

## 方法论节点

### Input：承接需求

方案分析必须以需求说明为输入。

如果需求对象、范围、期望行为和验收目标还不清楚，方案分析会变成猜测。此时应先回到需求拆解，而不是急着列方案。

进入方案阶段前，至少应确认：

```text
要解决的核心问题是什么
哪些目标必须满足
哪些范围明确不处理
有哪些约束或待确认点会影响方案选择
```

这里的输入不要求需求已经“完全无疑问”。更重要的是判断剩余不确定点的性质：

```text
阻塞需求理解的问题
└── 回到需求拆解阶段

影响方案取舍的问题
└── 带入方案分析阶段，作为约束或风险处理

影响实施细节的问题
└── 进入技术设计或任务拆分时继续确认
```

### Options：生成方案

生成方案不是罗列想法，而是构造不同解决路径。

一个有效的候选方案，应能说明：

```text
它如何回应核心需求
它改变哪些流程、规则或展示
它保留哪些既有逻辑
它不解决哪些相邻问题
```

候选方案之间应有真实差异。常见差异包括：

```text
轻量兜底 vs 完整处理
前端展示修正 vs 后端规则补齐
局部增强 vs 链路重构
自动处理 vs 人工确认
一次性改动 vs 分阶段演进
```

### Criteria：设定标准

方案不能只凭直觉比较。

评估前应先确定本次真正重要的判断维度。维度不宜过多，只保留会影响当前决策的标准。

常见评估维度包括：

```text
需求满足度
用户理解成本
实现复杂度
接入成本
对既有流程的影响
误判或误展示风险
可测试性
可维护性
后续扩展空间
```

### Evaluate：评估取舍

评估不是给方案打分，而是说明取舍。

每个方案都应同时看到收益和代价。一个方案如果只有优点，没有约束，通常说明分析还没有完成。

取舍分析通常回答：

```text
这个方案最大的收益是什么
它付出的主要成本是什么
它引入什么风险
它适合什么前提
它在哪些条件下不应采用
```

如果不同方案适合不同阶段，应明确阶段关系，而不是强行选出唯一答案。

### Advice：形成建议

方案分析最终应形成建议。

建议不是“我喜欢哪个方案”，而是在当前需求、约束、风险和阶段下的推荐方向。

有效建议通常包含：

```text
推荐哪条路径
推荐理由是什么
不推荐哪些路径以及原因
推荐路径的前置条件
需要保留哪些风险和待确认点
后续何时需要重新评估
```

如果信息不足以给出单一建议，应给出分条件建议，而不是制造确定性。

## 判断标准

方案分析是否完成，看四个标准：

```text
1. 读者能看出多个候选方案之间的本质差异
2. 读者能理解评估维度为什么适用于当前需求
3. 读者能判断推荐建议为什么成立
4. 读者知道推荐建议依赖哪些前提、风险和待确认点
```

如果只写一个方案，它更像方案说明，不是方案分析。

如果只列多个方案但没有比较标准，它只是方案清单。

如果只给结论但没有取舍过程，它不是可讨论的建议。

如果已经开始定义接口字段、数据库结构、调用时序和异常码，说明文档进入了技术设计阶段。

## 误用

方案分析不是需求拆解。

当需求本身还不清楚时，不应通过方案分析替代需求补全。

方案分析也不是技术设计。

它可以描述解决路径，但不负责落到接口、字段、模块、类结构、调用链、数据表或具体排期。

方案分析不是评审表格。

表格可以作为表达工具，但方法的核心是建立取舍判断，而不是机械打分。

方案分析也不是为了证明某个既定方案正确。

如果结论已经预设，候选方案和评估维度就会失去讨论价值。

## 适用边界

方案分析与建议适合处理以下场景：

```text
需求已经基本清楚，但实现路径不唯一
需要比较轻量方案和完整方案
需要解释为什么不直接做最大改造
需要给产品、研发、测试一个可讨论的推荐方向
需要在进入技术设计前暴露关键风险
```

它不适合处理以下场景：

```text
原始诉求仍然模糊
需求范围尚未确定
只有一个确定方案且无需取舍
已经进入接口、模块、数据结构和任务拆分
```

当候选方案已经确定且需要做实现细节，应进入技术设计。

当多个成熟方案需要正式决策、打分、投票或评审归档时，可以在本方法之上进一步抽象出独立的方案评审方法。
