# 需求拆解与补充说明方法论

> 把模糊诉求通过理解、质疑、澄清和补充，推进成可实现、可验收的明确需求。

```text
需求拆解与补充说明
├── Object：需求对象
│   └── 判断需求真正要改变什么，而不是照单复述
├── Assumption：隐含假设
│   └── 拆出原话中默认成立、但尚未确认的前提
├── Scenario：场景样例
│   └── 用代表性样例、边界值和反例暴露问题
├── State：状态语义
│   └── 判断系统应表达什么状态，以及哪些状态不能混淆
└── Question：澄清问题
    └── 把不确定、矛盾和风险转成可回答的问题
```

## 核心命题

需求拆解处理的不是“用户没有写完整文档”，而是“真实需求进入工程协作时，经常仍然没有被想清楚”。

真实需求可能模糊，也可能看起来清晰但隐含大量前提。业务方给出的通常是一段压缩表达：里面混有目标、限制、期望、样例、局部方案和默认假设。程序员不能只按字面执行，也不能替业务方擅自补完所有空白。

需求拆解与补充说明的任务，是帮助程序员理解需求、质疑需求、发现缺口、反问业务方，并在多轮澄清后把诉求推进成明确需求。

它的核心转化是：

```text
模糊诉求 -> 初步理解 -> 隐含假设 -> 场景样例 -> 状态语义 -> 澄清问题 -> 明确需求
```

这里的“拆解”，不是把需求拆成开发任务，而是把需求中未说清的对象、前提、状态、边界和风险拆出来。

这里的“补充”，不是替业务方增加新需求，而是把原话中已经明确或明显暗含、但沟通和验收必须知道的信息显性化。

这里的“反问”，不是否定业务诉求，而是拒绝把不确定内容伪装成确定需求。

## 迭代理解

需求拆解与补充说明不是一次性流水线，而是可以多次执行的澄清循环。

每一轮拆解都应产出两类结果：

```text
当前已经能确认的需求理解
下一轮必须向业务方澄清的问题
```

当业务方补充信息后，再重新检查需求对象、隐含假设、场景样例、状态语义和澄清问题。只有当关键问题被回答，需求才能进入方案分析或技术设计。

一个典型循环是：

```text
收到诉求
-> 形成初步理解
-> 拆出隐含假设和场景样例
-> 发现矛盾、缺口和风险
-> 向业务方提问
-> 根据回答修正需求理解
-> 再次检查是否可实现、可验收
```

这个循环不要求每次都写成完整文档。它可以是一段评论、一组澄清问题、一次会议纪要，也可以是正式需求说明。

## 方法论节点

这些节点不是输出目录，而是程序员面对不完整需求时应执行的判断动作。真实需求不同，节点权重可以不同，输出结构也可以不同。

### Object：需求对象

先判断需求真正要改变什么。

用户常常用表层表现描述需求，例如“结果显示绿字”“按钮置灰”“支持 AI 检测”。这些表达可能是样式、规则、状态、流程或局部方案，不一定就是需求对象本身。

对象化要完成的不是“给需求起标题”，而是判断这次变化真正作用在哪个业务对象上。

有效的对象判断通常回答：

```text
用户表面在要求什么
真正被改变的业务对象是什么
这个对象为什么值得被单独处理
哪些相邻对象不应被默认纳入本次需求
```

如果对象没有定住，后续讨论会漂移：业务方说展示，研发可能改流程；业务方说检测，研发可能只改颜色。

### Assumption：隐含假设

拆出原话中默认成立、但尚未被确认的前提。

一句话需求里最容易出问题的，不是已说出的内容，而是没说却被默认成立的内容。程序员需要把这些默认前提显性化，否则实现时就会把猜测写进系统。

隐含假设通常来自：

```text
某个外部能力一定可用
某个判断标准已经存在
某类输入一定有效
某个状态可以复用
某个展示语义不会引发误解
某个异常场景可以忽略
某个统计或导出入口不受影响
```

拆出假设后，不要直接把假设当事实。能从原话推出的，可以作为当前理解；不能从原话推出的，应进入澄清问题。

### Scenario：场景样例

用代表性样例、边界值和反例暴露需求缺口。

需求越短，越需要用场景样例逼出边界。样例不是写作装饰，而是理解需求的工具。它能把抽象条件变成可讨论对象，也能暴露业务方没有说清的特殊情况。

有效的场景样例通常包括：

```text
正常样例
边界样例
异常样例
反例
容易误判的样例
```

当原话出现数字、阈值、状态、颜色、角色、权限、时间、第三方限制时，通常都应该拆样例。

例如，“低于八字符”隐含了 `是`、`OK`、`好的`、七字符文本、八字符边界值、空白、纯标点、变量占位符等样例。没有这些样例，就很难确认哪些内容应处理、哪些内容应排除、哪些内容必须业务方确认。

### State：状态语义

判断系统应表达什么状态，以及哪些状态不能混淆。

很多需求表面是在说动作或样式，实际需要拆的是状态语义。比如“绿字”不是单纯颜色，而是业务状态表达：它可能代表“已检测且正常”，也可能被误解成“默认通过”。

状态语义不只处理 UI 状态，也处理动作、规则、检测或流转完成后，系统应进入的结果语义。

状态语义通常回答：

```text
系统可能有哪些状态
每个状态表达什么业务含义
用户看到这个状态应理解什么
哪些状态不能混用
哪些失败、异常或未知状态不能被伪装成正常
```

当需求涉及展示、检测、审批、校验、同步、发布、导出、统计时，状态语义通常比动作清单更重要。

### Question：澄清问题

把不确定、矛盾和风险转成可回答的问题。

需求拆解的输出不一定是一份完整说明，很多时候应该是一组高质量问题。问题的价值在于推动业务方补充客观条件，而不是让研发在不确定中猜。

澄清问题应优先来自：

```text
原话无法推出的判断
隐含假设中的高风险前提
场景样例暴露出的边界
状态语义中的混淆点
影响验收、统计、导出、权限或既有流程的变化
```

好的问题应能被业务方回答，而不是泛泛追问“这个需求具体是什么”。例如，“八字符是否按去空格后的 Unicode 字符数计算”比“字符怎么算”更容易推动确认。

## 可视化表达

可视化服务理解，不服务装饰。需求拆解中可以使用简单图表，但它们是表达工具，不是固定流程。

常用表达包括：

### 假设表

| 隐含假设 | 风险 | 处理方式 |
| --- | --- | --- |
| AI 检测可以替代第三方检测短文本 | 检测来源不同但展示一致，可能影响可信度 | 询问业务是否接受 |
| 绿字表示检测正常 | 如果 AI 失败也绿字，会误导用户 | 明确绿字只能绑定正常结果 |
| 短对白都属于有效对白 | 空白、纯标点、变量可能不应检测 | 拆样例确认 |

### 场景样例

| 样例 | 为什么要看 |
| --- | --- |
| `是`、`OK`、`好的` | 验证低于阈值的正常短对白 |
| 七字符文本 | 验证低于阈值边界 |
| 八字符文本 | 验证第三方检测边界 |
| 空白 / 标点 | 验证是否属于有效对白 |
| 变量 / 占位符 | 验证是否影响检测规则 |

### 状态矩阵

| 场景 | 状态语义 | 用户可见表现 |
| --- | --- | --- |
| AI 检测正常 | 已检测且正常 | 可以展示为正常状态 |
| AI 检测异常 | 已检测但异常 | 不能展示为正常状态 |
| AI 检测失败 | 未获得可信结果 | 不能展示为正常状态 |
| 达到第三方门槛 | 走既有检测逻辑 | 不因本需求改变 |

只有当图表能帮助读者理解假设、样例、状态或问题时，才需要使用。不要为了“看起来完整”而把需求写成表格堆叠。

## 输出形态

需求拆解与补充说明最终应推动需求变明确。它可以产出一段回复、一组澄清问题、一段 issue 补充、一段 PRD 草稿，也可以是正式文档中的一个章节。

推荐输出结构是：

```text
原始需求
当前理解
隐含假设
场景样例
状态语义
澄清问题
已确认后的需求说明
```

这只是推荐结构，不是固定模板。

输出结构应根据当前不确定性裁剪；不要为了完整而强行填满每一栏。

当需求仍然模糊时，优先输出当前理解和澄清问题；当业务方已经回答关键问题后，再整理成需求说明；当需求准备进入研发和测试时，再补验收口径。

## 示例组

本示例组采用 `Few-shot Learning` 思路，用经典少样本案例校准 `Object / Assumption / Scenario / State / Question` 的使用方式。

这里不重新定义 `Few-shot Learning`，也不把它作为需求拆解的业务节点。它只作为示例校准机制，帮助读者和模型避免只记住抽象节点而不会拆真实需求。示例不是模板，不要求每次照抄；它们分别说明正确拆解、错误用法和适用边界。

### 正例：导出权限需求

#### 原始需求

```text
希望导出按钮在没有权限时置灰，避免用户点击后报错。
```

#### 当前理解

这条需求表面是“按钮置灰”，但真正对象不是按钮样式，而是用户在导出动作前能否获得清晰、稳定的权限状态。

因此，不能把“没有权限”直接等同为“置灰”这一种表现。需求需要先确认权限判断来源、按钮状态语义、无权限提示方式，以及是否影响导出接口本身。

#### 隐含假设

| 隐含假设 | 为什么需要确认 |
| --- | --- |
| 前端可以提前知道用户是否有导出权限 | 如果权限只能由后端在导出时判断，单纯置灰可能不可靠 |
| 置灰表示用户当前不可导出 | 如果置灰原因不明确，用户不知道是无权限、无数据还是系统异常 |
| 无权限用户不应触发导出接口 | 如果后端仍允许调用，前端置灰只是体验优化，不是权限控制 |
| 权限状态不会随筛选条件变化 | 如果不同数据范围权限不同，按钮状态需要随上下文变化 |

#### 场景样例

| 样例类型 | 示例 | 需要暴露的问题 |
| --- | --- | --- |
| 有权限且有数据 | 用户有导出权限，列表有可导出数据 | 按钮是否可点击 |
| 无权限 | 用户没有导出权限 | 按钮置灰还是隐藏，是否展示原因 |
| 权限未知 | 权限接口加载中或失败 | 按钮应加载中、置灰还是允许点击 |
| 无数据 | 用户有权限但当前筛选结果为空 | 是否与无权限使用同一种置灰状态 |
| 后端拒绝 | 前端判断有权限，但后端导出返回 403 | 是否需要提示、刷新权限或记录异常 |

#### 状态语义

| 场景 | 状态语义 | 期望表现 |
| --- | --- | --- |
| 有权限且满足导出条件 | 用户当前可以执行导出 | 按钮可点击 |
| 无权限 | 用户不能执行导出 | 按钮置灰或隐藏，并说明原因 |
| 权限加载中 | 系统尚未获得可信权限状态 | 按钮展示加载或暂不可点击 |
| 权限判断失败 | 系统无法确认是否可导出 | 不能伪装成有权限，应提示稍后重试 |
| 有权限但无数据 | 动作条件不满足，不是权限问题 | 应与无权限状态区分 |

#### 澄清问题

- 权限判断来源是什么：前端已有权限配置、用户角色、后端权限接口，还是导出接口返回结果。
- 无权限时按钮应该置灰、隐藏，还是可点击后提示申请权限。
- 置灰原因是否需要区分无权限、无数据、权限加载中和系统异常。
- 后端导出接口是否仍需要做权限校验，前端置灰是否只是体验层防误触。
- 权限状态是否受当前筛选条件、数据范围、组织空间或资源归属影响。
- 权限接口失败时，页面应展示什么状态。

### 反例：只润色原话

下面这种写法看起来像需求说明，但没有完成需求拆解：

```text
需求：没有导出权限时按钮置灰。
展示：无权限用户不能点击导出按钮。
范围：导出功能入口。
验收：无权限用户看到置灰按钮。
```

它的问题是：

- 没有判断真正对象，仍然把需求停留在“按钮置灰”或“结果显示”。
- 没有拆出权限来源、状态可信度、后端兜底校验等隐含假设。
- 没有用有权限、无权限、权限加载失败、无数据等样例暴露边界。
- 没有区分可执行、不可执行、加载中、异常和条件不满足等状态。
- 没有形成可回答的澄清问题，研发仍然只能猜。

这种输出只是整理原话，不是需求拆解。

### 边界例：不应继续拆解的场景

如果业务已经提供完整 PRD，明确了对象、边界、状态、验收标准、异常处理和影响范围，继续按“一句话需求”拆解会显得过轻。此时更适合做需求评审、风险检查或验收标准补强。

如果讨论已经进入“使用哪个接口、哪个依赖包、什么数据结构、什么调用链路”，说明问题已经进入方案分析或技术设计阶段。此时不应继续用需求拆解替代方案判断。

## 判断标准

需求拆解是否完成，只看一个标准：

**读者是否已经能说清当前需求对象、隐含假设、代表性样例、状态语义和必须业务确认的问题，并知道下一步是继续澄清还是进入方案分析。**

如果文档已经开始定义接口、字段、算法、调用链、数据结构，说明它进入了解决方案阶段。

如果文档只是把原话整理成对象、范围、验收、待确认几栏，但没有提出假设、反例和澄清问题，说明它仍然是机械整理，没有完成需求拆解。

如果文档能推动业务方回答关键问题，并在回答后逐步形成可实现、可验收的需求，拆解已经成立。

## 误用

需求拆解不是解决方案设计。

它不负责决定接口字段、数据库结构、模型选择、调用链路、缓存策略或实现优先级。

需求拆解也不是简单润色。

润色只是让原话更顺；需求拆解要把原话中的对象、隐含假设、场景样例、状态语义和澄清问题拆出来。

需求拆解也不是固定流程。

`Object / Assumption / Scenario / State / Question` 是方法论节点，不是输出目录。真实需求不同，节点权重可以不同，输出结构也可以不同。

需求拆解也不是待办清单。

待办清单告诉团队要做哪些动作；需求拆解要先说明这些动作为什么成立、依赖哪些假设、有哪些反例、应该向业务方确认什么。

## 适用边界

需求拆解与补充说明适合处理用户只给出一句话、几段话、标题加描述、会议口头记录或粗略 issue 的场景。

当需求里出现阈值、颜色、状态、异常、角色、权限、时间、边界值或第三方限制时，尤其需要拆出隐含假设、场景样例和状态语义。

当需求已经进入技术方案、接口设计、排期拆分或实现评审阶段，应换用对应的方案设计或任务拆分方法。

如果用户已经提供了完整 PRD，继续使用这套方法可能显得过轻；此时更适合做需求评审、风险检查或验收标准补强。
