# 需求拆解与补充说明方法论

> 把一句话或一段话需求，补充成对象清楚、边界可讨论、结果可验收的需求说明。

```text
需求拆解与补充说明
├── Intent：诉求识别
│   └── 判断用户真正想改变什么，而不是只复述原话
├── Context：语境补全
│   └── 补出当前状态、触发条件、影响对象和问题后果
├── Scope：范围切分
│   └── 区分本次需求要覆盖什么、不覆盖什么
├── Behavior：期望行为
│   └── 描述系统应该表现成什么样，以及什么结果算满足需求
└── Confirm：待确认点
    └── 标出仍需用户或业务方确认的关键问题
```

## 核心命题

需求拆解处理的不是“用户没有写完整文档”，而是“用户只给了一个压缩过的业务表达”。

真实需求经常以一句话、几段话、标题加描述、会议记录或粗略 issue 出现。它们通常已经包含问题、限制、期望和隐含边界，但这些信息混在一起，没有形成稳定结构。

需求拆解与补充说明的任务，是把短表达转成可沟通、可确认、可验收的需求说明，而不是直接进入解决方案设计。

它的核心转化是：

```text
压缩表达 -> 需求对象 -> 行为预期 -> 验收口径 -> 确认清单
```

这里的“补充”，不是替业务方增加新需求，而是把原话中已经明确或明显暗含、但沟通和验收必须知道的信息显性化。

这里的“拆解”，不是把需求拆成开发任务，而是把对象、条件、行为、边界和不确定点拆开。

## 节点必要性

需求拆解与补充说明不是流水线，也不是固定写作模板。它有稳定的判断节点，但真实输出可以按需求复杂度压缩、合并或换序。

主节点分为两类：

| 节点 | 必要性 | 作用 |
| --- | --- | --- |
| `Intent` | 必要 | 防止只复述原话，先判断需求真正想解决什么问题 |
| `Context` | 必要 | 补出理解需求所需的现状、触发条件、影响对象和问题后果 |
| `Scope` | 必要 | 防止把短需求扩写成无边界改造，明确覆盖和不覆盖范围 |
| `Behavior` | 必要 | 把诉求翻译成用户可感知、测试可验证的系统表现 |
| `Confirm` | 必要 | 把不能擅自决定的内容集中暴露，进入下一轮确认 |

可选输出组件包括：

| 组件 | 何时使用 | 说明 |
| --- | --- | --- |
| 原始需求 | 通常建议保留 | 保留输入，方便对照补充内容是否偏离原意 |
| 需求补充说明 | 原话过短或上下文压缩时使用 | 用自然段把隐含语境展开，但不替业务方做强结论 |
| 拆解表 | 对象、条件、行为较多时使用 | 帮助读者快速扫描结构，不是必须格式 |
| 验收口径 | 需求需要进入实现、测试或排期前使用 | 可单列，也可并入 `Behavior` |
| 示例表达 | 方法教学、模板沉淀或需求容易误解时使用 | 用例子帮助理解，不代表每次输出都要写示例 |

判断一个节点是否需要显式写出来，不看模板是否完整，而看缺少它后需求是否仍然可理解、可讨论、可验收。

## 方法论节点

这些节点不是栏目要求，而是判断一个短需求是否已经被拆清楚的抽象动作。

### Intent：诉求识别

先判断用户真正想解决什么问题。

用户的原话常常会把问题、原因、期望和局部实现想法混在一起。需求拆解不能只改写原句，而要识别背后的主诉求。

有效的诉求识别通常回答：

```text
用户想优化什么
当前哪里不符合预期
这个需求针对的是展示、流程、规则、数据，还是体验理解
```

### Context：语境补全

补出需求发生的业务语境。

短需求通常默认了很多背景：当前系统如何工作、哪里有限制、谁受影响、在什么条件下触发、问题出现后造成什么困扰。

语境补全要把这些隐含信息显性化：

```text
当前状态是什么
触发条件是什么
影响对象是谁
问题出现后造成什么困扰
```

语境补全不是编造需求。原话明确表达或明显暗含的信息可以补出来；原话没有给出的判断，应进入待确认点。

### Scope：范围切分

确认本次需求覆盖什么、不覆盖什么。

短需求最容易被过度扩写成完整系统改造。范围切分的目的，是防止需求拆解滑向解决方案或无边界扩张。

范围切分通常回答：

```text
本次只处理哪类对象
哪些相邻问题暂不处理
哪些既有逻辑不应被默认改变
```

### Behavior：期望行为

描述系统应该让用户看到什么变化。

需求说明应表达期望行为，而不是提前指定实现路径。尤其当原始诉求混合了技术手段、展示样式或局部实现想法时，要先把它翻译成用户可感知的业务行为。

有效的期望行为通常包含：

```text
满足什么条件时
系统应该表现为什么状态
用户应该能理解什么结果
什么结果可以算作需求满足
哪些情况不能被误展示或误处理
```

验收口径不一定要单独成章，但需求进入实现或测试前，至少要能从 `Behavior` 中推出可验证结果。

### Confirm：待确认点

标出拆解后仍不能确定的部分。

需求拆解不是把所有问题都替用户决定，而是把不确定点集中暴露出来，方便下一轮沟通。

待确认点通常来自：

```text
计数口径
边界场景
展示位置
状态命名
是否需要人工确认
是否影响既有流程
```

待确认点不是文档缺陷。相反，能把不能擅自决定的内容标出来，说明需求拆解没有越界。

## 输出形态

需求拆解与补充说明最终应产出可沟通的需求说明。它可以是一段回复、一段 issue 补充、一段 PRD 草稿，也可以是正式文档中的一个章节，不要求每次都新建独立文档。

推荐的轻量输出结构是：

```text
原始需求
补充说明
拆解结果
验收口径
待确认点
```

这只是推荐结构，不是固定流水线。

当需求很简单时，可以合并成几段自然语言；当需求容易误解时，可以补一张拆解表；当需求要交给研发和测试时，应明确验收口径；当方法论用于教学或模板沉淀时，才需要完整示例。

## 示例

下面的示例展示如何处理“一段话需求”。示例用于说明方法，不代表每次输出都要照抄这个版式。

### 原始需求

```text
4.3 多语言翻译检测工具展示表现优化
第三方检测限制了八字符的限制，导致低于此字符的对白内容都不会进行检测
期望低于八字符的对白检测支持ai检测，检测正常则按照绿字显示出来
```

### 补充说明

多语言翻译检测工具目前依赖第三方检测能力，但第三方检测存在八字符门槛，导致低于八字符的对白无法进入原检测流程。用户在查看检测结果时，可能无法确认这类短对白是否已经被检测处理。

本次需求希望补齐短对白的检测和展示表现：当对白内容低于第三方检测支持的八字符门槛时，系统应通过 AI 检测补充判断；如果 AI 检测结果正常，则该对白应按照“检测正常”的视觉状态展示为绿字。

这里的绿字不应表示“因为字符过短所以默认通过”，而应表示“短对白已经完成 AI 检测，且检测结果正常”。

### 拆解结果

| 拆解项 | 说明 |
| --- | --- |
| 需求对象 | 多语言翻译检测工具中的短对白检测结果 |
| 触发条件 | 对白内容低于第三方检测支持的八字符门槛 |
| 当前问题 | 第三方检测不处理八字符以下内容，导致短对白没有正常检测结果 |
| 期望行为 | 八字符以下对白进入 AI 检测 |
| 正常展示 | AI 检测正常时，按绿字展示 |
| 异常边界 | AI 检测异常、失败或无结果时，不能误展示为绿字 |
| 不默认改变 | 八字符及以上对白仍沿用既有第三方检测逻辑 |

### 验收口径

- 输入低于八字符的对白时，系统不会仅因第三方检测限制而直接跳过检测。
- 低于八字符的对白可以进入 AI 检测，并产生可用于展示的检测结果。
- AI 检测结果正常时，该对白在检测结果中按绿色正常状态展示。
- AI 检测异常、失败或无结果时，该对白不会被展示为绿色正常状态。
- 八字符及以上对白的既有检测和展示逻辑不因本需求改变。

### 待确认点

- “八字符”的计数口径是什么，是否包含空格、标点、变量占位符和特殊符号。
- “低于八字符”是否包含空字符串、纯标点、纯数字或纯空白内容。
- AI 检测正常的判定标准是什么，是否需要覆盖漏译、错译、语言不一致、格式符和占位符问题。
- AI 检测失败、超时或无结果时，前端应展示为未检测、检测失败，还是保留原状态。
- 绿字展示只影响检测结果列表，还是也影响详情页、导出结果和统计汇总。

## 判断标准

需求拆解是否完成，只看一个标准：

**读者是否已经能清楚知道这个需求要解决什么、影响什么、期望系统表现什么、可以如何验收、还有哪些点需要确认。**

如果文档已经开始定义接口、字段、算法、调用链、数据结构，说明它进入了解决方案阶段。

如果文档能让产品、设计、研发、测试对需求对象和行为预期形成一致理解，但仍保留必要的待确认点，拆解已经成立。

更具体地说，完成后的需求说明应能经得起以下检查：

```text
是否识别了真实诉求
是否补出了必要背景
是否说明了影响对象
是否写清了触发条件
是否表达了期望行为
是否说明了不覆盖什么
是否能推出验收口径
是否把不确定内容列为待确认点
是否避免进入技术方案
```

## 误用

需求拆解不是解决方案设计。

它不负责决定接口字段、数据库结构、模型选择、调用链路、缓存策略或实现优先级。

需求拆解也不是简单润色。

润色只是让原话更顺；需求拆解要把原话中的对象、问题、范围、期望和待确认点拆出来。

需求拆解也不是固定流程。

`Intent / Context / Scope / Behavior / Confirm` 是方法论节点，不是输出目录。真实需求不同，节点权重可以不同，输出结构也可以不同。

## 适用边界

需求拆解与补充说明适合处理用户只给出一句话、几段话、标题加描述、会议口头记录或粗略 issue 的场景。

当需求已经进入技术方案、接口设计、排期拆分或实现评审阶段，应换用对应的方案设计或任务拆分方法。

如果用户已经提供了完整 PRD，继续使用这套方法可能显得过轻；此时更适合做需求评审、风险检查或验收标准补强。
