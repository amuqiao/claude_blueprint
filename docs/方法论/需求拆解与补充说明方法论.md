# 需求拆解与补充说明方法论

> 把一句话需求，从模糊诉求拆成可讨论、可确认、可验收的需求说明。

```text
需求拆解与补充说明
├── Intent：诉求识别
│   └── 判断用户真正想改变什么，而不是只复述原话
├── Context：语境补全
│   └── 补出当前状态、触发条件、影响对象和问题后果
├── Scope：范围切分
│   └── 区分本次需求要覆盖什么、不覆盖什么
├── Behavior：期望行为
│   └── 描述用户期望系统表现成什么样
└── Confirm：待确认点
    └── 标出仍需用户或业务方确认的关键问题
```

## 核心命题

需求拆解处理的不是“用户没有写完整文档”，而是“用户只给了一个很短的业务表达”。

真实需求经常以一句话、几段话、一个标题加一句期望出现。它们通常包含问题、限制、期望和隐含边界，但没有把这些信息拆开。

需求拆解的任务，是把短表达转成可沟通的需求说明，而不是直接给解决方案。

它的核心转化是：

```text
简短诉求 -> 需求对象 -> 行为预期 -> 确认清单
```

这里的“需求说明”，不是技术设计，也不是实现方案，而是让产品、设计、研发、测试能对同一个需求形成一致理解。

## 方法论节点

需求拆解与补充说明不是固定模板，但它有稳定的分析节点。

这些节点不是栏目要求，而是判断一个短需求是否已经被拆清楚的抽象动作。

### Intent：诉求识别

先判断用户真正想解决什么问题。

用户的原话常常会把问题、原因和期望混在一起。需求拆解不能只改写原句，而要识别它背后的主诉求。

有效的诉求识别通常回答：

```text
用户想优化什么
当前哪里不符合预期
这个需求针对的是展示、流程、规则、数据，还是体验理解
```

### Context：语境补全

补出需求发生的业务语境。

短需求通常默认了很多背景：当前系统如何工作、哪里有限制、谁受影响、在什么条件下触发、产生什么后果。

语境补全要把这些隐含信息显性化：

```text
当前状态是什么
触发条件是什么
影响对象是谁
问题出现后造成什么困扰
```

语境补全不是编造需求，而是把原话中已经暗含的信息整理出来；不确定的部分应进入待确认点。

### Scope：范围切分

确认本次需求覆盖什么、不覆盖什么。

短需求最容易被过度扩写成完整系统改造。范围切分的目的，是防止需求拆解滑向解决方案或无边界扩张。

范围切分通常回答：

```text
本次只处理哪类对象
哪些相邻问题暂不处理
哪些既有逻辑不应被默认改变
```

### Behavior：期望行为

描述系统应该让用户看到什么变化。

需求说明应表达期望行为，而不是提前指定实现路径。尤其当原始诉求混合了技术手段、展示样式或局部实现想法时，要先把它翻译成用户可感知的业务行为。

有效的期望行为通常包含：

```text
满足什么条件时
系统应该表现为什么状态
用户应该能理解什么结果
哪些情况不能被误展示
```

### Confirm：待确认点

标出拆解后仍不能确定的部分。

需求拆解不是把所有问题都替用户决定，而是把不确定点集中暴露出来，方便下一轮沟通。

待确认点通常来自：

```text
计数口径
边界场景
展示位置
状态命名
是否需要人工确认
是否影响既有流程
```

## 判断标准

需求拆解是否完成，只看一个标准：

**读者是否已经能清楚知道这个需求要解决什么、影响什么、期望系统表现什么、还有哪些点需要确认。**

如果文档已经开始定义接口、字段、算法、调用链、数据结构，说明它进入了解决方案阶段。

如果文档能让产品、设计、研发、测试对需求对象和行为预期形成一致理解，但仍保留必要的待确认点，拆解已经成立。

## 误用

需求拆解不是解决方案设计。

它不负责决定接口字段、数据库结构、模型选择、调用链路、缓存策略或实现优先级。

需求拆解也不是简单润色。

润色只是让原话更顺；需求拆解要把原话中的对象、问题、范围、期望和待确认点拆出来。

`Intent / Context / Scope / Behavior / Confirm` 是分析节点，不是输出目录。真实文档可以按业务需要组织，但不应跳过这些判断动作。

## 适用边界

需求拆解与补充说明适合处理用户只给出一句话、几段话、标题加描述、会议口头记录或粗略 issue 的场景。

当需求已经进入技术方案、接口设计、排期拆分或实现评审阶段，应换用对应的方案设计或任务拆分方法。

如果用户已经提供了完整 PRD，继续使用这套方法可能显得过轻；此时更适合做需求评审、风险检查或验收标准补强。
