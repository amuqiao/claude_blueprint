# 用户心智模型

> 本文档回答：为什么这套 blueprint 不能只针对 Claude/Codex 设计，而必须同时区分“人类层”和“Claude 层”。

---

## 先给结论

这套 blueprint 不是单纯的提示词仓库，也不是只给 Claude/Codex 用的运行时配置集合。

它是一个**人和 Claude 协作开发项目的系统**。

所以它天然需要两层：

- **人类层**：帮助你判断现在处于哪个阶段、该调用哪种能力、如何组织开发流程
- **Claude 层**：让 Claude 在运行时按规范执行

如果只做 Claude 层，你会得到一套“能运行”的配置；  
但如果没有人类层，你很难长期稳定地驾驭这套系统。

---

## 为什么不能只设计 Claude 层

Claude 可以帮你：

- 写代码
- 写文档
- 做分析
- 做审查
- 执行固定工作流

但 Claude 并不能替你稳定承担下面这些上位判断：

- 现在是该先定盘，还是该先建模块
- 这次需求要不要先设计
- 是该直接 `/new-module`，还是先 `@arch`
- 是该更新架构文档，还是只更新模块设计文档
- 这条新约束该沉淀到哪里

这些都属于：

**人类如何使用 Claude 的判断。**

如果这层缺失，就会出现：

- 命令越来越多
- skill 越来越多
- hooks 越来越多
- 但你自己越来越不知道什么时候该用哪个

所以这套系统不能只针对 Claude/Codex 设计。

---

## 两层分别解决什么问题

| 层级 | 主要对象 | 解决什么问题 | 典型文件 |
|------|---------|-------------|---------|
| 人类层 | 你 | 先做什么、后做什么、如何使用这套系统 | `README.md`、`PLAYBOOK.md`、`docs/能力地图.md`、`docs/工作流参考.md` |
| Claude 层 | Claude / Codex | 运行时该遵守什么规则、该如何执行工作流 | `CLAUDE.md`、`commands/`、`skills/`、`agents/`、`hooks/`、`templates/`、`settings.json` |

一句话理解：

- 人类层解决：**现在该做什么**
- Claude 层解决：**这件事具体怎么做**

---

## 这几份文档为什么不是重复

### README

回答：
- 这是什么
- 怎么部署
- 怎么开始用
- 这套系统应该怎么整体理解

它是入口，不是完整方法论文档。

### PLAYBOOK

回答：
- 开发范式是什么
- 项目定盘、模块设计、实现落地、回写抽象怎么切换
- 先做什么，后做什么

它是方法层，不是运行时配置。

### 能力地图

回答：
- `commands / agents / skills / hooks / templates` 的区别是什么
- 什么时候该用哪种能力

它是能力分工说明，不是范式文档。

### 工作流参考

回答：
- 一轮真实开发应该怎么走
- 不同场景下，哪条路径最合适

它是从方法层到执行层之间的桥。

所以它们的关系不是重复，而是：

- `README` 负责入口
- `PLAYBOOK` 负责方法
- `能力地图` 负责能力边界
- `工作流参考` 负责流程串联

---

## 你在 Claude TUI 中应该怎么用这套系统

推荐顺序是：

### 第一步：先判断自己处在哪个阶段

问自己：

- 我是在项目定盘？
- 模块设计？
- 代码实现？
- 还是回写与抽象？

这一步看 `PLAYBOOK.md`。

### 第二步：再判断该调用什么能力

问自己：

- 我现在需要阶段入口吗？→ `command`
- 我现在需要判断吗？→ `agent`
- 我现在需要按规范完成某类任务吗？→ `skill`
- 我现在需要自动保护吗？→ `hook`
- 我现在需要一个稳定骨架吗？→ `template`

这一步看 `docs/能力地图.md`。

### 第三步：让 Claude 按运行时规范执行

真正执行时，Claude 会读：

- `CLAUDE.md`
- `commands/*.md`
- `skills/*/SKILL.md`
- `agents/*.md`
- `hooks/*.sh`
- `templates/*.md`

这一步是 Claude 层工作。

---

## 什么时候要警惕设计过头

双层设计是合理的，但不能无限膨胀。

需要警惕的信号有：

- README 开始变成完整手册
- PLAYBOOK 开始重复命令细节
- 能力地图开始讨论项目方法论
- 工作流参考开始变成设计决策文档

只要守住边界：

- README：入口
- PLAYBOOK：方法
- 能力地图：能力边界
- 工作流参考：流程桥接
- Claude 层文件：运行时执行

这套结构就不会过度设计。

---

## 一句话总结

这套 blueprint 不是只让 Claude 更“听话”，而是让你能长期、稳定地**驾驭 Claude 开发项目**。

所以保留“人类层 + Claude 层”的双层设计，不是多余，而是这套系统成立的前提。
