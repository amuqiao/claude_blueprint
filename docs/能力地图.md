# 能力地图

> 本文档回答：在 Claude TUI 里，`commands`、`agents`、`skills`、`hooks`、`templates` 分别是什么，什么时候用哪个。

---

## 总览

| 类型 | 作用 | 触发方式 | 适合什么 |
|------|------|---------|---------|
| `commands` | 进入一个明确的工作流阶段 | 手动输入 `/命令名` | 初始化、建模块、同步文档、归档规则 |
| `agents` | 做带角色的分析和审查 | 手动输入 `@agent` | 架构判断、代码审查 |
| `skills` | 规定某类任务怎么做，或把任务路由到正确方法论 | 通常按任务语义触发，也可用明确表达引导 | 设计文档、写作、脚本、运维类任务，或复杂项目方法分流 |
| `hooks` | 做强制拦截或提醒 | 自动触发 | 阻止危险动作、提醒关键文件变更 |
| `templates` | 提供稳定起点 | 通常不手动触发，由 command 使用 | 项目初始化、架构文档、实施清单 |

一句话判断：

- 想“进入一个阶段” → 用 `command`
- 想“做一个判断” → 用 `agent`
- 想“按某类规范完成任务” → 用 `skill`
- 想“强制执行某种保护” → 用 `hook`
- 想“拿到稳定骨架” → 用 `template`

---

## commands

### 作用

`commands` 是显式工作流入口。  
它负责把你带进某个阶段，但不负责定义整套方法论。

### 当前典型命令

- `/init-architecture`
- `/new-module`
- `/update-docs`
- `/distill-draft`
- `/add-rule`
- `/smoke-test`

### 最小示例

```text
/init-architecture
/new-module 用户认证
/update-docs 用户认证
/distill-draft drafts/docs/某篇草稿.md
```

### 什么时候用

- 你已经明确“现在要进入哪个阶段”
- 你不想靠自由对话临时组织流程

---

## agents

### 作用

`agents` 是带角色的分析器。  
它们负责给判断、给审查结果，不负责默默启动整条工作流。

### 当前角色

- `@arch`：架构判断
- `@rev`：规范审查

### 最小示例

```text
@arch 帮我判断这个模块是否会影响全局架构
@rev 检查 app/services/user_service.py 是否违反分层约束
```

### 什么时候用

- 需要“先判断，再决定”
- 需要“先审查，再修正”

---

## skills

### 作用

`skills` 负责“这类任务怎么做”。  
它更像任务规范，而不是阶段入口。

补充：
- 对复杂项目场景，也可以存在**薄调度 skill**
- 它不复制完整方法论，只负责判断当前层次并路由到正确方法论文档

### 当前典型领域

- 设计文档
- 运维维护
- Python 脚本
- 技术写作
- 复杂项目方法论分流

### 最小示例

```text
帮我写一份新功能的设计文档
帮我整理一个运维诊断 CLI 脚本
```

### 什么时候用

- 你已经知道要做什么
- 但希望 Claude 按某类稳定规范来做
- 或者你还不知道当前问题属于哪一层，希望先做方法论路由

---

## hooks

### 作用

`hooks` 用来处理必须强制执行、不能只靠软提示词约束的行为。

### 当前行为

- 保护关键配置
- 阻止 `git push`
- 提醒关键文件变更

### 最小示例

- 让 Claude 修改 `CLAUDE.md` → 二次确认
- 让 Claude 执行 `git push` → 被拒绝

### 什么时候用

- 某个动作必须 100% 触发
- 出错代价高，不能靠模型“记得遵守”

---

## templates

### 作用

`templates` 提供稳定起点。  
它本身不负责工作流逻辑，通常由 command 间接使用。

### 当前模板

- 项目 `CLAUDE.md`
- `docs/design/INDEX.md`
- `docs/design/架构设计方案.md`
- `docs/实施清单.md`

### 最小示例

- `/init-architecture` 会使用项目模板、架构模板、实施清单模板
- `/new-module` 会创建模块设计文档和页面 mock

### 什么时候用

- 你需要一个标准骨架
- 但不希望每次从空白文档开始

## 当前没有默认纳入的能力

本 blueprint 当前没有把 `rules/` 和 `plugins/` 纳入默认主结构。

当前仓库有一个例外：
- 已保留最小 `rules/writing.md` 示例，用于验证 rules 机制可用
- 这不是“默认依赖 rules 分层”的意思，而是一次最小启用

- `rules`：适合按路径、按子域拆项目规则时再引入
- `plugins`：适合需要扩展新能力时再引入
