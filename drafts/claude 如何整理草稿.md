## 复述你的需求

你当前的痛点是：

**草稿管理缺少便捷的整理机制**

具体来说：

1. **草稿来源多样**：
   - 零散想法
   - 一段思维链
   - 未成熟的判断
   - 对话中的灵感

2. **草稿有两条升格路径**（MAINTAINING.md 已定义）：
   - `drafts/` → `drafts/incubating/`（重点草稿）
   - `drafts/incubating/` → 正式文档（PLAYBOOK/WHY/MAINTAINING/README）

3. **但缺少执行机制**：
   - 没有固定的整理流程
   - 不知道什么时候该整理
   - 整理时没有引导性的 prompt
   - 没有辅助判断"这篇草稿应该去哪里"的工具

你想要的是：
- **一个便捷的入口**（command/skill/工作流）
- **能引用具体草稿**
- **辅助完成**: 阅读 → 抽象 → 判断去向 → 迁移/升格

---

## 分析：这是一个标准的"阶段入口"需求

对照你的 PLAYBOOK.md 中的组件职责分工：

| 组件    | 职责                           | 是否适合草稿整理 |
| ------- | ------------------------------ | ---------------- |
| command | 阶段入口和显式工作流           | ✓ **最适合**     |
| skill   | 某一类任务的思维方式和写法规范 | 可作为辅助       |
| agent   | 带角色的分析和判断             | 不适合           |
| hook    | 强制和提醒                     | 不适合           |

**结论**：应该创建 `/refine-draft` command，配合在 MAINTAINING.md 中补充具体的整理 prompt 模板。

# 草稿整理工作流

你负责帮用户整理 `drafts/` 目录下的草稿文件。

## 执行步骤

### 1. 读取草稿

如果用户指定了具体文件路径（如 `/refine-draft drafts/某想法.md`）：
- 直接读取该文件

如果用户没有指定路径：
- 列出 `drafts/*.md` 文件
- 让用户选择要整理的草稿

### 2. 分析草稿

读取后，按以下维度分析：

**内容类型**：
- 零散想法 / 思维链 / 方法论探索 / 设计决策草案 / 踩坑记录

**成熟度**：
- 仅记录：只是记下来，还没形成观点
- 初步判断：有观点但还不稳定
- 接近定稿：已经形成稳定判断，可以升格

**影响范围**：
- 仅当前项目
- 跨项目通用
- 影响开发范式
- 影响维护流程

### 3. 判断去向

根据分析结果，给出升格建议：

| 成熟度 | 影响范围 | 建议去向 |
|--------|---------|---------|
| 仅记录 | 任意 | 保留在 `drafts/` |
| 初步判断 | 会反复影响设计判断 | 升格到 `drafts/incubating/` |
| 接近定稿 | 方法层 | 升格到 `PLAYBOOK.md` |
| 接近定稿 | 设计决策 | 升格到 `WHY.md` |
| 接近定稿 | 维护流程 | 升格到 `MAINTAINING.md` |
| 接近定稿 | 使用方式 | 升格到 `README.md` |
| 接近定稿 | 项目专属 | 如果在具体项目中，升格到项目文档 |

### 4. 输出整理建议

格式：

```markdown
## 草稿分析

**文件**: drafts/某想法.md

**核心观点**:
{用 1-3 句话总结核心观点}

**内容类型**: {零散想法 / 思维链 / ...}

**成熟度**: {仅记录 / 初步判断 / 接近定稿}

**影响范围**: {仅当前项目 / 跨项目通用 / ...}

## 升格建议

**建议去向**: {drafts/incubating/ / PLAYBOOK.md / WHY.md / ...}

**理由**: 
{为什么应该去这里}

**如何整理**:
{具体的整理建议，例如：}
- 提炼成 3 条原则
- 补充"为什么"和"什么时候用"
- 整合到目标文档的第 X 节
```

### 5. 辅助迁移（可选）

如果用户确认升格建议，询问是否需要：
- 直接整合到目标文档
- 还是只输出整理后的内容让用户自己复制

## 注意事项

- 不要擅自删除草稿原文件
- 升格到正式文档后，询问用户是否要在草稿中标注"已升格"
- 如果草稿中有多个独立观点，建议拆分成多篇
```
