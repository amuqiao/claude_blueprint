# FastAPI 多环境配置管理方案

> **文档职责**：梳理 FastAPI 项目在本地直接运行、docker-compose、K8s 三种部署方式下的配置加载逻辑与文件规划
> **适用场景**：需要在本地开发、开发服务器、生产环境间切换，且维护者分离（开发者 vs 运维）
> **目标读者**：开发者（需要在本地和开发服务器间切换）、运维（需要在 K8s 部署）
> **维护规范**：配置文件规划变更时更新 §3；环境切换命令变更时更新 §4

---

## 一、问题背景

FastAPI 项目实际部署在三种环境，运行方式不同：

| 环境 | 运行方式 | 维护者 | 配置来源 |
|---|---|---|---|
| 本地开发 | 直接运行 uvicorn | 开发者 | `.env` 文件 |
| 开发服务器 | docker-compose | 开发者 | `.env.dev` 文件 |
| 生产 | K8s | 运维 | K8s ConfigMap / Secret |

核心问题：**不同运行方式如何加载配置？如何用一套代码适配三种环境？**

---

## 二、思维链：逐步厘清问题

### 第一步：识别三种环境的配置加载差异

**本地直接运行（uvicorn）：**
```
Python 应用启动
  → pydantic-settings 默认读取 .env 文件
  → 加载配置，启动服务
```

**开发服务器（docker-compose）：**
```
docker-compose 读取 env_file 指定的文件
  → 注入为容器环境变量
  → 容器内 Python 应用启动
  → pydantic-settings 从环境变量读配置（优先级高于文件）
```

**生产（K8s）：**
```
运维在 K8s 侧配置 ConfigMap / Secret
  → K8s 将配置作为环境变量注入容器
  → 容器内 Python 应用启动
  → pydantic-settings 从环境变量读配置
  → 不读任何 .env 文件
```

关键认知：
- 本地直接运行时，只有 Python 应用层在读文件
- docker-compose 时，有两层：docker-compose 先注入环境变量，Python 应用再读（但环境变量优先）
- K8s 时，只有环境变量，文件机制完全不参与

### 第二步：对比两种环境切换方案

**方案 A：APP_ENV 方案（推荐）**

```yaml
# docker-compose.yml
env_file:
  - .env.${APP_ENV:-.env}
```

```bash
# 本地直接运行（默认读 .env）
uvicorn main:app --reload

# 开发服务器
APP_ENV=dev docker-compose up -d
```

文件规划：
- `.env` - 不提交，本地真实配置
- `.env.dev` - 提交（脱敏），开发服务器配置
- `.env.example` - 提交，配置模板

**方案 B：ENV_FILE 方案**

```yaml
# docker-compose.yml
env_file:
  - ${ENV_FILE:-.env}
```

```bash
# 本地直接运行
uvicorn main:app --reload  # 仍然读 .env

# 开发服务器
ENV_FILE=.env.dev docker-compose up -d
```

文件规划相同。

### 第三步：选择 APP_ENV 方案的原因

| 对比维度 | APP_ENV 方案 | ENV_FILE 方案 |
|---|---|---|
| 本地直接运行 | pydantic-settings 默认读 `.env`，零配置 ✅ | 需要额外处理环境变量 ❌ |
| 变量语义 | 表达环境含义（dev/prod） ✅ | 只是文件路径 ❌ |
| 本地配置安全 | `.env` 不提交，天然保护密码 ✅ | 需要额外 gitignore ❌ |
| 新人理解成本 | `APP_ENV=dev` 一看就懂 ✅ | `ENV_FILE=.env.dev` 需要理解映射关系 ❌ |

**一句话理由：** 本地直接运行时 pydantic-settings 天然读 `.env`，不提交正好保护本地真实密码；开发服务器用 `APP_ENV=dev` 切换，语义清晰，符合"用环境变量表达环境"的直觉。

### 第四步：理解运维侧的配置流程

运维不会把 `.env.prod` 放进 Git，他们的流程是：

```
1. 拉取仓库（只需代码 + Dockerfile）
2. 阅读 .env.example，了解需要哪些配置项
3. 在 K8s 侧创建 ConfigMap / Secret，填入生产真实值
4. K8s 启动容器时，将这些值作为环境变量注入
5. pydantic-settings 从环境变量读配置，不读任何文件
```

给运维的不是配置文件，而是**配置项清单**（`.env.example`）。

### 第五步：划清三种环境的边界

| | 本地直接运行 | 开发服务器 | 生产 K8s |
|---|---|---|---|
| 配置来源 | `.env` 文件 | `.env.dev` 文件 | K8s ConfigMap / Secret |
| 启动命令 | `uvicorn main:app` | `APP_ENV=dev docker-compose up` | 运维在 K8s 配置 |
| `.env.*` 文件作用 | 有用（本地读） | 有用（docker-compose 读） | 无关（只读环境变量） |

## 三、最终方案

### 3.1 配置文件规划

| 文件 | 是否提交 Git | 用途 | 读取者 |
|---|---|---|---|
| `.env` | ❌ 不提交 | 本地真实配置（包含真实密码） | 本地直接运行时读取 |
| `.env.dev` | ✅ 提交（脱敏） | 开发服务器配置 | docker-compose 读取并注入 |
| `.env.example` | ✅ 提交 | 配置模板，列出所有配置项和注释 | 运维参考，创建 K8s 配置 |
| `.env.prod` | ❌ 不提交 | （无需此文件，K8s 直接注入环境变量） | 不使用 |

`.gitignore` 追加：

```gitignore
.env
.env.prod
.env.*.override
```

### 3.2 docker-compose.yml 配置

```yaml
services:
  app:
    env_file:
      - .env.${APP_ENV:-.env}
    # 其他配置...
```

逻辑：
- 未设置 `APP_ENV` 时，默认读 `.env`（兼容本地 docker-compose 启动）
- 设置 `APP_ENV=dev` 时，读 `.env.dev`

### 3.3 .env.example 写法规范

```bash
# ==================== 应用配置 ====================
APP_ENV=prod
API_PORT=8000
FLOWER_PORT=5555

# ==================== 数据库 ====================
DATABASE_URL=postgresql+asyncpg://user:password@host:5432/dbname

# ==================== Redis ====================
REDIS_URL=redis://:password@host:6379/0

# ==================== Celery ====================
CELERY_BROKER_URL=redis://:password@host:6379/1
CELERY_RESULT_BACKEND=redis://:password@host:6379/2

# ==================== 安全 ====================
SECRET_KEY=your-secret-key-here
```

## 四、不同环境的维护方式和工作流

### 4.1 本地开发环境

**初始化配置：**

```bash
# 1. 复制模板
cp .env.example .env

# 2. 填入本地真实配置（数据库密码、Redis 密码等）
vim .env
```

**日常启动：**

```bash
# 方式 1：直接运行（推荐，方便调试）
uvicorn main:app --reload

# 方式 2：用 docker-compose（如果需要完整容器环境）
docker-compose up -d
```

**维护规则：**
- `.env` 包含真实密码，不提交 Git
- 新增配置项时，同步更新 `.env.example`

### 4.2 开发服务器环境

**初始化配置：**

```bash
# 1. 在仓库创建 .env.dev（脱敏版本）
cp .env.example .env.dev

# 2. 填入开发服务器的配置（连接开发服务器的 DB / Redis）
vim .env.dev

# 3. 提交到 Git
git add .env.dev
git commit -m "新增开发服务器配置"
```

**日常启动：**

```bash
# SSH 到开发服务器后
APP_ENV=dev docker-compose up -d
```

**维护规则：**
- `.env.dev` 提交 Git，但必须脱敏（不能包含真实生产密码）
- 配置变更后，重启服务：`APP_ENV=dev docker-compose restart`

### 4.3 生产环境（K8s）

**运维侧初始化流程：**

```bash
# 1. 拉取仓库
git clone <repo-url>

# 2. 阅读配置模板
cat .env.example

# 3. 在 K8s 创建 ConfigMap（非敏感配置）
kubectl create configmap app-config \
  --from-literal=APP_ENV=prod \
  --from-literal=API_PORT=8000 \
  --from-literal=FLOWER_PORT=5555

# 4. 在 K8s 创建 Secret（敏感配置）
kubectl create secret generic app-secret \
  --from-literal=DATABASE_URL=postgresql+asyncpg://user:password@prod-db:5432/prod \
  --from-literal=REDIS_URL=redis://:prod-password@prod-redis:6379/0 \
  --from-literal=SECRET_KEY=prod-secret-key-here

# 5. 部署时引用这些配置
kubectl apply -f deployment.yaml
```

**deployment.yaml 示例：**

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: fastapi-app
spec:
  template:
    spec:
      containers:
      - name: app
        image: your-image:latest
        envFrom:
        - configMapRef:
            name: app-config
        - secretRef:
            name: app-secret
```

**维护规则：**
- 配置全部在 K8s 侧管理，不在仓库里
- 新增配置项时，开发者更新 `.env.example`，运维同步更新 K8s ConfigMap / Secret

### 4.4 配置变更协作流程

**开发者新增配置项：**

1. 在 `.env.example` 添加新配置项（带注释）
2. 更新 `.env.dev`（脱敏值）
3. 提交 PR，告知运维需要同步配置
4. 在 PR 描述中写明：
   ```
   新增配置项：
   - NEW_CONFIG_ITEM：[说明用途]
   
   运维需要在 K8s 侧添加该环境变量。
   ```

**运维同步配置：**

```bash
# 更新 ConfigMap / Secret
kubectl edit configmap app-config
kubectl edit secret app-secret

# 滚动重启（让新配置生效）
kubectl rollout restart deployment/fastapi-app
```

### 4.5 交给运维的材料

提供两样东西：

1. **GitLab 仓库地址**（含 Dockerfile）
2. **配置说明**：

> 配置模板在仓库根目录 `.env.example`，按模板在 K8s 侧创建 ConfigMap（非敏感配置）和 Secret（敏感配置），以环境变量方式注入容器。应用启动时 pydantic-settings 会自动从环境变量读取配置，无需提供任何 `.env` 文件。

## 五、全局视图

### 三种环境的配置流程

```
┌─────────────────────────────────────────────────────────────┐
│ 本地开发（直接运行）                                          │
├─────────────────────────────────────────────────────────────┤
│ uvicorn main:app --reload                                   │
│   ↓                                                         │
│ pydantic-settings 读取 .env 文件                             │
│   ↓                                                         │
│ 加载配置，启动服务                                           │
│                                                             │
│ 配置文件：.env（不提交，包含真实密码）                        │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│ 开发服务器（docker-compose）                                  │
├─────────────────────────────────────────────────────────────┤
│ APP_ENV=dev docker-compose up -d                            │
│   ↓                                                         │
│ docker-compose 读取 .env.dev 文件                            │
│   ↓                                                         │
│ 将配置注入为容器环境变量                                      │
│   ↓                                                         │
│ 容器内 pydantic-settings 从环境变量读配置                     │
│   ↓                                                         │
│ 启动服务                                                     │
│                                                             │
│ 配置文件：.env.dev（提交 Git，脱敏）                          │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│ 生产环境（K8s）                                               │
├─────────────────────────────────────────────────────────────┤
│ 运维在 K8s 侧配置 ConfigMap / Secret                         │
│   ↓                                                         │
│ K8s 将配置作为环境变量注入容器                                │
│   ↓                                                         │
│ 容器内 pydantic-settings 从环境变量读配置                     │
│   ↓                                                         │
│ 启动服务                                                     │
│                                                             │
│ 配置文件：不使用任何 .env 文件                                │
│ 配置来源：K8s ConfigMap / Secret                             │
└─────────────────────────────────────────────────────────────┘
```

### 边界划分

| 维度 | 本地开发 | 开发服务器 | 生产 K8s |
|---|---|---|---|
| 配置文件 | `.env` | `.env.dev` | 不使用 |
| 提交 Git | ❌ | ✅（脱敏） | ❌ |
| 维护者 | 开发者本地 | 开发者（仓库） | 运维（K8s 侧） |
| 切换方式 | 无需切换（默认读 `.env`） | `APP_ENV=dev` | 运维配置 K8s |
| 密码安全 | 本地文件，不提交 | 脱敏值，可提交 | K8s Secret 管理 |

### 核心原则

- **本地零配置**：直接运行时默认读 `.env`，不需要设置任何环境变量
- **开发服务器一个变量切换**：`APP_ENV=dev` 即可
- **生产环境与仓库解耦**：运维在 K8s 侧管理配置，不依赖仓库里的 `.env` 文件
- **新人快速上手**：看 `.env.example` 即可了解所有配置项
