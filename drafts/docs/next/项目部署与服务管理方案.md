# 项目部署与服务管理方案

> **文档职责**：定义当前项目支持的部署方式，以及各方式下的服务边界与管理方式。
> **适用场景**：开发者或维护者需要判断当前项目如何启动、管理和交付服务时。
> **目标读者**：开发者、维护者。
> **维护规范**：部署方式边界、配置入口或服务管理方式变更时更新本文件。

---

## 一、当前主线约束

当前部署与服务管理统一基于这组约束：

- 配置模板维护在 `.env.example`
- `.env` 是默认统一入口
- `.env`、`.env.dev` 都不提交 Git
- 本地和开发环境可通过环境变量在 `.env`、`.env.dev` 间切换
- 环境变量优先级高于本地文件
- compose 只维护一个 `docker-compose.yml`

不再使用：

- `docker-compose.prod.yml`
- `.env.prod`
- `.env.prod.example`

---

## 二、支持的 3 种部署方式

当前项目统一按以下 3 种方式理解和维护：

1. 本地开发
2. `docker-compose` 全量部署
3. 基于 `Dockerfile` 的独立部署

这三种方式都允许存在，但不能混写成一套流程。

---

## 三、方式一：本地开发

### 3.1 适用场景

- 本地调试代码
- 快速开发和联调
- 希望应用直接在宿主机运行

### 3.2 服务边界

- 应用服务在本机直接运行
- PostgreSQL、Redis 这类依赖服务可由 `docker-compose` 提供

### 3.3 配置要求

- 默认使用 `.env`
- 如果要切换到 `.env.dev`，统一通过环境变量指定
- 环境变量优先于本地文件

### 3.4 启动方式

```bash
# 应用本机运行
uvicorn main:app --reload
```

如果本地需要用 compose 提供依赖服务，则仍然使用同一个 `docker-compose.yml`。

### 3.5 服务管理

- 应用服务由本机进程管理
- 依赖服务由 `docker-compose` 管理

---

## 四、方式二：docker-compose 全量部署

### 4.1 适用场景

- 需要在本地或开发环境模拟完整运行环境
- 希望应用服务和依赖服务统一由 compose 管理

### 4.2 服务边界

- 应用服务和依赖服务都通过 `docker-compose.yml` 启动
- PostgreSQL、Redis 在 compose 网络内提供服务

### 4.3 配置要求

- 默认使用 `.env`
- 需要时通过环境变量切换到 `.env.dev`
- 统一通过 `ENV_FILE` 切换

### 4.4 启动方式

```bash
# 默认读取 .env
docker-compose up -d

# 显式切换到 .env.dev
ENV_FILE=.env.dev docker-compose up -d
```

### 4.5 服务管理

```bash
docker-compose ps
docker-compose logs -f
docker-compose restart
```

切换到 `.env.dev` 时：

```bash
ENV_FILE=.env.dev docker-compose ps
ENV_FILE=.env.dev docker-compose logs -f
ENV_FILE=.env.dev docker-compose restart
```

---

## 五、方式三：基于 Dockerfile 的独立部署

### 5.1 适用场景

- 需要把应用镜像交给运维或平台单独运行
- 需要验证 `Dockerfile` 能否独立支撑应用启动

### 5.2 服务边界

- `Dockerfile` 负责构建应用镜像
- 应用服务独立运行
- PostgreSQL、Redis 由外部提供

### 5.3 配置要求

- 仍以 `.env` 作为默认统一入口
- 如需切换，可通过环境变量指定 `.env.dev`
- 外部依赖地址应填写真实 IP、域名或服务地址
- 环境变量优先于本地文件

### 5.4 启动方式

典型形态是分别启动应用服务容器，由外部数据库和 Redis 提供依赖。

### 5.5 服务管理

- 应用服务由运行平台或容器管理方式负责
- 外部 PostgreSQL、Redis 不由当前应用镜像负责

---

## 六、配置与部署的统一口径

当前项目需要统一遵守下面这组约束：

- 维护 `.env.example` 作为配置模板
- 默认统一入口是 `.env`
- `.env`、`.env.dev` 都不提交
- 通过环境变量在 `.env`、`.env.dev` 之间切换
- 只维护一个 `docker-compose.yml`

---

## 七、如何选择

- 只想本地开发调试：用“本地开发”
- 想跑完整容器环境：用“docker-compose 全量部署”
- 想交付镜像给运维或平台：用“基于 Dockerfile 的独立部署”

---

## 八、结论

当前主线不是维护多套 compose / env 文件，而是：

- 一个模板：`.env.example`
- 一个默认入口：`.env`
- 一个可选环境文件：`.env.dev`
- 一个 compose 文件：`docker-compose.yml`
- 三种部署理解：本地开发、compose 全量、Dockerfile 独立部署

配置管理和部署方式应统一口径，但文档职责分开维护。
