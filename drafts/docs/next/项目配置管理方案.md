# FastAPI 项目配置管理方案

> **文档职责**：定义当前项目的主线配置管理方式。
> **适用场景**：开发者需要了解 `.env`、`.env.dev`、`docker-compose.yml` 的职责和切换方式时。
> **目标读者**：开发者、维护者。
> **维护规范**：配置文件职责或切换方式变更时更新本文件。

---

## 一、当前主线

当前项目统一使用这一套配置模型：

- 本地开发使用 `.env`
- 开发服务器使用 `.env.dev`
- `docker-compose` 只使用一个 `docker-compose.yml`
- 通过 `ENV_FILE` 切换 `docker-compose` 读取的环境文件

不再使用：

- `docker-compose.prod.yml`
- `.env.prod`
- `.env.prod.example`

---

## 二、文件职责

| 文件 | 是否提交 Git | 用途 |
|---|---|---|
| `.env` | ❌ 不提交 | 本地开发真实配置 |
| `.env.dev` | ✅ 提交（脱敏） | 开发服务器配置 |
| `docker-compose.yml` | ✅ 提交 | 唯一的 compose 编排文件 |

---

## 三、切换方式

### 3.1 本地直接运行

本地直接运行应用时，默认读取 `.env`。

```bash
uvicorn main:app --reload
```

### 3.2 docker-compose 启动

`docker-compose` 统一通过 `ENV_FILE` 切换环境文件。

```bash
# 默认使用 .env
docker-compose up -d

# 明确使用 .env.dev
ENV_FILE=.env.dev docker-compose up -d
```

对应的 `docker-compose.yml` 应统一保持这一类写法：

```yaml
services:
  app:
    env_file:
      - ${ENV_FILE:-.env}
```

逻辑说明：

- 未设置 `ENV_FILE` 时，默认读取 `.env`
- 设置 `ENV_FILE=.env.dev` 时，读取 `.env.dev`

---

## 四、使用规则

### 4.1 `.env`

- 只用于本地开发
- 包含本地真实配置
- 不提交 Git

### 4.2 `.env.dev`

- 用于开发服务器或共享开发环境
- 必须脱敏
- 可以提交 Git

### 4.3 `docker-compose.yml`

- 是当前唯一维护的 compose 文件
- 不再拆出 `docker-compose.prod.yml`
- 所有 compose 相关说明都应以这个文件为准

---

## 五、协作约束

- 新增配置项时，同时更新本地 `.env` 和开发环境 `.env.dev`
- 不要再引入 `.env.prod`、`.env.prod.example`
- 不要再新增第二个 compose 文件
- 文档、提示词、脚本都应统一使用这套口径

---

## 六、常见问题

### Q1：为什么不再使用 `docker-compose.prod.yml`？

因为当前方案已经收敛为单一模型：

- 一个 `docker-compose.yml`
- 两个环境文件：`.env`、`.env.dev`
- 一个切换入口：`ENV_FILE`

继续保留 `docker-compose.prod.yml` 只会制造第二套部署认知。

### Q2：为什么不再使用 `.env.prod`、`.env.prod.example`？

因为它们不属于当前主线配置体系，也没有对应的 compose 主线入口。

保留它们只会让人误以为仓库还维护另一套生产环境文件方案。
