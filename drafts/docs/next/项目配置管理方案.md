# FastAPI 项目配置管理方案

> **文档职责**：定义当前项目的主线配置管理方式。
> **适用场景**：开发者需要了解 `.env.example`、`.env`、`.env.dev`、`docker-compose.yml` 的职责和切换方式时。
> **目标读者**：开发者、维护者。
> **维护规范**：配置文件职责、优先级或切换方式变更时更新本文件。

---

## 一、当前主线

当前项目统一使用这一套配置模型：

- 仓库维护 `.env.example` 作为配置模板
- `.env` 作为默认统一入口
- 本地和开发环境都可以在 `.env`、`.env.dev` 之间灵活切换
- 统一通过环境变量指定当前要使用的环境文件
- 环境变量优先级高于本地文件
- `docker-compose` 只维护一个 `docker-compose.yml`

不再使用：

- `docker-compose.prod.yml`
- `.env.prod`
- `.env.prod.example`

---

## 二、文件职责

| 文件 | 是否提交 Git | 用途 |
|---|---|---|
| `.env.example` | ✅ 提交 | 配置模板，列出所有配置项 |
| `.env` | ❌ 不提交 | 默认本地环境文件，也是统一入口 |
| `.env.dev` | ❌ 不提交 | 可选的开发环境文件 |
| `docker-compose.yml` | ✅ 提交 | 唯一维护的 compose 编排文件 |

---

## 三、优先级与切换原则

### 3.1 配置优先级

统一原则：

1. 环境变量优先
2. 本地环境文件次之
3. 默认值最后

这意味着：

- 如果某个配置已经通过环境变量传入，就不应再被 `.env` 或 `.env.dev` 覆盖
- 本地文件主要用于开发期提供默认配置

### 3.2 默认入口

默认情况下使用 `.env` 作为统一入口。

### 3.3 切换方式

当需要切换到 `.env.dev` 时，统一通过环境变量指定。

例如：

```bash
# 默认使用 .env
docker-compose up -d

# 切换到 .env.dev
ENV_FILE=.env.dev docker-compose up -d
```

如果应用本机直接运行，也应遵循同样的切换思路：默认读 `.env`，需要时通过环境变量切换到 `.env.dev`。

## 四、docker-compose 约束

`docker-compose` 统一只维护一个 `docker-compose.yml`。

推荐保持这一类写法：

```yaml
services:
  app:
    env_file:
      - ${ENV_FILE:-.env}
```

逻辑说明：

- 未设置 `ENV_FILE` 时，默认读取 `.env`
- 设置 `ENV_FILE=.env.dev` 时，读取 `.env.dev`

---

## 五、使用规则

### 5.1 `.env.example`

- 必须维护
- 作为所有配置项的模板来源
- 新增配置项时，优先更新它

### 5.2 `.env`

- 作为默认统一入口
- 本地默认使用它
- 不提交 Git

### 5.3 `.env.dev`

- 作为可选开发环境文件
- 只在需要切换时使用
- 不提交 Git

### 5.4 `docker-compose.yml`

- 是当前唯一维护的 compose 文件
- 不再拆出 `docker-compose.prod.yml`

---

## 六、协作约束

- 只维护 `.env.example`、`.env`、`.env.dev`、`docker-compose.yml`
- `.env` 和 `.env.dev` 都不提交 Git
- 新增配置项时，先更新 `.env.example`
- 文档、提示词、脚本都应统一使用这套口径

---

## 七、常见问题

### Q1：为什么 `.env.example` 必须保留？

因为仓库里需要有一份可维护的配置模板，方便开发者初始化环境，也方便后续核对配置项。

### Q2：为什么 `.env` 是统一入口？

因为默认入口必须只有一个，才能减少理解成本。`.env` 作为默认入口最直观，切换到 `.env.dev` 时再通过环境变量显式指定即可。

### Q3：为什么 `.env` 和 `.env.dev` 都不提交？

因为这两个文件都属于真实环境文件，应该只在本地或具体环境中使用，不应作为仓库主线的一部分。
