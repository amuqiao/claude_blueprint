# skill 为什么存在，以及应该承接什么

> **文档职责**：解释为什么 `CLAUDE.md` 会臃肿、为什么需要把部分内容提取成 skill，以及提取后的 skill 应该承接什么内容。
> **适用场景**：重构 `CLAUDE.md`、新增 skill、判断一条规范该放在 `CLAUDE.md` 还是 `skills/` 时使用。
> **目标读者**：本仓库维护者；已经接受“全局入口 + skills 下沉”的结构，但还没有稳定形成判断标准。
> **维护规范**：如果后续对 `CLAUDE.md` 与 skill 的分工理解发生变化，先更新本文档，再调整对应 skill。

---

## 1. 真正的问题不是“怎么写 skill”

表面上看，问题像是：

- skill 应该怎么写
- skill 里该放哪些章节
- 要不要写 why / how / what

但更本质的问题是：

**`CLAUDE.md` 承载不了所有规范。**

当一套规范既重要、又频繁复用、还带有明确工作方式时，如果继续堆在 `CLAUDE.md` 里，就会出现三个问题：

1. 全局入口越来越长，主干不清
2. 不同任务类型的细节互相污染
3. Claude 每次都要加载一大段并不总是相关的内容

所以 skill 的出现，本质上不是为了“分类更好看”，而是为了：

**把已经稳定、会反复出现、又不适合继续堆在 `CLAUDE.md` 里的任务规范下沉出去。**

---

## 2. 什么内容应该留在 `CLAUDE.md`

`CLAUDE.md` 适合留下的是：

- 跨项目都成立的总原则
- 各类能力的入口指针
- 最少量、最稳定的全局约束

它更像：

**总入口、总开关、总边界。**

典型例子：

- 分层架构约束
- 通用工作流约束
- skill 指针
- 最小的输出规则入口

一句话：

**`CLAUDE.md` 负责告诉 Claude“现在先看哪类东西”，而不是把所有细节都自己讲完。**

---

## 3. 什么内容应该提取成 skill

当一类内容同时满足下面几个条件时，就值得提取成 skill：

1. **它不是一句话能讲清的**
2. **它代表一种稳定的任务工作方式**
3. **它会在不同项目里反复出现**
4. **如果不单独下沉，`CLAUDE.md` 会越来越臃肿**

典型适合提成 skill 的内容：

- 设计文档怎么写
- 运维 CLI 怎么组织
- 批量脚本怎么设计
- FastAPI 后端项目默认怎么起盘
- 文档写作工作流怎么做

这些内容的共同点是：

**它们不是单条规则，而是一整类任务的方法。**

---

## 4. 提取 skill 时，真正要承接的是什么

skill 不是知识库条目，也不是概念说明书。  
它承接的应该是：

- 这类任务的稳定工作流
- 这类任务的判断顺序
- 这类任务的输出要求
- 这类任务最容易犯的错

所以一个合格的 skill，通常至少要能回答：

1. 这类任务什么时候适用
2. 这类任务应该按什么顺序推进
3. 最终至少要产出什么
4. 最低合格标准是什么

换句话说：

**skill 承接的不是“知识点”，而是“任务方法”。**

---

## 5. 为什么很多 skill 会写坏

最常见的坏法有 3 种：

### 5.1 把 skill 写成说明文

只解释理念，不给步骤。  
结果是：

- 看起来有道理
- 真要执行时不知道从哪里开始

### 5.2 把 skill 写成碎片清单

一堆要求、一堆 checklist，但没有主干。  
结果是：

- 条目很多
- 但不知道先做什么、后做什么

### 5.3 把 skill 写成另一个 `CLAUDE.md`

把总原则、全局边界、任务细节全塞进去。  
结果是：

- skill 本身又膨胀
- 和 `CLAUDE.md` 重复
- 以后继续漂移

---

## 6. 一个更实用的理解：Why / How / What

如果还想用 `why / how / what` 来理解，也可以，但它不该是死模板，而应该服务于上面那个问题。

### `Why`
不是为了写理念，而是回答：

**为什么这类内容必须从 `CLAUDE.md` 里下沉出来。**

### `How`
不是泛泛讲原则，而是回答：

**这类任务到底应该怎么推进。**

### `What`
不是罗列所有细节，而是回答：

**最后至少要交付什么，什么算合格。**

所以更准确地说：

- `Why` 解释下沉理由
- `How` 承接任务方法
- `What` 收住输出标准

---

## 7. 一个简单判断法

以后遇到一条规范时，可以先问：

### Q1：这是总原则，还是任务方法？

- 总原则 → 更可能留在 `CLAUDE.md`
- 任务方法 → 更可能提成 skill

### Q2：这条内容会不会继续膨胀？

- 不会 → 直接放 `CLAUDE.md`
- 会 → 提前下沉 skill

### Q3：这条内容是不是一类任务的稳定做法？

- 是 → 适合 skill
- 不是 → 可能只是普通规则、rule 或项目文档内容

---

## 8. 一句话收口

skill 的存在，不是为了把规范“分类存放”，而是为了：

**把已经稳定的一类任务方法，从臃肿的 `CLAUDE.md` 中提取出来，让全局入口保持轻，让任务方法保持清。**

所以写 skill 时，最先想的不是“我该写几个章节”，而是：

**这是不是一类已经稳定到值得单独承接的方法。**
