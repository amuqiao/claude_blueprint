我理解你的意思。

你的需求不是“再写一份普通文档”，而是：

**在没有真实业务需求的前提下，给这套 `~/.claude` 模具做一套可重复的验收机制。**

你担心的是这几类东西部署后“看起来在，实际没生效”：

- `skills/` 是否能被正确触发
- `agents/` 是否真的按预期角色工作
- `commands/` 是否能正常调用
- `hooks/` 是否真的拦截/提醒
- `templates/` 是否可用
- `deploy-manifest.txt` + `deploy-to-claude.sh` 是否把该部署的都部署了

而且你已经意识到一个关键问题：

**单纯写“验收文档”不够，因为没有真实需求时，很多机制不会自然触发。**  
所以你在想：是不是应该专门设计一组“验收用示例”或“验收用 skill/agent/command”，部署到 `~/.claude` 后，再进入 Claude TUI 里按步骤验证。

这个理解是对的。

## 分析

这里要先分清两类东西：

### 1. 可确定验收的
这些可以做成**稳定、可重复、低歧义**的测试。

- 文件是否部署到 `~/.claude`
- `commands/` 是否可见、可执行
- `agents/` 是否能被调用、输出格式是否符合预期
- `hooks/` 是否在关键动作时触发
- `templates/` 是否存在、内容是否完整

### 2. 不可完全确定验收的
这些最多只能做“高概率验证”，不能做成绝对测试。

- 某个 skill 是否会被 Claude **自动**触发
- 某个自然语言提示是否一定命中某个 skill
- 某个 agent 是否会被主对话**自动**选中

原因很简单：  
这类行为本质上带有模型判断成分，不是纯规则引擎。

所以你的验收设计不能只靠“自动触发”，否则会变成：

- 失败时不知道是没部署成功
- 还是部署成功了，但模型这次没选中

## 结论

**应该新增一套“验收包”，但不要只做文档。**  
应该是：

1. 一份验收说明文档
2. 一组验收用示例
3. 少量“专门用于验收的机制”
4. 明确区分“确定性测试”和“启发式测试”

## 我建议的方案

### A. 新增一个 `acceptance/` 目录
放在仓库根目录，**不部署到 `~/.claude`**。

例如：

```text
acceptance/
├── ACCEPTANCE.md          # 验收手册
├── prompts.md             # 在 Claude TUI 里逐条输入的测试提示词
├── expected.md            # 每条测试的期望结果
└── checklist.md           # 最终打勾清单
```

作用：
- 仓库维护者在本项目根目录阅读
- 部署完成后切到 Claude TUI 按步骤验收

### B. 新增一个专门的“验收 command”
这是最值得做的，**比新增验收 skill 更好**。

比如：

```text
commands/smoke-test.md
```

原因：
- `command` 是显式触发，确定性高
- `skill` 自动触发不稳定，不适合做验收基线

这个 `/smoke-test` 可以做的事：
- 读取 `~/.claude/CLAUDE.md`
- 检查关键目录是否存在
- 输出当前安装版本/结构摘要
- 提示下一步手工测试 hooks / agents / skills

也就是说，它不是替代全部验收，而是做“入口”和“环境确认”。

### C. skills 不要靠“自动触发”验收，改成“双层验收”
对 `skill` 我建议这样验：

#### 第一层：部署验收
确认对应文件存在：
- `skills/personal-os/references/design-doc.md`
- `~/.claude/skills/...`

这证明“已部署”。

#### 第二层：行为验收
用一条**高度定向的提示词**测试：
- 看 Claude 输出是否明显符合该 skill 的格式特征

比如对 `design-doc`：
- 输入：`我想写一份新功能的设计文档`
- 预期：输出里出现 `问题定义 / 目标与非目标 / 方案设计 / 数据模型 / 接口规范`

这不是 100% 规则式验证，但足够做 smoke test。

### D. agents 最好新增一个“验收 agent”或给现有 agent 加明确验收入口
这个要看你想不想污染正式结构。

我更建议：

- **不新增长期保留的“测试 agent”**
- 而是在 `ACCEPTANCE.md` 里给现有 `arch` / `rev` 写专门测试题

例如：

#### `arch` 验收题
输入一个明确的架构问题，预期输出必须包含：
1. 决策表
2. 推荐理由
3. 风险提示
4. 文档影响

#### `rev` 验收题
给一个故意违规的小样例代码，预期输出：
- `文件路径:行号 → 问题描述 → 修复建议`

这样更干净，不会为了测试污染正式 agent 集。

### E. hooks 用“手工触发脚本场景”验收
hooks 是最容易做成确定性验收的。

建议在 `ACCEPTANCE.md` 里写成固定步骤：

#### `protect-config.sh`
- 尝试修改 `~/.claude/CLAUDE.md`
- 预期：出现二次确认

#### `notify-file-changed.sh`
- 修改 `~/.claude/skills/...`
- 预期：出现提醒信息

#### `block-git-push.sh`
- 在 Claude 环境里尝试执行 `git push`
- 预期：被 deny

这部分不需要新增 skill/agent，直接用场景验收就够了。

### F. templates 只验“存在 + 可复制”
template 不需要复杂化。

验收项：
- `~/.claude/templates/project-CLAUDE.md` 存在
- `~/.claude/templates/docs-INDEX.md` 存在
- 手动 `cp` 一次没有报错

## 最关键的建议

### 不建议：新增“特别的验收 skill”
因为你想解决的是“是否生效”，但 `skill` 的核心问题恰恰是**自动触发不稳定**。  
用它去验证“验收机制”本身，会把不确定性再放大一层。

### 建议：新增“验收 command + 验收文档 + 验收样例”
这是最稳的组合：

- `acceptance/ACCEPTANCE.md`
- `commands/smoke-test.md`
- `acceptance/fixtures/` 里放一些故意用于测试的示例内容

这样：
- command 提供确定性入口
- fixtures 提供标准输入
- doc 提供人工验收步骤
- skill/agent/hook 各自用最合适的方式验

## 我建议的最终结构

```text
acceptance/
├── ACCEPTANCE.md
├── prompts.md
├── expected.md
└── fixtures/
    ├── rev-bad-sample.py
    ├── arch-question.md
    └── design-doc-request.md

commands/
└── smoke-test.md
```

## 最后判断

你的方向是对的，但要修正成：

- **不是“新增一个验收 skill”**
- 而是“新增一套验收机制”

最优解是：

1. `acceptance/` 文档与样例
2. 一个显式的 `/smoke-test` command
3. 对 `skills/agents/hooks/templates` 分别定义验收方式
4. 把“自动触发”只当辅助验证，不当唯一验收标准
