# 文档体系与职责边界

> **文档职责**：收口“各类文档分别做什么、放哪里、为什么不该混写”的判断，避免文档职责继续在普通草稿中分散。
> **当前状态**：重点草稿。已经形成稳定边界，但仍适合作为方法层草稿继续观察。
> **后续可能去向**：`README.md`、`PLAYBOOK.md`、`MAINTAINING.md`

---

## 1. 这篇草稿要解决什么问题

当文档、规则、模板、记忆位置逐渐增多时，最容易出现的不是“没文档”，而是：

- 不知道某段内容该写到哪里
- 同一个问题被写进多个文档
- 结构约定和运行时规则混在一起
- 草稿、正式文档、模板、README 的边界越来越模糊

所以这篇草稿真正要解决的是：

- `README / PLAYBOOK / WHY / MAINTAINING / docs / drafts` 各自负责什么
- 项目记忆位置为什么这样约定
- 哪些属于结构约定，哪些不该写成 rule
- 如何避免文档重复、漂移和失控

---

## 2. 当前的核心判断

### 2.1 文档体系必须分层

文档失控的根因，通常不是数量，而是职责混写。

一旦不同问题域混进同一份文档，就会出现：

- 用户不知道哪个才是准的
- 改了一处，另一处忘记同步
- README 开始像手册
- PLAYBOOK 开始像命令说明
- MAINTAINING 开始像方法论文档

所以真正重要的，不是“多写文档”，而是：

**每类文档只回答一类问题。**

### 2.2 不同文档承担不同问题域

当前最稳的文档层级模型是：

- `README.md`
  - 入口层
  - 回答“这是什么、怎么用、怎么开始”

- `PLAYBOOK.md`
  - 方法层
  - 回答“先做什么、后做什么、阶段怎么切”

- `WHY.md`
  - 决策层
  - 回答“为什么这样设计，而不是另一种设计”

- `MAINTAINING.md`
  - 维护层
  - 回答“以后怎么改、改哪里、改完怎么验”

- `docs/`
  - 总览与桥接层
  - 回答“怎么理解系统、怎么选能力、主干流程是什么、哪些典型场景如何进入”

- `drafts/` 与 `drafts/incubating/`
  - 草稿层
  - 回答“哪些判断还在形成、还没进入正式体系”

一句话：

**不同文档不是按文件名区分，而是按问题域区分。**

### 2.3 结构约定不等于 rule

这是一个很容易混淆的点。

像这些内容：

- 项目共享记忆放哪里
- 用户全局记忆放哪里
- 项目个人补充放哪里
- `.claude/CLAUDE.md` 是否作为标准位置

它们本质上不是“某类代码如何写”的规则，而是：

**仓库结构约定 / 记忆位置约定 / 维护约定。**

所以它们不适合写成 `rule`。

更合适的落点是：

- `README.md`：对使用者说明
- `MAINTAINING.md`：对维护者约束
- `templates/project-CLAUDE.md`：固化到模板

### 2.4 文档越多不等于越好

文档过多带来的问题，不是“写得辛苦”，而是：

- 同一件事重复写
- 维护成本开始反噬开发
- 文档体系本身需要被治理

所以更合理的方向不是无节制扩张，而是：

- 先立核心文档
- 再加桥接文档
- 最后才按需补辅助文档

## 3. 当前文档层级模型

### 3.1 入口层

- `README.md`

作用：
- 帮用户第一次接触时不误解这套系统
- 给出安装、部署、更新、验收和入口说明

### 3.2 方法层

- `PLAYBOOK.md`

作用：
- 定义开发范式
- 说明项目生命周期和阶段边界

### 3.3 决策层

- `WHY.md`

作用：
- 记录关键设计决策
- 解释为什么这样设计

### 3.4 维护层

- `MAINTAINING.md`

作用：
- 约束仓库如何继续演进
- 规定新增内容时该改哪里

### 3.5 总览桥接层

- `docs/`

作用：
- 不替代主文档
- 只负责解释、桥接、分类和导航

### 3.6 草稿层

- `drafts/`
- `drafts/incubating/`

作用：
- 先收纳模糊判断
- 再决定是否升格

## 4. 记忆位置与文档位置约定

### 4.1 项目共享记忆

标准位置应是：

- 项目根 `CLAUDE.md`

它承载的是：

- 当前项目的共享规则
- 当前项目的文档入口
- 当前项目的方法边界

### 4.2 用户全局记忆

标准位置应是：

- `~/.claude/CLAUDE.md`

它承载的是：

- 跨项目通用规则
- 全局能力调用边界
- 全局主控原则

### 4.3 项目个人补充

更推荐的方式是：

- 项目根 `CLAUDE.local.md`（兼容方案）
- 或在项目根 `CLAUDE.md` 中通过 `@import` 引入个人文件

它承载的是：

- 个人私有偏好
- 不想进入项目共享主记忆的个人说明

### 4.4 为什么 `.claude/CLAUDE.md` 不是标准位置

原因不是“不能用”，而是：

- 它不等价于标准项目主记忆位置
- 容易和项目私有记忆混淆
- 容易让项目共享内容和个人私有内容缠在一起

所以更稳的模型是：

- 项目共享 → 根 `CLAUDE.md`
- 用户全局 → `~/.claude/CLAUDE.md`
- 项目个人 → `CLAUDE.local.md` 或 `@import`

## 5. 常见混淆点

### 5.1 什么内容该进 `README`

适合：

- 这是什么
- 怎么开始用
- 一级入口认知

不适合：

- 长篇方法论
- 详细维护规则

### 5.2 什么内容该进 `PLAYBOOK`

适合：

- 开发范式
- 阶段边界
- 先做什么、后做什么

不适合：

- 能力分类细节
- TUI 操作细节

### 5.3 什么内容该进 `docs/`

适合：

- 理解框架
- 能力区别
- 主干流程
- 场景导航

不适合：

- 逐项命令说明书
- 另一套完整主文档体系

### 5.4 什么内容只是草稿，不应过早转正

如果一段内容仍然处于：

- 刚形成
- 还没稳定
- 还不确定归宿

就不该急着塞进正式文档。

它更适合先留在：

- `drafts/`
- `drafts/incubating/`

### 5.5 什么内容属于仓库结构约定，而不是运行时规则

凡是回答：

- 放哪里
- 哪类文件负责什么
- 哪种记忆位置是标准方案
- 哪些内容不应进某类文档

这类内容，通常属于结构约定，而不是运行时规则。

## 6. 后续判断标准

当新内容出现时，可以先问自己 3 个问题：

1. 这段内容回答的是什么问题域？
2. 这段内容是给谁看的？
3. 这段内容是在约束运行时行为，还是在约束结构和维护边界？

如果回答是：

- 给用户看 → 更可能进 `README`
- 给方法层看 → 更可能进 `PLAYBOOK`
- 给维护者看 → 更可能进 `MAINTAINING`
- 给系统解释和桥接看 → 更可能进 `docs/`
- 还没稳定 → 先留在 `drafts/`
