# 新约束归属判断

> **文档职责**：说明新增一条规范、约束、模板或经验时，应该优先放到运行层、项目层、文档层还是 prompt 草稿层。
> **适用场景**：维护 blueprint、想新增一条规则、判断某段内容是否应该进 `CLAUDE.md`、`skills/`、`docs/` 或 `drafts/prompts/` 时。
> **目标读者**：本仓库维护者。
> **维护规范**：如果仓库对运行层、文档层和 prompt 层的定位发生变化，需要同步更新本判断表。

## 1. 先判断：这是在改变 AI 行为，还是在沉淀方法与经验

先回答这个问题：

- 这段内容是不是希望 Claude / AI 在运行时长期遵守？
- 还是只是你对某类任务的做法总结、模板、检查项、写作母版？

如果是后者，默认不要先放运行层。

## 2. 判断顺序

```text
新约束出现
│
├─ Q1：这是让 AI 在运行时改变行为，还是记录方法 / 经验 / 决策？
│   │
│   ├─ 方法 / 经验 / 决策
│   │   ├─ 更像长期说明 → docs/
│   │   └─ 更像任务模板 / 可复制提示词 → drafts/prompts/
│   │
│   └─ 运行时行为
│       ├─ 跨项目长期成立
│       │   ├─ 很短、很硬 → CLAUDE.md
│       │   └─ 需要完整模板或长说明 → 先放 drafts/prompts/ 观察
│       │
│       └─ 只在当前项目或当前仓库成立
│           ├─ 项目专属 → 项目 CLAUDE.md / 项目文档
│           └─ 仓库维护专属 → docs/ 或 MAINTAINING 类文档
```

## 3. 当前仓库的默认立场

当前更推荐：

- `CLAUDE.md`
  只保留最小硬约束
- `skills/`
  谨慎使用，优先保留成熟、稳定、真正像“运行时操作手册”的能力
- `docs/`
  承担长期说明、边界解释、方法沉淀
- `drafts/prompts/`
  承担提示词母版、任务模板、检查清单、方法草案

也就是说：

- 不确定是否值得进运行层时，先放 `drafts/prompts/`
- 不确定是否已经稳定为正式规范时，先放 `drafts/docs/` 或 `drafts/prompts/`

## 4. 什么时候不要再升 skill

以下情况默认不要升成 skill：

- 只是方法论文档
- 只是模板或文档母版
- 只是某类任务的检查清单
- 只是你当前的一套个人偏好总结
- 还没有经过足够多真实使用验证

这些内容更适合先作为 prompt 文档存在。

## 5. 一个实用判断标准

只有同时满足下面 4 条，才考虑把内容从 `drafts/prompts/` 升到运行层：

- 触发场景稳定
- 输出结构稳定
- 边界清晰，能说清楚“什么时候不要用它”
- 它更像操作手册，而不是方法总结
