# 系统级蓝图 vs 项目级蓝图

> **文档职责**：收口“系统级蓝图”和“项目级蓝图”之间的边界问题，避免相关判断继续散落在多个普通草稿中。
> **当前状态**：重点草稿。已经形成清晰判断，但还不急着直接升格为正式文档。
> **后续可能去向**：`docs/用户心智模型.md`、`PLAYBOOK.md`、`WHY.md`

---

## 1. 这篇草稿要解决什么问题

当前最容易混淆的，不是某个命令怎么写，而是 3 个维度容易缠在一起：

1. **系统级** vs **项目级**
2. **给人看的** vs **给 Claude 运行时用的**
3. **方法论** vs **命令 / 能力**

一旦这 3 个维度混在一起，就会出现几个典型症状：

- 明明在讨论项目起盘方法，却试图让系统级 command 一次解决
- 明明是项目级文档边界问题，却误以为应该继续加全局配置
- 明明是“人如何使用 Claude”的问题，却只盯着运行时提示词和 agent

所以这篇草稿真正要回答的是：

- 什么属于系统级蓝图
- 什么属于项目级蓝图
- 两者分别服务谁
- 两者如何衔接，而不是互相替代

---

## 2. 当前的核心判断

### 2.1 系统级蓝图负责什么

系统级蓝图的职责，不是替你决定每个项目怎么做，而是提供一套跨项目复用的基础能力。

它负责：

- 统一方法
- 统一能力
- 统一模板
- 统一入口

更具体一点，它承载的是：

- `~/.claude/CLAUDE.md`
- `commands/`
- `skills/`
- `agents/`
- `hooks/`
- `templates/`
- `settings.json`

这些内容的共同特点是：

- 跨项目复用
- 一次维护，多项目受益
- 重点是“提供稳定底座”，不是“替项目做具体判断”

### 2.2 项目级蓝图负责什么

项目级蓝图解决的是“当前这个项目到底怎么落地”的问题。

它负责：

- 项目专属边界
- 项目级文档体系
- 当前项目的起盘方式
- 当前项目的模块设计与演化

更具体一点，它承载的是：

- 项目根 `CLAUDE.md`
- `docs/design/...`
- `docs/实施清单.md`
- 项目自己的需求、专题、部署、运维文档

这些内容的共同特点是：

- 只对当前项目成立
- 对当前项目的范围、分层、模块、演化路径负责
- 重点是“把一个真实项目讲清楚”，不是追求跨项目通用

### 2.3 二者为什么不能混成一层

它们不能混的原因非常直接：

- 系统级强调**跨项目复用**
- 项目级强调**对当前项目负责**

一旦混在一起，最先失控的就是边界：

- command 会开始替代方法论
- 模板会开始承载项目现实
- agent 会开始假设项目级文档天然存在
- 记忆位置和文档位置会变得含糊

一句话：

**系统级蓝图提供通用底座，项目级蓝图承接具体现实。**

## 3. 一个推荐模型

### 3.1 系统级：给所有项目复用

系统级应该回答：

- 我有哪些稳定能力可以复用
- 我如何与 Claude 协作
- 我有哪些可复用的命令、规则、模板和保护机制

它应该尽量避免回答：

- 某个项目的模块划分
- 某个项目的技术栈细节
- 某个项目的文档映射表

### 3.2 项目级：对单个项目负责

项目级应该回答：

- 这个项目现在要解决什么问题
- 当前阶段做什么，不做什么
- 当前项目有哪些模块与边界
- 当前项目的文档如何组织
- 当前项目下一步最该推进什么

它应该尽量避免回答：

- 所有项目通用的协作机制
- 所有项目都要用的命令和模板

### 3.3 二者之间的连接点

二者的连接点，不是“系统级替代项目级”，而是：

- 系统级蓝图提供起盘工具
- 项目级蓝图承接项目现实
- 真实项目跑通后，再把稳定模式抽象回系统级

也就是说，顺序是：

**系统级提供底座 → 项目级落地 → 项目经验反哺系统级。**

## 4. 典型混淆点

### 4.1 系统级 command 是否在替代项目级方法判断

不能。

`/init-architecture` 可以补最小骨架，但不能替代“项目到底如何起盘”的方法判断。

所以更准确的理解应该是：

- `command` 提供入口
- 方法论决定顺序和边界

### 4.2 系统级 agent 是否可以直接假设项目级文档存在

不能直接假设。

系统级 agent 可以读取项目级文档，但它不应该把项目现实当成系统前提。

这意味着：

- `arch` / `rev` 依赖项目级文档时，要么检查存在性
- 要么明确要求先补最小项目骨架

### 4.3 项目级 `CLAUDE.md` 与系统级 `~/.claude/CLAUDE.md` 的职责边界

推荐模型是：

- `~/.claude/CLAUDE.md`：全局主控，跨项目通用
- 项目根 `CLAUDE.md`：项目共享主记忆，承接项目现实

不推荐把 `.claude/CLAUDE.md` 当成标准项目记忆位置。

### 4.4 项目模板和项目现实之间的边界

模板只该保留：

- 方法
- 骨架
- 起点

不该保留：

- 项目特有文档映射表
- 项目特有模块清单
- 项目已经长出来的复杂现实

一句话：

**模板只保留“如何起步”，不替代“项目已经发展成什么样”。**

## 5. `title` 给 blueprint 的启发

### 5.1 已被验证的模式

`title` 至少验证了这些模式是成立的：

- 项目级 `CLAUDE.md` 不只是命令清单，还应该承担文档工作流规则
- 项目初始化和日常开发要分开
- 架构设计、模块设计、实施清单这三类文档能形成稳定主线

### 5.2 不该直接模板化的内容

`title` 里有些东西虽然有价值，但不适合原样进入 blueprint：

- 项目特有的文档映射表
- 项目特有模块清单
- 已经长出来的复杂基础设施清单

这些是项目资产，不是系统级模板。

### 5.3 可以抽象回 blueprint 的内容

真正适合抽回系统级的，是这些“稳定模式”：

- 新模块先设计后实现
- 实现后必须做文档回写检查
- 文档入口和文档边界应该显式约定
- 项目初始化与日常迭代必须分开

一句话：

**`title` 最有价值的作用，是当“已验证样板间”；blueprint 要抽的是稳定模式，不是项目细节。**

## 6. 二者如何协作

更稳的协作方式是：

### 第一步：系统级提供底座

例如：

- `commands/`
- `skills/`
- `templates/`
- `hooks/`

### 第二步：项目级建立最小骨架

例如：

- 项目根 `CLAUDE.md`
- `docs/design/INDEX.md`
- `docs/design/架构设计方案.md`
- `docs/实施清单.md`

### 第三步：项目在真实迭代中长出自己的文档与模块

例如：

- 模块设计文档
- 基础设施文档
- 专题文档

### 第四步：把被反复验证的模式抽回系统级

例如：

- 模板优化
- 命令优化
- 文档边界优化
- 方法论抽象

## 7. 当前最稳的边界判断

遇到新内容时，可以先问这 3 个问题：

1. 这条内容在下一个完全不同的项目里还成立吗？
2. 这条内容是在提供“通用底座”，还是在描述“当前项目现实”？
3. 这条内容是在讲能力与机制，还是在讲项目方法与文档边界？

如果答案偏向：

- 跨项目成立
- 通用底座
- 能力与机制

就更可能属于**系统级蓝图**。

如果答案偏向：

- 只对当前项目成立
- 当前项目现实
- 项目方法与文档边界

就更可能属于**项目级蓝图**。
