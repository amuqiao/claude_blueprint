"输出载体"关注的不是某一次具体要选哪个结果，而是：

- 信息最终以什么形式交付

## 核心选型维度

### 1. 输出载体形式

你说需要"可视化图表"，这决定了我用什么方式生成：

- **方案A：交互式 HTML artifact**（内嵌图表+代码示例，可点击切换，一份自包含的文档）
- **方案B：Markdown 文件**（纯文字+代码块，图表用 ASCII 或 Mermaid，适合放进 repo 或 Notion）
- **方案C：分段输出**（先图表，再文字说明，边看边讲）

这部分后续可继续补充：

- 各方案适用场景
- 各方案优缺点
- 各方案维护成本
- 各方案对读者阅读路径的影响

### 2. 代码示例的粒度

你说要"参考规范写法"，我需要确认覆盖哪些：

**Level A（最小可用）：**

```json
// settings.json 最简权限规则
{ "permissions": { "allow": ["Bash(git commit:*)"] } }
```

**Level B（完整模板）：** 给出一份可以直接 fork 用的完整文件，包含常用字段和注释

**Level C（场景化示例）：** 比如"团队统一代码风格"、"CI 只读模式"、"个人本地覆盖"各一份

这部分后续可继续补充：

- 不同粒度分别服务什么读者
- 不同粒度适合放在哪类文档里
- 示例粒度与正文篇幅之间的关系

### 3. 叙述主线

两个工具的地位是"并列"还是"以一为主"？

- **并列**：每个能力都同等篇幅讲两边，用户自己选
- **以 Claude Code 为主**：先讲 Claude Code 建立基础，再讲 Codex 的差异和独有概念
- **以用户场景为主**：用户带着"我要做 X"来查，自然引出两边的做法

这影响整个文档的叙述逻辑。

这部分后续可继续补充：

- 不同叙述主线适配的读者类型
- 同一主题在不同叙述主线下的目录差异
- 哪些内容适合先讲概念，哪些适合先讲任务

### 4. 视角边界

"管理员视角"的范围边界涉及：

- **A. 个人开发者作为自己的"管理员"**（设置自己机器的全局策略）
- **B. 团队 Tech Lead**（在 repo 里设置项目级规范，让团队成员继承）
- **C. 企业 IT 管理员**（通过 managed-settings / requirements.toml 强制全机器策略，用户无法覆盖）

这部分后续可继续补充：

- 各角色的权限边界
- 各角色对应的配置层级
- 同一能力在不同角色视角下的差异

### 5. 文档生命周期

文档的"生命周期"会影响结构设计：

- 是一次性参考（查完就走）→ 适合索引式结构
- 是持续学习路径（从入门到精通）→ 适合渐进式结构
- 是团队 onboarding 材料（让新成员快速上手）→ 适合任务导向结构

这部分后续可继续补充：

- 生命周期与更新频率的关系
- 生命周期与目录稳定性的关系
- 生命周期与示例密度的关系
