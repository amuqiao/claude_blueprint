## 直观对比示例

### 示例 1: 普通格式（.env）

```bash
# ========== .env 文件内容 ==========
LLM_API_KEY=sk-test-123
```

**执行流程：**
```python
# 1. Pydantic Settings 读取文件
with open('.env') as f:
    content = f.read()  # "LLM_API_KEY=sk-test-123"

# 2. 直接解析
LLM_API_KEY = "sk-test-123"  # ← 最终值

# 3. FastAPI 应用使用
settings.LLM_API_KEY  # → "sk-test-123"
```

---

### 示例 2: 变量替换格式（.env.dev）

```bash
# ========== .env.dev 文件内容 ==========
LLM_API_KEY=${LLM_API_KEY:-sk-dev-api-key-here}
```

#### 场景 A: 有环境变量

```bash
# 1. 服务器上设置环境变量
export LLM_API_KEY=sk-production-real-key

# 2. 启动 Docker Compose
ENV_FILE=.env.dev docker compose up -d

# 3. Docker Compose 读取 .env.dev 并替换
#    ${LLM_API_KEY:-sk-dev-api-key-here}
#    → sk-production-real-key (来自环境变量)

# 4. 传递给容器
docker run -e LLM_API_KEY=sk-production-real-key ...

# 5. FastAPI 应用使用
settings.LLM_API_KEY  # → "sk-production-real-key"
```

#### 场景 B: 无环境变量

```bash
# 1. 服务器上没有设置环境变量
# (未执行 export LLM_API_KEY=...)

# 2. 启动 Docker Compose
ENV_FILE=.env.dev docker compose up -d

# 3. Docker Compose 读取 .env.dev 并替换
#    ${LLM_API_KEY:-sk-dev-api-key-here}
#    → sk-dev-api-key-here (使用默认值)

# 4. 传递给容器
docker run -e LLM_API_KEY=sk-dev-api-key-here ...

# 5. FastAPI 应用使用
settings.LLM_API_KEY  # → "sk-dev-api-key-here"
```

---

## 两种格式的核心区别

| 特性                  | 普通格式    | 变量替换格式              |
| --------------------- | ----------- | ------------------------- |
| **语法**              | `VAR=value` | `VAR=${VAR:-default}`     |
| **是否支持覆盖**      | ❌ 否       | ✅ 是                     |
| **值来源**            | 固定值      | 环境变量 > 默认值         |
| **Pydantic Settings** | ✅ 支持     | ❌ 不支持（会读到字面值） |
| **docker run --env-file** | ✅ 支持 | ❌ 不支持（不会自动替换） |
| **docker run -e**     | ✅ 支持     | ✅ 支持（由 shell 先展开） |
| **Docker Compose**    | ✅ 支持     | ✅ 支持（会自动替换）     |
| **使用场景**          | 本地开发 / 简单容器启动 | Compose 部署 / 需要环境变量覆盖 |

## 三种使用方式怎么选

### 1. FastAPI 本地开发部署

```bash
# 直接在宿主机启动应用
uvicorn app.main:app --reload
```

这种场景通常是 **Pydantic Settings 直接读取 `.env` 文件**。

- 推荐：**普通格式**
- 不推荐：**变量替换格式**

原因：

- 普通格式会被 Python 正常解析
- 变量替换格式里的 `${VAR:-default}` 不会被 Pydantic 自动展开

---

### 2. Dockerfile + docker run 部署

#### 方式 A: 直接传 `-e`

```bash
docker run -e LLM_API_KEY="${LLM_API_KEY:-sk-dev-api-key-here}" my-app
```

这种写法是 **shell 先展开**，再把最终值传给容器。

- 普通格式思路：✅ 可以
- 变量替换格式思路：✅ 也可以，但要写在命令里，不是写在 `env-file` 里

#### 方式 B: 使用 `--env-file`

```bash
docker run --env-file .env my-app
```

这种写法只适合 **普通格式** 的 env 文件：

```bash
LLM_API_KEY=sk-test-123
```

如果你写成：

```bash
LLM_API_KEY=${LLM_API_KEY:-sk-dev-api-key-here}
```

`docker run --env-file` 不会自动替换，容器里拿到的会是字面值。

所以这一类部署要分开看：

- `docker run -e`：普通格式 / 变量替换思路都能用
- `docker run --env-file`：只适合普通格式

---

### 3. Docker Compose 部署

```bash
ENV_FILE=.env.dev docker compose up -d
```

这种场景下，**Docker Compose 会先读取 env 文件并处理变量替换**，再把结果传给容器。

- 普通格式：✅ 支持
- 变量替换格式：✅ 支持

所以服务器上如果希望：

- 优先使用系统环境变量
- 没有时再回退默认值

那 `VAR=${VAR:-default}` 这种格式更适合 `docker compose`。

---

## 快速记忆

```bash
# 普通格式 = 固定值
LLM_API_KEY=sk-test-123

# 变量替换格式 = 灵活值（环境变量 or 默认值）
LLM_API_KEY=${LLM_API_KEY:-sk-test-123}
              ^^^^^^^^^^^^          ^^^^^^^^^^^^
              ｜                    ｜
              检查环境变量          如果没有，用这个默认值
```

**一句话总结：**
- **普通格式**：直接写死值，简单但不灵活
- **变量替换格式**：优先用环境变量，没有就用默认值，适合 `docker compose` 或 `docker run -e`，不适合直接给 `docker run --env-file`
