## 📋 Dockerfile 部署方案分析报告

### 一、当前部署架构分析

#### 1.1 现有部署方式对比

| 部署方式           | 场景               | 使用工具       | 服务数量                    | 依赖管理                   |
| ------------------ | ------------------ | -------------- | --------------------------- | -------------------------- |
| **dev.sh**         | 本地开发（宿主机） | Bash + PID     | 3 个（api、worker、flower） | PostgreSQL/Redis 需 Docker |
| **docker-compose** | 本地开发（容器）   | Docker Compose | 5 个（含 DB）               | 自动编排                   |
| **Dockerfile**     | 镜像构建           | Docker         | 单个镜像，3 个 target       | 需外部编排                 |

#### 1.2 当前 Dockerfile 特点

**优点：**
- ✅ 多阶段构建，优化镜像大小
- ✅ 使用 uv 管理依赖，快速安装
- ✅ 三个 target（fastapi、celery-worker、flower），职责清晰

**问题：**
- ⚠️ 缺少数据库迁移步骤（Alembic）
- ⚠️ 缺少健康检查（HEALTHCHECK）
- ⚠️ 没有非 root 用户（安全风险）
- ⚠️ 缺少生产环境优化（如静态文件处理）

---

### 二、改为 Dockerfile 部署的三种理解

#### 理解 A：纯 Docker 命令部署（不用 docker-compose）

**场景：** 手动管理容器，适合简单部署或学习

**需要做的：**
1. 创建 Docker 网络
2. 手动启动 PostgreSQL 和 Redis 容器
3. 构建应用镜像
4. 按顺序启动 FastAPI、Worker、Flower 容器
5. 手动配置环境变量和容器间连接

**优点：** 理解容器工作原理
**缺点：** 繁琐，容易出错，不适合生产

---

#### 理解 B：K8s 生产部署（推荐）⭐

**场景：** 生产环境、云平台部署

**需要做的：**
1. 优化 Dockerfile（生产版本）
2. 创建 K8s 资源定义（Deployment、Service、Ingress）
3. 配置管理（ConfigMap、Secret）
4. 存储管理（PVC）
5. 监控和日志采集

**优点：** 高可用、可扩展、符合云原生
**缺点：** 复杂度高，需要 K8s 知识

---

#### 理解 C：简化容器部署（docker-compose 生产版）

**场景：** 小规模生产、VPS 部署

**需要做的：**
1. 创建 `docker-compose.prod.yml`（生产配置）
2. 优化 Dockerfile
3. 移除开发特性（热重载、volume 挂载）
4. 添加健康检查和重启策略

**优点：** 简单易用，适合小团队
**缺点：** 单机部署，扩展性有限

### 三、各方案需要修改的配置和代码

## 方案 A：纯 Docker 命令部署

### 需要修改的地方

#### 1. Dockerfile 调整

**需要添加：**
```dockerfile
# 添加健康检查
FROM base AS fastapi
EXPOSE 8000
HEALTHCHECK --interval=30s --timeout=3s --start-period=5s --retries=3 \
  CMD curl -f http://localhost:8000/health || exit 1
CMD ["uv", "run", "uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]
```

#### 2. 手动启动脚本

**需要创建：** `scripts/docker-deploy.sh`

```bash
#!/usr/bin/env bash
# 纯 Docker 部署脚本

# 1. 创建网络
docker network create translate-net

# 2. 启动 PostgreSQL
docker run -d \
  --name translate-postgres \
  --network translate-net \
  -e POSTGRES_USER=postgres \
  -e POSTGRES_PASSWORD=your_password \
  -e POSTGRES_DB=translate_db \
  -p 5432:5432 \
  postgres:16-alpine

# 3. 启动 Redis
docker run -d \
  --name translate-redis \
  --network translate-net \
  -p 6379:6379 \
  redis:7-alpine

# 4. 构建应用镜像
docker build -t translate-app:latest .

# 5. 运行数据库迁移
docker run --rm \
  --network translate-net \
  -e DATABASE_URL="postgresql+asyncpg://postgres:your_password@translate-postgres:5432/translate_db" \
  translate-app:latest \
  uv run alembic upgrade head

# 6. 启动 FastAPI
docker run -d \
  --name translate-api \
  --network translate-net \
  -p 8000:8000 \
  -e DATABASE_URL="postgresql+asyncpg://postgres:your_password@translate-postgres:5432/translate_db" \
  -e REDIS_URL="redis://translate-redis:6379/0" \
  --env-file .env \
  translate-app:latest

# 7. 启动 Celery Worker
docker run -d \
  --name translate-worker \
  --network translate-net \
  -e DATABASE_URL="postgresql+asyncpg://postgres:your_password@translate-postgres:5432/translate_db" \
  -e REDIS_URL="redis://translate-redis:6379/0" \
  --env-file .env \
  translate-app:latest \
  uv run celery -A app.celery_app worker --loglevel=info

# 8. 启动 Flower
docker run -d \
  --name translate-flower \
  --network translate-net \
  -p 5555:5555 \
  -e REDIS_URL="redis://translate-redis:6379/0" \
  translate-app:latest \
  uv run celery -A app.celery_app flower --port=5555
```

#### 3. 配置管理调整

**问题：** `--env-file` 需要在宿主机上有 `.env` 文件

**解决方案：**
- 方式 1：在服务器上创建 `.env` 文件
- 方式 2：改用 `-e` 参数逐个传递环境变量

## 方案 B：K8s 生产部署（推荐）⭐

### 需要修改的地方

#### 1. Dockerfile 生产优化

**需要修改：** `Dockerfile` → `Dockerfile.prod`

```dockerfile
# 生产环境优化版本
FROM python:3.11-slim AS base

# 安装必要的系统依赖
RUN apt-get update && apt-get install -y --no-install-recommends \
    curl \
    && rm -rf /var/lib/apt/lists/*

# 创建非 root 用户
RUN useradd -m -u 1000 appuser

# 安装 uv
COPY --from=ghcr.io/astral-sh/uv:latest /uv /usr/local/bin/uv

# 设置工作目录
WORKDIR /app

# 设置环境变量
ENV PYTHONUNBUFFERED=1 \
    PYTHONDONTWRITEBYTECODE=1 \
    UV_COMPILE_BYTECODE=1 \
    UV_LINK_MODE=copy

# 复制依赖文件
COPY --chown=appuser:appuser pyproject.toml ./

# 安装依赖
RUN uv sync --frozen --no-dev

# 复制应用代码
COPY --chown=appuser:appuser app ./app
COPY --chown=appuser:appuser alembic ./alembic
COPY --chown=appuser:appuser alembic.ini ./

# 切换到非 root 用户
USER appuser

# FastAPI 服务
FROM base AS fastapi
EXPOSE 8000
HEALTHCHECK --interval=30s --timeout=3s --start-period=10s --retries=3 \
  CMD curl -f http://localhost:8000/health || exit 1
CMD ["uv", "run", "uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000", "--workers", "4"]

# Celery Worker
FROM base AS celery-worker
HEALTHCHECK --interval=60s --timeout=10s --start-period=30s --retries=3 \
  CMD uv run celery -A app.celery_app inspect ping -d celery@$HOSTNAME || exit 1
CMD ["uv", "run", "celery", "-A", "app.celery_app", "worker", "--loglevel=info", "--concurrency=4"]

# Flower 监控
FROM base AS flower
EXPOSE 5555
CMD ["uv", "run", "celery", "-A", "app.celery_app", "flower", "--port=5555"]

# 数据库迁移（一次性任务）
FROM base AS migration
CMD ["uv", "run", "alembic", "upgrade", "head"]
```

#### 3. 配置管理调整

**需要删除：**
- ❌ `.env` 文件相关逻辑（K8s 不使用）
- ❌ `ENV_FILE` 环境变量支持

**需要修改：** `app/core/config.py`

```python
def get_env_file() -> str:
    """
    K8s 环境下不读取文件，直接从环境变量读取
    """
    app_env = os.getenv("APP_ENV", "").strip().lower()
    
    if app_env == "prod" or app_env == "production":
        # 生产环境：不读文件，直接从环境变量读
        print("[Config] 生产环境，从环境变量加载配置（K8s）")
        return ""  # pydantic 会从环境变量读取
    
    # 本地开发保持不变
    if not app_env:
        if os.path.exists(".env"):
            print("[Config] 加载配置: .env（本地开发）")
            return ".env"
    
    # 开发服务器
    env_file = f".env.{app_env}"
    if os.path.exists(env_file):
        print(f"[Config] 加载配置: {env_file}")
        return env_file
    
    return ""
```

#### 4. 部署流程调整

**需要创建：** `scripts/k8s-deploy.sh`

```bash
#!/usr/bin/env bash
# K8s 部署脚本

# 1. 构建并推送镜像
docker build -f Dockerfile.prod -t your-registry/translate-app:latest --target fastapi .
docker push your-registry/translate-app:latest

# 2. 应用 K8s 配置
kubectl apply -f k8s/namespace.yaml
kubectl apply -f k8s/configmap.yaml
kubectl apply -f k8s/secret.yaml  # 需要先创建 secret

# 3. 部署数据库和 Redis（或使用云服务）
kubectl apply -f k8s/postgres-deployment.yaml
kubectl apply -f k8s/postgres-service.yaml
kubectl apply -f k8s/redis-deployment.yaml
kubectl apply -f k8s/redis-service.yaml

# 4. 运行数据库迁移
kubectl apply -f k8s/migration-job.yaml
kubectl wait --for=condition=complete --timeout=300s job/translate-migration

# 5. 部署应用
kubectl apply -f k8s/api-deployment.yaml
kubectl apply -f k8s/api-service.yaml
kubectl apply -f k8s/api-ingress.yaml
kubectl apply -f k8s/worker-deployment.yaml
kubectl apply -f k8s/flower-deployment.yaml
kubectl apply -f k8s/flower-service.yaml

# 6. 验证部署
kubectl -n translate get pods
kubectl -n translate get svc
```

## 方案 C：docker-compose 生产版

### 需要修改的地方

#### 1. 创建生产版 docker-compose

**需要创建：** `docker-compose.prod.yml`

```yaml
version: '3.8'

services:
  postgres:
    image: postgres:16-alpine
    container_name: translate-postgres
    environment:
      POSTGRES_USER: ${DB_USER}
      POSTGRES_PASSWORD: ${DB_PASSWORD}
      POSTGRES_DB: ${DB_NAME}
    volumes:
      - postgres_data:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U ${DB_USER}"]
      interval: 10s
      timeout: 5s
      retries: 5
    restart: always
    networks:
      - translate-net

  redis:
    image: redis:7-alpine
    container_name: translate-redis
    command: redis-server --requirepass ${REDIS_PASSWORD}
    volumes:
      - redis_data:/data
    healthcheck:
      test: ["CMD", "redis-cli", "--pass", "${REDIS_PASSWORD}", "ping"]
      interval: 10s
      timeout: 3s
      retries: 5
    restart: always
    networks:
      - translate-net

  migration:
    build:
      context: .
      dockerfile: Dockerfile.prod
      target: migration
    container_name: translate-migration
    env_file:
      - .env.prod
    environment:
      DATABASE_URL: postgresql+asyncpg://${DB_USER}:${DB_PASSWORD}@postgres:5432/${DB_NAME}
    depends_on:
      postgres:
        condition: service_healthy
    networks:
      - translate-net
    restart: "no"

  api:
    build:
      context: .
      dockerfile: Dockerfile.prod
      target: fastapi
    container_name: translate-api
    ports:
      - "${API_PORT:-8000}:8000"
    env_file:
      - .env.prod
    environment:
      DATABASE_URL: postgresql+asyncpg://${DB_USER}:${DB_PASSWORD}@postgres:5432/${DB_NAME}
      REDIS_URL: redis://:${REDIS_PASSWORD}@redis:6379/0
    depends_on:
      postgres:
        condition: service_healthy
      redis:
        condition: service_healthy
      migration:
        condition: service_completed_successfully
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8000/health"]
      interval: 30s
      timeout: 3s
      retries: 3
    restart: always
    networks:
      - translate-net

  worker:
    build:
      context: .
      dockerfile: Dockerfile.prod
      target: celery-worker
    container_name: translate-worker
    env_file:
      - .env.prod
    environment:
      DATABASE_URL: postgresql+asyncpg://${DB_USER}:${DB_PASSWORD}@postgres:5432/${DB_NAME}
      REDIS_URL: redis://:${REDIS_PASSWORD}@redis:6379/0
    depends_on:
      postgres:
        condition: service_healthy
      redis:
        condition: service_healthy
    restart: always
    networks:
      - translate-net

  flower:
    build:
      context: .
      dockerfile: Dockerfile.prod
      target: flower
    container_name: translate-flower
    ports:
      - "${FLOWER_PORT:-5555}:5555"
    env_file:
      - .env.prod
    environment:
      REDIS_URL: redis://:${REDIS_PASSWORD}@redis:6379/0
    depends_on:
      - redis
      - worker
    restart: always
    networks:
      - translate-net

volumes:
  postgres_data:
  redis_data:

networks:
  translate-net:
    driver: bridge
```

#### 2. 部署脚本

**需要创建：** `scripts/deploy-prod.sh`

```bash
#!/usr/bin/env bash
# 生产环境部署脚本

set -euo pipefail

# 1. 检查环境变量
if [[ ! -f ".env.prod" ]]; then
    echo "错误: .env.prod 不存在"
    echo "请从模板创建: cp .env.prod.example .env.prod"
    exit 1
fi

# 2. 构建镜像
docker-compose -f docker-compose.prod.yml build

# 3. 启动服务
docker-compose -f docker-compose.prod.yml up -d

# 4. 查看状态
docker-compose -f docker-compose.prod.yml ps

# 5. 查看日志
docker-compose -f docker-compose.prod.yml logs -f
```

#### 3. 无需修改代码

**优点：** 配置管理逻辑保持不变，只需创建 `.env.prod`

### 五、通用需要修改的地方（所有方案）

#### 1. Dockerfile 必须优化

| 项目         | 当前状态    | 需要添加            | 原因         |
| ------------ | ----------- | ------------------- | ------------ |
| 健康检查     | ❌ 无       | ✅ HEALTHCHECK      | 容器编排需要 |
| 非 root 用户 | ❌ root     | ✅ appuser          | 安全最佳实践 |
| 数据库迁移   | ❌ 无       | ✅ migration target | 自动化部署   |
| 生产优化     | ❌ 开发模式 | ✅ workers 参数     | 性能优化     |
| curl 工具    | ❌ 无       | ✅ 安装 curl        | 健康检查需要 |

#### 2. 配置管理调整

**当前问题：**
- `app/core/config.py` 的 `get_env_file()` 假设文件存在
- 容器环境下 `.env` 文件可能不存在

**解决方案：**
```python
# 优化后的配置加载
class Settings(BaseSettings):
    model_config = SettingsConfigDict(
        env_file=get_env_file() if get_env_file() else None,  # 文件不存在时为 None
        env_file_encoding="utf-8",
        case_sensitive=False,
        extra="ignore",
    )
```

#### 3. 日志输出调整

**当前问题：**
- dev.sh 将日志写入文件（`logs/api.log`）
- 容器环境应输出到 stdout/stderr

**需要修改：** 无需修改，uvicorn 和 celery 默认输出到 stdout

**但需要注意：** 删除 dev.sh 中的日志重定向逻辑

#### 4. 数据库迁移集成

**当前问题：**
- 没有自动运行 Alembic 迁移
- 需要手动执行 `alembic upgrade head`

**解决方案：**
- 方案 A：启动脚本中添加迁移步骤
- 方案 B：K8s Job 运行迁移
- 方案 C：docker-compose 添加 migration 服务

### 七、总结

- ✅ 当前的 dev.sh（本地开发保留）
