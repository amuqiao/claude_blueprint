## 📊 两种部署方式的关系

### 1. 层级关系

```
Dockerfile（基础层）
    ↓ 构建镜像
docker-compose.yml（编排层）
    ↓ 本地开发/测试
生产部署（部署层）
    ├─ K8s
    ├─ 云平台
    └─ VPS + docker-compose.prod.yml
```

**关键理解：**
- **Dockerfile 是基础**：定义如何构建应用镜像
- **docker-compose 是编排**：定义如何组合多个服务（包括应用镜像和第三方镜像）
- **两者不是替代关系，而是依赖关系**

## 📋 具体配置策略

### 策略 1：Dockerfile（基础，必须有）

**职责：** 定义应用镜像构建方式

```dockerfile
# Dockerfile - 所有环境共用
FROM python:3.11-slim AS base

# 安装 uv
COPY --from=ghcr.io/astral-sh/uv:latest /uv /usr/local/bin/uv

# 设置工作目录
WORKDIR /app

# 复制依赖文件
COPY pyproject.toml ./

# 安装依赖
RUN uv sync --frozen --no-dev

# 复制应用代码
COPY app ./app

# 三个 target
FROM base AS fastapi
EXPOSE 8000
CMD ["uv", "run", "uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]

FROM base AS celery-worker
CMD ["uv", "run", "celery", "-A", "app.celery_app", "worker", "--loglevel=info"]

FROM base AS flower
EXPOSE 5555
CMD ["uv", "run", "celery", "-A", "app.celery_app", "flower", "--port=5555"]
```

**特点：**
- ✅ 所有部署方式都使用这个 Dockerfile
- ✅ 多阶段构建，支持不同服务
- ✅ 不包含环境特定配置

---

### 策略 2：docker-compose.yml（本地开发，全量）

**职责：** 本地开发时一键启动所有服务

```yaml
# docker-compose.yml - 本地开发全量部署
services:
  # 基础设施（本地 Docker）
  postgres:
    image: postgres:16-alpine
    environment:
      POSTGRES_USER: postgres
      POSTGRES_PASSWORD: postgres
      POSTGRES_DB: translate_db
    ports:
      - "5432:5432"
    volumes:
      - postgres_data:/var/lib/postgresql/data

  redis:
    image: redis:7-alpine
    ports:
      - "6379:6379"
    volumes:
      - redis_data:/data

  # 应用服务（使用 Dockerfile 构建）
  fastapi:
    build:
      context: .
      target: fastapi          # ← 使用 Dockerfile 的 fastapi target
    ports:
      - "8000:8000"
    env_file:
      - ${ENV_FILE:-.env}
    environment:
      DATABASE_URL: postgresql+asyncpg://postgres:postgres@postgres:5432/translate_db
      REDIS_URL: redis://redis:6379/0
    depends_on:
      - postgres
      - redis
    volumes:
      - ./app:/app/app         # 开发时热重载

  celery-worker:
    build:
      context: .
      target: celery-worker    # ← 使用 Dockerfile 的 celery-worker target
    env_file:
      - ${ENV_FILE:-.env}
    environment:
      DATABASE_URL: postgresql+asyncpg://postgres:postgres@postgres:5432/translate_db
      REDIS_URL: redis://redis:6379/0
    depends_on:
      - postgres
      - redis
    volumes:
      - ./app:/app/app

  flower:
    build:
      context: .
      target: flower           # ← 使用 Dockerfile 的 flower target
    ports:
      - "5555:5555"
    env_file:
      - ${ENV_FILE:-.env}
    environment:
      REDIS_URL: redis://redis:6379/0
    depends_on:
      - redis
    volumes:
      - ./app:/app/app

volumes:
  postgres_data:
  redis_data:
```

**使用方式：**
```bash
# 本地开发（一键启动全部）
docker-compose up -d

# 或指定环境
ENV_FILE=.env.dev docker-compose up -d
```

**特点：**
- ✅ 包含基础设施（PostgreSQL、Redis）
- ✅ 使用 Dockerfile 构建应用镜像
- ✅ 适合本地开发，有热重载
- ✅ 一个命令启动全部服务

### 策略 3：docker-compose.prod.yml（生产，VPS 单机）

**职责：** VPS 或单机生产环境部署

```yaml
# docker-compose.prod.yml - 生产环境（单机）
services:
  # 选项 A：使用本地 PostgreSQL（适合小规模）
  postgres:
    image: postgres:16-alpine
    environment:
      POSTGRES_USER: ${DB_USER}
      POSTGRES_PASSWORD: ${DB_PASSWORD}
      POSTGRES_DB: ${DB_NAME}
    volumes:
      - postgres_data:/var/lib/postgresql/data
    restart: always

  redis:
    image: redis:7-alpine
    command: redis-server --requirepass ${REDIS_PASSWORD}
    volumes:
      - redis_data:/data
    restart: always

  # 应用服务（生产优化）
  fastapi:
    build:
      context: .
      target: fastapi
    ports:
      - "${API_PORT:-8000}:8000"
    env_file:
      - .env.prod
    environment:
      DATABASE_URL: postgresql+asyncpg://${DB_USER}:${DB_PASSWORD}@postgres:5432/${DB_NAME}
      REDIS_URL: redis://:${REDIS_PASSWORD}@redis:6379/0
      APP_ENV: prod                    # ← 强制生产模式
    depends_on:
      - postgres
      - redis
    restart: always
    # 注意：生产环境不挂载 volumes（无热重载）

  celery-worker:
    build:
      context: .
      target: celery-worker
    env_file:
      - .env.prod
    environment:
      DATABASE_URL: postgresql+asyncpg://${DB_USER}:${DB_PASSWORD}@postgres:5432/${DB_NAME}
      REDIS_URL: redis://:${REDIS_PASSWORD}@redis:6379/0
      APP_ENV: prod
    depends_on:
      - postgres
      - redis
    restart: always

  flower:
    build:
      context: .
      target: flower
    ports:
      - "${FLOWER_PORT:-5555}:5555"
    env_file:
      - .env.prod
    environment:
      REDIS_URL: redis://:${REDIS_PASSWORD}@redis:6379/0
    depends_on:
      - redis
    restart: always

volumes:
  postgres_data:
  redis_data:
```

**使用方式：**
```bash
# 生产部署
docker-compose -f docker-compose.prod.yml up -d

# 查看状态
docker-compose -f docker-compose.prod.yml ps

# 查看日志
docker-compose -f docker-compose.prod.yml logs -f fastapi
```

**特点：**
- ✅ 使用同一个 Dockerfile
- ✅ 生产优化（restart: always，无 volumes）
- ✅ 可选包含 PostgreSQL/Redis（或连接外部云服务）
- ✅ 适合 VPS、单机生产环境

### 策略 4：纯 Dockerfile 部署（云平台，分离基础设施）

**职责：** K8s 或云平台部署，基础设施分离

**场景：**
- PostgreSQL：使用云数据库（RDS、阿里云 RDS）
- Redis：使用云缓存（ElastiCache、阿里云 Redis）
- 应用：只部署 FastAPI、Worker、Flower

**使用方式：**
```bash
# 1. 构建镜像（使用同一个 Dockerfile）
docker build -t your-registry/translate-app:latest --target fastapi .
docker push your-registry/translate-app:latest

# 2. K8s 部署（不包含 PostgreSQL/Redis）
kubectl apply -f k8s/api-deployment.yaml
kubectl apply -f k8s/worker-deployment.yaml
kubectl apply -f k8s/flower-deployment.yaml
```

**K8s Deployment 示例：**
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: translate-api
spec:
  replicas: 3
  template:
    spec:
      containers:
      - name: fastapi
        image: your-registry/translate-app:latest  # ← 使用 Dockerfile 构建的镜像
        env:
        - name: DATABASE_URL
          value: "postgresql+asyncpg://user:pass@rds.example.com:5432/db"  # ← 云数据库
        - name: REDIS_URL
          value: "redis://:pass@elasticache.example.com:6379/0"  # ← 云 Redis
```

**特点：**
- ✅ 使用同一个 Dockerfile
- ✅ 基础设施分离（云服务）
- ✅ 高可用、可扩展
- ✅ 适合大规模生产环境

## 🎯 共存的最佳实践

### 1. 一个 Dockerfile，多种部署方式

```
Dockerfile（基础）
    ├─ docker-compose.yml（本地开发：含 DB/Redis）
    ├─ docker-compose.prod.yml（VPS 生产：含 DB/Redis）
    └─ K8s YAML（云平台生产：分离 DB/Redis）
```

### 2. 环境变量分层覆盖

```
.env.example           # 模板（提交）
.env                   # 本地开发（不提交）
.env.dev               # 开发服务器（提交，脱敏）
.env.prod              # 生产环境（不提交）

docker-compose.yml     # 使用 ${ENV_FILE:-.env}
docker-compose.prod.yml # 使用 .env.prod
K8s                    # 使用 ConfigMap/Secret
```

### 3. 部署脚本清晰分离

```bash
# scripts/dev.sh - 本地宿主机开发（不用 Docker）
./dev.sh start

# scripts/docker-dev.sh - 本地 Docker 开发（全量）
docker-compose up -d

# scripts/deploy-vps.sh - VPS 生产部署
docker-compose -f docker-compose.prod.yml up -d

# scripts/deploy-k8s.sh - K8s 生产部署
kubectl apply -f k8s/
```

## 🎯 总结

### ✅ 可以且应该同时存在

| 关系         | 说明                                                                 |
| ------------ | -------------------------------------------------------------------- |
| **依赖关系** | docker-compose **依赖** Dockerfile 构建应用镜像                      |
| **互补关系** | Dockerfile 定义镜像，docker-compose 定义编排                         |
| **分层关系** | Dockerfile（基础层） → docker-compose（编排层） → 部署脚本（部署层） |

### 🎯 最佳实践

1. **一个 Dockerfile，多种部署方式**
   - 本地开发：docker-compose.yml（全量）
   - VPS 生产：docker-compose.prod.yml（全量或混合）
   - 云平台：K8s（分离基础设施）

2. **环境配置分层管理**
   - .env：本地开发
   - .env.dev：开发服务器
   - .env.prod：生产环境
   - ConfigMap/Secret：K8s 生产

3. **部署脚本清晰分离**
   - dev.sh：宿主机开发
   - docker-dev.sh：Docker 开发
   - deploy-vps.sh：VPS 部署
   - deploy-k8s.sh：K8s 部署

**核心理念：Dockerfile 是基础，docker-compose 和 K8s 是编排层，三者和谐共存，各司其职。**
