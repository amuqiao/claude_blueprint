# 项目开发方法主干

> **文档职责**：收口“项目开发到底按哪条主线推进”的方法层思考，作为后续正式方法论文档的母稿。
> **当前状态**：重点草稿。已经形成主干判断，但仍以方法论为主，不展开成完整操作清单。
> **后续可能去向**：`docs/项目开发主干流程.md`、`PLAYBOOK.md`

---

## 1. 这篇草稿要解决什么问题

真正的问题，不是“缺少某个命令”，而是：

- 你知道很多能力，但不知道当前该用哪一个
- 你知道模块开发大致有固定套路，但缺少一条能串起全局的主干
- 你知道每一步都重要，但不知道现在最该补哪一步
- 你知道某个节点出了问题会影响后面，但缺少一套可检查的标准

所以这篇草稿要回答的，不是某个具体项目怎么做，也不是某个 command 怎么写，而是：

**项目开发有没有一条稳定可复用的主干，以及这条主干为什么成立。**

---

## 2. 当前的核心判断

### 2.1 你真正需要的是主干，不是更多工具

`/init-architecture`、`/new-module`、`@arch`、`@rev` 这些都很重要，但它们本质上只是载体。

如果没有主干，你会遇到同一个问题：

- 知道工具在手里
- 但不知道现在该先用哪个
- 也不知道当前卡住的是“阶段判断”还是“具体实现”

所以真正的上位问题不是工具数量，而是：

**项目开发有没有一条稳定的主线，能把阶段、能力、文档和质量标准串起来。**

### 2.2 主干应该先于细节存在

做项目时，先有主干，再展开细节。

主干决定的是：

- 当前处在哪个节点
- 这一节点要解决什么问题
- 这一节点要产出什么结果
- 这一节点做到什么程度才适合继续

如果主干没有先立起来，就很容易出现：

- 一会儿在谈页面
- 一会儿在谈接口
- 一会儿又跳回基础设施
- 最后做了很多事，但主线仍然模糊

### 2.3 每个节点都应有输入、输出、质量标准

主干之所以成立，不是因为它列出了 8 个步骤，而是因为每个节点都应该具备可检查的结构：

- 输入是什么
- 输出是什么
- 什么叫做到位
- 什么叫还不能进入下一步

没有这些，节点就只是阶段名，不能形成真正可用的方法论。

### 2.4 节点之间存在依赖关系和回退关系

项目开发不是完全线性流水线，但也不是无序循环。

更真实的情况是：

- 前一节点质量不足，会直接拖累后一节点
- 后一节点暴露出问题时，往往需要回退到前面补强

所以这条主干不是“按部就班永不回头”，而是：

**有稳定顺序，也允许基于问题回退修正。**

---

## 3. 当前主干候选

当前最稳的主干，可以收敛成 8 个节点：

1. 项目定盘
2. 架构设计
3. 首个核心模块选择
4. 模块设计
5. 最小基础设施落地
6. 模块实现
7. 联调与验证
8. 回写与治理

这 8 个节点的价值，不在于“列出来就完了”，而在于它们把几个最重要的问题分开了：

- 什么属于项目级边界
- 什么属于模块级边界
- 什么属于基础设施投入
- 什么属于实现阶段
- 什么属于回写与治理

---

## 4. 为什么这条主干成立

### 4.1 它解决了什么混乱

它解决的是这种典型混乱：

- 一上来就在实现和设计之间反复横跳
- 还没选首个核心模块，就开始铺全套基础设施
- 模块设计没闭环，就直接进入代码
- 实现完成后，没有文档回写和规则沉淀

主干的意义不是让项目显得更正式，而是：

**让你知道当前真正卡在哪个层级。**

### 4.2 它为什么比“项目起盘说明”更上位

“项目起盘说明”回答的是：

- 新项目第一步做什么
- 老项目第一步做什么

而“项目开发方法主干”回答的是：

- 从项目定盘到模块设计，再到实现与治理，整条主线怎么走
- 为什么这些节点必须存在
- 节点之间为什么有依赖关系

所以前者更像一篇“如何开始”，后者更像“整个开发过程的骨架”。

### 4.3 它为什么适合跨项目复用

不同项目的业务内容不同，但这些上位结构通常是相似的：

- 都要先定边界
- 都要先把模块讲清楚
- 都要围绕当前核心模块补最小基础设施
- 都要在实现后做文档回写和规则沉淀

也就是说：

**项目会不同，但主干方法往往是稳定的。**

## 5. 这条主干如何挂接能力

### 5.1 command 挂在“阶段入口”

`command` 最适合承担：

- 进入某个阶段
- 启动一个明确工作流

例如：

- `/init-architecture`
- `/new-module`
- `/update-docs`
- `/add-rule`

它们不定义方法论，只负责把你带进某个阶段。

### 5.2 agent 挂在“判断节点”

`agent` 最适合承担：

- 架构判断
- 规范审查
- 边界确认

也就是说，主干上的某些节点需要判断，但不需要 agent 自己承担整条流程。

### 5.3 skill 挂在“任务规范”

`skill` 最适合承担：

- 这一类任务该怎么做
- 文档怎么写
- 分析怎么输出
- 某类脚本或运维任务怎么组织

它不负责决定“现在处在哪个节点”，只负责“这一节点里的任务怎么做”。

### 5.4 template 挂在“稳定起点”

`template` 最适合承担：

- 给项目或文档一个稳定骨架
- 避免每次从空白开始

它不替代方法，不替代流程，只提供起点。

---

## 6. 质量标准应该如何表达

### 6.1 节点级质量标准

每个节点都应该至少回答：

- 当前输出是否足以支撑下一节点
- 是否还存在关键边界不清
- 是否只是“写出来了”，还是“真的讲清楚了”

### 6.2 常见失败信号

常见失败信号通常有这些：

- 还没定边界就进入实现
- 还没选核心模块就铺基础设施
- 模块设计没闭环就开始写代码
- 实现完成后不回写文档
- 新约束反复出现却始终不沉淀

### 6.3 何时进入下一节点

标准不应该是“我已经做了一些内容”，而应该是：

**当前节点的产物已经足以支撑下一节点，而不是需要靠下一节点来倒逼补前置判断。**

这篇草稿现在更像“方法论母稿”，不直接替代正式文档。

它和正式文档的关系可以这样理解：

- `PLAYBOOK.md`
  负责给出开发范式与阶段原则

- `docs/项目开发主干流程.md`
  负责把节点结构写成可读、可用、可挂接能力的主干文档

而这篇草稿负责的是：

**解释为什么你真正需要的是“项目开发主干”，而不是继续往 command 或说明文档里堆东西。**
