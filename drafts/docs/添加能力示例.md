你现在问的是：**这条约定应该落到哪种机制里：`rule`、`skill` 还是 `command`。**

## 先给结论

如果你的意思是：

**“只要项目后端选 FastAPI，就统一用 `uv` 管理依赖、环境和运行命令。”**

那最合理的是：

1. **核心落点：`skill`**
2. **必要时再补一条短 `rule`**
3. **只有要做显式初始化流程时才用 `command`**

---

## 怎么判断

### 用 `rule` 的场景
`rule` 适合放：

- 短、硬、稳定的约束
- 不需要讲很多背景
- 更像“底线”或“默认约定”

比如：

- FastAPI 项目统一使用 `uv`
- 不使用 `pip install` 作为主流程
- 新增依赖统一走 `uv add`

如果你只想表达这类**简短工程约束**，`rule` 可以。

但问题是：  
你这条约定通常不只是 1 句话，它往往还包括：

- 目录怎么组织
- `uv` 具体怎么用
- 怎么启动开发环境
- 怎么加依赖 / 同步依赖
- 怎么跑脚本 / 测试
- FastAPI 项目的推荐分层

这些内容一多，`rule` 就不够了。

---

### 用 `skill` 的场景
`skill` 适合放：

- 一整类任务的方法
- 某个技术栈的完整工作方式
- 不只是“必须怎样”，还包括“具体怎么做”

所以如果你真正想要的是：

**“做 FastAPI 架构 / 初始化 / 开发时，默认采用一套 uv 驱动的工程方法。”**

那最适合的是 `skill`。

例如可以有：

- `skills/personal-os/references/fastapi-backend.md`
- 或按你的体系继续收编到 `personal-os` 下的其他主题 reference

里面写：

- 什么时候适用
- FastAPI 项目默认使用 `uv`
- 推荐目录结构
- 推荐分层
- 依赖管理方式
- 常用命令
- 禁止什么旧做法
- 和项目 `CLAUDE.md` / 架构文档怎么配合

**这是最符合你这套 blueprint 思路的。**

---

### 用 `command` 的场景
`command` 适合：

- 显式进入一个工作流阶段
- 有清晰输入输出
- 像“现在就执行一套流程”

比如：

- `/init-fastapi-backend`
- `/new-fastapi-service`

如果你希望的是：

**“我输入一个命令，就按 FastAPI + uv 的约定起一个后端骨架。”**

那才需要 `command`。

所以：
- `command` 负责“入口”
- `skill` 负责“方法”
- `rule` 负责“硬约束”

---

## 对你这个需求的最佳方案

我建议用**两层结构**：

### 第一层：`skill`
作为主承载。

比如：
- `skills/personal-os/references/fastapi-backend.md`

负责说明：
- FastAPI 项目为什么统一用 `uv`
- 怎么管理依赖、环境、运行
- 后端推荐分层
- 常用命令
- 架构设计时该考虑什么

### 第二层：短 `rule`
如果你想加一个硬约束，就再补一条轻量 `rule`。

比如：
- `rules/python-fastapi.md`

只写：
- FastAPI 项目统一使用 `uv`
- 不使用 `pip install` / `requirements.txt` 作为主流程

但如果现在你还不想把 `rules/` 扩大，我建议先不要加这层。

---

## 我不建议现在只做 `command`

因为你现在想解决的不是：

- “如何执行一次初始化”

而是：

- **“做 FastAPI 项目时应该采用什么工程方法”**

这明显更像 `skill`，不是 `command`。

---

## 一句话判断

### 如果你要的是：
- **方法和规范** → `skill`
- **硬性底线** → `rule`
- **显式启动流程** → `command`
