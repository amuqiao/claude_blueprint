# 独立全栈开发者 OS — 方案思路 v4

> **文档职责**：记录这套开发 OS 的设计思路、分层逻辑和关键决策，供自己回顾和对外解释用。
> **不是什么**：不是操作手册，不是文件内容清单，不是安装步骤。
> **对应实现**：https://github.com/amuqiao/claude_blueprint

---

## 一、核心问题是什么

用 Claude Code 开发全栈项目，遇到的不是"Claude 不够聪明"，而是三个工程问题：

- **规则越来越多，不知道放哪里**：发现新约束，塞进 CLAUDE.md，越来越长，Claude 遵守率越来越低。
- **设计文档和代码现实脱节**：文档写的是当时的决策，代码改了文档没跟上，AI 读到的是历史版本。
- **没有可复用的开发起点**：每个新项目从零配置，重复劳动，没有积累。

这三个问题的答案都不是"写更多规则"，而是**建立正确的分层**。

---

## 二、整体思路：三层分离

```
Layer 1  全局模具  ~/.claude/
         ↓ 跨所有项目复用，维护一次，所有项目受益
         判断标准：这条内容在下一个完全不同的项目里也成立吗？

Layer 2  项目壳    项目根/CLAUDE.md + .claude/
         ↓ 每个项目初始化一次，只填项目专属内容
         判断标准：这条内容只在本项目成立，换个项目就不一样？

Layer 3  业务内容  项目根/docs/
         随需求持续迭代，记录本项目的设计决策和知识沉淀
```

这个分层解决了"新约束不知道放哪里"的问题。每次发现新约束，先问属于哪一层，再放进去。不会再堆在一起。

## 三、全局模具的内部结构

全局模具里有四种东西，各司其职：

```
CLAUDE.md      始终加载，< 80 行
               只放：通用分层约束 + skill 指针 + 写作原则
               不放：具体路径、具体工具名、项目技术栈

skills/        按需加载，闲置时仅占 ~100 token（只加载 description）
               每个 skill = 一份专业手册，Claude 判断何时读取
               这是规范迭代的主战场，CLAUDE.md 不动，改 skill

agents/        只放 2 个（arch + rev）
               选择标准：任务量大、上下文脏、结果独立、不需要多轮沟通

commands/      用户主动触发的工作流（/new-module /update-docs /add-rule）
               底层机制和 skill 相同，区别是触发方式

hooks/         100% 执行的强制机制，不依赖 Claude 遵守
               CLAUDE.md 管引导（~80% 遵守），hooks 管强制（100% 执行）

templates/     手动 cp 用的起点文件，不是自动机制
```

---

## 四、为什么只有 2 个 agent

最初设计了 5 个角色：arch 谋士、coder 匠人、rev 御史、doc 史官、ops 值守。

最后只保留 2 个，是刻意收敛。判断逻辑：

```
arch 谋士  → 读大量文档做分析，上下文容易变脏，结果独立  → 保留为 subagent
rev 御史   → 读大量代码做检查，上下文容易变脏，结果独立  → 保留为 subagent

doc 史官   → 需要频繁和我确认内容细节，多轮沟通          → 改为主对话 + design-doc skill
coder 匠人 → 强依赖实时反馈，没设计文档不该动代码        → 直接主对话
ops 值守   → 诊断命令需要我确认后才执行                  → 改为主对话 + python-ops-cli skill
```

核心判断：拆 agent 解决的是"上下文太脏"，不是"规则太多"。多数任务用 skill 就够了。subagent token 消耗是主对话的 4-7 倍，独立开发者的任务是顺序的，不是并行的，subagent 的并行优势用不上。

## 五、为什么 hooks 必须有

纯靠 CLAUDE.md 的约束是软约束，遵守率约 80%。有些事不能有 20% 的失败率：

| hook | 触发时机 | 解决什么问题 |
|------|---------|------------|
| protect-config.sh | 修改配置文件前 | Claude 会通过禁用 lint 规则来"修复"报错 |
| block-git-push.sh | 执行 git push 前 | 自动 push 可能触发 CI/CD 污染队友分支 |
| notify-file-changed.sh | 修改关键文件后 | 提醒检查相关文档和设置是否还对得上 |

设计原则：**需要 100% 执行的约束放 hook，需要表达判断原则的约束放 CLAUDE.md，需要大量细节的任务规范放 skill。**

## 六、新约束的处置决策树

发现任何新约束，三秒判断放哪里：

```
新约束出现
│
├─ 改变 AI 行为？
│   ├─ 下个完全不同的项目也成立？
│   │   ├─ 超过 3 行 → 放对应 skill，CLAUDE.md 加一行指针
│   │   └─ 3 行以内 → 直接放 ~/.claude/CLAUDE.md
│   └─ 只在本项目成立 → 放项目 CLAUDE.md 或 .claude/rules/
│
└─ 记录项目决策？
    ├─ 产品/架构决策 → 放 docs/design/ 对应文档
    └─ 过程经验/踩坑 → 放 docs/专题/（只增不改）
```

这个决策树也实现为 `/add-rule` command，发现新约束时直接运行，引导完成归档。

## 七、模具的自维护机制

模具本身也是一个需要维护的项目。有三个机制保证它不失控：

**os-maintenance skill**：记录三层定义、决策树、迭代流程。需要修改模具时读它，不用凭记忆判断。

**WHY.md**：记录每个设计决策的理由。几个月后自己都会忘为什么这样设计，这份文档是答案。

**CLAUDE.md 行数上限 80 行**：超了说明有细节没下沉到 skill，定期用 `wc -l` 检查。

**drafts/ 目录**：思考草稿箱，无脑扔进去，不用想放哪里，攒多了再整理提炼。

## 八、对标参考

同类仓库：`faizkhairi/claude-code-blueprint`

两者的定位差异：

| 维度 | faizkhairi/claude-code-blueprint | 本方案 |
|------|----------------------------------|--------|
| 目标用户 | 所有 Claude Code 用户 | 独立全栈开发者 |
| 核心价值 | 行为约束和防护机制 | 工程流程和规范体系 |
| skills 内容 | 通用工作流（git-commit/create-pr） | 生产验证的专项规范（脚本/运维/文档） |
| agents 数量 | 11 个 | 2 个（刻意收敛） |
| hooks 数量 | 10 个 | 3 个（够用即止） |
| CLAUDE.md 行数 | 144 行（偏长） | 67 行（控制严格） |

值得借鉴的部分：hooks 的 WHY 文档模式（每个 hook 追溯到一次真实事故）、model tiering 思路（任务复杂度决定模型，不是重要程度）、`permissionMode: plan` 用于只读 agent。
