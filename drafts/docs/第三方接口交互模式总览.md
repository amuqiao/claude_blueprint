# 第三方接口交互模式

理解这个话题需要区分两个维度，混淆它们是很多文档的通病：

- **维度一：接口设计模式** — 后端如何组织处理逻辑
- **维度二：结果获取策略** — 客户端如何拿到结果

两者是正交的。先选接口模式，再选取结果策略。

---

## 维度一：接口设计模式

### 1. 同步请求-响应

一次 HTTP 请求，服务端在**连接不断开的情况下**处理完毕直接返回。

```
Client ──── POST /translate ────► Server [处理中]
Client ◄─── 200 OK + 结果 ──────  Server
```

适合处理时间短、结果体小的场景。接入成本最低，但受网关超时限制，不适合耗时操作。

---

### 2. 异步任务（Asynchronous Job API）

提交与执行解耦。客户端先提交任务拿到 `job_id`，服务端后台处理，客户端**事后**再取结果。

```
Client ──── POST /jobs ──────────► Server
Client ◄─── 202 Accepted + job_id  Server

[后台处理中...]

Client  取结果（见维度二）
```

适合耗时任务、大批量处理。代价是调用链变长，接入方需要处理任务状态。

> **注意**：轮询、长轮询、Webhook 都不是独立的"接口模式"，它们是异步任务下取结果的不同手段，属于维度二。

---

### 3. 流式响应（Streaming Response）

一次请求建立后，服务端**边处理边分块返回**，不等全部完成。

```
Client ──── POST /chat ──────────► Server
Client ◄─── chunk 1 ─────────────  Server
Client ◄─── chunk 2 ─────────────  Server
Client ◄─── chunk 3 (done) ──────  Server
```

常见于 AI 生成内容、日志流。首字节时间短，用户感知好。接入方需要处理增量数据。

---

### 4. 离线批处理（Batch Processing）

上传文件或提交批次，服务端**离线处理**，完成后提供结果文件下载。

```
Client ──── 上传文件 / 提交批次 ──► Server
                                    [离线处理，可能数小时]
Client ──── 下载结果文件 ──────────► Server
```

适合超大规模数据，交互周期长，不适合实时业务。

---

## 维度二：结果获取策略

> 以下策略主要配合**异步任务**使用，也可用于其他需要持续通信的场景。

### 1. 轮询（Polling）

客户端**定期主动查询**任务状态，直到完成。

```
Client ──── GET /jobs/{id} ──────► Server → "processing"
Client ──── GET /jobs/{id} ──────► Server → "processing"
Client ──── GET /jobs/{id} ──────► Server → "done" + 结果
```

实现最简单，兼容性最好。缺点是存在无效请求，实时性一般。

---

### 2. 长轮询（Long Polling）

客户端发起查询，服务端**挂起连接不立即返回**，等到状态变化或超时才响应。

```
Client ──── GET /jobs/{id} ──────► Server
                                   Server [挂起，等待变化...]
Client ◄─── 200 OK + 结果 ──────   Server（状态变化时才返回）
```

本质仍是 HTTP Pull，减少了空轮询，实时性更好。代价是服务端连接占用更高。

---

### 3. Webhook

客户端提交任务时带上 `callback_url`，服务端完成后**主动 POST 到该地址**。

```
Client ──── POST /jobs (带 callback_url) ──► Server
                                             [后台处理...]
Client ◄─── POST {callback_url} + 结果 ────  Server（完成时主动推送）
```

时效性好，客户端无需频繁查询。要求对方必须提供公网可达地址，还需处理签名校验、重试、幂等。

---

### 4. SSE（Server-Sent Events）

客户端建立一个 HTTP 长连接，服务端**持续单向推送**事件流。

```
Client ──── GET /stream ─────────► Server
Client ◄─── event: progress ─────  Server
Client ◄─── event: progress ─────  Server
Client ◄─── event: done + 结果 ──  Server
```

比轮询更实时，协议比 WebSocket 简单。更适合前端场景，不是最典型的第三方开放平台方案。

---

### 5. WebSocket

客户端与服务端建立**持久双向连接**，双方均可主动发送消息。

```
Client ◄──────────── WebSocket ────────────► Server
         （全双工，任意一方可随时主动发送）
```

实时性最好。鉴权、连接管理、运维复杂度高，不适合大多数普通业务 API。

---

## 两个维度的组合方式

```
接口设计模式          结果获取策略
──────────────        ────────────────────────────────
同步请求-响应    →    不需要（结果直接返回）

异步任务         →    轮询
                 →    长轮询
                 →    Webhook
                 →    Webhook + 轮询兜底（推荐组合）

流式响应         →    不需要（连接内持续返回）

离线批处理       →    轮询查状态 + 下载结果
```

## 开放平台选型建议

对于翻译类 API，推荐分阶段演进：

**阶段一：优先落地**

| 场景 | 接口模式 | 取结果策略 |
|------|----------|------------|
| 小批量、低延迟 | 同步请求-响应 | — |
| 大批量、耗时任务 | 异步任务 | 轮询 |

**阶段二：平台成熟后补充**

| 场景 | 接口模式 | 取结果策略 |
|------|----------|------------|
| 高频调用、对实时性有要求 | 异步任务 | Webhook + 轮询兜底 |

## 术语速查

| 术语 | 所属维度 | 说明 |
|------|----------|------|
| 同步请求-响应 | 接口设计模式 | 一次请求内直接返回结果 |
| 异步任务 API | 接口设计模式 | 提交与执行解耦，返回 job_id |
| 流式响应 | 接口设计模式 | 一次请求内分块持续返回 |
| 离线批处理 | 接口设计模式 | 提交批次，后台处理，下载结果 |
| 轮询 | 结果获取策略 | 客户端定期主动查询 |
| 长轮询 | 结果获取策略 | 服务端挂起连接，有变化才返回 |
| Webhook | 结果获取策略 | 服务端完成后主动推送 |
| SSE | 结果获取策略 | HTTP 长连接，服务端单向持续推送 |
| WebSocket | 结果获取策略 | 持久双向连接，全双工通信 |
