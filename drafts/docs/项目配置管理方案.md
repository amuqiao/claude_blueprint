# FastAPI 项目配置管理方案

> **文档职责**：定义 FastAPI 项目在本地开发、开发服务器、生产环境的配置管理规范
> **适用场景**：初始化 FastAPI 项目配置体系；团队成员需要了解配置规范；运维需要了解如何部署
> **目标读者**：开发者、运维人员
> **维护规范**：配置文件规划变更时更新 §2；环境切换命令变更时更新 §3；新增常见问题时更新 §5

---

## 一、方案总览

采用 **APP_ENV 方案**：通过 `APP_ENV` 环境变量区分环境，本地直接运行读 `.env`，开发服务器通过 `APP_ENV=dev` 读 `.env.dev`，生产环境由 K8s 直接注入环境变量。

**核心原则：**
- 本地零配置：直接运行默认读 `.env`，不需要设置任何环境变量
- 开发服务器一个变量切换：`APP_ENV=dev` 即可
- 生产环境与仓库解耦：运维在 K8s 侧管理配置，不依赖仓库里的 `.env` 文件
- 密码分离：本地 `.env` 不提交（包含真实密码），`.env.dev` 提交但必须脱敏

---

## 二、配置文件规划

| 文件 | 是否提交 Git | 用途 | 读取方式 |
|---|---|---|---|
| `.env` | ❌ 不提交 | 本地开发真实配置（包含真实密码） | pydantic-settings 默认读取 |
| `.env.dev` | ✅ 提交（脱敏） | 开发服务器配置 | docker-compose 通过 `APP_ENV=dev` 读取 |
| `.env.example` | ✅ 提交 | 配置模板，列出所有配置项和注释 | 开发者初始化时复制，运维参考 |
| `.env.prod` | ❌ 不提交 | （不使用此文件） | K8s 直接注入环境变量 |

### .gitignore 配置

```gitignore
.env
.env.prod
.env.*.override
```

### .env.example 模板规范

```bash
# ==================== 应用配置 ====================
APP_ENV=prod
API_PORT=8000
FLOWER_PORT=5555

# ==================== 数据库 ====================
DATABASE_URL=postgresql+asyncpg://user:password@host:5432/dbname

# ==================== Redis ====================
REDIS_URL=redis://:password@host:6379/0

# ==================== Celery ====================
CELERY_BROKER_URL=redis://:password@host:6379/1
CELERY_RESULT_BACKEND=redis://:password@host:6379/2

# ==================== 安全 ====================
SECRET_KEY=your-secret-key-here
```

## 三、技术实现

### 3.1 docker-compose.yml 配置

```yaml
services:
  app:
    env_file:
      - .env.${APP_ENV:-.env}
    # 其他配置...
```

**逻辑说明：**
- 未设置 `APP_ENV` 时，默认读 `.env`（兼容本地 docker-compose 启动）
- 设置 `APP_ENV=dev` 时，读 `.env.dev`
- `${APP_ENV:-.env}` 是 shell 语法，表示如果 `APP_ENV` 未设置则使用 `.env`

### 3.2 pydantic-settings 配置加载

应用启动时，pydantic-settings 按以下优先级读取配置：

1. **环境变量**（最高优先级）
2. `.env` 文件
3. 代码中的默认值

因此：
- 本地直接运行时，读 `.env` 文件
- docker-compose 启动时，docker-compose 先将文件内容注入为环境变量，应用从环境变量读取
- K8s 部署时，K8s 将 ConfigMap/Secret 注入为环境变量，应用从环境变量读取

## 四、不同环境的使用方式

### 4.1 本地开发环境

**初始化配置（新项目或新成员加入）：**

```bash
# 1. 复制模板
cp .env.example .env

# 2. 填入本地真实配置
vim .env  # 填入本地数据库密码、Redis 密码等
```

**日常启动：**

```bash
# 方式 1：直接运行（推荐，方便调试）
uvicorn main:app --reload

# 方式 2：用 docker-compose（如果需要完整容器环境）
docker-compose up -d
```

**维护规则：**
- `.env` 包含真实密码，绝对不提交 Git
- 新增配置项时，必须同步更新 `.env.example`

### 4.2 开发服务器环境

**初始化配置（首次部署）：**

```bash
# 1. 在仓库创建 .env.dev（脱敏版本）
cp .env.example .env.dev

# 2. 填入开发服务器的配置
vim .env.dev  # 连接开发服务器的 DB / Redis，使用脱敏密码

# 3. 提交到 Git
git add .env.dev
git commit -m "新增开发服务器配置"
```

**日常启动：**

```bash
# SSH 到开发服务器后
APP_ENV=dev docker-compose up -d

# 查看日志
APP_ENV=dev docker-compose logs -f

# 重启服务
APP_ENV=dev docker-compose restart
```

**维护规则：**
- `.env.dev` 提交 Git，但必须脱敏（不能包含生产级密码）
- 配置变更后，提交 Git 并重启服务

### 4.3 生产环境（K8s）

**运维侧初始化流程：**

```bash
# 1. 拉取仓库
git clone <repo-url>

# 2. 阅读配置模板
cat .env.example

# 3. 在 K8s 创建 ConfigMap（非敏感配置）
kubectl create configmap app-config \
  --from-literal=APP_ENV=prod \
  --from-literal=API_PORT=8000 \
  --from-literal=FLOWER_PORT=5555

# 4. 在 K8s 创建 Secret（敏感配置）
kubectl create secret generic app-secret \
  --from-literal=DATABASE_URL=postgresql+asyncpg://user:password@prod-db:5432/prod \
  --from-literal=REDIS_URL=redis://:prod-password@prod-redis:6379/0 \
  --from-literal=SECRET_KEY=prod-secret-key-here

# 5. 部署应用
kubectl apply -f deployment.yaml
```

**deployment.yaml 示例：**

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: fastapi-app
spec:
  template:
    spec:
      containers:
      - name: app
        image: your-image:latest
        envFrom:
        - configMapRef:
            name: app-config
        - secretRef:
            name: app-secret
```

**维护规则：**
- 配置全部在 K8s 侧管理，不在仓库里
- 配置变更后，需要滚动重启：`kubectl rollout restart deployment/fastapi-app`

## 五、配置变更协作流程

### 开发者新增配置项

1. 在 `.env.example` 添加新配置项（带注释说明用途）
2. 更新 `.env.dev`（填入脱敏值）
3. 更新本地 `.env`（填入本地测试值）
4. 提交 PR，在 PR 描述中告知运维：

```markdown
## 配置变更

新增配置项：
- `NEW_CONFIG_ITEM`：[说明用途]

**运维需要操作：**
在 K8s 侧添加该环境变量到 ConfigMap 或 Secret。
```

### 运维同步配置

```bash
# 更新 ConfigMap（非敏感配置）
kubectl edit configmap app-config

# 或更新 Secret（敏感配置）
kubectl edit secret app-secret

# 滚动重启（让新配置生效）
kubectl rollout restart deployment/fastapi-app
```

## 六、环境对比总览

| 维度 | 本地开发 | 开发服务器 | 生产 K8s |
|---|---|---|---|
| 运行方式 | 直接运行 uvicorn | docker-compose | K8s Deployment |
| 配置文件 | `.env` | `.env.dev` | 不使用文件 |
| 提交 Git | ❌ | ✅（脱敏） | ❌ |
| 启动命令 | `uvicorn main:app --reload` | `APP_ENV=dev docker-compose up -d` | `kubectl apply -f deployment.yaml` |
| 配置维护者 | 开发者本地 | 开发者（仓库） | 运维（K8s 侧） |
| 密码安全 | 本地文件，不提交 | 脱敏值，可提交 | K8s Secret 加密存储 |

## 七、常见问题

### Q1：为什么本地用 `.env` 而不是 `.env.local`？

**A：** 本地直接运行时，pydantic-settings 默认读 `.env`，零配置。如果用 `.env.local`，需要额外设置 `APP_ENV=local` 或修改代码，增加复杂度。`.env` 不提交 Git，天然保护本地真实密码。

### Q2：`.env.dev` 可以包含真实密码吗？

**A：** 不可以。`.env.dev` 提交到 Git，所有团队成员可见。必须使用脱敏密码（如 `dev_password_123`），生产级密码只能在 K8s Secret 中维护。

### Q3：运维需要提供 `.env.prod` 吗？

**A：** 不需要。K8s 部署时，配置全部通过 ConfigMap / Secret 注入为环境变量，不需要任何 `.env` 文件。运维只需要参考 `.env.example` 了解需要配置哪些项。

### Q4：如果本地也想用 docker-compose 怎么办？

**A：** 直接运行 `docker-compose up -d`（不设置 `APP_ENV`），会读取 `.env` 文件。docker-compose.yml 中配置的 `${APP_ENV:-.env}` 在未设置 `APP_ENV` 时默认读 `.env`。

### Q5：如何快速验证配置是否生效？

**A：** 可以在应用启动时打印关键配置项（注意脱敏）：

```python
from config import settings

logger.info(f"APP_ENV: {settings.APP_ENV}")
logger.info(f"DATABASE_URL: {settings.DATABASE_URL[:20]}...")  # 只打印前20个字符
logger.info(f"API_PORT: {settings.API_PORT}")
```

### Q6：为什么不用 `ENV_FILE` 方案？

**A：** `ENV_FILE` 方案的变量名只表达"文件路径"，语义不如 `APP_ENV`（表达"环境"）清晰。新人看到 `APP_ENV=dev` 立刻明白是开发环境，看到 `ENV_FILE=.env.dev` 需要理解映射关系。

## 八、交付给运维的材料

提供两样东西：

1. **GitLab 仓库地址**（含 Dockerfile）
2. **配置说明**：

> 配置模板在仓库根目录 `.env.example`，按模板在 K8s 侧创建 ConfigMap（非敏感配置）和 Secret（敏感配置），以环境变量方式注入容器。应用启动时 pydantic-settings 会自动从环境变量读取配置，无需提供任何 `.env` 文件。
>
> **注意事项：**
> - `DATABASE_URL`、`REDIS_URL`、`SECRET_KEY` 等敏感配置必须放在 Secret 中
> - `APP_ENV`、`API_PORT` 等非敏感配置放在 ConfigMap 中
> - 配置变更后需要滚动重启：`kubectl rollout restart deployment/fastapi-app`
