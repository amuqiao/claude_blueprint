## 标注系统定义

在 Claude Code 目录结构参考文档中，我们使用统一的三级标注系统来标识目录/文件的来源和状态：

### 【官方标准】

**含义**: 官方文档明确定义的核心目录/文件，是标准结构的一部分

**特征**:
- ✅ 官方文档中有明确描述
- ✅ 是 Claude Code 核心功能的一部分
- ✅ 应该存在于标准部署中
- ✅ 用户应该了解和使用

**适用于**:
- `CLAUDE.md` - 全局/项目指令文件
- `settings.json` - 配置文件
- `commands/` - 自定义命令目录
- `skills/` - 可复用工作流目录
- `agents/` - 子智能体定义目录
- `plugins/` - 插件管理目录
- `projects/` - 项目运行时数据目录
- `.mcp.json` - MCP 服务器配置
- `.worktreeinclude` - Worktree 配置

### 【官方支持】

**含义**: 官方文档提及支持的功能，按需手动创建或使用时自动创建

**特征**:
- ✅ 官方文档中有提及
- ⚠️ 不是默认创建的
- ⚠️ 需要用户主动创建或使用特定功能时才会出现
- ✅ 是可选但推荐的功能

**适用于**:
- `keybindings.json` - 自定义键盘快捷键（需手动创建）
- `rules/` - 模块化指令目录（按需创建）
- `output-styles/` - 自定义输出风格（按需创建）
- `agent-memory/` - 子智能体记忆（使用 agents 时自动创建）

**何时创建**:
- **手动创建**: keybindings.json, rules/, output-styles/
- **自动创建**: agent-memory/（使用 agents 功能时）

### 【自动生成】

**含义**: Claude Code 运行时自动创建，无需用户手动管理

**特征**:
- ⚠️ 不需要用户创建或编辑
- ⚠️ 由 Claude Code 内部管理
- ✅ 可能在使用过程中自动出现
- ❌ 通常不应手动修改

**适用于**:
- `plugins/installed_plugins.json` - 插件安装记录
- `plugins/known_marketplaces.json` - 插件市场记录
- `plugins/cache/` - 插件缓存
- `projects/<hash>/<session-id>.jsonl` - 会话记录
- `projects/<hash>/<session-id>/` - 会话数据
- 运行时目录: downloads/, file-history/, history.jsonl, ide/, paste-cache/, plans/, session-env/, sessions/, shell-snapshots/, tasks/, telemetry/ 等

**用户操作**:
- ✅ 可以查看了解
- ✅ 可以在 .gitignore 中忽略
- ❌ 不建议手动编辑
- ❌ 不建议删除（除非了解后果）

## 标注应用示例

### 系统级 ~/.claude/ 目录

```
~/.claude/
├── CLAUDE.md              # 【官方标准】
├── settings.json          # 【官方标准】
├── keybindings.json       # 【官方支持】需手动创建
│
├── rules/                 # 【官方支持】按需创建
├── commands/              # 【官方标准】
├── skills/                # 【官方标准】
├── agents/                # 【官方标准】
├── agent-memory/          # 【官方支持】使用时自动创建
├── output-styles/         # 【官方支持】按需创建
│
├── plugins/               # 【官方标准】
│   ├── installed_plugins.json  # 【自动生成】
│   ├── known_marketplaces.json # 【自动生成】
│   └── cache/             # 【自动生成】
│
├── projects/              # 【官方标准】
│   └── <hash>/
│       ├── <session>.jsonl     # 【自动生成】
│       └── <session>/          # 【自动生成】
│
├── downloads/             # 【自动生成】
├── history.jsonl          # 【自动生成】
└── sessions/              # 【自动生成】
```

### 项目级 .claude/ 目录

```
your-project/
├── CLAUDE.md              # 【官方标准】
├── CLAUDE.local.md        # 【官方标准】（可选，需手动创建）
├── .mcp.json              # 【官方标准】
├── .worktreeinclude       # 【官方标准】
│
└── .claude/
    ├── settings.json          # 【官方标准】
    ├── settings.local.json    # 【官方标准】（自动 gitignored）
    │
    ├── rules/                 # 【官方支持】按需创建
    ├── commands/              # 【官方标准】
    ├── skills/                # 【官方标准】
    ├── agents/                # 【官方标准】
    ├── agent-memory/          # 【官方支持】使用时自动创建
    └── output-styles/         # 【官方支持】按需创建
```

## 标注变更历史

### v1.0 (2026-05-12)
**初始版本 - 统一标注系统**

**问题**: 之前使用了不一致的标注
- 【官方确认】
- 【官方文档确认】
- 【官方支持，按需创建】
- 【官方支持，需手动创建】
- 【官方支持，使用时自动创建】
- 【运行时生成】
- 【安装插件后生成】

**解决**: 统一为三级标注系统
- **【官方标准】** - 核心结构
- **【官方支持】** - 按需创建
- **【自动生成】** - 运行时创建

**影响范围**:
- `Claude Code 目录结构完整参考(文字版).md`
- `Claude Code 目录结构完整参考(可视化版).html`

## 使用建议

### 对于用户

**【官方标准】目录/文件**:
- 应该了解其作用
- 应该按官方文档正确使用
- 可以在蓝图项目中提供模板或示例

**【官方支持】目录/文件**:
- 了解其功能和触发条件
- 需要时再创建（不强制）
- 可以在文档中提供创建示例

**【自动生成】目录/文件**:
- 无需关注内部结构
- 在 .gitignore 中正确忽略
- 不要手动编辑或删除

### 对于蓝图维护者

**部署到 ~/.claude/ 时应该包含**:
- 所有【官方标准】目录/文件（提供模板或示例）
- 可选：部分【官方支持】目录/文件（作为使用示例）
- 不包含：【自动生成】目录/文件（运行时自动创建）

**在 deploy-manifest.txt 中应该列出**:
```
# 官方标准
CLAUDE.md
settings.json
rules/
commands/
skills/
agents/

# 可选：官方支持（作为示例）
# keybindings.json
# output-styles/

# 不包含自动生成的内容
```

## 常见问题

### Q1: 为什么 keybindings.json 是【官方支持】而不是【官方标准】？

A: 虽然官方文档提及了这个功能，但：
1. 默认情况下不存在这个文件
2. 需要用户有自定义快捷键需求时才创建
3. 不是 Claude Code 核心功能必需的

因此标注为【官方支持】更准确。

### Q2: rules/ 目录是【官方支持】，我应该创建吗？

A: 取决于你的需求：
- **如果** CLAUDE.md 超过 200 行，或者需要按路径生效的规则 → 建议创建
- **如果** CLAUDE.md 简洁清晰，没有模块化需求 → 暂不创建

本蓝图项目提供了 rules/collaboration.md 作为使用示例。

### Q3: 【自动生成】的目录可以删除吗？

A: 技术上可以，但：
- ✅ projects/, cache/, sessions/ 等可以安全删除（会重新生成）
- ⚠️ history.jsonl 包含会话历史，删除会丢失历史记录
- ⚠️ plugins/ 相关文件删除可能需要重新配置插件

建议：不要删除，正确配置 .gitignore 忽略即可。

### Q4: 如何判断一个新功能属于哪个标注级别？

判断流程：
1. **官方文档有明确定义 + 默认存在** → 【官方标准】
2. **官方文档有提及 + 需要手动创建或按需使用** → 【官方支持】
3. **运行时自动创建 + 不需要用户管理** → 【自动生成】
