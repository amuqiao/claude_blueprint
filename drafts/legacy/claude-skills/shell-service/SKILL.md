# Shell 服务管理规范

> **文档职责**：定义跨项目可复用的 Shell 服务生命周期管理脚本标准，覆盖脚本结构、服务定义、命令契约、进程控制、日志与输出规范；不含技术栈特定的启动命令、端口号或路径。
> **适用场景**：新建服务管理脚本前；向 Claude 描述脚本需求前；为现有脚本新增服务或命令时评审合规性。
> **目标读者**：全栈项目开发者及 Claude CLI；具备 Bash 基础，了解进程信号与 PID 管理。
> **维护规范**：通用约束变化时更新本文档；项目特有的启动命令、端口号、路径等在各项目脚本的配置函数中定义，不写入本规范。

---

## 0. 设计原则

| 原则 | 含义 |
|---|---|
| 单一入口 | 一个项目只有一个服务管理脚本，所有服务通过它统一操作 |
| 配置与逻辑分离 | 启动命令、端口、路径在配置函数中声明；核心操作函数不内联任何服务信息 |
| 命令契约稳定 | 五个标准命令的接口与行为不因新增服务而改变 |
| 失败即报错 | 端口冲突、启动崩溃、未知参数必须立即报错，不静默失败 |

---

## 1. 脚本文件结构

模块内节顺序固定，不可调换：

1. Shebang + 模式设置（`set -euo pipefail`）
2. Header 注释块（见 §2）
3. 路径常量（`SCRIPT_DIR`、`ROOT`、各服务日志根目录）
4. 服务列表常量（`SERVICES` 数组，定义合法服务名与 `run_for` 的遍历顺序）
5. 颜色常量（终端检测后赋值，非终端时赋空字符串）
6. 服务配置函数（`_log_file`、`_pid_file`、`_port`、`_urls`）
7. 辅助函数（`is_running`、`_check_port`）
8. 核心操作函数（`do_start`、`do_stop`、`do_restart`、`do_status`、`do_logs`）
9. 派发函数（`usage`、`validate_service`、`run_for`）
10. 主入口（参数解析 + `case` 派发）

---

## 2. Header 注释规范

Header 注释块位于 Shebang 行之后，各节顺序固定，不可省略：

| 节 | 内容 |
|---|---|
| 脚本名 + 一句话说明 | 标题行，说明脚本管理什么 |
| 管理的服务清单 | 每行一个服务：名称、技术栈、端口（无端口则注明"无端口"） |
| 用法 | 调用格式，含 `[service]` 可选标注 |
| 命令 | 五个标准命令，每条标注不传 service 时的默认行为 |
| 典型工作流 | 3–5 条常用命令示例（行尾注释说明用途） |
| 日志与 PID 文件位置 | 每个服务的 `.log` 和 `.pid` 文件路径 |
| 边界说明 | 不管理什么（如数据库）、日志策略、启动顺序策略 |

---

## 3. 服务定义规范

每个服务通过四个配置函数描述，函数内用 `case` 语句按服务名分支：

| 函数 | 返回值 | 约束 |
|---|---|---|
| `_log_file <svc>` | 日志文件绝对路径 | 必须实现；与 `_pid_file` 同目录 |
| `_pid_file <svc>` | PID 文件绝对路径 | 必须实现；与 `_log_file` 同目录 |
| `_port <svc>` | 端口号字符串 | 必须实现；无端口服务显式返回空字符串 |
| `_urls <svc>` | 单行字符串，条目间用双空格分隔；每条目可加 `label: url` 前缀标签 | 必须实现；无 URL 返回空字符串 |

**规则：**
- 四个函数必须覆盖 `SERVICES` 数组中的全部服务名，缺失分支视为配置错误
- 无端口服务不可省略 `_port` 的 `case` 分支，必须显式返回空字符串；`_check_port` 收到空字符串时直接返回 0，不做检测

**新增服务时须修改以下四处，且仅这四处：**
1. `SERVICES` 数组：追加服务名
2. `_log_file`：追加 `case` 分支
3. `_pid_file`：追加 `case` 分支
4. `_port` / `_urls`：追加 `case` 分支

## 4. 命令接口契约

脚本必须实现以下五个命令，行为契约不可更改：

| 命令 | service 参数 | 不传 service 时 | 禁止行为 |
|---|---|---|---|
| `start` | 可选 | 启动全部服务 | 静默忽略启动失败 |
| `stop` | 可选 | 停止全部服务 | 跳过 SIGTERM 直接 SIGKILL |
| `restart` | 可选 | 重启全部服务 | — |
| `status` | 可选 | 显示全部服务状态 | running 时不显示 URL |
| `logs` | 必填 | 报错退出 | 接受多个服务或空值 |

传入未知 service 名时，脚本必须报错并打印 usage，不可静默忽略。

## 5. 进程控制规范

### PID 管理

- 每个服务对应一个 PID 文件，路径由 `_pid_file` 返回
- `is_running <svc>`：PID 文件存在且 `kill -0 <pid>` 成功则返回 0，否则返回 1
- 检测到 stale PID（文件存在但进程不存在）时，`do_status` 和 `do_stop` 均自动删除该文件

### 启动（`do_start`）

后台启动固定格式：

```bash
(cd <workdir> && exec <cmd>) >> <log> 2>&1 &
```

`exec` 替换子 shell，保证 `$!` 准确指向实际进程 PID。

启动成功路径输出顺序：
1. 端口冲突（见 §6）→ 打印错误 + kill 命令，跳过该服务，继续其他服务
2. 打印日志文件路径（蓝色）
3. 等待存活检测通过后，调用 `do_status` 显示运行状态（不自行拼接 running 行）

**存活检测**：启动后等待 **1.5 秒**（推荐默认值，可按项目调整），再检查 PID 是否存活；进程已退出则删除 PID 文件并以红色打印错误，提示用户执行 `logs <svc>` 查看原因。

### 停止（`do_stop`）

- 先发 SIGTERM，每 **0.3 秒**轮询，等待最多 **6 秒**（均为推荐默认值，可按项目调整）
- 超时后发 SIGKILL，输出标注 `force killed`
- 无论以何种方式退出，均删除 PID 文件

---

## 6. 端口管理规范

- `do_start` 启动前必须调用 `_check_port`；`_port` 返回空字符串时跳过检测
- 使用 `lsof -ti :<port>` 获取全部占用该端口的 PID，不只取第一个
- 端口冲突时跳过该服务，继续处理其他服务（不中断 `start` 流程）
- 错误输出格式：

```text
[<svc>   ] Error: port <N> in use
         run: kill <pid1> <pid2>
```

---

## 7. 日志规范

- 日志文件追加写入（`>>`），不覆盖、不轮转
- 每次 `do_start` 在日志文件写入时间戳分隔线：`=== YYYY-MM-DD HH:MM:SS start ===`
- `do_logs` 执行前打印日志文件路径与退出提示，再执行 `tail -f`

---

## 8. 输出规范

### 颜色

`[[ -t 1 ]]` 为真时启用颜色；否则所有颜色变量赋空字符串，输出纯文本。

| 语义 | 颜色 |
|---|---|
| 成功 / running | 绿色 |
| 警告 / already running / not running | 黄色 |
| 错误 / stopped / force killed | 红色 |
| 信息 / 日志路径 | 蓝色 |

### 格式

- 服务名字段宽度取 `SERVICES` 数组中最长服务名的字符数，保证多服务输出列对齐
- `do_status` running 行格式：`[<svc>   ] running  (PID <N>)  <urls>`
- 有 URL 的服务在 running 状态下必须显示 URL；stopped / dead 状态不显示

### 错误消息

```text
[<svc>   ] Error: <描述>
           <操作提示>（可选，缩进与首行文字对齐）
```
