# Blueprint Governance

## 核心口径

维护个人开发 OS 时，默认先判断：

1. 这是在改变 AI 行为方式，还是在记录项目事实
2. 这是跨项目稳定成立，还是只在当前项目成立
3. 它应该进入真源层，还是只保留在使用层

## 三层理解

| 层 | 位置 | 判断标准 |
|---|---|---|
| 真源层 | `skills/`、正式方法文档 | 需要长期维护、跨任务复用、后续其他入口都应以它为准 |
| 使用层 | `drafts/prompts/` | 日常直接调用、任务启动器、派生入口 |
| 事实层 | `docs/`、项目设计文档 | 记录当前项目事实、业务决策、实现状态 |

## 什么时候进真源层

满足以下任一条件时，优先考虑进 `skills/`：

- 这是跨项目稳定成立的规则或判断框架
- 这是你准备长期维护的方法论
- 后续会反复从它派生 prompt 或文档
- 规则已经不适合继续散落在多个 prompt 中

## 什么时候留在使用层

满足以下任一条件时，继续留在 `drafts/prompts/`：

- 它只是任务启动器
- 它主要服务日常复制使用
- 它不是规则真源，只是顺手的说法
- 它仍在探索，边界还不稳定

## 什么时候进事实层

满足以下任一条件时，进 `docs/`：

- 这是项目业务决策
- 这是接口、数据模型、架构边界
- 这是当前项目已实现的真实状态

## skill 新建口径

默认不要轻易再开新的 skill。优先判断：

- 这个内容能不能进现有真源
- 它是不是只是现有真源的一个章节
- 它是不是只是一个 prompt，不值得独立成 skill

只有当现有真源明显承载不下，或领域边界已经稳定时，再拆新 skill。

## 当前边界

- `personal-os` 只管“个人工作流治理”
- 现在它也承担文档表达、设计文档、FastAPI 后端、脚本模式、代码讲解等真源收口
- `project-methodology` 仍是例外真源
- 它不直接维护项目业务事实
