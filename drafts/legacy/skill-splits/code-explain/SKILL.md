# Code Explain Skill

这是一个**代码讲解真源 skill**。  
它不在正文里重复展开所有讲解细节。

它主要负责：

- 判断当前代码是否值得写讲解文档
- 读取代码讲解真源
- 先建立心智模型，再展开关键逻辑单元讲解
- 避免把讲解文档写成逐行翻译

## Canonical Source

这个 skill 的讲解真源在：

- [`references/code-explain-rules.md`](references/code-explain-rules.md)

默认只读取这一份 reference。  
如果 reference 缺失：

- 明确说明当前缺少代码讲解真源
- 仅保守遵守“只讲难点、先讲心智模型、优先解释 WHY”的最小原则

## 标准使用方式

这个 skill 的默认动作顺序是：

1. 先判断当前代码是否真的需要讲解
2. 读取 `references/code-explain-rules.md`
3. 先输出心智模型或讲解骨架
4. 再按关键逻辑单元展开
5. 如存在固定依赖顺序，再补执行顺序说明

## 什么时候使用

当问题主要是**代码讲解怎么组织、哪些逻辑该讲、该如何让讲解更容易理解**时，使用这个 skill。

### 必须使用

- 需要为函数、模块、脚本写代码讲解文档
- 需要解释算法、关键流程、关键设计
- 需要判断某段代码是否值得单独讲解

### 推荐使用

- 代码逻辑不长，但设计点容易误解
- 讲解对象包含参数、阈值、状态变化或顺序依赖
- 需要给维护者写“为什么这样做”的解释

### 不适合使用

- 纯实现任务
- 命名已能自解释、没有关键设计点的简单逻辑
- 只想要一句话概括，不需要正式讲解文档

## 输出契约

默认先输出：

```text
讲解对象：
目标读者：
是否值得单独讲解：
建议讲解结构：
建议最小产物：
```

只有在这一步明确后，才继续展开正文。

## 最小判断规则

- 如果问题是“这段代码值不值得讲” -> 先判断是否有难点、混淆点、关键设计
- 如果问题是“讲解怎么起手” -> 第一节先写心智模型
- 如果问题是“这段逻辑怎么解释” -> 优先解释 WHY，而不是翻译 WHAT
- 如果问题是“多个逻辑先后有依赖” -> 最后补执行顺序

## 重要边界

- 这是代码讲解真源，不是代码实现指南
- 不要把讲解文档写成逐行转述
- 不要为显而易见的简单逻辑硬写讲解
- 永远优先解释关键设计、混淆点和顺序依赖
