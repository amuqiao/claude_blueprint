# Code Explain Rules

## 文档定位

代码讲解文档是**可选产出**，仅在明确有讲解需求时创建。

选题原则：

- 只讲难点
- 只讲混淆点
- 只讲关键设计

以下内容通常不需要讲：

- 命名已经能自解释的逻辑
- 目标读者已知的通用模式
- 单纯 CRUD、胶水代码、没有设计张力的样板代码

推荐命名：

- `{主题}代码讲解.md`

## 默认结构

| 节 | 是否必须 | 触发条件 |
|---|---|---|
| 心智模型 | 必须 | 始终第一节 |
| 关键逻辑单元讲解 | 有则必须 | 存在值得讲的逻辑单元 |
| 执行顺序 | 按需 | 多个单元间存在固定依赖顺序 |

## 心智模型

第一节固定为“心智模型”。

目的：

- 帮读者建立理解后续内容所需的认知前提
- 先说明这段代码到底在解决什么问题
- 降低后续细节理解门槛

内容可以是：

- 运行上下文
- 数据结构理解
- 领域概念
- 状态流或输入输出关系

## 关键逻辑单元讲解

每个讲解单元按以下顺序判断：

### 1. 问题段（可选）

说明为什么需要这段代码：

- 它解决什么问题
- 不这样做会出现什么问题

只有当原因不显而易见时才写。

### 2. 原理段（必须）

必须至少包含一种更适合理解的表达方式：

- 公式
- 示意图思路
- 伪代码
- 状态变化描述

禁止只做纯自然语言翻译。

### 3. 代码段（必须）

规则：

- 只贴关键 3 到 10 行
- 不贴完整函数
- 注释重点放在 WHY，而不是 WHAT
- 标明完整函数名，方便读者回源码定位

### 4. 参数段（有参数时必须）

至少说明：

- 参数含义
- 参数如何影响行为

如果参数存在多个合理取值，再补调参指南。

## 执行顺序

满足以下任一条件时补充：

- 多个单元有固定先后顺序
- 前一单元的输出是后一单元的输入
- 先后顺序不能调换

推荐表达：

- Mermaid `flowchart`
- 配套说明顺序约束的原因

## 输出原则

- 先讲整体理解，再讲细节
- 优先解释 WHY
- 只保留真正值得讲的单元
- 不要堆满代码片段
- 不要把讲解文档写成逐行注释版源码
