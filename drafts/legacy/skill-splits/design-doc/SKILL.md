# Design Doc Skill

这是一个**设计文档真源 skill**。  
它不在正文里重复展开全部规范细节。

它主要负责：

- 区分项目级架构文档与模块设计文档的边界
- 约束设计文档应该写什么、不该写什么
- 统一数据模型、接口规范、模块边界等核心设计表达
- 为设计评审和质检提供最小判断基线

## Canonical Source

这个 skill 的设计真源在：

- [`references/design-doc-rules.md`](references/design-doc-rules.md)

默认只读取这一份 reference。  
如果 reference 缺失：

- 明确说明当前缺少设计文档真源
- 仅保守遵守“架构文档写骨架、模块设计写约束与契约、质检优先找结构性问题”的最小原则

## 标准使用方式

这个 skill 的默认动作顺序是：

1. 先判断当前产物是“架构文档 / 模块设计文档 / 设计质检”中的哪一种
2. 读取 `references/design-doc-rules.md`
3. 先输出建议结构或评审框架
4. 再展开正文或问题清单
5. 如涉及接口、数据模型、状态流转，优先检查契约是否完整

## 什么时候使用

当问题主要是**怎么写设计文档、怎么区分设计边界、怎么质检方案是否可落地**时，使用这个 skill。

### 必须使用

- 编写项目级架构文档
- 编写模块设计文档
- 整理接口规范、数据模型、状态机、模块边界
- 对设计方案做系统性质检或评审

### 推荐使用

- 需要判断某段内容该写进架构文档还是模块设计文档
- 需要判断方案是否存在边界模糊、职责混叠、过度设计
- 需要在落地前做一次结构性设计审查

### 不适合使用

- 纯业务实现任务
- 只想写 README 或普通说明文档
- 已经有明确设计基线，只需局部措辞润色

## 输出契约

默认先输出：

```text
当前设计类型：
目标读者：
建议最小结构：
关键缺失项：
建议最小产物：
```

只有在这一步明确后，才继续展开正文或评审结果。

## 最小判断规则

- 如果问题是“整个项目怎么设计” -> 用架构文档视角，只写骨架与边界
- 如果问题是“某个模块怎么落地” -> 用模块设计视角，写规则、契约和约束
- 如果问题是“方案有没有问题” -> 优先找结构性缺口，而不是润色措辞
- 如果问题涉及接口和数据模型 -> 必须检查契约完整性

## 重要边界

- 这是设计文档真源，不是项目方法总入口
- 不要把实现细节塞进项目级架构文档
- 不要把 mock、UI 细节误当成模块设计真源
- 永远优先保证边界清楚、契约完整、方案可落地
