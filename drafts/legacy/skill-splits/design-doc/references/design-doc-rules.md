# Design Doc Rules

## 两类文档的内容边界

### 架构文档

项目级架构文档一份即可。  
它只写系统骨架，不写会随某个模块内部变化而频繁变动的实现细节。

适合写：

- 技术选型与理由
- 分层结构与目录边界
- 核心业务流、核心调用链、核心数据流
- 全局约束：认证、错误处理、日志、契约同步方式

不适合写：

- 具体函数签名
- 某个模块内部算法细节
- 某个接口的完整字段表
- 页面布局和细部交互

判断标准：

- 如果这段内容会随着某个模块内部实现变化而频繁变化，它更可能属于模块设计文档

### 模块设计文档

每个功能模块一份。  
它是该模块的设计基线，负责回答“这个模块具体如何被约束和落地”。

适合写：

- 问题定义和约束
- 目标与非目标
- 核心流程与跨模块边界
- 数据模型
- 接口契约
- 状态机与关键规则
- 备选方案与未决事项

不适合写：

- UI 排版细节
- 单纯的页面视觉描述
- 与模块设计无关的项目级方法论

## 模块设计文档的建议结构

1. 问题定义
   说清核心痛点、已知约束、时间边界和依赖关系。
2. 目标与非目标
   目标要可验收；非目标要明确写出不做什么以及原因。
3. 方案设计
   用流程图、决策表、边界描述说明方案主体。
4. 数据模型
   有持久化变更时必须写完整字段定义与约束。
5. 接口规范
   有接口时必须写请求体、响应体、错误码和认证方式。
6. 前端集成与交互协议
   只在复杂交互、流式响应、SSE、WS、多状态流转等情况下展开。
7. 备选方案
   至少保留 2 个可比对方案，并写清否决理由。
8. 待定事项
   只记录真实未决问题，并写决策时机。

## 数据模型规范

每张表建议至少说明：

- 字段名
- 类型
- 可空性
- 默认值
- 说明
- 约束 / 索引

如果包含枚举：

- 必须列出全部合法值

如果包含表间关系：

- 优先补 Mermaid `erDiagram` 或明确关系说明

## 接口规范规范

每个接口建议至少包含四部分：

1. 基本信息
   方法、路径、描述、认证方式。
2. 请求体
   字段、类型、是否必填、说明、校验规则。
3. 响应体
   成功状态码及字段说明。
4. 错误码
   HTTP 状态码、错误码、触发条件。

## 设计质检最小清单

评审一份设计文档时，优先检查：

1. 问题是否说清楚
2. 边界是否清楚
3. 模块职责是否冲突
4. 技术方案是否可落地
5. 核心链路是否闭环
6. 关键契约是否完整
7. 风险、取舍和演化是否交代

如果整体方向没问题：

- 也要指出哪些部分不足以支撑后续实现、联调或演进

如果某部分已经足够：

- 不要为了显得严格而硬挑问题

- 普通文档表达问题仍交给 `document-writing`
