# 文档成型 Skill

这是一个**轻量文档写作 skill**。  
文档表达真源位于本 skill 的 `references/`。

它不在 `SKILL.md` 正文里重复展开完整规则。  
它主要负责：

- 判断当前输出物是否属于正式文档
- 读取文档表达真源
- 先搭建文档骨架，再展开正文
- 在必要时补充更适合扫描和理解的结构化表达

## Canonical Source

这个 skill 的表达真源在：

- [`references/writing-rules.md`](references/writing-rules.md)

默认只读取这一份 reference。  
如果 reference 缺失：

- 明确说明当前缺少文档表达真源
- 仅保守遵守“先整体、后细节；先骨架、后展开；优先降低理解成本”的最小原则

## 标准使用方式

这个 skill 的默认动作顺序是：

1. 先判断当前要写的是哪类文档
2. 读取 `references/writing-rules.md`
3. 先输出整体理解或骨架，再展开正文
4. 如内容较复杂，优先改成更容易扫描的结构化表达
5. 如需新建文档，遵守文档命名规则

## 什么时候使用

当问题主要是**文档成型与表达**时，使用这个 skill。

### 必须使用

- 编写设计文档、架构文档、方法论文档、实施说明、README、索引页
- 需要把零散信息整理成正式文档
- 需要重写文档结构，使其更容易理解和维护
- 需要判断文档开头是否应明确职责、适用场景、目标读者、维护规范

### 推荐使用

- 文档内容已经有了，但结构混乱
- 内容抽象度较高，不确定怎样表达更清楚
- 需要补表格、清单、文本树、Mermaid 等可视化表达
- 需要新建文档并给出稳定命名建议

### 不适合使用

- 纯闲聊或即时回复
- 纯代码实现任务
- 短到不需要正式结构的临时备注
- 只做局部措辞润色，不涉及结构与表达判断

## 输出契约

默认先输出：

```text
文档类型：
目标读者：
文档目标：
建议结构：
建议最小产物：
```

只有在骨架明确后，才继续展开正文。

## 最小判断规则

- 如果核心问题是“这份文档怎么组织” -> 先搭骨架，再展开
- 如果核心问题是“这段内容太抽象” -> 先补心智模型，再讲细节
- 如果核心问题是“文档不好扫描” -> 优先改成清单、表格、文本树或图示
- 如果核心问题是“README / 索引怎么链接” -> 优先使用相对路径
- 如果核心问题是“新文档怎么命名” -> 默认使用英文小写加连字符

## 重要边界

- 这是一个写作方法 skill，不是具体业务真相来源
- 不要为了看起来完整而堆砌内容
- 不要跳过整体理解，直接陷入局部细节
- 不要把所有短文本都强行写成重型文档
- 永远优先选择最小充分、最容易理解的表达方式
