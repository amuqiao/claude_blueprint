# Python Ops CLI Skill

这是一个**Python 运维诊断 CLI 真源 skill**。  
它不在正文里重复展开全部 CLI 规范细节。

它主要负责：

- 统一以子命令组织的运维诊断工具结构
- 区分只读诊断与可选写入动作
- 统一 `--apply`、dry-run 和终端输出的安全口径
- 约束 DB、外部服务、进程信息和富化逻辑的分层

## 标准使用方式

这个 skill 的默认动作顺序是：

1. 先判断当前是不是运维诊断 CLI 场景
2. 读取 `references/python-ops-cli-rules.md`
3. 先输出子命令结构和数据访问分层
4. 再进入 dry-run、`--apply` 和实现细节
5. 优先让脚本适合一线排障直接执行

## 什么时候使用

当问题主要是**怎么用 Python 写一个面向终端使用的运维诊断 CLI**时，使用这个 skill。

### 必须使用

- 新建运维诊断脚本
- 为现有诊断工具新增子命令
- 评审只读诊断、写入动作和数据访问分层是否清楚

### 推荐使用

- 需要跨 DB、Redis、进程状态、日志等多数据源诊断
- 需要把诊断和修复动作放进一个 CLI
- 需要默认 dry-run，只有显式 `--apply` 才真正写入

### 不适合使用

- 多服务生命周期管理脚本
- 批量 API 调用脚本
- 单纯一次性查询脚本，没有稳定 CLI 结构

## 输出契约

默认先输出：

```text
当前 CLI 类型：
建议子命令结构：
建议数据访问分层：
建议 dry-run / --apply 策略：
建议最小产物：
```

只有在这一步明确后，才继续展开实现。

## 最小判断规则

- 如果有多个诊断动作 -> 用子命令，而不是巨型入口函数
- 如果涉及写入 -> 默认 dry-run，并要求 `--apply`
- 如果涉及多数据源 -> 先分清 DB、外部服务、进程信息和富化层
- 如果输出只给终端看 -> 优先即时可读，不默认写报告文件

## 重要边界

- 这是运维诊断 CLI 真源，不是服务总控脚本真源
- 不要让子命令之间互相调用
- 不要让写入逻辑绕开 `--apply`
- 永远优先排障安全、结构清楚、终端可扫
