# Python Ops CLI Rules

## 适用边界

这个真源只处理一类工具：

- 通过子命令组织不同诊断动作
- 同时存在只读诊断和可选写入动作
- 可能跨 DB、Redis、进程状态、日志等多数据源
- 输出主要直接给终端使用

如果核心问题是：

- 服务生命周期管理 -> 用 `shell-service`
- 批量 API 调用、文件产物输出 -> 用 `python-script`

## 最小结构

推荐至少包含这些部分：

1. 模块文档字符串
2. 标准库 / 第三方库 import
3. `PROJECT_ROOT` 与 `sys.path` 注入
4. 项目内部 import
5. 颜色常量
6. 项目常量
7. dataclass
8. 基础设施函数
9. 工具函数
10. 数据访问函数
11. 富化函数
12. 子命令函数
13. `main`

## 子命令契约

规则：

- 一个主脚本 + 多个子命令
- 子命令之间不互相调用
- 只读诊断和写入动作严格区分
- 写入动作必须支持 `--apply`

推荐命名：

- 概览 / 列表：名词
- 单对象诊断：`inspect`
- 批量诊断：形容词，如 `stuck`
- 变更动作：`fix-` 前缀

## 写入控制

所有写入动作默认 dry-run。

最低要求：

- 不加 `--apply` 时，只打印将要执行什么
- 加 `--apply` 时，才真正写入
- dry-run 和 apply 走同一个入口函数，不维护两套逻辑

不要：

- 默认直接写入
- 只打印“完成”，却不逐条给出结果

## 数据访问分层

至少区分：

- DB 查询
- 外部服务访问
- 进程状态读取
- 富化整合

规则：

- DB 层只做查询 / 更新，不做业务判断
- 富化层只做数据合并，不再写 DB
- 子命令层负责组织和输出，不自己内嵌所有访问细节

## 输出规则

输出默认面向终端，不默认生成报告文件。

最低要求：

- help 清楚
- 每个子命令的输出结构稳定
- dry-run 与 apply 输出能一眼区分
- 状态、警告、错误在终端中易于扫描
