# Python Script Skill

这是一个**批量 API 调用脚本真源 skill**。  
它不在正文里重复展开全部脚本模式细节。

它主要负责：

- 约束批量 API 调用脚本的最小结构
- 区分 `Config / Task / Job / Result` 等核心数据层次
- 统一 `call_one / retry_call / build_job_list / write_report / main` 的职责边界
- 统一并发、重试、日志、dry-run 和输出目录的基本口径

## Canonical Source

这个 skill 的脚本模式真源在：

- [`references/python-script-rules.md`](references/python-script-rules.md)

默认只读取这一份 reference。  
如果 reference 缺失：

- 明确说明当前缺少批量脚本真源
- 仅保守遵守“结构清楚、职责单一、最小并发、最小重试、最小报告”的原则

## 标准使用方式

这个 skill 的默认动作顺序是：

1. 先判断当前是不是“批量 API 调用脚本”场景
2. 读取 `references/python-script-rules.md`
3. 先输出脚本结构和数据流
4. 再进入并发、重试、报告和实现细节
5. 始终优先最小充分，不把单次脚本做成重型框架

## 什么时候使用

当问题主要是**一批任务调用外部 API，并为每个任务生成文件产物的 Python 脚本怎么组织**时，使用这个 skill。

### 必须使用

- 编写批量 API 调用脚本
- 评审现有批量脚本的结构是否清楚
- 判断并发、重试、dry-run、输出目录该怎么组织

### 推荐使用

- 输入是一批 task，输出是每个 task 的文件产物
- 关键难点是并发控制、失败重试和报告整理
- 需要为脚本建立稳定的数据结构和函数边界

### 不适合使用

- 纯运维诊断 CLI
- 纯服务生命周期管理脚本
- 普通一次性小脚本，没有批量任务和文件产物

## 输出契约

默认先输出：

```text
当前脚本类型：
输入与输出：
建议最小结构：
建议并发与重试策略：
建议最小产物：
```

只有在这一步明确后，才继续展开实现。

## 最小判断规则

- 如果输入是一批 task，输出是文件产物 -> 优先按 `Task / Job / Result` 分层
- 如果核心难点是 API 调用失败 -> 把重试逻辑独立到 `retry_call`
- 如果需要报告 -> 把报告写出和主流程解耦
- 如果需要并发 -> 先明确并发边界和失败后的继续策略

## 重要边界

- 这是批量 API 调用脚本真源，不是后端服务真源
- 不要把业务参数散装传来传去
- 不要把所有逻辑塞进 `main()`
- 永远优先结构清楚、职责单一、输出稳定
