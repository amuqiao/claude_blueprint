## 建议的执行路径

### 第一步：初始化项目骨架（约 30 分钟）

**为什么先做这个**：
- 翻译执行模块依赖 Redis、配置管理、数据库连接
- 先建立开发环境，后续模块开发更顺畅
- 避免后期大规模重构目录结构

**具体动作**：
- 创建完整目录结构（按架构文档 §8）
- 配置管理（`config.py`，读取环境变量）
- Redis 连接池（`redis.py`）
- 数据库连接（`session.py`，SQLAlchemy async）
- 数据模型（`models.py`，TranslateTask 和 TranslateResult）
- Docker Compose 配置（5 个容器）
- 依赖管理（`requirements.txt` 或 `pyproject.toml`）

---

### 第二步：实现翻译执行模块（约 2 小时）

按照刚才的设计文档，实现三个文件：

1. **`llm_client.py`**（大模型客户端）
   - Prompt 模板
   - HTTP 调用 + 重试逻辑
   - 超时控制

2. **`cache_service.py`**（缓存服务）
   - 批量查缓存
   - 写入缓存
   - Key 生成（MD5 hash）

3. **`translate_executor.py`**（翻译流程编排）
   - 调用缓存查询
   - 调用大模型（缓存未命中时）
   - 合并结果

---

### 第三步：写验证脚本测试（约 30 分钟）

**创建 `scripts/test_translate.py`**：
- 测试缓存未命中场景（首次翻译）
- 测试缓存命中场景（第二次翻译同一标签）
- 测试大模型返回异常格式（手动 Mock）
- 打印缓存命中率和响应时间

**验证目标**：
- Prompt 设计是否有效（大模型能否返回正确格式）
- 缓存逻辑是否正常
- 容错处理是否完善

---

### 第四步：实现任务调度层（约 1.5 小时）

**`tasks/translate_task.py`**（Celery 任务）：
- 调用翻译执行模块
- 写入 `translate_results` 表
- 原子更新 `done`/`failed` 计数
- 推进下一个 pending 任务

---

### 第五步：实现 API 层（约 1 小时）

**`api/v1/translate.py`**（FastAPI 路由）：
- `POST /api/v1/translate/batch`（提交任务）
- `GET /api/v1/translate/batch/{task_id}`（查询进度）
- `GET /api/v1/translate/batch/{task_id}/results`（获取结果）
