# 新建正式文档 Prompt

> 状态：派生自真源 skill 的日常使用 prompt，不再作为唯一真源维护。

适用场景：
- 需要从零开始写一篇正式 Markdown 文档
- 目标不是随手记录，而是形成可长期维护的项目文档

使用方式：
把下面这段 prompt 作为任务说明发给 Claude，再补充你的主题、读者和上下文。

```text
请把下面这份内容整理成一篇正式 Markdown 文档。

要求：
1. 先帮助读者建立整体理解，再展开细节。
2. 默认使用下面这个开头：

# 文档标题

> **文档职责**：...
> **适用场景**：...
> **目标读者**：...
> **维护规范**：...

3. 主干结构优先，先讲“这篇文档到底在讲什么”，再讲层级关系、边界区别和执行方式。
4. 当内容出现明显分层、对比、依赖、覆盖关系时，优先使用结构化清单、表格或文本树；只有在文字开始增加理解成本时才使用 Mermaid。
5. 不要一上来堆背景，不要先讲枝节再讲主干。
6. 如果文档里存在规则、步骤、建议，请明确区分：
   - 什么是必须遵守的
   - 什么是推荐做法
   - 什么是按需补充

我的主题和原始材料如下：
【把你的原始内容贴在这里】
```

补充建议：
- 如果主题偏“方法论 / 认知 / 说明书”，先写整体理解，再写主干结构。
- 如果主题偏“流程 / 操作步骤”，先写总流程，再写阶段划分和分支判断。

## 重构版

适用场景：
- 已有一篇文档，但难读、难扫、层级混乱
- 目标是重构结构，不是只做润色

```text
请把下面这篇文档重构成“更适合阅读和维护”的正式 Markdown 文档。

不要只做措辞润色，要按下面顺序处理：
1. 先找这篇文档真正要回答的主问题。
2. 先补整体理解，让读者知道这篇文档在讲什么。
3. 重排主干结构，先讲主干，再讲细节。
4. 把抽象、堆叠的段落改成更易扫描的结构。
5. 最后再修句子。

输出要求：
- 如果适合，使用正式文档开头：

# 文档标题

> **文档职责**：...
> **适用场景**：...
> **目标读者**：...
> **维护规范**：...

- 讲清楚每一段“在讲哪一层”和“和别的层有什么区别”。
- 对比关系优先用表格。
- 分层关系优先用结构化清单或文本树。
- 不要为了好看而加图，只有在文字已经开始增加理解成本时才补 Mermaid。

原文如下：
【把原文贴在这里】
```

## 自检清单

- [ ] 这篇文档是否先帮助读者建立整体理解
- [ ] 是否先讲主干，再讲细节
- [ ] 是否讲清楚层级关系和边界区别
- [ ] 是否避免一上来堆背景和概念
- [ ] 是否只在真正需要时使用表格、文本树或 Mermaid
- [ ] 如果是正式文档，是否使用了统一开头契约
- [ ] 规则、建议、按需项是否被明确区分
