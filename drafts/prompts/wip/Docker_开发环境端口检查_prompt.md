# Docker 开发环境端口检查命令 Prompt

> **文档职责**：沉淀“根据项目配置，生成可执行的端口检查命令并判断可用端口”的可复用 prompt。
> **适用场景**：项目准备以 Docker 方式部署到本地开发环境，需要先根据现有配置生成一组可执行检查命令，判断端口是否可用、是否冲突时。
> **目标读者**：本仓库维护者。
> **维护规范**：如果这类 prompt 的输入边界和输出结构稳定，可迁入 `prompts/`；如果后续被更完整的部署检查能力吸收，再迁入 `drafts/prompts/archived/`。

## 简单版

```text
请基于当前项目配置，帮我生成一组“开发环境 Docker 部署前的端口检查命令”。

要求：
1. 先从项目配置里找出当前声明的端口
2. 再输出可直接执行的检查命令，判断这些端口是否已被占用
3. 如果端口冲突，再给出一组推荐可检查的替代端口
4. 说明这些命令分别在检查什么
5. 不要直接修改代码或配置
```

## 详细版

```text
请基于当前项目的 Docker 部署配置，帮我输出一份“开发环境端口检查命令清单”。

目标：
- 根据项目现有配置找出需要关注的端口
- 给出可直接执行的命令来检查端口占用情况
- 如果端口冲突，给出可替代端口和对应检查命令

请按以下顺序处理：

1. 检查当前项目配置里声明的端口
- `docker-compose.yml` / `compose.yaml`
- `Dockerfile`
- `.env` / `.env.example`
- 启动脚本
- README / 部署文档中写死的端口

2. 提取需要重点检查的宿主机端口
- Web 服务端口
- API 服务端口
- 数据库 / Redis / MQ / 管理界面端口
- 其他实际暴露到宿主机的端口

3. 为这些端口生成可执行检查命令
- 优先输出适合本机环境的命令
- 命令要能直接检查端口是否被占用
- 如果一个命令不够稳，可以给 2 到 3 种等价命令

4. 如果发现端口可能冲突
- 给出推荐替代端口
- 给出这些替代端口的检查命令
- 简要说明推荐理由

5. 最后给出修改提示
- 如果端口需要调整，应优先改哪里（如 `.env`、`docker-compose.yml`、启动脚本）
- 但本次输出以“检查命令和判断依据”为主，不直接改配置

输出要求：
1. 当前项目涉及的关键端口
2. 每个端口对应的检查命令
3. 命令用途说明
4. 如有冲突，给出推荐替代端口及对应检查命令
5. 最后给出一份简洁结论
```

## 强执行版

```text
请把自己当成开发环境部署助手，针对这个 Docker 项目输出一套可直接执行的端口检查命令。

不要只说“换一个空端口”，而要结合项目实际配置输出真正可执行的检查步骤：

1. 当前项目有哪些服务
2. 哪些服务对外暴露宿主机端口
3. 当前端口配置散落在哪些文件
4. 针对这些端口，给出逐条检查命令
5. 如果默认端口冲突，再给一组推荐端口和检查命令

补充要求：
- 检查命令应尽量贴近本地开发常用环境
- 如果某些端口只供容器内部通信，请明确说明无需检查宿主机占用
- 输出应以“命令 + 说明”为主，而不是泛泛建议
```

## 当前判断

- 这类 prompt 的核心不是“做抽象端口规划”，而是“根据项目配置生成一组可执行的端口检查命令”
- 当前更适合作为 prompt 资产，不急着直接升 skill
- 后续如果你频繁做本地 Docker 部署前检查，可以继续提炼为正式 prompt，或并入更完整的部署检查能力
