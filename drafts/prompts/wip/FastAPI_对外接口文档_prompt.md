# FastAPI 对外接口文档 Prompt

> **文档职责**：沉淀“基于已实现的 FastAPI 项目，整理面向调用方的接口文档”的可复用 prompt。
> **适用场景**：项目已经实现完成，需要基于当前代码、路由、模型和鉴权方式，输出一份可交给前端、第三方或联调方使用的接口文档时。
> **目标读者**：本仓库维护者。
> **维护规范**：如果这类 prompt 的输入边界和输出结构稳定，可迁入 `prompts/`；如果后续被更完整的接口文档方法吸收，再迁入 `drafts/prompts/archived/`。

## 简单版

```text
请基于当前已实现的 FastAPI 项目代码，整理一份面向调用方的接口文档。

要求：
1. 以当前代码和实际路由为准，不要参考历史草案脑补
2. 说明接口用途、请求方法、URL、参数、请求体、返回结果
3. 说明鉴权方式、必要请求头和调用前提
4. 给出可复制的请求示例和响应示例
5. 在文档开头标注生成日期和时间，明确这是一份当前时点的接口文档快照
6. 输出为正式 Markdown 文档，适合直接发给调用方
```

## 详细版

```text
请基于当前已实现的 FastAPI 项目代码、路由、Pydantic 模型、鉴权逻辑和配置，整理一份“面向调用方的接口文档”。

目标读者：
- 前端开发
- 联调人员
- 第三方调用方

要求：
1. 先扫描当前真实存在的接口，不要按理想设计补接口
2. 如果接口很多，请按模块或业务域分组输出
3. 每个接口至少包含：
   - 接口名称 / 用途
   - 请求方法
   - URL
   - path 参数
   - query 参数
   - request body
   - headers（尤其是鉴权相关）
   - 成功响应示例
   - 常见失败响应示例
4. 说明当前项目的鉴权方式，以及调用前需要准备什么
5. 如果某些接口只能在特定前置条件下调用，明确写出前置条件
6. 对于上传、分页、筛选、排序、幂等、回调等特殊接口，按实际情况补充调用说明
7. 在文档开头标注生成日期和时间，明确这是一份当前时点的接口文档快照
8. 输出为正式 Markdown 文档，结构清晰，适合直接给调用方使用

补充要求：
- 优先写调用方真正关心的信息，不要写成后端内部实现分析
- 不要只列字段定义，要给完整可复制的示例
- 如果某些接口当前实现不完整或文档信息不足，明确标注
```

## 严格审查版

```text
请把自己当成“接口文档整理人 + 联调审查人”，基于当前已实现的 FastAPI 项目，产出一份真正可交付给调用方的接口文档。

不要直接照抄 `/docs`，要先判断当前项目的接口文档信息是否足够支撑对外使用。

请按以下顺序处理：

1. 先审查当前接口信息是否完整
- 路由是否清楚
- 请求模型和响应模型是否清楚
- 鉴权方式是否清楚
- 是否存在 `/docs` 里能看见但调用方仍无法直接理解的接口
- 是否存在缺少示例、缺少前置条件、缺少错误说明的问题

2. 再输出正式接口文档
- 按模块分组
- 每个接口写清调用方式
- 给出可复制的请求示例
- 给出成功 / 失败响应示例
- 写清鉴权、前置条件、限制和注意事项

3. 最后补一个“当前接口文档仍存在的缺口”
- 哪些地方需要补示例
- 哪些接口命名或返回结构不够友好
- 哪些信息还不足以直接交付第三方

要求：
- 只基于当前代码和实际实现输出
- 不要把理想设计写成既定事实
- 在文档开头标注生成日期和时间，明确这是一份当前时点的接口文档快照
- 输出目标是“调用方可直接使用的文档”，不是内部代码评审报告
- 最终输出为正式 Markdown 文档
```

## 当前判断

- 这类 prompt 和“后台接口请求示例”不完全一样，它更偏完整的对外接口文档
- 当前更适合作为独立 prompt 保留
- 如果后续频繁出现“基于现有代码整理给调用方的接口文档”场景，可以继续细分为“单接口 / 模块接口 / 全量接口文档”
