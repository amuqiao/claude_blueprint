# 从粗架构到实现 Prompt

> **文档职责**：沉淀“从粗架构设计文档逐步推进到模块设计与实现”的可复用 prompt。
> **适用场景**：已经有一份较粗的架构设计文档，但还没进入清晰的模块设计和实现阶段时。
> **目标读者**：本仓库维护者。
> **维护规范**：如果这组 prompt 的阶段划分和输出结构长期稳定，可迁入 `prompts/`；如果后续被方法文档或 skill 吸收，再迁入 `drafts/prompts/archived/`。

## 通用主干

这组 prompt 不是几个并列问题，而是一条有前后依赖的链路：

**粗架构文档 → 首个核心模块 → 模块设计文档 → 实现前方案 → 代码实现 → 回写文档**

### 阶段 1：从粗架构文档选出首个核心模块

**输入**：一份较粗的架构设计文档  
**目标**：确定当前最值得优先落地的首个核心模块  
**输出**：模块选择结论 + 选择理由 + 模块设计范围  
**进入下一阶段条件**：已经知道“先做哪个模块，以及为什么先做它”
**当前阶段只产出**：模块选择与模块范围判断  
**当前阶段不进入**：项目骨架初始化、文件级实现拆分、API 细节设计、任务排期、代码实现  
**到此为止**：停在“先做哪个模块 + 为什么 + 做到什么边界”

```text
我已经有一份较粗的架构设计文档，请不要直接开始写代码。

请基于这份架构文档，先帮我确定“首个核心模块”。

要求：
1. 判断当前最适合优先落地的模块是什么
2. 说明为什么这个模块应该优先做，而不是别的模块
3. 输出这个模块的设计范围，包括：
   - 模块职责
   - 边界
   - 需要覆盖的核心链路
   - 当前先做什么、不做什么
4. 如果架构文档里有缺失，先指出缺什么，再给出最小可落地判断
5. 输出为正式 Markdown 文档，遵循 writing 规范
6. 不要进入项目骨架初始化、文件拆分、实现顺序或代码建议
```

### 阶段 2：把首个核心模块写成模块设计文档

**输入**：阶段 1 选出的首个核心模块及其范围  
**目标**：把“先做哪个模块”细化为“这个模块具体怎么设计”  
**输出**：模块设计文档  
**进入下一阶段条件**：模块的职责、边界、接口/交互、数据模型和分层职责已经清楚
**当前阶段只产出**：模块设计文档  
**当前阶段不进入**：项目骨架初始化、任务排期、文件级实现清单、代码实现  
**到此为止**：停在“模块怎么设计清楚”，不展开“怎么实施”

```text
我已经确定了首个核心模块，请不要直接写代码。

请把这个模块展开成模块设计文档。

要求：
1. 先确认该模块在整体架构中的位置
2. 说明模块职责、输入、输出、依赖和边界
3. 输出模块设计，至少包括：
   - 核心链路
   - 接口/交互
   - 数据模型
   - 依赖关系
   - 分层职责
4. 如果设计仍然缺关键信息，先指出缺口，不要硬补
5. 输出为正式 Markdown 文档，遵循 writing 规范
6. 不要给出项目初始化步骤、文件清单、开发排期或代码建议
```

### 阶段 3：从模块设计进入实现前方案

**输入**：阶段 2 的模块设计文档  
**目标**：形成可以指导编码的实现前方案  
**输出**：实现顺序、最小闭环、依赖清单、可后置项  
**进入下一阶段条件**：已经知道先实现什么、按什么顺序实现、哪些基础设施必须先补
**当前阶段只产出**：实现前方案  
**当前阶段不进入**：具体代码实现与实现后的文档回写  
**到此为止**：停在“先做什么、后做什么、哪些基础设施必须先补”

```text
我已经有模块设计文档，现在请基于这份设计，生成实现前方案。

不要直接写代码，先输出落地方案。

要求：
1. 拆出推荐实现顺序
2. 说明当前模块的最小闭环是什么
3. 指出哪些基础设施是当前必须的，哪些可以后置
4. 明确本次实现的边界和不做项
5. 如果设计还不足以支撑实现，先指出缺口
6. 不要直接开始写代码
```

### 阶段 4：从实现前方案进入代码实现

**输入**：阶段 3 的实现前方案  
**目标**：开始代码实现  
**输出**：代码 + 需要回写的设计点  
**完成条件**：实现了最小闭环，并知道哪些内容要回写到设计文档
**当前阶段只产出**：代码与回写点清单  
**当前阶段不进入**：直接代替文档回写  
**到此为止**：停在“代码完成 + 知道哪些地方要回写”

```text
我已经有实现前方案，请基于现有文档开始实现。

要求：
1. 先简要复述这次实现的目标和边界
2. 按文档中的分层约束实现，不要跳层
3. 优先实现最小闭环，不要一次铺满所有基础设施
4. 如果文档不足以支撑实现，先指出缺口，不要硬写
5. 实现后给出需要回写到设计文档的内容
6. 不要在本阶段直接改写设计文档正文
```

### 阶段 5：实现后回写设计文档

**输入**：阶段 4 的实现结果与实现过程中发现的偏差  
**目标**：把实现后的真实情况同步回设计文档，避免设计与代码脱节  
**输出**：更新后的设计文档 + 同步状态说明  
**完成条件**：设计文档、实施清单、索引状态与当前实现保持一致
**当前阶段只产出**：文档同步结果  
**当前阶段不进入**：新一轮功能设计或额外代码实现  
**到此为止**：停在“文档与当前实现一致”

```text
我已经完成了一轮实现，请不要只停留在代码结果。

请基于本次实现结果，回写并同步设计文档。

要求：
1. 先列出这次实现与原设计相比，哪些内容被确认、补充或修正
2. 更新对应的设计文档内容，不要只写笼统总结
3. 如果实现过程中改变了模块边界、接口、数据模型或依赖关系，明确写回文档
4. 检查是否需要更新实施清单和设计索引状态
5. 输出为正式 Markdown 文档，遵循 writing 规范
```

## 场景分支

### 全栈项目补充

```text
如果这个项目是全栈项目，请在上面的主干基础上，额外补充：

- 用户流程
- 页面结构
- 前后端交互
- 是否需要先产出 `mock.html`

要求：
- 不要只讲页面，也要讲数据和调用链
- 如果页面结构会影响后端接口设计，明确指出
```

### 后端 API 服务补充

```text
如果这个项目是纯后端 API 服务，请在上面的主干基础上，额外补充：

- API 契约
- 请求/响应模型
- 数据模型
- 调用链
- 外部依赖与基础设施

要求：
- 不要展开页面和前端交互
- 重点说明服务边界、接口设计、存储模型和分层职责
```

### 前端项目补充

```text
如果这个项目是纯前端项目，请在上面的主干基础上，额外补充：

- 页面层
- 组件层
- 状态层
- 服务层
- 页面交互与数据流

要求：
- 不要假设一定有后端改动
- 重点说明页面结构、状态流动和组件职责边界
```

## 当前判断

- 这组 prompt 适合做“架构 → 模块 → 实现”的过渡链路
- 最稳的组织方式不是三套完全独立 prompt，而是“通用主干 + 场景分支”
- 当前更适合作为 prompt 资产，不急着直接升为 skill
- 如果后续阶段划分、输入输出和回写动作持续稳定，可以进一步提炼成正式 prompt 或方法文档
