# 基于已实现代码补强架构文档 Prompt

> **文档职责**：沉淀“基于已实现代码初始化或补强正式架构设计文档”的可复用 prompt。
> **适用场景**：项目已经实现了一部分代码，希望以代码为准生成或持续维护正式架构文档时。
> **目标读者**：本仓库维护者。
> **维护规范**：如果这类 prompt 的输入边界和输出结构稳定，可迁入 `prompts/`；如果后续被方法文档或能力吸收，再迁入 `drafts/prompts/archived/`。

## 阶段 1：基于已实现代码初始化 / 补强正式架构文档

```text
请不要参考历史草案，直接以当前已实现代码为准，帮我初始化或补强一份正式架构设计文档。

要求：
1. 先从当前代码中识别真实存在的模块、分层、关键依赖和运行链路
2. 区分哪些是已经落地的事实，哪些只是代码里尚未成型的推测，不要把推测写成既定事实
3. 输出一份正式架构文档，至少包括：
   - 系统整体定位
   - 当前目录 / 分层结构
   - 核心组件
   - 关键依赖
   - 核心运行链路
   - 当前已知限制与待补充部分
4. 如果代码结构和分层存在明显混乱，也请直接指出，不要为了“像架构文档”而强行美化
5. 输出为正式 Markdown 文档，遵循 writing 规范
```

## 阶段 2：判断并补图

```text
在完成正式架构文档文字内容后，请继续判断这份架构文档最应该补哪些图。

要求：
1. 优先选择最有助于理解当前真实系统的图，不要为了完整而堆图
2. 先判断是否需要：
   - 整体结构图
   - 分层能力结构图
   - 核心组件图
   - 核心链路图
   - 部署图
3. 对每种图说明：
   - 是否需要
   - 解决什么理解问题
   - 应该画到什么粒度
4. 如适合，直接给出 Mermaid 初稿
5. 只画代码和运行形态中已经真实成立的东西，不画尚未落地的理想结构图
```

## 当前判断

- 这类 prompt 的核心不是“回写旧草案”，而是“从已实现代码反推并维护当前真实架构文档”
- 对当前使用方式来说，分成“文字文档补强”和“按需补图”两个阶段更顺
- 后续如果“补图”开始频繁独立发生，再考虑单独拆出
