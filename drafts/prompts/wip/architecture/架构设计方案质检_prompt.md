# 架构设计方案质检 Prompt

> **文档职责**：沉淀“质检架构设计方案”的可复用 prompt。
> **适用场景**：架构设计文档初稿写完后，想系统检查其完整性、清晰度和可落地性时。
> **目标读者**：本仓库维护者。
> **维护规范**：如果这个 prompt 的输入边界和输出结构长期稳定，可迁入 `prompts/`；如果后续被更完整的方法或 skill 吸收，再迁入 `drafts/prompts/archived/`。

## 简单版

```text
请帮我质检这份架构设计方案。

要求：
1. 先判断整体是否完整、清晰、可落地
2. 再指出最关键的问题和缺漏
3. 优先找会影响后续开发和迭代的结构性问题
4. 不要泛泛夸好，重点讲问题
5. 输出时先给总体结论，再列问题，再给改进建议
```

## 详细版

```text
请帮我对这份“架构设计方案”做一次系统性质检，不要直接重写文档，先以评审和质检的方式输出。

请重点检查：

1. 问题定义是否清楚
   - 这份方案是否说清楚要解决什么问题
   - 项目范围、目标和非目标是否明确

2. 架构边界是否清楚
   - 模块划分是否合理
   - 分层职责是否清楚
   - 是否存在职责混叠或边界模糊

3. 技术方案是否可落地
   - 技术选型是否有理由
   - 有没有明显过度设计
   - 有没有关键实现前提没交代

4. 核心链路是否闭环
   - 关键流程是否讲清楚
   - 数据流、调用链、模块协作是否能串起来

5. 风险和后续演化是否有交代
   - 已知风险有没有明确写出
   - 后续扩展点和阶段性取舍是否清楚

输出格式：
1. 总体判断
2. 主要问题
3. 缺失内容
4. 优先修改建议

要求：
- 优先指出结构性问题，而不是抠措辞
- 如果方案整体方向没问题，也要指出哪些部分还不够支撑后续开发
- 如果某部分已经足够，不必为了“显得严格”而硬挑问题
```

## 严格审查版

```text
请把自己当成架构评审人，对这份架构设计方案做一次严格质检。

要求：
- 不要帮我润色
- 不要默认方案正确
- 重点找出“会导致后续实现、协作、扩展或维护出问题”的地方

请从以下角度审查：
1. 是否真的围绕问题设计，而不是只是在堆技术名词
2. 分层、模块、职责是否存在模糊或冲突
3. 是否遗漏了关键链路、关键数据、关键约束
4. 是否有不必要的复杂度
5. 是否能支持后续模块设计和实施落地

输出时请分成：
1. 总体评价
2. 高优先级问题
3. 中优先级问题
4. 建议如何补强
```
