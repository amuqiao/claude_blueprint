# 需求分析与技术栈选型 Prompt

> **文档职责**：沉淀“先分析需求，再确定技术栈”的可复用 prompt。
> **适用场景**：新项目起盘、旧项目重构前、需要先收敛需求和技术路线时。
> **目标读者**：本仓库维护者。
> **维护规范**：如果这个 prompt 的结构和输出长期稳定，可迁入 `prompts/`；如果后续被方法论或 skill 吸收，再迁入 `drafts/prompts/archived/`。

## 简单版

```text
请先不要急着给实现方案，也不要直接开始写代码。

先帮我做两件事：
1. 分析需求，明确这是在解决什么问题
2. 在需求基础上，再判断技术栈应该怎么选

要求：
- 先澄清目标、范围、核心用户和关键约束
- 再判断适合的技术栈，而不是先拍脑袋选框架
- 输出时先给结论，再给理由
- 如果信息不足，先列出需要我补充的关键问题
```

## 详细版

```text
请先帮我做“需求分析 + 技术栈选型”，不要急着进入实现细节，也不要直接写代码。

我希望你按下面顺序来：

1. 先分析需求本身
   - 这个项目/功能真正要解决什么问题
   - 目标用户是谁
   - 核心使用场景是什么
   - 当前范围和非目标是什么
   - 有哪些明确约束（时间、性能、成本、部署、维护、团队规模）

2. 再判断技术栈
   - 后端、前端、数据库、部署方式是否需要分开选
   - 哪些技术选型是核心决策，哪些只是实现细节
   - 优先给“够用、可维护、可迭代”的方案，不要一上来过度设计
   - 如果有多个可行方案，请给出推荐方案和不推荐的原因

3. 输出格式
   - 先给需求判断结论
   - 再给技术栈建议
   - 再给选型理由
   - 最后列出仍需澄清的问题

要求：
- 先做问题定义，再做技术选型
- 不要把“流行”当成理由
- 不要跳过约束直接推荐框架
- 如果需求本身还不清楚，明确指出应先补需求，而不是硬选技术栈
```

## 项目起盘版

```text
请把这次对话当成“项目起盘前的需求分析与技术栈选型”。

先不要写代码，也不要直接进入目录结构设计。

请按这个顺序输出：
1. 需求理解
2. 关键约束
3. 项目类型判断（内容站 / 工具型产品 / SaaS / API 服务 / AI 应用等）
4. 技术栈建议
5. 为什么这样选
6. 哪些问题还需要我确认

要求：
- 优先帮助我判断“这个项目到底是什么”
- 技术栈服务于需求，不要反过来
- 如果适合走最小可用路线，明确指出哪些基础设施可以后置
```

## 当前判断

- 这类 prompt 已经具备复用价值
- 当前更像“起盘前分析 prompt”，还不适合直接升 skill
- 后续如果“需求分析 → 技术栈选型 → 项目定盘”的输出结构稳定，可以继续提炼成正式 prompt 或方法文档
