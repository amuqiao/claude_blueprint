# FastAPI 改动后验证与报告 Prompt

> **状态**：派生自 [skills/personal-os/references/fastapi-backend.md](/Users/admin/Downloads/Code/claude_blueprint/skills/personal-os/references/fastapi-backend.md:1) 的日常使用 prompt，当前保留使用，不作为唯一真源维护。

> **文档职责**：沉淀“FastAPI 服务改动后，先判断最合适的验证方式，再输出验证清单与结果报告”的可复用 prompt。
> **适用场景**：FastAPI 服务改动后，需要检查接口是否正常、是否已有测试资产、是否需要补最小验证方案时。
> **目标读者**：本仓库维护者。
> **维护规范**：如果这类 prompt 的输入边界和输出结构稳定，可迁入 `prompts/`；如果后续被更完整的测试或质检能力吸收，再迁入 `drafts/prompts/archived/`。

## 标准版

```text
请基于当前 FastAPI 项目和这次改动，先检查现有测试与验证方式，再判断我应该如何验证受影响接口是否正常；如果现有验证不足，请给出最小补齐方案，并输出一份简洁的验证清单和结果报告模板。
```

## 强执行版

```text
请把自己当成后端开发验证助手，针对这次 FastAPI 改动，帮我制定接口验证方案。

请按以下顺序处理：
1. 检查当前项目是否已有：
   - `tests/` 目录
   - `pytest` 用例
   - 接口联调脚本
   - 健康检查或 smoke test
2. 结合本次改动范围，找出受影响的接口和核心链路
3. 判断应该：
   - 直接复用现有测试
   - 增加最小测试
   - 用脚本或 `curl` 做接口验证
   - 只做手工验证
4. 如果现有验证不足，请给出“最小补齐方案”，不要过度设计
5. 输出结果包括：
   - 当前可用的验证手段
   - 本次建议验证的接口
   - 推荐验证步骤
   - 是否需要补测试模块
   - 是否建议输出验证报告
```

## 报告版

```text
请基于当前项目和这次改动，帮我整理一份“接口验证与结果报告”模板。

目标：
- 方便我每次改完 FastAPI 服务后快速验证接口
- 记录这次验证了什么、结果如何、还有什么风险

要求：
1. 先判断项目当前已有的测试与验证资产
2. 再给出适合本项目的验证方式
3. 最后输出一份简洁的验证报告模板，包含：
   - 改动范围
   - 验证接口列表
   - 验证方式
   - 验证结果
   - 未覆盖项
   - 风险与后续建议
```

## 当前判断

- 这类 prompt 的核心不是“立刻补完整单元测试体系”，而是“结合现状找最合适的验证方式”
- 当前保留为 `personal-os` 下 FastAPI 真源之上的快捷入口，不再承担后端规范本体职责
- 后续视真实使用频率决定是否继续保留在 `wip/`
