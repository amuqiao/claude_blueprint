# dev.sh 服务总控脚本 Prompt

> **文档职责**：沉淀“为本地 FastAPI 服务编写 dev.sh 总控脚本”的可复用 prompt。
> **适用场景**：本地已部署或准备部署 FastAPI 服务，需要统一管理启动、停止、重启、状态查看，并友好展示访问地址时。
> **目标读者**：本仓库维护者。
> **维护规范**：如果这个 prompt 的输入边界和输出结构稳定，可迁入 `prompts/`；如果后续被 `shell-service` 或 `fastapi-backend` 相关能力吸收，再迁入 `drafts/prompts/archived/`。

## 简单版

```text
请帮我写一个 `dev.sh` 总控脚本，用来管理本地 FastAPI 服务。

要求：
1. 支持 `start`、`stop`、`restart`、`status`
2. 输出要清晰，适合本地开发使用
3. `status` 时友好显示服务状态和可访问 URL
4. 如果服务未启动，给出明确提示
5. 脚本尽量稳，不要依赖复杂外部工具
```

## 详细版

```text
请帮我写一个本地开发用的 `dev.sh` 总控脚本，用来管理 FastAPI 服务。

我希望它至少支持以下命令：
- `./dev.sh start`
- `./dev.sh stop`
- `./dev.sh restart`
- `./dev.sh status`

要求：
1. `start`
   - 启动 FastAPI 服务
   - 避免重复启动
   - 启动成功后输出访问地址

2. `stop`
   - 停止当前服务进程
   - 如果服务未启动，不要报错，要友好提示

3. `restart`
   - 按“先停后启”执行
   - 输出过程清楚

4. `status`
   - 显示服务是否运行
   - 显示 PID（如果有）
   - 显示可访问 URL
   - 如果有健康检查接口，优先顺手检查一次

5. 脚本设计要求
   - 输出清晰、适合开发者日常使用
   - 尽量使用 bash 原生和常见命令
   - 注意 pid 文件、重复启动、僵尸进程等问题
   - 如果项目里已有启动命令、虚拟环境、端口配置，请基于项目现状生成

请先给出脚本设计思路，再给出完整脚本。
```

## 严格版

```text
请帮我为本地 FastAPI 服务设计一个可维护的 `dev.sh` 总控脚本。

不要只给最短可运行版本，请按“开发期长期使用”的标准来写。

目标：
- 用一个脚本统一管理服务生命周期
- 让开发者一眼知道服务是否正常、PID 是多少、URL 是什么
- 减少重复启动、端口占用、停不干净这类问题

功能要求：
1. `start`
2. `stop`
3. `restart`
4. `status`

输出要求：
- 状态信息清晰
- 关键步骤有提示
- 成功、失败、未启动三类情况能区分
- `status` 要显示服务访问地址

实现要求：
- 结合当前项目已有的启动方式、虚拟环境、端口、日志文件、健康检查接口
- 如果某些信息缺失，先指出缺什么，再给一个最小稳妥实现
- 先说明脚本结构，再给完整脚本
```
