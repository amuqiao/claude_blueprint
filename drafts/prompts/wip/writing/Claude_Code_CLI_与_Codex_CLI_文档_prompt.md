# Claude Code CLI 与 Codex CLI 文档 Prompt

> **文档职责**：沉淀一组用于编写并质检 `Claude Code CLI`、`Codex CLI` 及两者对照文档的 prompt。
> **适用场景**：需要把 Agent CLI 相关知识整理成团队 onboarding 文档、能力说明文档或对照型知识库时。
> **目标读者**：本仓库维护者。
> **维护规范**：当前作为写作文档 prompt 保存在 `drafts/prompts/wip/writing/`；如果后续输入边界和输出结构稳定，可迁入正式 prompts。

## 设计判断

这类需求适合维护为**单文件、多阶段、多版本**。

推荐组织方式不是拆成多份文档，而是在一个文件里区分：

- **阶段 1：生成**
  - `Claude Code CLI` 文档版
  - `Codex CLI` 文档版
  - 两者对照整合版
- **阶段 2：质检**
  - 简单版
  - 详细版
  - 严格审查版
  - 快速检查版

原因：

- 单工具文档和对照文档的叙述目标不同
- 单工具文档重在建立完整心智模型
- 对照文档重在概念映射、差异解释、决策和排错
- 生成阶段和质检阶段的任务性质也不同
- 如果拆成多份文档，后续维护和查找成本会变高

推荐使用方式：

1. 先用阶段 1 生成文档初稿
2. 再用阶段 2 做质检
3. 如果最后要收成一篇团队 onboarding 主文档，优先以“对照整合版 + 详细质检版”为主路径

---

## 使用前提

这组 prompt 在使用前，默认有一个前置要求：

- 只要涉及 `Claude Code CLI` / `Codex CLI` 的能力定义、命令、配置项、规则优先级、文件职责边界，就应先参考官方资料，再生成和质检

原因：

- 这类文档不是纯方法论文章，而是工具知识库
- 一旦核心概念、命令、配置字段或优先级关系偏离官方，读者会直接学歪
- 后续即使结构和表达很好，也会因为事实层不准而失去价值

执行原则：

- 事实层要忠于官方
- 解释层允许整理和归纳
- 但整理后的表达不能歪曲官方语义
- 如果某处是作者自己的抽象总结，最好明确区分，不要伪装成官方原话

## 阶段 1：生成

### Prompt 1：Claude Code CLI 文档版

适用场景：

- 需要单独写一份 `Claude Code CLI` 能力说明文档
- 目标是帮助新成员建立完整心智模型
- 文档以 Markdown 为 source of truth

```text
请帮我写一份面向团队 onboarding 的 `Claude Code CLI` 完整说明文档。

文档目标：
- 让第一次接触该工具的成员建立完整心智模型
- 不只是知道某个文件怎么写，还能理解整个能力体系如何协同
- 能够独立完成区分、使用、决策、排错四类任务

读者起点：
- 知道 `CLAUDE.md` 是给 AI 的指令文件
- 但不清楚它和其他配置文件、能力机制的分工
- 不了解 `Claude Code CLI` 的完整能力体系

请覆盖以下能力维度，不遗漏：
- Instructions
- Rule
- Hook
- Skill
- Command
- Plugin
- MCP
- Agent
- Profile
- Memory
- Settings 配置

请覆盖以下四类认知任务：
1. 区分
   - 概念定义
   - 容易混淆的边界
   - 命名差异
2. 使用
   - 操作路径
   - 文件位置
   - 格式示例
3. 决策
   - 用什么
   - 放哪层
   - 为什么
4. 排错
   - 优先级规则
   - Trust 机制
   - 常见陷阱

输出要求：
- 输出为正式 Markdown 文档
- 面向团队 onboarding，而不是零散问答
- 先建立整体理解，再展开细节
- 结构优先任务导向，让读者带着“我要做 X”来查
- 使用 Markdown 原生表达可视化：
  - ASCII 图
  - 表格
  - 结构化清单
  - 代码块
- 不依赖外部渲染
- 文件名、命令、配置字段保持原始英文
- 正文使用中文
- 代码示例使用 Level B：
  - 完整模板
  - 可直接 fork
  - 含中文注释
- 不只讲“怎么做”，也解释“为什么这样设计”
- 管理员视角至少覆盖：
  - 个人开发者
  - Tech Lead

文档组织建议：
1. 先给整体能力地图
2. 再按任务维度展开
3. 再讲配置层级、优先级与生效路径
4. 最后补决策树和排错部分

如果内容过多：
- 先给完整总览版
- 再标出适合后续拆分的章节
```

### Prompt 3：Claude Code CLI 与 Codex CLI 对照整合版

适用场景：

- 需要写一份并列对照文档
- 目标是让团队成员同时看懂两个工具
- 重点是概念映射、差异解释和使用决策

```text
请帮我写一份面向团队 onboarding 的 `Claude Code CLI` 与 `Codex CLI` 对照整合文档。

文档目标：
- 让新成员建立两个工具的完整对照心智模型
- 不只是分别知道各自怎么用，还知道它们在概念、结构、配置、能力机制上的映射关系与差异
- 让读者能够独立完成区分、使用、决策、排错四类任务

读者起点：
- 知道 `CLAUDE.md` / `AGENTS.md` 这类文件是给 AI 的指令文件
- 但不清楚它们和其他配置文件、能力机制的分工
- 不了解两个工具的完整能力体系

请覆盖以下能力维度，不遗漏，并且两个工具同等篇幅、并列讲解：
- Instructions
- Rule
- Hook
- Skill
- Command
- Plugin
- MCP
- Agent
- Profile
- Memory
- Settings 配置

请覆盖以下四类认知任务：
1. 区分
   - 概念映射
   - 命名差异
   - 容易混淆的边界
2. 使用
   - 操作路径
   - 文件位置
   - 格式示例
3. 决策
   - 这个需求该用哪类能力
   - 应该放在哪一层
   - 两边分别怎么判断
4. 排错
   - 优先级规则
   - Trust 机制
   - 常见陷阱

输出要求：
- 输出为正式 Markdown 文档
- 以 Markdown 作为 source of truth
- 面向团队 onboarding
- 两个工具并列，同等篇幅，不要一边作为附录
- 结构采用任务导向，而不是单纯从头讲到尾
- 优先让读者带着“我要做 X”来查
- 使用 Markdown 原生可视化表达：
  - ASCII 图
  - 表格
  - 结构化清单
  - 代码块
- 不依赖外部渲染
- 文件名、命令、配置字段保持原始英文
- 正文使用中文
- 代码示例使用 Level B：
  - 完整模板
  - 可直接 fork
  - 含中文注释
- 不只讲“怎么做”，也解释“为什么这样设计”
- 管理员视角至少覆盖：
  - 个人开发者
  - Tech Lead

推荐组织方式：
1. 先给两者整体能力地图
2. 再给概念映射表
3. 再按“区分 / 使用 / 决策 / 排错”四类任务展开
4. 再补配置层级、优先级、Trust、生效路径
5. 最后给任务导向的决策树

关键要求：
- 对照关系优先用表格
- 层级关系优先用结构化清单或 ASCII 图
- 同一能力维度下，要同时说明：
  - Claude Code CLI 怎么做
  - Codex CLI 怎么做
  - 两者差异在哪里
  - 什么场景下更容易混淆

如果内容过多：
- 先给完整总览版
- 再标出适合后续拆分的章节
```

## 阶段 2：质检

### Prompt 4：对照文档质检简单版

```text
请帮我质检这份 `Claude Code CLI` 与 `Codex CLI` 的对照文档。

要求：
1. 先判断整体是否完整、清晰、适合作为团队 onboarding 文档
2. 再指出最关键的问题和缺漏
3. 重点检查两个工具是否真正并列，而不是一主一辅
4. 重点检查是否覆盖了区分、使用、决策、排错四类任务
5. 重点检查关键概念、命令、配置字段是否与官方规范冲突
6. 不要泛泛夸好，优先指出结构性问题

输出格式：
1. 总体判断
2. 最关键的 3 个问题
3. 建议优先补强的方向
```

### Prompt 5：对照文档质检详细版

```text
请帮我对这份 `Claude Code CLI` 与 `Codex CLI` 的对照文档做一次系统性质检。

不要直接重写文档，先以评审和质检的方式输出。

请重点检查以下内容：

1. 文档定位是否清楚
   - 是否真的面向团队 onboarding
   - 是否先帮助读者建立整体心智模型
   - 是否讲清楚这篇文档要解决什么问题

2. 两个工具是否真正并列
   - 是否同等篇幅
   - 是否同等深度
   - 是否存在一边写得完整、另一边像补充说明
   - 是否存在默认偏向某一工具的叙述方式

3. 概念映射是否完整
   - 是否覆盖了核心能力维度
   - 是否讲清了命名差异、边界差异、结构差异
   - 是否存在只列术语、不解释映射关系的问题

4. 四类任务是否完整覆盖
   - 区分：概念定义、映射关系、命名差异是否清楚
   - 使用：操作路径、文件位置、格式示例是否清楚
   - 决策：是否说明“该用什么、放哪层、为什么”
   - 排错：是否说明优先级规则、Trust 机制、常见陷阱

5. 结构是否任务导向
   - 是否让读者带着“我要做 X”来查
   - 是否避免从头到尾只讲概念而缺少任务入口
   - 是否把主干结构放在前面

6. 示例和可视化是否达标
   - 是否给出了足够完整的示例
   - 是否达到了 Level B 粒度
   - 是否优先使用了 Markdown 原生可视化
   - 是否存在表格、结构化清单、ASCII 图该用但没用的地方

7. 管理员视角和解释深度是否到位
   - 是否至少覆盖个人开发者和 Tech Lead
   - 是否不只讲“怎么做”，也讲“为什么这样设计”

8. 是否符合官方规范与官方定义
   - 文件名、命令、配置字段是否与官方写法一致
   - 核心概念的定义和边界是否与官方描述冲突
   - 是否把作者自己的归纳误写成官方规则
   - 是否为了方便对照而过度类比，导致失真
   - 如果某处是整理后的解释，是否仍然忠于官方语义

输出格式：
1. 总体判断
2. 高优先级问题
3. 中优先级问题
4. 缺失内容
5. 优先修改建议

要求：
- 优先指出结构性问题，而不是抠措辞
- 如果整体方向没问题，也要指出最影响文档质量的薄弱点
- 如果某部分已经足够，不必为了显得严格而硬挑问题
```

### Prompt 6：对照文档严格审查版

```text
请把自己当成团队知识库评审人，对这份 `Claude Code CLI` 与 `Codex CLI` 对照文档做一次严格质检。

要求：
- 不要帮我润色
- 不要默认这份文档已经达标
- 重点找出“会导致新成员理解偏差、选型错误、配置误用或排错困难”的地方

请从以下角度审查：
1. 是否真正建立了双工具的整体心智模型
2. 是否真的完成了概念映射，而不是把两个工具的内容简单并排摆放
3. 是否存在明显的篇幅失衡、深度失衡或叙述偏向
4. 是否遗漏关键能力维度
5. 是否遗漏关键任务场景
6. 是否缺少足够的示例、层级说明或优先级解释
7. 是否存在与官方规范冲突或可能误导读者的表述
8. 是否适合作为团队 onboarding 主文档

输出时请分成：
1. 总体评价
2. 高优先级问题
3. 中优先级问题
4. 最该补上的内容
5. 是否建议继续作为主文档推进
```

### Prompt 7：对照文档快速检查版

```text
请快速检查这份 `Claude Code CLI` 与 `Codex CLI` 对照文档是否达标。

只回答下面这些问题：
1. 两个工具是否真正并列
2. 能力维度是否覆盖完整
3. 是否覆盖区分、使用、决策、排错四类任务
4. 关键概念、命令、配置字段是否与官方规范一致
5. 是否适合作为团队 onboarding 文档
6. 最大缺陷是什么
7. 最终判断：
   - 可以继续推进
   - 修改后再推进
   - 不建议作为主文档
```

## 我的建议

- 如果你的最终目标是一份团队主文档，**优先使用 Prompt 3**
- 如果你还在分别摸清两个工具，再先各跑一次 Prompt 1 和 Prompt 2
- 写完对照稿后，优先先跑 Prompt 5
- 如果只是想快速判断是否偏题，先跑 Prompt 7
- 如果这篇文档准备成为团队主文档，最后再跑一次 Prompt 6

## 当前判断

- 这类需求更适合做成“一个文件里的多阶段多版本 prompt”
- 最稳的结构是：阶段 1 负责生成，阶段 2 负责质检
- 其中“Prompt 3 + Prompt 5 + Prompt 6”最接近你当前这份草稿的主路径
