# 后台接口请求示例 Prompt

> **文档职责**：沉淀“为后台管理 UI 对应接口编写请求示例文档”的可复用 prompt。
> **适用场景**：需要给前端开发、联调人员或未来维护者补接口请求示例时。
> **目标读者**：本仓库维护者。
> **维护规范**：如果这类 prompt 的输入边界和输出结构稳定，可迁入 `prompts/`；如果后续被更完整的接口文档方法吸收，再迁入 `drafts/prompts/archived/`。

## 简单版

```text
请帮我为这个后台接口写一份请求示例，面向前端开发联调使用。

要求：
1. 写清请求方法、URL、参数和请求体
2. 给出完整请求示例
3. 给出成功响应示例
4. 如有必要，补一个常见失败响应示例
5. 输出为清晰的 Markdown 文档，便于直接复制给前端
```

## 详细版

```text
请帮我为这个后台管理 UI 对应的接口，整理一份前端联调用的请求示例文档。

目标读者：前端开发 / 联调人员

要求：
1. 先说明这个接口的用途
2. 写清：
   - 请求方法
   - URL
   - path 参数
   - query 参数
   - request body
   - headers（如鉴权）
3. 给一个完整可复制的请求示例
   - curl 示例优先
   - 如适合，也可补 fetch / axios 示例
4. 给出成功响应示例
5. 给出常见失败响应示例
6. 如果这个接口和页面交互流程有关，顺手说明它在页面里的触发时机
7. 输出为正式 Markdown 文档，结构清晰，便于直接发给前端

补充要求：
- 不要只给字段说明，要给完整可复制的示例
- 优先输出前端真正会复制使用的内容，而不是后端视角的抽象说明
```

## 模块接口版

```text
请帮我为这个后台模块整理“接口请求示例文档”，面向前端联调。

要求：
1. 不只是列接口名，要按页面或操作流程组织
2. 每个接口都给：
   - 用途
   - 方法和 URL
   - 参数说明
   - 请求示例
   - 成功响应示例
3. 如果多个接口有先后依赖，写清调用顺序
4. 输出为 Markdown 文档，适合直接作为模块联调文档
```

## 当前判断

- 这类 prompt 已具备复用价值
- 当前更适合作为 prompt 资产，不急着直接升 skill
- 如果后续经常需要产出“前端联调接口示例文档”，可以继续提炼为正式 prompt，或并入更完整的接口文档方法
