# 服务安全检查 Prompt

> **文档职责**：沉淀“先做服务安全审计，再整理安全接入与使用说明”的可复用 prompt。
> **适用场景**：需要检查项目中鉴权、限流、输入校验、配置暴露等安全设置，并基于当前代码整理安全说明时。
> **目标读者**：本仓库维护者。
> **维护规范**：如果这类 prompt 的输入边界和输出结构稳定，可迁入 `prompts/`；如果后续被更完整的安全审查能力吸收，再迁入 `drafts/prompts/archived/`。

## 阶段 1：服务安全审计

### 简单版

```text
请基于当前项目代码、配置、接口实现和运行方式，对这个服务做一次安全审计。

要求：
1. 先基于当前项目实际情况分析，不要凭空假设
2. 重点检查接口鉴权、JWT / Session / API Key 机制、越权风险、限流、输入校验、敏感信息暴露和暴露面
3. 如果项目当前没有成型的鉴权机制，请结合现状给出推荐方案
4. 按“问题 / 风险 / 建议修复方案”的格式输出
5. 如果某项目前无法判断，请明确说明缺少什么信息
```

### 详细版

```text
请基于当前项目代码、配置、接口实现和运行方式，对这个服务做一次安全审计。

要求：
1. 先基于当前项目实际情况分析，不要凭空假设
2. 重点检查：
   - 接口鉴权是否完整，JWT / Session / API Key 是否真的在后端生效
   - Token 是否校验签名、过期时间、类型和必要声明
   - 是否存在公开接口、越权访问、权限边界缺失
   - 是否缺少限流、防刷、输入校验、上传或外部请求限制
   - 是否存在敏感信息暴露、调试接口暴露、CORS 过宽、docs / health / admin 暴露面问题
3. 如果项目当前没有成型的鉴权机制，请结合项目现状给出推荐鉴权方案

输出：
1. 总体安全判断
2. 高优先级问题
3. 每个问题的“位置 / 风险 / 修复建议”
4. 推荐鉴权方案
5. 按优先级排序的整改清单
```

### 强审查版

```text
请把自己当成安全审查人，对这个服务做一次偏实战的安全检查。

不要急着给“最佳实践清单”，先基于当前项目实际情况找问题。

重点关注：
- 未鉴权接口
- JWT / Session / API Key 鉴权是否真正成立
- 越权风险
- 限流缺失
- 调试接口暴露
- 敏感配置泄露
- 输入校验不足
- 文件上传 / 外部请求 / 回调接口风险
- CORS、docs、health、admin 等暴露面

输出格式：
1. 总体判断
2. 高风险问题
3. 中风险问题
4. 推荐鉴权方案
5. 暂未发现但建议关注的点
6. 按优先级排序的修复方案
```

## 阶段 2：整理安全接入与使用说明

```text
请基于当前已实现代码和安全机制，整理一份“安全接入与使用说明”。

要求：
1. 不要参考历史草案，直接以当前代码和配置为准
2. 说明当前项目采用的鉴权方式，以及哪些接口公开、哪些接口需要保护
3. 说明前端如何调用、第三方如何调用、是否需要 token / API Key / 密钥，以及这些配置放在哪里
4. 如果项目提供 `http://localhost:8000/docs`，说明它在开发期如何使用，以及哪些内容可以参考、哪些不能作为生产安全边界
5. 如果项目当前安全实现不完整，请区分：
   - 已落地的安全机制
   - 建议补上的安全机制

输出：
1. 当前安全接入方式概览
2. 鉴权与调用方式说明
3. 密钥 / Token / 配置项说明
4. 开发期 `/docs` 的使用说明
5. 仍待补齐的安全能力
6. 输出为正式 Markdown 文档，适合交给前端、联调方或第三方接入方阅读
```

## 当前判断

- 对你当前的使用习惯来说，更适合做成同一主题的二阶段，而不是拆成两份独立 prompt
- 阶段 1 负责找问题和给方案，阶段 2 负责把当前已落实的安全机制整理成可用文档
- 后续如果你发现“只做安全审计”和“只写安全接入说明”会频繁独立发生，再考虑拆分也不迟
