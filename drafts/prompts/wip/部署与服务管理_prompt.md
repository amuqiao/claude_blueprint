# 部署与服务管理 Prompt

## 简单版

```text
请基于当前项目代码、配置、脚本、Docker 相关文件和已有文档，整理一份“部署与服务管理文档”。

当前口径固定为：
- 只维护 `docker-compose.yml`
- 只维护 `.env`、`.env.dev`
- 通过 `ENV_FILE` 切换 docker-compose 读取的环境文件
- 不再使用 `docker-compose.prod.yml`、`.env.prod`、`.env.prod.example`

这份文档需要覆盖 3 种部署方式：
1. 本地开发
2. docker-compose 全量部署
3. 基于 Dockerfile 的独立部署

要求：
1. 先基于当前项目实际情况分析，不要凭空补全
2. 按这 3 种部署方式分别输出，不要混写
3. 每种方式至少说明：
   - 适用场景
   - 服务边界
   - PostgreSQL、Redis 等外部依赖服务由谁提供、如何连接
   - 配置要求
   - 启动方式
   - 服务管理方式
   - 常见问题与排查思路
4. 如果某种方式当前项目并不成立，直接说明原因
5. 输出为正式 Markdown 文档
```

## 详细版

### 阶段 1：判断这 3 种部署方式是否成立

```text
请先不要直接输出部署文档。

请基于当前项目代码、配置、脚本、Docker 相关文件和已有文档，先判断以下 3 种部署方式在当前项目中是否成立：

1. 本地开发
2. docker-compose 全量部署
3. 基于 Dockerfile 的独立部署

当前口径固定为：
- 只维护 `docker-compose.yml`
- 只维护 `.env`、`.env.dev`
- docker-compose 统一通过 `ENV_FILE` 切换
- 不再使用 `docker-compose.prod.yml`、`.env.prod`、`.env.prod.example`

输出要求：
1. 分别判断这 3 种方式是否支持
2. 说明判断依据
3. 如果不支持，指出缺什么
4. 说明每种方式的服务边界和依赖来源
5. 特别说明 PostgreSQL、Redis 等外部依赖服务在这 3 种方式下的边界是否清楚
6. 不要进入正式文档输出
```

### 阶段 2：输出部署与服务管理文档

```text
请基于前一阶段对 3 种部署方式的判断结果，输出一份“部署与服务管理文档”。

要求：
1. 先给三种部署方式的区别总览
2. 再按三种方式分别输出文档，不要混写
3. 每种方式统一包含：
   - 适用场景
   - 服务边界
   - 依赖来源
   - PostgreSQL、Redis 等外部依赖服务由谁提供、如何连接
   - 配置要求
   - 启动方式
   - 服务管理方式
   - 验证方式
   - 常见问题与排查思路
4. 对于前一阶段判定为不支持的方式，明确标注“当前不成立”或“需要补充什么”
5. 最后补一个“如何选择这三种部署方式”的简短建议

要求：
- 以阶段 1 的判断为准，不要重新假设
- 不要把三种部署方式混成一套命令或一套配置
- 输出为正式 Markdown 文档
```

## 严格审查版

### 阶段 1：严格审查这 3 种部署方式是否真正成立

```text
请把自己当成项目维护者和部署审查人，先不要急着输出部署文档。

请基于当前项目真实情况，严格审查以下 3 种部署方式在当前项目中是否真正成立：

1. 本地开发
2. docker-compose 全量部署
3. 基于 Dockerfile 的独立部署

当前口径固定为：
- 只维护 `docker-compose.yml`
- 只维护 `.env`、`.env.dev`
- docker-compose 统一通过 `ENV_FILE` 切换
- 不再使用 `docker-compose.prod.yml`、`.env.prod`、`.env.prod.example`

请重点检查：
1. 仓库里是否真的存在支撑这些方式的配置、脚本、Docker 文件和环境变量方案
2. 哪些方式是已经能落地的
3. 哪些方式只是理论可行，但当前仓库没有足够支撑
4. 哪些关键前提缺失
5. 三种方式的服务边界、依赖来源、配置方式是否清楚
6. PostgreSQL、Redis 等外部依赖服务在不同方式下由谁提供、如何连接，是否表达清楚

输出要求：
1. 分别给出“成立 / 基本成立 / 不成立”的判断
2. 说明每个判断的依据
3. 指出缺口和风险
4. 特别指出 PostgreSQL、Redis 等外部依赖服务的边界是否明确
5. 不要进入正式文档输出
```

### 阶段 2：基于审查结论输出正式文档

```text
请基于前一阶段的审查结论，产出一份可长期维护的“部署与服务管理文档”。

要求：
1. 先给“部署方式总览”，说明三者的边界和差异
2. 再按三种部署方式分别写文档，不要混写
3. 每种方式统一包含：
   - 适用场景
   - 服务边界
   - 依赖来源
   - PostgreSQL、Redis 等外部依赖服务由谁提供、如何连接
   - 配置要求
   - 启动方式
   - 服务管理方式
   - 验证方式
   - 常见故障排查
4. 对于前一阶段判定为“基本成立”或“不成立”的方式，明确写出限制、缺口或风险
5. 最后给出推荐使用建议

限制：
- 必须以前一阶段的审查结论为准
- 不要脱离当前项目现实去补理想化方案
- 如果信息不足，明确指出缺口
- 输出为正式 Markdown 文档
```

## 当前判断

- 当前主线已经固定为“一个 `docker-compose.yml` + `.env` / `.env.dev` + `ENV_FILE`”
- 这份 prompt 的任务是基于这套口径输出部署与服务管理文档
- `简单版` 保持单阶段，`详细版` 和 `严格审查版` 保持两阶段
