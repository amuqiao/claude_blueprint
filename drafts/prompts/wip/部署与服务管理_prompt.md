# 部署与服务管理 Prompt

## 简单版

```text
请基于当前项目代码、配置、脚本、Docker 相关文件和已有文档，整理一份“部署与服务管理文档”。

这份文档需要支持 3 种部署方式：
1. 本地开发
2. docker-compose 全量部署
3. 基于 Dockerfile 的独立部署

要求：
1. 先基于当前项目实际情况分析，不要凭空补全
2. 按这 3 种部署方式分别输出，不要混写
3. 每种方式至少说明：
   - 适用场景
   - 服务边界
   - 配置要求
   - 启动方式
   - 服务管理方式
   - 常见问题与排查思路
4. 如果某种方式当前项目并不成立，直接说明原因
5. 输出为正式 Markdown 文档
```

## 详细版

### 阶段 1：判断这 3 种部署方式是否成立

```text
请先不要直接输出部署文档。

请基于当前项目代码、配置、脚本、Docker 相关文件和已有文档，先判断以下 3 种部署方式在当前项目中是否成立：

1. 本地开发
- 应用在宿主机本机运行
- docker-compose 只负责依赖服务

2. docker-compose 全量部署
- 应用服务和依赖服务都由 docker-compose 启动

3. 基于 Dockerfile 的独立部署
- Dockerfile 用于构建应用镜像
- 应用服务独立运行
- PostgreSQL / Redis 由外部提供

输出要求：
1. 分别判断这 3 种方式是否支持
2. 说明判断依据
3. 如果不支持，指出缺什么
4. 说明每种方式的服务边界和依赖来源
5. 不要进入正式文档输出
```

### 阶段 2：输出部署与服务管理文档

```text
请基于前一阶段对 3 种部署方式的判断结果，输出一份“部署与服务管理文档”。

要求：
1. 先给三种部署方式的区别总览
2. 再按三种方式分别输出文档，不要混写
3. 每种方式统一包含：
   - 适用场景
   - 服务边界
   - 依赖来源
   - 配置要求
   - 启动方式
   - 停止 / 重启 / 日志 / 状态查看方式
   - 验证方式
   - 常见问题与排查思路
4. 对于前一阶段判定为不支持的方式，明确标注“当前不成立”或“需要补充什么”
5. 最后补一个“如何选择这三种部署方式”的简短建议

要求：
- 以阶段 1 的判断为准，不要重新假设
- 不要把三种部署方式混成一套命令或一套配置
- 输出为正式 Markdown 文档
```

## 严格审查版

### 阶段 1：严格审查这 3 种部署方式是否真正成立

```text
请把自己当成项目维护者和部署审查人，先不要急着输出部署文档。

请基于当前项目真实情况，严格审查以下 3 种部署方式在当前项目中是否真正成立：

1. 本地开发
2. docker-compose 全量部署
3. 基于 Dockerfile 的独立部署

请重点检查：
1. 仓库里是否真的存在支撑这些方式的配置、脚本、Docker 文件和环境变量方案
2. 哪些方式是已经能落地的
3. 哪些方式只是理论可行，但当前仓库没有足够支撑
4. 哪些关键前提缺失
5. 三种方式的服务边界、依赖来源、配置方式是否清楚

输出要求：
1. 分别给出“成立 / 基本成立 / 不成立”的判断
2. 说明每个判断的依据
3. 指出缺口和风险
4. 不要进入正式文档输出
```

### 阶段 2：基于审查结论输出正式文档

```text
请基于前一阶段的审查结论，产出一份可长期维护的“部署与服务管理文档”。

要求：
1. 先给“部署方式总览”，说明三者的边界和差异
2. 再按三种部署方式分别写文档，不要混写
3. 每种方式统一包含：
   - 适用场景
   - 服务边界
   - 依赖来源
   - 配置要求
   - 启动方式
   - 服务管理方式
   - 验证方式
   - 常见故障排查
4. 对于前一阶段判定为“基本成立”或“不成立”的方式，明确写出限制、缺口或风险
5. 最后给出推荐使用建议

限制：
- 必须以前一阶段的审查结论为准
- 不要为了完整而写成泛泛教程
- 不要脱离当前项目现实去补理想化方案
- 如果信息不足，明确指出缺口
- 输出为正式 Markdown 文档
```

## 当前判断

- `简单版` 适合单阶段，方便直接复制使用
- `详细版` 和 `严格审查版` 更适合分成两阶段
- 第一阶段的核心任务是先判断三种部署方式是否成立
- 第二阶段再基于判断结果输出文档，这样更不容易混写或误判
