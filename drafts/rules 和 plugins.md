## `rules` 到底什么时候该用

### 你现在不急着用 `rules` 的原因
因为你当前的约束，大多已经有明确归宿：

- 跨项目通用约束 → `~/.claude/CLAUDE.md`
- 复杂任务规范 → `skills/`
- 项目级工作流与边界 → 项目 `CLAUDE.md`
- 阶段入口 → `commands/`
- 强制约束 → `hooks`

所以 `rules` 现在没有强刚需。

### `rules` 真正适合的场景
当你出现下面这种需求时，`rules` 才开始有价值：

- **按路径生效的约束**
  - 例如：`frontend/` 下有一套约束，`backend/` 下另一套
- **项目内多子域、多目录、多边界**
  - 例如：`pipeline/`、`adapters/`、`pages/` 各有不同规则
- **不想把项目 `CLAUDE.md` 撑太长**
  - 这时可以把项目规则按主题拆成模块化文件

### 一句话判断
**当项目级规则开始“按目录、按子域、按文件类型”分化时，再引入 `rules/`。**

所以对你现在的 blueprint，我建议：

- **系统级先不做 `rules/`**
- **项目级只在项目复杂度明显上来时才用 `.claude/rules/`**
- 当前先以：
  - 项目根 `CLAUDE.md`
  - `docs/design/...`
  - `skills/`
  为主

## `plugins` 到底什么时候该用

### 你现在不急着用 `plugins` 的原因
`plugins` 不是“组织规范”的工具，而是“扩展 Claude Code 能力”的工具。

- 阶段边界

这些问题主要靠文档、commands、skills、hooks 就能解决。

### `plugins` 真正适合的场景
当你需要的是**新能力**，而不是新规范时，才该考虑 plugin。

例如：
- 接一个外部系统
- 加一个专门的数据源
- 增加某种工具能力
- 做可复用的自动化能力封装

### 一句话判断
**plugin 用来扩能力，不用来补方法论。**

所以你现在不用 plugin 是合理的。  
因为你当前的核心矛盾是“开发范式”，不是“Claude 还缺某个外接能力”。
