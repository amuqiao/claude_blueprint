# 图示表达原则

> **文档职责**：记录本仓库对“文字图 / ASCII 图 / Mermaid / 普通分点”之间边界的当前思考，用于后续再决定是否更新全局写作原则。
> **适用场景**：发现某类图示表达写得难看、过重或不够直观时，用来判断应该如何调整规范。
> **目标读者**：本仓库维护者。
> **维护规范**：遇到新的正例或反例时追加，不急于收敛成硬规则；等积累足够案例后，再考虑是否同步更新 `CLAUDE.md` 或 `writing` skill。

---

## 1. 当前问题不是“能不能用 ASCII 图”，而是“怎样既有效又好看”

过去的表述容易滑向两个极端：

- 一刀切禁止 ASCII 图
- 为了保留灵活性，什么图都放开

这两个方向都不理想。

真正的问题不是“形式是否合法”，而是：

- 这种表达是否足够直观
- 是否比其他表达更省力
- 是否在中文场景下仍然整洁
- 最终观感是否足够好

也就是说，判断标准应该是**信息效率和美观性**，而不是格式统一。

---

## 2. 文字图有时比 Mermaid 更有效

像“线性流程”“少量节点”“一步接一步”的说明，文字图常常更直接。

典型特征：
- 节点数量少
- 基本没有复杂分支
- 读者目标是快速扫一眼
- 不值得为了正式性引入 Mermaid 的额外重量

这类场景里，简洁的文字图往往更好，因为：
- 写起来快
- 看起来轻
- 不打断正文节奏
- 一眼能懂

因此不能把“所有非 Mermaid 图示”都视为低质量表达。

---

## 3. 真正应该避免的是“难看的图”

需要警惕的不是“ASCII 图”这个类别本身，而是以下情况：

- 中文或宽字符导致边缘错位
- 为了凑方框而让内容拥挤
- 对齐不稳定，读起来费劲
- 本来是简单结构，却画成笨重的形式

一旦图示出现这些问题，即使 technically 可用，也应该判定为表达失败。

所以应该避免的不是“文字图”，而是：

**为了省事而输出难看的图。**

---

## 4. 暂定的判断原则

当前更适合采用软性原则，而不是硬性禁令。

### 适合文字图 / 简洁 ASCII 图的场景

- 线性流程
- 节点少
- 只是帮助理解顺序关系
- 不需要精确布局
- 图本身比 Mermaid 更轻、更直观

### 更适合 Mermaid 的场景

- 多角色交互
- 多分支状态流转
- 多模块边界
- 需要更明确的视觉分组
- 图一旦不用正式图就会显得混乱

### 更适合普通分点或文本树的场景

- 根本不需要图
- 只是列层级或清单
- 图反而让表达更重

---

## 5. 现阶段不急着把规则写死

现阶段不建议立刻把 `CLAUDE.md` 里的相关提示改成严格新规则。

原因：
- 目前还缺少足够多的正例和反例
- 一旦写成硬规则，模型容易过度服从
- 过早收紧会伤害本来高效的简洁表达

更稳的做法是：
- 先积累案例
- 先区分“有效的文字图”和“失败的 ASCII 方框图”
- 等边界更清楚后，再更新全局原则

---

## 6. 暂时可接受的一句话原则

如果现在必须用一句话概括这件事，最接近真实意图的版本是：

**表达方式服从信息效率和观感，不服从形式统一。**

延伸解释：
- 简单场景下，文字图可以优先
- 复杂关系里，Mermaid 更适合
- 一旦 ASCII 图显得错位、拥挤或难看，就应该换表达方式

## 8. 当前收录的一个正例

来源：`title/docs/design/模块/任务模块_拓扑模式手册.md`

原文里“1.2 一次任务的完整事件流（文字图）”属于一个典型正例，因为它满足：

- 结构是线性的
- 节点虽然多，但关系单一，核心是“按顺序发生什么”
- 读者重点不是看拓扑布局，而是快速理解事件推进过程
- 如果强行改成 Mermaid，图会更重，正文节奏也会被打断

可以简化提炼成这种形式：

```text
[HTTP 请求]
   ↓
services.create_task
   ├─ 写 tasks / jobs
   └─ dispatch(...)
         ↓
workflows registry.build
   ├─ build_canvas
   ├─ 批量插 work_items
   └─ 投递到 Redis Broker
         ↓
[Celery Worker] 取消息执行
   ├─ work_item: pending → running
   ├─ 执行业务函数
   └─ work_item: running → done / failed
         ↓
[SSE] 推送 step_started / step_done / task_done / task_failed
```

这个例子说明：

- 文字图不是天然低级表达
- 当信息本身是“顺序感”而不是“空间关系”时，文字图反而更有效
- 真正应该避免的是中文环境下为了凑方框而出现的错位与拥挤

所以当前更合理的原则不是“禁止文字图”，而是：

**允许简洁、整齐、顺手的文字图；避免难看、错位、过度装饰的 ASCII 方框图。**

## 9. 当前收录的第二个正例：轻量拓扑图

同样来源于：`title/docs/design/模块/任务模块_拓扑模式手册.md`

像下面这种“扇出并行，最后合流”的轻量拓扑图，也是一个值得保留的正例：

```text
              ┌─► worker[0]    ─┐
   start ─────┼─► worker[1]    ─┼─► finalize
              └─► worker[N-1]  ─┘
```

这个例子的价值在于：

- 它不是纯线性流程
- 但结构仍然足够简单
- 用极少字符就表达了“扇出 → 并行 → 合流”
- 比起 Mermaid，它更轻、更快、更适合嵌在正文中

它成立的前提是：

- 节点标签短
- 关系简单
- 布局容易对齐
- 图的目标是快速表达拓扑，不是做正式架构图

这类图说明：

- 文字图的可用范围不只限于“线性事件流”
- 在简单拓扑场景下，轻量图示也可能比 Mermaid 更有效

但它也有明显边界。一旦出现以下情况，就应该切回 Mermaid：

- 标签变长
- 中文过多
- 分支层级继续增加
- 需要补充大量旁注
- 对齐开始不稳定

所以这个例子进一步支持当前判断：

**不该禁止所有 ASCII 图，而应该允许那些结构简单、标签短、观感稳定的轻量图示。**
