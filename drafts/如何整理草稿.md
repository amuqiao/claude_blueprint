我理解你的需求是：

**你不是单纯想“存草稿”，而是想让草稿有一条低摩擦、可重复的整理路径。**

更具体地说，你要解决的是这几个问题：

- 一个想法、一段思维链、一次对话摘录，先怎么快速落下来
- 过几天再看时，怎么方便地把它提炼成：
  - 普通草稿
  - 重点草稿
  - 正式决策
- 这个过程最好不要每次都靠你手动想“该怎么整理”
- 你希望有一个固定入口，最好能配合 Claude TUI 用

这不是“再写一篇规则文档”的问题，而是：

**你需要一套“草稿整理工作流”。**

## 先给结论

**最适合你的，不是先加 skill，也不是只写一段固定 prompt，而是先加一个显式 command。**

也就是：

**新增一个草稿整理命令。**

比如名字可以是：

- `/distill-draft`
- `/refine-draft`
- `/process-draft`

我更建议：

**`/distill-draft`**

因为它最贴近你的真实意图：  
不是简单“整理”，而是**提炼**。

---

## 为什么 command 最合适

### 不是 hook
因为草稿整理不是必须自动发生的动作。  
它需要你主动决定“现在要整理这篇草稿”。

### 不只是固定 prompt
固定 prompt 当然能用，但问题是：
- 它不在系统里
- 每次容易改来改去
- 不容易形成标准输出格式

### 不是先做 skill
skill 更适合“某类任务怎么做”。  
而你现在要的是一个**元工作流入口**：
- 读取草稿
- 分类
- 提炼
- 推荐去向

这更像 command。

---

## 我建议的最小方案

### 新增一个 command
比如：

```text
/distill-draft 草稿路径
```

例如：

```text
/distill-draft drafts/真正述求.md
/distill-draft drafts/incubating/项目文档应该写到什么程度.md
```

### 它应该做什么
我建议只做 5 件事：

1. **读取草稿原文**
2. **提炼核心判断**
   - 这篇草稿真正讲了什么
3. **识别成熟度**
   - 普通草稿
   - 重点草稿
   - 可升格正式文档
4. **推荐去向**
   - 留在 `drafts/`
   - 移到 `drafts/incubating/`
   - 提炼进 `PLAYBOOK.md`
   - 提炼进 `WHY.md`
   - 提炼进 `MAINTAINING.md`
   - 提炼进 `README.md`
5. **输出一个标准整理结果**

---

## 我建议的标准输出格式

这样最实用：

### 1. 草稿主题
一句话说这篇在讨论什么。

### 2. 核心判断
列 3 到 5 条真正值得保留的判断。

### 3. 当前成熟度
三选一：
- 保持普通草稿
- 升为重点草稿
- 可以提炼为正式文档

### 4. 推荐归宿
只能选一个主去向：
- `drafts/`
- `drafts/incubating/`
- `PLAYBOOK.md`
- `WHY.md`
- `MAINTAINING.md`
- `README.md`

### 5. 下一步建议
例如：
- 暂不升格，继续观察
- 迁入 `incubating/`
- 提炼成 `PLAYBOOK` 一节
- 拆成两部分分别进入 `WHY` 和 `MAINTAINING`

## 这套工作流的真正价值

它会把你的草稿处理从：

**“我想到一个东西，先存一下，以后再说”**

变成：

**“我想到一个东西，先存；之后有固定入口把它提炼成系统资产。”**

这就把草稿从“仓库里的角落文件”，变成了：
- 方法演化的原料
- 正式规范的前置阶段
