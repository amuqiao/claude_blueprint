## 先把三种东西分清

### 1. command
适合做：**显式工作流入口**

比如：
- `/init-architecture`
- `/new-module`
- `/update-docs`

特点：
- 确定性强
- 适合“现在就执行”

不适合：
- 承担整套方法论
- 替你做高层判断

### 2. skill
适合做：**某一类任务的思维和写法规范**

比如：
- 设计文档怎么写
- 架构分析怎么输出
- 运维 CLI 怎么组织

特点：
- 约束“怎么做”
- 不负责决定“什么时候做”

### 3. 说明文档
适合做：**范式定义**

也就是：
- 新项目怎么起盘
- 老项目怎么纳入规范
- 新模块怎么走
- 小改动和结构性改动怎么区分

特点：
- 负责“方法层”
- 是所有 command / skill 的上位说明

## 所以你应该怎么设计

### 你真正需要的是三层结构

### 第一层：开发范式
这是最上层，先定义“开发阶段”。

建议你把范式抽象成 4 个阶段：

1. **项目定盘**
目标：明确目标、范围、技术栈、分层边界  
产物：
- 项目 `CLAUDE.md`
- `docs/design/架构设计方案.md`
- `docs/design/INDEX.md`
- `docs/实施清单.md`

2. **模块设计**
目标：把一个具体功能模块讲清楚  
产物：
- `docs/design/模块/.../设计_v1.md`
- `页面布局_mock.html`（如有前端）

3. **实现落地**
目标：按设计实现代码  
产物：
- 代码
- 测试/验证结果

4. **回写与抽象**
目标：同步文档、抽取可复用规则  
产物：
- 更新后的设计文档
- `/add-rule` 归档的新规则

这才是你的“开发范式”。

---

## 第二层：阶段判断规则

这是你最关心的“先做什么”。

### 新项目
先做：
- 目标/范围
- 技术栈与关键选型
- 架构与分层
- 再进入模块设计

也就是：

**不是先写代码，也不是先铺满基础设施。**

### 老项目纳入规范
先做：
- 补项目级骨架
- 让项目能进入同一套范式
- 再开始新模块开发

### 新模块
先做：
- 问题定义
- 数据模型 / API 契约 / 页面结构
- 再实现

### 小改动 / bugfix
可以：
- 先实现
- 再回写文档

## 第三层：工具承载

这时才轮到 command / skill / 文档。

### 说明文档
负责定义范式本身。  
这是你现在最缺的。

### `/init-architecture`
负责项目定盘阶段的**入口**。  
它不应该承担整套范式，只负责把项目带入起点状态。

### `/new-module`
负责模块设计阶段的**入口**。

### `/update-docs`
负责实现后回写阶段的**入口**。

### skills
负责每一阶段里“具体怎么写”。

## 回到你那几个关键问题

### 先做架构，还是先考虑技术栈？
先问题与范围，再架构与技术栈一起定。  
不是先单独“选技术栈”，也不是先盲画架构。

### 先做基础设施，还是先做功能模块？
先做**支撑首个核心模块所需的最小基础设施**。  
不要一开始把全套基础设施铺满。

### 先写设计文档，还是先写项目代码？
- 新模块、结构性改动：先设计文档
- 小改动、bugfix：先代码，后同步文档

### 到功能设计时，先后端设计还是先页面布局？
先**用户流程 / 页面结构 / API 契约 / 数据模型一起收敛**，不要只从后端或只从前端单边推进。

更实用的顺序是：

1. 用户要完成什么
2. 页面大致怎么承载
3. 前后端交互是什么
4. 数据模型是什么
5. 再分别落后端和前端实现
