# Decision Prompts

> **目录职责**：沉淀“认知补盲 + 选型判断 + 决策整理”相关的通用 prompt。
> **适用场景**：经验不足、概念模糊、方案空间看不全、需要先补齐认知再做判断时。
> **目标读者**：本仓库维护者。
> **维护规范**：本文件同时作为目录索引、使用入口和维护规范；新增、调整、合并 `decision` 目录内容时，优先以本文件为准。

## 这个目录解决什么问题

这个目录不是为了直接回答某个领域里的具体问题，而是为下面这类高价值场景提供通用提问能力：

- 我还不知道这个问题到底在选什么
- 我只想到了 1 到 2 个熟悉方案
- 我缺少完整概念地图
- 我不知道行业里还有哪些默认选项
- 我想先补齐判断框架，再做推荐或决策

它的定位不是替代 `rule` 或 `skill`，而是作为它们的上游能力来源：

- `prompt`：负责补齐认知、展开方案空间、组织提问
- `rule`：负责沉淀稳定判断标准和约束边界
- `skill`：负责封装稳定工作流和执行方式

推荐演进路径：

1. 先用 prompt 反复补盲和提问
2. 从多次对话中提炼稳定判断维度
3. 把稳定标准沉淀为 rule / checklist
4. 把稳定流程升级为 skill

---

## 目录层级

`decision` 目录现在按角色分层，而不是继续平铺。

这样做的目的不是为了好看，而是为了避免把不同类型的文件误当成同一类对象：

- `README.md`
  目录索引和维护规范，不参与子 prompt 质检

- `core/`
  放母 prompt。这里的文件负责补盲和展开方案空间，不按“领域子 prompt”标准质检

- `domains/`
  放真正的领域子 prompt。这里才是“子 Prompt 质检”的默认作用对象

- `tools/`
  放辅助型 prompt，例如补问模板、质检 prompt。它们服务使用过程，但不等同于领域子 prompt

这条边界要明确：

- [tools/子 Prompt 质检_prompt.md](./tools/%E5%AD%90%20Prompt%20%E8%B4%A8%E6%A3%80_prompt.md)
  默认只用于检查 `domains/` 下的候选文件
- 它**不默认检查** `README.md`、`core/` 和 `tools/` 下的文件，除非你明确指定

## 使用顺序

### 场景 1：完全陌生

先用 `core/` 下的母 prompt：

- 先补齐概念
- 先展开常见方案空间
- 先建立比较维度

### 场景 2：已经知道主题，但领域判断还模糊

先用 `domains/` 下的领域子 prompt：

- 直接聚焦到该主题下的常见方案
- 输出更贴近该领域的结构化结果

### 场景 3：对话进行到一半卡住

直接用 `tools/` 下的辅助文件：

- 补问模板：追问遗漏方案、概念边界、判断前提
- 口令版质检：临时丢一句话，让模型快速判断方向
- 快速质检：先做 30 秒级判断，看值不值得继续深入检查
- 子 prompt 质检：做完整质检，判断是否适合正式收录

## 命名规范

`decision` 目录下的文件命名优先遵循下面规则：

- `core/`：保留少量母 prompt，名称直接体现“通用 / 总入口”
- `domains/`：使用 `主题名_prompt.md` 或 `主题名选型_prompt.md`
- `tools/`：使用明确工具功能名，例如 `提问补问模板.md`、`子 Prompt 口令版质检_prompt.md`、`子 Prompt 快速质检_prompt.md`、`子 Prompt 质检_prompt.md`

命名原则：

- 文件名要能直接暴露“决策对象”或“工具职责”
- 避免过泛名称，例如 `架构_prompt.md`、`规范_prompt.md`
- 避免只写工具名，不写决策对象
- 同一层级尽量保持风格一致，不混用太多句式

## 子 Prompt 准入标准

这里的“子 prompt”默认特指 `domains/` 下的候选文件，不包括 `core/` 和 `tools/`。

不是每个主题都应该立刻拆成独立文件。新增 `domains/` 子 prompt 前，至少满足下面三条：

- 你已经反复遇到这个领域的选型问题
- 通用母 prompt 已经不够高效
- 这个领域存在稳定的概念集、方案空间和比较维度

如果只满足下面情况，优先不要新建 `domains/` 文件：

- 只是一次性问题
- 只是已有子 prompt 的一个小分支
- 还没有形成稳定决策对象
- 只是想先收集一些追问短句

这几种情况更适合：

- 先追加到现有 `domains/` 子 prompt
- 先放进 [tools/提问补问模板.md](./tools/%E6%8F%90%E9%97%AE%E8%A1%A5%E9%97%AE%E6%A8%A1%E6%9D%BF.md)
- 先在真实使用中验证，再决定是否拆分

## 职责边界

`decision` 目录内的内容现在按四类角色理解：

- **README**：索引和维护规范，不参与质检
- **Core**：母 prompt，负责补盲和展开方案空间
- **Domains**：领域子 prompt，负责某一类稳定决策对象
- **Tools**：辅助 prompt，负责补问、质检、快速检查

边界要求：

- 每个 `domains/` 文件必须有明确主决策对象
- `domains/` 文件允许有少量交叉，但不能高度重叠
- `tools/` 文件不按 `domains/` 标准质检
- `core/` 文件不默认走“子 Prompt 质检”
- 如果一个文件主要在回答“最终规范怎么执行”，它更接近 rule，而不是 decision prompt

---

## 演进规则

`decision` 目录里的内容默认按下面路径演进：

1. 先在真实问题里反复使用 `core/` 母 prompt
2. 高频追问、口令式调用和快速检查先沉淀到 `tools/`
3. 稳定结构再升级为 `domains/` 子 prompt
4. 稳定判断标准再升级为 checklist / rule
5. 稳定工作流再升级为 skill

判断是否该升级：

- 如果一个短句被反复补问，它适合进 `tools/`
- 如果一个动作主要是“一句话就能触发”，它也适合进 `tools/`
- 如果一个动作主要是“快速判断是否值得继续”，它也适合进 `tools/`
- 如果一个主题反复出现固定结构，它适合进 `domains/`
- 如果一个判断标准已经相对稳定，它适合进 rule
- 如果一整套流程可重复执行，它适合进 skill

## 维护动作规范

对 `decision` 目录做变更时，优先遵循下面顺序：

1. 先判断是新增、补充、合并还是归档
2. 先判断它属于 `core/`、`domains/` 还是 `tools/`
3. 先更新本 `README`
4. 再新增或调整具体文件
5. 保持“索引说明”和“实际结构”同步

具体要求：

- 新增 `domains/` 子 prompt 时，要在“当前结构”中登记
- 新增或重构 `domains/` 子 prompt 后，优先过一遍 [tools/子 Prompt 质检_prompt.md](./tools/%E5%AD%90%20Prompt%20%E8%B4%A8%E6%A3%80_prompt.md)
- 如果一个主题已经不再独立，应从“当前结构”中移除并合并说明
- 如果扩展方向变化，要同步更新“当前优先扩展方向”
- 不要让 `README` 落后于实际文件结构

## 合并与归档规则

当两个 `domains/` 子 prompt 长期高度重叠时，应考虑合并，而不是继续并存。

适合合并的信号：

- 决策对象几乎相同
- 比较维度高度相同
- 高频补问大面积重复
- 使用时经常分不清该选哪个文件

适合归档或降级为 `tools/` 的信号：

- 长期低频使用
- 结构不稳定，尚未形成独立主题
- 更适合作为某个大主题中的一个小节

处理原则：

- 能合并就先合并
- 能降级为 `tools/` 就不要强留为 `domains/`
- 归档前先在 `README` 中移除入口或注明状态

## 新建子 Prompt 模板

写完一个新的 `domains/` 子 prompt 后，建议用 [tools/子 Prompt 质检_prompt.md](./tools/%E5%AD%90%20Prompt%20%E8%B4%A8%E6%A3%80_prompt.md) 过一遍，再决定是否收进目录。

如果你后续继续扩展，可以沿用下面的最小模板：

```text
# 子 Prompt 标题

> 文档职责：
> 适用场景：
> 目标读者：
> 维护规范：

## 这个子 Prompt 解决什么问题

## 常见概念

## 常见方案空间

## 标准版 Prompt

## 高频补问

## 适合沉淀成笔记的输出结构

## 当前判断
```
