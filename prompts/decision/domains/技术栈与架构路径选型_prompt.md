# 技术栈与架构路径选型 Prompt

> **文档职责**：沉淀一套更贴近“技术栈选择 / 架构路径判断 / 系统边界划分”场景的决策 prompt。
> **适用场景**：新项目起盘、旧项目重构、功能扩展前，需要先看清技术路线和架构方案空间时。
> **目标读者**：本仓库维护者。
> **维护规范**：作为 `decision` 目录下的领域子 prompt 维护；当某些子问题明显膨胀后，再拆成更细的专题 prompt。

## 这个子 Prompt 解决什么问题

这类 prompt 主要解决下面几类常见困惑：

- 我脑子里只有熟悉的框架名，但不知道到底在做哪一层决策
- 我会直接问“用什么技术栈”，但还没有先定义需求和约束
- 我不知道哪些是核心架构决策，哪些只是实现细节
- 我不知道行业里常见的几条技术路线分别适合什么条件
- 我不清楚应该优先追求最快上线、最好维护，还是最强扩展性

---

## 常见概念

这类问题里，最容易混在一起的概念包括：

- **需求问题**：到底在解决什么业务或用户问题
- **技术栈**：选用哪些语言、框架、数据库、部署组件
- **架构路径**：系统整体按什么方式组织，例如单体、模块化单体、前后端分离、事件驱动
- **系统边界**：哪些能力要放进当前系统，哪些要外置或后置
- **核心决策**：会显著影响后续迭代成本的选择
- **实现细节**：在主路径已确定后再决定也可以的部分

## 标准版

```text
请把这次对话当成一次“技术栈与架构路径选型”分析，不要急着推荐框架，也不要直接写代码。

我当前的问题是：
【把你的问题贴在这里】

我目前想到的方案是：
【写下你目前想到的技术、框架、架构方式；如果没有，就写“目前只有模糊想法”】

请按下面顺序帮我展开：

1. 先定义这个问题到底在决定什么
   - 是需求问题
   - 是技术栈问题
   - 是架构路径问题
   - 还是系统边界问题
   - 如果我把几个层级混在一起了，请拆开说明

2. 先补齐常见方案空间
   - 这个场景下通常有哪些主流路径
   - 请至少列出 A / B / C
   - 如果有新手容易漏掉但现实中很常见的过渡方案，也请补上

3. 澄清核心概念
   - 技术栈、架构路径、系统边界、核心决策、实现细节分别是什么意思
   - 哪些概念经常被混用
   - 它们分别影响什么

4. 比较这些方案
   - 从开发效率、维护成本、扩展性、团队门槛、部署复杂度、风险等角度比较
   - 明确每种方案更适合什么，不适合什么

5. 结合我的场景给出建议
   - 当前更推荐什么方向
   - 为什么
   - 这个建议依赖哪些前提
   - 如果我的目标从“尽快上线”变成“长期演进”，推荐会怎么变化

6. 如果现在还不适合拍板
   - 请列出最值得继续确认的 3 到 5 个问题

输出要求：
- 先补齐方案空间，再进入推荐
- 不要把“流行”直接当成理由
- 如果我默认跳过了关键约束，请明确指出
- 如果我的问题其实跨了多个决策层级，请拆成多个问题回答
```

## 项目起盘版

```text
请把这次对话当成一次“项目起盘前的技术栈与架构路径判断”。

先不要写代码，也不要直接进入目录结构设计。

项目/功能背景如下：
【把背景贴在这里】

请按下面结构输出：
1. 这个项目真正要解决什么问题
2. 当前关键约束
3. 这属于什么类型的系统
4. 常见技术路线有哪些
5. 哪条路线更适合当前场景
6. 哪些能力应该先做，哪些应该后置
7. 还需要继续确认什么

要求：
- 先定义问题，再谈技术
- 技术选型服务于需求，不反过来
- 如果适合走最小可用路线，请明确指出哪些基础设施可以后置
- 如果存在更简单但足够用的方案，请不要省略
```

## 思路决策文档版

```text
请把下面这份关于“技术栈 / 架构路径 / 系统边界”的草稿，整理成一份“思路决策文档”。

不要把它写成最终拍板文档，而是写成一个帮助我持续判断的文档。

请按下面结构输出：
1. 这个问题在决定什么
2. 核心概念与边界
3. 常见方案空间
4. 比较维度
5. 当前草稿已提到的方案
6. 当前草稿遗漏但值得补充的方案
7. 当前待确认问题
8. 阶段性倾向
9. 后续建议继续追问的方向

要求：
- 把“需求判断”和“技术推荐”分开
- 把“架构路径”和“框架选择”分开
- 如果草稿里直接跳到了熟悉框架，请回补前面的决策层
- 输出要适合长期维护

原始草稿如下：
【把草稿贴在这里】
```

## 高频补问

- `请先告诉我，我现在是在做需求判断、技术栈判断，还是架构路径判断。`
- `除了我熟悉的框架外，这个场景下常见的主流路线还有哪些？`
- `请把“技术栈选择”和“系统边界划分”分开解释。`
- `哪些决策现在必须做，哪些可以后置？`
- `如果我要优先追求尽快上线，推荐会怎么变化？`
- `如果我要优先追求长期可维护，推荐会怎么变化？`
- `请告诉我这个场景下最容易过度设计的点是什么。`
