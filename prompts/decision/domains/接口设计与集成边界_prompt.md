# 接口设计与集成边界 Prompt

> **文档职责**：沉淀一套更贴近“接口设计 / 系统集成 / 边界划分”场景的决策 prompt。
> **适用场景**：需要判断系统之间如何协作、接口应该暴露到什么程度、集成边界如何划分、职责如何切开时。
> **目标读者**：本仓库维护者。
> **维护规范**：作为 `decision` 目录下的领域子 prompt 维护；当 API 风格、事件集成、第三方接入、模块边界等子问题明显膨胀后，再拆出更细专题。

## 这个子 Prompt 解决什么问题

这类 prompt 主要解决下面几类常见困惑：

- 我知道系统之间需要交互，但不知道边界应该切在哪里
- 我不清楚接口应该暴露粗粒度能力，还是细粒度操作
- 我会直接问“API 怎么设计”，但还没有先定义消费者是谁、集成方式是什么
- 我不知道应该同步调用、异步事件、文件交换，还是内部直接调用
- 我不清楚应该先追求复用性、稳定性，还是交付速度

---

## 常见概念

这类问题里，最容易混在一起的概念包括：

- **接口**：系统对外或对内暴露的可调用能力
- **集成边界**：两个系统、模块或服务之间的职责分界线
- **消费者**：谁来调用这个接口，为什么调用
- **交互方式**：同步请求、异步消息、回调、批处理、文件交换等
- **契约稳定性**：接口变更是否容易影响调用方
- **边界泄漏**：一个系统把内部实现细节暴露给了外部
- **编排与编排权**：由谁负责组织多个能力的调用顺序

## 标准版

```text
请把这次对话当成一次“接口设计与集成边界”分析，不要急着直接画 API 或写协议。

我当前的问题是：
【把你的问题贴在这里】

我目前想到的方案是：
【写下你现在想到的接口形式、集成方式或边界划分；如果没有，就写“目前只有模糊想法”】

请按下面顺序帮我展开：

1. 先定义这个问题到底在决定什么
   - 是在决定接口形态
   - 是在决定系统边界
   - 是在决定集成方式
   - 还是在决定职责归属
   - 如果我把几个层级混在一起了，请拆开说明

2. 先补齐常见方案空间
   - 这个场景下通常有哪些主流路径
   - 请至少列出 A / B / C
   - 如果存在新手容易漏掉但现实中很常见的过渡方案，也请补上

3. 澄清核心概念
   - 接口、集成边界、消费者、交互方式、契约稳定性、边界泄漏、编排权分别是什么意思
   - 哪些概念经常被混用
   - 它们分别影响什么

4. 比较这些方案
   - 从耦合度、演进成本、稳定性、开发速度、调试难度、复用性等角度比较
   - 明确每种方案更适合什么，不适合什么

5. 结合我的场景给出建议
   - 当前更推荐什么方向
   - 为什么
   - 这个建议依赖哪些前提
   - 如果我的目标从“快速打通”变成“长期稳定集成”，推荐会怎么变化

6. 如果现在还不适合拍板
   - 请列出最值得继续确认的 3 到 5 个问题

输出要求：
- 先补齐方案空间，再进入推荐
- 不要把“接口多”直接等同于“设计好”
- 如果我跳过了消费者、调用方式或变更成本，请明确指出
- 如果我的问题其实跨了 API 设计、模块边界、系统协作多个层级，请拆开回答
```

## 模块边界版

```text
我正在判断模块 / 服务之间的边界和接口应该如何设计。

当前背景如下：
【把背景贴在这里】

请按下面结构输出：
1. 当前有哪些参与方
2. 各自应该承担什么职责
3. 哪些能力应该作为接口暴露
4. 哪些细节不应该泄漏到边界之外
5. 更适合采用什么交互方式
6. 为什么
7. 后续如果规模变大，应该怎么演进

要求：
- 把“职责边界”和“调用方式”分开
- 把“接口契约”和“内部实现”分开
- 如果更粗粒度的接口已经足够，请明确指出
```

## 高频补问

- `请先告诉我，我现在是在决定接口形态、系统边界，还是集成方式。`
- `这个场景下，消费者到底是谁，他们需要什么粒度的能力？`
- `哪些能力应该暴露成接口，哪些不应该跨边界泄漏？`
- `这里更适合同步请求、异步事件，还是其他交互方式？`
- `如果我优先追求快速打通，推荐会怎么变化？`
- `如果我优先追求长期稳定契约，推荐会怎么变化？`
- `这个场景下最容易出现的边界泄漏是什么？`
