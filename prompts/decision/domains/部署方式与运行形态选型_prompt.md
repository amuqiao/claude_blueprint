# 部署方式与运行形态选型 Prompt

> **文档职责**：沉淀一套更贴近“部署方式 / 运行形态 / 服务暴露方式”场景的决策 prompt。
> **适用场景**：需要判断服务应该如何启动、部署、编排、暴露、维护与排障时。
> **目标读者**：本仓库维护者。
> **维护规范**：作为 `decision` 目录下的领域子 prompt 维护；当部署层问题继续增长后，再拆分为更细专题。

## 这个子 Prompt 解决什么问题

这类 prompt 主要解决下面几类常见困惑：

- 我知道服务要跑起来，但不知道应该本地直跑、容器化、脚本托管还是交给平台
- 我不清楚“开发运行方式”和“生产部署方式”是不是同一个决策
- 我不知道服务应该如何暴露端口、管理进程、处理配置和日志
- 我不知道哪些能力应该自己维护，哪些更适合交给基础设施
- 我会直接问“怎么部署”，但还没有先定义运行目标和约束

---

## 常见概念

这类问题里，最容易混在一起的概念包括：

- **运行形态**：服务在什么环境、以什么方式运行
- **部署方式**：代码如何从仓库变成可运行实例
- **编排方式**：多个服务如何协同启动、管理和通信
- **暴露方式**：服务通过什么路径被访问
- **配置策略**：环境变量、配置文件、密钥如何管理
- **运维边界**：哪些事情自己管，哪些交给平台或外部组件

## 标准版

```text
请把这次对话当成一次“部署方式与运行形态选型”分析，不要急着直接给部署命令。

我当前的问题是：
【把你的问题贴在这里】

我目前想到的方案是：
【写下你现在想到的运行或部署方式；如果没有，就写“目前只有模糊想法”】

请按下面顺序帮我展开：

1. 先定义这个问题到底在决定什么
   - 是在决定开发运行方式
   - 是在决定生产部署方式
   - 是在决定服务编排方式
   - 还是在决定暴露和运维边界
   - 如果我把几个层级混在一起了，请拆开说明

2. 先补齐常见方案空间
   - 这个场景下通常有哪些主流路径
   - 请至少列出 A / B / C
   - 如果有新手容易漏掉但现实中很常见的过渡方案，也请补上

3. 澄清核心概念
   - 运行形态、部署方式、编排方式、暴露方式、配置策略、运维边界分别是什么意思
   - 哪些概念经常被混用
   - 它们分别影响什么

4. 比较这些方案
   - 从复杂度、维护成本、可迁移性、可观测性、稳定性、协作成本等角度比较
   - 明确每种方案更适合什么，不适合什么

5. 结合我的场景给出建议
   - 当前更推荐什么方向
   - 为什么
   - 这个建议依赖哪些前提
   - 如果目标从“本地开发方便”变成“线上稳定维护”，推荐会怎么变化

6. 如果现在还不适合拍板
   - 请列出最值得继续确认的 3 到 5 个问题

输出要求：
- 先补齐方案空间，再进入推荐
- 不要把“更复杂”误当成“更专业”
- 如果我跳过了环境、流量、协作方式等关键约束，请明确指出
- 如果我的问题其实跨了开发、测试、部署、运维多个层级，请拆开回答
```

## 服务落地版

```text
我正在判断一个服务应该如何运行和部署。

当前背景如下：
【把背景贴在这里】

请按下面结构输出：
1. 这个服务当前处于什么阶段
2. 现在真正需要解决的运行问题是什么
3. 常见可选路径有哪些
4. 当前场景更适合哪条路径
5. 为什么
6. 哪些能力现在要做，哪些可以后置
7. 后续如果规模变大，应该怎么演进

要求：
- 不要直接上最复杂方案
- 如果更轻量的运行方式已经足够，请明确指出
- 把“短期跑起来”和“长期稳定运行”区分开
```

```text
请把下面这份关于“部署方式 / 运行形态 / 服务暴露”的草稿，整理成一份“思路决策文档”。

不要把它写成最终部署方案，而是写成帮助我持续判断的文档。

请按下面结构输出：
1. 这个问题在决定什么
2. 核心概念与边界
3. 常见方案空间
4. 比较维度
5. 当前草稿已提到的方案
6. 当前草稿遗漏但值得补充的方案
7. 当前待确认问题
8. 阶段性倾向
9. 后续建议继续追问的方向

要求：
- 把“运行形态”“部署方式”“编排方式”“暴露方式”分开
- 把“开发方便”和“线上稳定”分开
- 如果草稿里直接跳到了具体命令，请回补前面的决策层
- 输出要适合长期维护

原始草稿如下：
【把草稿贴在这里】
```

## 高频补问

- `请先告诉我，我现在是在决定运行形态、部署方式，还是服务暴露方式。`
- `这个场景下更适合本地直跑、容器化，还是平台托管？`
- `开发运行方式和生产部署方式应该分开判断吗？`
- `哪些能力现在必须做，哪些可以后置？`
- `如果我优先追求本地开发效率，推荐会怎么变化？`
- `如果我优先追求线上稳定性，推荐会怎么变化？`
- `这个场景下最容易过度设计的部署点是什么？`

## 当前判断

- 这是对母 prompt 在“运行与部署判断”场景下的领域化收窄
- 它适合承接很多“服务怎么跑、怎么暴露、怎么维护”的问题
- 后续如果“本地开发环境”“容器编排”“线上运维”持续增长，可以从本文件继续拆出更细主题
