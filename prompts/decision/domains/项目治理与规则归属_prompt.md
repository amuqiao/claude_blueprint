# 项目治理与规则归属 Prompt

> **文档职责**：沉淀一套更贴近“项目治理 / 规则归属 / 职责边界”场景的决策 prompt。
> **适用场景**：需要判断某条规则、约束、配置或流程应该放在哪里、由谁负责、以什么粒度生效时。
> **目标读者**：本仓库维护者。
> **维护规范**：作为 `decision` 目录下的领域子 prompt 维护；当规则体系、治理层级或工具边界持续增长后，再拆出更细专题。

## 这个子 Prompt 解决什么问题

这类 prompt 主要解决下面几类常见困惑：

- 我知道“需要一个规则”，但不知道它应该放在系统级、项目级还是局部级
- 我不清楚某条约束应该写成文档、规则、脚本检查，还是人工约定
- 我不知道这件事应该由个人维护、团队维护，还是由管理员统一收口
- 我会把“能力实现”“治理约束”“执行流程”混在一起
- 我不知道某条规则是长期稳定约束，还是当前阶段的临时策略

---

## 常见概念

这类问题里，最容易混在一起的概念包括：

- **规则归属**：这条规则应该放在哪一层
- **治理对象**：规则约束的是代码、流程、文档、环境，还是人员行为
- **生效范围**：个人、本项目、团队、组织级
- **执行方式**：文档说明、人工约定、自动检查、强制拦截
- **责任主体**：谁来制定、谁来维护、谁来执行
- **临时策略**：短期适配当前阶段的做法
- **长期规则**：值得稳定沉淀下来的治理约束

## 标准版

```text
请把这次对话当成一次“项目治理与规则归属”分析，不要急着直接替我下规则。

我当前的问题是：
【把你的问题贴在这里】

我目前的想法是：
【写下你现在想到的归属层级、约束方式或执行方式；如果没有，就写“目前只有模糊想法”】

请按下面顺序帮我展开：

1. 先定义这个问题到底在决定什么
   - 是在决定规则放在哪一层
   - 是在决定谁负责维护
   - 是在决定如何执行
   - 还是在决定是否值得上升为正式治理规则
   - 如果我把几个层级混在一起了，请拆开说明

2. 先补齐常见治理路径
   - 这个场景下常见的做法有哪些
   - 请至少列出 A / B / C
   - 如果存在新手容易忽略的中间层方案，也请补上

3. 澄清核心概念
   - 规则归属、生效范围、责任主体、执行方式、临时策略、长期规则分别是什么意思
   - 哪些概念经常被混用
   - 它们分别影响什么

4. 比较这些方案
   - 从治理成本、执行稳定性、可维护性、误伤风险、团队协作成本等角度比较
   - 明确每种方案更适合什么，不适合什么

5. 结合我的场景给出建议
   - 当前更推荐哪种归属和执行方式
   - 为什么
   - 这个建议依赖哪些前提
   - 如果团队规模、稳定性或约束强度变化，推荐会怎么变化

6. 如果现在还不适合拍板
   - 请列出最值得继续确认的 3 到 5 个问题

输出要求：
- 先补齐治理路径，再进入推荐
- 不要把“能做自动化”直接等同于“就应该自动化”
- 如果我跳过了责任主体或生效范围，请明确指出
- 如果我的问题其实跨了规则归属、执行方式和流程设计多个层级，请拆开回答
```

## 规则落点判断版

```text
我正在判断某条规则或约束应该放在哪里。

规则 / 约束内容如下：
【把内容贴在这里】

请按下面结构输出：
1. 这条规则到底在约束什么
2. 它属于长期规则还是阶段性策略
3. 它更适合放在个人级、项目级、团队级还是组织级
4. 它更适合写成文档、规则文件、脚本检查，还是人工约定
5. 为什么
6. 这样做的风险和边界是什么
7. 如果现在还不适合正式固化，过渡方案是什么

要求：
- 明确区分“规则内容”和“执行方式”
- 明确区分“值得长期沉淀”和“先临时收口”
- 如果存在更轻量的过渡方案，请不要省略
```

## 思路决策文档版

```text
请把下面这份关于“项目治理 / 规则归属 / 职责边界”的草稿，整理成一份“思路决策文档”。

不要把它写成最终定版规则，而是写成帮助我判断治理方式的文档。

请按下面结构输出：
1. 这个问题在决定什么
2. 核心概念与边界
3. 常见治理路径
4. 比较维度
5. 当前草稿已提到的方案
6. 当前草稿遗漏但值得补充的方案
7. 当前待确认问题
8. 阶段性倾向
9. 后续建议继续追问的方向

要求：
- 把“规则内容”“生效范围”“执行方式”分开
- 把“阶段性策略”和“长期规则”分开
- 如果草稿里直接跳到了某种实现方式，请回补治理层判断
- 输出要适合长期维护

原始草稿如下：
【把草稿贴在这里】
```

## 高频补问

- `请先告诉我，我现在是在决定规则归属、生效范围，还是执行方式。`
- `这条约束更适合项目级还是团队级？为什么？`
- `它应该写成规则、文档、脚本检查，还是人工约定？`
- `如果现在不适合正式固化，过渡方案应该是什么？`
- `这条规则的责任主体应该是谁？`
- `自动化检查和人工约定各自的边界是什么？`
- `这个场景下最容易把什么治理问题混在一起？`

## 当前判断

- 这是对母 prompt 在“治理与归属判断”场景下的领域化收窄
- 它适合承接你仓库里很多“规则到底放哪、由谁负责”的问题
- 后续如果“系统级 vs 项目级”“规则 vs hooks vs scripts”持续增长，可以从本文件继续拆出更细主题
