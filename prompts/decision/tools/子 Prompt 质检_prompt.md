# 子 Prompt 质检 Prompt

> **文档职责**：为 `decision/domains/` 下的领域子 prompt 提供统一的最小质检标准，并提供可直接投喂给大模型的质检 prompt。
> **适用场景**：新建子 prompt、重构子 prompt、怀疑某个文件边界模糊或质量不稳定时。
> **目标读者**：本仓库维护者。
> **维护规范**：这份文件既是检查标准，也是可执行 prompt；优先保持短、硬、可直接使用，不要堆成泛泛建议。

## 这份文件的定位

这份文件不只是“给人看的清单”，也应该是“给大模型执行的质检 prompt”。

原因很直接：

- 你是使用 prompt 的人，不是人工审稿工具
- 这套标准如果不能直接交给模型执行，复用成本会偏高
- 作为 `decision` 目录的配套工具，它应该服务“快速检查”，而不是要求你每次手工对照

默认作用域要明确：

- 默认检查对象是 `decision/domains/` 下的候选领域子 prompt
- 不默认检查 `README.md`
- 不默认检查 `decision/core/`
- 不默认检查 `decision/tools/`

所以它同时承担两件事：

- **质检标准**：定义什么样的子 prompt 才算过关
- **质检 prompt**：让模型按统一标准帮你检查

## 直接可用版 Prompt

适用场景：

- 你刚写完一个新的子 prompt
- 你想让大模型帮你判断它是否适合收进 `decision` 目录
- 你怀疑某个文件边界不清、结构不稳、和别的文件重叠

```text
请把下面这份文档当成 `decision/domains/` 下候选“子 prompt”，按严格质检方式帮我检查。

你的任务不是润色，而是判断它是否真的适合作为 `decision/domains/` 中的独立子 prompt。

请按下面顺序输出：

1. 先给总判断
   - 适合直接收录
   - 适合修改后收录
   - 暂不适合收录

2. 一票否决项检查
   - 是否有明确主决策对象
   - 是否已经高频复用，而不是一次性问题
   - 是否和现有子 prompt 边界基本清楚
   - 是否其实更像补问模板
   - 是否其实更像 rule / 规范文档

3. 结构检查
   - 是否具备清晰的职责、适用场景、目标读者、维护规范
   - 是否有“这个子 Prompt 解决什么问题”
   - 是否有“常见概念”
   - 是否有“标准版”或等价主体 prompt
   - 是否有“高频补问”
   - 是否有“适合沉淀成笔记的输出结构”
   - 是否有“当前判断”

4. 决策对象检查
   - 文件名是否能直接说明它在帮助决定什么
   - 正文是否明确说清“这个问题到底在决定什么”
   - 是否把多个大决策层级混在一起
   - 它是否真的是一个稳定决策主题

5. 方案空间检查
   - 是否会先补齐常见方案空间，再进入推荐
   - 是否显式提醒新手容易漏掉的方案
   - 是否默认用户已经知道行业常见路径
   - 是否一上来就让模型拍板

6. 概念边界检查
   - 是否先澄清核心概念
   - 是否指出容易混淆的概念
   - 是否把“概念、方案、推荐、待确认问题”分开
   - 是否把实现细节误当成核心决策

7. 边界重叠检查
   - 它和相邻子 prompt 的区别是否清楚
   - 是否像从别的文件复制后只改标题
   - 高频补问是否和别的文件大面积重复

8. 维护成本检查
   - 未来是否有明确维护价值
   - 是否主题过大，容易失控
   - 是否主题过小，可能低频
   - 是否已经能看出未来拆分方向

9. 最后给出动作建议
   - 直接收录
   - 并入现有子 prompt
   - 降级为补问模板
   - 暂缓收录，继续验证
   - 更适合沉淀为 rule / checklist

输出要求：
- 先判断是否适合收录，不要先帮我润色
- 要明确指出最关键的问题，不要泛泛而谈
- 如果你认为不适合独立存在，请明确说出它更适合放到哪里
- 如果你认为适合收录，也要指出最需要补强的 1 到 3 个点

候选子 prompt 如下：
【把文档贴在这里】
```

## 精简版 Prompt

适用场景：

- 你只想快速判断“这个文件能不能进 `decision`”
- 你不需要完整质检报告，只要短判断

```text
请帮我快速检查下面这份文档，判断它是否适合收进 `decision/domains/`，作为一个独立子 prompt。

请只回答这 5 件事：
1. 它的主决策对象是否明确
2. 它是否和现有子 prompt 明显重叠
3. 它是否具备稳定结构
4. 它是否符合 `decision` 风格（先补盲、再比较、再推荐）
5. 最终建议是：收录 / 修改后收录 / 暂不收录

如果不适合，请直接告诉我它更适合：
- 并入现有子 prompt
- 放进补问模板
- 做成 rule / checklist

候选文档如下：
【把文档贴在这里】
```

## 重构建议版 Prompt

适用场景：

- 你已经知道这份子 prompt 不够好
- 你希望模型不只检查，还告诉你应该怎么改

```text
请把下面这份候选子 prompt 当成 `decision/domains/` 的候选文件，先严格质检，再给出重构建议。

请按下面结构输出：
1. 是否适合收录
2. 最关键的 3 个问题
3. 它当前更像什么
   - 独立子 prompt
   - 补问模板
   - rule / checklist
   - 还不稳定的草稿
4. 如果要收录，最少要补哪几个部分
5. 如果不该独立，建议并到哪个现有文件
6. 给一个更合适的结构草案

要求：
- 重点是结构和边界，不是措辞润色
- 如果问题在于决策对象不清，要明确指出
- 如果问题在于和现有文件重叠，要明确指出重叠点

候选文档如下：
【把文档贴在这里】
```

## 人工检查标准

如果你不想直接把文档丢给模型，也可以手工按下面标准快速过一遍。

### 一票否决项

只要下面任一项不满足，这个文件就不应该直接收进 `decision/domains/`：

- [ ] 这个文件有明确的主决策对象，不是在泛泛讲一个领域
- [ ] 这个主题已经高频复用，不是一次性问题
- [ ] 它和现有子 prompt 的边界基本清楚，不会明显重复
- [ ] 它不是补问模板伪装成子 prompt
- [ ] 它不是 rule / 规范文档伪装成子 prompt

### 结构检查

- [ ] 文件开头有清晰的职责、适用场景、目标读者、维护规范
- [ ] 有“这个子 Prompt 解决什么问题”小节
- [ ] 有“常见概念”小节
- [ ] 有“标准版”或等价的主体 prompt
- [ ] 有“高频补问”小节
- [ ] 有“适合沉淀成笔记的输出结构”小节
- [ ] 有“当前判断”小节

如果缺了 2 项以上，通常说明结构还没稳定。

### 决策对象检查

- [ ] 文件名能直接说明“它在帮助决定什么”
- [ ] 文档正文里能明确说清“这个问题到底在决定什么”
- [ ] 没有把两个以上的大决策层级长期混在一起
- [ ] 它更像一个稳定决策主题，而不是某个主题下的零散小问题

常见反例：

- 文件标题很大，但正文其实只覆盖一个小角落
- 文件标题很小，但正文其实混了多个层级
- 主题名是工具名，不是决策对象

### 方案空间检查

- [ ] 这个子 prompt 会先补齐常见方案空间，再进入推荐
- [ ] 它会显式提醒“新手容易漏掉的方案”
- [ ] 它不会默认用户已经知道行业常见路径
- [ ] 它不是一上来就让模型拍板

### 概念边界检查

- [ ] 它能帮助用户先澄清核心概念，而不是直接跳结论
- [ ] 它会指出容易混淆的概念
- [ ] 它会把“概念、方案、推荐、待确认问题”分开
- [ ] 它没有把“实现细节”误当成“核心决策”

### 输出质量检查

- [ ] 它要求“先解释，再比较，再推荐”
- [ ] 它要求说明适用场景，而不只讲优点
- [ ] 它要求暴露前提和不确定性
- [ ] 它会要求列出仍待确认的问题
- [ ] 它能产出可沉淀为“思路决策文档”的结果

### 边界重叠检查

- [ ] 我能清楚说出它和相邻子 prompt 的区别
- [ ] 这个文件不是通过复制别的子 prompt 再小改标题得到的
- [ ] 高频补问没有和别的文件大面积重复
- [ ] 如果有少量交叉，这种交叉是合理的，不会导致使用时选不出来

如果你无法在 1 到 2 句话里讲清它和现有文件的区别，通常就不该新建。

### 维护成本检查

- [ ] 这个文件未来有明确维护价值
- [ ] 它不会因为主题过大而很快失控
- [ ] 它不会因为主题过小而几乎不再使用
- [ ] 如果未来继续膨胀，我已经能看出可拆分方向

---

## 最终判断

如果大部分项都满足，可以收进 `decision/domains/`。

如果问题主要集中在下面情况，优先这样处理：

- 主题太小：先并入现有子 prompt
- 结构未稳：先放真实使用里继续验证
- 只是追问短句：先并入 [提问补问模板.md](./%E6%8F%90%E9%97%AE%E8%A1%A5%E9%97%AE%E6%A8%A1%E6%9D%BF.md)
- 已经是稳定约束：考虑沉淀为 rule / checklist
