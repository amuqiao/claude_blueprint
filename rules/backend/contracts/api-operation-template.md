# 新接口接入模板

本模板规定新增 HTTP 接口时必须形成的接口契约文档结构。它用于保证接口文档能直接说明字段含义、必选/可选、类型、约束、`null` 与省略语义、成功/失败示例、错误码、日志和验收要求。

本文只定义接口文档模板和接入检查，不替代 `service-contract.md` 的 envelope、错误码和时间格式规则，也不替代具体业务 schema。

## 接口基本信息

每个公开接口必须先定义基本信息：

| 项目 | 说明 |
| --- | --- |
| `operation_id` | 稳定操作 id，用于 OpenAPI、SDK、日志和 contract test 对齐。 |
| 接口名称 | 面向维护者的人读名称。 |
| 方法与路径 | 例如 `POST /api/v1/jobs`。 |
| 调用方 | 哪些服务、用户或租户可以调用。 |
| 鉴权 | 鉴权方式、凭证来源、权限边界。 |
| 模式 | 同步、异步 Job、callback、只读查询或写入操作。 |
| 幂等 | 幂等键字段、唯一范围、重复请求语义。 |
| 副作用 | 是否创建任务、写数据库、调用模型、写第三方系统。 |
| 版本 | 接口版本、兼容策略和废弃策略。 |

## 请求字段

请求字段必须用字段表说明。字段表至少包含：

| 字段 | 必选 | 类型 | 约束 | `null` / 省略语义 | 说明 |
| --- | --- | --- | --- | --- | --- |
| `field_name` | 是 / 否 | string / integer / object / array / enum | 长度、范围、枚举、格式、大小限制 | 是否允许 `null`；省略时如何处理 | 字段业务含义和来源 |

规则：

- 未声明字段默认拒绝。
- 必选字段必须说明缺失时的错误码。
- 可选字段必须说明省略、`null`、空字符串、空数组和默认值的差异。
- 对象字段必须继续展开子字段表；不要只写 “object”。
- 数组字段必须说明 item 类型、最小/最大长度、去重和顺序语义。
- 枚举字段必须列出所有允许值和兼容新增策略。
- 时间字段必须遵守 `service-contract.md` 的时间格式规则。
- 大文本、文件、URL 和对象存储引用必须说明大小、类型、hash 或安全限制。

异步 Job 创建接口的业务字段应位于 `job_params`，不要把具体 `job_type` 字段摊平到通用顶层。

## 成功响应

成功响应必须套用统一 envelope，并说明 `data` 的业务结构。

成功响应文档至少包含：

| 项目 | 说明 |
| --- | --- |
| HTTP status | 例如同步成功 `200`，异步创建成功 `202`。 |
| `code` | 成功固定为 `OK`。 |
| `msg` | 成功短消息；不作为机器判断依据。 |
| `data` | 业务成功数据结构，字段表必须展开。 |
| `request_id` | 请求追踪 id。 |
| `server_time` | Unix 秒级服务端时间。 |

`data` 字段也必须提供字段表：

| 字段 | 必选 | 类型 | 约束 | `null` / 省略语义 | 说明 |
| --- | --- | --- | --- | --- | --- |
| `id` | 是 | string | 稳定资源 id | 不允许 `null` | 资源唯一标识 |

无数据接口必须稳定声明 `data` 为 `{}` 还是 `null`，不能同一接口混用。

成功 `data` 字段的展开规则与请求字段一致：对象字段必须继续展开子字段表，数组字段必须说明 item 类型、最小/最大长度、去重和顺序语义，枚举字段必须列出允许值和兼容新增策略。不要只写 “object” 或 “array” 让调用方自行猜结构。

## 错误响应

错误响应必须使用服务级错误码、项目子错误码和真实 HTTP status。不要为了统一 envelope 全部返回 `200`。

错误表至少包含：

| HTTP status | `code` | `sub_code` | 场景 | `retryable` | 说明 |
| --- | --- | --- | --- | --- | --- |
| `422` | `INVALID_INPUT` | `COMMON.INVALID_FIELD` | 字段缺失、类型错误、schema 校验失败 | false | 调用方修正输入后重试 |

错误响应示例必须至少覆盖一个请求校验失败。涉及外部依赖、AI provider、数据库或权限时，应补对应错误示例。

对外错误不得包含底层库异常类名、堆栈、密钥、完整供应商响应、完整 Prompt、隐私文本或大载荷。

## 示例要求

接口文档必须提供：

- 最小合法请求示例。
- 典型完整请求示例。
- 成功响应示例。
- 至少一个错误响应示例。
- 如果是异步 Job，补创建成功、查询运行中、成功终态、失败终态和 callback 示例或链接到对应 `job_type` 文档。

示例必须与 schema、OpenAPI 和 contract tests 对齐。示例不是事实源；冲突时修正 schema 或示例，不在文档里临时解释差异。

## 可观测与日志

接口文档必须说明可观测字段：

| 字段 | 说明 |
| --- | --- |
| `request_id` | 串联 HTTP 请求日志。 |
| `trace_id` | 跨进程链路 id；异步 Job 应传给 Worker。 |
| `caller_id` | 调用方或租户摘要。 |
| 业务 id | 资源 id、`job_id` 或其他可排查对象。 |
| 日志事件 | 至少包括请求完成、失败分类和关键外部调用。 |

日志只记录摘要，不记录完整请求体、完整响应体、密钥、token、完整 Prompt、隐私文本或大文件内容。

## 接入实现清单

新增接口实现前必须确认：

- [ ] `operation_id` 已定义且稳定。
- [ ] 请求字段表完整，包含字段说明、必选/可选、类型、约束、`null` 与省略语义。
- [ ] 成功 `data` 字段表完整。
- [ ] 错误码表包含 HTTP status、服务级 `code`、项目 `sub_code` 和 `retryable`。
- [ ] 请求 schema 拒绝未声明字段。
- [ ] route 不复制 envelope、不返回裸错误对象。
- [ ] Service 承担业务判断，Repository 只做数据访问。
- [ ] 外部依赖通过 adapter 转换错误。
- [ ] OpenAPI 或 schema 快照已更新。
- [ ] Contract tests 覆盖成功、校验失败和至少一个业务失败。
- [ ] 日志字段可通过 `request_id`、`caller_id` 和业务 id 排查。

## 文档维护

接口文档、OpenAPI、SDK 类型和 README 示例都是投影。字段事实源应落在 schema、错误码注册表、handler registry 和 contract tests。接口变更时必须先判断是否为兼容变更；破坏性变更必须定义新版本、迁移窗口、调用方灰度和回滚方案。
