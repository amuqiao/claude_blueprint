# 黑盒学习文档规则

## 核心目标

学习型文档应帮助读者先会使用和判断，再逐步理解内部实现。

不要一开始解释源码、算法或底层机制；应先说明每个关键模块解决什么问题、什么时候需要、输入是什么、输出是什么、成功标志是什么、下一步接哪里。

## 适用场景

- 新用户需要理解复杂项目如何跑通。
- 项目包含多个脚本、页面、模型、服务、配置或流水线。
- 用户不需要马上研究内部算法，但需要知道每一步为什么存在。
- 需要把项目整理成可学习、可复述、可迁移的文档。

## 文档定位

这类文档不是纯快速上手，也不是源码内部机制说明，而是中间层：

场景化流程 + 黑盒模块说明 + 输入输出参数 + 判断信号。

## 每个模块应说明

- 模块名
- 它解决什么问题
- 什么时候需要
- 输入是什么
- 输出是什么
- 关键参数会影响什么
- 成功标志
- 常见错误
- 是否可跳过
- 下一步接哪里

## 禁止倾向

- 不要把文档写成按钮说明书。
- 不要一开始拆内部算法。
- 不要只列目录、命令或配置项。
- 不要让读者自己拼流程。
- 不要把主流程、可选流程和排查内容混在同一层级。
