# 实现选型规则

## 核心原则

小而美的应用应优先复用成熟组件和平台原生能力，用少量适配承接产品差异，不默认重新造轮子。

## 选型顺序

实现 UI、交互、状态、存储、路由、动画、手势或基础设施能力时，优先按以下顺序判断：

1. 平台原生能力是否可以直接承接。
2. 项目已接入的成熟组件是否可以复用。
3. 成熟组件是否可以通过少量参数、包装或组合完成适配。
4. 只有在上述方案无法表达产品关键差异时，才新增自定义实现。

## 自定义边界

允许自定义：

- 产品识别度强、成熟组件无法表达的核心视觉或交互。
- 业务领域模型、数据生命周期、权限边界等领域规则。
- 对成熟组件的薄封装，用于统一项目语义、样式或错误处理。

避免自定义：

- 平台或成熟组件已经稳定支持的基础行为。
- 为了“更灵活”提前抽象出的通用组件。
- 复制成熟组件内部机制后再局部改造的实现。
- 用大量手写状态、偏移量、手势计算或隐式同步维护的 UI。

## 适配要求

少量适配应遵守成熟组件的原始契约。封装组件前必须理解其约束、断言和失败模式，不应通过绕开组件契约来获得短期效果。

当封装平台组件时：

- 保留平台组件的核心行为和可访问性语义。
- 只暴露项目真正需要的参数。
- 对运行时约束补测试，尤其是只在特定组合下触发的断言。
- 不吞掉底层错误；违反契约时应快速暴露并修正封装。

## 判断信号

如果一个实现出现以下信号，应回到选型层重新判断：

- 需要大量魔法数或手写布局补丁。
- 需要模拟成熟组件已经提供的状态机。
- 需要持续修补平台组件本来会处理的边界情况。
- 组件封装比直接使用成熟组件更难理解、更难测试。
- 产品差异很小，但自定义代码承担了大量基础设施责任。
