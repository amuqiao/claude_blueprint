# 项目解析规则

目标：面对陌生项目，建立"可运行、可使用、可追踪、可定位"的心智模型，并产出读者可自学的解析文档。目标不是讲完所有文件，而是让读者能理解主线、跑通项目、追踪主流程，并知道从哪里深入。

职责边界：本规则只约束项目解析的执行顺序、取证要求、可信度标注和产出结构。通用文风、排版规范和通用心智模型抽象不在本规则范围内；当其他规则同时适用时，本规则只提供项目解析特有约束，不与通用规则争夺表达层职责。本规则单独生效时完整可执行，不依赖任何外部规则文件。

## 1. 解析执行顺序（分析侧）

按以下顺序读项目，禁止一上来通读全部源码：

```text
1. README / 文档目录          -> 项目自述的目标和用法
2. 构建与依赖清单             -> package.json / pyproject.toml / go.mod /
                                Cargo.toml / Makefile / Dockerfile / compose 文件
3. 入口文件                   -> main / cli / server 启动点 / scripts 定义的命令
4. 配置文件与环境变量示例      -> .env.example / config 目录 / 默认值
5. 沿一条真实主流程追调用链    -> 从入口到输出，只追一条，不发散
6. 数据与产物目录             -> 输入样例、输出目录、缓存、日志
7. 有条件时实际运行验证        -> 安装、启动、跑通最小路径
```

规则：

- 每一步只为回答一个问题服务，答案够了就前进，不恋战。
- 若项目提供明确的 quickstart 或一键 demo，可在第 3 或第 4 步后提前做最小运行验证，尽早校准 README 声称与实际行为。
- 追调用链时遇到的工具函数、边缘分支，记录位置即可，不展开。
- 测试目录是理解模块预期行为的捷径，优先于读实现。
- 无法运行的环境下，明确降级为"静态推断模式"并在产出中声明（见第 2 节）。

## 2. 可信度约束

产出中的每个操作性结论必须可区分来源：

- **已验证**：实际执行过命令、看到过输出。可直接陈述。
- **代码推断**：从源码/配置读出但未运行。须标注"（依据 xx 文件推断，未运行验证）"。
- **不确定**：找不到依据。须明确写"未能确认"，禁止编造。

硬性禁止：

- 禁止编造未验证的启动命令、端口、输出路径、成功输出样例。
- 禁止把 README 的声称当作已验证事实转述（README 可能过时）。
- "成功信号"字段只允许写已验证内容或明确标注的推断。

## 3. 产出契约

默认交付物与拆分标准：

- **小项目**（单入口、单进程、核心代码可在数小时内读完）：产出一份解析文档，覆盖第 5 节认知路径的全部节点，放在项目根目录或指定输出目录，命名 `PROJECT-ANALYSIS.md`。
- **大项目**（多服务、多入口、monorepo）：先产出一份「项目总览」，再按第 6 节视角拆分独立文档，放入 `docs/analysis/` 之类的独立目录。总览必须先交付，其余按需展开。
- 用户指定了范围（如"只看数据管道"）时，契约收窄到该范围，但仍须包含输入、输出、成功信号。
- 上述命名和目录为无约定时的默认值；若仓库已有文档命名或目录约定（如 `docs/`、`wiki/`），优先沿用现有约定。

每篇文档开头固定回答四问：

```text
这篇文档解决什么问题 / 适合谁读 / 读完获得什么能力 / 不解决什么问题
```

## 4. 核心写作原则

- 从功能入手，不从目录入手。
- 先跑通最小路径，再解释完整链路；先场景闭环，再模块边界。
- 先讲输入、输出和成功信号，再讲内部实现。
- 一篇文档承担一个稳定视角，独立可读，不依赖读者先读另一篇。
- 目录树、类名、算法名只有在解释流程、边界或判断信号时才进入正文。
- 保持三层信息分离，不混写在同一段：
  - 操作层（怎么跑通）/ 结构层（由哪些稳定部分组成）/ 判断层（遇到变化怎么选路径）。

## 5. 文档认知路径（组织侧）

```text
项目目标 -> 环境安装 -> 快速上手 -> 应用场景
        -> 完整调用链 -> 功能模块 -> 数据和产物
        -> 配置和运行时 -> 深入专题入口
```

这是认知顺序而非固定目录。可按项目规模合并章节，但默认不打乱前后依赖：调用链默认不应早于快速上手出现。例外：库、SDK、编译器等非应用型项目，或无法本地运行的项目，可先给出静态调用链，须按第 2 节标注为静态推断模式，并说明为何调整顺序。

单篇内部顺序：心智模型 -> 主流程 -> 输入输出 -> 关键参数或分支 -> 成功信号 -> 常见失败 -> 深入方向。操作型文档主流程优先，解释型文档心智模型优先；两者都不得省略"输入、输出、成功信号"。

## 6. 视角、取证入口与默认字段

| 视角 | 回答的问题 | 优先取证位置 | 默认字段 | 禁止 |
| --- | --- | --- | --- | --- |
| 项目总览 | 做什么，边界在哪 | README、依赖清单、入口文件 | 目标、能力、入口、主干流程、关键产物 | 展开源码细节 |
| 环境安装 | 怎么装起来 | 依赖清单、Dockerfile、CI 配置、lock 文件 | 依赖、版本、配置、启动前检查 | 混入业务原理 |
| 快速上手 | 第一个成功结果 | scripts 命令、examples、Makefile、实际运行 | 最短路径、最小输入、命令、成功信号 | 覆盖所有场景 |
| 场景工作流 | 有 X 想得到 Y 怎么走 | WebUI/API/CLI 入口、examples、测试用例 | 见下方场景字段 | 深入算法内部 |
| 调用链 | 一次任务穿过哪里 | 入口文件、路由/命令注册、核心服务 | 见下方节点字段 | 逐行源码注释 |
| 功能模块 | 每块负责什么 | 包/目录划分、导出接口、测试目录 | 能力、触发入口、主要文件、读写数据、边界、修改入口 | 机械翻译目录树 |
| 数据产物 | 运行后留下什么 | output/logs/cache 目录、migrations、.gitignore | 见下方产物字段 | 只列文件名 |
| 配置运行时 | 行为由什么控制 | .env.example、config 目录、代码内默认值 | 环境变量、配置文件、端口、模式、开关、默认值 | 重复安装步骤 |
| 深入入口 | 想改某块从哪读 | 模块主文件、对应测试、扩展点/插件注册 | 修改目标、优先入口、验证方式、风险边界 | 写成临时排查记录 |

小项目可合并或精简默认字段，但四项底线不得省略：输入、输出、成功信号、可信度标注。

**场景字段**（每个场景完整闭环，写清脚本/按钮吃什么、吐什么、谁消费）：

```text
适用条件 / 目标 / 准备 / 步骤（不跳步）/ 每步输出 / 成功信号 / 失败信号 / 下一步
```

**调用链节点字段**（只收录改变理解、排错、扩展或脚本化能力的节点）：

```text
职责 / 输入 / 输出 / 关键参数 / 失败点 / 可替换性（能否单独执行、重跑、替换）
```

**产物字段**（中间产物不是噪声，是新手理解流程的骨架）：

```text
名称或路径 / 来源步骤 / 消费者 / 格式关键字段 / 生命周期（可否删除重建）/ 排错价值
```

## 7. 规模适配

- 小项目只需回答第 8 节质量检查的前四问，其余视角一笔带过或省略。
- 边缘配置、罕见分支不为完整性而收录；收录实现细节的唯一条件是它解释了关键输入输出、调用链断点、模块边界、配置行为，或深入入口之一。否则下沉到源码注释或省略。

## 8. 质量检查

交付前逐条自检，任一为否则先补理解节点，不补概念：

```text
1. 读者是否知道项目能做什么
2. 读者是否能跑出第一个成功结果
3. 读者是否知道典型场景该走哪条路
4. 读者是否知道一次任务经过哪些入口和模块
5. 读者是否知道关键输入、输出和中间产物
6. 读者是否知道哪些步骤可跳过、可重跑、可替换
7. 读者是否知道想深入某块该从哪里继续
8. 所有操作性结论是否已按第 2 节标注可信度
```

## 9. 反模式

- 一上来贴目录树或讲框架、算法、类名，没有先讲目标和最小使用路径。
- 把快速上手、场景、调用链、模块混在一篇里互相打断层级。
- 只写操作步骤不写输入输出和成功信号；只写模块职责不写上下游和触发入口。
- 只写最终产物，不写中间产物如何连接流程。
- 为完整性堆边缘配置，破坏主线。
- 把 README 声称、未运行的命令当作已验证事实写入文档。
