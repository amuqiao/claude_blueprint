# Shell 与 Python 脚本稳定合同

脚本入口是稳定行为合同，不是临时命令拼装。入口必须让用户和自动化都能判断：怎么调用、读取什么、输出什么、留下什么、会改变什么、失败如何分类，以及用什么验证防止漂移。

本文面向仓库内项目脚本。仓库脚本应定位项目根目录并复用公共 helper；独立运维脚本或单文件工具可以定位脚本目录、安装前缀或显式配置目录，但仍应遵守本规则的合同、错误、输出、退出码和安全边界。

本文不讲 shell 教程，也不规定每个产物的文件名或目录布局。它只约束入口行为维度和合同稳定性。

## 设计原则

- 好用优先：脚本是为了降低使用成本；低风险、高频、只读命令不应被改造成参数堆砌。
- 合同唯一：help、参数、配置、输入、输出、JSON、产物、副作用、退出码和验证都必须有唯一 owner。
- 配置收口：配置真源、优先级和 profile 映射只在参数与配置合同中定义，其他层只引用。
- 高风险显式：生产、远端、写入、删除、上传、费用、迁移和真实任务目标不得藏在隐式配置或脚本默认值里。
- 输出可判定：默认人读输出能判断状态、目标、路径和失败原因；机器读输出保持纯净；退出码语义稳定。
- 错误快显：非法参数、缺少依赖、非法配置、危险运行环境和互斥选项直接失败，不做 silent fallback、自动修正或吞错兼容。

## 入口模型

写脚本前先回答两个问题：

```text
这个入口的合同 owner 是谁？
  shell、Python 脚本、Python 辅助能力，还是正式 Python CLI。

这个入口需要稳定哪些合同面？
  help、参数、配置、输入、输出、产物、副作用、退出码、验证。
```

一个合同面只能有一个事实源。shell wrapper 和 Python CLI 不得重复维护命令目录、参数列表、JSON 字段语义或退出码说明；外层只能转发、补足或引用。

入口形态按职责和用户合同选择：

| 模式 | 用户合同 owner | 使用边界 |
|---|---|---|
| 单 shell | shell | 外部命令编排、环境保护、生命周期动作、轻量分发。 |
| 纯 Python 脚本 | Python 脚本 | 单一职责、结构化处理、少量参数或一个 `--json`。 |
| shell + py | shell | shell 负责入口和编排，Python 只做单一复杂能力。 |
| shell + CLI Typer | Python CLI | 多子命令、复杂 help、多输出格式、安全确认组合或长期稳定接口。 |

入口演进是建议，不是强制结构。功能少且职责单一时可以保留独立小入口；同一职责域内动作增多且共享运行环境、安全边界、输出和退出码时，优先稳定入口加子命令；无法自然归入现有入口时，可以新增顶层入口。

## 稳定合同总表

| 合同面 | 必须回答 | 事实源 | 最小验证 |
|---|---|---|---|
| Help | 入口是什么、最小怎么开始、风险在哪里。 | `-h|--help`、子命令 help。 | help smoke。 |
| 参数 | 命令、参数、默认值、互斥、确认、非交互行为。 | shell dispatch、`argparse`、Typer、Click。 | 正常参数和失败参数。 |
| 配置与输入 | 读取哪些文件、环境变量、URL、stdin、远端资源；如何合并和失败。 | parser、配置读取函数、长期文档。 | 合法输入和非法输入。 |
| 输出 | 默认人读输出、JSON、stderr/stdout owner、敏感字段。 | 输出 helper、格式化模块、CLI parser。 | 人读 smoke、JSON 纯度。 |
| 产物 | 会生成哪些文件、日志、PID、manifest、报告；路径如何发现。 | 入口实现、help、成功输出。 | 路径可发现和产物存在。 |
| 副作用 | 是否写库、删除、上传、远端写入、启动进程、触发费用或迁移。 | help、确认参数、保护检查。 | 拒绝路径和确认路径。 |
| 退出码 | 各类失败如何分类，何时透传下层命令。 | 入口实现、help、测试。 | 失败分类 smoke。 |
| 验证 | 哪些命令证明合同没有漂移。 | `verify.sh`、CI、脚本 smoke。 | 语法、help、成功、失败、机器输出。 |

临时内部脚本只有同时满足“不被用户直接调用、不被 README/runbook/CI 引用、不被稳定入口外的自动化调用、不承载独立合同”时，才可以不按稳定入口维护。任一条件不满足，就按稳定入口处理。

## Help 合同

`-h|--help` 是入口合同 envelope，不是完整手册。用户打开 help 后，应能判断入口职责、最小用法、默认读写、输出位置、风险动作和退出码。

Help 只展示调用必须知道的合同面；业务背景、完整协议、排障手册放长期文档。实现变更影响调用、解析、产物、安全边界或失败处理时，必须同步更新 help 或被 help 指向的合同事实源。

Help envelope 栏目名必须从下列集合中选择。没有对应行为时可以省略；有对应行为时必须使用这里的固定栏目名，不得在单个脚本里临时发明 `Tips`、`Notes`、`Config`、`Artifacts`、`Danger`、`Examples` 等并行叫法。

```text
Usage / 用法
职责
Options / 选项
Arguments / 参数
Commands / 命令
作用域
不负责
运行环境
配置与环境变量
默认行为
输入要求
输出
关键概念
进阶用法
运行产物
副作用与保护边界
幂等性
成功标准
常用示例
补充格式
Exit Codes
```

栏目使用规则：

- 带 `/` 的栏目名表示中英文等价别名，例如 `Usage / 用法`、`Options / 选项`；框架生成的 `Usage`、`Options`、`Arguments`、`Commands` 可以直接作为对应栏目。
- 基础栏目是 `Usage / 用法`、`职责`、适用的 `Options / 选项`、`Arguments / 参数` 或 `Commands / 命令`、`输出`、`常用示例`、`Exit Codes`；单命令小工具按实际合同面裁剪，不强行补不适用栏目。
- 其他栏目按行为出现：有环境或工具链前提写 `运行环境`，有配置读取写 `配置与环境变量`，有默认动作或默认窗口写 `默认行为`，有输入文件或远端资源写 `输入要求`，产生文件或远端产物写 `运行产物`，有副作用写 `副作用与保护边界`，有复杂概念或高级路径分别写 `关键概念`、`进阶用法`。
- `补充格式` 只用于最小输入/输出格式、JSON 字段、manifest 片段或文件格式说明，不得作为杂项说明桶。
- 高风险命令必须包含 `副作用与保护边界`、`幂等性`、`常用示例` 和 `Exit Codes`；远端、写入、费用、迁移或生命周期命令还应说明 `作用域` 或 `不负责`。
- `幂等性` 说明重复执行会覆盖、跳过、追加、复用、清理还是失败；如果动作不幂等，必须说明二次执行风险。
- `成功标准` 用于 check、status、inspect、聚合命令、生命周期命令和长耗时任务，说明什么状态算成功完成。
- `常用示例` 必须优先选择可复制、频繁使用、最低前置条件的路径；示例命令复制后应只需要替换少量显式参数即可运行。
- 远端、写入、上传、费用、迁移或生命周期示例必须显式展示关键目标和保护参数；不得使用依赖隐藏上下文、未声明环境变量、未说明 `.env`、临时文件名或当前目录假设的示例凑数。
- 所有稳定入口至少提供一条常用示例；复杂、高风险或多输出命令按场景补充 2 到 3 条示例，不枚举所有参数组合。
- 新增栏目名必须先更新本规范，不能在单个脚本里局部扩展。
- 顶层 help 回答“入口是什么、有哪些命令、最小怎么开始”；子命令 help 回答“这个动作如何配置、复制和修改”。
- 框架 CLI 应让框架生成 `Usage`、`Options`、`Arguments` 和 `Commands`；手写 epilog 只补作用域、默认行为、配置、输出、产物、安全确认和退出码。
- `-h|--help` 必须在任何执行动作前处理，尤其是写库、上传、远端访问、费用、服务生命周期、迁移和长耗时任务。

## 参数与配置合同

参数名、默认值、互斥关系、确认参数和非交互行为属于稳定用户合同，不得随实现方便任意变更。

- 参数解析必须有唯一事实源。shell 手写解析、`argparse`、Typer 或 Click 中只能有一层拥有参数合同。
- 非法命令、未知参数、缺少必填值、互斥冲突、非法枚举和非法范围必须非 0 退出，通常按 `2` 处理。
- 布尔风险开关使用清晰确认参数，例如 `--yes`、`--confirm-*`、`--allow-*`、`--force`、`--dry-run`；不得用隐式环境或交互猜测替代。
- 机器读模式统一使用 `--json` 作为参数名。是否支持由具体命令决定；除非底层正式 CLI 已有稳定合同，不另造 `--format json`、`--machine`、`--raw-json`。
- 部署、运维、远端访问、写入和费用目标默认使用显式参数，例如 host、目录、URL、端口、target、namespace、kube-context、profile、service、远端路径。
- 改名、删除或改变参数默认值会改变用户合同；除非仍是临时内部脚本，否则应在 help、README 或迁移说明中显式说明。

配置真源优先级固定：

```text
显式命令参数
  > 环境变量
  > --profile 指定配置文件
  > 默认 .env
  > 脚本低风险 const 默认值
```

配置规则：

- 默认 `.env` 是项目默认配置真源；dotenv 可以服务低风险本地开发或只读命令，但必须在 help 或长期合同中可发现。
- 配置文件切换统一使用 `--profile <name>`；profile 到配置文件的映射、路径解析、缺失行为和非法格式行为必须明确。
- 除非底层正式 CLI 已有稳定合同，否则不得为配置切换另造 `--env-file`、`--config`、`--target-file` 等并行入口。
- 环境变量优先级高于 profile 和默认 `.env`，但低于本次显式命令参数；读取环境变量时声明关键变量和覆盖规则，不默认读取或打印整个环境。
- 超时、tail 行数、limit、本地端口、缓存目录、默认输出路径、默认输出模式等低风险运行体验项，可以保留脚本 const 默认值，并允许参数覆盖。
- 生产、远端、写入、删除、上传、真实费用、迁移和真实任务目标不得由默认 `.env`、隐藏配置源或脚本 const 默认值单独决定；必须由显式参数、profile、环境变量、确认参数或执行前目标展示共同约束。
- wrapper 或公共 helper 不得叠加未声明的多套 dotenv、target 文件、隐藏配置目录或脚本内高风险默认值。

## 输入合同

输入合同定义入口读取什么，以及格式和失败方式是什么。

- 读取文件、路径、URL、service name、target name 等外部输入时，先校验允许字符或允许集合。
- 配置文件默认按白名单 key 解析；不得对不可信配置使用 `source` 或动态执行。
- 输入文件必须声明格式、路径解析基准、缺失行为、非法格式行为和是否允许空文件。
- 配置、dotenv、profile 和环境变量读取必须遵守参数与配置合同；本节只补充具体输入格式、路径基准、缺失和非法输入行为。
- 网络调用必须显式设置超时和失败判定；不得把 HTTP、DNS、TLS 或连接失败当作成功继续。
- 使用 `curl` 时保留证书校验；除非入口合同明确是调试命令，否则不得使用 `--insecure` 或等价跳过校验选项。
- 脚本必须声明平台假设。使用 GNU/BSD 行为差异明显的 `sed`、`date`、`readlink`、`stat`、`xargs` 等工具时，写成项目验证环境可稳定运行的形式，或在前置检查中 fail-fast。
- 执行前或成功输出中应展示最终解析后的关键目标，至少覆盖会改变风险边界的 host、目录、URL、端口、环境、namespace、service、target、profile；`--json` 模式下进入该命令自己的 JSON 合同。

## 输出与产物合同

默认输出面向人读，必须能直接判断状态、摘要、关键对象、路径和失败原因；不得默认输出原始 JSON、大段 payload、二进制内容、CSV 字段直排或未整理工具噪声。

人读输出：

- 正常状态、摘要、路径、表格输出到 `stdout`。
- 错误原因、非法参数、缺少依赖、warning、进度、确认提示和诊断输出到 `stderr`。
- 多对象、多资源或多阶段输出必须提供摘要、单位、分组和关键关联关系。
- 长路径、大字段、重复字段和低频诊断默认压缩或分层展示；完整值通过 `--json`、`--verbose`、`--debug` 或入口声明的等价模式获取。

JSON 输出：

- `--json` 可以定义在顶层默认动作，也可以定义在具体子命令；只绑定它所附着的命令层级。
- 每个支持 `--json` 的命令独立拥有 JSON 合同；字段语义、默认窗口、过滤条件和输出结构不自动继承其他命令。
- 支持 `--json` 的命令必须在自己的 help 中可发现。
- 跨级写法例如 `entry --json <subcmd>` 不得依赖框架隐式行为；入口必须明确支持并路由，或按参数错误返回 `2`。
- 提供 `--json` 时，`stdout` 只能包含单个完整 JSON 文档，不混入说明文字、日志前缀、阶段标题、人读摘要、确认提示或后缀。
- 机器可读模式只能有一个 `stdout` owner。若 Python 或 Typer 输出 JSON，外层 shell 必须保持 `stdout` 静默，诊断走 `stderr`。
- 不支持 `--json` 的命令必须拒绝该参数并按 `2` 返回，不能静默忽略。
- JSON 字段语义属于机器合同；删除、改名或改变含义必须按合同变更处理。

敏感输出：

- 字段名、参数名或环境变量名命中 `token`、`password`、`passwd`、`secret`、`key`、`credential`、`authorization`、`cookie` 时默认按敏感信息处理。
- 日志和错误信息默认不得输出敏感值原文；确需定位时，只输出字段名、来源、长度、摘要或脱敏片段。
- 脚本不默认读取、打印或持久化 token、密码、完整请求体、完整模型响应、大型 payload 或二进制内容。

产物：

- 文件产物路径必须可发现。路径固定或可预测时写在 help 或长期合同；运行后才确定时，成功输出或诊断输出实际路径。
- 产物包括日志、PID、manifest、CSV、HTML、下载目录、模型 repository、报告、临时保留目录、远端创建目录等。
- 产物应区分默认生成、参数触发生成、临时自动清理和失败后保留。
- 危险覆盖、删除或远端清理必须有显式确认参数。
- 含敏感内容的临时文件、日志片段、报告或中间产物必须限制为当前用户可读写；不能保证权限时不得写入敏感内容。
- 临时文件使用 `mktemp` 并用 `trap` 清理；清理逻辑不得吞掉原始退出码。

## 副作用与安全合同

只读入口不得顺手修复环境、安装依赖、下载模型、写库、清理远端或启动后台任务。

副作用规则：

- 写库、删除、恢复、上传、远端写入、远端生命周期、真实费用、迁移、启动/停止进程、真实任务、压测放大必须有显式确认参数。
- 非交互环境下，没有显式确认参数必须失败；用户取消必须稳定非 0，不能伪装成功或空结果。
- 确认提示只能输出到 `stderr`；`--json` 模式不得混入确认提示。
- 入口发现未知资源、冲突运行模式或目标不归自己管理时 fail-fast，默认按 `3` 处理，不得静默复用、覆盖或清理。
- 单实例、锁、PID、端口、后台任务、远端任务句柄必须定义冲突行为；默认发现冲突即 fail-fast，只有显式参数允许等待、重试或抢占。
- 持有临时文件、锁、子进程、端口、后台任务或远端任务句柄的脚本，必须在 `EXIT`、`SIGINT`、`SIGTERM` 下走一致清理路径，并保留原始失败或中断语义。

Shell 安全基线：

- 变量展开默认加引号，例如 `"$path"`、`"$@"`。
- 参数列表默认使用数组或直接参数转发，不用字符串拼接命令。
- 禁止对不可信输入使用 `eval`、`source`、`bash -c "$user_input"` 或同类动态执行。
- 调用外部命令时，让失败自然中断或显式传播。只有需要收集失败信息时，才临时关闭 `errexit` 或用条件判断；收集后按原始状态返回。
- wrapper 能把控制权交给下层 CLI 时使用 `exec`；如果持有锁、临时文件、后台进程、端口、trap 清理或收尾日志，就捕获下层状态，清理后 `exit "$status"`。
- wrapper 默认透传下层退出码。下层命令已经开始执行后，wrapper 不得把下层失败重新映射成成功或模糊失败。
- shell 不适合承载复杂业务模型、JSON/YAML/CSV 非平凡读写、DB 查询、HTTP 响应解析、业务诊断或统计聚合；这些能力应下沉到 Python 或正式 CLI。

## 退出码合同

退出码由入口合同 owner 负责。wrapper 已把控制权交给下层 CLI、远端脚本或外部工具后，应透传下层退出码，并在 help 中说明透传边界。

新增或修改稳定入口时，应优先收敛到本节语义。既有入口暂时不能迁移时，必须在 help 或长期合同文档中声明实际退出码，不得让调用者猜测。

仓库入口自发状态推荐收敛到：

| 退出码 | 含义 |
|---|---|
| `0` | 成功。 |
| `1` | 脚本运行成功，但检查不通过、期望不满足或业务语义失败。 |
| `2` | 参数、用法、配置、安全确认或静态前置条件错误。 |
| `3` | 入口自己稳定判定目标对象、目标状态、运行模式或用户选择导致动作无法继续，例如对象不存在、用户取消、锁冲突、运行模式冲突、环境保护拒绝。 |
| `4` | 入口自身发起运行后遇到外部依赖、文件产物、报告读取、网络、数据库、子任务或运行时资源失败，且不是下层退出码直接透传。 |
| 其他非 0 | 底层工具或子任务透传；help 必须说明。 |

判定顺序：

```text
已经交给下层命令执行
  -> 原则上透传。

尚未执行下层命令，且是参数、配置、确认或静态前置条件错误
  -> 2。

入口能稳定判定对象不存在、用户取消、锁冲突、运行模式冲突或保护拒绝
  -> 3。

入口自身运行过程中遇到外部系统、网络、报告、文件产物或子任务失败
  -> 4。

脚本正常完成检查，但结果不满足预期或业务语义失败
  -> 1。
```

能归入 `2`、`3` 或 `4` 的失败不得降级成 `1`。危险运行环境按来源区分：缺少依赖、权限、必要环境变量、非法配置或静态环境检查失败归 `2`；入口已识别目标对象、运行模式或资源状态，并因保护策略拒绝继续，归 `3`。动作前探测按目的归 `2/3`；进入自身运行阶段后的外部失败才归 `4` 或透传。

## 实现模式

单 shell：

- 适合外部命令编排、环境保护、生命周期动作、轻量入口、简单 `case` 分发、PID/端口/日志/运行模式检查。
- 不适合复杂参数解析、结构化数据非平凡读写、DB/HTTP 解析、业务诊断、统计聚合、大量复制粘贴子命令。
- 推荐骨架：`shebang + set -euo pipefail`、根目录定位、公共 helper、usage、`run_<command>`、`case` 分发。
- 非法命令把 help 输出到 `stderr` 并返回 `2`；已停止、空结果、只读状态可以人读展示，但不能掩盖真正错误。

纯 Python 脚本：

- 适合单一职责原子工具，例如格式化、解析、扫描、配置校验、文件检查、端口检查、URL 编码、迁移元数据验证。
- 适合结构化数据、文件格式、网络响应、路径处理、JSON 输出或可测试函数；CLI 表面较薄。
- 推荐骨架：`build_parser()`、业务函数、`main(argv: list[str] | None = None) -> int`、`raise SystemExit(main())`。
- 使用 `argparse` 承担参数事实源；用户输入错误转 `stderr` 和 `2`；检查不通过但运行正常返回 `1`。
- 当出现多命令、长期用户合同、复杂 help、安全确认组合或多输出格式时，升级为 Typer CLI。

shell + py：

- shell 仍拥有入口、help、任务顺序、环境保护、外部命令编排；Python 只做单一复杂能力。
- Python 可以用 `argparse`，不必为了单个动作引入 Typer。
- shell 调 Python 时使用数组或直接参数转发，保留 Python 的 `stderr` 和退出码。
- 人读模式 shell 可以包装阶段标题；`--json` 或机器模式下 shell 不得向 `stdout` 写任何非 JSON 内容。
- Python 侧开始承担统一 `--json`、exit code、help 或稳定命令树时，升级为正式 CLI。

shell + CLI Typer：

- Typer 拥有命令树、参数类型、默认值、help、JSON、人读输出、安全确认、退出码和结构化读写。
- shell wrapper 只定位根目录和 Python，设置必要环境，加载最小 helper，然后 `exec "$python_bin" -m package.cli "$@"`。
- shell wrapper 不重复解析 Typer 参数，不维护 Typer 命令目录，不解析业务 JSON/DB/HTTP 响应，不把写操作确认、远端允许或真实费用确认藏在 shell 层，不污染 Typer 的 `--json stdout`。
- Typer 命令无参数时必须明确行为：显示 help 并返回 `2`，或执行稳定默认只读命令。

## 验证合同

验证按合同面选择最小必要命令，不只机械运行 checklist。声称完成前必须说明哪些合同面已验证，哪些因工具链或环境缺失未验证。

基础验证：

```bash
bash -n <changed-script>.sh
shellcheck <changed-script>.sh
./<changed-entry>.sh -h
python -m py_compile <changed-file>.py
python <changed-file>.py -h
```

按实际入口选择：shell 入口验证语法、shellcheck、help；纯 Python 入口验证 py_compile 和 Python help；Python 由 `.sh` wrapper 暴露时，同时验证 wrapper help。`shellcheck` 是默认验证项；项目工具链暂未提供时，完成说明必须写明未运行原因，不得说成已验证。

按改动合同面补齐：

- 参数：合法参数路径和预期失败参数路径。
- 配置：固定优先级、`--profile`、缺失或非法配置、默认 `.env` 或 const 默认值不会单独决定高风险目标。
- 输入：合法输入和非法输入。
- 人读输出：状态、摘要、关键对象、路径和失败原因可扫描。
- JSON：`stdout` 是纯 JSON，诊断不污染机器输出；自动化消费时补字段、golden 或 schema 断言。
- 产物：路径可发现，必要时确认文件存在、权限和覆盖行为。
- 副作用：高风险命令至少验证未确认时拒绝执行；安全时再验证确认路径。
- 退出码：预期失败路径输出到 `stderr`；涉及退出码分类时断言预期码或透传边界。

修改公共 helper、入口分发、help envelope、输出合同、安全边界或多脚本共享规则时，应运行项目已有最小集成验证；没有时，至少补齐语法检查、help smoke 和关键只读命令 smoke。

## 漂移检查

审查脚本变更时，优先检查：

- help、README、长期文档和实现是否重复维护同一命令目录。
- shell wrapper 和 Python CLI 是否重复维护参数、命令索引、JSON 字段或退出码。
- 默认人读输出是否退化成 CSV 直排、原始 JSON 或未整理工具噪声。
- `--json` 是否被阶段标题、warning、确认提示或日志污染。
- 配置优先级、profile 映射和输入格式是否只存在于实现里。
- 产物路径是否只存在于实现里，成功输出或 help 没有告诉用户位置。
- 只读入口是否顺手安装、下载、写库、启动服务、清理远端或修复环境。
- 退出码是否把参数错误、对象不存在、外部依赖失败和业务检查失败混成同一种失败。
- 验证是否只跑 help smoke，没有覆盖失败路径、JSON 纯度、产物发现或副作用保护。
