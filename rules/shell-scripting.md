# Shell 与 Python 脚本稳定合同

脚本入口不是临时命令拼装，而是一组稳定行为合同。入口应明确自己如何被调用、读取什么、输出什么、产生什么产物、有什么副作用、失败时如何判断，以及用什么验证防止漂移。

本文默认面向仓库内项目脚本。仓库脚本应定位项目根目录并复用公共 helper；独立运维脚本或单文件工具可以改为定位脚本目录、安装前缀或显式配置目录，但仍应遵守合同、错误、输出、退出码和安全边界规则。

本文不是完整 shell 教程，也不规定每个产物的文件名或目录布局。它约束的是入口行为维度和合同稳定性。

## 核心模型

写脚本前先判断两个问题：

```text
这个入口的合同 owner 是谁？
  shell、Python 脚本、Python 辅助能力，还是正式 Python CLI。

这个入口需要稳定哪些合同面？
  help、入参、输入源、人读输出、JSON、产物、副作用、退出码、验证。
```

四类模式不是按技术偏好选择，而是按职责复杂度和用户合同选择：

| 模式 | 用户入口合同归属 | 硬判定 |
|---|---|---|
| 单 shell | shell | 命令主要是外部进程编排、环境保护、生命周期动作或轻量分发。 |
| 纯 Python 脚本 | Python 脚本自身 | 单命令，Python 自身即可完成，不依赖 shell 编排；用户直接使用 Python 参数、help、输出和退出码。 |
| shell + py | shell | shell 负责入口、help、任务顺序、环境探测和外部命令编排；Python 只做结构化计算、检查或格式处理。 |
| shell + CLI Typer | Python CLI | Python 直接承载用户 CLI 合同；出现多子命令、多个输出格式、确认参数、复杂 help 或长期稳定接口时使用。 |

如果 shell 只是薄包装，而参数、help、输出、退出码和安全确认已经由 Python 维护，应按 `shell + CLI Typer` 或纯 Python CLI 思路收口，不要继续把它当作 `shell + py`。

反过来，只要 shell 暴露了用户可见参数、help、确认语义或输出合同，这个入口就不再是纯 Python 脚本；此时应按 `shell + py` 或 `shell + CLI Typer` 维护 wrapper 边界。

## 稳定合同模型

一个脚本入口至少要能回答下列合同问题。`-h|--help` 是展示和发现合同的重要入口，但不是唯一合同事实源；实现、helper、CLI parser、长期文档和验证也必须与合同一致。

| 合同面 | 必须回答的问题 | 典型事实源 | 最小验证 |
|---|---|---|---|
| Help 合同 | 入口是什么、最小怎么开始、风险在哪里。 | `-h|--help`、子命令 help。 | help smoke。 |
| 入参合同 | 支持哪些命令、参数、默认值、互斥关系、确认参数和非交互行为。 | shell dispatch、`argparse`、Typer、Click。 | 正常参数和预期失败参数。 |
| 输入源合同 | 读取哪些文件、目录、URL、环境变量、stdin 或远端资源；格式和失败方式是什么。 | parser、配置读取函数、长期文档。 | 合法输入和非法输入 smoke。 |
| 人读输出合同 | 默认 stdout/stderr 如何让人判断状态、摘要、对象、路径和失败原因。 | 输出 helper、入口实现、help 输出说明。 | 默认输出 smoke。 |
| JSON 合同 | 是否支持 `--json`、stdout 谁拥有、字段语义和诊断边界是什么。 | CLI parser、格式化模块、自动化调用方。 | JSON 纯度和字段 smoke。 |
| 产物合同 | 会生成哪些文件、日志、PID、manifest、CSV、HTML、下载目录或临时保留产物；路径如何发现。 | 入口实现、help、成功输出、长期文档。 | 产物存在性和路径可发现性。 |
| 副作用合同 | 是否写库、删除、恢复、上传、远端写入、远端生命周期动作、启动进程、触发费用、迁移或真实任务。 | help、确认参数、保护检查。 | 拒绝执行路径和确认路径。 |
| 退出码合同 | 各类失败如何分类，何时透传下层命令。 | 入口实现、help、测试。 | 失败分类 smoke。 |
| 验证合同 | 哪些命令证明上述合同没有漂移。 | `verify.sh`、CI、脚本 smoke。 | 语法、help、成功、失败和机器输出检查。 |

稳定合同必须有 owner。一个合同面只能有一个事实源；其他层只能转发、补足或引用。不得在 shell wrapper 和 Python CLI 两层重复维护命令目录、参数列表、JSON 字段语义或退出码说明。

## 入口演进建议

入口形态按职责域、生命周期、安全边界和用户合同自然演进，不按代码行数、目录整齐或实现方便程度决定。以下是推荐路线，不是强制结构。

- 功能少、职责单一、合同简单时，可以保留独立小入口，不必提前聚合到总脚本。
- 同一职责域内动作增多，且共享运行环境、安全边界、输出和退出码合同时，优先考虑稳定入口加子命令。
- 无法自然归入现有入口的功能，可以新增顶层入口；新增时应能说明职责域、生命周期、安全边界或用户合同差异。
- 入口负责定义职责域，子命令负责定义具体动作或工作流；子命令不应成为无边界的杂项集合。
- shell 子命令树复杂到参数互斥、输出格式、安全确认或命令索引难维护时，建议升级为 Python CLI 或 Typer；shell 只保留 wrapper 和运行环境保护职责。

## 通用规则

顶层入口按职责边界拆分，不按实现方便程度拆分。新增或合并顶层入口前，先判断它和已有入口是否属于同一职责域、生命周期、安全边界和用户合同。

shell 入口默认使用：

```bash
#!/usr/bin/env bash
set -euo pipefail
```

纯 Python 脚本默认使用 `main() -> int` 和 `raise SystemExit(main())`。需要直接执行时可以加 Python shebang，但仓库内工具优先通过项目入口或 `python path/to/script.py` 调用，避免多个入口事实源。

入口脚本应先定位运行根目录，再加载公共 helper。不要在多个入口中重复实现路径解析、Python 定位、输出格式、环境读取或安全校验。

错误必须快速暴露。非法参数、缺少依赖、非法配置、危险运行环境和互斥选项应直接失败，不做 silent fallback、自动修正或吞错兼容。

wrapper 默认不得改写下层退出码。能把控制权交给下层 CLI 时使用 `exec`；不能 `exec` 时，捕获状态后必须 `exit "$status"`。清理逻辑、`trap`、日志收尾和提示信息不得覆盖原始失败码。

只有 wrapper 尚未调用下层命令时，前置校验失败才能使用 wrapper 自己的退出码。下层命令已经开始执行后，wrapper 不得把下层失败重新映射成另一个成功或模糊失败。

## Help 合同

`-h|--help` 不是完整手册，而是可执行入口的合同 envelope。用户打开 help 后，应能判断这个入口负责什么、最小怎么开始、默认会读什么和写什么、输出在哪里、哪些动作有风险、失败时看哪个退出码。

Help 合同只展示用户调用需要知道的合同面；长期解释、业务背景、排障手册和完整业务协议放到长期文档。实现变更影响调用、解析、产物追踪、安全边界或失败处理时，必须同步更新 help 或被 help 指向的合同事实源。

### 必选信息

所有入口都应覆盖：

- `Usage / 用法`
- 一句话职责
- `Options`、`Arguments` 或 `Commands`
- `输出`
- `Exit Codes`

### 条件必选信息

存在对应行为时必须覆盖：

- 有职责误用风险时，写 `作用域` 和 `不负责`。
- 有环境或配置依赖时，写 `运行环境`、`配置与环境变量`。
- 有默认动作、默认窗口、默认 limit 或默认输出模式时，写 `默认行为`。
- 读取输入文件、manifest、CSV、HTML、目录、stdin 或 URL 时，写 `输入要求`。
- 有日志、PID、manifest、CSV、HTML、下载目录等文件产物时，写 `运行产物`；如果精确路径只在运行后才能确定，help 必须说明产物类型、触发条件和路径发现方式。
- 有写库、上传、费用、远端写入、远端生命周期动作、启动/停止进程、迁移或真实任务触发时，写 `副作用与保护边界`。
- 有聚合命令或生命周期命令时，写 `成功标准`。

脚本族入口通常需要更完整的 envelope；单一小工具可以只保留必选信息和少量条件必选信息。

### 顶层 Help

顶层 help 回答“这个入口是什么、有哪些命令、最小怎么开始”。推荐区块顺序：

```text
Usage / 用法
一句话职责
Options / 选项
Commands / 命令
作用域
不负责
运行环境
默认行为
配置与环境变量
输入要求
输出
关键概念
成功标准
运行产物
副作用与保护边界
常用示例
进阶用法
Exit Codes
```

顶层 help 规则：

- `Usage`、`Options`、`Commands` 只负责命令形态和索引，不塞长解释。
- `作用域` 和 `不负责` 必须切清职责边界，尤其是部署、远端、写库、费用和生产运维等易误用领域。
- `默认行为` 必须说明无参行为、默认窗口、默认 limit 和默认输出模式；没有默认动作时可以省略。
- `配置与环境变量` 说明加载顺序、关键环境变量和运行时覆盖规则。
- `输入要求` 说明输入文件、目录、stdin、URL、manifest 或远端资源的格式、路径解析基准和非法输入行为。
- `输出` 明确 `stdout`、`stderr`、`--json`、日志和文件产物边界。
- `运行产物` 说明用户需要知道的持久产物、临时保留产物、日志、PID、manifest、报告或下载目录。
- `副作用与保护边界` 覆盖写库、上传、费用、远端写入、远端生命周期动作、启动/停止进程、迁移、真实任务触发、幂等性和 fail-fast。
- `常用示例` 选择 3 到 6 条最高频、最低前置条件、可复制修改的路径，不公平枚举所有命令。
- `Exit Codes` 写稳定退出码；若透传底层工具，明确“其他非 0 由子任务返回”。

### 子命令 Help

子命令 help 回答“这个具体动作如何配置、复制和修改”。推荐区块顺序：

```text
Usage / 用法
一句话职责
Arguments / Options
默认行为
输入要求
输出
运行产物
副作用与保护边界
常用示例
补充格式
Exit Codes
```

子命令 help 规则：

- `-h|--help` 必须在任何执行动作前拦截，尤其是写库、上传、远端访问、费用、启动/停止服务、迁移和长耗时任务。
- 至少提供 1 条可复制示例。
- 有 `--json`、过滤条件、确认参数、远端环境、批量输入或输出文件时，提供 2 到 3 条示例。
- 高风险或高复杂度命令可以更多示例，但应按场景分组。
- 复杂输入格式只放最小可用结构；完整业务合同链接到长期文档。
- 子命令 help 可以更具体，但不要重复顶层完整手册。

### 命令索引事实源

命令索引只能有一个事实源：

- 纯 shell 入口可以手写 `Usage`、`Options`、`Commands`、示例和 exit codes。
- `argparse`、Typer、Click 等框架入口，应让框架生成 `Usage`、`Options`、`Arguments` 和 `Commands`。
- shell wrapper + Python CLI 时，wrapper 只负责定位运行时、设置环境和转发参数；help 事实源放在真正 CLI 层。
- 如果框架已经生成 `Commands`，手写 epilog 不再维护“命令列表”“子命令索引”或“命令说明大全”。
- 手写 epilog 只补足框架不擅长表达的内容：作用域、默认行为、心智模型、常用路径、配置加载、输出产物、安全确认、副作用和退出码。

## 入参合同

入参合同定义用户如何调用入口。参数名、默认值、互斥关系、确认参数和非交互行为属于稳定用户合同，不得随实现方便任意变更。

- 参数解析必须有唯一事实源。shell 手写解析、`argparse`、Typer 或 Click 中只能有一层拥有参数合同。
- `-h|--help` 必须在任何执行动作前处理。
- 非法命令、未知参数、缺少必填值、互斥参数冲突、非法枚举值和非法范围必须非 0 退出。
- 布尔风险开关使用清晰的确认参数，例如 `--yes`、`--confirm-*`、`--allow-*`、`--force`、`--dry-run`；不得用隐式环境或交互猜测替代。
- 机器读模式统一使用 `--json` 作为参数名；`--json` 绑定在它所附着的命令层级上，顶层和子命令可各自按需决定是否支持。除非底层正式 CLI 框架已有稳定合同，否则不得为同类能力另造 `--format json`、`--machine`、`--raw-json` 等参数名。
- 改名、删除或改变参数默认值会改变用户合同；除非入口仍是临时内部脚本，否则应在 help、README 或迁移说明中显式说明。
- 临时内部脚本必须同时满足：不被用户直接调用，不被 README、runbook 或 CI 作为入口引用，不被其他稳定入口以外的自动化直接调用，且不承载独立 help、输出、产物、副作用或退出码合同。不满足任一条件时，应按稳定入口维护合同。

## 输入源合同

输入源合同定义入口会读取什么。参数、环境变量、配置文件、manifest、stdin、URL、本地路径和远端资源都必须有明确边界。

- 读取文件、路径、URL、service name、target name 等外部输入时，先校验允许字符或允许集合。
- 配置文件默认按白名单 key 解析；不得对不可信配置使用 `source` 或动态执行。
- 输入文件必须声明格式、路径解析基准、缺失行为、非法格式行为和是否允许空文件。
- 读取环境变量时必须说明关键变量、优先级和运行时覆盖规则；不得默认读取或打印整个环境。
- 网络调用必须显式设置超时和失败判定；不得使用无超时的阻塞调用，不得把 HTTP 失败、DNS 失败、TLS 失败或连接失败当作成功路径继续执行。
- 使用 `curl` 时必须保留证书校验；除非入口合同明确是调试命令，否则不得使用 `--insecure` 或等价跳过校验选项。
- 脚本必须声明自己的平台假设。仓库脚本默认以项目验证环境为准；使用 GNU/BSD 行为差异明显的 `sed`、`date`、`readlink`、`stat`、`xargs` 等工具时，必须写成项目验证环境可稳定运行的形式，或在前置检查中 fail-fast。

## 输出合同

输出合同定义默认人读输出、机器读输出、stdout/stderr owner，以及大字段、敏感字段和工具输出如何展示。

### 人读输出

- 默认输出面向人读，必须能直接判断状态、摘要、关键路径和失败原因；不得默认输出原始 JSON、大段 payload、二进制内容、CSV 字段直排或未整理的工具噪声。
- 正常状态、摘要、路径、表格输出到 `stdout`。
- 错误原因、非法参数、缺少依赖、warning、进度、确认提示和诊断信息输出到 `stderr`。
- 人读输出使用稳定阶段和事件，例如 `section`、`event`、`row`、`detail`。
- 多对象、多资源或多阶段输出必须提供摘要、单位、分组和关键关联关系；例如进程、任务、文件、容器、GPU、端口等列表必须能关联到所属资源或动作。
- 长路径、大字段、重复字段和低频诊断默认应压缩或分层展示；完整值应通过 `--json`、`--verbose`、`--debug` 或入口明确声明的等价模式获取。

### JSON 输出

- `--json` 不要求出现在顶层；它可以定义在顶层默认动作，也可以定义在具体子命令。是否支持 `--json` 由对应命令按需决定。
- 每个支持 `--json` 的命令独立拥有自己的 JSON 合同；字段语义、默认窗口、过滤条件和输出结构不自动继承其他命令。
- `--json` 只绑定它所附着的命令层级。顶层 `--json` 只对顶层默认动作有意义；子命令 `--json` 应定义在子命令自己的 parser 和 help 上。
- 跨级写法例如 `entry --json <subcmd>` 不得依赖框架隐式行为；入口必须明确支持并路由，或按参数错误返回 `2`。
- 支持 `--json` 的命令必须在该命令自己的 help 中可发现。
- 提供 `--json` 时，`stdout` 必须只包含单个完整 JSON 文档，不混入说明文字、日志前缀、人读摘要、前缀或后缀。
- 机器可读模式只能有一个 `stdout` owner。若 Python 或 Typer 输出 JSON，外层 shell 必须保持 `stdout` 静默；阶段标题、日志和诊断只能走 `stderr`，或完全下沉到 Python/Typer 统一输出。
- 不支持 `--json` 的命令必须拒绝该参数并按参数错误返回 `2`，不能静默忽略。
- JSON 字段语义属于机器合同；新增字段通常可以兼容，删除、改名或改变含义必须按入口合同变更处理。

### 敏感输出

- 字段名、参数名或环境变量名命中 `token`、`password`、`passwd`、`secret`、`key`、`credential`、`authorization`、`cookie` 时默认按敏感信息处理。匹配按大小写不敏感的名称片段判断，优先匹配完整字段、分隔符片段和常见后缀；误命中的非凭据字段必须由入口合同明确声明例外。
- 日志和错误信息默认不得输出敏感值原文；确实需要定位时，只能输出字段名、来源、长度、摘要或脱敏后的前后少量字符。
- 脚本不应默认读取、打印或持久化 token、密码、完整请求体、完整模型响应、大型 payload 或二进制内容；必须读取时，只读取入口声明需要的字段或环境变量。

## 产物合同

产物合同定义脚本会留下什么，以及用户如何找到它们。本文不规定产物文件名或目录布局，但要求产物可发现、权限合理、清理语义明确。

- 脚本产生文件产物时，必须保证用户能发现产物路径。路径固定或可预测时，help 或长期合同文档应说明；路径运行后才确定时，成功输出或诊断输出必须写出实际路径。
- 产物包括日志、PID、manifest、CSV、HTML、下载目录、模型 repository、报告、临时保留目录、生成后的输入素材和远端创建的目录。
- 产物应区分默认生成、参数触发生成、临时自动清理和失败后保留。
- 覆盖已有产物必须有明确规则；危险覆盖、删除或远端清理必须需要显式确认参数。
- 含敏感内容的临时文件、日志片段、报告或中间产物必须限制为当前用户可读写；不能保证权限时不得写入敏感内容。
- 临时文件使用 `mktemp`，并用 `trap` 清理；清理逻辑不得吞掉原始退出码。

## 副作用合同

副作用合同定义入口会改变什么。只读入口不得顺手修复环境、安装依赖、下载模型、写库、清理远端或启动后台任务。

- 写库、删除、恢复、上传、远端写入、远端生命周期动作、真实费用、迁移、启动/停止进程、触发真实任务或压测放大必须有显式确认参数。
- 非交互环境下，危险动作没有显式 `--yes`、`--confirm-*` 或等价确认参数时必须直接失败，不得猜测用户意图。
- 确认提示只能输出到 `stderr`；`--json` 模式不得混入确认提示。
- 用户取消必须产生稳定非 0 退出行为，不能伪装成成功或空结果。
- 本地脚本应校验目标资源是否符合本地预期，发现远端或冲突运行模式时直接失败。
- 持有临时文件、锁、子进程、端口、后台任务或远端任务句柄的脚本，必须在 `EXIT`、`SIGINT` 和 `SIGTERM` 下走一致清理路径，并保留原始失败或中断语义。
- 单实例、锁文件或 PID 文件保护必须定义冲突行为：默认发现冲突即 fail-fast；只有显式参数声明等待、重试或抢占时，才允许改变默认行为。
- 发现锁冲突、运行模式冲突或目标资源不属于当前入口管理范围时，按 `3` 处理；不得静默复用、覆盖或清理未知资源。

## Shell 安全基线

- 变量展开默认加引号，例如 `"$path"`、`"$@"`。
- 参数列表默认使用数组或直接参数转发，不用字符串拼接命令。
- 禁止对不可信输入使用 `eval`、`source`、`bash -c "$user_input"` 或同类动态执行。
- 调用外部命令时，让失败自然中断或显式传播状态。只有在确实需要收集失败信息时，才临时关闭 `errexit` 或使用条件判断；收集完必须按原始状态返回。
- wrapper 可以把控制权 `exec` 给下层命令；但如果 wrapper 持有锁、临时文件、后台进程、端口、trap 清理或收尾日志，就不能直接 `exec`，应捕获下层状态，完成清理后 `exit "$status"`。
- shell 不适合承载复杂业务模型、JSON/YAML/CSV 的非平凡读写、DB 查询、HTTP 响应解析、业务诊断或统计聚合；这些能力应下沉到 Python 或正式 CLI。

## 退出码合同

退出码合同由入口合同 owner 负责。下列 `0/1/2/3/4` 是仓库入口自发状态的推荐稳定语义；当 wrapper 已把控制权交给下层 CLI、远端脚本或外部工具后，应透传下层退出码，并在 help 中说明透传边界。

新增或修改稳定入口时，应优先收敛到本节语义。既有入口暂时不能迁移时，必须在 help 或长期合同文档中声明实际退出码，不得让调用者猜测。

退出码应保持稳定：

- `0` 表示成功。
- `1` 表示脚本运行成功，但检查不通过、期望不满足或业务语义失败。
- `2` 表示参数、用法、配置、安全确认或前置条件错误。
- `3` 表示入口自己完成了校验或查询，并能稳定判定目标对象、目标状态、运行模式或用户选择导致动作无法继续，例如查询对象不存在、用户取消、锁冲突、运行模式冲突或环境保护拒绝执行。
- `4` 表示入口自身发起运行后遇到外部依赖、文件产物、报告读取、网络、数据库、子任务或运行时资源失败，且该失败不是底层命令退出码的直接透传。
- 其他非 0 可以由底层工具或子任务透传，但 help 中必须说明。

退出码判定优先级：

```text
已经交给下层命令执行
  -> 原则上透传下层退出码。

尚未执行下层命令，且是参数、配置、安全确认或静态前置条件错误
  -> 2。

入口自己能稳定判定为对象不存在、用户取消、锁冲突、运行模式冲突或环境保护拒绝
  -> 3。

入口自身运行过程中遇到外部系统、网络、报告、文件产物或子任务运行失败
  -> 4。

脚本正常完成检查，但检查结果不满足预期或业务语义失败
  -> 1。
```

能归入 `2`、`3` 或 `4` 的失败不得降级成 `1`。`4` 只用于入口自身能稳定归因、且不破坏下层退出码透传原则的运行失败。下层命令已经开始执行后，非 0 原则上透传。

危险运行环境按判定来源区分：缺少依赖、缺少必要权限、缺少必要环境变量、非法配置或静态环境检查失败，归入 `2`；入口已经识别到目标对象、运行模式或资源状态，并出于保护策略拒绝继续，归入 `3`。

动作前探测按目的区分：为了判断参数、配置、依赖、权限或目标是否允许执行而做的探测失败，归入 `2` 或 `3`；入口已经进入自身运行阶段后，调用外部系统、网络、报告、文件产物或子任务失败，才归入 `4` 或透传下层退出码。

## 实现模式

### 单 shell

单 shell 适合外部命令编排和轻量入口，不适合承载复杂业务模型。

适合留在 shell 的内容：

- 运行根目录定位、公共 helper 加载和运行时变量组装。
- 简单 `case` 子命令分发。
- `-h|--help` 早拦截。
- 调用 `docker compose`、`pytest`、`alembic`、`curl`、`tail`、`bash -n` 等外部命令。
- PID、日志路径、端口占用、进程状态和运行模式冲突检查。
- 输出阶段 framing，以及必要的工具输出透传。

不适合放在 shell 的内容：

- 复杂参数解析和多层互斥关系。
- JSON/YAML/CSV 的非平凡读写。
- DB 查询、HTTP 响应解析、业务诊断、统计聚合。
- 用字符串拼接实现结构化数据或领域规则。
- 大量复制粘贴的子命令实现。

单 shell 的推荐结构：

```text
shebang + set -euo pipefail
ROOT_DIR 或 SCRIPT_DIR 定位
source 公共 helper
usage / command_usage
run_<command> 函数
case "$command" in ... esac
```

非法命令应把 help 输出到 `stderr` 并返回 `2`。已停止、空结果、只读状态这类可预期结果可以人读展示，但不能掩盖真正错误。

### 纯 Python 脚本

纯 Python 脚本适合单一职责的原子工具，例如格式化、解析、扫描、配置校验、文件检查、端口检查、URL 编码或迁移元数据验证。

适合使用纯 Python 的场景：

- 主要复杂度来自结构化数据、文件格式、网络响应、路径处理、JSON 输出或可测试函数。
- CLI 表面较薄，只有少量参数或一个 `--json` 机器读模式。
- 需要被 shell 入口、CI 或其他 Python 模块调用，但本身不需要成为长期多命令入口。
- 逻辑可以拆成 `build_parser()`、业务函数和 `main()`，便于单测和复用。

不适合继续使用纯 Python 的场景：

- 出现多个稳定子命令、长 help、复杂过滤、分页、批量输入、远端写入、远端生命周期动作、费用或写操作确认组合。
- Python 脚本开始承担直接用户合同，被多个 wrapper 或自动化入口复用。
- `argparse` epilog、互斥参数和示例已经难维护。
- 脚本需要管理服务生命周期、进程、端口、日志、compose 或多个外部命令编排，这类更适合 shell 或 shell + py。

纯 Python 推荐结构：

```text
build_parser() / parse_args()
纯函数或小型业务函数
main(argv: list[str] | None = None) -> int
if __name__ == "__main__":
    raise SystemExit(main())
```

参数和错误规则：

- 使用 `argparse` 承担参数事实源，必要时用 `choices`、互斥组、范围校验和必填组合。
- `-h|--help` 必须由 parser 在任何动作前处理。
- 用户输入错误使用 `ValueError` 或 `parser.error()`，转为 `ERROR: ...` 到 `stderr` 和 exit `2`。
- 检查不通过但脚本运行正常时，逐条输出问题到 `stderr`，返回 `1`。
- 成功返回 `0`；只有明确参数如 `--allow-*` 才允许把非理想状态转为成功。
- 确认参数本身不是升级 Typer 的充分条件；当确认参数叠加多命令、复杂 help、互斥关系、多输出格式或长期用户合同时，才应升级为 Typer CLI。
- 当纯 Python 脚本开始出现多命令、长期用户合同、复杂 help、安全确认组合或多输出格式，应升级为 Typer CLI。

### shell + py

`shell + py` 适合 shell 仍是主入口，但某个子任务需要 Python 完成可靠的结构化处理。

典型边界：

```text
shell
  负责入口、help、任务顺序、环境保护、外部命令编排。
  人读模式可负责 section 输出；机器可读模式必须尊重唯一 stdout owner。

Python
  负责单一复杂能力，例如 URL 编码、配置校验、图片检查、JSON 处理、数据库迁移检查。
```

Python 辅助脚本应保持单一职责。它可以使用 `argparse`，不必为了单个动作引入 Typer。错误处理应把参数、配置和输入错误转为稳定 exit code，例如 `2`；运行失败或外部系统失败可使用更具体的非 0 exit code。

shell 调 Python 时应：

- 先确定项目 Python 或系统 Python，缺失时 fail-fast。
- 使用数组或直接参数转发，避免字符串拼接命令。
- 让 Python 的错误原样暴露到 `stderr`。
- 在人读模式下，shell 可以包装阶段标题；在 `--json` 或其他机器可读模式下，shell 不得向 `stdout` 写任何非 JSON 内容。
- 默认传播 Python 的退出码；不得因为打印了说明或清理了临时文件而返回 `0`。

当 Python 侧成为直接用户合同、被多个 wrapper 或自动化入口复用，开始承担统一 `--json`、exit code、help 或稳定命令树时，必须升级为正式 CLI。升级后 help、参数、输出和错误合同收口到 Python CLI，shell 只保留 bootstrap 和 wrapper 职责。

### shell + CLI Typer

`shell + CLI Typer` 适合长期维护的命令行产品。此时 `.sh` 只是稳定入口，Python Typer 才是命令合同事实源。

shell wrapper 只应做：

- 定位 `ROOT_DIR`、`SCRIPT_DIR` 或安装前缀。
- 加载最小公共 helper。
- 定位 Python，缺失时 fail-fast。
- 设置必要运行环境，例如 `PYTHONPATH`、`PYTHONUNBUFFERED`。
- 使用 `exec "$python_bin" -m package.cli "$@"` 转交控制权。

shell wrapper 不应做：

- 重复 Typer 的参数解析。
- 重复维护 Typer 自动生成的命令目录。
- 解析业务 JSON、DB 结果或 HTTP 响应。
- 把写操作确认、远端允许、真实费用确认藏在 shell 层。
- 在 Typer 的 `--json` 模式下向 `stdout` 写任何额外内容。

Typer CLI 应负责：

- 命令树、参数类型、默认值、help、epilog 和子命令说明。
- `--json` 与人读输出的完整合同。
- 参数校验、互斥选项、安全确认和 exit code。
- 结构化读写，例如 DB、HTTP、OSS、manifest、CSV、报告摘要。
- 业务对象格式化和错误信息。

Typer 命令无参数时应明确选择行为：显示 help 并返回 `2`，或执行一个稳定的默认只读命令。不要让无参数行为依赖隐式猜测。

## 验证合同

验证应按合同面选择最小必要命令，不只机械运行固定 checklist。声称完成前必须说明哪些合同面已验证，哪些因工具链或环境缺失未验证。

新增或修改 shell 入口后至少验证语法、help 和本次改动涉及的合同面：

```bash
bash -n <changed-script>.sh
shellcheck <changed-script>.sh
./<changed-entry>.sh -h
```

修改子命令 help 时，验证对应子命令：

```bash
./<changed-entry>.sh <command> -h
```

修改纯 Python 入口时，至少验证：

```bash
python -m py_compile <changed-file>.py
python <changed-file>.py -h
```

如果该 Python 入口由 `.sh` wrapper 暴露，同时验证 wrapper help：

```bash
./<changed-entry>.sh -h
```

修改 Python 辅助脚本时，至少验证语法和拥有入口合同的 wrapper 或调用方 smoke；辅助脚本只有在自己承载用户入口合同时才需要验证 Python help。

按实际入口选择验证命令：没有 `.sh` wrapper 的纯 Python 入口验证 Python help；有 wrapper 的入口同时验证 wrapper help 和 Python 语法。

`shellcheck` 是 shell 脚本默认验证项。项目工具链暂未提供 `shellcheck` 时，完成说明必须明确写出未运行原因；不得把未运行的静态检查描述为已验证。

修改公共 helper、入口分发、help envelope、输出合同、安全边界或多脚本共享规则时，应运行项目已有的最小集成验证入口。没有项目级验证入口时，至少补齐语法检查、help smoke 和关键只读命令 smoke。

按本次改动涉及的合同面补齐验证：

- 入参合同：至少覆盖一个合法参数路径和一个预期失败参数路径。
- 输入源合同：修改输入文件、manifest、环境变量或 URL 读取时，覆盖合法输入和非法输入。
- 人读输出合同：确认默认输出能扫描状态、摘要、关键对象、路径和失败原因。
- JSON 合同：按支持 `--json` 的具体命令验证，断言 `stdout` 是纯 JSON，诊断信息不污染机器可读输出；被自动化消费的 JSON 命令应补字段、golden 或 schema 断言。
- 产物合同：确认产物路径可发现，必要时确认文件存在、权限和覆盖行为。
- 副作用合同：高风险命令至少验证未确认时拒绝执行；能安全验证时再覆盖确认路径。
- 退出码合同：预期失败路径必须确认错误输出到 `stderr`。改动涉及退出码分类时，必须断言预期退出码或预期透传边界。

高风险脚本还应补充：

- 一个成功路径 smoke。
- 一个预期失败路径 smoke，确认错误输出到 `stderr`，并在入口定义稳定码时断言预期退出码。

## 漂移检查清单

审查脚本变更时，优先检查这些漂移点：

- help、README、长期合同文档和实现是否重复维护同一命令目录。
- shell wrapper 和 Python CLI 是否重复维护参数、命令索引或 JSON 字段语义。
- 默认人读输出是否退化成 CSV 字段直排、原始 JSON 或未整理工具噪声。
- `--json` 是否被阶段标题、warning、确认提示或日志污染。
- 输入文件、manifest、环境变量和配置优先级是否只存在于实现里，help 或长期合同不可发现。
- 产物路径是否只存在于实现里，成功输出或 help 没有告诉用户位置。
- 只读入口是否顺手安装、下载、写库、启动服务、清理远端或修复环境。
- 退出码是否把参数错误、对象不存在、外部依赖失败和业务检查失败混成同一种失败。
- 验证是否只跑了 help smoke，没有覆盖失败路径、JSON 纯度、产物发现或副作用保护。
