# 架构反模式

实现前先检查是否落入这些陷阱。

## 需求混淆

- 没识别真实需求，就直接实现用户提出的机制。
- 没把用户原话翻译成工程需求，就直接给架构结论。
- 只问开放式澄清问题，不给用户可选解释和推荐默认项。
- 把调试脚本当成产品工作流。
- 把模板抽象当成业务 API 合同。
- 把业务项目里的特定实现当成通用平台模式。
- 把单个模式当成完整方案，忽略它必须配套的幂等、重试、观测和恢复路径。

## 可靠性债务

- DB 写入成功后，消息发布可能消失，且没有持久化发布意图。
- 对非幂等任务做盲目重试。
- 长任务只依赖内存状态，不可恢复。
- 吞掉运维必须知道的错误。
- 所有失败一律重试，不区分可重试错误和终态错误。
- 失败任务被隐藏，没有 dead letter、补偿或人工修复路径。

## 数据与计费债务

- 直接更新余额或成本，没有不可变 ledger。
- 当资金准确性重要时，却依赖可能静默失败的日志来计算费用。
- 混合 estimated 和 finalized 成本，但没有状态字段。
- 写了 summary 后丢失原始 usage 或审计事实。

## API 与产品债务

- 客户端需要强类型合同和可发现能力时，却只给一个过度泛化接口。
- 生命周期、权限、状态完全一致时，却为每个类型复制一套 bespoke endpoint。
- 把脚本生成的 summary 当成原始 API 响应展示。
- 未经设计就把内部状态名暴露为公共合同。

## 复杂度债务

- 为本地同步问题引入分布式基础设施。
- 在没有重复使用压力前，先造内部框架。
- 增加会静默改变语义的 fallback。
- 为单一实现创建抽象层，但没有清晰扩展压力。
- 为了“生产级”堆叠模式，但没有明确失败代价、流量规模、团队运维能力和恢复责任。
