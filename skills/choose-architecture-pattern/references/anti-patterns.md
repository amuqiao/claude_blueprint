# 推理类反模式

实现前先检查是否落入这些坑。本文件聚焦"思考过程"的反模式，stack 无关，对 Flutter/移动、前端/Web、Python/后端、全栈都成立。各 stack 框架特有的反模式放在对应 `patterns-*.md` 的"反模式提示"小节，不在这里堆砌。

## 需求混淆

- 没识别真实需求，就直接实现用户提出的机制。
- 没把用户原话翻译成工程需求，就直接给架构结论。
- 只问开放式澄清问题，不给用户可选解释和推荐默认项。
- 把调试脚本或临时诊断工具当成产品工作流。
- 把某个具体项目里的实现当成通用平台模式。
- 把单个模式当成完整方案，忽略它必须配套的幂等、重试、观测和恢复路径。

## 可靠性债务

- 状态写入成功后，依赖的副作用（消息发布、缓存失效、远端同步）可能静默消失，且没有持久化这份意图。
- 对非幂等操作做盲目重试。
- 长任务或长流程只依赖内存/进程状态，不可恢复。
- 吞掉运维或用户必须知道的错误。
- 所有失败一律重试，不区分可重试错误和终态错误。
- 失败的工作被隐藏，没有死信、补偿或人工修复路径。
- 只验证了 happy path，没有考虑崩溃、重启、并发、重复、部分失败下的正确性。

## 数据与一致性债务

- 直接覆盖关键事实（余额、计数、状态），没有不可变记录或乐观并发控制。
- 当准确性重要时，却依赖可能静默失败的旁路（日志、事件）来计算结果。
- 混合中间态和终态，但没有状态字段加以区分。
- 写了派生汇总后，丢失原始事实或审计依据。

## 边界与契约债务

- 调用方需要稳定、可发现的契约时，却只给一个过度泛化的接口。
- 生命周期、权限、状态完全一致时，却为每个类型复制一套 bespoke 实现。
- 未经设计就把内部状态名暴露为对外契约。
- 让契约、实现、类型三方各说各话，没有单一真相源。

## 复杂度债务

- 为本地、同步、低风险的问题引入分布式或重型基础设施。
- 在没有重复使用压力前，先造内部框架或抽象层。
- 增加会静默改变语义的 fallback 或默认值兜底。
- 为了"生产级"堆叠模式，但没有明确失败代价、流量规模、团队运维能力和恢复责任。
