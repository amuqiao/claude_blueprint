# 决策评估清单

架构敏感问题先按这份清单判断，不要直接进入实现。

## 先澄清

- 用户真正需要的结果是什么？
- 用户是否已经把某个实现方式误当成需求？
- 用户表面在问什么，真正要保证什么？
- 哪些事实如果不同，会改变推荐架构？
- 这是产品功能、平台能力、运维保护、迁移方案，还是临时诊断工具？
- 失败影响范围有多大？
- 出问题后谁来排查、恢复和兜底？

## 需求翻译

先把用户输入翻译成工程语言，再判断方案：

- 表面诉求：用户直接问的内容或提出的实现。
- 真实需求：用户真正要保护的业务事实、状态一致性、恢复能力或运维边界。
- 成熟模式信号：这个需求是否对应已有模式，而不是需要自定义机制。
- 非目标：本次不需要保证的能力，避免过度设计。

如果翻译有歧义，先给用户 2-3 个可选解释，并标出推荐默认解释。

## 适配性标准

比较每个方案时至少看这些维度：

- 并发、重试、崩溃、部分失败下是否正确。
- 实现复杂度和运维复杂度。
- 可观测性：能否看到状态、失败、重试、恢复过程。
- 可回滚性和迁移成本。
- 数据一致性要求：强一致、最终一致、best-effort，还是不需要一致性。
- 幂等性和重复请求/重复消息处理。
- 安全边界和滥用风险。
- 扩展路径和瓶颈。
- 成本、延迟和用户体验影响。
- 是否符合当前代码库已有模式。

## 推荐规则

优先采用成熟模式，当：

- 问题是分布式系统或全栈架构里的常见问题。
- 设计必须承受进程崩溃、重启、重复事件或外部服务失败。
- 涉及资金、权限、数据丢失、用户可见状态或合规。
- 运维人员需要清晰地查看和修复失败工作。

优先采用简单本地设计，当：

- 流程是同步的，失败能直接暴露给调用方。
- 功能是内部低风险能力，且可以手动重跑。
- 成熟模式引入的运维面明显大于风险本身。
- 团队还没有能力安全运维该成熟模式。

## 必须保持怀疑

下面这些理由不能单独构成架构决策依据：

- “这样写最快。”
- “队列/库会帮我们重试。”
- “这个应该不会失败。”
- “可观测性以后再加。”
- “用户就是这么要求的。”

先把实现请求翻译回真实需求，再选择方案。

如果用户只是要求“怎么实现 X”，仍然要检查 X 是需求还是手段。只有当 X 已经是合理边界时，才进入实现细节。
