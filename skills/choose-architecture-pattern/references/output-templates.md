# 输出模板

复杂决策默认使用"架构比较"模板；窄问题用"简短建议"；审查既有/拟议设计用"Review 发现"。模板里不要写死任何框架名，框架候选从已加载的 `patterns-*.md` 里取。

## 架构比较

```text
需求翻译：
- 用户原话/表面诉求：...
- 用户提出的实现方式：...
- 我理解的真实需求：...
- 不在解决的问题：...

关键假设：
- ...

问题类别：
- 项目类型：Flutter/跨端移动 | SwiftUI/iOS 原生 | 前端/Web | Python/后端 | 全栈/接缝
- 问题维度：状态 / 数据一致性 / 异步可靠性 / API 合同 / 安全访问 / 可观测运维（取相关项）

命名（用户能描述效果但不知道叫什么时必填）：
- 业界通用名称：...（英文为主，列出所有相关叫法：平台官方命名 + 社区俗称）
- 最典型出处：...（哪个产品 / 设计规范最早或最典型地引入）
- 可搜索关键词：keyword1 / keyword2 / keyword3（3-5 个可直接搜的英文词）

可枚举候选（来自 patterns-<stack>.md + 生态；官方 API 优先，其次高星库）：
- 候选 A｜canonical name：... ｜成熟度：官方 API / star 量级 + 维护状态 ｜适用边界：... ｜坑：... ｜谁在用：...
- 候选 B｜canonical name：... ｜成熟度：... ｜适用边界：... ｜坑：... ｜谁在用：...
- 候选 C｜canonical name：... ｜成熟度：... ｜适用边界：... ｜坑：... ｜谁在用：...

推荐选项（推荐 + 枚举并存，不做盲盒决策）：
- 推荐：从上面候选里选一个 ...
- 为什么：...
- 成立前提：...
- 何时改选其他候选：如果 ... 则候选 B/C 更合适（把决策权交还用户）

需要确认：
- 问题 1：选项 A / 选项 B（推荐）/ 选项 C。
- 问题 2：...

失败模式：
- 崩溃/重启：...
- 重复请求/消息：...
- 并发/竞争：...
- 外部服务或网络部分失败：...
- 运维/用户侧恢复：...

不建议：
- 避免 ...，因为 ...

最小可信落地：
- 步骤 1
- 步骤 2
- 步骤 3

需要验证：
- ...
```
