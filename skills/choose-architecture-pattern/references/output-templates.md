# 输出模板

复杂决策默认使用"架构比较"模板；窄问题用"简短建议"；审查既有/拟议设计用"Review 发现"。模板里不要写死任何框架名，框架候选从已加载的 `patterns-*.md` 里取。

## 架构比较

```text
需求翻译：
- 用户原话/表面诉求：...
- 用户提出的实现方式：...
- 我理解的真实需求：...
- 不在解决的问题：...

关键假设：
- ...

问题类别：
- 项目类型：Flutter/移动 | 前端/Web | Python/后端 | 全栈/接缝
- 问题维度：状态 / 数据一致性 / 异步可靠性 / API 合同 / 安全访问 / 可观测运维（取相关项）

成熟模式候选（来自 patterns-<stack>.md）：
- 模式 A：适用/不适用原因。
- 模式 B：适用/不适用原因。
- 模式 C：适用/不适用原因。

候选方案：
- 方案 A：...
- 方案 B：...
- 方案 C：...

推荐选项：
- 推荐：...
- 为什么：...
- 成立前提：...

需要确认：
- 问题 1：选项 A / 选项 B（推荐）/ 选项 C。
- 问题 2：...

失败模式：
- 崩溃/重启：...
- 重复请求/消息：...
- 并发/竞争：...
- 外部服务或网络部分失败：...
- 运维/用户侧恢复：...

不建议：
- 避免 ...，因为 ...

最小可信落地：
- 步骤 1
- 步骤 2
- 步骤 3

需要验证：
- ...
```

## 简短建议

```text
需求翻译：你真正要解决的是 ...
推荐：选择 ...
理由：...
不建议：避免 ...，除非 ...
需要确认：如果 ... 不成立，方案要调整。
```

## Review 发现

```text
发现：
- 当前/拟议设计存在 ... 风险。

需求翻译：
- 表面上是在 ...，真实风险是 ...

成熟模式：
- ...（来自 patterns-<stack>.md）

为什么重要：
- ...

建议修改：
- ...

剩余风险：
- ...
```
