# 设计文档规范

## §1 两类文档的内容边界

**架构文档**（整个项目一份）：只写骨架，不写实现细节。
判断标准：这个内容会随某个模块的内部变化而变化吗？如果会，它属于模块设计文档。

| 章节 | 写什么 | 不写什么 |
|------|--------|---------|
| 技术选型 | 选了什么技术 + 为什么选 | 该技术如何使用、具体 API 调用方式 |
| 目录结构 | 目录层级 + 每个目录所属的层分类 | 具体文件名 |
| 核心业务流 | 哪个层做了什么步骤、层与层之间的数据流向 | 具体算法名、模型名、函数签名 |

**模块设计文档**（每个功能模块一份，命名 `设计_v1.md`）：该模块的唯一设计基线。
- 负责：架构决策、数据模型、接口契约、状态机、产品规则
- 不负责：UI 布局细节（主参考同目录的 `mock.html`）
- 冲突时：先更新本文档规则，再同步 mock.html

## §2 模块设计文档结构

| 节 | 写什么 | 是否必须 |
|----|--------|---------|
| §1 问题定义 | 核心痛点 + 已知约束（技术、时间、依赖关系） | 必须 |
| §2 目标与非目标 | 目标每条对应一个可验收能力；非目标加 ❌ 前缀，注明不做的原因 | 必须 |
| §3 方案设计 | 核心流程 Mermaid 图；关键决策表；与上下游模块的边界 | 必须 |
| §4 数据模型 | 每张表的完整字段定义 | 有 DB 改动时必须 |
| §5 后端接口规范 | 每个接口：基本信息、请求体、响应体、错误码 | 有接口时必须 |
| §6 前端集成与交互协议 | SSE/WS 协议、状态流转、调用入口映射 | 有复杂交互时必须 |
| §7 备选方案 | 表格（方案 \| 否决理由 \| 可能切换场景），至少 2 个 | 必须 |
| §8 待定事项 | 每条格式：`⏳ 问题描述。**决策时机**：xxx` | 有则必须 |

**迁移草案约束**：不产出独立迁移文件，schema 决策直接写进 §4，代码落库后由 ORM 文件和迁移工具承担。

## §3 数据模型格式

每张表独立一节，字段表格格式固定：

| 字段名 | 类型 | 可空 | 默认值 | 说明 | 约束/索引 |
|---|---|---|---|---|---|
| `id` | `UUID` | 否 | DB 生成 | 主键 | PK |
| `status` | `enum` | 否 | `{初始值}` | 记录状态 | 枚举值：{列出全部} |

规则：
- 枚举类型必须列出所有合法值
- 表之间关联关系用 Mermaid `erDiagram` 补充

## §4 接口规范格式

每个接口独立一节，四项缺一不可：

**① 基本信息**

    Method + Path：{HTTP 方法} /api/{版本}/{resource}
    描述：{一句话说明接口做什么}
    认证：{认证方式}

**② 请求体**：表格（字段 \| 类型 \| 必填 \| 说明 \| 校验规则）

**③ 响应体**（标注成功 HTTP 状态码）：表格（字段 \| 类型 \| 说明）

**④ 错误码**：表格（HTTP 状态码 \| 错误码 \| 触发条件）

## §5 前端集成节触发条件

以下任一情况时必须写 §6：
- 有多步骤状态流转（超出"加载中/成功/失败"三态）
- 涉及 SSE / WebSocket / 长轮询等非标准 REST 交互
- 文件上传进度、流式响应、实时推送

纯 CRUD（请求 → 响应 → 刷新列表）可跳过。

## §6 自检清单

**架构文档**
- [ ] 技术选型：每行只写选型结论和理由，不写使用方式；已否决方案有记录
- [ ] 后端分层：各层均有对应目录；每层有"禁止做的事"
- [ ] 前端分层：各层均有对应目录；每层有"禁止做的事"
- [ ] 全局约定覆盖：命名、错误处理、认证、日志、前后端类型契约
- [ ] 业务流时序图：参与者为层，步骤描述层行为，无具体函数签名

**模块设计文档**
- [ ] §2 目标每条可验收；非目标有否决原因
- [ ] §3 核心流程有 Mermaid 图；跨模块边界已明确
- [ ] §4 每张表有完整字段表格（含可空、默认值、约束）；枚举值已列出
- [ ] §5 每个接口有四项（基本信息、请求体、响应体、错误码）
- [ ] §6 满足触发条件时已写
- [ ] §7 至少 2 个备选方案，格式完整
- [ ] §8 只记录真实未决事项
- [ ] 有前端页面时：同目录存在 mock.html
