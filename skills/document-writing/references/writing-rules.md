# 文档表达规则

## 1. 表达思路

- 写文档时，先帮助读者建立心智模型，再展开细节
- 先给整体理解，再讲主干结构、层级关系、边界区别
- 当主题容易混淆时，先切清对象边界，再展开内容
- 文档应优先降低理解成本，而不是单纯堆信息

## 2. 组织倾向

组织倾向关注整篇文档的骨架和开头组织，不负责逐章检查；章节之间的顺序、分层和跳步检查见“章节逻辑链”。

正式文档应有稳定骨架。优先先主干、后分支，先核心、后补充；高抽象内容应优先转成更容易扫描的结构。

- 对长期维护、说明型、方法论型文档，开头应优先帮助读者快速判断:
  - 这篇文档是什么
  - 它解决什么问题
  - 谁应该读
  - 哪些问题不应由它解决
  - 后续如何维护或扩展
- 正式文档开头推荐优先写一句话主旨，用 1 句话压缩本文的核心作用、认知转化或使用价值
- 根据文档类型按需补充以下开头组件:
  - 文档职责：本文负责什么，不负责什么
  - 适用读者：谁应该读
  - 不适用于：哪些问题不应由本文解决
  - 维护规则：目录入口、索引页、规范文档、方法论集合页等长期维护型文档才需要
- 上述开头组件不是固定模板，不要求每篇文档都写全
- 是否显式写出这些组件，应以是否降低理解成本、明确文档边界、防止文档膨胀为判断标准

## 3. 章节逻辑链

章节不是信息容器，而是读者理解路径上的节点。判断章节是否合理，重点看三件事：

```text
章节职责：每个章节是否承担明确作用
章节顺序：章节是否按读者理解或操作顺序推进
章节分层：主流程、可选流程、补充说明和常见问题是否分层清楚
```

一级章节应承担稳定的逻辑作用，例如建立背景、定义边界、完成主流程、处理分支、验证结果或排查问题。

章节之间应形成连续链条。后文如果依赖某个概念、工具或动作，前文应先完成必要说明。面向新用户、安装配置、操作流程或问题排查的文档，应特别检查是否存在关键跳步。

主流程、可选流程、补充说明和常见问题应分层放置，不要混在同一层级。可选安装、按需处理、暂时可忽略的内容必须显式标注，避免读者误以为都是必做步骤。

质检时优先检查：

```text
每个章节是否有明确职责
章节顺序是否符合读者的实际理解或操作顺序
是否缺少关键前置节点
是否存在主流程和可选流程混排
是否存在后文引用但前文未解释的概念、工具或动作
是否有章节只是堆信息，没有推进理解
```

如果新增、删除或移动章节后，读者完成任务的路径更清楚，说明章节调整是结构优化；如果只是让文档看起来更完整，但没有降低理解成本，应避免增加章节。

## 4. 禁止行为

- 不要堆概念，不要让读者自行拼结构
- 不要一上来陷入细节，而没有先建立整体理解
- 不要为了"看起来完整"而堆无助于理解的内容
- 不要把不同范畴、不同层级的内容硬塞进一篇文档
- 不要让章节只按信息类别堆叠，而没有形成清晰的理解或操作路径

## 5. 可视化原则

- 可视化服务理解，不服务装饰
- 文字图、对比表、结构化清单、Mermaid 都只是表达工具
- 简单结构优先用简单表达，复杂关系再使用更强的可视化方式
- 图示应优先帮助读者理解主干结构、层级关系和边界区别

## 6. Markdown 索引与链接

- Markdown 索引、目录、README 导航默认使用相对路径，不使用绝对路径
- 在仓库内引用其他 Markdown、目录或资源时，优先保证相对链接可随目录迁移而保持有效
- 只有在明确需要依赖外部固定地址时，才使用非相对路径

## 7. 文档文件命名

- 新建文档时，文件名应保持稳定、可预测、便于链接、引用与迁移，并尽量直接表达主题
- 默认优先使用英文小写加连字符，例如：`writing-rules.md`、`api-overview.md`、`deployment-checklist.md`
- 英文命名不是硬性要求；当文档主要面向中文读者，且英文名会显著变长或降低可读性时，可以使用简短、稳定、语义明确的中文文件名
- 技术文档文件名优先采用“主题/技术 + 具体对象/目标 + 文档类型”的自然命名，例如：`swiftui-heatmoment-0-to-1.md`
- 文件名不使用空格，不混用多种分隔方式；如无特殊理由，不使用下划线、驼峰或临时性缩写
- 若仓库内已有明确命名惯例，应优先沿用，保持同层级文档命名一致
