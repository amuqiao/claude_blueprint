# 开发 OS 维护规范

## §1 三层定义

| 层 | 位置 | 判断标准 | 修改频率 |
|----|------|---------|---------|
| 全局层 | `~/.claude/` | 在我的下一个完全不同的项目里也成立 | 低，谨慎修改 |
| 项目层 | `项目根/.claude/` + `CLAUDE.md` | 只在本项目成立，换个项目就不一样 | 中，按项目演进 |
| 文档层 | `项目根/docs/` | 本项目的业务决策和知识沉淀 | 高，随需求迭代 |

## §2 新约束分类决策树

遇到任何新约束，按顺序回答以下问题：

    新约束出现
    │
    ├─ Q1: 这是"让 AI 改变行为方式"还是"记录项目决策"？
    │   │
    │   ├─ 记录项目决策 → Q4
    │   │
    │   └─ 改变 AI 行为 → Q2
    │       │
    │       ├─ Q2: 这条规则在下一个技术栈完全不同的项目里还成立吗？
    │       │   │
    │       │   ├─ 是（通用原则）→ Q3
    │       │   │
    │       │   └─ 否（项目专属）→ 放项目 CLAUDE.md 或 .claude/rules/
    │       │
    │       └─ Q3: 内容超过 3 行吗？
    │           │
    │           ├─ 是 → 放对应 skill（在 CLAUDE.md 保留一行指针）
    │           │
    │           └─ 否 → 直接放 ~/.claude/CLAUDE.md
    │
    └─ Q4: 这是产品/架构决策，还是过程经验/踩坑？
        │
        ├─ 产品/架构决策 → 放 docs/design/ 对应文档
        │
        └─ 过程经验/踩坑 → 放 docs/专题/（只增不改）

## §3 各层内容边界（举例说明）

**应该在全局层（`~/.claude/CLAUDE.md` 或 skills/）的内容**：
- 分层架构约束（入口层不写业务逻辑）
- 文档格式规范（写作规范、设计文档格式）
- 通用脚本规范（批量 API 脚本的模块结构要求）
- 通用工作流原则（先设计文档再动代码）

**应该在项目层（项目 `CLAUDE.md` 或 `.claude/rules/`）的内容**：
- 具体的开发命令（`./scripts/dev.sh start`）
- 具体的日志路径（`backend/logs/celery.log`）
- 目录名到层级的映射（`backend/app/services/` → 应用层）
- 项目专属技术约束（如 `Celery 任务内使用 NullPool`，只在用 Celery 的项目成立）
- 项目专属实现约束（如 `图片必须用 AuthImage`，只在有这个组件的项目成立）

**应该在文档层（`docs/design/`）的内容**：
- 技术选型决策（选了什么技术以及为什么）
- 接口规范（URL、请求体、响应体、错误码）
- 数据模型（表结构、枚举值）
- 产品规则（业务状态机、权限规则）

## §4 迭代 skill 的流程

当发现某个 skill 输出不符合预期：

1. 定位是哪个 skill 的问题
2. 用自然语言描述：`当前行为是什么 / 期望行为是什么 / 反例是什么`
3. 让 Claude 修改对应的 SKILL.md：

       design-doc skill 在输出模块设计文档时，没有提醒要同步 mock.html，
       请在 ~/.claude/skills/design-doc/SKILL.md 的自检清单里加上这条

4. 用同类任务测试修改后的效果

## §5 新增 skill 的时机

满足以下任一条件时，新建一个 skill：
- 某类任务有超过 5 条通用规范，且这类任务会被频繁触发
- 现有 skill 内容超过 500 行，需要拆分
- 遇到现有 skill 都未覆盖的新任务类型

新 skill 的 `description` 字段必须包含：做什么（1 句话）+ 识别方法 + 触发词列表。

## §6 全局 CLAUDE.md 的行数控制

- 上限 80 行（超出说明细节没有下沉到 skill）
- 检查方法：`wc -l ~/.claude/CLAUDE.md`
- 超出时：把某一主题的全部细节提取到新 skill，CLAUDE.md 只保留一行指针

## §7 `.claude/rules/` 的路径匹配机制

rules/ 下的文件**默认全局加载**（每次会话都生效），不会按文件名自动匹配目录。

要实现"只在处理某目录时才执行这条规则"，需在文件头加 YAML frontmatter：

    ---
    paths:
      - backend/**
    ---

    （后续为规则正文）

不加 `paths` frontmatter → 全局规则。

`paths` 的作用不是触发加载，而是告诉 Claude：这条规则只在匹配路径时适用。

维护约定：
- 默认不写 `paths`
- 不要写 `paths: []`
- 只有在不同目录存在不同约束，或两条规则需要明确区分作用域时，才添加 `paths`

## §8 定期回顾（每月一次，约 30 分钟）

1. 检查 `docs/design/INDEX.md` 中有无 ⚠️ 待更新 的文档，补上
2. 看过去一个月的专题文档，提炼可以升格为 skill 规则的经验
3. 检查 `~/.claude/CLAUDE.md` 行数，超 80 行则裁剪
4. 检查各 skill 的触发是否准确（description 字段的触发词是否够用）
5. 运行 `/doctor`（Claude Code 内置命令）检查 skill 描述预算是否溢出
