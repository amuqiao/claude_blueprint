# Design Doc

## 两类文档的内容边界

### 架构文档

项目级架构文档一份即可。它只写系统骨架，不写会随模块内部变化而频繁变动的实现细节。

适合写：

- 技术选型与理由
- 分层结构与目录边界
- 核心业务流、调用链、数据流
- 全局约束：认证、错误处理、日志、契约同步方式

不适合写：

- 具体函数签名
- 模块内部算法细节
- 单个接口的完整字段表
- 页面布局和细部交互

判断标准：

- 如果内容会随某个模块内部实现频繁变化，它更可能属于模块设计文档

### 模块设计文档

每个功能模块一份。它是模块的设计基线，回答“这个模块如何被约束和落地”。

适合写：

- 问题定义和约束
- 目标与非目标
- 核心流程与跨模块边界
- 数据模型
- 接口契约
- 状态机与关键规则
- 备选方案与未决事项

不适合写：

- UI 排版细节
- 单纯页面视觉描述
- 项目级方法论正文

## 模块设计文档建议结构

1. 问题定义
2. 目标与非目标
3. 方案设计
4. 数据模型
5. 接口规范
6. 前端集成与交互协议
7. 备选方案
8. 待定事项

## 数据模型规范

每张表建议至少说明：

- 字段名
- 类型
- 可空性
- 默认值
- 说明
- 约束 / 索引

如果包含枚举，必须列出全部合法值。  
如果包含表间关系，优先补 Mermaid `erDiagram` 或明确关系说明。

## 接口规范

每个接口建议至少包含四部分：

1. 基本信息
2. 请求体
3. 响应体
4. 错误码

## 设计质检最小清单

评审设计文档时，优先检查：

1. 问题是否说清楚
2. 边界是否清楚
3. 模块职责是否冲突
4. 技术方案是否可落地
5. 核心链路是否闭环
6. 关键契约是否完整
7. 风险、取舍和演化是否交代
