# Document Writing

## 适用边界

这份真源只处理：

- README
- 索引页
- 说明型文档
- 方法论文档
- 长期维护的正式文档

如果核心问题是：

- 项目阶段与演进判断 -> 用 `project-methodology`
- 设计文档边界与契约 -> 看 `design-doc.md`
- 代码讲解 -> 看 `code-explain.md`

## 表达思路

- 先帮助读者建立心智模型，再展开细节
- 先给整体理解，再讲主干结构、层级关系和边界区别
- 文档优先降低理解成本，而不是单纯堆信息

## 组织倾向

- 正式文档应有稳定骨架
- 优先先主干、后分支，先核心、后补充
- 高抽象内容优先转成更容易扫描的结构
- 对长期维护、说明型、方法论型文档，优先在开头明确：
  - 文档职责
  - 适用场景
  - 目标读者
  - 维护规范

不是所有文档都必须机械套这 4 项。短文档、索引页、FAQ、快速参考页可以按需简化。

## 禁止行为

- 不要堆概念，让读者自己拼结构
- 不要一上来陷入细节，没有先建立整体理解
- 不要为了“看起来完整”而堆无助于理解的内容

## 可视化原则

- 可视化服务理解，不服务装饰
- 文字图、对比表、结构化清单、Mermaid 都只是表达工具
- 简单结构优先用简单表达，复杂关系再使用更强的可视化方式
- 图示应优先帮助读者理解主干结构、层级关系和边界区别

## Markdown 链接

- 仓库内 Markdown 索引、目录、README 导航默认使用相对路径
- 优先保证文档移动后链接仍然可用

## 文件命名

- 默认使用英文小写加连字符
- 文件名尽量稳定、可预测、便于链接和迁移
- 仓库内已有明确命名惯例时，优先沿用
