# 项目方法论调度 Skill

这是一个**薄调度 skill**。  
方法论真源位于本 skill 的 `references/`。

它不在 `SKILL.md` 正文里重复展开完整方法论。  
它只负责判断：

- 当前问题真正属于哪一层
- 当前应优先调用哪份方法论文档
- 当前最小下一步和最小产物是什么

## Canonical Sources

这个 skill 的方法论真源在本 skill 的 `references/`。

读取顺序：

1. 先读 [`references/source-map.md`](references/source-map.md)
2. 再根据当前问题，到 `references/` 中读取对应的方法论文档
3. 默认只读取一个最相关的真源文档；只有在边界交叉时，才补读第二个

如果当前工作区不存在这些 reference 文件：

- 不要假装知道完整方法论内容
- 先明确说明当前缺少方法论真源
- 仅使用本 skill 里的最小路由逻辑做保守判断

## 标准使用方式

这个 skill 的标准动作顺序是：

1. 先判断当前问题更像项目、模块、基础设施、版本，还是文档问题
2. 输出最小路由结果
3. 读取 `references/source-map.md`
4. 只打开一个最相关的 canonical source
5. 基于真源继续分析，而不是在本 skill 中重复构造方法论

## 什么时候使用

当问题主要是**方法判断**，而不是直接实现时，使用这个 skill。

### 必须使用

- 从一个模糊想法开始做复杂项目、0-1 全栈项目
- 判断当前项目处在哪个阶段
- 判断一个新事项属于项目问题、模块问题、基础设施问题，还是版本问题
- 项目开始变复杂，不知道先收敛版本还是先加功能
- 基础设施开始成为阻塞，但不确定该不该升级
- 新功能要接入已有骨架，但不确定先走哪条方法论

### 推荐使用

- 项目已经有代码，但不知道下一步该先做什么
- 需求不断进入，版本边界开始模糊
- 某个功能看起来不大，但可能影响架构或基础设施
- 不确定应该开新模块、扩旧模块，还是暂时延后

### 不适合使用

- 设计和阶段都已经清楚，只剩纯实现
- 简单 bug 修复，没有阶段歧义
- 纯文档润色，不需要方法论分流
- 纯基础设施执行任务，升级方向已经确定

## 路由逻辑

默认按下面顺序分流：

1. 这是否主要是**项目级**问题？
   则路由到：`项目成型方法论`

2. 这是否主要是**功能模块接入**问题？
   则路由到：`功能模块成型方法论`

3. 这是否主要是**基础设施演进**问题？
   则路由到：`基础设施演进方法论`

4. 这是否主要是**需求 / 版本收敛**问题？
   则路由到：`需求与版本演进方法论`

5. 这是否主要是**输出 / 文档成型**问题？
   则路由到：`文档成型规则`

对应路径见 [`references/source-map.md`](references/source-map.md)。

## 输出契约

使用这个 skill 时，第一步只输出：

```text
当前层次：
当前阶段：
当前最主要问题：
建议先调用的方法论文档：
建议最小产物：
```

只有在这一步完成后，才继续进入对应的方法论文档。

## 最小判断规则

- 如果核心问题是“这个项目应该从哪里开始” -> `项目成型方法论`
- 如果核心问题是“这个功能如何接入现有系统” -> `功能模块成型方法论`
- 如果核心问题是“当前基础设施该保持轻还是升级” -> `基础设施演进方法论`
- 如果核心问题是“这个需求该不该进当前版本” -> `需求与版本演进方法论`
- 如果核心问题是“输出物该怎么写、怎么组织” -> `文档成型规则`

## 重要边界

- 这是一个路由器，不是真源
- 不要在这里复制完整方法论文档
- 不要让所有任务都强制走完整方法论主线
- 永远优先选择最小充分的方法论路径
- 当 `references/` 真源可用时，优先读取真源，不要只靠本文件做推断
