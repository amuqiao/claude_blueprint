# 项目方法论总览

这个 `references/` 目录不是普通资料收纳区，而是 `project-methodology` skill 的**方法论真源骨架**。

它回答的不是某个具体项目怎么写代码，而是：

- 项目怎么起盘
- 模块怎么健康接入
- 基础设施怎么逐步演进
- 文档怎么沉淀成长期资产
- 需求和版本如何持续收敛，而不是越做越散

---

解决：所有输出物如何建立心智模型、组织结构、表达清楚并沉淀成长期资产。

## 怎么搭配使用

### 场景 1：一个项目还只是想法

先用：
- `项目成型方法论`

如果过程中涉及文档输出：
- 同时受 `文档成型规则` 约束

### 场景 2：项目骨架已经有了，要继续加模块

先用：
- `功能模块成型方法论`

如果模块接入中暴露基础设施问题：
- 串 `基础设施演进方法论`

### 场景 3：项目开始变复杂，版本边界容易失控

先用：
- `需求与版本演进方法论`

它不是事后纠偏，而是持续决定：
- 这轮做什么
- 不做什么
- 什么延后
- 什么该拒绝

### 场景 4：基础设施开始成为真实阻塞

先用：
- `基础设施演进方法论`

它解决的不是“选什么中间件”，而是：
- 当前最轻可行方案是什么
- 什么时候值得升级
- 升级前该留什么缝

## 一个稳定心智模型

你可以把这 5 份文档理解成：

- `项目成型`
  负责把项目搭起来

- `功能模块`
  负责把模块健康接进去

- `基础设施演进`
  负责让底层能力按阶段升级

- `需求与版本演进`
  负责决定每一轮该推进什么、该延后什么

- `文档成型规则`
  负责把过程和结果写成长期资产

一句话：

**项目成型是主干，功能模块和基础设施演进是执行支线，需求与版本演进是调度层，文档成型规则是横切表达层。**

---

## 快速使用

- 项目还只是想法：先看 `项目成型`
- 项目骨架有了，要加功能：先看 `功能模块成型`
- 基础设施开始成为阻塞：先看 `基础设施演进`
- 版本开始失控、需求越堆越多：先看 `需求与版本演进`
- 任何输出物怎么写：都受 `文档成型规则` 约束
