# 功能模块成型方法论

> 用于在项目骨架已经存在的前提下，判断一个功能模块如何健康接入现有系统，并在不破坏架构的前提下完成设计、实现、验证与回写。

> 主线框架：`Point -> Fit -> Design -> Integrate -> Implement -> Verify -> Governance`

## 1. 思考顺序

- 做功能模块时，先按 `Point -> Fit -> Design -> Integrate -> Implement -> Verify -> Governance` 的主线顺序思考
- `Point`：先确认当前要做的到底是一个独立模块、一个子功能、一个页面改动，还是一个技术债修复
- `Fit`：再判断它在现有项目骨架里落在哪一层、复用什么、是否会突破边界
- `Design`：再把模块讲清楚，形成可执行设计，而不是靠边写边想
- `Integrate`：再判断它如何接入已有基础设施、接口、数据流和页面结构
- `Implement`：再围绕已明确的接入方式实现，而不是一边实现一边改骨架
- `Verify`：实现后验证主路径、边界和对现有系统的影响
- `Governance`：最后回写模块文档、沉淀约束、更新项目资产，让模块真正被系统吸收
- `Decision`：不是默认主线；只有在模块归位、实现路线、表达载体或复用策略存在明显分叉时，才作为可选分支进入比较与判断
- 不要跳过前置阶段，直接把模块硬塞进现有项目

## 2. 核心方法

- 先判断模块是否成立，再判断怎么做
- 先判断模块如何贴合现有骨架，再谈实现
- 先复用已有能力，再决定是否新增基础设施
- 先让模块与系统边界对齐，再进入编码
- 不要把“加一个功能”误当成“只要把代码写出来就行”

## 3. 模块起点判断

- 第一阶段不要默认当前需求已经是“独立模块”
- 先确认当前对象更像：
  - 一个独立模块
  - 一个已有模块下的子功能
  - 一个页面级增强
  - 一个接口级增强
  - 一个技术债修复或结构重整
- 起点不同，后续归位和设计方式也不同
- 如果当前只是一个零散需求点，应先判断它是否值得抽成独立模块
- 不要把“功能增量”直接等同于“需要新开模块”

## 4. 归位优先原则

- 模块接入前，优先回答：
  - 它落在哪一层
  - 它依赖哪些现有模块或服务
  - 它复用什么
  - 它会改变哪些边界
  - 它是否引入新的长期约束
- 如果这些问题答不出来，模块就还不适合直接进入实现
- 模块设计的第一件事，不是“接口怎么写”，而是“它在系统里放哪”

## 5. 复用与边界意识

- 默认优先复用已有：
  - 项目级分层
  - 基础设施
  - 接口风格
  - 数据模型约定
  - 状态管理或任务流机制
- 只有当现有骨架确实无法承载时，才考虑新增横向能力或基础设施
- 新模块不应通过绕过原有边界来追求短期速度
- 如果模块需要突破既有边界，应显式说明“为什么原边界不够用”

## 6. 设计优先原则

- 模块设计时，优先讲清：
  - 模块真正要解决的问题
  - 它服务的用户动作或系统动作
  - 它和已有模块的关系
  - 页面结构或交互轮廓
  - 接口契约与数据模型
  - 状态变化与异常路径
  - 分层职责
- 如果设计还没闭环，不要急着开始实现
- 如果模块会明显影响交互路径，应尽早用文本树、表格、结构图或 `mock.html` 对齐

## 7. 接入优先原则

- 模块实现前，优先判断：
  - 它接入现有系统的最短路径是什么
  - 哪些基础设施可以直接复用
  - 哪些接入点最容易产生耦合问题
  - 它是否会改变已有模块的责任边界
- 正确问题不是“从前端还是后端开始”，而是“从哪里接入，最不容易破坏系统骨架”
- 不要为了让模块上线，悄悄在多个层打洞

## 8. 实现与验证意识

- 模块实现时，默认优先保证：
  - 主路径可运行
  - 与已有骨架不冲突
  - 模块边界没有被偷偷扩大
- 验证时，除了验证模块本身，还应验证：
  - 是否影响已有主路径
  - 是否引入额外技术债
  - 是否暴露出项目级约束需要回写
- 不要把“功能能跑”误当成“模块已健康接入”

## 9. 治理与回写意识

- 模块完成后，默认继续回答：
  - 哪些模块文档要回写
  - 哪些架构文档要同步
  - 哪些约束值得沉淀成规则
  - 哪些清单或设计索引要更新
  - 这个模块是否改变了原有骨架认知
- 模块完成不应只留下代码，还应留下能支撑下一轮开发的长期资产

## 10. 可视化原则

- 可视化服务理解，不服务装饰
- 涉及模块归位时，优先用文本树、分层图或关系图
- 涉及页面或交互时，优先用页面结构图、交互轮廓或 `mock.html`
- 涉及接口与状态变化时，优先用表格、状态清单或简化流程图
- 涉及模块与已有系统关系时，优先用“已有模块 / 新模块 / 依赖关系”的结构化表达
- 如果没有图会明显抬高理解成本，应明确补图

## 11. 禁止行为

- 不要把功能增量直接当成新模块
- 不要跳过模块归位，直接开始设计和实现
- 不要在没判断复用边界前先造新基础设施
- 不要通过跨层打洞来追求短期速度
- 不要让模块实现反向定义系统骨架
- 不要模块做完却不回写项目资产

## 12. 默认输出要求

- 输出模块方法或模块判断时，优先体现：
  - 当前起点
  - 当前模块在系统中的归位
  - 当前模块与已有骨架的关系
  - 当前设计和接入方式
  - 当前是否已足以进入实现
  - 当前需要补的图示或结构化表达
  - 当前完成后应回写什么
- 如果信息不足，先暴露缺口，再继续展开
- 如果存在多个合理接入路径，应先给推荐路径，再说明备选
- 如果当前更适合先沉淀模块骨架，不要默认直接展开完整实现方案

## 13. 分阶段 Prompt 示例

这一节不是替代上面的规则，而是给出可直接使用的参考模板。

推荐用法：

1. 先把这份规则文档作为高层约束
2. 再按下面的阶段 prompt 去跑具体模块
3. 默认优先按主线分阶段跑，不要一上来把所有问题混在一次输出里

### Phase 0：Point

适用场景：

- 当前只是一个功能想法、需求点或改造点
- 还不确定它是不是独立模块

进入本阶段前，至少确认：

- 当前确实有一个值得进一步判断的功能点
- 你现在缺的是“它到底是不是一个模块”的判断

何时足以进入下一阶段：

- 已经知道它是独立模块、子功能，还是局部改动
- 已经知道它是否值得进入模块级梳理

```text
请先不要直接设计或实现。

请遵守我提供的《功能模块成型方法论》，先按 Point 阶段帮我判断：我当前这个功能点，是否值得长成一个独立模块。

当前起点是：
【把功能想法、需求点、问题或改造点贴在这里】

请按下面顺序输出：
1. 这更像独立模块、子功能、页面增强，还是技术债修复
2. 它是否值得进入独立模块级梳理
3. 如果值得，真正的模块问题可能是什么
4. 如果不值得，应该归到哪里
5. 下一步更适合进入 Fit，还是暂时先停
```

### Phase 1：Fit

适用场景：

- 已经知道当前对象值得继续
- 但还不确定它如何落进现有骨架

进入本阶段前，至少确认：

- 当前对象已经不是完全模糊的功能点
- 已经值得进入模块归位判断

何时足以进入下一阶段：

- 已经知道模块落在哪一层
- 已经知道它与已有模块和基础设施的关系
- 已经知道它主要复用什么，以及潜在边界风险是什么

```text
请遵守我提供的《功能模块成型方法论》，按 Fit 阶段帮我判断这个模块如何落进现有项目骨架。

模块或功能点是：
【把对象写在这里】

当前项目背景是：
【把当前项目的分层、已有模块、基础设施、约束贴在这里】

请按下面顺序输出：
1. 它应落在哪一层
2. 它和哪些已有模块或服务有关
3. 它应复用什么
4. 它会改变哪些边界
5. 它是否值得进入 Design
```

### Phase 2：Design

适用场景：

- 模块已经基本归位
- 现在要把它讲清楚到可执行

进入本阶段前，至少确认：

- 模块归位已经基本成立
- 主要复用关系和边界已经清楚

何时足以进入下一阶段：

- 模块问题、交互、契约、数据模型和职责已经闭环
- 已经知道哪些部分需要图、表或 `mock.html`
- 当前设计已经足以进入接入判断

```text
请遵守我提供的《功能模块成型方法论》，按 Design 阶段帮我把这个模块讲清楚。

模块是：
【把模块写在这里】

当前归位信息是：
【粘贴 Fit 阶段结果】

请按下面顺序输出：
1. 模块真正要解决的问题
2. 它服务的用户动作或系统动作
3. 页面结构或交互轮廓
4. 接口契约与数据模型
5. 状态变化与异常路径
6. 分层职责
7. 哪些部分需要图、表、文本树或 `mock.html`
8. 当前设计是否足以进入 Integrate
```

### Phase 3：Integrate

适用场景：

- 设计已闭环
- 现在需要判断它如何最健康地接入现有系统

进入本阶段前，至少确认：

- 设计已经闭环
- 归位与复用边界已经清楚

何时足以进入下一阶段：

- 已经知道最短接入路径
- 已经知道哪些基础设施可直接复用
- 已经识别主要耦合风险和边界变化点

```text
请遵守我提供的《功能模块成型方法论》，按 Integrate 阶段帮我判断这个模块如何接入现有系统。

模块是：
【把模块写在这里】

当前设计是：
【粘贴 Design 阶段结果】

当前项目上下文是：
【把相关分层、模块、基础设施信息贴在这里】

请按下面顺序输出：
1. 最合理的接入路径
2. 哪些已有能力可以直接复用
3. 哪些地方最容易产生耦合问题
4. 是否需要新增基础设施或新约束
5. 当前是否足以进入 Implement
```

### Phase 4：Implement

适用场景：

- 接入方式已明确
- 现在要开始实现

进入本阶段前，至少确认：

- 接入方式和复用边界已明确
- 当前实现不需要再靠边写边改骨架

何时足以进入下一阶段：

- 模块主路径已具备可运行实现
- 主要边界没有被偷偷扩大
- 已经具备进入 Verify 的条件

```text
请遵守我提供的《功能模块成型方法论》，按 Implement 阶段帮我判断这个模块最合理的实现顺序。

模块是：
【把模块写在这里】

当前接入信息是：
【粘贴 Integrate 阶段结果】

请按下面顺序输出：
1. 最合理的实现顺序
2. 哪些部分应先落地
3. 哪些部分现在不要做
4. 哪些实现动作最容易破坏现有骨架
5. 当前实现是否已具备进入 Verify 的条件
```

### Phase 5：Verify

适用场景：

- 已有一轮实现
- 需要验证模块是否健康接入

进入本阶段前，至少确认：

- 当前已有一轮可运行实现
- 已经具备可验证的主路径

何时足以进入下一阶段：

- 模块主路径已经明确验证
- 对现有系统的影响已基本清楚
- 已经知道该回写哪些资产和约束

```text
请遵守我提供的《功能模块成型方法论》，按 Verify 阶段帮我检查这个模块是否已经健康接入现有项目。

模块是：
【把模块写在这里】

当前实现情况是：
【把实现结果、联调情况、测试情况贴在这里】

请按下面顺序输出：
1. 主路径是否已经跑通
2. 是否影响已有模块或主路径
3. 是否引入新的耦合问题或技术债
4. 哪些问题应回退到 Design 或 Integrate 修正
5. 当前是否足以进入 Governance
```

### Phase 6：Governance

适用场景：

- 模块已经过了一轮验证
- 需要把它回写成长期资产

进入本阶段前，至少确认：

- 当前模块已经过基本验证
- 当前结果值得沉淀，而不是还处于不稳定状态

何时足以形成闭环：

- 该回写的模块文档、架构文档或规则已明确
- 已经知道这个模块是否改变了原有骨架认知
- 已经知道下一轮应继续哪个模块或哪个节点

```text
请遵守我提供的《功能模块成型方法论》，按 Governance 阶段帮我判断这个模块完成后应沉淀什么。

模块是：
【把模块写在这里】

当前结果是：
【把实现、验证、经验、偏差贴在这里】

请按下面顺序输出：
1. 哪些模块文档应回写
2. 哪些项目级文档应同步
3. 哪些约束值得沉淀成规则
4. 这个模块是否改变了原有骨架认知
5. 下一轮最合理进入哪个节点
```
