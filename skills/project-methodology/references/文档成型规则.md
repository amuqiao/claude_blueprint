# 文档成型规则

> 用于从一个模糊起点出发，逐步收拢为正式文档输出时，统一思考顺序、表达方式和分阶段调用方式。

> 主线框架：`Point -> Knowledge -> Positioning -> Understanding -> Output`

## 1. 思考顺序

- 写文档前，先按 `Point -> Knowledge -> Positioning -> Understanding -> Output` 的主线顺序思考
- `Point`：先确认这次整理的起点到底是什么，是一个想法、问题、任务、兴趣点，还是零散观察
- `Knowledge`：先把模糊想法、碎片概念和零散经验收成知识骨架
- `Positioning`：再判断这份文档的职责、生命周期、读者和结构类型
- `Understanding`：基于骨架和定位，先帮助读者建立心智模型，再组织成文
- `Output`：最后判断这次应该沉淀成什么产物，以及用什么表达方式落地
- `Decision`：不是默认主线；只有在确实存在分叉方案、结构路线或输出载体选择时，才作为可选分支进入比较与判断
- 不要跳过前置阶段，直接进入成稿或拍板

## 2. 表达思路

- 写文档时，先帮助读者建立心智模型，再展开细节
- 先给整体理解，再讲主干结构、层级关系、边界区别
- 文档应优先降低理解成本，而不是单纯堆信息
- 如果主题本身还不清晰，先收束主题，不要急着写正文
- 如果文档职责还不明确，先做定位，不要急着展开章节

## 3. 文档定位与生命周期

- 写文档前，先判断它主要是给谁看、解决什么场景、会如何被使用
- 先区分它更偏：
  - 一次性参考
  - 持续学习路径
  - 团队 onboarding 材料
  - 规范 / 规则文档
  - 决策参考文档
- 再判断它更适合：
  - 索引式结构
  - 渐进式结构
  - 任务导向结构
  - 并列对照结构
- 不要把“文档在讲什么”和“文档应该长成什么样”混成一步
- 文档的定位会直接影响结构、章节顺序、示例方式和可视化需求

## 4. 起点判断

- 第一阶段不要默认自己已经有 `Knowledge`
- 先确认当前起点更像：
  - 一个模糊想法
  - 一个具体问题
  - 一个任务需求
  - 一个值得沉淀的观察
  - 一个已经半成型的主题
- 起点不同，后续梳理方式也会不同
- 如果当前只有一个点，而不是一个稳定主题，应先帮助它长成可整理主题
- 不要把“我有兴趣整理它”和“它已经可以直接成文”混为一谈

## 5. 组织倾向

- 正式文档应有稳定骨架
- 优先先主干、后分支，先核心、后补充
- 高抽象内容应优先转成更容易扫描的结构
- 如果一个主题同时包含多个层级，应先拆层，再组织
- 如果一个主题过大，应先收敛成可维护版本，不要为了完整而铺太大
- 默认优先让读者先看懂主干，再决定是否继续深入

## 6. 可视化原则

- 可视化服务理解，不服务装饰
- 文字图、对比表、结构化清单、Mermaid 都只是表达工具
- 简单结构优先用简单表达，复杂关系再使用更强的可视化方式
- 图示应优先帮助读者理解主干结构、层级关系和边界区别
- 涉及层级关系时，优先考虑文字图、ASCII 图或层级图
- 涉及流程和先后顺序时，优先考虑步骤清单、流程图或时序表达
- 涉及并列比较时，优先考虑对比表
- 涉及组成结构时，优先考虑结构图、分块图或图文对照
- 不要为了“图文并茂”而强行增加图示；但如果没有图会明显抬高理解成本，应明确补图

## 7. 讲解方式

- 优先先讲“它是什么、解决什么、在整体里放哪”，再讲“它如何工作”
- 先讲必须先懂的内容，再讲可后补的内容
- 先讲稳定结构，再讲实现细节、扩展分支和边界案例
- 如果读者主要是查阅，优先提高扫描效率
- 如果读者主要是学习，优先建立渐进式理解路径
- 如果读者主要是 onboarding，优先帮助其尽快开始做事
- 如果一个概念容易混淆，应显式给出边界区别，而不要让读者自行拼结构

## 8. 输出意识

- 不要把 `Decision` 当成链路终点
- 默认要继续回答：这次最终应该留下什么产物
- 输出结果可以是：
  - 极简记录
  - 主题文档骨架
  - 学习路径文档
  - onboarding 文档
  - 决策参考文档
  - 索引页或导航页
- 输出结果的粒度应与当前目标匹配，不要默认一上来就写完整长文
- 如果当前更适合先沉淀骨架，也可以只输出骨架文档，而不是完整正文

## 9. 禁止行为

- 不要堆概念，不要让读者自行拼结构
- 不要一上来陷入细节，而没有先建立整体理解
- 不要为了“看起来完整”而堆无助于理解的内容
- 不要把一个点直接当成完整主题
- 不要跳过文档定位，直接进入章节展开
- 不要把暂时不需要的细节提前塞进主干
- 不要把可视化当成装饰素材，而忽略它的理解职责

## 10. Markdown 索引与链接

- Markdown 索引、目录、README 导航默认使用相对路径，不使用绝对路径
- 在仓库内引用其他 Markdown、目录或资源时，优先保证相对链接可随目录迁移而保持有效
- 只有在明确需要依赖外部固定地址时，才使用非相对路径

## 11. 默认输出要求

- 输出正式文档时，优先体现：
  - 当前起点
  - 主题骨架
  - 文档定位
  - 主干结构
  - 边界区别
  - 最终输出形态
  - 必要的可视化建议
- 如果信息不足，先暴露缺口，再继续展开
- 如果存在合理但不同的定位或结构路线，应先给推荐判断，再说明备选
- 如果某部分天然更适合图、表或结构化清单，应主动指出，而不是全部压成大段文字

## 12. 分阶段 Prompt 示例

这一节不是替代上面的规则，而是给出可直接使用的参考模板。

推荐用法：

1. 先把这份规则文档作为高层约束
2. 再按下面的阶段 prompt 去跑具体主题
3. 默认优先按主线分阶段跑，不要一上来把所有问题混在一次输出里

### Phase 0：Point

适用场景：

- 现在还只有一个点
- 还不是稳定主题
- 不确定这件事值不值得整理成文档

```text
请先不要把它当成一个完整主题，也不要直接写文档。

请遵守我提供的文档规则，先按 Point 阶段帮我判断：我当前这个点，是否值得长成一个可整理的主题。

我当前的起点是：
【把你的想法、问题、任务、兴趣点或观察贴在这里】

请按下面顺序输出：
1. 这更像一个想法、问题、任务，还是一个已经成型的主题
2. 它是否值得继续整理成文档主题
3. 如果值得，真正的主题可能是什么
4. 如果还不值得，缺的是什么
5. 如果要继续，下一步更适合进入 Knowledge，还是暂时先停

要求：
- 先判断起点，不要提前进入正文写作
- 不要默认我现在已经有稳定主题
```

### Phase 1：Knowledge

适用场景：

- 只有模糊想法
- 知道一些概念，但很碎
- 还没建立知识骨架

```text
请先不要直接写成正式文档。

你需要遵守我提供的文档规则，先按 Knowledge 阶段帮我梳理这个主题。

我的主题想法是：
【把主题或模糊想法贴在这里】

我当前已知的内容是：
【把你知道的零散概念、片段理解、经验印象贴在这里】

请按下面顺序输出：
1. 我真正想整理的主题可能是什么
2. 它更像独立主题，还是更大主题下的子主题
3. 它在知识体系中的位置
4. 当前必须先有的主干概念
5. 这些概念之间的层级、依赖和边界
6. 我现在已经知道什么
7. 我还缺哪些会影响骨架稳定的关键内容
8. 哪些细节现在可以先暂缓
9. 如果现在开始整理，最合理的“可收敛版本”是什么

要求：
- 目标是先建立知识骨架
- 不要直接进入正文写作
- 如果主题过大，请先帮我收缩
```

### Phase 2：Positioning

适用场景：

- 已经大概知道主题是什么
- 但还不确定这份文档应该成为什么类型
- 不确定它适合什么结构

```text
请先不要展开正文，而是遵守我提供的文档规则，按 Positioning 阶段帮我判断这份文档的职责和生命周期。

主题是：
【把主题写在这里】

当前背景是：
【写清读者、使用场景、维护方式、预期用途】

请按下面顺序输出：
1. 这份文档主要解决什么使用场景
2. 它的核心读者是谁
3. 读者使用它时，主要是为了查阅、学习、快速上手、对齐规则，还是做判断
4. 它更像一次性参考、持续学习路径、团队 onboarding 材料、规范文档，还是决策参考文档
5. 它更适合索引式、渐进式、任务导向式、并列对照式，还是其他结构
6. 为什么这样定位
7. 哪些内容应该进入主干
8. 哪些内容应该降级成附录、补充说明或跳转链接

要求：
- 先做文档定位，不要直接展开章节
- 如果存在两种合理定位，请给推荐和备选
```

### Phase 3：Understanding

适用场景：

- 已经有知识骨架
- 也已经知道文档大致定位
- 现在要把它整理成文档结构

```text
请遵守我提供的文档规则，按 Understanding 阶段帮我把主题整理成适合长期维护的正式文档结构。

主题是：
【把主题写在这里】

当前知识骨架是：
【粘贴 Knowledge 阶段结果】

当前文档定位是：
【粘贴 Positioning 阶段结果】

请按下面顺序输出：
1. 更合理的文档标题建议
2. 这份文档的职责
3. 它到底在讲什么，不讲什么
4. 最适合长期维护的主干结构
5. 哪些章节必须先写
6. 哪些章节可以后补
7. 哪些内容适合以后拆成子文档
8. 哪些部分建议用图、表或结构化清单辅助理解

要求：
- 先帮助读者建立整体理解，再展开结构
- 不要为了完整而把结构铺得过大
- 如果主题混了多个层级，请先拆开
```

### Phase 4：Output

适用场景：

- 已经有骨架、定位和文档结构
- 现在要决定这次最终沉淀成什么

```text
请遵守我提供的文档规则，按 Output 阶段帮我判断：这次最终最适合沉淀成什么产物。

主题是：
【把主题写在这里】

当前背景是：
【把目标、读者、维护方式、是否要长期维护等背景贴在这里】

当前已有结果是：
【可选，粘贴 Knowledge / Positioning / Understanding 阶段结果】

请按下面顺序输出：
1. 这次最适合留下什么结果
2. 为什么是这种输出，而不是更大或更小的输出
3. 它更适合是极简记录、主题骨架、正式文档、索引页、学习路径，还是其他形态
4. 这份输出最合理的粒度和边界是什么
5. 哪些部分必须有
6. 哪些部分可以暂时不写
7. 哪些部分建议配图、表或结构化清单

要求：
- 不要默认必须直接输出完整长文
- 输出形态要匹配当前目标和维护成本
```

### Optional Phase：Decision

适用场景：

- 已经理解主题
- 但遇到了分叉方案、结构路线或输出载体选择

```text
请遵守我提供的文档规则，按 Decision 阶段帮我处理这个主题下的分叉判断。

主题是：
【把主题写在这里】

当前问题是：
【把具体分叉问题写在这里】

当前背景是：
【把约束、目标、读者、维护成本等背景贴在这里】

请按下面顺序输出：
1. 这个问题本质上在决定什么
2. 我当前可能混淆了哪些概念
3. 常见方案有哪些
4. 我最容易遗漏的方案或视角是什么
5. 这些方案分别适合什么场景
6. 应该从哪些维度比较
7. 基于我当前情况，更推荐哪种方向
8. 如果现在还不能稳妥决策，最该继续确认什么

要求：
- 先补齐方案空间，再比较，再推荐
- 不要直接拍板，不要把“常见”当成“最适合”
```

## 13. 完整使用示例

### 示例：Transformer

如果你想梳理 `深度学习模型 Transformer`，更推荐这样使用：

1. 如果当前还只是一个想法，先跑 `Phase 0：Point`
   先确认它是不是一个值得整理的主题
2. 再跑 `Phase 1：Knowledge`
   先确认它在知识体系中的位置、核心主干和当前缺口
3. 再跑 `Phase 2：Positioning`
   判断这份文档更适合写成学习型文档、查阅型文档，还是 onboarding 材料
4. 再跑 `Phase 3：Understanding`
   把骨架和定位收成正式文档结构
5. 再跑 `Phase 4：Output`
   判断这次最终应该沉淀成主题骨架、正式文档，还是学习路径
6. 只有真的出现分叉时，再跑 `Optional Phase：Decision`
   例如是否要重点对比 `RNN / CNN / Transformer`，是否要选 Markdown 还是 HTML

### 示例：直接调用方式

```text
请遵守我提供的《文档成型规则》。

现在先只运行 Phase 0：Point。

我当前的起点是：我想梳理深度学习模型 Transformer。
我当前知道的内容有：attention、self-attention、encoder、decoder、大模型。
但我的认知很碎，我更关注先建立知识骨架，而不是一开始啃完所有细节。

请严格只做 Point 阶段，不要提前进入 Knowledge、Positioning、Understanding、Output 或 Decision。
```
