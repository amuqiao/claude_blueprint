# 文档写作工作流

## §1 核心目标

写文档时，优先保证 3 件事：

1. 文档有稳定骨架
2. 文档先帮读者建立整体理解
3. 文档在需要时适当图文并茂，但不花哨

其中：

- 稳定骨架由 `rules/writing.md` 强约束
- 本 skill 负责把文档写得更好读

## §2 默认写法

正式文档默认按这个思路展开：

1. 先给出这篇文档到底在讲什么
2. 再给读者一个整体理解
3. 再展开主干结构
4. 再讲层级关系和边界区别
5. 最后补充例外、细节或快速参考

重点：

- 不要一上来堆背景和概念
- 不要先讲枝节，再讲主干
- 如果读者还没建立整体理解，就不要急着下钻

## §3 什么时候该补结构化表达

当内容出现下面这些特征时，不要只写连续段落：

- 抽象概念多
- 分层明显
- 边界容易混
- 有优先级、覆盖关系、依赖关系
- 需要横向比较

优先顺序：

1. 结构化清单
2. 对比表
3. 文本树
4. Mermaid

原则：

- 能用简单结构讲清，就不要上复杂图
- Mermaid 只在文字已经开始增加理解成本时使用

## §4 常见写法判断

### 方法类 / 认知类文档

优先写：

- 整体理解
- 主干结构
- 层级关系
- 边界区别

### 流程类文档

优先写：

- 总流程
- 阶段划分
- 分支判断
- 例外情况

### 对比类文档

优先写：

- 对比维度
- 核心差异
- 适用场景
- 推荐结论

对比类内容优先使用表格。

## §5 重构文档时怎么做

如果用户要“整理成更适合阅读的文档”，默认不是润色，而是重构。

顺序：

1. 先找主问题
2. 先补整体理解
3. 重排主干结构
4. 把抽象段落改成更易扫描的结构
5. 最后再修句子

如果一段内容已经让读者难以快速回答下面两个问题，就说明还没整理好：

- 这段在讲哪一层
- 这段和别的层有什么区别

## §6 最小检查点

- 开头是否用了稳定骨架
- 是否先帮助读者建立整体理解
- 是否先讲主干，再讲细节
- 是否把层级关系和边界区别讲清楚
- 是否避免概念堆砌
- 是否只在真正需要时补图或表
