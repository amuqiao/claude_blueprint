# 写作工作流

## §1 文档开头模板

每份文档开头结构固定：先一级标题，再引用块。

```markdown
# 文档标题（中文，与文件名语义一致）

> **文档职责**：[这份文档覆盖什么内容，一句话]
> **适用场景**：[什么时候翻这份文档，一句话]
> **目标读者**：[谁在读，知识背景是什么]
> **维护规范**：[谁改了什么要同步更新哪里，禁止写"保持文档最新"之类的空话]
```

规则：
- `# 标题` 必须在引用块之前，不得省略
- `维护规范` 必须具体说明"谁改了什么要同步更新哪里"

## §2 写作流程

### 新建文档时

1. **确定文档职责** - 明确这份文档的边界
2. **使用标准开头** - 套用 §1 模板
3. **规划主体结构** - 确定 2-4 个主要模块（`##` 级）
4. **填充内容** - 遵循 `rules/writing.md` 的所有约束
5. **自检** - 使用 §3 检查清单

### Review 已有文档时

1. 检查文档开头是否符合 §1 格式
2. 使用 §3 检查清单逐项验证
3. 确认是否遵守 `rules/writing.md` 的约束

## §3 文档检查清单

### 开头格式 ✓
- [ ] 有 `#` 标题在最前
- [ ] 有引用块，包含职责/场景/读者/维护规范
- [ ] 维护规范是具体的，不是空话

### 内容质量 ✓
- [ ] 先结论后展开
- [ ] 语言简洁、可执行
- [ ] 每节聚焦单一主题
- [ ] 内容范围符合文档职责

### 结构规范 ✓
- [ ] `#` 只用于文档标题
- [ ] 层级不超过三层
- [ ] 同层标题不重叠

### 图表质量 ✓
- [ ] 复杂图用 Mermaid
- [ ] Mermaid 用了 `classDef`
- [ ] 没有难看的 ASCII 方框图

### 注释规范 ✓
- [ ] 只写 WHY，不写 WHAT
- [ ] 文件/类注释在名字无法表达职责时才写
- [ ] 行内注释只解释非显然约束

## §4 触发条件

调用本 skill 的场景：
- 用户明确说"写文档"、"新建文档"、"创建文档"
- 用户要求 review 文档格式
- 用户询问文档头部、引用块格式

不调用本 skill 的场景：
- 用户只是问文档内容，不涉及格式
- 用户要求写代码注释（直接应用 `rules/writing.md` 的注释规范）
- 用户要求画图（直接应用 `rules/writing.md` 的图表规范）
