## 文档前置条件

**适用范围**：实现新功能、新模块、对已有模块做结构性修改时触发。纯 bugfix、纯配置调整、纯文案修改默认不触发。

### 新模块：先设计后实现

满足以下条件时，视为需要先建设计文档的新模块：
- 新增后端数据模型或核心数据结构
- 新增 API / 队列任务 / 外部服务集成中的至少一种
- 新增前端页面 / store / service 中的至少一种

开始实现前：
1. 先读取 `docs/design/架构设计方案.md`，确认全局分层边界
2. 在 `docs/design/模块/` 下建设计文档，明确数据模型、接口契约、分层职责
3. 等待用户确认设计方向后再实现

如果设计文档不存在，不要自行跳过设计阶段直接实现。

### 已有模块内扩展：先实现，后同步文档

如果只是：
- 新增字段
- 新增分支
- 调整已有流水线
- 扩展已有页面或服务

则先读取对应设计文档理解边界，再实现；实现完成后必须执行“文档后置检查”。

## 文档后置检查

每次代码落库后、回复用户前，检查本次改动是否影响了以下内容：

- `docs/design/架构设计方案.md`
- 对应模块设计文档
- 对应基础设施设计文档
- `docs/design/INDEX.md`
- `docs/实施清单.md`

判断原则：
- 涉及全局分层、技术选型、核心数据流变化 → 更新 `docs/design/架构设计方案.md`
- 涉及单模块的数据模型、接口、前端集成、内部流程变化 → 更新对应模块设计文档
- 新增任何设计文档 → 必须更新 `docs/design/INDEX.md`
- 当前模块开发任务状态发生变化 → 更新 `docs/实施清单.md`

## 文档边界

- `docs/design/架构设计方案.md` 只记录项目级全局架构
- `docs/design/模块/...` 记录单模块设计，不重复全局架构
- `docs/design/基础设施/...` 记录跨模块基础设施约定
- `docs/专题/...` 记录复盘、踩坑、故障排查，不当作主设计文档

写设计文档前，先确认这次信息应该落在哪一层，不要把所有内容都写进同一份文档。
